Section 2.3: IGP Convergence Tuning.

Convergence isn't a single event — it's a pipeline. Failure detection, then notification, then SPF calculation, then RIB update, then FIB programming. Each stage has knobs. Tuning them wrong gives you either slow failover — which costs revenue — or instability — which also costs revenue. This section covers the convergence pipeline end-to-end for both IS-IS and OSPF, with real SP-grade timer values and the math behind them.

The Convergence Pipeline.

Every link failure traverses five stages in order.

Stage 1 is Detection — BFD or IGP hello timers notice the failure. This takes about 150 milliseconds with BFD.

Stage 2 is Notification — the router generates and floods a new LSP for IS-IS or LSA for OSPF. This takes about 10 milliseconds.

Stage 3 is SPF Computation — all routers in the flooding domain run Dijkstra's algorithm. This takes 50 to 200 milliseconds.

Stage 4 is RIB Update — the routing table is updated with new paths. About 10 milliseconds.

Stage 5 is FIB Programming — the hardware forwarding table is reprogrammed. This is the wildcard — anywhere from 50 to 500 milliseconds depending on the platform.

Total convergence equals the sum of all stages. Typical SP target is under 1 second for any single failure. With optimization, under 200 milliseconds. And with fast reroute mechanisms like TI-LFA, it approaches 50 milliseconds.

Stage 1: Failure Detection.

BFD, Bidirectional Forwarding Detection, is the standard for sub-second failure detection. It runs in hardware on modern linecards.

The SP standard is 50 millisecond intervals with a multiplier of 3, giving 150 millisecond detection time. But be warned: BFD at 50 milliseconds is aggressive. On congested control planes or software-based platforms, false positives will happen. Test before deploying. Some service providers use 300 milliseconds times 3 — 900 milliseconds — as a safer default.

Physical layer detection is fastest — about 10 milliseconds for SONET/SDH, 50 milliseconds for Ethernet. Modern optics report loss-of-light instantly. This is faster than BFD but only catches physical failures, not Layer 3 or forwarding plane issues.

IGP hello timers are the last resort. IS-IS defaults to 10-second hellos with a 30-second hold time. OSPF defaults to 10-second hellos with a 40-second dead interval. Never rely on IGP hellos for fast detection — that's BFD's job.

Stage 2: LSP and LSA Generation and Flooding.

Once a failure is detected, the router generates a new LSP or LSA and floods it to all routers in the domain.

Throttling uses exponential backoff. The first LSP is sent after an initial delay of 50 milliseconds. The second after a secondary delay of 100 milliseconds. Subsequent delays double each time up to a maximum of 5 seconds. The timer resets to the initial value after a quiet period.

Stage 3: SPF Computation.

Both IS-IS and OSPF use the same three-timer backoff model for SPF scheduling: initial wait, second wait, and max wait. The recommended SP values are 50 milliseconds initial, 100 milliseconds secondary, and 5 seconds maximum.

Partial Route Computation, or PRC, optimizes the case where only a leaf prefix changes rather than the topology itself. In that case, a full SPF recomputation is wasteful — PRC only recalculates the affected prefixes.

Stage 4: RIB and FIB Update.

After SPF computes new paths, the routing table and forwarding table must update. RIB update is a software operation, typically 10 to 50 milliseconds. FIB programming is hardware-dependent. Modern NPUs handle bulk updates in about 50 milliseconds. Older TCAM-based platforms take 100 to 500 milliseconds. Software forwarding can take up to seconds.

This stage is often the bottleneck. You can tune SPF to 50 milliseconds, but if FIB programming takes 500 milliseconds, your total convergence is still over 500 milliseconds.

Putting It All Together — The Convergence Budget.

With default timers, total convergence is about 35 seconds — dominated by the 30-second IGP hello dead timer.

With tuned timers — BFD at 150 milliseconds, SPF at 50 milliseconds initial — total convergence drops to 300 to 800 milliseconds.

With FRR mechanisms like TI-LFA, convergence approaches 150 milliseconds. The backup path is pre-computed and pre-installed in the FIB. When BFD detects the failure, traffic switches to the backup in the data plane — no SPF needed.

IP Fast Reroute Mechanisms.

There are three generations of IP FRR.

LFA, or Loop-Free Alternate, defined in RFC 5286, is the simplest. It uses a directly connected neighbor as a backup if that neighbor provides a loop-free path. The coverage problem is that LFA can't always find a backup, especially in ring topologies. Typical coverage: 40 to 80 percent of prefixes.

Remote LFA, defined in RFC 7490, extends LFA by tunneling to a remote repair node via LDP. This covers most gaps that basic LFA misses, with typical coverage of 90 to 95 percent.

TI-LFA, Topology-Independent LFA, defined in RFC 8400, is the gold standard. It uses Segment Routing to construct explicit backup paths via segment lists. It provides 100 percent coverage for any topology.

TI-LFA plus BFD equals sub-50 millisecond failover for any single failure. This is the endgame for SP convergence.

Key Configuration Recommendations.

For BFD timers, use a tiered approach: aggressive on core inter-POP links — 50 milliseconds with hardware offload — and conservative on access-facing links — 300 milliseconds. This avoids the false-positive feedback loop.

For SPF timers, 50 milliseconds initial wait is safe and fast. The risk is minimal because SPF itself is a lightweight computation.

For max-wait across all timers, 5 seconds is the standard ceiling. A lower ceiling means faster recovery from flap storms.

For hello intervals, keep the default 10 seconds. Only go to 1-second hellos if BFD is not available — the CPU cost isn't worth it.

Verification and Monitoring.

The SPF log should show runs triggered by real events, not constant churn. BFD sessions should be stable with zero or near-zero packet loss. TI-LFA coverage should be 100 percent for all protected prefixes. The FIB should show backup next-hops installed. And a convergence test — shutting and re-enabling a link — should complete in under 200 milliseconds with FRR.

War Story: The BFD That Cried Wolf.

A large SP deployed BFD at 50 milliseconds times 3 across their entire backbone. Worked great — until a new software release increased control plane CPU utilization by 15 percent. Suddenly, BFD sessions started flapping during peak traffic hours. Not because links were failing, but because the CPU couldn't process BFD keepalives fast enough.

Each BFD flap triggered SPF, which triggered more CPU load, which caused more BFD flaps. Classic positive feedback loop.

The fix: moved BFD to hardware offload, which was supported on the platform but not enabled, and set a floor of 100 milliseconds for non-critical links. Core inter-POP links stayed at 50 milliseconds with hardware BFD. Access-facing links got 300 milliseconds.

Lesson: BFD timers are not one-size-fits-all. Tiered BFD — aggressive on core, conservative on access — is the right pattern.
