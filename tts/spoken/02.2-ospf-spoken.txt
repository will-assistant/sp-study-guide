Section 2.2: OSPF in Service Provider Networks.

OSPF is the world's most deployed IGP — but in service provider backbones, it plays second fiddle to IS-IS. That doesn't mean you can ignore it. OSPF dominates enterprise aggregation, metro Ethernet access layers, and many regional and tier-2 SP networks. Understanding where OSPF fits — and where it breaks — in SP-scale environments is essential for anyone sitting a IE-SP or IE-SP exam. You'll encounter it. You need to own it.

Where OSPF Lives in SP Networks.

OSPF rarely runs as the core backbone IGP in tier-1 providers. Where you will find it: metro and access aggregation — PE-to-CE or PE-to-aggregation in metro Ethernet rings. Regional ISPs — smaller providers with hundreds, not thousands, of routers. Data center underlay — especially with OSPFv3 for IPv6 fabrics. Acquired networks — post-M&A, you inherit whatever the other guy ran. And legacy cores — migrating from OSPF to IS-IS is painful enough that some just don't.

That said, OSPF is not technically inferior in all cases. For networks under about 500 routers with straightforward hierarchy, OSPF works fine.

OSPF Packet Types.

OSPF uses five packet types. Type 1 is Hello, for neighbor discovery and keepalive. Type 2 is DBD, or Database Description, for LSDB summary exchange. Type 3 is LSR, Link State Request, to request specific LSAs. Type 4 is LSU, Link State Update, which carries the LSAs. And Type 5 is LSAck, to acknowledge received LSAs.

LSA Types — The Full Picture.

This is where OSPF gets complex. Every LSA type exists because someone needed to solve a specific flooding or reachability problem. In SP networks, you must know all of them cold.

Type 1 is the Router LSA, originated by every router, flooded within the area, describing links and costs for the originating router.

Type 2 is the Network LSA, originated by the designated router, flooded within the area, representing a multi-access segment.

Type 3 is the Summary LSA, originated by an ABR, flooded inter-area, carrying prefixes from other areas.

Type 4 is the ASBR Summary, originated by an ABR, flooded inter-area, providing reachability to ASBRs.

Type 5 is the AS External LSA, originated by an ASBR, flooded AS-wide, carrying external routes with type 1 or type 2 metrics.

Type 7 is the NSSA External, originated by an ASBR in an NSSA area, flooded only within the NSSA area, and converted to Type 5 at the ABR.

Types 9, 10, and 11 are the Opaque LSAs — link-local, area-scoped, and AS-wide respectively. These are how OSPF supports MPLS-TE and Segment Routing. Without opaque LSAs, OSPF can't play in modern SP networks.

Area Design for Scale.

OSPF mandates Area 0 as the backbone. All inter-area traffic transits Area 0. This creates a single flooding domain at the core — the exact place where you want the least churn.

The SP-scale implications: every link-state change in Area 0 triggers SPF across all Area 0 routers. ABRs must maintain the LSDB for Area 0 plus their attached areas. And virtual links — the duct tape of OSPF — are fragile and operationally painful.

For area types: Standard areas allow everything. Stub areas block external routes but allow inter-area Type 3 LSAs plus a default route. Totally stubby areas block both external and inter-area routes, leaving only a default. NSSA areas allow external injection via Type 7 LSAs that get converted to Type 5 at the ABR. And totally NSSA combines NSSA with totally stubby behavior.

SP design guidance: keep Area 0 lean — only P routers and PE uplinks. Use NSSA for areas that originate external routes. Use totally stubby or totally NSSA for pure leaf sites to minimize LSDB size. And never use virtual links in production if you can avoid it. Redesign instead.

Summarization at ABRs.

Unlike IS-IS, which has no inter-level summarization by default, OSPF requires you to think about summarization at every ABR. Good summarization at ABRs is the number one way to keep OSPF stable at scale. Bad summarization is the number one way to get paged at 3 AM.

OSPF Convergence Mechanics.

Modern OSPF implementations use throttled SPF with exponential backoff. Cisco defaults are aggressive at 5 seconds initial delay. Junos defaults to 200 milliseconds. For SP use, tune to 50 millisecond initial delay, 100 millisecond secondary, and 5 second max holdtime. RFC 8405 defines this same backoff algorithm used by IS-IS.

BFD integration is essential for sub-second failure detection. BFD plus OSPF gives you approximately 150 millisecond failure detection — 3 times 50 milliseconds. Compare that to the OSPF dead timer default of 40 seconds. Night and day.

For fast reroute, Loop-Free Alternates — or LFA — provide pre-computed backup paths. LFA gives you sub-50 millisecond failover without MPLS-TE FRR. Remote LFA extends coverage when no direct LFA exists.

OSPFv3 — The IPv6 Story.

OSPFv3, defined in RFC 5340, was a full rewrite to support IPv6. It runs over IPv6 link-local addresses, uses IPsec for authentication instead of in-protocol authentication, and supports multiple instances via an Instance ID field. With Address Families defined in RFC 5838, OSPFv3 can carry both IPv4 and IPv6 in a single process. However, most service providers just use IS-IS with multi-topology instead.

Key Configuration Recommendations.

Always set the reference bandwidth to 1 terabit to differentiate modern link speeds. The 100 megabit default makes 10 gig and 100 gig links the same cost — that's how you get asymmetric routing nightmares.

Always use point-to-point on SP links to eliminate DR/BDR election and get faster convergence. Enable BFD on all production links. Tune SPF timers to 50/100/5000 milliseconds. And enable prefix suppression to reduce transit link LSAs.

Verification and Monitoring.

For health checks: all expected neighbors should be in FULL state. LSDB counts should be stable, not incrementing rapidly. SPF run count should be low and stable. BFD sessions should be Up on all OSPF interfaces. No EXSTART or EXCHANGE stuck states — those indicate MTU mismatch.

Troubleshooting Common Issues.

Neighbors stuck in INIT usually means an ACL is blocking OSPF protocol 89 or multicast addresses. Neighbors stuck in EXSTART is the classic MTU mismatch indicator. Flapping adjacencies point to unstable links or overly aggressive BFD timers. Missing routes from other areas suggest the ABR isn't in Area 0 or there's filtering. And if SPF is running constantly, you have a flapping link in the flooding domain — find the churning LSA.

War Story: The MTU Mismatch That Wasn't.

A regional ISP reported intermittent OSPF flaps on their ABR links. Neighbors would establish, exchange a few LSAs, then drop back to EXSTART. Classic MTU mismatch, right? Except both sides showed 9000 bytes.

Turns out, the DWDM transport between sites was doing transparent Layer 2 switching with a 16-byte overhead for OAM. The physical MTU was 9000, but the effective payload was 8984. OSPF's DBD packets would sometimes exceed this depending on LSDB size. The fix: set interface MTU to 8900 on both sides, or better yet, fix the transport config.

Lesson: never trust the MTU your interface reports. Test with large pings across the path. Ping size 8972 with the do-not-fragment bit set is your friend.

Review Questions.

Question 1: You're migrating a 200-router metro network from OSPF to IS-IS. The network uses NSSA areas for external route injection at access PEs. What's your migration strategy that avoids downtime? What's the IS-IS equivalent of NSSA?

Question 2: A regional SP has 3 POPs connected in a triangle. Each POP has 10 to 15 access routers behind an ABR pair. Total network is about 80 routers. Would you recommend OSPF or IS-IS? Justify your answer considering operational complexity, IPv6 requirements, and future Segment Routing deployment.

Question 3: Your OSPF network has Area 0 with 6 P-routers and 4 ABRs. One ABR reports 3 times the expected LSA count in Area 0. The other 3 ABRs show normal counts. SPF is running every 2 seconds on the affected ABR. What's your triage approach?
