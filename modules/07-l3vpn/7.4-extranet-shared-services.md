# 7.4 — Extranet & Shared Services

## Overview

> Every SP eventually gets the request: "We need Customer A to reach Customer B's database." Or: "All VPN customers need access to a central DNS/NTP/internet gateway." This is extranet and shared services — the art of **controlled route leaking between VRFs** without blowing up isolation. It sounds simple. It breaks things spectacularly when misconfigured. The RT import/export model from 7.1 is your scalpel — extranet just uses it with more precision. Shared services takes it further by establishing a common VRF that every customer accesses. Get RT policy wrong and you've just merged two customers' networks. Get it right and you're selling a premium service on top of basic L3VPN.

> - how VRFs, route targets, and MP-BGP deliver tenant separation

## Theory

### Extranet vs Shared Services

| Concept | What It Is | RT Model | Scale |
|---------|-----------|----------|-------|
| **Extranet** | Selective connectivity between specific VRFs | Mutual RT import between VRF pairs | O(n²) — each pair needs explicit RT config |
| **Shared Services** | One central VRF accessible by all (or many) VPN customers | Hub VRF exports a common RT; spokes import it | O(n) — each new customer imports one RT |
| **Central Services** | SP-owned services (DNS, NTP, internet) injected into customer VRFs | Similar to shared services, often with default route | O(n) — same pattern, SP-operated |

The distinction matters for **scaling**. Extranet is peer-to-peer: Customer A and Customer B exchange routes bilaterally. Shared services is hub-and-spoke: everyone talks to the hub, not to each other. Most real deployments use shared services for common infrastructure and extranet for specific B2B peering.

### Extranet Mechanics

Extranet is controlled entirely through **Route Target import/export policy**. No special protocol extensions. No new address families. Just thoughtful RT design.

#### Simple Two-Party Extranet

Customer A (VRF-A) needs to reach Customer B (VRF-B) and vice versa:

```
VRF-A:  export RT 65000:100   →  import RT 65000:100, 65000:200
VRF-B:  export RT 65000:200   →  import RT 65000:100, 65000:200
```

Both VRFs import each other's export RT. Full mesh connectivity between the two.

#### Selective Extranet with Dedicated RT

Better approach — use a **dedicated extranet RT** separate from the base VPN RT:

```
VRF-A:  export RT 65000:100, 65000:999   →  import RT 65000:100, 65000:999
VRF-B:  export RT 65000:200, 65000:999   →  import RT 65000:200, 65000:999
```

RT 65000:999 is the "extranet community." Any VRF that exports and imports it joins the extranet group. The base VPN RTs (100, 200) remain isolated.

**Why this is better:** You can add RT 65000:999 to VRF-C without touching VRF-A or VRF-B config. You can also use route-policy to export only specific prefixes with the extranet RT, rather than leaking the entire VRF.

#### Selective Prefix Leaking

The nuclear option is leaking all routes between VRFs. The surgical option is leaking only specific prefixes. This requires **route-policy on export** to tag only selected prefixes with the extranet RT.

IOS-XR approach:
1. Define a prefix-set matching the shared prefixes
2. Apply a route-policy on the VRF's BGP export that adds the extranet RT only for matching prefixes
3. Other VRFs import the extranet RT normally

This gives per-prefix control over what leaks. Essential for compliance and security.

### Shared Services Architecture

Shared services creates a **hub VRF** (commonly called `VRF-SHARED` or `VRF-CENTRAL`) that provides services to multiple customer VRFs (spokes).

#### Hub-and-Spoke RT Model

```
Hub VRF (SHARED):
  export RT 65000:9999        ← "shared services available"
  import RT 65000:8888        ← "customer routes for return traffic"

Spoke VRF (CUST-A):
  export RT 65000:100, 65000:8888    ← normal VPN + shared-return
  import RT 65000:100, 65000:9999    ← normal VPN + shared-services

Spoke VRF (CUST-B):
  export RT 65000:200, 65000:8888
  import RT 65000:200, 65000:9999
```

**Route flow:**
1. Hub VRF exports its service prefixes (DNS, NTP, internet gateway) with RT 65000:9999
2. All spoke VRFs import RT 65000:9999 → they learn the shared service routes
3. Spoke VRFs export their routes with RT 65000:8888 (in addition to their normal VPN RT)
4. Hub VRF imports RT 65000:8888 → it learns return paths to all customer prefixes

**Critical design question: Do spokes see each other?**

With the above design, **no**. Customer A exports RT 65000:100 and RT 65000:8888. Customer B imports RT 65000:200 and RT 65000:9999. Customer B never imports RT 65000:100 or RT 65000:8888, so Customer A's routes are invisible to Customer B. The hub sees everyone; spokes see only the hub. This is by design.

If you accidentally have spokes import RT 65000:8888, you've just created a full mesh extranet. That's not shared services — that's a career-limiting event.

### Internet Access via Shared Services

The most common shared service: internet breakout. Every VPN customer needs internet access, but you don't want N separate internet connections.

#### Design Options

| Approach | How | Pros | Cons |
|----------|-----|------|------|
| **Default route injection** | Hub VRF exports 0/0 with shared RT | Simple, scalable | All traffic hits the hub — capacity planning critical |
| **NAT at the hub** | Hub VRF NATs customer traffic to a public pool | Full isolation, single public prefix block | NAT state, logging, troubleshooting complexity |
| **VRF route leaking to GRT** | Leak customer routes into global table for internet | No hub VRF needed | Dangerous — customer routes in GRT, overlap risk |
| **Internet VRF** | Dedicated VRF for internet (not the global table) | Clean separation, scales well | Extra VRF overhead |

**Recommended:** Internet VRF with default route injection. Keeps customer routes out of the global routing table, provides clean isolation, and scales linearly.

### Static Route Leaking vs RT-Based Leaking

On some platforms, you can leak routes between VRFs using **static routes with next-hop in another VRF** instead of RT import/export:

```
! IOS-XR: static route in VRF-A pointing to next-hop in VRF-B
router static
 vrf CUST-A
  address-family ipv4 unicast
   10.99.0.0/24 vrf SHARED 10.99.0.1
```

**When to use static leaking:**
- Small scale (1-2 prefixes, 1-2 VRF pairs)
- PE-local services (the shared service is on THIS PE only)
- Quick fix / proof of concept

**When NOT to use static leaking:**
- Multi-PE environments (static routes don't propagate via BGP)
- More than a handful of prefixes (unmanageable)
- Anything that needs to survive PE failover

RT-based leaking is always the scalable, production answer. Static leaking is the "I need this working before the call in 10 minutes" answer.

### Route Leaking Loops and Black Holes

The single biggest danger with extranet/shared services: **routing loops created by leaked routes being re-exported.**

Scenario: VRF-A exports a route with RT 65000:100. VRF-SHARED imports it (via RT 65000:8888). VRF-SHARED then re-exports it (because export policy adds RT 65000:9999 to ALL routes). VRF-B imports it (via RT 65000:9999). VRF-B now has a route to VRF-A's prefix — **which you didn't intend.**

**Prevention:**

1. **Export route-policy filtering** — Only export hub-originated prefixes with the shared RT, not imported customer routes
2. **SOO (Site of Origin)** — Tags routes with their source; reject routes with your own SOO on import
3. **RT rewrite** — Strip and rewrite RTs on import to prevent transitive leaking
4. **BGP community filtering** — Use a "do-not-leak" community on routes that must stay within their VRF

The export route-policy is the primary defense. If your hub VRF blindly re-exports everything it imports, you've built a full-mesh extranet whether you wanted one or not.

### Asymmetric Extranet (One-Way Access)

Sometimes Customer B needs to reach Customer A's services, but Customer A should NOT be able to initiate connections to Customer B:

```
VRF-A:  export RT 65000:100, 65000:999   →  import RT 65000:100  (no 999!)
VRF-B:  export RT 65000:200              →  import RT 65000:200, 65000:999
```

VRF-B imports RT 65000:999 → it learns VRF-A's routes. But VRF-A doesn't import RT 65000:200 → it never learns VRF-B's routes. VRF-B can reach VRF-A, but return traffic from A to B has no route.

**Wait, that breaks things.** If VRF-B sends a packet to VRF-A, VRF-A has no return route to VRF-B. The packet arrives, the reply gets dropped.

For asymmetric extranet to work, you need **return route injection** — the hub or a firewall in the path must provide a return path. Typically:
1. VRF-A exports its routes to a **firewall VRF**
2. The firewall VRF has a return route to VRF-B
3. The firewall enforces the one-way policy (permit B→A, deny A→B)
4. VRF-A has a route to VRF-B **via the firewall** (the firewall handles the policy)

True asymmetric routing without a stateful device in the path doesn't work at L3. You need a firewall or ACL at a choke point.

### Firewall Integration with Extranet

Real-world extranet almost always involves a firewall. Two integration patterns:

#### Pattern 1: Firewall-on-a-Stick (Single PE)

```
     VRF-A ──┐
              ├── PE ──── Firewall (sub-interfaces per VRF)
     VRF-B ──┘
```

- Firewall has interfaces in both VRFs
- Routes between VRFs via the firewall's routing table
- All inter-VRF traffic traverses the firewall
- Pros: Simple, all traffic inspected
- Cons: Bandwidth limited by firewall throughput, single PE failure domain

#### Pattern 2: Firewall VRF (Dedicated Transit VRF)

```
     VRF-A ──── PE1 ═══ MPLS ═══ PE2 ──── VRF-FW ──── Firewall
     VRF-B ──── PE1 ═══ MPLS ═══ PE2 ──── VRF-FW ──── Firewall
```

- Dedicated VRF-FW for firewall connectivity
- VRF-A and VRF-B export routes to VRF-FW (RT import)
- Firewall has full visibility and enforces policy
- Return traffic routed back through VRF-FW
- Pros: Firewall can be centralized, survives PE failure
- Cons: Extra VRF complexity, traffic tromboning through firewall PE

Pattern 2 is the production answer for SP-scale extranet. Pattern 1 is for enterprise deployments or small SPs.

## Configuration

### IOS-XR — Shared Services (Hub-and-Spoke)

```cisco
!! =============================================
!! Hub PE — Shared Services VRF
!! =============================================

! Hub VRF providing DNS, NTP, Internet
vrf SHARED-SERVICES
 address-family ipv4 unicast
  import route-target
   65000:8888            ! Import return routes from spokes
  !
  export route-policy SHARED-EXPORT
  export route-target
   65000:9999            ! Export shared services routes to spokes
  !
 !
!

! Route-policy: ONLY export hub-originated routes, NOT re-export imported customer routes
! NOTE: export route-policy filters WHICH routes get exported.
! export route-target sets WHAT RT is attached to passing routes.
! The policy does NOT need to set RTs — that's the route-target's job.
route-policy SHARED-EXPORT
  if destination in SHARED-PREFIXES then
    pass
  else
    drop
  endif
end-policy
!

prefix-set SHARED-PREFIXES
  10.99.0.0/24,           ! DNS servers
  10.99.1.0/24,           ! NTP servers
  10.99.2.0/24,           ! RADIUS/TACACS
  0.0.0.0/0               ! Default route for internet
end-set
!

! Interface to shared services infrastructure
interface GigabitEthernet0/0/0/10
 vrf SHARED-SERVICES
 ipv4 address 10.99.0.1 255.255.255.0
!

! Static routes to shared services (or redistribute connected)
router static
 vrf SHARED-SERVICES
  address-family ipv4 unicast
   10.99.1.0/24 10.99.0.254     ! NTP via services switch
   10.99.2.0/24 10.99.0.253     ! RADIUS via services switch
  !
 !
!

! BGP — advertise shared services and receive spoke routes
router bgp 65000
 vrf SHARED-SERVICES
  rd 65000:9000
  address-family ipv4 unicast
   redistribute connected
   redistribute static
  !
 !
!
```

### IOS-XR — Spoke VRF (Customer Importing Shared Services)

```cisco
!! =============================================
!! Spoke PE — Customer VRF importing shared services
!! =============================================

vrf CUST-A
 address-family ipv4 unicast
  import route-target
   65000:100              ! Normal VPN routes
   65000:9999             ! Shared services routes from hub
  !
  export route-policy SPOKE-EXPORT
  export route-target
   65000:100              ! Normal VPN export
   65000:8888             ! Return routes to shared hub
  !
 !
!

! Route-policy: export all customer routes with both RTs,
! but tag only specific return prefixes with the shared RT
! (Optional — full export is simpler but less controlled)
route-policy SPOKE-EXPORT
  pass
end-policy
!

! Required: IOS-XR eBGP neighbors MUST have route-policies
route-policy PASS-ALL
  pass
end-policy
!

! PE-CE interface
interface GigabitEthernet0/0/0/0.100
 vrf CUST-A
 ipv4 address 172.16.0.1 255.255.255.252
 encapsulation dot1q 100
!

router bgp 65000
 vrf CUST-A
  rd 65000:100
  address-family ipv4 unicast
   maximum-prefix 1000 75 warning-only
  !
  neighbor 172.16.0.2
   remote-as 64512
   address-family ipv4 unicast
    route-policy PASS-ALL in
    route-policy PASS-ALL out
   !
  !
 !
!
```

### IOS-XR — Extranet with Selective Prefix Leaking

```cisco
!! =============================================
!! Extranet: VRF-A leaks specific prefixes to VRF-B
!! =============================================

vrf CUST-A
 address-family ipv4 unicast
  import route-target
   65000:100              ! Own routes
   65000:201              ! Extranet routes from B
  !
  export route-policy EXTRANET-EXPORT-A
  export route-target
   65000:100              ! Normal VPN
  !
 !
!

! Tag ONLY the database subnet with the extranet RT
! (All routes still export with base RT 65000:100 — the extranet RT is additive)
route-policy EXTRANET-EXPORT-A
  if destination in EXTRANET-PREFIXES-A then
    set extcommunity rt (65000:901) additive
  endif
  pass
end-policy
!

prefix-set EXTRANET-PREFIXES-A
  10.1.99.0/24            ! Database server subnet
end-set
!

vrf CUST-B
 address-family ipv4 unicast
  import route-target
   65000:200              ! Own routes
   65000:901              ! Extranet routes from A (database only)
  !
  export route-policy EXTRANET-EXPORT-B
  export route-target
   65000:200              ! Normal VPN
  !
 !
!

route-policy EXTRANET-EXPORT-B
  if destination in EXTRANET-PREFIXES-B then
    set extcommunity rt (65000:201) additive
  endif
  pass
end-policy
!

prefix-set EXTRANET-PREFIXES-B
  10.2.50.0/24            ! Application server subnet
end-set
!
```

### Junos — Shared Services (Hub-and-Spoke)

```junos
## =============================================
## Hub PE — Shared Services VRF
## =============================================

routing-instances {
    SHARED-SERVICES {
        instance-type vrf;
        interface ge-0/0/10.0;
        route-distinguisher 65000:9000;
        vrf-import SHARED-IMPORT;       ## Import return routes from spokes
        vrf-export SHARED-EXPORT;       ## Export only hub-originated services
        vrf-table-label;

        routing-options {
            static {
                route 10.99.1.0/24 next-hop 10.99.0.254;   ## NTP
                route 10.99.2.0/24 next-hop 10.99.0.253;   ## RADIUS
            }
        }

        ## NOTE: PE-PE iBGP for VPNv4 is configured at the GLOBAL level
        ## (protocols bgp), not inside the routing-instance.
        ## VRF route exchange happens automatically via global MP-BGP + RT policies.
    }
}

## Import policy: accept routes tagged for shared services return
policy-options {
    policy-statement SHARED-IMPORT {
        term ACCEPT-SPOKE-ROUTES {
            from community SHARED-RETURN-RT;
            then accept;
        }
        term REJECT-REST {
            then reject;
        }
    }

    ## Export policy: tag ONLY hub-originated prefixes, not re-exported spoke routes
    policy-statement SHARED-EXPORT {
        term EXPORT-SERVICES {
            from {
                protocol [ static direct ];
                route-filter 10.99.0.0/22 orlonger;
                route-filter 0.0.0.0/0 exact;          ## Default route
            }
            then {
                community add SHARED-SERVICES-RT;
                accept;
            }
        }
        term REJECT-LEAKED {
            then reject;                                ## DO NOT re-export imported spoke routes
        }
    }

    community SHARED-SERVICES-RT members target:65000:9999;
    community SHARED-RETURN-RT members target:65000:8888;
}

interfaces {
    ge-0/0/10 {
        unit 0 {
            family inet {
                address 10.99.0.1/24;
            }
        }
    }
}
```

### Junos — Spoke VRF (Customer Importing Shared Services)

```junos
## =============================================
## Spoke PE — Customer VRF with shared services access
## =============================================

routing-instances {
    CUST-A {
        instance-type vrf;
        interface ge-0/0/0.100;
        route-distinguisher 65000:100;
        vrf-import CUST-A-IMPORT;
        vrf-export CUST-A-EXPORT;
        vrf-table-label;

        routing-options {
            autonomous-system 65000;    ## Required if not set globally
        }

        protocols {
            bgp {
                group CE-PEERS {
                    type external;
                    peer-as 64512;
                    neighbor 172.16.0.2;
                }
            }
        }
    }
}

policy-options {
    ## Import own VPN routes + shared services
    policy-statement CUST-A-IMPORT {
        term OWN-VPN {
            from community CUST-A-RT;
            then accept;
        }
        term SHARED-SERVICES {
            from community SHARED-SERVICES-RT;
            then accept;
        }
        term REJECT {
            then reject;
        }
    }

    ## Export customer routes with own RT + shared return RT
    ## WARNING: This exports ALL customer prefixes to the hub VRF.
    ## For large customers, filter to only necessary return prefixes
    ## (e.g., match protocol direct for connected subnets only).
    policy-statement CUST-A-EXPORT {
        term EXPORT-ALL {
            then {
                community add CUST-A-RT;
                community add SHARED-RETURN-RT;
                accept;
            }
        }
    }

    community CUST-A-RT members target:65000:100;
    community SHARED-SERVICES-RT members target:65000:9999;
    community SHARED-RETURN-RT members target:65000:8888;
}
```

### Junos — Extranet with Selective Prefix Leaking

```junos
## =============================================
## Extranet: CUST-A leaks database subnet to CUST-B
## =============================================

routing-instances {
    CUST-A {
        instance-type vrf;
        interface ge-0/0/0.100;
        route-distinguisher 65000:100;
        vrf-import CUST-A-IMPORT;
        vrf-export CUST-A-EXTRANET-EXPORT;
        vrf-table-label;
    }

    CUST-B {
        instance-type vrf;
        interface ge-0/0/1.200;
        route-distinguisher 65000:200;
        vrf-import CUST-B-IMPORT;
        vrf-export CUST-B-EXTRANET-EXPORT;
        vrf-table-label;
    }
}

policy-options {
    ## CUST-A export: normal RT + extranet RT on database prefix only
    policy-statement CUST-A-EXTRANET-EXPORT {
        term DATABASE-EXTRANET {
            from {
                route-filter 10.1.99.0/24 exact;
            }
            then {
                community add CUST-A-RT;
                community add EXTRANET-A-TO-B;
                accept;
            }
        }
        term NORMAL {
            then {
                community add CUST-A-RT;
                accept;
            }
        }
    }

    ## CUST-B export: normal RT + extranet RT on app server prefix only
    policy-statement CUST-B-EXTRANET-EXPORT {
        term APP-EXTRANET {
            from {
                route-filter 10.2.50.0/24 exact;
            }
            then {
                community add CUST-B-RT;
                community add EXTRANET-B-TO-A;
                accept;
            }
        }
        term NORMAL {
            then {
                community add CUST-B-RT;
                accept;
            }
        }
    }

    ## CUST-A import: own routes + extranet from B
    policy-statement CUST-A-IMPORT {
        term OWN {
            from community CUST-A-RT;
            then accept;
        }
        term EXTRANET-FROM-B {
            from community EXTRANET-B-TO-A;
            then accept;
        }
        term REJECT {
            then reject;
        }
    }

    ## CUST-B import: own routes + extranet from A
    policy-statement CUST-B-IMPORT {
        term OWN {
            from community CUST-B-RT;
            then accept;
        }
        term EXTRANET-FROM-A {
            from community EXTRANET-A-TO-B;
            then accept;
        }
        term REJECT {
            then reject;
        }
    }

    community CUST-A-RT members target:65000:100;
    community CUST-B-RT members target:65000:200;
    community EXTRANET-A-TO-B members target:65000:901;
    community EXTRANET-B-TO-A members target:65000:201;
}
```

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| `import route-target` | None | Explicit per-VRF | Only import what you need — never wildcard |
| `export route-policy` | Export all with configured RT | Filter on hub VRF | Prevents transitive route leaking |
| `maximum-prefix` (VRF) | Unlimited | Set per VRF | Prevents spoke from flooding hub with routes |
| `vrf-table-label` (Junos) | Off | On for shared VRFs | Per-VRF label allocation, needed for proper forwarding |
| `retain route-target all` | Off | On (RR only) | RR must keep all RT routes for extranet to work |
| RT-Constraint | Off | On (PE + RR) | Prevents unnecessary extranet routes hitting uninvolved PEs |

### What Good Looks Like

**Spoke VRF correctly imports shared services route:**
```
RP/0/RP0/CPU0:PE1# show bgp vrf CUST-A ipv4 unicast 10.99.0.0/24

BGP routing table entry for 10.99.0.0/24, Route Distinguisher: 65000:9000
  65000:9000:10.99.0.0/24
  Path: internal, best
  Origin: incomplete, localpref 100, weight 0
  Received from: 192.0.2.10 (Route Reflector)
  Extended Community: RT:65000:9999
  MPLS label: 24015
  Next-hop: 192.0.2.2 (resolved via LDP/SR)
```

Key checks:
- Route is present in the spoke VRF ✓
- RT matches the shared services RT (65000:9999) ✓
- RD is the hub VRF's RD (65000:9000) — confirms it came from the hub ✓
- MPLS label is allocated — forwarding will work ✓
- Next-hop resolves — transport path exists ✓

**Hub VRF correctly imports spoke return route:**
```
RP/0/RP0/CPU0:PE2# show bgp vrf SHARED-SERVICES ipv4 unicast 172.16.0.0/24

  Path: internal, best
  Extended Community: RT:65000:100 RT:65000:8888
  MPLS label: 24020
  Next-hop: 192.0.2.1 (resolved via LDP/SR)
```

Key checks:
- Hub VRF has return route to customer prefix ✓
- RT includes 65000:8888 (shared return RT) — that's why it was imported ✓
- Note: RT 65000:100 is also present (customer's base RT), but hub imported via 65000:8888

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Spoke can't reach shared services | Missing `import RT 65000:9999` on spoke VRF | Add the shared services RT to spoke import |
| Shared services can't reach spoke (return traffic fails) | Hub VRF doesn't import spoke's return RT | Add `import RT 65000:8888` on hub; verify spoke exports it |
| Spoke A can see Spoke B's routes (unintended) | Hub VRF re-exporting imported spoke routes | Add export route-policy to ONLY export hub-originated prefixes |
| Extranet route missing on remote PE | RR not retaining the extranet RT (no local VRF imports it) | Enable `retain route-target all` on RR, or enable RT-Constraint |
| Route exists in VRF but ping fails | Label stack broken — next-hop not resolved | Check `show cef vrf` for complete label stack; verify transport LSP |
| Routing loop between VRFs | Transitive leaking — imported routes re-exported with new RT | Filter exports: only export locally-originated routes with extranet RT |
| Spoke exports too many routes to hub | No maximum-prefix on hub VRF import | Add `maximum-prefix` per VRF; use route-policy to limit what spoke tags with return RT |
| Extranet works on one PE but not another | Inconsistent RT config across PEs for the same VRF | Audit RT import/export on ALL PEs hosting the VRF |
| RT-Constraint filtering extranet routes | PE doesn't advertise interest in extranet RT | VRF must import the RT for RT-Constraint to signal interest |
| Shared services default route missing | Hub VRF export policy doesn't include 0/0 | Add `0.0.0.0/0 exact` to export prefix-set |

### Methodology

1. **Verify RT config** — `show vrf CUST-A detail` / `show route instance CUST-A detail` — confirm import/export RTs are correct
2. **Check the RR** — `show bgp vpnv4 unicast rd <hub-rd>` — are hub routes being advertised?
3. **Check RT-Constraint** — Is the PE expressing interest in the shared RT? If RT-Constraint is on and the PE doesn't import the RT, the RR won't send it
4. **Check spoke VRF** — `show bgp vrf CUST-A ipv4 unicast 10.99.0.0/24` — is the route imported?
5. **Check FIB** — `show cef vrf CUST-A 10.99.0.0/24` — is there a valid label stack?
6. **Check return path** — From hub VRF, verify route to spoke prefix exists with correct next-hop
7. **Trace the packet** — If route and FIB look good, use `traceroute vrf` to follow the label-switched path
8. **Check for re-export** — On hub PE, verify the export route-policy is filtering correctly — `show bgp vrf SHARED-SERVICES ipv4 unicast | include RT` should NOT show spoke RTs on hub-exported routes

### War Stories

> **The Accidental Full-Mesh Extranet**
>
> A regional SP deployed shared services for internet breakout. Hub VRF on PE2 with DNS, NTP, and a default route. 47 spoke VRFs across 12 PEs. Design looked clean on paper.
>
> Six months later, a compliance audit found that Customer 12 (a healthcare provider) could ping Customer 37 (a competitor). HIPAA violation. Immediate escalation.
>
> Root cause: The hub VRF export policy used `pass` (accept all) instead of filtering to hub-originated routes. When spoke routes arrived at the hub VRF via the return RT, the hub PE re-exported them with the shared services RT. Every spoke that imported the shared services RT also imported every other spoke's routes. 47 VRFs, fully meshed, for six months. Nobody noticed because customers rarely try to ping each other — until an overzealous pentester ran a network scan.
>
> Fix: Two-line route-policy change on the hub PE. Total time to remediate: 4 minutes of config work, 6 weeks of compliance documentation, and one very uncomfortable conversation with the healthcare customer's CISO.
>
> **Lesson:** Always filter hub VRF exports. If your export policy is `pass`, you don't have shared services — you have a party line. Test by checking `show bgp vrf SHARED-SERVICES ipv4 unicast` for routes that shouldn't be there.

## Design Considerations

### RT Allocation Strategy for Extranet

| RT Range | Purpose | Example |
|----------|---------|---------|
| 65000:1-999 | Base VPN RTs (per-customer) | 65000:100 = Customer A |
| 65000:8000-8999 | Shared services (spoke→hub return) | 65000:8888 = return routes to hub |
| 65000:9000-9999 | Shared services (hub→spoke services) | 65000:9999 = services available |
| 65000:5000-5999 | Extranet groups | 65000:5001 = Extranet Group 1 |
| 65000:6000-6999 | Per-pair extranet | 65000:6012 = A↔B pair |

Document this in your SP's provisioning system. RT collisions between extranet groups are invisible until someone notices cross-VRF traffic that shouldn't exist.

### Scale Considerations

- **Extranet pairs:** O(n²) — 10 VRFs in a full-mesh extranet = 45 pairs = 90 directional RT imports. Use shared services instead.
- **Shared services spokes:** O(n) — adding a new spoke = 1 VRF config change (add 2 RTs). Hub config doesn't change.
- **RR impact:** Every extranet/shared RT is another VPNv4 path the RR must store and distribute. Monitor RR memory.
- **RT-Constraint saves bandwidth:** Without it, every PE receives every extranet route even if it has no matching VRF. With RT-Constraint, only interested PEs receive extranet routes.

### When NOT to Use Extranet

- **When a firewall should decide:** If the inter-VRF policy is "allow HTTP but deny SSH," that's a firewall job, not a routing job. Leak the routes through a firewall VRF.
- **When customers need full isolation:** If the contract says "dedicated network," don't export their RTs anywhere. Use separate physical infrastructure or convince them that VRF isolation with auditing is equivalent (it usually is, but lawyers gonna lawyer).
- **When scale exceeds RT manageability:** If you have 200 extranet relationships, you don't need smarter RT design — you need a different architecture (SD-WAN overlay, service mesh, etc.).

### RR Configuration for Extranet

**Mandatory on Route Reflectors** — without this, the RR drops extranet/shared-services routes that no local VRF imports:

```cisco
!! IOS-XR Route Reflector
router bgp 65000
 address-family vpnv4 unicast
  retain route-target all        ! Keep ALL VPNv4 routes regardless of local VRFs
 !
 address-family vpnv6 unicast
  retain route-target all
 !
!
```

```junos
## Junos Route Reflector — retains all VPN routes by default
## when acting as RR (route-reflection is implicit retain-all).
## No special knob needed on Junos RRs.
```

This is the **#1 operational gotcha** with extranet deployments. RRs with no local VRFs will silently discard extranet routes unless `retain route-target all` is configured. RT-Constraint (RFC 4684) is the smarter alternative — PEs advertise which RTs they need, and the RR only sends those.

### Modern Alternatives

IOS-XR 7.x supports `import from vrf <name> advertise-as-vpn` as a local-PE alternative to RT-based leaking. Junos has `auto-export` for automatic inter-instance route sharing. These are simpler for PE-local use cases but don't scale across the MPLS core like RT-based designs.

### SRv6 Considerations

Extranet and shared services work identically with SRv6 transport — the RT import/export logic is control-plane only. The data-plane label stack changes (SRv6 SIDs instead of MPLS labels), but the VRF and RT mechanics are unchanged. No special SRv6 config needed for extranet beyond the normal L3VPN SRv6 setup from Section 6.4.
