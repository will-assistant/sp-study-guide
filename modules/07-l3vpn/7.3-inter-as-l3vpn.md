# 7.3 — Inter-AS L3VPN

## Overview
> When a single VPN customer spans multiple autonomous systems — different providers, different regions within the same provider, or acquired networks not yet merged — standard iBGP VPNv4 breaks because it can't cross AS boundaries. Inter-AS L3VPN defines how to stitch VPN services across ASes while preserving isolation, scalability, and operational control. RFC 4364 (Section 10) defines three canonical options: A (back-to-back VRF), B (ASBR VPNv4 exchange), and C (multihop MP-BGP + labeled unicast). Each trades simplicity for scalability. This is bread-and-butter IE-SP material and one of the most heavily tested topics on both exams.


## Key Concepts
> **After this section you will understand:**
> - how VRFs, route targets, and MP-BGP deliver tenant separation
> - how VPN route import/export policy controls reachability scope
> - design options for inter-AS and shared-services connectivity
> - how scale and convergence choices impact large VPN deployments

## Theory

### The Problem
In a single-AS L3VPN, the PE-to-PE path works because:
1. iBGP VPNv4 carries the VPN prefix + label
2. The VPNv4 next-hop (originating PE loopback) is reachable via IGP + LDP/SR
3. A 2-label stack (transport + VPN) delivers the packet

Across AS boundaries, this breaks:
- **IGP doesn't span ASes** — PE2's loopback isn't in AS1's routing table
- **LDP/RSVP-TE sessions don't cross ASes** — no transport label to PE2
- **iBGP sessions require IGP reachability** — can't peer to unknown loopbacks

Each Inter-AS option solves this differently.

### Option A: Back-to-Back VRF (VRF-to-VRF)

**Architecture:**
```
CE1 --- PE1 (AS1) --- ASBR1 (AS1) === ASBR2 (AS2) --- PE2 (AS2) --- CE2
                       VRF ←→ VRF
                      (eBGP per VRF)
```

**Mechanics:**
- ASBRs face each other with a physical or logical link
- Each VPN gets a VRF on both ASBRs
- Per-VRF eBGP sessions (or static routes) exchange IPv4/IPv6 unicast between ASBRs
- Traffic is **decapsulated** at ASBR1, forwarded as plain IP, **re-encapsulated** at ASBR2
- From each AS's perspective, the ASBR looks like a PE with a directly-connected CE

**Label stack:**
```
AS1: [Transport-to-ASBR1][VPN-Label] → ASBR1 strips MPLS, forwards IP → 
AS2: ASBR2 pushes [Transport-to-PE2][VPN-Label] → PE2
```
Two independent label domains. No label stitching.

**Pros:**
- Simplest conceptually — each AS is fully independent
- Complete control: each AS sets its own RT import policy on the ASBR VRFs
- No VPNv4 routes leak between ASes — maximum isolation
- Works with any MPLS transport (LDP, RSVP-TE, SR) independently per AS
- Easy troubleshooting — standard VRF + eBGP

**Cons:**
- **Doesn't scale**: Every VPN needs a VRF + sub-interface + eBGP session on both ASBRs
- 1,000 VPNs = 1,000 VRFs + 1,000 eBGP sessions per ASBR pair. ASBRs become the bottleneck.
- ASBR must do full IP lookup per VPN (no cut-through MPLS switching)
- Physical links between ASBRs must carry all VPN traffic (no ECMP across MPLS core)
- Operational overhead: adding a VPN requires config on 4 devices (PE1, ASBR1, ASBR2, PE2)

**When to use:**
- Small number of VPNs across ASes (< 50)
- Maximum isolation required (regulatory, different providers with trust boundary)
- Migration/acquisition: quick interconnect before building proper Option B/C
- Lab environments and proof-of-concept

### Option B: ASBR VPNv4 Exchange (eBGP VPNv4 at ASBR)

**Architecture:**
```
CE1 --- PE1 (AS1) --- ASBR1 (AS1) === ASBR2 (AS2) --- PE2 (AS2) --- CE2
                       eBGP VPNv4
                      (next-hop-self)
```

**Mechanics:**
- ASBRs run eBGP VPNv4/VPNv6 between each other (single session, all VPNs)
- ASBR1 receives VPNv4 from PE1 via iBGP, re-advertises to ASBR2 via eBGP
- Critical: ASBR must do **next-hop-self** — rewrites VPNv4 next-hop to itself
- ASBR1 allocates a **new VPN label** for each re-advertised prefix
- When traffic arrives from AS2, ASBR1 swaps the VPN label and forwards into AS1's MPLS domain
- Label swap (not IP lookup) at ASBR — much faster than Option A

**Label stack:**
```
Ingress PE2 pushes: [Transport-to-ASBR2][ASBR2-VPN-Label]
ASBR2 swaps VPN label → [Transport-to-ASBR1][ASBR1-VPN-Label]  
  (actually: ASBR2 strips transport, looks up VPN label, pushes toward ASBR1)
ASBR1 swaps VPN label → [Transport-to-PE1][PE1-VPN-Label]
```

**Detailed packet walk (CE2 → CE1, ASBRs directly connected):**
1. PE2 pushes: `[Transport-to-ASBR2][VPN-label-from-ASBR2] | IP`
2. P routers in AS2 swap transport labels
3. Penultimate hop in AS2 pops transport (PHP) → ASBR2 receives: `[VPN-label-from-ASBR2] | IP`
4. ASBR2 looks up its VPN label in LFIB → action: swap to ASBR1's VPN label, forward to ASBR1
5. On inter-AS link: `[VPN-label-from-ASBR1] | IP` (single label — no transport needed between directly-connected ASBRs)
6. ASBR1 looks up its VPN label in LFIB → action: swap to PE1's VPN label, push transport to PE1
7. Packet into AS1: `[Transport-to-PE1][VPN-label-from-PE1] | IP`
8. P routers in AS1 swap transport labels
9. PE1 pops transport + VPN label → VRF → CE1

**The key insight:** ASBRs do label swaps, not IP lookups. They maintain a VPN label forwarding table but never need VRFs. Each ASBR acts as a "label stitching" point.

**Pros:**
- Scales better than Option A — single eBGP VPNv4 session carries all VPNs
- No per-VPN VRF or sub-interface on ASBRs
- MPLS-switched at ASBRs (label swap, not IP lookup)
- Straightforward eBGP — existing operational tooling works

**Cons:**
- **ASBRs must hold ALL VPNv4 routes** from the other AS (or at least those with matching RTs)
- With 500K VPNv4 routes, ASBRs need significant memory/RIB capacity
- RT filtering at ASBR is critical to avoid route explosion
- Adding VPNs doesn't require ASBR config changes (good), but ASBR is still in the data path for label allocation
- Next-hop-self means ASBRs are in the label forwarding path — failure of ASBR = loss of all inter-AS VPNs through it
- Route refresh/soft-clear across eBGP VPNv4 can be expensive

**RT filtering (critical for scale):**
- Without filtering, ASBR1 receives ALL VPNv4 routes from AS2, even VPNs that only exist in AS2
- Use `orf prefix-list` or explicit RT-based route-policy to accept only relevant RTs
- RT-Constraint (RFC 4684) works within each AS but does NOT cross eBGP boundaries natively
- Manual RT-based filtering at ASBR eBGP session is standard practice

**When to use:**
- Medium-scale inter-AS (50–5,000 VPNs)
- Single provider, multiple regions/ASes
- When ASBRs have sufficient RIB capacity
- Most common production choice for medium SPs

### Option C: Multihop MP-eBGP + Labeled Unicast

**Architecture:**
```
CE1 --- PE1 (AS1) --- ASBR1 (AS1) === ASBR2 (AS2) --- PE2 (AS2) --- CE2
                                  eBGP labeled-unicast
        RR1 (AS1) ========================================= RR2 (AS2)
                         multihop eBGP VPNv4
```

**Two BGP sessions, two different jobs:**
1. **ASBR-to-ASBR**: eBGP labeled-unicast (AFI 1/SAFI 4) — exchanges PE loopbacks with labels
2. **RR-to-RR** (or PE-to-PE): Multihop eBGP VPNv4/VPNv6 — exchanges VPN routes with original next-hops

**Mechanics (the elegant part):**
- RR1 advertises VPNv4 routes to RR2 with the **original PE1 next-hop preserved** (no next-hop-self)
- But PE1's loopback isn't in AS2's IGP. How does PE2 resolve it?
- ASBR1 advertises PE1's loopback to ASBR2 via eBGP labeled-unicast (with a label!)
- ASBR2 redistributes this into AS2's IGP (or iBGP) so PE2 can resolve PE1's next-hop
- The label from ASBR1 acts as a "stitching" label — it tells ASBR1 how to forward toward PE1

**Label stack (3 labels!):**
```
PE2 pushes: [Transport-to-ASBR2][Stitching-Label][VPN-Label]
                 ↑                      ↑              ↑
            LDP/SR in AS2    From labeled-unicast    From VPNv4 (PE1's label)
```

**How the label chain is built:**
- ASBR1 advertises PE1's loopback (10.0.0.1/32) to ASBR2 via eBGP labeled-unicast with label L1
- ASBR2 allocates its own label L2 for this FEC, advertises via iBGP labeled-unicast to PEs in AS2
- PE2 receives VPNv4 route with next-hop = PE1 (10.0.0.1), VPN label = V1
- PE2 resolves 10.0.0.1 via labeled-unicast: stitching label L2 (from ASBR2), transport to ASBR2 via LDP/SR = T2

**Packet walk per hop (CE2 → CE1):**
```
PE2:      [T2] [L2] [V1] | IP            ← 3-label stack
P (AS2):  [T2'] [L2] [V1] | IP           ← transport swap only
ASBR2:    strips T2, swaps L2→L1:  [L1] [V1] | IP
          (directly connected to ASBR1: single label + VPN label on inter-AS link)
ASBR1:    pops L1, pushes T1 (transport to PE1):  [T1] [V1] | IP
P (AS1):  [T1'] [V1] | IP                ← transport swap only
PE1:      pops T1 + V1 → VRF lookup → CE1
```

> **Key detail:** ASBR2 does a label *swap* (L2→L1), not a pop-and-push. L1 is the label ASBR1 advertised for PE1's loopback. ASBR1 allocated L1 knowing it maps to "forward toward PE1" — so ASBR1 pops L1 and imposes transport to PE1 within AS1.

**Pros:**
- **Maximum scalability** — ASBRs only carry PE loopbacks (tens to hundreds), not VPNv4 routes (hundreds of thousands)
- VPNv4 route exchange is between RRs — ASBRs are not in the control plane for VPN routes
- Adding new VPNs requires zero ASBR changes
- PE-to-PE label path preserved — original VPN labels from PE1 are used end-to-end
- Conceptually closest to "extending the single-AS model across boundaries"

**Cons:**
- **Most complex** — requires labeled-unicast, multihop eBGP, careful next-hop resolution
- 3-label stack required — some older hardware chokes on deep label stacks
- ASBRs must support eBGP labeled-unicast (not all platforms historically did)
- Multihop eBGP VPNv4 between RRs requires careful TTL/security config
- Next-hop resolution chain is longer: VPNv4 next-hop → labeled-unicast → IGP → transport
- Debugging is harder — label stitching adds indirection
- If ASBR1-ASBR2 labeled-unicast session flaps, ALL VPNs lose inter-AS reachability (same as Option B, but the failure mode is less obvious)

**When to use:**
- Large-scale: thousands of VPNs, hundreds of thousands of VPNv4 routes
- Tier-1 SPs with multiple ASes (regional, acquired, peering)
- When ASBR RIB is the bottleneck (Option B won't fit)
- Production standard for large SPs

### Option Comparison

| Criteria | Option A | Option B | Option C |
|----------|----------|----------|----------|
| **ASBR complexity** | VRF per VPN | VPNv4 RIB | Labeled-unicast only |
| **ASBR state** | All VPN routes (in VRFs) | All VPNv4 routes (in BGP) | PE loopbacks only |
| **Scale ceiling** | ~50 VPNs | ~5K VPNs (RIB dependent) | Unlimited (VPN state on RRs) |
| **Label stack** | 2 per AS (independent) | 2 per AS (stitched) | 3 (transport + stitch + VPN) |
| **Data plane at ASBR** | IP lookup (per-VRF) | Label swap | Label swap |
| **Provisioning per VPN** | ASBR config required | No ASBR config | No ASBR config |
| **Isolation** | Maximum | Medium | Medium |
| **Troubleshooting** | Easiest | Medium | Hardest |
| **Common deployment** | Lab/small/migration | Medium SP | Large SP |
| **RFC 4364 section** | 10a | 10b | 10c |

### Option AB (Hybrid)

A practical variant seen in production: Option A for high-value/regulated VPNs (banking, government) combined with Option B for bulk VPNs on the same ASBR pair. The ASBR runs VRFs for sensitive VPNs and eBGP VPNv4 for everything else. This gives per-VPN control where needed without the scale limit of pure Option A.

### Option D / BGP Classful Transport (Emerging)

RFC 9012 (BGP-CT) and related drafts introduce a transport-class concept where VPN routes carry a "transport class" indicating required SLA (e.g., low-latency, high-bandwidth). The underlay provides multiple transport paths with different characteristics, and the overlay selects the right one automatically. This effectively extends Inter-AS to be SLA-aware without per-VPN tunnels.

Not yet widely deployed (2026), but shows up in Junos 23.x and IOS-XR 7.11+. Mentioned here for awareness — this will likely appear on future cert revisions.

## Configuration

> **Transport prerequisite:** All configs below assume a working MPLS transport within each AS (LDP, RSVP-TE, or SR-MPLS). See Module 4 (MPLS Fundamentals) and Module 6 (Segment Routing) for transport configuration. The inter-AS link itself does NOT require MPLS — it carries either plain IP (Option A) or MPLS with locally-significant labels (Options B/C).

### Option A — IOS-XR

**ASBR1 (AS 65001):**
```cisco
! VRF definition (one per VPN)
vrf CUSTOMER-A
 address-family ipv4 unicast
  import route-target 65001:100
  export route-target 65001:100
 !
!
! Sub-interface toward ASBR2
interface GigabitEthernet0/0/0/1.100
 vrf CUSTOMER-A
 ipv4 address 10.255.0.1 255.255.255.252
 encapsulation dot1q 100
!
! eBGP to ASBR2 — per-VRF session
router bgp 65001
 vrf CUSTOMER-A
  rd 65001:100
  address-family ipv4 unicast
  !
  neighbor 10.255.0.2
   remote-as 65002
   address-family ipv4 unicast
    route-policy PASS-ALL in
    route-policy PASS-ALL out
   !
  !
 !
!
! Route-policy (mandatory on IOS-XR)
route-policy PASS-ALL
  pass
end-policy
```

**ASBR2 (AS 65002):**
```cisco
vrf CUSTOMER-A
 address-family ipv4 unicast
  import route-target 65002:100
  export route-target 65002:100
 !
!
interface GigabitEthernet0/0/0/1.100
 vrf CUSTOMER-A
 ipv4 address 10.255.0.2 255.255.255.252
 encapsulation dot1q 100
!
router bgp 65002
 vrf CUSTOMER-A
  rd 65002:100
  address-family ipv4 unicast
  !
  neighbor 10.255.0.1
   remote-as 65001
   address-family ipv4 unicast
    route-policy PASS-ALL in
    route-policy PASS-ALL out
   !
  !
 !
!
route-policy PASS-ALL
  pass
end-policy
```

> **Note:** RD values differ between ASes (65001:100 vs 65002:100). This is correct — RDs are locally significant for uniqueness. RTs handle import/export policy within each AS independently. The ASBR doesn't need to match RTs across ASes because it's doing plain IPv4 eBGP in the VRF.

### Option B — IOS-XR

**ASBR1 (AS 65001):**
```cisco
! eBGP VPNv4 to ASBR2 — single session, all VPNs
router bgp 65001
 address-family vpnv4 unicast
 !
 neighbor 10.255.0.2
  remote-as 65002
  address-family vpnv4 unicast
   ! CRITICAL: next-hop-self — ASBR1 becomes the next-hop for all VPNv4 from AS1
   next-hop-self
   ! RT-based filtering to limit what we accept
   route-policy ACCEPT-OUR-RTS in
   route-policy PASS-ALL out
   ! Safety: prevent route explosion from misconfigured peer
   maximum-prefix 100000 75 warning-only
  !
  ! BGP authentication (MD5 or TCP-AO in production)
  password encrypted <key>
 !
 !
 ! iBGP to RR (receives VPNv4 from PE routers in AS1)
 neighbor 10.0.0.200
  remote-as 65001
  update-source Loopback0
  address-family vpnv4 unicast
   ! CRITICAL: retain all VPNv4 routes even without local VRF import match
   ! Without this, ASBR drops all VPNv4 from RR (no local VRFs = no matching imports)
   retain route-target all
   ! NHS only for routes we re-advertise from AS2 into AS1
   ! (routes from AS1 PEs reflected by RR won't be re-advertised by ASBR due to iBGP loop prevention)
  !
 !
!
! Accept only RTs we care about from AS2
route-policy ACCEPT-OUR-RTS
  if extcommunity rt matches-any (65001:100, 65001:200, 65001:300) then
    pass
  else
    drop
  endif
end-policy
!
route-policy PASS-ALL
  pass
end-policy
```

**ASBR2 (AS 65002) — mirror config with appropriate AS/RT changes.**

> **Critical detail:** Both ASBRs must do `next-hop-self` for VPNv4. Without it, PE1's loopback becomes the VPNv4 next-hop in AS2, which is unresolvable. This is the #1 misconfiguration in Option B deployments.

> **Label allocation:** When ASBR1 does next-hop-self, it allocates a new local VPN label for each prefix it re-advertises. This label maps to the original PE1 VPN label + transport to PE1 in the ASBR's LFIB. This is the "label stitching" that makes Option B work.

### Option B — Junos

**ASBR1 (AS 65001):**
```junos
protocols {
    bgp {
        group INTER-AS-VPNV4 {
            type external;
            peer-as 65002;
            multihop {
                /* Only if not directly connected */
                ttl 2;
            }
            local-address 10.255.0.1;
            family inet-vpn {
                unicast;
            }
            export NEXT-HOP-SELF-POLICY;
            import ACCEPT-OUR-RTS;
            neighbor 10.255.0.2;
        }
        group IBGP-RR {
            type internal;
            local-address 10.0.0.10;
            family inet-vpn {
                unicast;
            }
            export NEXT-HOP-SELF-POLICY;
            neighbor 10.0.0.200;
        }
    }
}

policy-options {
    policy-statement NEXT-HOP-SELF-POLICY {
        term NHS {
            then {
                next-hop self;
            }
        }
    }
    policy-statement ACCEPT-OUR-RTS {
        term ALLOWED {
            from community [ RT-100 RT-200 RT-300 ];
            then accept;
        }
        term DEFAULT {
            then reject;
        }
    }
    community RT-100 members target:65001:100;
    community RT-200 members target:65001:200;
    community RT-300 members target:65001:300;
}

/* NOTE: Junos sets next-hop self automatically for eBGP-advertised routes.
   The explicit NEXT-HOP-SELF-POLICY on the eBGP group is technically redundant
   but shown for clarity and consistency with IOS-XR. On the iBGP group, it IS
   needed for routes learned from eBGP (AS2 VPNv4 routes re-advertised into AS1). */
```

### Option C — IOS-XR

**This requires two separate BGP configurations:**

**ASBR1 (AS 65001) — Labeled Unicast:**
```cisco
! Advertise PE loopbacks to ASBR2 with labels
router bgp 65001
 address-family ipv4 labeled-unicast
 !
 ! eBGP labeled-unicast to ASBR2
 neighbor 10.255.0.2
  remote-as 65002
  address-family ipv4 labeled-unicast
   route-policy ADVERTISE-PE-LOOPBACKS out
   route-policy ACCEPT-REMOTE-PES in
  !
 !
 ! CRITICAL: allocate labels for labeled-unicast prefixes
 ! This goes under GLOBAL address-family, NOT under a neighbor
 address-family ipv4 unicast
  allocate-label all
 !
 !
 ! iBGP labeled-unicast to redistribute PE loopbacks learned from AS2
 ! (Or redistribute into IGP — see design note below)
 neighbor 10.0.0.200
  remote-as 65001
  update-source Loopback0
  address-family ipv4 labeled-unicast
  !
 !
!
! Only advertise our PE loopbacks, not the entire IGP
route-policy ADVERTISE-PE-LOOPBACKS
  if destination in PE-LOOPBACKS then
    pass
  else
    drop
  endif
end-policy
!
prefix-set PE-LOOPBACKS
  10.0.0.1/32,    ! PE1
  10.0.0.2/32,    ! PE2
  10.0.0.3/32     ! PE3
end-set
!
route-policy ACCEPT-REMOTE-PES
  pass
end-policy
```

**RR1 (AS 65001) — Multihop eBGP VPNv4 to RR2:**
```cisco
router bgp 65001
 ! Normal iBGP VPNv4 to local PEs
 address-family vpnv4 unicast
 !
 ! Multihop eBGP VPNv4 to RR2 in AS2
 neighbor 10.0.1.200
  remote-as 65002
  ebgp-multihop 255
  update-source Loopback0
  address-family vpnv4 unicast
   ! CRITICAL: preserve original PE next-hops — do NOT let eBGP default rewrite them
   ! IOS-XR eBGP changes next-hop to self by default. You MUST override this.
   route-policy PRESERVE-NEXT-HOP out
   route-policy PASS-ALL in
   ! Optionally filter by RT
  !
 !
 ! iBGP to local PEs
 neighbor 10.0.0.1
  remote-as 65001
  update-source Loopback0
  address-family vpnv4 unicast
  !
 !
!
route-policy PASS-ALL
  pass
end-policy
!
! CRITICAL: Preserve original PE next-hops on eBGP VPNv4 to remote RR
! IOS-XR eBGP rewrites next-hop to self by default — this prevents that.
! Without this, you get the exact catastrophic failure from the war story below.
route-policy PRESERVE-NEXT-HOP
  set next-hop unchanged
end-policy
```

> **ASBR2 (AS 65002)** needs a mirror config: eBGP labeled-unicast to ASBR1, iBGP labeled-unicast to its RR, and `allocate-label all` under global `address-family ipv4 unicast`. Both sides must allocate labels for the stitching to work bidirectionally.

> **CRITICAL: IOS-XR eBGP changes next-hop to self by default.** On the RR-to-RR VPNv4 session, you MUST use a route-policy with `set next-hop unchanged` (shown above as `PRESERVE-NEXT-HOP`) to preserve the original PE next-hops. Without this, every VPNv4 route points to the RR as next-hop — the RR has no VPN labels — instant black hole. On Junos, eBGP also changes next-hop to self by default, so the RR-to-RR config must include an export policy with `next-hop unchanged` (shown below as `PRESERVE-NEXT-HOP`).

> **Next-hop resolution chain in PE2's brain:**
> 1. Receives VPNv4 for 192.168.1.0/24, next-hop 10.0.0.1 (PE1 in AS1), VPN label V1
> 2. Looks up 10.0.0.1 — found via iBGP labeled-unicast (learned from ASBR2 who learned from ASBR1), label L2
> 3. Looks up ASBR2 — found via IGP, transport label T2
> 4. Pushes: [T2][L2][V1]

### Option C — Junos

**ASBR1 (AS 65001):**
```junos
protocols {
    bgp {
        group INTER-AS-LABELED {
            type external;
            peer-as 65002;
            neighbor 10.255.0.2;
            family inet {
                labeled-unicast;
                /* labeled-unicast routes install into inet.3 by default on Junos,
                   which is the RIB used for BGP next-hop resolution of VPN routes.
                   No explicit rib configuration needed. */
            }
            export ADVERTISE-PE-LOOPBACKS;
            import ACCEPT-REMOTE-PES;
        }
        group IBGP-CORE {
            type internal;
            local-address 10.0.0.10;
            family inet {
                labeled-unicast;
            }
            neighbor 10.0.0.200;   /* RR */
        }
    }
}

policy-options {
    policy-statement ADVERTISE-PE-LOOPBACKS {
        term PE-ROUTES {
            from {
                protocol [ direct ospf isis ];
                prefix-list PE-LOOPBACKS;
            }
            then {
                next-hop self;
                accept;
            }
        }
        term DEFAULT {
            then reject;
        }
    }
    prefix-list PE-LOOPBACKS {
        10.0.0.1/32;
        10.0.0.2/32;
        10.0.0.3/32;
    }
    policy-statement ACCEPT-REMOTE-PES {
        then accept;
    }
}

/* Junos: labeled-unicast routes go into inet.3 by default,
   which is used for BGP next-hop resolution of VPNv4 routes.
   This is why Option C works naturally on Junos — inet.3 is the glue. */
```

**RR1 (AS 65001) — Multihop eBGP VPNv4:**
```junos
protocols {
    bgp {
        group INTER-AS-RR {
            type external;
            multihop {
                ttl 255;
            }
            peer-as 65002;
            local-address 10.0.0.200;
            family inet-vpn {
                unicast;
            }
            /* CRITICAL: preserve original PE next-hops.
               Junos eBGP also changes next-hop to self by default.
               This export policy overrides that behavior. */
            export PRESERVE-NEXT-HOP;
            neighbor 10.0.1.200;
        }
        group IBGP-CLIENTS {
            type internal;
            cluster 10.0.0.200;
            local-address 10.0.0.200;
            family inet-vpn {
                unicast;
            }
            neighbor 10.0.0.1;
            neighbor 10.0.0.2;
        }
    }
}

policy-options {
    policy-statement PRESERVE-NEXT-HOP {
        then {
            next-hop unchanged;
        }
    }
}

/* IMPORTANT: RR1's loopback (10.0.0.200) must be reachable from RR2 (10.0.1.200).
   This is typically achieved by:
   1. Including RR loopbacks in the labeled-unicast exchange at ASBRs, OR
   2. Static routes on ASBRs for RR loopbacks, OR
   3. Running the multihop eBGP session between ASBR loopbacks instead of RRs
   
   Option 3 is common in practice — ASBRs act as "pseudo-RRs" for inter-AS VPNv4.
   Real RRs handle intra-AS distribution only. */
```

### Key Knobs

| Parameter | Option A | Option B | Option C | Why |
|-----------|----------|----------|----------|-----|
| next-hop-self (ASBR) | N/A (IPv4 eBGP) | **Mandatory** | Only on labeled-unicast | Without NHS on Option B, VPNv4 next-hop is unreachable |
| next-hop-self (RR) | N/A | N/A | **Must NOT set** | Breaks label chain — PE next-hop must be preserved |
| RT filtering | Per-VRF import | eBGP route-policy | eBGP route-policy on RR | Prevents route explosion at ASBR/RR |
| VRF on ASBR | Required (per VPN) | Not required | Not required | Only Option A needs ASBR VRFs |
| labeled-unicast | Not needed | Not needed | **Required** at ASBR | Provides inter-AS label stitching for PE reachability |
| ebgp-multihop | Not needed | Not needed | **Required** for RR-RR | RR-to-RR session spans multiple hops |
| maximum-prefix | Recommended per VRF | **Critical** on VPNv4 | Less critical (few loopbacks) | Option B ASBRs can receive massive VPNv4 |
| BFD | Per-VRF eBGP | Single eBGP session | Both sessions | Detect ASBR link failure fast |

### What Good Looks Like

**Option A:**
- Per-VRF eBGP sessions to ASBR2 are Established
- VRF routes show next-hop as ASBR2's VRF interface IP
- Traceroute from CE1 to CE2 shows IP hops at ASBR (no MPLS labels visible at boundary)

**Option B:**
- Single eBGP VPNv4 session to ASBR2 is Established with correct prefix count
- VPNv4 routes show ASBR as next-hop (due to next-hop-self)
- `show mpls forwarding` on ASBR shows VPN label swap entries
- Traceroute with MPLS extensions shows label swap at ASBR

**Option C:**
- eBGP labeled-unicast session between ASBRs is Established (small prefix count = PE loopbacks)
- Multihop eBGP VPNv4 between RRs is Established
- VPNv4 routes on PE2 show PE1 as next-hop (NOT ASBR — this confirms Option C)
- `show route table inet.3` (Junos) or `show bgp ipv4 labeled-unicast` shows remote PE loopbacks with labels
- `show cef` on PE2 shows 3-label stack for inter-AS VPN prefix

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Option B: VPNv4 routes not received from ASBR2 | Missing `next-hop-self` on ASBR2 (next-hop unresolvable) | Add `next-hop-self` under VPNv4 AF on ASBR eBGP |
| Option B: VPNv4 routes received but no forwarding | ASBR LFIB missing VPN label swap entry | Check `allocate-label all` (IOS-XR) or `labeled-unicast` family |
| Option B: Only some VPNs work | RT filtering at ASBR dropping routes | Check route-policy/import filter on ASBR eBGP session |
| Option C: VPNv4 next-hop unresolvable on remote PE | Labeled-unicast not exchanging PE loopbacks | Verify ASBR labeled-unicast session + PE loopback in `inet.3` |
| Option C: Labeled-unicast up but VPN traffic black-holed | Label not installed in LFIB on ASBR | Check `show mpls forwarding` — label should map to transport to PE |
| Option C: RR-to-RR session won't establish | RR loopback not reachable across ASes | Include RR loopback in labeled-unicast or add static route |
| Option C: VPNv4 works but with wrong next-hop-self on RR | Someone set NHS on the multihop session | Remove `next-hop-self` from RR-to-RR VPNv4 — this breaks Option C |
| Option A: New VPN not working across ASes | Forgot to add VRF + sub-interface + eBGP on both ASBRs | Provision all 4 devices (PE1, ASBR1, ASBR2, PE2) |
| All options: Intermittent packet loss across ASes | MTU mismatch on inter-AS link (MPLS overhead) | Verify MTU ≥ 1518 (or higher for Option C 3-label stack: 1530+) |
| All options: VPN works one direction only | Asymmetric route advertisement or label allocation | Check route advertisement and label entries in both directions |

### Methodology

1. **Identify which option is deployed** — check ASBR config for VRFs (A), VPNv4 eBGP (B), or labeled-unicast + multihop VPNv4 (C)
2. **Verify BGP sessions** — all relevant sessions Established with expected prefix counts
3. **Check next-hop resolution** — can the remote PE's VPNv4 next-hop be resolved?
   - Option A: N/A (local VRF)
   - Option B: ASBR must be next-hop (next-hop-self)
   - Option C: Remote PE must be in labeled-unicast / inet.3
4. **Check label stack** — `show cef vrf <VRF> <prefix>` should show correct label count
   - Option A: 2 labels (within each AS)
   - Option B: 2 labels (within each AS, stitched at ASBR)
   - Option C: 3 labels (transport + stitch + VPN)
5. **Verify LFIB at ASBR** — label swap entries must exist
6. **Traceroute with MPLS** — end-to-end path validation
7. **Check MTU** — label stack adds overhead; inter-AS links often have default 1500 MTU

### War Story

> **The next-hop-self that shouldn't have been there**
>
> Large SP, two regional ASes, Option C deployment. Working fine for 18 months. New engineer joins, sees the multihop eBGP VPNv4 session between RRs doesn't have `next-hop-self` and "fixes" it — because every other eBGP session they'd ever configured used NHS.
>
> Result: All VPNv4 routes now point to the RR as next-hop instead of the originating PE. The RR doesn't have VPN labels for the prefixes (it's a reflector, not a PE). Every inter-AS VPN prefix becomes a black hole. 15,000 VPN customers lose connectivity simultaneously.
>
> Root cause took 45 minutes to find because the BGP sessions were all Established, prefix counts were correct, and `show bgp vpnv4` showed routes with valid-looking paths. The only tell was the next-hop field showing the RR loopback instead of a PE loopback — a subtlety that doesn't jump out when you're panicking.
>
> Lesson: Option C next-hop-self on the inter-AS VPNv4 session is not just wrong — it's catastrophic. Document it, protect it with a comment in the config, and test the config change in lab before touching production.

## Design Considerations

### Choosing Between Options

**Start with the question: "How many VPNs cross AS boundaries?"**

- **< 50 VPNs, high isolation need** → Option A
- **50–5,000 VPNs, single provider** → Option B
- **> 5,000 VPNs, or ASBR RIB is a concern** → Option C
- **Regulatory requirement for per-VPN control** → Option A (or hybrid A+B)

**Second question: "How often do VPNs get added?"**

- Rare (quarterly) → Option A is tolerable
- Frequent (weekly) → Option B or C (no ASBR provisioning needed)

### Multi-ASBR Redundancy

In production, you never run a single ASBR pair. Typical design:

- 2+ ASBR pairs (geographic diversity)
- Option B: eBGP VPNv4 on each pair, BGP multipath at PEs
- Option C: labeled-unicast on each pair, ADD-PATH or multipath for PE loopbacks
- BFD on all inter-AS eBGP sessions (50ms detect time)

### Security at the AS Boundary

The inter-AS link is a trust boundary:

- **GTSM** (Generalized TTL Security Mechanism, RFC 5082): Require TTL=255 on eBGP sessions
- **MD5/TCP-AO** authentication on all BGP sessions
- **Prefix limits**: `maximum-prefix` on all sessions (especially Option B VPNv4)
- **RT filtering**: Only accept RTs you need (Option B)
- **ACLs**: Only allow BGP (TCP/179) and ICMP on inter-AS links
- **RPKI/ROV**: If exchanging IPv4/IPv6 unicast alongside VPNv4

### SRv6 Considerations

With SRv6 transport, Inter-AS changes:
- Option C with SRv6: labeled-unicast is replaced by SRv6 SID exchange (locator prefixes via eBGP)
- VPN SIDs (End.DT4/DT6) are carried in VPNv4 NLRI instead of MPLS labels
- No label stitching — SRv6 SIDs are globally routable if using GUA (but ULA breaks this)
- Most deployments still use SRv6 within AS + MPLS inter-AS (interworking gateway from Section 6.6)

## Quick Reference

- **Key RFCs:** RFC 4364 (Section 10a/b/c), RFC 8277 (labeled-unicast), RFC 4659 (VPNv6), RFC 5082 (GTSM)
- **Cert relevance:** IE-SP (4.0 VPN Technologies — Inter-AS), IE-SP (VPN section — Inter-Provider VPN)
- **Critical rule:** Option C = NO next-hop-self on VPNv4 multihop session
