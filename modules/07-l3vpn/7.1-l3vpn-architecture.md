# 7.1 — MPLS L3VPN Architecture

## Overview

> MPLS L3VPN (RFC 4364, formerly RFC 2547bis) is the backbone revenue engine of every service provider on Earth. It virtualizes the routing table — each customer gets an isolated VRF on the PE, with routes exchanged via MP-BGP between PEs and transported across the MPLS core. The customer sees routed connectivity. The provider sees margin. The architecture is elegant in its simplicity: VRFs provide isolation, Route Distinguishers provide uniqueness, Route Targets provide policy, and the MPLS label stack provides forwarding. If you understand L3VPN deeply, you understand how SPs actually make money.


## Key Concepts
> **After this section you will understand:**

> - how push, swap, and pop operations are applied hop by hop
> - where control-plane and data-plane mismatches create black holes

## Theory

### The Problem L3VPN Solves

Without L3VPN, a service provider offering routed connectivity to multiple customers faces an impossible scaling challenge:

1. **Routing table pollution** — Customer A's 10.0.0.0/8 collides with Customer B's 10.0.0.0/8
2. **Security isolation** — Customers must not see or reach each other's networks
3. **Operational scalability** — Adding a new customer shouldn't require touching every router in the network

L3VPN solves all three by separating the **customer routing domain** from the **provider routing domain**.

### Architecture Components

```
Customer A                                                Customer A
Site 1                                                    Site 2
  CE1 -------- PE1 ========= P ========= PE2 -------- CE2
               |    MPLS Core (labels)    |
  CE3 -------- PE1                       PE2 -------- CE4
Customer B                                                Customer B
Site 1                                                    Site 2
```

**Roles:**

| Device | Role | Runs VRF? | Runs MP-BGP VPNv4? | Runs LDP/SR? |
|--------|------|-----------|---------------------|--------------|
| CE | Customer Edge | No | No (runs eBGP/OSPF/static to PE) | No |
| PE | Provider Edge | Yes | Yes (VPNv4/VPNv6 address families) | Yes |
| P  | Provider core | No | No (label-switches only) | Yes |

The PE is where all the magic happens. P routers are blissfully ignorant of VPN state — they just swap labels.

### VRF — Virtual Routing and Forwarding

A VRF is an isolated routing table instance on the PE. Each customer gets their own VRF. Think of it as a logical router inside the physical router.

**What a VRF contains:**
- Its own RIB (routing table) — completely separate from the global table
- Its own FIB (forwarding table) — separate CEF/forwarding entries
- Its own set of interfaces — physical or logical, assigned to the VRF
- Its own PE-CE routing protocol instance (eBGP, OSPF, static, etc.)

**Key insight:** A VRF is NOT a VLAN. It operates at Layer 3. Two VRFs on the same PE can use identical IP address space without conflict. The isolation is in the routing/forwarding plane, not the data link layer.

**VRF capacity planning:**
- Modern PE routers (NCS 5500, MX304, ASR 9000) support thousands of VRFs
- Practical limit is usually memory for route state, not VRF count
- Rule of thumb: each VRF costs ~2-5 MB base memory + per-route overhead
- Watch `show processes memory` and FIB TCAM utilization

### Route Distinguisher (RD)

The RD makes overlapping customer prefixes unique in BGP. It's a 64-bit value prepended to the IPv4 prefix to create a **VPNv4** prefix (96 bits total).

**Format options:**

| Type | Format | Example | When to Use |
|------|--------|---------|-------------|
| Type 0 | ASN:nn | 65000:100 | Single-AS networks |
| Type 1 | IP:nn | 192.0.2.1:100 | Multi-AS or inter-provider |
| Type 2 | 4-byte-ASN:nn | 4200000001:100 | 4-byte ASN environments |

**Critical misconception: The RD is NOT a routing policy tool.**

The RD exists solely to make prefixes unique in the BGP table. It has **zero influence** on route import/export. That's the Route Target's job.

**RD assignment strategies:**

| Strategy | Example | Pros | Cons |
|----------|---------|------|------|
| Per-VRF unique | PE-loopback:VRF-id | Best-path selection works correctly; enables per-CE path visibility | More RDs to track |
| Per-customer same | ASN:customer-id | Simple, fewer values | Breaks best-path for multi-homed customers (BGP sees same NLRI, picks one) |

**The per-VRF unique RD is the only correct answer for production networks.** If you use the same RD on two PEs for the same customer VRF, the route reflector will perform best-path selection on the VPNv4 prefix and may advertise only one path — silently breaking dual-homing. This is the #1 L3VPN design mistake.

### Route Target (RT)

Route Targets are BGP Extended Communities (64-bit) that control VPN membership. They are the **policy knobs** of L3VPN.

**Two directions:**

| Direction | Meaning | Config keyword |
|-----------|---------|----------------|
| Export RT | "Tag this route with RT X when sending to MP-BGP" | `route-target export` |
| Import RT | "Accept routes tagged with RT X into this VRF" | `route-target import` |

**Basic model — symmetric RT:**
```
VRF CUST-A on PE1: export 65000:100, import 65000:100
VRF CUST-A on PE2: export 65000:100, import 65000:100
→ Both PEs exchange routes. Customer A has connectivity.
```

**Advanced model — asymmetric RT (hub-and-spoke):**
```
Hub PE (HQ):    export 65000:100, import 65000:101
Spoke PE (branch): export 65000:101, import 65000:100
→ Spokes can reach hub. Hub can reach spokes. Spokes cannot reach each other directly.
```

**RT is where network architects earn their money.** Complex topologies (partial mesh, extranet, shared services) are all just RT import/export policy. Section 7.4 covers this in depth.

### The Label Stack

L3VPN forwarding uses a **two-label stack**:

```
┌──────────────┬──────────────┬──────────────────┐
│ Transport    │ VPN Label    │ Customer IP      │
│ Label (LDP/  │ (allocated   │ Packet           │
│ SR prefix-   │ per-VRF or   │                  │
│ SID)         │ per-prefix   │                  │
│              │ by egress PE)│                  │
└──────────────┴──────────────┴──────────────────┘
  ↑ Swapped        ↑ Untouched      ↑ Untouched
  by P routers     until egress PE  until egress PE
```

**Label roles:**

| Label | Allocated By | Purpose | Scope |
|-------|-------------|---------|-------|
| Transport (outer) | LDP or SR (prefix-SID) | Get the packet across the core to the egress PE | P-router hop-by-hop |
| VPN (inner) | Egress PE, advertised in MP-BGP | Identify the VRF on the egress PE | PE-to-PE, end-to-end |

**VPN label allocation modes:**

| Mode | Behavior | Scale Impact | Use Case |
|------|----------|-------------|----------|
| Per-prefix | Unique label per VPN route | High label consumption, best granularity | Small scale, traffic engineering per prefix |
| Per-VRF | One label for entire VRF | Low label consumption | Large scale (1000s of VRFs) |
| Per-CE | One label per CE neighbor | Middle ground | Common default on IOS-XR |
| Per-nexthop (aggregate) | One label per BGP next-hop | Very efficient | Massive scale |

IOS-XR default: **per-prefix** allocation (each VPN route gets its own label). Junos default without `vrf-table-label`: **per-next-hop** (one label per protocol next-hop in the VRF — routes sharing the same CE next-hop share one label). Adding `vrf-table-label` switches Junos to **per-VRF** allocation (single label for the entire routing-instance + IP lookup — most common production deployment).

### Control Plane: MP-BGP VPNv4/VPNv6

The control plane is pure BGP. No new protocols. MP-BGP (RFC 4760) carries VPN routes in the **VPNv4** (AFI 1, SAFI 128) and **VPNv6** (AFI 2, SAFI 128) address families.

**What travels in the MP-BGP update:**

| Field | Content |
|-------|---------|
| NLRI | RD:prefix (e.g., 192.0.2.1:100:10.1.1.0/24) |
| Next-hop | PE loopback (must be reachable via IGP + LDP/SR) |
| Extended Community | Route Target(s), SOO, etc. |
| Label | VPN label allocated by the advertising PE |
| Standard attributes | AS-path, local-pref, MED, communities — all work normally |

**Route Reflector placement for VPNv4:**

In a scaled SP network, PEs don't peer directly. Route Reflectors (RRs) centralize VPNv4 distribution.

```
        RR1 -------- RR2
       / | \        / | \
     PE1 PE2 PE3  PE4 PE5 PE6
```

**Critical RR requirement:** The RR does NOT need VRF configuration. It reflects VPNv4 routes without importing them. However, it MUST:
- Support the VPNv4/VPNv6 address family
- Have `next-hop-unchanged` behavior (default for iBGP reflected routes)
- NOT modify the VPN label (it doesn't — labels are opaque to the RR)

**RT-constrained route distribution (RFC 4684):**

Without RT-constraint, every PE receives every VPN route from the RR, even for VRFs it doesn't have. At scale (100K+ VPN routes), this wastes memory on every PE.

RT-constraint (AFI 1, SAFI 132) lets PEs advertise which RTs they're interested in. The RR only sends matching routes.

```
PE1 → RR: "I only want routes with RT 65000:100 and 65000:200"
RR → PE1: Only sends VPNv4 routes carrying those RTs
```

**This is mandatory at scale.** Any network with >50 VRFs should enable RT-constraint.

### Data Plane: Packet Walk

Let's trace a packet from CE1 (Customer A, Site 1) to CE2 (Customer A, Site 2):

```
Step 1: CE1 → PE1
  CE1 sends plain IP packet (dst: 10.2.1.1) to PE1
  PE1 receives on VRF-CUST-A interface

Step 2: PE1 VRF lookup
  PE1 looks up 10.2.1.1 in VRF CUST-A's FIB
  Finds: next-hop PE2 (192.0.2.2), VPN label 24001, transport label 16002

Step 3: PE1 → P (label push)
  PE1 pushes two labels:
    Bottom: 24001 (VPN label, identifies VRF CUST-A on PE2)
    Top:    16002 (transport label, gets packet to PE2)

Step 4: P router (label swap)
  P receives labeled packet
  Swaps transport label: 16002 → 16022 (or pops if PHP)
  VPN label untouched (P doesn't know or care about VPNs)

Step 5: PE2 (label pop + VRF forwarding)
  PE2 receives packet with VPN label 24001
  Looks up label 24001 → maps to VRF CUST-A
  Pops VPN label, does IP lookup in VRF CUST-A
  Forwards plain IP packet to CE2

Step 6: PE2 → CE2
  Plain IP packet delivered to CE2
```

**With Penultimate Hop Popping (PHP):**
The P router pops the transport label (top), so PE2 receives only the VPN label. This saves PE2 one lookup operation. With SR-MPLS, PHP behavior depends on the prefix-SID configuration (explicit-null vs implicit-null).

### PE-CE Routing Protocols

The PE-CE link runs a standard routing protocol. The PE redistributes CE-learned routes into MP-BGP VPNv4 (and vice versa).

| Protocol | Use Case | Pros | Cons |
|----------|----------|------|------|
| eBGP | Large/enterprise customers, multi-homed sites | Scalable, policy-rich, loop prevention via AS-path | More complex for small customers |
| OSPF | Enterprise customers with existing OSPF | Familiar to customers, sham-link for backdoor handling | DN bit, domain-id complexity; OSPF↔BGP redistribution |
| Static | Small sites, managed CPE | Simple, no protocol overhead | No dynamic failover |
| EIGRP | Cisco IOS-XE only (NOT IOS-XR) | Customers already running EIGRP | Vendor lock-in, not supported on IOS-XR PEs |
| RIPv2 | Legacy, very small sites | Dead simple | No one should be deploying this in 2026 |

**eBGP is the correct answer 90% of the time.** It's the most scalable, most policy-flexible, and provides natural loop prevention. OSPF is acceptable when the customer requires it, but introduces complexity (DN bit, sham-links, domain-id — covered in 7.5).

### SOO — Site of Origin

Site of Origin (SOO) is an extended community that prevents routing loops in multi-homed VPN sites.

**The problem:** Customer A has two CEs at Site 1, both connected to different PEs. Without SOO, a route learned from CE1 on PE1 gets advertised via MP-BGP to PE2, which re-advertises it to CE2 at Site 1. CE2 might prefer this MP-BGP-learned path over the direct path, creating a suboptimal routing loop through the MPLS core.

**The fix:**
```
PE1 tags routes from Site 1 CEs with SOO 65000:1001
PE2 tags routes from Site 1 CEs with SOO 65000:1001
When PE2 receives a VPNv4 route with SOO 65000:1001, the route IS
installed in the VRF RIB — but PE2 will NOT advertise it out to any
CE neighbor whose interface/session is also tagged with SOO 65000:1001.
The filtering happens at export to the CE, not at VRF import.
```

On IOS-XR, SOO is applied via route-policy on the BGP neighbor (not directly on the interface). On IOS-XE, it can be configured per-interface. On Junos, it's applied via import policy on the BGP neighbor.

### Junos

```junos
## === VRF (routing-instance) Definition ===
routing-instances {
    CUST-A {
        instance-type vrf;
        interface ge-0/0/1.0;
        route-distinguisher 192.0.2.1:100;
        vrf-target target:65000:100;
        ## Or asymmetric:
        ## vrf-import CUST-A-IMPORT;
        ## vrf-export CUST-A-EXPORT;
        vrf-table-label;             ## Allocates single VPN label per VRF

        ## Note: VRF inherits AS from global routing-options.
        ## If global AS not set, add: routing-options { autonomous-system 65000; }

        protocols {
            bgp {
                group CE-CUST-A {
                    type external;
                    peer-as 65100;
                    neighbor 10.0.1.2 {
                        family inet {
                            unicast;
                        }
                        import CUST-A-CE-IN;
                        export CUST-A-CE-OUT;
                        as-override;
                    }
                }
            }
        }
    }
}

## === Interface Assignment ===
interfaces {
    ge-0/0/1 {
        unit 0 {
            family inet {
                address 10.0.1.1/30;
            }
        }
    }
}

## === MP-BGP VPNv4 on PE ===
protocols {
    bgp {
        group IBGP-VPNv4 {
            type internal;
            local-address 192.0.2.1;
            family inet-vpn {
                unicast;
            }
            family inet6-vpn {
                unicast;
            }
            neighbor 192.0.2.250;
        }
    }
}

## === SOO Configuration ===
policy-options {
    community SOO-SITE1 members origin:65000:1001;

    policy-statement SOO-SITE1-IN {
        term SET-SOO {
            then {
                community add SOO-SITE1;
                accept;
            }
        }
    }
}

routing-instances {
    CUST-A {
        protocols {
            bgp {
                group CE-CUST-A {
                    neighbor 10.0.1.2 {
                        import SOO-SITE1-IN;
                    }
                }
            }
        }
    }
}

## === RT-Constrained Distribution (on RR) ===
protocols {
    bgp {
        group RR-CLIENTS {
            family route-target;     ## Enable RT-constraint
        }
    }
}
```

### Key Knobs

| Parameter | IOS-XR Default | Junos Default | Recommended | Why |
|-----------|---------------|---------------|-------------|-----|
| RD format | Manual | Manual | PE-loopback:VRF-id | Unique per PE, enables multi-path |
| Label mode | per-prefix | per-next-hop (no vrf-table-label) / per-vrf (with vrf-table-label) | per-vrf or per-ce | Balance scale vs granularity |
| RT-constraint | Disabled | Disabled | **Enable** | Mandatory at scale (>50 VRFs) |
| as-override | Disabled | Disabled | Use when needed | Required when customer uses same ASN at multiple sites |
| SOO | Not set | Not set | Set for multi-homed sites | Prevents routing loops |
| allowas-in | Disabled | loops 0 | Avoid if possible | Weakens loop prevention; use as-override + SOO instead |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| VRF route table empty | RT mismatch (import doesn't match remote export) | Verify `show bgp vpnv4 unicast` — routes exist but not imported? Check RT values. |
| Routes in BGP but not in VRF FIB | Next-hop unreachable (no LDP/SR label to remote PE) | `show cef vrf CUST-A` — check for "no label" or "unresolved". Verify `show mpls forwarding` has label for PE2 loopback. |
| Customer traffic black-holed | VPN label mismatch or stale label | `show mpls forwarding labels <vpn-label>` on remote PE — does the label still map to the right VRF? |
| Dual-homed site gets asymmetric routing | Same RD on both PEs → RR selects only one path | Use unique RD per PE. Check `show bgp vpnv4 unicast rd <rd>` on RR for path diversity. |
| Route learned from CE re-advertised back to same site | Missing SOO | Configure SOO on PE-CE interfaces for multi-homed sites. |
| PE-CE OSPF routes have DN bit, rejected | Routes redistributed from BGP into OSPF get DN bit set | Expected behavior — prevents loops. Use sham-link for backdoor OSPF connectivity (see 7.5). |
| VPNv4 session to RR won't establish | Missing address-family on RR | RR must have `address-family vpnv4 unicast` configured under the neighbor. |
| Massive memory use on PEs | No RT-constraint → every PE gets all VPN routes | Enable `address-family rtfilter` (IOS-XR) or `family route-target` (Junos). |

### Methodology

1. **Verify VRF exists and has interfaces:** `show vrf <name>` — confirm interfaces are UP
2. **Check PE-CE adjacency:** `show bgp vrf <name> summary` — is the CE neighbor established?
3. **Confirm local routes in VRF:** `show route vrf <name>` — are CE routes present?
4. **Check VPNv4 export:** `show bgp vpnv4 unicast rd <rd>` — are local routes in VPNv4 with correct RT?
5. **Check VPNv4 on remote PE:** Same command on remote PE — did routes arrive?
6. **Verify RT import:** Does remote VRF import the RT? `show vrf <name> detail`
7. **Check transport label:** `show cef vrf <name> <prefix>` — is there a valid label stack?
8. **Verify MPLS path:** `traceroute mpls ipv4 <PE-loopback>/32` — is the LSP intact?
9. **End-to-end ping:** `ping vrf <name> <destination>` from PE, then from CE

### War Story: The RD That Broke Dual-Homing

**Scenario:** A Tier 2 SP deployed L3VPN for a banking customer with 200 branches. HQ had dual-homed CE routers to two PEs (PE-NYC-1 and PE-NYC-2). The design team used `65000:CUSTID` as the RD on all PEs for simplicity.

**What happened:** Everything worked for 6 months. Then the customer reported intermittent connectivity from 30 branch offices to HQ. Not all branches — just some, and it changed over time.

**Root cause:** Both PEs advertised the HQ prefix (10.100.0.0/16) with the same RD (`65000:5001`). The Route Reflector saw two identical VPNv4 NLRIs (same RD + prefix), ran best-path selection, and picked ONE. The loser was withdrawn from all PEs. When the RR picked PE-NYC-1's path, branches connected to PE-NYC-2 lost their direct path to HQ. Traffic either took a suboptimal path through PE-NYC-1 or, if PE-NYC-2 had no alternative, was black-holed.

**The twist:** The RR's best-path selection was deterministic but unstable. Any route refresh, RR failover, or BGP flap could flip which PE won — explaining the intermittent, time-varying failures.

**Fix:** Changed to per-PE RD (`loopback:100`). Both paths now had unique VPNv4 NLRIs. The RR reflected BOTH. PEs could see multiple paths and use BGP Add-Path or best-external to maintain dual-homing. Time to fix: 15 minutes of config change. Time to diagnose: 3 weeks.

**Alternative fix:** BGP Add-Path (RFC 7911) allows the RR to advertise multiple paths for the same NLRI, even with identical RDs. However, unique-RD-per-PE is still the recommended design — Add-Path adds complexity and both solutions should be understood for expert-level exams.

**Lesson:** *Never use the same RD for the same VRF on different PEs.* The 15 characters you save in the config will cost you 3 weeks in troubleshooting.

## Design Considerations

### When to Use L3VPN

| Scenario | L3VPN? | Why |
|----------|--------|-----|
| Multi-tenant routed service | ✅ Yes | This is what it was built for |
| Enterprise WAN (managed by SP) | ✅ Yes | Customer gets routed connectivity, SP manages core |
| Internal network segmentation | ✅ Yes | Isolate departments, PCI zones, IoT |
| Layer 2 extension needed | ❌ No | Use L2VPN/EVPN (Module 8) |
| Point-to-point circuit emulation | ❌ No | Use VPWS |
| Internet transit | ❌ No | Use global table or dedicated VRF with full table |

### Scale Planning

| Metric | Small SP | Medium SP | Large SP |
|--------|----------|-----------|----------|
| VRFs per PE | <100 | 100-1000 | 1000-4000 |
| VPN routes per PE | <10K | 10K-100K | 100K-1M+ |
| Label mode | per-prefix OK | per-ce recommended | per-vrf mandatory |
| RT-constraint | Optional | Recommended | **Mandatory** |
| RR redundancy | 1 pair | 2 pairs (geo-diverse) | Hierarchical RR |

### RD/RT Allocation Scheme

A disciplined allocation scheme prevents operational chaos:

```
RD:   <PE-loopback>:<VRF-numeric-id>
      Example: 192.0.2.1:100

RT:   <ASN>:<service-type><customer-id>
      Example: 65000:1100  (1=L3VPN, 100=customer-id)
      
Service type prefix:
  1xxx = L3VPN
  2xxx = L2VPN
  3xxx = EVPN
  9xxx = Infrastructure/management
```

Document this. Put it in your NOC wiki. Tattoo it on the junior engineer's forearm if necessary.

### as-override vs allowas-in

When a customer uses the same AS number at multiple sites, BGP loop prevention will reject routes because the customer's ASN appears in the AS-path.

| Feature | as-override | allowas-in |
|---------|-------------|------------|
| How it works | PE replaces customer ASN with its own ASN in the AS-path | CE accepts routes despite own ASN in path |
| Configured on | PE | CE (or PE for the VRF neighbor) |
| Loop prevention | SOO required to prevent loops | Weak — manually set max occurrences |
| Recommended | ✅ Yes, with SOO | ⚠️ Last resort only |

**Always use as-override + SOO.** `allowas-in` is a band-aid that weakens BGP's fundamental loop prevention.
