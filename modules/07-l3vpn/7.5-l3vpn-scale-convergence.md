# 7.5 — L3VPN Scale & Convergence

## Overview

> The first four sections taught you how L3VPN works. This section teaches you how it works at **scale** — when you have 50,000 VPN prefixes on a PE, 400 VRFs on a route reflector, and a core link failure that needs sub-second recovery. L3VPN convergence is where theory meets physics: BGP timers, label allocation modes, RIB/FIB programming time, and prefix-independent convergence (PIC) all interact in ways that are non-obvious until you've debugged a 45-second convergence event at 2am. This is the section that separates "I can configure L3VPN" from "I can run L3VPN in production."

## Theory

### The Scale Problem

SP L3VPN networks regularly hit numbers that break assumptions built for enterprise:

| Metric | Small SP | Large SP | Hyperscale |
|--------|---------|---------|------------|
| VPN customers | 50–200 | 1,000–5,000 | 10,000+ |
| VRFs per PE | 10–50 | 100–500 | 500–2,000 |
| VPNv4 prefixes (total) | 10K–50K | 100K–500K | 1M–5M |
| VPNv4 prefixes per PE | 5K–20K | 50K–200K | 200K–1M |
| Route Reflectors | 2 | 4–8 | 8–16 (hierarchical) |
| BGP sessions per PE | 2–4 | 4–8 | 8–16 |

Every one of these numbers affects **memory**, **convergence time**, and **control plane CPU**. The relationship isn't linear — it's often worse.

### Where Scale Hurts

#### 1. Route Reflector Memory and Processing

The RR holds **all** VPNv4/VPNv6 routes from **all** PEs. With 500K VPNv4 prefixes and full path attributes, a single RR consumes 8–16 GB of BGP RIB memory. Each BGP update triggers:

1. Receive and parse the update
2. Run import policy (if any)
3. Run bestpath selection
4. For every client: check RT-Constraint, run export policy, format update
5. Serialize and send

Step 4 is the killer. Without RT-Constraint (Section 7.2), the RR sends **every** VPN route to **every** PE. A PE with 10 VRFs gets routes for all 500 VRFs. The PE then imports the 10 it cares about and throws away the other 490 VRFs worth of routes. This wastes bandwidth, memory, and CPU on both the RR and every PE.

**RT-Constraint (RFC 4684)** fixes this: PEs advertise which RTs they need, and the RR only sends matching routes. This is not optional at scale — it's survival.

#### 2. PE Memory: RIB vs FIB

PEs have two tables that matter for VPN:

| Table | Contents | Size Driver | Memory Impact |
|-------|----------|-------------|---------------|
| **BGP RIB** (Adj-RIB-In) | All received VPNv4/VPNv6 paths | Total routes × paths per route | Software memory (RAM) |
| **VRF RIB** | Imported routes per VRF | Routes matching import RT | Software memory (RAM) |
| **VRF FIB** (CEF/PFE) | Forwarding entries per VRF | Same as VRF RIB (minus inactive) | Hardware TCAM/LEM/LPM |

The FIB is the constraint. ASICs have finite forwarding table space:

| Platform (2026) | IPv4 FIB Entries | IPv6 FIB Entries | VPN Label Entries |
|-----------------|-----------------|-----------------|-------------------|
| NCS-5500 (eTCAM) | 4M | 2M | 2M |
| NCS-5700 (Jericho2) | 8M | 4M | 4M |
| ASR 9000 (Memory) | 8M+ | 4M+ | 4M+ |
| MX304 (Memory chipset) | 4M | 2M | 2M |
| MX10008 (LC) | 8M+ | 4M+ | 4M+ |

When the FIB fills up, new routes get **software-forwarded** (punt to CPU) or **dropped**. Neither is acceptable. Monitor FIB usage religiously.

#### 3. Label Allocation and FIB Pressure

Label allocation mode (Section 7.2) directly impacts FIB scale:

| Mode | Labels Consumed | FIB Entries | Use Case |
|------|----------------|-------------|----------|
| **Per-prefix** | One per VPN prefix | N (prefix count) | Default IOS-XR — maximum granularity |
| **Per-VRF** | One per VRF | V (VRF count) | IOS-XR `label mode per-vrf`, Junos `vrf-table-label` — best scale |
| **Per-CE** | One per CE neighbor per VRF | C × V | Middle ground — allows per-CE ECMP |
| **Per-next-hop** | One per unique next-hop per VRF | NH × V | Junos option (`per-nexthop` knob) |

**Per-VRF** is the scale winner: 500 VRFs = 500 labels regardless of prefix count. The trade-off: all traffic for a VRF arrives with the same label, so the egress PE must do an additional IP lookup in the VRF to determine the CE next-hop. On modern ASICs this lookup is done in hardware (additional pipeline stage), but it does consume an extra TCAM/LPM lookup cycle and slightly increases per-packet processing.

**Per-prefix** gives the cleanest forwarding (label directly maps to CE next-hop — no IP lookup) but consumes one label per VPN prefix. At 200K prefixes, that's 200K label entries in the FIB.

**Recommendation for scale:**
- < 50K VPN prefixes: per-prefix is fine
- 50K–200K: per-CE or per-next-hop
- > 200K: per-VRF mandatory

#### 4. BGP Update Processing Time

When a PE goes down, its RR must:
1. Detect the failure (BGP hold timer expiry — default **90 seconds**, or BFD — **sub-second**)
2. Withdraw all routes from that PE
3. Run bestpath for every affected prefix
4. Send updates to all affected PEs

Steps 2–4 scale linearly with prefix count. A PE advertising 100K VPN prefixes generates 100K withdrawals. The RR processes them, runs bestpath 100K times, then generates 100K updates to every affected PE. Processing rates vary dramatically by platform:

| RR Platform | Updates/Second | 100K Prefix Recovery |
|-------------|---------------|---------------------|
| Legacy (2015-era) | 10K–50K | 2–10s |
| Modern dedicated RR (2026) | 100K–500K | 0.2–1s |

Even modern RRs spend meaningful time on bulk updates. The point isn't the exact number — it's that processing time scales linearly with prefix count, and PIC eliminates this dependency entirely.

### Convergence Components

L3VPN convergence is the sum of multiple sequential stages:

```
Total Convergence = Detection + Propagation + Processing + Programming

Detection:    How long until the failure is noticed
Propagation:  How long for BGP updates to reach all PEs
Processing:   How long for PEs to run bestpath + policy
Programming:  How long for new entries to be installed in hardware FIB
```

#### Detection Time

| Method | Time | Notes |
|--------|------|-------|
| BGP Hold Timer (default) | 180s (IOS-XR: 3×60s) / 90s (Junos: 3×30s) | Unacceptable for production |
| BGP Hold Timer (tuned) | 9–15s (3 × 3–5s keepalive) | Better, still slow |
| BFD (single-hop) | 50–300ms (typical: 3 × 50ms) | Standard for PE-PE/PE-RR |
| BFD (multihop) | 300ms–1s | PE-to-RR over multiple hops |
| IGP convergence (transport) | 1–5s | For core link failures — triggers BGP next-hop change |
| Fast External Fallover | Immediate (interface down) | eBGP PE-CE only |

**Best practice:** BFD on all iBGP sessions (PE-RR). Fast external fallover on all eBGP PE-CE sessions. Never rely on BGP hold timer alone.

#### Propagation Time

BGP update propagation from RR to PE depends on:
- **TCP window size**: Default 16KB, should be tuned to 1MB+ for bulk updates
- **Update generation rate**: RR must serialize updates per-peer
- **MRAI timer**: Minimum Route Advertisement Interval — IOS-XR default 0s for iBGP (good), Junos default 0s for iBGP (good). eBGP default 30s — reduce to 5s for PE-CE
- **Pack multiple prefixes**: BGP packs multiple NLRIs per update message (up to 4096 bytes per message)

#### Processing Time

Bestpath runs per-prefix. For VPN routes, the algorithm is:
1. Standard BGP bestpath (weight → local-pref → AS-path → origin → MED → eBGP/iBGP → IGP metric)
2. **IGP metric to next-hop** is the typical tiebreaker for VPN routes (same AS, same policies)
3. Router-ID tiebreaker (last resort)

Processing time per prefix: ~10–50μs on modern hardware. For 100K prefixes: 1–5 seconds. This is unavoidable without PIC.

#### Programming Time

FIB programming is the silent killer. After BGP picks new best paths, each entry must be:
1. Converted to a forwarding entry (label, next-hop, interface)
2. Pushed to the line card ASIC via DMA/IPC
3. Acknowledged by the hardware

Rates vary dramatically:

| Platform | FIB Programming Rate | 100K Route Change |
|----------|---------------------|-------------------|
| NCS-5500 | ~50K entries/sec | ~2s |
| ASR 9000 | ~100K entries/sec | ~1s |
| MX Series (Memory) | ~60K entries/sec | ~1.7s |

During programming, traffic to not-yet-updated prefixes follows the **old** path — which may be a black hole if the old next-hop is down.

### BGP Prefix Independent Convergence (PIC)

PIC is the answer to the "processing + programming" bottleneck. Instead of re-computing and re-programming every affected prefix individually, PIC pre-computes backup paths and installs them **before** the failure.

#### How PIC Works

Without PIC (traditional convergence):
```
Failure → Detect → Withdraw 100K routes → Bestpath 100K times → Program 100K FIB entries
Total: Detection + 10s processing + 2s programming = 12+ seconds
```

With PIC:
```
Normal operation:  For each prefix, compute PRIMARY and BACKUP next-hop
                   Install both in FIB (primary active, backup standby)
Failure →          Detect → Update ONE pointer (next-hop group) → All prefixes switch
Total: Detection + ~50ms pointer update = sub-second
```

The key insight: if 100K prefixes all point to the same **next-hop** (PE), you don't need to update 100K FIB entries. You update the **next-hop entry** that all 100K prefixes reference. This is O(1) instead of O(n).

#### PIC Architecture

```
┌─────────────────────────────────────────────┐
│                VPN FIB                       │
│                                              │
│  Prefix A ──→ ┌─────────────────┐            │
│  Prefix B ──→ │  Next-Hop Group │──→ PE2     │
│  Prefix C ──→ │  (shared)       │──→ PE3 [backup]
│  ...          └─────────────────┘            │
│  Prefix N ──→                                │
└─────────────────────────────────────────────┘

Failure of PE2:
  - Update Next-Hop Group: PE2 → PE3 (one operation)
  - ALL N prefixes instantly use PE3
  - No per-prefix FIB update needed
```

#### PIC Variants

| Variant | Scope | What It Protects | Backup Source |
|---------|-------|-----------------|---------------|
| **BGP PIC Edge** | PE-CE link failure | CE next-hop down | Alternate CE or alternate PE (multi-homed) |
| **BGP PIC Core** | PE node/link failure | Remote PE unreachable | Alternate PE (via RR ADD-PATH or multi-homed CE) |

**BGP PIC Edge**: The local PE has multiple paths to the same CE (e.g., via two interfaces or two CEs at the same site). When the primary CE path fails, traffic switches to the backup instantly.

**BGP PIC Core**: A remote PE fails. All prefixes advertised by that PE need to switch to an alternate PE (which requires the prefix to be reachable via another PE — i.e., multi-homed sites). This requires:
- **ADD-PATH** on the RR (to advertise both primary and backup paths)
- Or **RD-per-PE** with different RDs generating distinct VPNv4 prefixes (both selected by RR)

#### PIC and ADD-PATH

Without ADD-PATH, the RR sends only the **best** path for each VPNv4 prefix. If PE1 and PE2 both advertise 10.0.0.0/24 with the same RD, the RR picks one and sends only that. When the chosen PE fails, the RR must re-run bestpath, select PE2's path, and send updates to all PEs. This is the slow path.

With ADD-PATH, the RR sends **both** paths. The receiving PE installs the best as primary and the second as backup. When the primary PE fails, the local PE switches to the backup immediately — no BGP update needed. The RR still sends a withdrawal eventually, but convergence already happened.

**RD-per-PE alternative**: If each PE uses a unique RD, the same customer prefix (e.g., 10.0.0.0/24) generates two distinct VPNv4 prefixes (RD1:10.0.0.0/24 and RD2:10.0.0.0/24). The RR naturally sends both because they're different NLRIs. The receiving PE imports both into the VRF and uses bestpath to select primary/backup. This achieves the same result as ADD-PATH without requiring ADD-PATH capability. **This is why the recommendation in Section 7.1 was RD-per-PE.**

### RR Scale Design

#### Flat vs Hierarchical RR

| Design | RRs | Sessions per PE | VPN Routes on RR | Failure Domain |
|--------|-----|----------------|-------------------|----------------|
| **Flat** (2 RRs) | 2 | 2 | ALL routes | Entire network |
| **Regional** (2 per region) | 4–8 | 2 | Regional routes only | Region |
| **Hierarchical** (tier-1 + tier-2) | 6–16 | 2 (to local tier-2) | Tier-2: regional; Tier-1: aggregated | Region (tier-2), Global (tier-1) |

Hierarchical RR adds complexity but solves two problems:
1. **Memory**: Regional RRs only hold routes for their region (with RT-Constraint, they don't even receive routes for other regions unless a local PE imports them)
2. **Convergence**: A PE failure only requires re-computation on the regional RR, not the global RR

#### Dedicated vs In-Path RR

**Dedicated RR** (recommended):
- Separate box (or VM), not in the forwarding path
- Can use a cheaper platform (doesn't need ASIC forwarding)
- Failure doesn't affect traffic forwarding
- IOS-XR: XRv 9000 or NCS-55A1 with route-reflector config
- Junos: vMX or MX150 with `cluster` configured

**In-path RR** (legacy):
- A PE or P router that also acts as RR
- Saves hardware but couples control plane and data plane failures
- A control plane crash on the RR disrupts both BGP reflection AND traffic forwarding
- Avoid in new deployments

#### RR Placement and Anycast

Best practice for RR resilience:
- Place RRs in **different physical locations** (different POPs, different power feeds)
- Use **anycast loopback** for the RR cluster: both RRs advertise the same /32 via IGP. PEs connect to "the nearest RR" automatically. If one RR fails, IGP reconverges and sessions move to the surviving RR.
- Set BGP **hold-timer to 30s** on RR sessions (more lenient than PE-CE) — RR failures should be rare, and aggressive timers cause false positives under CPU load.

#### Optimal Route Reflection (ORR)

Dedicated RRs have a known problem: they select bestpath based on **their own** IGP metric to the next-hop, not the client PE's metric. This can cause suboptimal routing — the RR picks a path that's best from the RR's perspective but not from the PE's.

**ORR (RFC 9107)** solves this: the RR computes bestpath from each client's IGP location, ensuring the PE receives the path it would have chosen if it had full visibility. This is critical for large networks where RRs are in a central location far from edge PEs.

- IOS-XR: `optimal-route-reflection <group-name>` under BGP address-family
- Junos: `optimal-route-reflection` under BGP group

See Section 7.2 for detailed ORR configuration.

### Graceful Restart (GR) and Non-Stop Routing (NSR)

Both features aim to preserve VPN forwarding during control plane restarts (process restart, RP switchover, software upgrade).

#### BGP Graceful Restart (RFC 4724)

During a BGP restart:
1. Restarting router sends BGP OPEN with GR capability (restart flag)
2. Peers **keep stale routes** in their RIB for the restart time (default 120s)
3. Restarting router comes back, re-establishes sessions, sends End-of-RIB marker
4. Peers delete any routes not refreshed by End-of-RIB

**VPN implication**: GR preserves VPN forwarding state in the FIB during RSP switchover. Traffic continues to flow using stale FIB entries while the control plane rebuilds. Without GR, peers immediately withdraw all routes from the restarting PE, causing a full convergence event for every customer.

**GR timers matter:**
- `restart-time`: How long peers wait before deleting stale routes (default 120s, set to match RSP switchover time + margin)
- `stale-path-time`: How long to keep stale paths after session re-establishment (default 360s)
- `End-of-RIB`: Marker signaling all routes have been re-advertised — peers then purge any remaining stale routes

#### Non-Stop Routing (NSR)

NSR is the alternative to GR. Instead of relying on peers to hold stale routes, the standby RP maintains a **synchronized copy** of the BGP RIB. During switchover:
1. Standby RP takes over
2. BGP sessions don't drop (TCP sessions are preserved via SSO/ISSU)
3. Peers don't even know a restart happened
4. No stale routes, no re-convergence

**NSR vs GR**:

| Feature | GR | NSR |
|---------|-----|-----|
| Peer cooperation required | Yes (peer must support GR) | No |
| Session drops | Yes (briefly) | No |
| Stale routes on peer | Yes (for restart-time) | No |
| Memory overhead | Low | High (dual RIB) |
| Complexity | Lower | Higher |
| Recommended for | PE (peer-facing) | RR (critical infrastructure) |

**Best practice**: NSR on Route Reflectors (they're too important for GR's stale-route window), GR on PEs (acceptable brief disruption during RSP switchover).

**Long-Lived Graceful Restart (LLGR, RFC 9494)**: Extension of GR for routes that should survive even longer outages (minutes to hours). LLGR routes are marked with a `LLGR_STALE` community and de-preferenced but kept in the RIB. Increasingly deployed for VPN services where a brief control plane blip shouldn't trigger full re-convergence. Both IOS-XR and Junos support LLGR — configure `long-lived-graceful-restart` stale-time per address-family.

### BGP Convergence Optimization Knobs

#### Batched Updates and Best-External

**Best-External (draft-ietf-idr-best-external)**: On a PE with multiple paths to the same prefix (e.g., two CE connections), BGP normally advertises only the best path to the RR. With best-external, the PE also advertises its **best external** path. This gives the RR (and other PEs) visibility into backup paths, enabling PIC Core.

On IOS-XR, best-external is `advertise best-external` under the VPNv4 address-family. On Junos, it's `advertise-external` under the BGP group.

#### BGP Additional Paths (ADD-PATH)

ADD-PATH (RFC 7911) allows a BGP speaker to advertise **multiple paths** for the same NLRI. Combined with PIC, this enables sub-second convergence for multi-homed VPN sites.

Requirements:
- RR must support ADD-PATH send
- PEs must support ADD-PATH receive
- Both sides negotiate via capability in BGP OPEN
- Path-ID field distinguishes multiple paths for the same prefix

#### Maximum Prefixes (Safety Valve)

Every VPN session and VRF should have prefix limits:

| Scope | IOS-XR | Junos | Purpose |
|-------|--------|-------|---------|
| Per-neighbor (PE-CE) | `maximum-prefix <N> <warn%>` | `prefix-limit maximum <N>` | Prevent CE from flooding PE |
| Per-VRF | `maximum prefix <N>` (under VRF) | `routing-options maximum-prefixes <N>` | Prevent VRF from consuming all FIB |
| Per-RR client | `maximum-prefix <N>` (under neighbor) | `prefix-limit maximum <N>` | Prevent rogue PE from overwhelming RR |

Without prefix limits, a single misconfigured CE can inject a full Internet table (900K+ routes) into a VRF, consume all FIB space, and take down forwarding for every customer on that PE. This has happened. More than once.

#### Update Generation Optimization

| Knob | IOS-XR | Junos | Effect |
|------|--------|-------|--------|
| Update groups | Automatic | Automatic | Peers with identical policy share one update — massive RR optimization |
| Socket buffer | `socket send-buffer-size 1048576` | System-level TCP tuning | Larger TCP windows = faster bulk updates |
| MRAI (iBGP) | 0s (default) | 0s (default) | No artificial delay for iBGP — leave it |
| MRAI (eBGP PE-CE) | 30s (default) | 30s (default) | Reduce to 5–10s for faster CE convergence |
| NH trigger delay | `nexthop trigger-delay critical 0` | `next-hop-tracking` (automatic) | Faster PIC convergence on next-hop change |

### VRF Convergence Specifics

#### Import/Export Processing

When a VPN prefix arrives at a PE:
1. Stored in VPNv4 Adj-RIB-In
2. RT checked against all local VRF import policies
3. If RT matches, prefix installed in VRF RIB
4. Bestpath run within VRF (if multiple paths)
5. Best path pushed to VRF FIB (hardware)

Steps 2–3 are the VRF-specific overhead. With 500 VRFs and 100K incoming updates, the PE checks each update against 500 import policies. This is O(VRFs × updates) in the worst case — though optimized implementations use RT hash tables to make it O(updates).

#### Selective VRF Download (SVD)

On platforms with limited FIB space, SVD allows the PE to install only **active** VRF routes in hardware. VRFs with no traffic are kept in the software RIB only and programmed on-demand when traffic arrives. This trades first-packet latency for FIB space.

IOS-XR: Removed from modern platforms entirely (NCS-5500/5700/ASR 9000 have sufficient FIB). Legacy feature — do not attempt to configure on current hardware.
Junos: No direct equivalent. Junos VRF routes are always installed in hardware if FIB space permits. Monitor `show pfe route summary` for capacity.

### Putting It All Together: Convergence Timeline

#### Scenario: Remote PE Failure (Without PIC)

```
T=0.000s   PE2 fails (power loss)
T=0.150s   BFD detects PE2 down (3 × 50ms)
T=0.200s   RR marks PE2 BGP session down via BFD, begins processing 100K withdrawals
T=1.200s   RR finishes bestpath for all affected prefixes (~100K/sec on modern RR), begins sending updates
T=2.200s   PE1 receives updates, begins FIB programming
T=4.200s   PE1 finishes FIB programming — convergence complete
           Total: ~4 seconds (modern RR), ~14+ seconds (legacy hardware)
```

#### Scenario: Remote PE Failure (With PIC + ADD-PATH)

```
T=0.000s   PE2 fails (power loss)
T=0.150s   BFD on RR detects PE2 down; IGP adjacency loss detected
T=0.500s   IGP SPF completes, PE2 loopback withdrawn from all nodes
T=0.550s   PE1 next-hop tracking detects PE2 unreachable
T=0.600s   PIC activates: next-hop group pointer updated (ONE operation)
           All 100K prefixes now use backup PE3 path
           Total: ~600ms
```

That's a **10–50x improvement** depending on the RR hardware. PIC turns an O(n) problem into O(1).

#### Scenario: CE Failure (PIC Edge)

```
T=0.000s   CE1 link fails
T=0.001s   Fast external fallover triggers (interface down)
T=0.002s   PE1 detects CE1 path down
T=0.003s   PIC Edge: switch to backup CE2 path (pre-installed)
           Total: ~3ms
```

## Configuration

### IOS-XR: Scale and Convergence Optimization

```cisco
!! === Route Reflector Configuration ===

router bgp 65000
 !
 ! -- NSR for hitless RSP switchover --
 nsr
 !
 ! -- Larger socket buffers for bulk updates --
 socket send-buffer-size 1048576
 socket receive-buffer-size 1048576
 !
 !
 address-family vpnv4 unicast
  !
  ! -- RT-Constraint to limit VPN route distribution --
  ! (Configured on PE side — RR supports automatically)
  !
  ! -- ADD-PATH: advertise multiple paths to PEs for PIC --
  additional-paths receive
  additional-paths send
  additional-paths selection route-policy ADD-PATH-SELECT
 !
 address-family vpnv6 unicast
  additional-paths receive
  additional-paths send
  additional-paths selection route-policy ADD-PATH-SELECT
 !
 neighbor-group RR-CLIENTS
  address-family vpnv4 unicast
   route-reflector-client
   maximum-prefix 500000 90
  !
  address-family vpnv6 unicast
   route-reflector-client
   maximum-prefix 500000 90
 !
 neighbor 10.0.0.1
  use neighbor-group RR-CLIENTS
  description PE1-WEST
  bfd fast-detect
  bfd minimum-interval 300
  bfd multiplier 3
 !
 neighbor 10.0.0.2
  use neighbor-group RR-CLIENTS
  description PE2-EAST
  bfd fast-detect
  bfd minimum-interval 300
  bfd multiplier 3
!
! -- ADD-PATH selection: advertise best + backup --
route-policy ADD-PATH-SELECT
  set additional-paths-selection best 2
end-policy
!

!! === PE Configuration ===

router bgp 65000
 !
 ! -- Graceful Restart (PE side) --
 bgp graceful-restart
 bgp graceful-restart restart-time 120
 bgp graceful-restart stalepath-time 360
 !
 !
 ! -- RT-Constraint: only receive routes for locally-configured RTs --
 address-family ipv4 rt-filter
 !
 address-family vpnv4 unicast
  ! -- ADD-PATH receive from RR --
  additional-paths receive
  !
  ! -- Best-external: advertise backup CE paths to RR for PIC Core --
  advertise best-external
 !
 ! NOTE: BGP PIC is enabled by default on most IOS-XR 7.x+ platforms for VPN prefixes.
 ! Verify with `show cef vrf <name> <prefix> detail` — look for backup path.
 ! For pre-7.x or platforms where PIC is not default, see explicit PIC config block below.
 !
 vrf CUSTOMER-A
  rd 1.1.1.1:0
  address-family ipv4 unicast
   !
   ! -- Per-VRF label allocation for scale --
   label mode per-vrf
   !
   ! -- Prefix limits: prevent CE flooding --
   maximum-prefix 10000 80
  !
  neighbor 192.168.1.1
   remote-as 64501
   address-family ipv4 unicast
    route-policy CUST-A-IN in
    route-policy CUST-A-OUT out
    maximum-prefix 5000 90
    !
    ! -- Fast detection for CE failure --
    bfd fast-detect
    bfd minimum-interval 100
    bfd multiplier 3
!
! -- Fast external fallover (default on, verify it's not disabled) --
! bgp fast-external-fallover   ← default enabled
```

### Junos: Scale and Convergence Optimization

```junos
## === Route Reflector Configuration ===

protocols {
    bgp {
        group RR-CLIENTS {
            type internal;
            cluster 10.0.0.100;
            family inet-vpn {
                unicast {
                    ## ADD-PATH: advertise multiple paths for PIC
                    add-path {
                        receive;
                        send {
                            path-count 2;
                        }
                    }
                    ## Prefix limits per client
                    prefix-limit {
                        maximum 500000;
                        teardown 90;
                    }
                }
            }
            family inet6-vpn {
                unicast {
                    add-path {
                        receive;
                        send {
                            path-count 2;
                        }
                    }
                }
            }
            ## BFD on RR-client sessions
            bfd-liveness-detection {
```

### What Good Looks Like

1. **FIB utilization < 80%** — anything higher means you're one routing event from overflow
2. **BGP convergence < 1s** for PE failures (with PIC) — check `show bgp convergence`
3. **Update groups = number of distinct policies**, not number of peers (efficient RR)
4. **All BFD sessions UP** with consistent timers
5. **NSR synchronized** on RR: `show bgp nsr` shows "TCP Initial Sync Complete"
6. **Label allocation mode matches design**: `show bgp vpnv4 unicast labels` shows per-vrf if that's the plan
7. **No FIB programming drops**: `show cef resource` shows 0 drops/failures

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| VPN convergence > 10s after PE failure | PIC not enabled or ADD-PATH not configured | Enable PIC + ADD-PATH (path-count 2) |
| FIB overflow (routes in software) | Too many per-prefix labels or no prefix limits | Switch to per-VRF labels, add max-prefix |
| RR CPU spikes during convergence | No RT-Constraint — RR sending all VPN routes to all PEs | Enable `address-family ipv4 rt-filter` on PEs |
| BGP sessions flap during RSP switchover | GR/NSR not configured | Enable NSR on RR, GR on PEs |
| Stale VPN routes after PE restart | GR stale-path-time too short or End-of-RIB delayed | Increase stale-path-time, check session re-establishment |
| Slow PE-CE failover (> 5s) | No BFD on PE-CE, fast-external-fallover disabled | Enable BFD (100ms × 3), verify fast-external-fallover |
| One PE gets routes for all VRFs | RT-Constraint not negotiated — check `show bgp neighbor <ip> capabilities` | Ensure both sides support RT-Constraint capability |
| Update groups not optimal | Different policies per peer preventing grouping | Standardize route-policy across same-role peers |
| Black hole during convergence | Old FIB entry points to dead next-hop, PIC not covering | Verify PIC pre-computation with `show cef vrf <name> <prefix> detail` — should show backup |
| Asymmetric VPN traffic after failover | IGP metric not matching, MED not set | Check bestpath: IGP metric to next-hop is the tiebreaker |

### Methodology

1. **Establish baseline**: Before any changes, record `show bgp scale`, `show cef resource`, `show bgp convergence` output
2. **Measure current convergence**: Use `show bgp convergence` (IOS-XR) or monitor syslog during a controlled failover
3. **Check PIC state**: `show cef vrf <vrf> <prefix> detail` — look for "backup path" or "repair" entry
4. **Verify ADD-PATH**: `show bgp vpnv4 unicast <prefix>` — should show multiple paths with path-IDs
5. **Check label mode**: `show bgp vpnv4 unicast labels` — verify per-vrf if scale is the goal
6. **Monitor FIB during event**: `show cef resource` before, during, and after — look for FIB programming rate and drops
7. **Check BFD**: `show bfd session detail` — verify negotiated intervals match design
8. **RR health**: `show bgp process performance-statistics` — check update generation rate, queue depth
9. **Validate GR/NSR**: Trigger a controlled RSP switchover, verify BGP sessions don't drop (NSR) or recover within restart-time (GR)
10. **Test under load**: Generate VPN traffic across multiple VRFs, then fail a PE — measure actual convergence with hardware timestamps

### War Stories

> **The 45-Second VPN Black Hole**
>
> Large metro SP, 300 VRFs, 180K VPN prefixes across 24 PEs. Two dedicated RRs, no PIC, no ADD-PATH. Business as usual until a PE in the downtown POP crashes — power supply failure, instant death.
>
> Here's what happened in slow motion:
>
> T=0s: PE-DOWNTOWN dies. 12K VPN prefixes go dark.
> T=0.9s: BFD on the RR detects the failure. Good — sub-second detection. RR begins processing.
> T=1-11s: RR processes 12K withdrawals, runs bestpath for each prefix. Most prefixes have a backup PE (multi-homed sites), but the RR has to discover this by running bestpath. Their RR was an in-path P router running BGP — not a dedicated RR — processing at roughly 10K bestpaths/second under load. That's ~1.2 seconds of computation, but the update generation and serialization to 23 clients without RT-Constraint turned it into 10 seconds of sustained CPU burn.
> T=11-15s: RR begins sending updates to the 23 remaining PEs. Without RT-Constraint (they hadn't deployed it), every PE gets all 12K updates even if only 500 are relevant.
> T=15-25s: PEs receive updates, run their own import processing, program FIBs.
> T=25-45s: Stragglers — some PEs have slower FIB programming, some updates are delayed by TCP retransmits (small socket buffers).
>
> 45 seconds of customer-impacting VPN black hole. For a power supply.
>
> The fix was four changes:
> 1. **ADD-PATH (path-count 2)** on the RR — PEs now pre-install backup paths
> 2. **PIC Core** enabled on all PEs — next-hop group pointer update instead of per-prefix
> 3. **RT-Constraint** on all PEs — RR sends only relevant routes (reduced RR CPU by 80%)
> 4. **Socket buffer tuning** — 1MB send buffers on RR
>
> After: same failure scenario converged in **~600ms**. BFD detection (150ms) + IGP convergence (~350ms with tuned SPF timers) + PIC pointer update (~100ms). The RR still processes the withdrawals and sends updates, but traffic has already moved. The updates just confirm what PIC already did.
>
> Lesson: PIC doesn't eliminate BGP convergence — it makes it **irrelevant to the data plane**. BGP still does its job, just after traffic has already reconverged.
