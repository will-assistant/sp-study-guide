# 4.1 — LDP & Label Distribution

## Overview
> Label Distribution Protocol (LDP) is the workhorse signaling protocol that builds Label Switched Paths (LSPs) across an MPLS domain. While RSVP-TE gets the glamour (and the traffic engineering), LDP quietly does the heavy lifting in the vast majority of SP cores — providing best-effort label paths that follow the IGP shortest path. Understanding LDP's mechanics — discovery, session establishment, label allocation, retention modes, and synchronization with the IGP — is fundamental to operating any MPLS network. Get LDP wrong and your VPNs black-hole traffic in ways that make you question your career choices.


## Key Concepts
> **After this section you will understand:**
> - how labels are distributed and installed in the MPLS forwarding plane
> - how push, swap, and pop operations are applied hop by hop
> - where control-plane and data-plane mismatches create black holes
> - which verification commands confirm end-to-end LSP correctness

## Theory

### Why LDP Exists

MPLS needs a way to distribute labels between routers. A label is just a 20-bit integer (0–1,048,575) that tells a router "swap this label and forward." But routers need to agree on which label maps to which FEC (Forwarding Equivalence Class). LDP automates this agreement.

There are three ways to distribute labels:
1. **LDP** (RFC 5036) — follows IGP best path, no traffic engineering
2. **RSVP-TE** (RFC 3209) — explicit path control, bandwidth reservation (Section 4.2)
3. **Segment Routing** — eliminates signaling entirely (Module 6)

LDP is the simplest. It creates LSPs that mirror the IGP topology — if IS-IS says "reach 10.0.0.1/32 via next-hop 10.1.1.2," LDP ensures there's a label binding for that prefix at every hop along the way.

### LDP Discovery

LDP uses a two-phase approach: discovery, then session.

#### Basic Discovery (Link-Level)
- Routers send **LDP Hello** messages to the all-routers multicast address **224.0.0.2** on UDP port **646**
- Hello interval: **5 seconds** (default on both IOS-XR and Junos)
- Hold time: **15 seconds** (3× hello interval)
- Hellos contain the **LDP Identifier** (Router-ID:label-space)
- Hellos are sent on every MPLS-enabled interface

**LDP Identifier format**: `<router-id>:<label-space>`
- Label space `0` = per-platform label space (most common)
- Non-zero = per-interface label space (rare, used in ATM/Frame Relay legacy)

#### Extended Discovery (Targeted LDP)
- Unicast UDP Hellos sent to a specific remote peer
- Used for: targeted LDP sessions (e.g., L2VPN pseudowires, MPLS over GRE)
- No multicast — the two endpoints must be explicitly configured
- Hello interval: **10 seconds** (default)
- Hold time: **90 seconds** (default on Junos), **180 seconds** (IOS-XR default)

### Session Establishment

Once two routers discover each other via Hellos, they establish a TCP session:

1. **Transport connection**: TCP session on port **646** (both ends use 646 — the router with the higher LDP Identifier becomes the passive/listening side)
2. **Initialization**: Exchange of Initialization messages containing:
   - Session parameters (keepalive time, label distribution method, loop detection)
   - Both sides must agree on parameters or the session fails
3. **Keepalive**: Default **60 seconds** on IOS-XR, **30 seconds** on Junos
4. **Label advertisement**: Once the session is up, both sides flood their label bindings

**Critical detail on TCP role determination**: The router with the *higher* transport address (the address used for the TCP session, usually the loopback/router-ID) is the **active** side (initiates TCP). The lower transport address is the **passive** side. This matters when debugging — if both sides think they should be passive, no session forms.

```
Router-A (10.0.0.1) ──── LDP Hello (UDP 646, mcast 224.0.0.2) ────→
                    ←──── LDP Hello (UDP 646, mcast 224.0.0.2) ────
                    
Router-A (higher ID) ──── TCP SYN to 10.0.0.2:646 ────→
                    ←──── TCP SYN-ACK ────
                    ──── Initialization Message ────→
                    ←──── Initialization Message ────
                    ──── Keepalive ────→
                    ←──── Keepalive ────
                    ←──── Label Mappings ────→ (bidirectional flood)
```

### Label Allocation and Distribution

#### Downstream Unsolicited (DU)
LDP uses **Downstream Unsolicited** mode by default — a router assigns a local label to every IGP prefix in its RIB and advertises that binding to all LDP peers without being asked. This is the only mode you'll see in modern networks.

(Downstream on Demand existed for ATM networks — if you encounter it in production in 2026, something has gone terribly wrong.)

#### Label Retention Modes

This is a classic exam topic and a real operational consideration:

| Mode | Behavior | Memory Usage | Convergence | Default |
|------|----------|-------------|-------------|---------|
| **Liberal** | Keep ALL received label bindings, even from non-best-path peers | Higher | Faster (backup labels pre-installed) | **IOS-XR default** |
| **Conservative** | Keep ONLY label bindings from the best-path next-hop peer | Lower | Slower (must request new label on failover) | **Junos default** |

**Liberal retention** means if you have three LDP neighbors and all advertise a label for 10.0.0.5/32, you store all three bindings. When the best path changes, you instantly have the new label — no signaling delay.

**Conservative retention** means you only store the binding from the current best-path neighbor. Less memory, but on IGP convergence you have to wait for a new label binding from the new next-hop. In practice, LDP convergence is fast enough that this rarely matters — but at scale (1M+ labels), memory savings from conservative mode are real.

**Junos behavior note**: Junos uses conservative retention by default, but in practice this is rarely a problem because Junos LDP convergence is very fast. You can change it with `set protocols ldp track-igp-metric` and related knobs, but most operators leave the default.

#### Label Control Modes

| Mode | Behavior | Default |
|------|----------|---------|
| **Independent** | Each router assigns labels independently, without waiting for a downstream binding | **IOS-XR and Junos** |
| **Ordered** | A router only assigns/advertises a label after receiving a binding from downstream (toward the FEC) | Optional on both |

**Independent control** means every router immediately allocates and advertises labels for all its prefixes. This is fast but can temporarily create incomplete LSPs during convergence (the upstream has a label but the downstream hasn't advertised one yet).

**Ordered control** means labels propagate from the egress back toward the ingress in order. The LSP is guaranteed complete before the ingress starts using it — but setup is slower.

In practice, independent control with liberal retention is the most common combination on IOS-XR. Junos uses independent control with conservative retention.

### Loop Detection

LDP has two loop-detection mechanisms, both disabled by default:

1. **Path Vector** — similar to BGP AS_PATH; the LDP Identifier is added to a path vector TLV. If a router sees its own ID, it's a loop.
2. **Hop Count** — incremented at each hop; if it exceeds a configured maximum, the label mapping is rejected.

**Why are they disabled by default?** Because MPLS forwarding follows the IGP path, and the IGP (IS-IS/OSPF) already handles loop prevention via SPF. Adding LDP loop detection adds overhead with no real benefit in a properly configured network.

**When to enable**: Only in rare topologies with route redistribution between LDP and other label distribution methods, or during migration scenarios.

### Targeted LDP

Regular LDP discovers neighbors on directly connected links. Targeted LDP establishes sessions between non-adjacent routers:

**Use cases**:
- **AToM/L2VPN pseudowires**: PE-to-PE LDP sessions for pseudowire signaling (before EVPN took over)
- **LDP over RSVP-TE tunnels**: Using RSVP-TE for TE in the core but LDP for VPN label distribution
- **MPLS over GRE**: Extending MPLS across non-MPLS domains
- **LDP session protection**: Maintaining sessions during link failures via backup paths

Targeted LDP sessions survive link failures as long as an IP path exists between the two endpoints. This is critical for pseudowire stability.

### LDP-IGP Synchronization

This is one of the most important operational features for MPLS networks, and a common source of outages when not configured.

**The problem**: When a link comes up, IGP converges first (installs the route), but LDP takes longer to establish a session and exchange labels. During this gap, traffic is forwarded to the link based on IGP, but there's no label — so labeled traffic (VPN, etc.) is black-holed.

**The solution**: LDP-IGP sync tells the IGP to advertise the link with maximum metric until LDP is fully converged on that interface. Traffic avoids the link until labels are ready.

```
Timeline without LDP-IGP sync:
  t=0s    Link comes up
  t=0.5s  IGP adjacency forms, installs route with normal metric
  t=3s    LDP session forms, but no labels yet
  t=5s    LDP labels exchanged — MPLS works
  t=0.5s–5s: BLACK HOLE for labeled traffic

Timeline with LDP-IGP sync:
  t=0s    Link comes up
  t=0.5s  IGP adjacency forms, but advertises MAX METRIC (IS-IS: 2^24-1, OSPF: 65535)
  t=5s    LDP session forms, labels exchanged
  t=5.1s  IGP re-advertises link with normal metric
  t=0s–5.1s: No black hole — traffic uses other paths
```

### Scaling Considerations

| Parameter | Cisco IOS-XR | Junos |
|-----------|-------------|-------|
| Max LDP sessions | Platform-dependent; ASR 1000: ~500 sessions | MX: ~1000 sessions |
| Max label bindings (LFIB) | ASR 1000: ~1M labels; ASR 9000: ~2M+ | MX204: ~512K; MX960: ~2M+ |
| Default label range | 16–1048575 (dynamic: 16–max) | 100000–1048575 |
| LDP Graceful Restart | Supported (RFC 3478) | Supported (RFC 3478) |
| Session protection | Supported | Supported |
| Entropy label (RFC 6790) | Supported on ASR 9000 | Supported on MX |

**Label allocation strategy**: By default, IOS-XR allocates a local label for every prefix in the RIB. In a network carrying a full Internet table (1M+ prefixes), this wastes labels. Best practice: allocate labels only for loopback/host routes (/32s and /128s) since those are the only FECs that matter for LSP establishment.

## Configuration

### IOS-XR

```cisco
!! === LDP Basic Configuration (IOS-XR: ASR 9000, NCS 5500) ===
!!
mpls ldp
 router-id 10.0.0.1
 !
 !! Only allocate labels for host routes (CRITICAL for scaling)
 address-family ipv4
  label
   local
    allocate for host-routes
   !
  !
 !
 !! LDP-IGP Synchronization
 igp sync delay on-session-up 10
 !
 !! Session Protection — maintains targeted session as backup on link failure
 session protection duration infinite
 !
 !! Graceful Restart
 graceful-restart
 !
 !! Targeted LDP (for pseudowires)
 address-family ipv4
  discovery targeted-hello accept
 !
 !! LDP Authentication (MD5 — use for all sessions in production)
 neighbor 10.0.0.2
  password encrypted MPLS-LDP-KEY-2026
 !
 !! Tuning Timers
 discovery hello interval 5
 discovery hello holdtime 15
 holdtime 180
 !
 !! Enable LDP on core-facing interfaces
 interface GigabitEthernet0/0/0/0
 !
 interface GigabitEthernet0/0/0/1
 !
!
```

> **Also on IOS-XE:** IOS-XE uses interface-level `mpls ip` to enable LDP per-interface, and global commands like `mpls ldp router-id Loopback0 force`, `mpls ldp label allocate global host-routes`. The IOS-XR model groups everything under `mpls ldp` router configuration, which is cleaner for large-scale deployments.

### Junos

```junos
# === LDP Basic Configuration ===
set protocols ldp interface ge-0/0/0.0
set protocols ldp interface ge-0/0/1.0
set protocols ldp interface lo0.0

# Transport address (use loopback — MUST match IGP reachability)
set protocols ldp transport-address router-id

# Deaggregate loopback — needed for LDP FEC resolution
set protocols ldp deaggregate

# === LDP-IGP Synchronization ===
# Junos: applied under the IGP protocol, not LDP
set protocols isis interface ge-0/0/0.0 ldp-synchronization hold-time 30
set protocols isis interface ge-0/0/1.0 ldp-synchronization hold-time 30
# OR globally:
set protocols isis ldp-synchronization hold-time 30

# OSPF variant:
set protocols ospf area 0.0.0.0 interface ge-0/0/0.0 ldp-synchronization hold-time 30

# === LDP Session Protection ===
set protocols ldp session-protection timeout 0

# === LDP Graceful Restart ===
set routing-options graceful-restart

# === Targeted LDP (for pseudowires or LDP-over-RSVP) ===
set protocols ldp targeted-hello 10.0.0.5

# === LDP Authentication ===
set protocols ldp session 10.0.0.2 authentication-key "MPLS-LDP-KEY-2026"

# === Label Retention Policy (change from conservative to liberal) ===
# Most operators leave conservative (default) — only change if you need faster convergence
# set protocols ldp label-retention-mode liberal

# === Track IGP Metric ===
set protocols ldp track-igp-metric

# === Filter Label Advertisement (only for loopback /32s) ===
set policy-options policy-statement LDP-LOOPBACKS-ONLY term LOOPBACKS from route-filter 10.0.0.0/24 prefix-length-range /32-/32
set policy-options policy-statement LDP-LOOPBACKS-ONLY term LOOPBACKS then accept
set policy-options policy-statement LDP-LOOPBACKS-ONLY term DEFAULT then reject
set protocols ldp egress-policy LDP-LOOPBACKS-ONLY
```

### Key Knobs

| Parameter | IOS-XE Default | Junos Default | Recommended | Why |
|-----------|---------------|---------------|-------------|-----|
| Hello interval | 5s | 5s | 5s | Default is fine; BFD handles fast failure detection |
| Hello holdtime | 15s | 15s | 15s | 3× hello |
| Keepalive | 60s | 30s | Default | Don't over-tune |
| Label allocation | All prefixes | All prefixes | Host routes only | Saves LFIB space at scale |
| LDP-IGP sync | Disabled | Disabled | **Enabled** | Prevents black holes on link-up |
| Session protection | Disabled | Disabled | **Enabled** | Survives link flaps |
| Graceful restart | Disabled | Disabled | **Enabled** | Survives RP/RE switchover |
| Label retention | Liberal (IOS-XR) | Conservative (Junos) | Platform default | Each vendor optimized for their default |

### What Good Looks Like

**IOS-XE — Healthy LDP neighbor**:
```
Router# show mpls ldp neighbor 10.0.0.2
    Peer LDP Ident: 10.0.0.2:0; Local LDP Ident 10.0.0.1:0
        TCP connection: 10.0.0.2.646 - 10.0.0.1.44238
        State: Oper; Msgs sent/rcvd: 4521/4518; Downstream
        Up time: 2w3d
        LDP discovery sources:
          GigabitEthernet0/0/0, Src IP addr: 10.1.12.2
        Addresses bound to peer LDP Ident:
          10.0.0.2        10.1.12.2       10.1.23.1
```

Key things to verify:
- **State: Oper** — session is operational
- **Up time** — long uptime = stable session
- **Discovery sources** — shows which interface(s) the neighbor was discovered on
- **Addresses bound** — the peer's IP addresses (used for next-hop resolution)

**Junos — Healthy LDP session**:
```
user@router> show ldp session
  Address           State       Connection  Hold time  Adv. Mode
  10.0.0.2          Operational Open          26         DU
```

**Verify LDP-IGP sync is active**:
```
Router# show mpls ldp igp sync interface GigabitEthernet0/0/0
    GigabitEthernet0/0/0:
        LDP configured; LDP-IGP Synchronization enabled.
        Sync status: sync achieved; peer reachable.
```

If sync status shows "not achieved" — LDP is not converged on that interface, and the IGP is advertising max metric. Check the LDP session on that interface.

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| No LDP neighbor discovered | MPLS not enabled on interface (`mpls ip` / interface not under `protocols ldp`) | Enable MPLS on the interface |
| LDP Hello seen but no session | Transport address unreachable (loopback not in IGP) | Ensure loopback is advertised in IGP; verify `mpls ldp router-id` |
| Session flapping | MTU mismatch on link (LDP uses TCP, fragmentation issues) | Match MTU on both sides; use `mpls mtu` if needed |
| Session up but no labels for a prefix | Label allocation filter too restrictive; prefix not in RIB | Check `mpls ldp label allocate` / `egress-policy` |
| Labels exist but traffic black-holed | PHP delivering unlabeled packet to wrong interface; recursive lookup failure | Check `show mpls forwarding-table`; verify next-hop resolution |
| VPN traffic black-holed after link recovery | LDP-IGP sync not configured; IGP converged before LDP | Enable `mpls ldp sync` under IGP |
| Targeted LDP session won't form | No IP reachability between endpoints; wrong transport address | Verify IGP reachability for loopbacks; check `targeted-hello` config |
| LDP session up but wrong transport address | Router-ID changed or multiple loopbacks | Use `mpls ldp router-id Loopback0 force` / verify `transport-address router-id` |
| Liberal retention consuming too much memory | Full Internet table with per-prefix label allocation | Switch to host-route-only label allocation |

### Methodology

1. **Check discovery**: `show mpls ldp discovery` / `show ldp interface` — are Hellos being sent and received?
2. **Check transport**: Can the router reach the peer's transport address? `ping <peer-loopback>` — if this fails, no TCP session will form
3. **Check session**: `show mpls ldp neighbor` / `show ldp session` — is the TCP session established?
4. **Check bindings**: `show mpls ldp bindings <prefix>` / `show ldp database <prefix>` — does a label binding exist for the destination?
5. **Check forwarding**: `show mpls forwarding-table <prefix>` / `show route table mpls.0 label <X>` — is the LFIB programmed correctly?
6. **Check data plane**: `traceroute mpls ipv4 <prefix>` — does the LSP actually work end-to-end?

### War Story: The Case of the Asymmetric Transport Address

A Tier-2 ISP had an MPLS core with 40 P routers running IS-IS and LDP. During a routine maintenance window, a junior engineer added a second loopback (Loopback1) on a P router for management purposes. They didn't think twice about it.

The next morning, LDP sessions on that router started cycling every 30–60 minutes. VPN traffic was intermittently black-holed. The NOC spent 6 hours chasing "flapping LDP" before someone noticed the new loopback.

What happened: The new Loopback1 had a higher IP address than Loopback0. On IOS-XE, `mpls ldp router-id` selects the highest loopback by default (without the `force` keyword). After a process restart during the maintenance, LDP picked Loopback1 as its transport address. But Loopback1 wasn't advertised in IS-IS. So:

- LDP Hellos went out with the new transport address
- Peers tried to establish TCP to the new address
- TCP failed because the address wasn't routable via IGP
- LDP fell back and sometimes connected on the old address
- Flap city

**Fix**: `mpls ldp router-id Loopback0 force` — the `force` keyword means "use this loopback, always, even if higher addresses exist." This should be in every MPLS router config. In Junos, `transport-address router-id` under `protocols ldp` achieves the same thing.

**Lesson**: Always use the `force` keyword. Always. And always verify that your LDP transport address matches what the IGP advertises as the router's loopback.
