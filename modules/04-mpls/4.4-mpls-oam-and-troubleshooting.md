# 4.4 — MPLS OAM & Troubleshooting

## Overview
> MPLS is a beautiful forwarding paradigm right up until something breaks and you realize you can't just `ping` and `traceroute` your way out of it. IP-based tools don't test the MPLS data plane — a ping to a loopback might succeed via an IP route while the MPLS LSP is completely broken. MPLS OAM (Operations, Administration, and Maintenance) tools — LSP Ping, LSP Traceroute, BFD for MPLS, and MPLS OAM per RFC 8029 — are purpose-built to verify the MPLS forwarding plane. Master these tools and you'll find MPLS problems in minutes. Ignore them and you'll spend days wondering why VPN traffic is silently vanishing.


## Key Concepts
> **After this section you will understand:**
> - how labels are distributed and installed in the MPLS forwarding plane
> - how push, swap, and pop operations are applied hop by hop
> - where control-plane and data-plane mismatches create black holes
> - which verification commands confirm end-to-end LSP correctness

## Theory

### Why Standard Ping/Traceroute Fail for MPLS

Consider this scenario: PE1 wants to verify the LSP to PE5 through the MPLS core.

**`ping 10.0.0.5` from PE1**: This IP ping might:
1. Be forwarded as a labeled packet if an LSP exists (great, tests the data plane!)
2. Be forwarded as an IP packet if no LSP exists but an IP route exists (misleading — says "reachable" but MPLS is broken)
3. Succeed via a different path than the LSP under test

**The fundamental problem**: IP tools test IP reachability. MPLS OAM tools test **label forwarding** — they send labeled packets and verify that each hop performs the correct label operation.

### LSP Ping (RFC 8029, originally RFC 4379)

LSP Ping is the MPLS equivalent of IP ping. It sends a **UDP packet encapsulated in MPLS labels** to verify end-to-end LSP connectivity.

#### How It Works

1. **Ingress (sender)** constructs a special UDP packet:
   - Destination: **127.0.0.0/8** (a loopback address — guaranteed to never be routed externally)
   - UDP port: **3503** (MPLS echo)
   - Payload: MPLS Echo Request containing:
     - FEC Stack TLV — identifies the LSP being tested (e.g., "LDP FEC for 10.0.0.5/32")
     - Timestamp, sequence number, reply mode
2. **Ingress pushes the same labels** the LSP under test uses
3. Packet is forwarded through the MPLS domain **as a regular labeled packet**
4. **Egress (responder)** receives the packet:
   - Pops the label(s)
   - Sees destination 127.0.0.x → sends to local control plane
   - Validates FEC Stack TLV against its local label database
   - Sends MPLS Echo Reply (UDP, same port) back to the ingress
5. Ingress receives reply → LSP is working

**Key insight**: Because the Echo Request travels as a labeled packet, it tests the **actual data plane path**. If any transit router has a corrupt LFIB entry, the packet will be misdelivered or dropped — exactly what you want to detect.

#### Reply Modes

| Mode | Value | Description | Use Case |
|------|-------|-------------|----------|
| Do not reply | 1 | No response (one-way test) | Loss measurement |
| Reply via IPv4/IPv6 UDP | 2 | Reply as IP packet (not via LSP) | Default, most common |
| Reply via control channel | 3 | Reply via application-level control channel | When IP return path is unavailable |
| Reply via specified path | 4 | Reply via specified path (Reverse LSP) | Tests bidirectional LSP |

#### Return Codes

| Code | Meaning | Action |
|------|---------|--------|
| 0 | No return code | Problem — no response at all |
| 1 | Malformed echo request | Check sender implementation |
| 2 | Unknown TLV | Version mismatch between sender and responder |
| 3 | **Replying router is egress** | **Success** — the LSP works |
| 4 | Replying router has no FEC mapping | FEC mismatch — label exists but maps to wrong prefix |
| 5 | Downstream mapping mismatch | Label database inconsistency |
| 8 | Label switched at stack depth | Transit — used in traceroute mode |
| 10 | Mapping for this FEC not associated with receiving interface | Packet arrived on wrong interface (routing loop or label error) |
| 15 | Label switched but FEC not found | Transit router forwarded but can't validate |

### LSP Traceroute

LSP Traceroute works like IP traceroute but for MPLS:

1. Send MPLS Echo Request with MPLS TTL = 1
2. First hop expires the TTL → sends MPLS Echo Reply with return code 8 ("Label switched at stack depth") and includes the downstream label mapping
3. Increment TTL and repeat
4. When the egress is reached → return code 3 ("Replying router is egress")

The reply includes:
- The **downstream mapping TLV** — shows what label the replying router would push and which interface it would use
- This allows the sender to verify the **entire label path** hop-by-hop

```
Router# traceroute mpls ipv4 10.0.0.5/32
  0 10.0.0.1 MRU 9216 [Labels: 24001 Exp: 0]
  1 10.1.12.2 MRU 9216 [Labels: 24015 Exp: 0] 3 ms
  2 10.1.23.2 MRU 9216 [Labels: implicit-null Exp: 0] 5 ms
  3 10.0.0.5 7 ms, ret code 3
```

This output tells you:
- Hop 1 (10.1.12.2): Swaps label 24001 → 24015, MRU 9216 ✓
- Hop 2 (10.1.23.2): Swaps label 24015 → Implicit NULL (PHP), MRU 9216 ✓
- Hop 3 (10.0.0.5): Egress, return code 3 (success) ✓

### BFD for MPLS (RFC 5884)

BFD provides sub-second failure detection for MPLS LSPs. While RSVP Hello and LDP keepalives detect failures in seconds to minutes, BFD can detect in **milliseconds** (typically 50ms × 3 = 150ms detection).

#### BFD for LDP LSPs

BFD sessions are established between head-end and tail-end of each LDP LSP you want to monitor:

1. BFD control packets are sent as **labeled packets** through the LSP
2. BFD session monitors the data plane health of the specific LSP
3. On failure → triggers fast IGP convergence or LDP failover

#### BFD for RSVP-TE LSPs

BFD runs over the TE tunnel:
1. BFD packets follow the exact path of the TE tunnel
2. On failure → triggers FRR switchover or tunnel re-signal
3. Provides faster detection than RSVP Hello (which takes 9+ seconds by default)

#### BFD Timers for MPLS

| Parameter | Recommended | Notes |
|-----------|-------------|-------|
| Min TX interval | 100ms | Don't go below 50ms on software-based platforms |
| Min RX interval | 100ms | Must be supported by both endpoints |
| Multiplier | 3 | Detection time = 300ms |
| Echo mode | Disabled for MPLS BFD | Echo mode doesn't work well over MPLS |

### MPLS OAM Framework (RFC 8029)

RFC 8029 consolidates and updates the MPLS OAM mechanisms from RFC 4379. Key additions:

- **Nil FEC**: Test a label path without knowing the FEC (useful for generic label verification)
- **FEC Stacking**: Test multi-label stacks (VPN over LDP, L2VPN, etc.)
- **P2MP LSP Ping**: Test point-to-multipoint LSPs (multicast MPLS)
- **Downstream Detailed Mapping TLV**: More granular path information including:
  - Interface index
  - Address type (IPv4, IPv6, unnumbered)
  - Label stack at each hop
  - Multipath information (shows ECMP paths)

### Proactive OAM: Performance Monitoring

Beyond fault detection, MPLS OAM supports performance measurement:

- **Delay measurement**: RFC 6374 — measures one-way and two-way delay through an LSP
- **Loss measurement**: RFC 6374 — measures packet loss on an LSP
- **Throughput measurement**: Active testing with calibrated traffic

These are increasingly important for SLA monitoring in SP networks, especially for L3VPN/EVPN services with strict SLA guarantees.

### What Good Looks Like

**IOS-XE — Successful LSP Ping**:
```
Router# ping mpls ipv4 10.0.0.5/32
Sending 5, 100-byte MPLS Echos to 10.0.0.5/32,
     timeout is 2 seconds, send interval is 0 msec:

Codes: '!' - success, 'Q' - request not sent, '.' - timeout,
  'L' - labeled output interface, 'B' - unlabeled output interface,
  'D' - DS Map mismatch, 'F' - no FEC mapping, 'f' - FEC mismatch,
  'M' - malformed request, 'm' - unsupported tlvs, 'N' - no label entry,
  'P' - no rx intf label prot, 'p' - premature termination of LSP,
  'R' - transit router, 'I' - unknown upstream index,
  'X' - unknown return code, 'x' - return code 0

Type escape sequence to abort.
!!!!!
Success rate is 100 percent (5/5), round-trip min/avg/max = 2/3/5 ms
```

**IOS-XE — Successful LSP Traceroute**:
```
Router# traceroute mpls ipv4 10.0.0.5/32
Tracing MPLS Label Switched Path to 10.0.0.5/32, timeout is 2 seconds

  0 10.0.0.1 MRU 9216 [Labels: 24001 Exp: 0]
L 1 10.1.12.2 MRU 9216 [Labels: 24015 Exp: 0] 3 ms
L 2 10.1.23.2 MRU 9216 [Labels: implicit-null Exp: 0] 5 ms
! 3 10.0.0.5 7 ms
```

Key: `L` = labeled output, `!` = egress reached. Every hop should show `L` except the last hop which shows `!`.

**Junos — Successful LSP Ping**:
```
user@router> ping mpls ldp 10.0.0.5/32
!!!!!
--- lsping statistics ---
5 packets transmitted, 5 packets received, 0% packet loss
round-trip min/avg/max/stddev = 1.924/2.437/3.112/0.389 ms
```

**Failure indicators to watch for**:
- `.` (timeout) — packet never reached egress or reply never came back
- `N` (no label entry) — broken LFIB at some transit hop
- `B` (unlabeled output) — transit router forwarding as IP instead of MPLS
- `D` (DS Map mismatch) — label database inconsistency between hops

## Troubleshooting

### Common MPLS Failure Scenarios

| Scenario | Symptoms | Detection | Root Cause | Fix |
|----------|----------|-----------|------------|-----|
| **LDP session down** | VPN traffic black-holed; LSP ping fails | `show mpls ldp neighbor` shows no session | Transport address unreachable; ACL blocking TCP/646 | Fix IGP route to peer loopback; check ACLs |
| **Partial LSP (label gap)** | Traffic works to some destinations, not others | LSP traceroute shows `N` at specific hop | One router missing label for a FEC | Check LDP bindings on that router; verify label allocation policy |
| **RSVP-TE tunnel down** | TE traffic reroutes to LDP or drops | `show mpls traffic-eng tunnels` shows Oper: down | CSPF failure; bandwidth exhaustion; explicit path invalid | Check TED; verify bandwidth availability; update explicit path |
| **PHP causing QoS issues** | QoS policy not applied correctly at egress | Customer SLA violations on specific traffic classes | TC bits lost during PHP | Enable Explicit NULL on egress PE |
| **MTU black hole** | Large packets dropped; small packets OK; TCP retransmits | LSP ping with large size fails; `ping mpls ipv4 X size 9000` | Core MTU too small for label stack | Set `mpls mtu 9216` on all core interfaces |
| **ECMP imbalance** | Congestion on some core links, others underutilized | Interface counter disparity on parallel links | Hash only on top label | Enable deep inspection / entropy labels |
| **IGP-LDP mismatch** | Brief traffic loss after link recovery | Monitoring shows drops for ~5-10s after interface up | LDP-IGP sync not configured | Enable `mpls ldp sync` under IGP |
| **Label corruption** | Intermittent packet loss; packets arriving at wrong destination | Hard to detect — may need packet capture in core | Hardware fault; software bug | Isolate affected router; check hardware diagnostics |
| **FRR not engaging** | Slow failover (seconds instead of sub-50ms) | Monitor convergence time during failure test | No bypass tunnel; FRR not enabled on LSP | Configure bypass tunnels on PLR; enable FRR on LSP |
| **BFD false positives** | Frequent unnecessary failovers | BFD session flapping logs | Timer too aggressive; CPU overload | Increase BFD interval; check CPU; consider hardware-offloaded BFD |

### Comprehensive Troubleshooting Methodology

#### Step 1: Identify the Failure Domain

```
Question: Is the problem...
├── All MPLS traffic? → Core issue (IGP, LDP global, interface)
├── Specific destination? → Targeted LSP issue
├── Specific VPN? → VPN label issue (not transport)
├── Intermittent? → ECMP, BFD flap, or MTU
└── After a change? → Config review
```

#### Step 2: Verify Control Plane (Bottom-Up)

1. **IGP**: Is the destination reachable via IGP?
   ```
   show ip route 10.0.0.5     / show route 10.0.0.5
   ```

2. **LDP/RSVP Session**: Is the signaling protocol up?
   ```
   show mpls ldp neighbor      / show ldp session
   show mpls traffic-eng tunnels / show mpls lsp
   ```

3. **Label Binding**: Does a label exist for the destination?
   ```
   show mpls ldp bindings 10.0.0.5 32  / show ldp database 10.0.0.5/32
   ```

4. **LFIB**: Is the forwarding table programmed?
   ```
   show mpls forwarding-table 10.0.0.5  / show route table mpls.0 label X
   ```

#### Step 3: Verify Data Plane

5. **LSP Ping**: Does the data plane work end-to-end?
   ```
   ping mpls ipv4 10.0.0.5/32  / ping mpls ldp 10.0.0.5/32
   ```

6. **LSP Traceroute**: Where does it break?
   ```
   traceroute mpls ipv4 10.0.0.5/32  / traceroute mpls ldp 10.0.0.5/32
   ```

7. **MTU Test**: Can full-size packets traverse the LSP?
   ```
   ping mpls ipv4 10.0.0.5/32 size 8986  / ping mpls ldp 10.0.0.5/32 size 8986
   ```

#### Step 4: Correlate and Fix

8. Cross-reference LSP traceroute output with IGP topology
9. Check the specific router where the failure occurs
10. Verify configuration, interface state, and hardware health on that device

### War Story: The Phantom LSP

A national backbone operator ran a 60-router MPLS core with LDP. Their NOC monitoring showed all LDP sessions up and all LFIB entries present. Yet customer VPN traffic between two specific PE routers was intermittently dropping — about 2% packet loss, consistent, all day.

Standard troubleshooting showed:
- ✅ LDP sessions: all up
- ✅ Label bindings: present for both PE loopbacks
- ✅ IP ping between PEs: 0% loss
- ✅ LSP ping: 0% loss (?!)

Wait — LSP ping showed 0% loss but customer traffic was losing 2%? Something was wrong.

The engineer ran LSP traceroute and noticed the path was 4 hops. But IP traceroute with TTL propagation showed 5 hops. The LSP was taking a different path than the IGP best path.

**Root cause**: A P router in the middle of the IGP best path had `mpls ip` disabled on one interface (it had been removed during a "cleanup" that accidentally deleted the config). The IGP route existed through that router, but there was no label for the outgoing interface. LDP treated that link as unlabeled and didn't include it in LSP setup.

Traffic flow split:
- **LSP ping** (labeled traffic): Took the LDP path (4 hops, avoiding the broken link) → 0% loss
- **IP ping** (unlabeled): Took the IGP best path (5 hops, through the broken router) → 0% loss (IP forwarding worked fine)
- **VPN traffic** (labeled): Sometimes took the LDP path (fine), sometimes hashed to ECMP path that included the broken link → packets arriving at the interface without MPLS enabled were dropped

The 2% loss corresponded to exactly the traffic that ECMP hashed to the path through the broken interface.

**Fix**: Re-enabled `mpls ip` on the missing interface. Loss dropped to 0% instantly.

**Lesson**: `ping` and `LSP ping` can both show 100% success while actual traffic is failing. Always check **ECMP paths** — test with different flow parameters (source/destination ports in LSP ping) to exercise all ECMP paths. And always verify `mpls ip` is enabled on every core-facing interface with `show mpls interfaces` / `show mpls interface`.

Also: use `traceroute mpls multipath ipv4 <prefix>` (IOS-XE) or `traceroute mpls ldp <prefix> multipath` (Junos) to discover ALL ECMP paths, not just one.
