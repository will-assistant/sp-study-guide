# 4.2 — RSVP-TE: Traffic Engineered Label Switching

## Overview
> RSVP-TE (Resource Reservation Protocol — Traffic Engineering) is MPLS's answer to the question: "What if I don't *want* traffic to follow the IGP shortest path?" While LDP blindly follows the IGP, RSVP-TE lets you engineer explicit paths through the network — reserving bandwidth, avoiding congested links, computing constraint-based paths, and providing sub-50ms failover with Fast Reroute. It's the foundation of traffic engineering in traditional MPLS networks and, despite Segment Routing's rise, remains deeply deployed in SP cores worldwide. If LDP is a GPS that always picks the "fastest route," RSVP-TE is the rally driver who picks the exact roads they want.

## Theory

### RSVP-TE Fundamentals

RSVP-TE extends the original RSVP protocol (RFC 2205, designed for IntServ QoS — which nobody uses) with LSP tunnel objects (RFC 3209). It provides:

1. **Explicit path control** — route traffic on any path, not just IGP shortest
2. **Bandwidth reservation** — reserve link bandwidth for LSPs
3. **Preemption** — higher-priority LSPs can steal bandwidth from lower-priority ones
4. **Fast Reroute** — pre-computed backup paths for sub-50ms failover
5. **Make-before-break** — seamless re-optimization without traffic loss

### Signaling: PATH and RESV

RSVP-TE uses two core messages that travel in opposite directions:

```
Ingress (Head-End)                                    Egress (Tail-End)
       ─── PATH message ──────────────────────────────→
       ←── RESV message ──────────────────────────────
```

**PATH message** (downstream, ingress → egress):
- Carries the **Explicit Route Object (ERO)** — the desired path
- Carries **SENDER_TSPEC** — requested bandwidth
- Carries **SESSION_ATTRIBUTE** — priority, preemption, FRR flags
- Each hop records itself in the **Record Route Object (RRO)**
- Each hop checks: "Can I satisfy these constraints?"

**RESV message** (upstream, egress → ingress):
- Carries the **LABEL object** — each hop allocates and returns its label
- Carries **RECORD_ROUTE** — confirmed path (with labels)
- Carries **FLOWSPEC** — confirmed bandwidth reservation
- This is where labels are actually distributed — opposite direction to LDP

**Refresh mechanism**: RSVP-TE is a soft-state protocol. Both PATH and RESV must be refreshed periodically (default: **30 seconds**). If three consecutive refreshes are missed (90 seconds), the LSP is torn down. This creates overhead — a router with 10,000 LSPs sends/processes hundreds of refresh messages per second.

**Summary Refresh (RFC 2961)**: Bundles multiple refresh states into a single message, dramatically reducing overhead. Enabled by default on modern platforms. Essential for scaling.

**RSVP Hello (RFC 3209)**: Node-to-node keepalive separate from PATH/RESV refresh. Provides faster failure detection than waiting for refresh timeout. Default: **9 seconds** detection (3 × 3s hellos). For faster detection, use BFD instead.

### Explicit Route Object (ERO)

The ERO defines the path the LSP should take. It can be:

- **Strict hop**: The LSP MUST transit this exact node/link — no deviation
- **Loose hop**: The LSP MUST reach this node but can take any IGP path to get there
- **Empty ERO**: The head-end doesn't specify any path — CSPF computes it entirely

```
Example ERO: strict 10.0.0.2, strict 10.0.0.3, loose 10.0.0.5
                                                              
Means: Go through P2, then P3 (exact links), then somehow reach PE5 (IGP decides)
```

### Record Route Object (RRO)

The RRO records the actual path taken, including:
- Node addresses at each hop
- Label values assigned at each hop
- Link protection flags (which hops have FRR protection)

The RRO is essential for **loop detection** (a router sees its own address) and for **make-before-break** (the head-end knows exactly what the old path looks like).

### CSPF — Constraint-Shortest Path First

CSPF is the algorithm that computes TE paths. It runs on the **head-end router** (not distributed like SPF):

1. Start with the TE database (TED) — built from IGP TE extensions (IS-IS TLV 22 / OSPF Type 10 LSA)
2. **Prune** links that don't meet constraints:
   - Not enough available bandwidth
   - Wrong admin-group (affinity/color)
   - Same SRLG as a link to avoid
3. Run **Dijkstra** on the remaining topology
4. Output: ERO with strict hops

**TE Database contents** (from IGP TE extensions):
- Maximum link bandwidth
- Maximum reservable bandwidth
- Unreserved bandwidth (per priority level, 0–7)
- TE metric (can differ from IGP metric)
- Admin groups (link coloring — 32-bit bitmask)
- SRLG values (Shared Risk Link Groups)

**CSPF tie-breaking** (when multiple equal-cost TE paths exist):
1. Least-loaded path (most available bandwidth)
2. Fewest hops
3. Random (implementation-dependent)

### Bandwidth Reservation and Preemption

RSVP-TE uses **8 priority levels** (0–7, where 0 is highest):
- **Setup priority**: The priority used when establishing the LSP
- **Hold priority**: The priority used to maintain the LSP once established
- **Rule**: Setup priority must be ≤ hold priority (you can't set up with higher priority than you hold — this prevents oscillation)

**Preemption**: If a new LSP with setup priority X wants bandwidth on a link, it can preempt existing LSPs with hold priority > X (lower priority). The preempted LSP is torn down and must re-signal on an alternate path.

**Default**: Setup 7, Hold 7 on both IOS-XE and Junos. This means all default-priority LSPs are equal — none can preempt any other. Most operators leave 7/7 for best-effort LSPs and define an explicit priority scheme (e.g., 2/2 for gold, 4/4 for silver) when preemption is needed.

### Bandwidth Overbooking

RSVP-TE tracks "reserved bandwidth" per priority per link. But this is a **control-plane accounting mechanism**, not a data-plane policer. If you reserve 5 Gbps on a 10G link, nothing prevents 8 Gbps of actual traffic from flowing. The reservation just tells CSPF "only 5 Gbps of reservable bandwidth remains."

Many operators intentionally overbook — reserving bandwidth at, say, 50% of physical capacity, knowing that not all LSPs will be fully loaded simultaneously. This is a design decision, not a bug.

### Fast Reroute (FRR)

FRR is RSVP-TE's killer feature: pre-computed backup paths that activate in sub-50ms on failure. RFC 4090 defines two methods:

#### Facility Backup (Bypass Tunnels)

```
                    Primary LSP: A → B → C → D
                    
                    A ───── B ───── C ───── D
                            │               │
                            └───── E ───────┘
                            
                    Bypass tunnel: B → E → D (protects link B-C and node C)
```

- **Point of Local Repair (PLR)**: Router B — detects the failure
- **Merge Point (MP)**: Router D — where protected and bypass paths merge
- The PLR pushes an **additional label** (the bypass tunnel label) onto the packet and sends it through the bypass
- **Link protection**: Bypass protects against a specific link failure (B→C)
- **Node protection**: Bypass protects against node C failing entirely (bypass goes B→E→D, skipping C)
- **Scalable**: One bypass tunnel protects ALL primary LSPs traversing that link/node — N:1 protection
- **Label stack**: During FRR, packets carry 3+ labels: bypass label → primary label → service label

#### One-to-One Backup (Detour LSPs)

- Each protected LSP gets its **own** dedicated backup path
- Signaled end-to-end from the PLR to the tail-end
- More granular (per-LSP bandwidth guarantees on backup path)
- **Less scalable**: N backup LSPs for N protected LSPs = 2N total state
- Rarely used in production due to scaling concerns

#### FRR Comparison

| Aspect | Facility Backup | One-to-One Backup |
|--------|----------------|-------------------|
| State overhead | Low (one bypass per link/node) | High (one detour per LSP) |
| Bandwidth guarantee | No (backup is best-effort) | Yes (per-LSP reservation) |
| Setup complexity | Lower | Higher |
| Scalability | Excellent (N:1) | Poor (1:1) |
| Label stack depth | +1 label during FRR | Same depth |
| Industry preference | **Dominant** | Rarely deployed |

### Make-Before-Break (MBB)

When an LSP needs to be re-optimized (better path available, manual reroute, bandwidth change), RSVP-TE signals the **new** path before tearing down the **old** one:

1. Head-end computes new path via CSPF
2. Signals new LSP using a different **LSP-ID** but same **Tunnel-ID** (shared SESSION object)
3. New LSP established end-to-end (new PATH/RESV complete)
4. Head-end switches traffic from old to new LSP (atomic in forwarding)
5. Old LSP torn down (PathTear)

**Key**: The shared SESSION object tells transit routers "these two LSPs are related" — they won't double-count bandwidth reservations during the transition.

**Re-optimization timer**: By default, the head-end periodically re-runs CSPF to check if a better path exists. IOS-XE default: **300 seconds** (5 minutes). Junos default: **300 seconds**.

### Scaling Considerations

| Parameter | Cisco IOS-XE | Junos |
|-----------|-------------|-------|
| Max RSVP-TE LSPs per router | ASR 1000: ~2,000; ASR 9000: ~20,000+ | MX960: ~10,000–30,000+ (depends on FPC) |
| Refresh reduction (RFC 2961) | Supported, enabled by default | Supported, enabled by default |
| RSVP Hello interval | 3s default | 3s default |
| RSVP authentication | MD5 per-neighbor | MD5 per-neighbor |
| Graceful Restart | Supported (RFC 3473 extensions) | Supported |
| Max bypass tunnels per interface | Platform-dependent; typically 8–16 | Platform-dependent |

**Scaling reality**: RSVP-TE state is per-LSP, per-hop. A full mesh of 100 PE routers = ~10,000 LSPs. Each transit router maintains PATH and RESV state for every LSP it transits. This is the primary scaling limitation of RSVP-TE versus Segment Routing.

## Configuration

### IOS-XE

```cisco
! === Enable RSVP-TE on interfaces ===
! MUST enable TE extensions in IGP first
!
! Step 1: Enable MPLS TE globally
mpls traffic-eng tunnels
!
! Step 2: Enable IS-IS TE extensions
router isis CORE
 mpls traffic-eng router-id Loopback0
 mpls traffic-eng level-2
 metric-style wide
!
! Step 3: Enable RSVP and TE on interfaces
interface GigabitEthernet0/0/0
 description TO-P2-CORE (10G)
 ip address 10.1.12.1 255.255.255.252
 mpls traffic-eng tunnels
 ip rsvp bandwidth 8000000
 ! Reserve max 8 Gbps on a 10G link (80% — leave headroom)
!
interface GigabitEthernet0/0/1
 description TO-P3-CORE (10G)
 ip address 10.1.13.1 255.255.255.252
 mpls traffic-eng tunnels
 ip rsvp bandwidth 8000000
!
! === RSVP-TE Tunnel (Head-End Configuration) ===
interface Tunnel10
 description TE-LSP to PE5 (primary path via P2-P4)
 ip unnumbered Loopback0
 tunnel mode mpls traffic-eng
 tunnel destination 10.0.0.5
 tunnel mpls traffic-eng bandwidth 1000000
 ! Reserve 1 Gbps
 tunnel mpls traffic-eng priority 4 4
 ! Setup priority 4, Hold priority 4
 tunnel mpls traffic-eng path-option 1 explicit name PATH-VIA-P2-P4
 tunnel mpls traffic-eng path-option 2 dynamic
 ! Fallback to CSPF-computed path if explicit fails
 tunnel mpls traffic-eng fast-reroute node-protect
 ! Enable node-protection FRR
 tunnel mpls traffic-eng autoroute announce
 ! Install this tunnel in IGP for automatic traffic steering
!
! === Explicit Path Definition ===
ip explicit-path name PATH-VIA-P2-P4 enable
 next-address strict 10.0.0.2
 next-address strict 10.0.0.4
!
! === FRR Bypass Tunnel (configured on PLR, e.g., P2) ===
interface Tunnel100
 description BYPASS-P2-to-P4-protecting-link-P2-P3
 ip unnumbered Loopback0
 tunnel mode mpls traffic-eng
 tunnel destination 10.0.0.4
 tunnel mpls traffic-eng path-option 1 explicit name BYPASS-PATH
 ! Bypass path avoids the protected link/node
!
! === RSVP Authentication ===
ip rsvp authentication key RSVP-KEY-2026
ip rsvp authentication type md5
!
! === RSVP Refresh Reduction ===
ip rsvp signalling refresh reduction
!
! === Re-optimization Timer ===
mpls traffic-eng reoptimize timers frequency 300
```

### Junos

```junos
# === Enable MPLS and RSVP on interfaces ===
set protocols mpls interface ge-0/0/0.0
set protocols mpls interface ge-0/0/1.0

set protocols rsvp interface ge-0/0/0.0 bandwidth 8g
set protocols rsvp interface ge-0/0/1.0 bandwidth 8g

# Subscription (overbooking) — allow reserving up to 100% (default)
# To allow 150% overbooking:
# set protocols rsvp interface ge-0/0/0.0 subscription 150

# === IS-IS TE Extensions ===
set protocols isis traffic-engineering

# === RSVP-TE LSP (Head-End Configuration) ===
set protocols mpls label-switched-path TO-PE5 to 10.0.0.5
set protocols mpls label-switched-path TO-PE5 bandwidth 1g
set protocols mpls label-switched-path TO-PE5 priority 4 4
set protocols mpls label-switched-path TO-PE5 primary PATH-VIA-P2-P4
set protocols mpls label-switched-path TO-PE5 secondary DYNAMIC-BACKUP standby
set protocols mpls label-switched-path TO-PE5 fast-reroute

# === Explicit Path Definition ===
set protocols mpls path PATH-VIA-P2-P4 10.0.0.2 strict
set protocols mpls path PATH-VIA-P2-P4 10.0.0.4 strict

# === Node-Link Protection (FRR) ===
# Applied per-interface on the PLR
set protocols rsvp interface ge-0/0/0.0 link-protection
# For node protection:
set protocols rsvp interface ge-0/0/0.0 link-protection node-link-protection

# === Make-Before-Break (automatic) ===
# Junos does MBB by default for re-optimization. To enable adaptive:
set protocols mpls label-switched-path TO-PE5 adaptive

# === Re-optimization ===
set protocols mpls optimize-timer 300

# === RSVP Authentication ===
set protocols rsvp interface ge-0/0/0.0 authentication-key "RSVP-KEY-2026"

# === RSVP Refresh Reduction (enabled by default on Junos) ===
# To verify: show rsvp interface

# === Admin Groups (Link Coloring) ===
set protocols mpls admin-groups GOLD 0
set protocols mpls admin-groups SILVER 1
set protocols mpls admin-groups AVOID-SATELLITE 2

set protocols mpls interface ge-0/0/0.0 admin-group GOLD
set protocols mpls interface ge-0/0/1.0 admin-group SILVER

# LSP with affinity constraint:
set protocols mpls label-switched-path TO-PE5-GOLD to 10.0.0.5
set protocols mpls label-switched-path TO-PE5-GOLD admin-group include-any GOLD
set protocols mpls label-switched-path TO-PE5-GOLD admin-group exclude AVOID-SATELLITE

# === SRLG Configuration ===
set routing-options srlg FIBER-RING-1 srlg-value 100
set protocols rsvp interface ge-0/0/0.0 srlg FIBER-RING-1
```

### Key Knobs

| Parameter | IOS-XE Default | Junos Default | Recommended | Why |
|-----------|---------------|---------------|-------------|-----|
| RSVP bandwidth per interface | 0 (no reservation) | 0 | 75–80% of physical BW | Leave headroom for control plane and best-effort |
| Re-optimization timer | 300s | 300s | 300s | Balance between optimality and churn |
| Setup/Hold priority | 7/7 | 7/7 | 7/7 (or design a priority scheme) | All default LSPs equal; design preemption tiers deliberately |
| FRR type | Not enabled by default | Not enabled by default | Node-protection where possible | Node protection > link protection |
| RSVP Hello interval | 3s | 3s | 3s (or use BFD) | 9s detection; BFD is faster |
| Refresh reduction | Available | Enabled by default | Enable everywhere | Essential for scale |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| LSP down, "no route to destination" | CSPF can't find a path meeting constraints | Check TED (`show ted database`); reduce constraints; verify TE extensions in IGP |
| LSP down, "admission control failure" | Not enough unreserved bandwidth on a link | Increase interface RSVP BW; reduce LSP BW; allow preemption |
| LSP oscillating (up/down/up) | Preemption war between LSPs | Review priority scheme; ensure setup ≤ hold priority |
| FRR not protecting an LSP | Bypass tunnel doesn't exist or can't reach merge point | Verify bypass configuration on PLR; check NNHOP reachability |
| LSP up but traffic not using it | `autoroute announce` missing; or IGP metric not favorable | Add autoroute; or use `tunnel mpls traffic-eng forwarding-adjacency` |
| RSVP session flapping | Refresh timeouts due to CPU load | Enable refresh reduction; check CPU; increase hello interval |
| TED missing link information | IGP TE extensions not enabled | Enable `mpls traffic-eng` under IGP and on interfaces |
| Bandwidth accounting wrong | Preempted LSP not cleaned up; stale reservation | Clear stale reservations; check `show ip rsvp reservation` |

### Methodology

1. **Check the TED**: `show mpls traffic-eng topology` / `show ted database` — does the head-end see all links with TE attributes?
2. **Check CSPF**: Can CSPF compute a path? On Junos: `show mpls lsp name X extensive` shows CSPF output. On IOS-XE: enable debug or check path-option status.
3. **Check signaling**: Is PATH reaching the tail-end? Is RESV coming back? Check `show ip rsvp sender` and `show ip rsvp reservation` on transit hops.
4. **Check FRR**: Is bypass tunnel up? Does `show mpls traffic-eng fast-reroute database` show protection?
5. **Check forwarding**: Is traffic actually entering the tunnel? Check interface counters on the tunnel interface.

### War Story: The Preemption Cascade

A large European ISP deployed RSVP-TE with a tiered priority scheme: gold VPN LSPs at priority 2/2, silver at 4/4, and best-effort at 7/7. During a fiber cut that removed 40% of east-west capacity, CSPF rerouted gold LSPs to remaining paths, preempting silver LSPs. The silver LSPs then rerouted, preempting best-effort LSPs. The best-effort LSPs had nowhere to go and failed entirely.

But here's the nasty part: some of the rerouted gold LSPs chose paths that were actually *longer* and consumed *more* bandwidth than their original paths. This left even less room for silver, triggering more preemptions. The cascade took 45 minutes to stabilize — far longer than the actual failover.

**Root cause**: No preemption dampening and no bandwidth caps per priority class. The CSPF was technically correct but operationally destructive.

**Fix**:
1. Implemented **preemption hold-timer** — don't preempt an LSP within 60 seconds of it being established (prevents oscillation)
2. Set per-class **bandwidth pools** — gold can only use max 40% of any link's reservable bandwidth
3. Reduced re-optimization frequency to 600 seconds after a network event
4. Added **soft-preemption** (RFC 5712) — signals to the preempted LSP "you're about to be preempted, reroute gracefully" instead of hard-killing it

**Lesson**: Priority without bandwidth governance turns a controlled failover into a thundering herd. Always model your preemption scheme under worst-case capacity loss.
