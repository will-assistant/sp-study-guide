# 6.5 SRv6 Network Programming

## Overview

SRv6 Network Programming is what separates SRv6 from "MPLS with IPv6 headers." Where SR-MPLS gives you label-based forwarding, SRv6 gives you a *programmable data plane* — each SID encodes not just a destination, but a *behavior* to execute at that destination. Chain these behaviors together and you have a program that executes hop-by-hop across the network.

This section deep-dives into RFC 8986 endpoint behaviors, SRv6 flavors (PSP, USP, USD), micro-SID (uSID) compression, advanced service chaining, and the operational reality of deploying programmable SRv6 in production SP networks.

If Section 6.4 was "what SRv6 is," this section is "what SRv6 *does*."

## Theory: The Network Programming Model

### From Forwarding to Programming

Traditional routing: packet arrives → FIB lookup → forward toward destination.

SR-MPLS: packet arrives → pop label → execute implied action (forward, VPN lookup, etc.).

SRv6 Network Programming: packet arrives → match SID against Local SID Table → execute the *specific behavior* encoded in the SID's function field → process the SRH → continue.

The fundamental shift: **SIDs are instructions, not just addresses.** An SRv6 segment list is literally a program — a sequence of instructions that the network executes on your packet as it traverses the path.

```
Segment List (= Program):

  SID 1: fc00:1::100  → END         (transit node 1: decrement SL, update DA)
  SID 2: fc00:2::200  → END.X       (node 2: forward to specific adjacency)
  SID 3: fc00:3::46   → END.DT4     (node 3: decap, lookup in IPv4 VRF)
```

Each SID triggers a specific behavior at a specific node. The ingress PE "writes the program" by constructing the segment list; transit and egress nodes "execute" it.

### RFC 8986: SRv6 Network Programming Specification

RFC 8986 defines the canonical set of SRv6 endpoint behaviors. These are the "instruction set" of the SRv6 data plane.

#### Endpoint Behaviors (Full Taxonomy)

| Behavior | Description | Use Case |
|----------|-------------|----------|
| **End** | Endpoint — decrement SL, update DA to next SID | Transit steering |
| **End.X** | Endpoint with Layer-3 cross-connect | Adjacency-specific forwarding (like Adj-SID) |
| **End.T** | Endpoint with specific table lookup | Multi-topology / table-based forwarding |
| **End.DX2** | Endpoint with L2 cross-connect | L2VPN (pseudowire replacement) |
| **End.DX4** | Endpoint with IPv4 cross-connect | Per-CE IPv4 VPN (like per-CE label) |
| **End.DX6** | Endpoint with IPv6 cross-connect | Per-CE IPv6 VPN |
| **End.DT4** | Endpoint with IPv4 table lookup | IPv4 L3VPN (like VPNv4 label) |
| **End.DT6** | Endpoint with IPv6 table lookup | IPv6 L3VPN (like VPNv6 label) |
| **End.DT46** | Endpoint with IPv4/IPv6 table lookup | Dual-stack VPN in single SID |
| **End.B6.Encaps** | Endpoint bound to SRv6 policy (encaps) | Policy insertion / re-encapsulation |
| **End.B6.Encaps.Red** | Same as above, reduced SRH | Fewer bytes overhead |
| **End.BM** | Endpoint bound to SR-MPLS policy | SRv6-to-SR-MPLS interworking |
| **End.S** ² | Endpoint to SR-MPLS (push label) | SRv6→MPLS gateway |
| **End.AS** ² | Endpoint with static proxy (service chaining) | Service function insertion without SRv6 awareness |

> ² End.S and End.AS are from draft-ietf-spring-sr-service-programming, not RFC 8986. End.AS is a *static proxy* for service chaining (sends to SRv6-unaware appliance with pre-configured SRH restoration), NOT an SR-MPLS interworking function. Verify vendor support.

> **Key insight:** The `D` prefix means "decapsulation" — the SRv6 header is removed and the inner packet is forwarded. `X` means cross-connect (specific next-hop). `T` means table lookup. `B` means binding (insert into another policy). `S` means SR-MPLS interworking.

#### Transit Behaviors

Transit behaviors apply to nodes that are NOT the SID endpoint — they just forward the packet because the destination address doesn't match their Local SID Table.

| Behavior | Description |
|----------|-------------|
| **T** | Transit — standard IPv6 forwarding (SRH present but not processed) |
| **T.Insert** | Transit with SRH insertion (don't encap, just add SRH to existing packet) |
| **T.Insert.Red** | Transit insert with reduced SRH |
| **T.Encaps** | Transit with full encapsulation (outer IPv6 + SRH) |
| **T.Encaps.Red** | Transit encaps with reduced SRH |

**T.Insert vs T.Encaps — the critical difference:**
- **T.Insert**: Modifies the *existing* IPv6 header — adds an SRH extension header, copies original DA to segment list, sets DA to first SID. The original packet's IPv6 header is preserved.
- **T.Encaps**: Adds a *new* outer IPv6 header + SRH. The original packet (including its IPv6 header) becomes the payload. This is full encapsulation.

**When to use which:**
- T.Encaps is the safe default — works for any inner payload (IPv4, IPv6, Ethernet)
- T.Insert only works when the original packet is already IPv6 (you're inserting SRH into it)
- T.Insert saves ~40 bytes (no outer IPv6 header) but has security implications — the inner packet's source/destination are visible to transit nodes
- Most production deployments use T.Encaps for VPN traffic and T.Insert for intra-domain steering

### SRv6 Flavors: PSP, USP, USD

RFC 8986 defines three "flavors" that modify how the *penultimate* and *ultimate* segment endpoints process the SRH. These are critical for performance and interoperability.

#### Penultimate Segment Pop (PSP)

**What it does:** The *penultimate* SID endpoint (Segments Left = 1) removes the SRH before forwarding to the final SID.

**Why it matters:** Without PSP, the ultimate segment endpoint receives a packet with an SRH that has SL=0 — it has to process and remove the (now useless) SRH. With PSP, the penultimate node does this work, so the ultimate node just sees a plain IPv6 packet.

```
Without PSP:
  Node A (SL=2) → Node B (SL=1) → Node C (SL=0, still has SRH!)
                                    Node C must process + strip SRH

With PSP:
  Node A (SL=2) → Node B (SL=1, PSP: remove SRH) → Node C (plain IPv6)
                                                      Node C does normal IPv6 forwarding
```

**Operational benefit:**
- Final node doesn't need to be SRv6-aware (interop with legacy IPv6 routers)
- Reduces processing at the egress node
- Enables "SRv6 transit" through non-SRv6 domains

**Configuration (IOS-XR):**
```cisco
segment-routing
 srv6
  locators
   locator MAIN
    micro-segment behavior unode psp-usd
    !
```

**Configuration (Junos):**
```junos
routing-options {
    segment-routing {
        srv6 {
            locator MAIN {
                end-flavors {
                    psp;
                }
            }
        }
    }
}
```

#### Ultimate Segment Pop (USP)

**What it does:** The *ultimate* SID endpoint (Segments Left = 0) processes the SID and removes the SRH.

**Why it matters:** USP is a **distinct flavor, NOT implicit.** The base END behavior per RFC 8986 **requires SL > 0** — if a packet arrives at an END SID with SL=0 and USP is not enabled, the packet is **dropped** (error condition). USP explicitly allows an END SID to process and pop the SRH when SL=0.

> ⚠️ **Common lab mistake:** Candidates see packets dropped at the ultimate node with SL=0 and don't realize it's because USP flavor isn't enabled on that SID. Without PSP (to remove SRH at penultimate) or USP (to handle SL=0 at ultimate), the base END behavior treats SL=0 as an error.

**Distinction from PSP:** With PSP, the SRH is removed *before* reaching the ultimate node (at SL=1). With USP, the SRH is removed *at* the ultimate node (at SL=0). They serve different operational needs:
- PSP = optimize for egress node (offload SRH removal to penultimate)
- USP = explicit handling at egress when SRH still present (no PSP available)

#### Ultimate Segment Decapsulation (USD)

**What it does:** The ultimate SID endpoint performs decapsulation — removes the outer IPv6 header (and SRH) and forwards the inner packet.

**Important distinction:** D-class functions (End.DT4, End.DT6, End.DX4, etc.) **inherently perform decapsulation** as part of their behavior — they don't need the USD flavor for this. USD flavor is specifically for when you want a **non-D SID** (like plain End) to trigger decapsulation at SL=0. In practice, USD is most useful as a combined PSP-USD flavor, where it serves as a fallback: if PSP didn't strip the SRH (e.g., single-hop path where there is no penultimate node), USD ensures the ultimate node can still decap.

**In the VPN context:** At a VPN egress PE, the End.DT4 function itself strips the outer IPv6 header and does VRF lookup — that's what the "D" means. USD flavor adds the ability for this to work cleanly when arriving with SL=0 and SRH still present.

**The combo you'll see in production: PSP-USD**
- PSP at the penultimate node removes the SRH
- USD at the ultimate node removes the outer IPv6 header and forwards the inner packet
- Result: minimal processing at the egress PE, clean decapsulation

```
Ingress PE          Transit (penultimate)       Egress PE
[IPv6 + SRH + IPv4] → [IPv6 + SRH + IPv4]    → [IPv6 + IPv4]     → [IPv4]
                       PSP: strip SRH,           USD: Local SID lookup on DA,
                       set DA to last SID,        decap outer IPv6, forward
                       forward plain IPv6         inner packet into VRF
```

> **How PSP-USD actually works:** The penultimate node (SL=1) processes its SID, then — because PSP is enabled — removes the SRH and sets the DA to the final SID (SL[0]). The packet forwarded to the ultimate node is a plain IPv6 packet with the ultimate SID as DA. The ultimate node matches the DA against its Local SID Table, finds the USD-flavored SID, and performs decapsulation — it strips the outer IPv6 header and forwards the inner packet. The key: the ultimate node does NOT need to process an SRH (PSP already removed it). It just does a Local SID Table lookup and decaps.

#### Flavor Summary

| Flavor | Who Acts | What's Removed | When to Use |
|--------|----------|----------------|-------------|
| PSP | Penultimate node (SL=1) | SRH only | Offload egress, interop with legacy |
| USP | Ultimate node (SL=0) | SRH only | Default, explicit egress processing |
| USD | Ultimate node (SL=0) | Outer IPv6 + SRH | VPN decapsulation at egress PE |

> **Production recommendation:** Use **PSP-USD** for VPN services. The penultimate node strips the SRH (PSP), then the egress PE decapsulates the outer IPv6 header (USD). This is the most hardware-friendly combination and the de facto standard in deployments.

### Micro-SID (uSID): The Compression Game-Changer

#### The Problem: SRv6 Overhead

Standard SRv6 SIDs are 128 bits (16 bytes) each. A segment list with 6 SIDs = 96 bytes just for the segment list, plus the 8-byte SRH header, plus the 40-byte outer IPv6 header = **144 bytes of overhead**.

Compare to SR-MPLS: 6 labels × 4 bytes = 24 bytes. SRv6 is 6x heavier.

This matters because:
1. **MTU pressure** — 144 bytes of overhead on a 1500-byte MTU leaves only 1356 bytes for payload
2. **Hardware TCAM** — Longer headers consume more entries in forwarding ASICs
3. **Bandwidth waste** — At scale, 6x more overhead per packet adds up

#### The Solution: Micro-SID (uSID)

Micro-SID (RFC 9602, "Compressed SRv6 Segment List Encoding," published 2024) compresses multiple logical SIDs into a single 128-bit container.

**How it works:**

Instead of each SID being a full 128-bit address, a micro-SID is only 16 or 32 bits. Multiple uSIDs are packed into a single 128-bit "carrier" SID:

```
Standard SRv6 (6 SIDs = 96 bytes in segment list):
┌──────────────────┐
│ SID 1 (128 bits) │
├──────────────────┤
│ SID 2 (128 bits) │
├──────────────────┤
│ SID 3 (128 bits) │
├──────────────────┤
│ SID 4 (128 bits) │
├──────────────────┤
│ SID 5 (128 bits) │
├──────────────────┤
│ SID 6 (128 bits) │
└──────────────────┘

Micro-SID (6 uSIDs packed into 1 carrier = 16 bytes in segment list):
┌───────────────────────────────────────────────────┐
│ Locator (32b) │ uSID1 │ uSID2 │ uSID3 │ uSID4 │ uSID5 │ uSID6 │
│               │ (16b) │ (16b) │ (16b) │ (16b) │ (16b) │ (16b) │
└───────────────────────────────────────────────────┘
```

With 16-bit uSIDs and a 32-bit block (locator prefix), you can fit **6 micro-SIDs** in a single 128-bit carrier. That's 6 hops in 16 bytes instead of 96 bytes — **an 83% reduction**.

#### uSID Structure

```
┌──────────────────┬───────┬───────┬───────┬───────┬───────┬───────┐
│   uSID Block     │ uSID  │ uSID  │ uSID  │ uSID  │ uSID  │ uSID  │
│   (32 bits)      │  1    │  2    │  3    │  4    │  5    │  6    │
│   e.g. fcbb:bb00 │(16b)  │(16b)  │(16b)  │(16b)  │(16b)  │(16b)  │
└──────────────────┴───────┴───────┴───────┴───────┴───────┴───────┘
```

- **uSID Block**: A /32 or /48 prefix that identifies the SRv6 domain (e.g., `fcbb:bb00::/32`)
- **uSID**: 16-bit identifier per node/behavior (e.g., `0001` = node 1 END, `0046` = node VPN decap)
- **Shift operation**: After processing a uSID, the node left-shifts the remaining uSIDs by 16 bits and pads with zeros. When all uSIDs in the carrier are consumed (all zeros after the block), move to the next carrier SID in the SRH.

#### uSID Forwarding Example

```
Carrier SID: fcbb:bb00:0001:0002:0003:0046:0000:0000

At Node 1 (uSID 0001 = END):
  - Process uSID 0001
  - Left-shift: fcbb:bb00:0002:0003:0046:0000:0000:0000
  - Update DA, forward

At Node 2 (uSID 0002 = END):
  - Process uSID 0002
  - Left-shift: fcbb:bb00:0003:0046:0000:0000:0000:0000
  - Update DA, forward

At Node 3 (uSID 0003 = END.X):
  - Process uSID 0003
  - Left-shift: fcbb:bb00:0046:0000:0000:0000:0000:0000
  - Update DA, forward

At Node 4 (uSID 0046 = END.DT4):
  - Process uSID 0046
  - Left-shift: fcbb:bb00:0000:0000:0000:0000:0000:0000
  - All zeros after block → move to next carrier SID (or SL--)
  - Decapsulate, forward inner IPv4 into VRF
```

#### uSID vs Standard SRv6 Overhead Comparison

| Metric | Standard SRv6 (6 SIDs) | uSID (6 uSIDs) | SR-MPLS (6 labels) |
|--------|------------------------|-----------------|---------------------|
| Segment list size | 96 bytes (6 × 16) | 16 bytes (1 carrier) | 24 bytes (6 × 4) |
| SRH header (base + segments) | 104 bytes (8 + 96) | 24 bytes (8 + 16) | N/A |
| Outer IPv6 header | 40 bytes | 40 bytes | N/A |
| **Total overhead** | **144 bytes** | **64 bytes** | **24 bytes** |
| MTU impact (1500) | 1356 payload | 1436 payload | 1476 payload |

> **Note on SRH size:** The SRH base header is 8 bytes. Each 128-bit segment entry adds 16 bytes. With standard SRv6 and 6 SIDs, the SRH is 8 + (6 × 16) = 104 bytes. With uSID, 6 micro-SIDs fit in 1 carrier (1 segment entry), so the SRH is 8 + 16 = 24 bytes.
>
> **T.Encaps.Red optimization:** With reduced encapsulation (T.Encaps.Red) and a single segment, the first SID goes directly into the IPv6 DA and the SRH is omitted entirely — overhead drops to just 40 bytes (outer IPv6 header only). This is the best-case scenario for uSID with 6 or fewer hops.

uSID brings SRv6 overhead much closer to SR-MPLS, making it practical for real SP deployments.

> **Critical caveat:** uSID requires all nodes in the SRv6 domain to support the micro-SID shift operation. You cannot mix standard SRv6 and uSID nodes in the same segment list carrier — a non-uSID node won't know how to left-shift. You CAN have standard SIDs and uSID carriers in different positions of the same segment list (different SRH entries), but within a single 128-bit carrier, all referenced nodes must support uSID.

### Advanced Service Chaining

#### SRv6 as a Service Function Chaining (SFC) Platform

Traditional SFC (RFC 7665) requires Network Service Headers (NSH, RFC 8300) or other overlays to steer traffic through service functions (firewalls, DPI, load balancers). SRv6 can replace NSH by encoding service functions as SIDs.

**Example: Firewall insertion via SRv6**

```
Segment List:
  SID 1: fcbb:bb00:0001::  → Transit to firewall site
  SID 2: fcbb:bb00:FW01::  → END.AS (send to firewall appliance)
  SID 3: fcbb:bb00:0003::  → Transit to egress
  SID 4: fcbb:bb00:0004:46:: → END.DT4 (VPN decap at egress)
```

The firewall doesn't need to understand SRv6 — the node hosting SID 2 (END.AS) decapsulates before sending to the firewall and re-encapsulates on return.

#### End.AD (Endpoint with Dynamic Proxy)

> **Note:** End.AD and End.AM are defined in draft-ietf-spring-sr-service-programming, not RFC 8986. They are widely implemented but not yet RFC-track as of 2026. Verify vendor support.

End.AD enables service chaining through SRv6-unaware appliances:

1. Packet arrives at SRv6 proxy node with End.AD SID
2. Proxy caches the SRH (stores in local state, keyed by inner flow)
3. Proxy strips SRv6 headers, forwards bare packet to service function
4. Service function processes and returns the packet
5. Proxy restores cached SRH, continues SRv6 forwarding

**Trade-off:** End.AD requires per-flow state at the proxy, which breaks the stateless-core principle of SR. Use sparingly and only when the service function cannot be SRv6-enabled.

#### End.AM (Endpoint with Masquerading)

End.AM is a lighter alternative to End.AD:

1. Packet arrives with SRv6 headers
2. Proxy "masquerades" the packet — swaps the DA to the service function's address while keeping the SRH intact (moves SRH to a different extension header position)
3. Service function sees a normal IPv6 packet, processes it, returns it
4. Proxy "de-masquerades" — restores the DA from the SRH, continues forwarding

**Advantage over End.AD:** No per-flow state required. The SRH survives the trip through the service function (it's just repositioned, not cached).

**Limitation:** Only works if the service function preserves IPv6 extension headers it doesn't understand. Some firewalls strip unknown extension headers, which would destroy the SRH.

### SRv6 + EVPN Integration

SRv6 integrates with EVPN (RFC 9252) to provide L2/L3 VPN services without MPLS:

**EVPN Type-2 (MAC/IP) with SRv6:**
- BGP EVPN carries SRv6 SID in the SRv6-VPN SID attribute (instead of MPLS label)
- Egress PE advertises an End.DX2 SID for per-CE L2 service or End.DT2 for VLAN-aware bundle
- Ingress PE encapsulates with outer IPv6 header + SRH pointing to the EVPN SID

**EVPN Type-5 (IP Prefix) with SRv6:**
- Same as L3VPN — End.DT4/DT6/DT46 SIDs advertised via BGP EVPN
- Used for symmetric IRB (Integrated Routing and Bridging) in DC/SP edge

**Key SIDs for EVPN:**

| EVPN Service | SRv6 Behavior | Equivalent MPLS |
|-------------|---------------|-----------------|
| Per-CE L2 | End.DX2 | PW label |
| VLAN bundle | End.DT2 ¹ | VPLS label |
| Per-CE L3 (v4) | End.DX4 | Per-CE VPN label |
| Per-CE L3 (v6) | End.DX6 | Per-CE VPN label |
| Per-VRF L3 (v4) | End.DT4 | Per-VRF VPN label |
| Per-VRF L3 (v6) | End.DT6 | Per-VRF VPN label |
| Dual-stack L3 | End.DT46 ¹ | Two labels (v4 + v6) |

> ¹ End.DT2 and End.DT46 are NOT defined in RFC 8986. They come from draft-ietf-spring-srv6-network-programming-ext (the SRv6 Network Programming extensions draft). Widely implemented but verify vendor support for your specific platform/version.


### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| uSID block size | /48 | /32 for large SPs, /48 for enterprise | /32 = more uSIDs per carrier (6 × 16-bit), /48 = easier allocation |
| Encap source | Auto (loopback) | Explicit loopback | Predictable, filterable in ACLs and telemetry |
| SRv6 flavor | None | PSP-USD | De facto standard, best hardware support |
| Alloc mode | per-prefix | per-vrf | per-vrf = fewer SIDs, simpler FIB; per-prefix = finer control |
| SRH insertion | T.Encaps | T.Encaps for VPN, T.Insert for intra-domain | T.Encaps is safer (full encap), T.Insert saves 40 bytes |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| SID shows `Error` state | Locator prefix conflict or SID allocation overlap | Check `show segment-routing srv6 sid` for conflicts; ensure unique locator per node |
| uSID shift fails (packet dropped) | Non-uSID node in uSID carrier path | Verify ALL nodes in uSID segment support micro-SID; use standard SIDs for non-uSID nodes |
| VPN traffic black-holed at egress | Wrong SID behavior (End instead of End.DT4) | Verify BGP advertises correct SID type; check `alloc mode` (per-vrf vs per-prefix) |
| PSP not stripping SRH | Penultimate node doesn't support PSP flavor | Check penultimate node's locator config; must have `psp` flavor enabled |
| T.Insert packet dropped at transit | Transit node strips unknown extension headers | Common with firewalls/middleboxes; switch to T.Encaps or whitelist SRH at firewall |
| MTU black hole with uSID | uSID overhead calculation wrong | Even with uSID, still have 40B outer IPv6 + 8B SRH minimum = 48B overhead; set path MTU accordingly |
| SRv6 SID not in IGP | Locator not bound to IGP or IGP SRv6 not enabled | Check `router isis` / `protocols isis` for `segment-routing srv6 locator` binding |
| BGP VPN routes missing SRv6 SID | BGP session not configured for SRv6 encapsulation | Add `encapsulation-type srv6` (IOS-XR) or `extended-nexthop` (Junos) to BGP neighbor/family |

### Methodology

1. **Verify SRv6 is globally enabled**: `show segment-routing srv6` — confirm locators are configured and active
2. **Check Local SID Table**: `show segment-routing srv6 sid` — are the expected SIDs present with correct behaviors?
3. **Verify IGP advertisement**: `show isis database detail` — is the locator being advertised to neighbors?
4. **Check remote SID reachability**: `show route ipv6 <remote-locator>` — can you reach the remote node's locator?
5. **Inspect BGP SRv6 signaling**: `show bgp vpnv4 unicast <prefix>` — does the BGP route carry an SRv6 SID?
6. **Verify CEF/FIB entry**: `show cef ipv6 <sid> detail` — is the SID programmed in hardware?
7. **Test end-to-end**: `ping vrf CUSTOMER-A <ce-address> source <ce-interface>` — does VPN traffic flow?
8. **Check counters**: `show segment-routing srv6 sid <sid> detail` — are packets hitting the SID? Incrementing?
9. **Packet capture**: If all else fails, capture at ingress and egress to verify SRH construction, SID list ordering (remember: reverse order per RFC 8754), and proper decapsulation

### War Story: The uSID Shift That Wasn't

> **Scenario (composite/illustrative):** A Tier-1 SP deploys SRv6 with uSID across their backbone. Everything works in the lab. In production, traffic through one specific path keeps getting dropped at the third transit node.
>
> **Investigation:** The third transit node was a different hardware platform (older linecard) that had been upgraded to an OS version supporting SRv6 — but the ASIC firmware didn't support the uSID shift operation. The node saw the packet's destination address (a uSID carrier), matched it against its Local SID Table for the first uSID, processed it correctly... but then couldn't perform the left-shift to advance to the next uSID. Instead of shifting, it tried to do a standard IPv6 lookup on the post-processing address, which didn't match any route. Drop.
>
> **The fix:** The team had to either (a) upgrade the linecard firmware, (b) replace the linecard, or (c) route around the node using standard (non-uSID) SIDs for paths traversing that hardware.
>
> **The lesson:**
> 1. uSID support isn't just a software feature — it requires ASIC-level support for the shift operation
> 2. "SRv6-capable" ≠ "uSID-capable" — always verify micro-SID support per platform/linecard
> 3. Lab testing must use the EXACT hardware mix you'll deploy in production
> 4. Have a fallback: maintain standard SRv6 SIDs for paths that might cross non-uSID hardware
> 5. The shift operation is unique to uSID — there's no equivalent in SR-MPLS or standard SRv6 — so it's a new failure mode that teams aren't used to looking for

## Design Considerations

### When to Use SRv6 Network Programming

**Use SRv6 Network Programming when:**
- Greenfield IPv6-native deployment (no legacy MPLS to maintain)
- Service chaining requirements (steering through VNFs/appliances)
- Unified L2/L3 VPN with SRv6 + EVPN
- Network programmability is a strategic goal (API-driven, intent-based)
- Your hardware supports SRv6/uSID in ASIC (not just software forwarding)

**Stick with SR-MPLS when:**
- Existing MPLS deployment with investment to protect
- Mixed-vendor environment (SR-MPLS has broader vendor support today)
- MTU-constrained paths (SR-MPLS is still lighter than uSID)
- Hardware doesn't support SRv6 forwarding in silicon
- Operations team is MPLS-native and retraining cost is high

### uSID Block Allocation Strategy

| SP Size | Recommended Block | uSID Width | Nodes per Block | Notes |
|---------|------------------|------------|-----------------|-------|
| Small (< 100 nodes) | /48 | 16-bit | 65,534 | Simple, plenty of space |
| Medium (100–1000) | /40 or /48 | 16-bit | 65,534 | /40 gives more carrier space |
| Large (1000+) | /32 | 16-bit | 65,534 | Maximum compression (6 uSIDs/carrier) |
| Global (multi-domain) | /32 per domain | 16-bit | Unique block per domain | Prevents cross-domain conflicts |

**Allocation rules:**
1. Use `fcbb:bb00::/32` (conventional uSID block from Cisco docs/IETF examples) or allocate from fd00::/8 (ULA locally-assigned space per RFC 4193). Note: `fcbb::/16` technically falls in fc00::/8 (ULA centrally-assigned, no registry exists), but it's the de facto standard for uSID examples. For production, use your own ULA prefix from fd00::/8 or coordinate with your addressing team. Do NOT use GUA (Global Unicast) for internal SRv6 SIDs
2. Assign unique uSID values per node within a block — there's no SRGB-like automatic protection, so use an IPAM tool
3. Reserve uSID ranges for functions: e.g., 0x0001–0x00FF for END (transit), 0x0100–0x01FF for END.DT4, etc.
4. Document your allocation scheme — uSID conflicts are silent killers (traffic goes to wrong node)

### MTU Planning with uSID

```
Minimum overhead (uSID, 1 carrier):
  Outer IPv6 header:  40 bytes
  SRH (1 segment):    24 bytes  (8 base + 16 per segment)
  ─────────────────────────────
  Total:              64 bytes

Maximum payload in 1500 MTU:  1436 bytes
Maximum payload in 9000 MTU:  8936 bytes

With 2 carriers (12 uSIDs):
  SRH (2 segments):   40 bytes  (8 base + 2×16)
  Outer IPv6:         40 bytes
  Total:              80 bytes
  Payload (1500):     1420 bytes
```

**Recommendation:** Set core MTU to 9000+ (jumbo frames). With jumbo frames, even worst-case SRv6 overhead is negligible. On 1500 MTU paths, uSID is the only practical option for multi-hop steering.
