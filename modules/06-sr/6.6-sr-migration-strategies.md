# 6.6 SR Migration Strategies

## Overview

Nobody rips out a running MPLS core on a Friday night. Migration from LDP, RSVP-TE, or SR-MPLS to the next-generation transport is the most operationally critical phase of any SR deployment. Get it wrong and you're explaining to the NOC why 40% of your L3VPN traffic just disappeared.

This section covers the three major migration paths in service provider networks: LDP→SR-MPLS (the most common today), RSVP-TE→SR-TE (less common but strategically important), and SR-MPLS→SRv6 (the long game). Each migration supports hitless coexistence — the entire point is that you never need a maintenance window for the cutover itself.

## Theory: Migration Principles

### Why Migrate?

**LDP → SR-MPLS:**
- LDP has no traffic engineering capability
- LDP requires full-mesh targeted sessions for RSVP-TE FRR interop
- LDP follows IGP shortest path only — no optimization possible
- SR-MPLS eliminates LDP state (no per-prefix label bindings, no keepalives)
- TI-LFA (Section 6.3) provides sub-50ms protection without RSVP-TE

**RSVP-TE → SR-TE:**
- RSVP-TE state scales O(n²) with LSP count — thousands of LSPs = millions of RSVP messages
- RSVP-TE requires per-hop state for every LSP (label bindings on every transit router)
- SR-TE is stateless in the core — only head-end maintains policy state
- SR-TE policies with ODN (On-Demand Next-hop) replace static RSVP-TE tunnel provisioning
- PCEP-based computation works identically for SR-TE (same PCE, different LSP type)

**SR-MPLS → SRv6:**
- Future-proofing: IPv6-native transport, no MPLS dependency
- Network programming: SRv6 functions (END.DT4, END.DX6, etc.) replace VPN label stacks
- Simplified stack: one protocol (IPv6) vs two (IPv6 + MPLS)
- Trade-off: higher overhead, IPv6 underlay required, less vendor maturity (as of 2025)

### The Golden Rule: Ships in the Night

Every SR migration follows the **ships-in-the-night** principle: old and new protocols run simultaneously, both distributing labels/SIDs, both building forwarding entries. Traffic follows whichever has a valid path. You migrate by:

1. **Enable SR alongside existing protocol** (both active, SR unused)
2. **Shift preference** so SR paths are preferred
3. **Validate** that all services work on SR paths
4. **Remove old protocol** configuration

At no point does a single prefix lose reachability. If SR breaks, traffic falls back to LDP/RSVP-TE automatically.

## Migration Path 1: LDP to SR-MPLS

This is the most common SP migration happening globally (2023-2026). It's well-documented, well-tested, and supported by every major vendor.

### Phase 1: Enable SR-MPLS (Coexistence)

Enable Segment Routing in the IGP alongside LDP. Both protocols allocate labels for every prefix. The router installs **both** labels in the forwarding table.

**Key concept — Mapping Server (SRMS):**
If you can't upgrade all routers simultaneously, the SR Mapping Server bridges the gap. It advertises SR prefix-SID mappings on behalf of LDP-only nodes that don't understand SR. This allows SR-capable routers to build end-to-end SR paths even when some transit nodes only speak LDP.

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  PE-1    │─────│  P-1     │─────│  P-2     │─────│  PE-2    │
│ SR+LDP   │     │ LDP-only │     │ SR+LDP   │     │ SR+LDP   │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
                       │
                 ┌─────┴──────┐
                 │ SR Mapping │
                 │   Server   │
                 └────────────┘
                 Advertises Prefix-SIDs
                 for P-1's prefixes
```

The Mapping Server advertises SID bindings via IS-IS/OSPF prefix-SID sub-TLVs. SR-capable routers receive these and can build SR label stacks that include P-1, even though P-1 itself only speaks LDP. The LDP label for P-1 is used in the actual forwarding — the SR head-end "stitches" the SR label stack to LDP labels at the SR/LDP boundary.

### Phase 2: Prefer SR Labels

Once all routers (or enough, with Mapping Server coverage) have SR enabled, configure the IGP to **prefer SR labels over LDP labels** for the same prefix.

This is the key moment. The control plane switches from LDP-resolved paths to SR-resolved paths. Because both resolve to the same next-hop and same forwarding behavior, traffic is unaffected.

### Phase 3: Remove LDP

After validation (traffic stats, label verification, BFD sessions stable), remove LDP configuration. This eliminates:
- LDP session state (TCP sessions, keepalives)
- LDP label bindings (per-prefix, per-neighbor)
- LDP FIB entries

**Net result:** Dramatically simplified control plane. A network with 500 routers and 5,000 prefixes goes from millions of LDP label bindings to zero.

### Configuration: LDP to SR-MPLS

#### IOS-XR

**Step 1: Enable SR in IGP (alongside LDP)**

```cisco
! Enable SRGB (Segment Routing Global Block)
segment-routing
 global-block 16000 23999
!
! Enable SR in IS-IS (LDP stays running)
router isis CORE
 address-family ipv4 unicast
  segment-routing mpls
  !
 interface Loopback0
  address-family ipv4 unicast
   prefix-sid index 100
  !
 interface GigabitEthernet0/0/0/0
  address-family ipv4 unicast
   ! Adj-SIDs allocated automatically
  !
!
! LDP remains active — both protocols distribute labels
mpls ldp
 router-id 10.0.0.1
 address-family ipv4
 !
 interface GigabitEthernet0/0/0/0
 !
```

**Step 2: Enable SR Mapping Server (for LDP-only nodes)**

```cisco
! On the Mapping Server(s) — typically 2 for redundancy
segment-routing
 mapping-server
  prefix-sid-map
   address-family ipv4
    ! Map LDP-only router loopbacks to SIDs
    10.0.0.2/32 200    ! P-1 (LDP-only)
    10.0.0.5/32 500    ! P-4 (LDP-only)
   !
  !
 !
!
! Advertise mappings via IGP
router isis CORE
 address-family ipv4 unicast
  segment-routing prefix-sid-map advertise-local
 !
!
! On all SR-capable routers — receive mappings
router isis CORE
 address-family ipv4 unicast
  segment-routing prefix-sid-map receive
 !
```

**Step 3: Prefer SR over LDP**

```cisco
! Tell the router to prefer SR labels when both SR and LDP
! resolve to the same prefix
segment-routing
 global-block 16000 23999
 traffic-eng
  ! Not needed for basic SR preference — just IGP config
 !
!
router isis CORE
 address-family ipv4 unicast
  segment-routing mpls sr-prefer
  ! ^^^^^^^^^^^^^ This is the key command
  ! Without sr-prefer, LDP labels are used when available
  ! With sr-prefer, SR labels take priority
 !
```

**Step 4: Remove LDP**

```cisco
! GATE CHECK: Verify NO BGP next-hop still resolves via LDP
! If any VPN next-hop uses LDP labels, removing LDP = instant VPN outage
show bgp vpnv4 unicast labels
! Every next-hop should show SR label (16xxx range), not LDP label

! Also verify all IGP prefixes have SR labels
show cef | include "labeled SR"
! Should see "labeled SR" for all /32 loopbacks

! Only after gate checks pass:
no mpls ldp
```

#### Junos

**Step 1: Enable SR in IGP (alongside LDP)**

```junos
# Enable SRGB
routing-options {
    segment-routing {
        srgb start-label 16000 index-range 8000;
    }
}

# Enable SR in IS-IS (LDP stays running)
protocols {
    isis {
        source-packet-routing {
            srgb start-label 16000 index-range 8000;
            node-segment {
                ipv4-index 100;
            }
        }
        interface ge-0/0/0.0;
        interface lo0.0;
    }
    # LDP remains active
    ldp {
        interface ge-0/0/0.0;
    }
    mpls {
        interface ge-0/0/0.0;
    }
}
```

**Step 2: Mapping Server (Junos)**

```junos
# On the Mapping Server
protocols {
    isis {
        source-packet-routing {
            mapping-server {
                entry ldp-node-p1 {
                    prefix 10.0.0.2/32;
                    start-sid 200;
                    range 1;
                }
                entry ldp-node-p4 {
                    prefix 10.0.0.5/32;
                    start-sid 500;
                    range 1;
                }
            }
        }
    }
}
```

**Step 3: Prefer SR over LDP**

```junos
# Junos SPRING routes install in inet.3 with protocol preference 8
# LDP routes install with preference 9
# SR is NATURALLY preferred — no explicit preference config needed
#
# To verify:
#   show route table inet.3 10.0.0.2/32 detail
#   Look for "Protocol Preference: [8]" (SPRING) vs [9] (LDP)
#
# If you need LDP-SR interworking (SR ingress, LDP transit):
protocols {
    isis {
        source-packet-routing {
            ldp-stitching;  # Enables label stitching at SR/LDP boundary
        }
    }
}
# ldp-stitching allows SR head-end to use LDP labels for transit
# nodes that don't support SR (same concept as Mapping Server on IOS-XR)
```

**Step 4: Remove LDP**

```junos
# Only after full validation
delete protocols ldp
delete protocols mpls interface ge-0/0/0.0
# Keep 'protocols mpls' if needed for other features
```

## Migration Path 2: RSVP-TE to SR-TE

Less common than LDP→SR but strategically important for large SPs running thousands of RSVP-TE tunnels. The migration eliminates per-hop RSVP state while preserving traffic engineering capabilities.

### The Challenge

RSVP-TE tunnels are often deeply integrated into the network:
- BGP next-hops resolved over RSVP-TE tunnels
- Static routes pointing to tunnel interfaces
- QoS policies referencing tunnel names
- RSVP bandwidth reservations for capacity planning

You can't just delete RSVP-TE tunnels. You need to replace them with equivalent SR-TE policies that steer the same traffic with the same constraints.

### Phase 1: Deploy SR-TE Policies Alongside RSVP-TE

Create SR-TE policies that mirror existing RSVP-TE tunnels:
- Same destination (endpoint)
- Same constraints (affinity, bandwidth — though SR doesn't do bandwidth reservation natively)
- Same color community for BGP steering

Both RSVP-TE tunnels and SR-TE policies coexist. BGP next-hops resolve over RSVP-TE (higher preference).

### Phase 2: Shift BGP Steering to SR-TE

**Critical ordering:** Most RSVP-TE deployments use `autoroute announce` to install tunnel interfaces as BGP next-hops. SR-TE uses a completely different mechanism — color extended communities. You must transition the steering mechanism **before** removing tunnels:

1. Apply color extended communities to VPN routes
2. Configure ODN templates matching those colors
3. SR-TE policies are instantiated on-demand
4. Remove `autoroute announce` from RSVP-TE tunnels (traffic shifts to SR-TE)
5. Verify BGP next-hops now resolve via SR-TE color, not RSVP-TE autoroute
6. RSVP-TE tunnels become unused (safe to remove)

### Phase 3: Remove RSVP-TE Tunnels

After validation, remove RSVP-TE tunnel configuration. This eliminates:
- RSVP PATH/RESV messages (periodic refresh, per-hop state)
- RSVP-TE bandwidth reservations (link-level bookkeeping)
- ERO (Explicit Route Object) computation and maintenance

### Configuration: RSVP-TE to SR-TE

#### IOS-XR

**Step 1: Create SR-TE policies mirroring RSVP-TE tunnels**

```cisco
! Existing RSVP-TE tunnel (to be replaced)
interface tunnel-te100
 ipv4 unnumbered Loopback0
 destination 10.0.0.2
 path-option 1 explicit name PATH-TO-PE2
 autoroute announce
!

! Equivalent SR-TE policy
segment-routing
 traffic-eng
  policy TO-PE2-LOW-LATENCY
   color 100 end-point ipv4 10.0.0.2
   candidate-paths
    preference 100
     explicit segment-list SL-PE2-VIA-P1
     !
    preference 50
     dynamic
      pcep
      !
      metric
       type latency
      !
     !
    !
   !
  !
 !
```

**Step 2: Steer BGP traffic to SR-TE policies**

```cisco
! Apply color extended community to VPN routes
route-policy SET-COLOR-100
  set extcommunity color (100) additive
end-policy
!
! Apply on PE for VPN export
router bgp 65000
 vrf CUSTOMER-A
  address-family ipv4 unicast
   network 192.168.0.0/24 route-policy SET-COLOR-100
  !
 !
!
! ODN template (auto-creates SR-TE policy for any color 100 destination)
segment-routing
 traffic-eng
  on-demand color 100
   dynamic
    pcep
    !
    metric
     type latency
    !
   !
  !
 !
```

**Step 3: Validate and remove RSVP-TE**

```cisco
! Verify SR-TE policy is active and forwarding
show segment-routing traffic-eng policy color 100

! Verify BGP next-hop is resolved via SR-TE, not RSVP-TE
show bgp vrf CUSTOMER-A ipv4 unicast

! Remove RSVP-TE tunnel
no interface tunnel-te100

! Remove RSVP (after ALL tunnels are gone)
! Best practice: remove tunnel-te interfaces first, then RSVP per-interface,
! then the RSVP process. 'no rsvp' removes all RSVP config including
! authentication, bandwidth preemption, and BFD — verify nothing else depends on it.
no rsvp
```

#### Junos

**Step 1: Create SR-TE policy mirroring RSVP-TE LSP**

```junos
# Existing RSVP-TE LSP (to be replaced)
protocols {
    mpls {
        label-switched-path to-pe2 {
            to 10.0.0.2;
            primary path-via-p1;
        }
        path path-via-p1 {
            10.0.0.3 strict;
            10.0.0.2 strict;
        }
    }
    rsvp {
        interface ge-0/0/0.0;
    }
}

# Equivalent SR-TE policy
protocols {
    source-packet-routing {
        segment-list sl-pe2-via-p1 {
            hop1 label 16300;  # P-1 Prefix-SID
            hop2 label 16200;  # PE-2 Prefix-SID
        }
        sr-policy to-pe2-low-latency {
            to 10.0.0.2;
            color 100;
            segment-list sl-pe2-via-p1 {
                priority 100;
            }
        }
    }
}
```

**Step 2: Steer BGP traffic**

```junos
# Color community for traffic steering
policy-options {
    community color-100 members color:0:100;
    policy-statement set-color-100 {
        term add-color {
            then {
                community add color-100;
                accept;
            }
        }
    }
}

# Apply to VPN export
routing-instances {
    CUSTOMER-A {
        vrf-export set-color-100;
    }
}
```

**Step 3: Remove RSVP-TE**

```junos
# After validation
delete protocols mpls label-switched-path to-pe2
delete protocols rsvp
# Keep 'protocols mpls' if SR-MPLS still needs it
```

### Key Differences: RSVP-TE vs SR-TE

| Feature | RSVP-TE | SR-TE |
|---------|---------|-------|
| State location | Every hop in path | Head-end only |
| Bandwidth reservation | Native (RSVP RESV) | External (PCE/controller) |
| Path computation | CSPF or PCE | PCE or head-end |
| FRR | Facility backup (RSVP FRR) | TI-LFA (no tunnels) |
| Scale | O(n²) state growth | O(n) — head-end only |
| Provisioning | Per-tunnel config on every head-end | ODN + color (zero-touch) |
| Signaling overhead | PATH/RESV refresh every 30s per LSP | None (stateless core) |

### Important Caveat: Bandwidth Reservation

SR-TE does **not** natively support bandwidth reservation. RSVP-TE's killer feature is per-LSP bandwidth guarantees enforced at every hop. SR-TE relies on:

1. **External controllers** (PCE with bandwidth-aware CSPF) — the PCE computes paths that fit available bandwidth, but routers don't enforce it
2. **Telemetry + reactive rerouting** — monitor link utilization and adjust policies
3. **Interface QoS** — classify SR-TE traffic into queues with guaranteed bandwidth

If your network relies heavily on RSVP bandwidth reservation for capacity planning, the SR-TE migration requires a shift to controller-based bandwidth management. This is architecturally cleaner (centralized view of network state) but operationally different.

## Migration Path 3: SR-MPLS to SRv6

This is the least mature migration path but strategically the most important for forward-looking SPs. As of 2025, most operators are in the "lab evaluation" or "greenfield SRv6" phase — very few are migrating production SR-MPLS cores to SRv6.

### Prerequisites for SRv6 Migration

1. **Native IPv6 underlay** — Every router in the SR domain must have IPv6 reachability
2. **Hardware support** — ASICs must support SRH processing (see 6.4 hardware matrix)
3. **Software versions** — IOS-XR 7.3.1+, Junos 21.1R1+ (varies by platform and feature)
4. **uSID support recommended** — Without micro-SID compression, SRv6 overhead is significant

### The Interworking Gateway

During migration, you'll have SR-MPLS domains and SRv6 domains coexisting. The **interworking gateway** (IWG) sits at the boundary and translates between the two:

```
┌─────────────────────┐          ┌──────────────────────┐
│   SR-MPLS Domain    │          │    SRv6 Domain        │
│                     │          │                       │
│  PE-1 ── P-1 ── [GW-1] ──── [GW-2] ── P-3 ── PE-3  │
│                     │          │                       │
│  Labels: 16xxx      │          │  SIDs: fd00::/48      │
│  IGP: IS-IS (MPLS)  │          │  IGP: IS-IS (SRv6)   │
└─────────────────────┘          └──────────────────────┘
```

**Gateway functions:**
- **SR-MPLS → SRv6:** Receives MPLS-encapsulated packet, pops SR-MPLS labels, encapsulates in IPv6 with SRH containing SRv6 SIDs
- **SRv6 → SR-MPLS:** Receives IPv6/SRH packet, removes SRH, pushes SR-MPLS label stack
- **BGP next-hop translation:** Re-advertises VPN routes with appropriate next-hop type (MPLS label vs SRv6 SID)

### Phase 1: Deploy SRv6 in a New Domain

Don't migrate existing SR-MPLS routers. Deploy SRv6 on **new** routers or a new region:

1. Enable IPv6 on all interfaces in the new domain
2. Configure SRv6 locators and IS-IS SRv6 extensions
3. Configure VPN services with SRv6 (END.DT4, END.DT6)
4. Validate intra-domain connectivity

### Phase 2: Deploy Interworking Gateways

Gateway routers run **both** SR-MPLS and SRv6:

```cisco
! IOS-XR — Interworking Gateway
! SR-MPLS side (existing)
segment-routing
 global-block 16000 23999
!
router isis CORE
 address-family ipv4 unicast
  segment-routing mpls
  segment-routing mpls sr-prefer
 !
 interface Loopback0
  address-family ipv4 unicast
   prefix-sid index 100
 !

! SRv6 side (new)
segment-routing
 srv6
  locators
   locator MAIN
    prefix fd00:0:100::/48
   !
  !
 !
!
router isis CORE
 address-family ipv6 unicast
  segment-routing srv6
   locator MAIN
   !
 !
!

! BGP: Advertise VPN routes with both label types
router bgp 65000
 address-family vpnv4 unicast
  ! Existing SR-MPLS VPN label allocation
 !
 neighbor 10.0.0.3
  ! SRv6 domain PE — use SRv6 VPN SID
  address-family vpnv4 unicast
   encapsulation-type srv6
  !
 !
```

```junos
# Junos — Interworking Gateway
# Note: SRv6 uses 'source-packet-routing' hierarchy, not 'segment-routing'

protocols {
    isis {
        source-packet-routing {
            # SR-MPLS (existing)
            srgb start-label 16000 index-range 8000;
            node-segment {
                ipv4-index 100;
            }
            # SRv6 (new) — coexists in same stanza
            srv6 {
                locator MAIN {
                    fd00:0:100::/48;
                }
            }
        }
        interface ge-0/0/0.0;
        interface lo0.0;
    }
    bgp {
        group srv6-peers {
            neighbor 10.0.0.3 {
                family inet-vpn {
                    unicast {
                        # Junos uses extended-nexthop for SRv6 VPN signaling
                        # NOT 'encapsulation srv6' (that's IOS-XR syntax)
                        extended-nexthop;
                    }
                }
            }
        }
    }
}
```

### Phase 3: Gradual Domain Expansion

Expand the SRv6 domain by upgrading routers one at a time:

1. Upgrade router hardware/software to support SRv6
2. Enable SRv6 locator and IS-IS SRv6 extensions
3. Router becomes part of SRv6 domain
4. Gateway shifts — the boundary moves inward
5. Eventually, all routers are SRv6-capable
6. Remove SR-MPLS configuration and gateways

### Key Challenges: SR-MPLS to SRv6

| Challenge | Impact | Mitigation |
|-----------|--------|------------|
| IPv6 underlay requirement | All routers need IPv6 | Deploy dual-stack incrementally |
| MTU overhead | SRv6 adds 40-88+ bytes vs MPLS 4-16 bytes | Use uSID compression (RFC 9602), increase MTU to 9100+ |
| Hardware support | Not all ASICs support SRv6/SRH | Check vendor roadmaps, replace incompatible line cards |
| Operational complexity | Two transport technologies during migration | Automate gateway configs, use centralized controller |
| BGP VPN interworking | VPN routes need both MPLS and SRv6 advertisements | Gateway re-originates routes with correct encapsulation |
| Tooling | NOC tools may not understand SRv6 | Update monitoring, add SRv6-aware dashboards |

## Troubleshooting Migrations

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Traffic black-holed after `sr-prefer` | SR label not installed for some prefixes | Check `show cef` — missing prefix-SIDs? Mapping Server advertising? |
| L3VPN traffic lost after LDP removal | BGP next-hop resolved via LDP, not SR | Verify BGP next-hop has SR label: `show bgp vpnv4 unicast labels` |
| SR-TE policy down | Segment list uses SID not in SRGB | Verify SRGB range matches across all routers |
| RSVP-TE tunnel still preferred over SR-TE | Autoroute precedence higher than SR-TE color | Remove `autoroute announce` from RSVP-TE tunnel or set lower metric |
| SRv6 packets dropped at transit router | Router doesn't understand SRH | Check `show ipv6 segment-routing` — SRv6 enabled? Right software version? |
| Mapping Server SIDs not received | IGP not configured to advertise/receive | `prefix-sid-map advertise-local` on server, `receive` on clients |
| Label collision (traffic to wrong VRF) | SRGB too small, SID index exceeds range, overlap with dynamic labels | Standardize SRGB with headroom (8000+ labels), verify no overlap with `show mpls label range` |

### Migration Validation Checklist

Before removing the old protocol:

```
□ All prefixes have SR labels installed (show cef / show route table inet.3)
□ BGP next-hops resolve via SR (not LDP/RSVP-TE)
□ All VPN services operational (ping across VRFs)
□ BFD sessions stable on all SR paths
□ No MPLS TTL expiry or label stack depth issues
□ Traffic counters show traffic flowing on SR paths
□ RSVP-TE / LDP session count is zero (after removal)
□ Mapping Server entries correct (if used)
□ TI-LFA providing protection (replaces RSVP FRR)
□ PCE computing SR-TE paths correctly (if used)
□ Telemetry/monitoring updated for SR metrics
□ NOC runbooks updated with SR commands
```

### Methodology: Staged Validation

1. **Lab first** — Build migration scenario in EVE-NG/CML, test failure cases
2. **Single router** — Enable SR on one P-router, verify coexistence
3. **One PE pair** — Migrate two PEs, test VPN services end-to-end
4. **Region** — Expand to a full POP or region
5. **Network-wide** — Roll out to remaining routers
6. **Remove old protocol** — Only after 2+ weeks of stable SR operation

**Never skip steps 1-2.** The number of SPs who've black-holed production traffic by skipping lab validation is... non-trivial.

## War Story: The SRGB That Wasn't Big Enough

> **Scenario:** A Tier 2 SP migrating from LDP to SR-MPLS across 200 routers. Phase 1 (core P-routers) went flawlessly — 30 routers, prefix-SID indices 1-30, SRGB 16000-16999 ("we'll never need more than 1000").
>
> **The mistake:** Six months later, the PE team started deploying SR on 170 edge routers. Their prefix-SID index allocation scheme started at 1000 (offset from core). Router PE-171 got index 1170. Index 1170 exceeds the SRGB range of 1000 labels (16000-16999). IOS-XR rejected the config. The engineer, under pressure to finish the rollout, changed PE-171's SRGB to 16000-17999 to fit. "Just this one router."
>
> **What happened:** PE-171's expanded SRGB (16000-17999) overlapped with the dynamic label range on **PE-171 itself**. When PE-171 acted as a transit router, it received SR label 17100 from a neighbor (computed as PE-171's SRGB base + SID index). But PE-171 had also dynamically allocated label 17100 to an LDP binding for a different prefix. The label collision on PE-171's own forwarding table meant incoming SR-labeled packets got forwarded based on the stale LDP binding instead. Traffic for VRF CUSTOMER-A silently ended up in VRF CUSTOMER-B. No alarms. No drops. Just the wrong customer receiving the wrong traffic.
>
> **How they found it:** A customer reported receiving someone else's routing table in their VRF. The NOC spent 14 hours chasing a "BGP route leak" before an SR-aware engineer checked `show mpls label table` and found the overlap.
>
> **The fix:** Standardize SRGB across the entire SR domain with enough headroom. They moved to 16000-23999 (8000 labels) on all 200 routers. The real fix: pre-deployment validation that checks SRGB consistency and ensures SID indices fit within every router's range.
>
> **The lesson:** SRGB size planning matters. Each router advertises its own SRGB via IGP, and the ingress PE computes per-hop labels using each transit node's SRGB — so different ranges CAN work. But the moment someone's SRGB is too small and they "fix" it locally, label collisions become possible. Plan for 2-3x your expected prefix-SID count, standardize across the domain, and automate the validation.
>
> *(Composite scenario based on multiple documented migration incidents.)*

## Design Considerations

### When to Migrate

| Current State | Recommended Path | Urgency |
|--------------|-----------------|---------|
| LDP + RSVP-TE | SR-MPLS + SR-TE (skip SRv6 for now) | High — LDP/RSVP-TE are maintenance liabilities |
| LDP only | SR-MPLS | Medium — LDP works but limits future capabilities |
| RSVP-TE only (no LDP) | SR-TE + TI-LFA | Medium — RSVP-TE scale ceiling will eventually bite |
| SR-MPLS (already) | SRv6 (evaluate, don't rush) | Low — SR-MPLS is fine for 5+ years |
| Greenfield | SRv6 directly | High — no legacy to migrate, start modern |

### Migration Anti-Patterns

1. **Big bang cutover** — Don't disable LDP/RSVP-TE network-wide simultaneously. Ever. Use phased rollout.
2. **Skipping Mapping Server** — If you have LDP-only nodes, you need SRMS. "We'll upgrade everything first" is a lie you tell yourself at 2 PM and regret at 2 AM.
3. **Undersized or inconsistent SRGB** — Plan for 2-3x expected prefix-SID count. While different SRGB ranges technically work (each router advertises its own via IGP), inconsistency invites label overlap with dynamic ranges. Standardize across the domain.
4. **Ignoring MTU** — SR label stacks (and especially SRv6 SRH) add overhead. If your MTU is 1500, you will fragment. Plan for 9100+ on all core links.
5. **No rollback plan** — Always keep old protocol config ready to re-enable. `sr-prefer` is reversible; LDP deletion is not (without reconfig).
6. **Migrating without telemetry** — If you can't see per-prefix label resolution in real-time, you're flying blind. Deploy streaming telemetry before migration.

### Mapping Server Best Practices

- Deploy **at least 2** Mapping Servers for redundancy (different geographic locations)
- Mapping Server doesn't need to be in the forwarding path — it's control-plane only
- Use the same SID index for a prefix across all Mapping Servers (consistency)
- Remove Mapping Server entries as nodes upgrade to native SR
- Monitor SRMS reachability — if both Mapping Servers go down, LDP-only nodes lose SR reachability

### Timeline Expectations

| Migration | Typical Duration | Notes |
|-----------|-----------------|-------|
| LDP → SR-MPLS (100 routers) | 3-6 months | Well-understood, vendor-supported |
| RSVP-TE → SR-TE (1000 LSPs) | 6-12 months | Service mapping is the bottleneck |
| SR-MPLS → SRv6 | 12-36 months | Hardware refresh cycle drives timeline |
