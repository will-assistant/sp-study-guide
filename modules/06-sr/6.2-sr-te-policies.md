# 6.2 — SR-TE Policies

## Overview
SR-TE (Segment Routing Traffic Engineering) policies are the mechanism for source-routing traffic along explicit paths using segment lists. Unlike RSVP-TE (stateful, per-tunnel signaling), SR-TE is **stateless in the core** — the headend builds a label stack that steers the packet through a specific path, and midpoint routers just forward based on labels. This section covers how to define policies, steer traffic using color communities, integrate with a PCE, and use binding SIDs for hierarchical TE.


## Key Concepts
> **After this section you will understand:**
> - how constrained paths are computed and signaled through the network
> - how bandwidth, affinity, and path constraints affect tunnel placement
> - tradeoffs between RSVP-TE and SR-based traffic engineering models
> - how to validate tunnel liveness, path intent, and failover behavior

## Theory

### Concepts

**SR Policy Basics:**
- An **SR-TE policy** is a set of one or more **candidate paths**, each with a **segment list** (ordered list of SIDs)
- **Headend** router installs the policy and builds the label stack
- **Endpoint** is the destination (identified by loopback/prefix-SID)
- **Color** is a 32-bit identifier that allows services (BGP routes, L3VPN) to select a specific policy
- **Preference** determines which candidate path is active (highest wins)

**Policy Identifier:** `(headend, color, endpoint)`
- **Headend**: The router installing the policy (local)
- **Color**: A community value (1–4294967295) that services can reference
- **Endpoint**: The destination router (typically a loopback)

Example: Policy `(R1, color=100, endpoint=10.0.0.5)` on R1 steers traffic toward 10.0.0.5 using color 100.

**Segment List:**
- Ordered list of SIDs that define the path
- Can be:
  - **Prefix-SIDs** (loose hops — IGP shortest path between segments)
  - **Adjacency-SIDs** (strict hops — explicit link traversal)
  - **Binding-SIDs** (hierarchical TE — represent another policy)

Example segment list for R1→R2→R4→R5:
```
segment-list path-via-R2-R4
 index 10 mpls label 16002  ! R2 Prefix-SID (from SRGB)
 index 20 mpls label 24004  ! R2→R4 Adj-SID (dynamic range, platform-specific)
 index 30 mpls label 16005  ! R5 Prefix-SID (from SRGB)
```
**Note:** Adjacency-SID labels are dynamically allocated outside the SRGB. IOS-XR defaults to 24000-1048575, Junos uses 1000000-1048575 (post-17.2). Check your platform's dynamic label range.

**Candidate Paths:**
- A policy can have multiple candidate paths (primary, backup, PCE-computed)
- Only one is active at a time (highest preference)
- Paths can be:
  - **Explicit** (manually configured segment list)
  - **Dynamic** (computed by PCE or local CSPF)
  - **PCE-initiated** (pushed from controller)

**Policy Installation:**
- Policies can be configured **locally** (on-box CLI/NETCONF) or **remotely** (via PCEP from a PCE controller)
- Locally configured policies are typically for testing or static TE
- PCE-based policies are typical in production (centralized optimization)

### Traffic Steering

**How do services select an SR policy?**

1. **Color Extended Community** (BGP):
   - BGP routes carry a color extended community (type `0x030b`)
   - Format: `color:0:{color-value}`
   - When a route has color 100, the headend looks for a policy with `(local-router, 100, next-hop)`
   - If found, traffic is forwarded using the policy's segment list instead of IGP shortest path

Example:
```
BGP route: 192.0.2.0/24, next-hop 10.0.0.5, color:0:100
Policy on R1: (R1, color=100, endpoint=10.0.0.5)
Result: Traffic to 192.0.2.0/24 uses the SR-TE policy, not IGP path
```

2. **On-Demand Next-hop (ODN)**:
   - Automatic policy creation triggered by receiving a BGP route with a color
   - Headend router sees color, has no policy → asks PCE to compute one
   - PCE responds with a dynamic candidate path
   - Policy is installed on-demand, removed when no longer needed
   - Requires PCE integration (PCEP session configured)

3. **Manual Steering** (per-VRF or per-prefix):
   - Static routes or policy-based routing can explicitly steer into an SR policy
   - Less common — color-based steering is preferred for scalability

### Architecture

**Distributed vs Centralized:**

**Distributed (On-Box Config):**
- Policies configured on each headend
- Segment lists computed manually or by local CSPF (Constrained Shortest Path First)
- Pros: Simple, no external dependencies
- Cons: No global optimization, hard to scale

**Centralized (PCE-Based):**
- **PCE (Path Computation Element)**: Controller that computes optimal paths
- **PCEP (PCE Protocol)**: Signaling between router and PCE (RFC 5440, 8231, 8281)
- **BGP-LS (Link State)**: Routers advertise topology to PCE via BGP (RFC 7752, RFC 9085)
  - Each IGP domain runs BGP-LS to export link/node/prefix data
  - PCE subscribes to BGP-LS feed to learn topology, TE metrics, bandwidth
  - **CRITICAL**: Without BGP-LS, PCE cannot compute paths (blind controller)
- PCE computes paths considering constraints (latency, bandwidth, disjointness)
- PCE pushes policies to routers via PCEP
- Pros: Global optimization, centralized policy management, ODN support
- Cons: PCE is a critical dependency (but can be HA paired); requires BGP-LS deployment

**Example PCE Workflow (ODN):**
1. PE1 receives BGP route 192.0.2.0/24, next-hop PE2, color 100
2. PE1 has no policy for (PE1, 100, PE2) → triggers ODN
3. PE1 sends PCEP PCInitiate request to PCE: "I need a path to PE2 with color 100"
4. PCE computes path (e.g., PE1→P1→P3→PE2 avoiding congested P2)
5. PCE sends PCEP PCInitiate reply with segment list
6. PE1 installs policy and forwards traffic using the computed path

### Design Considerations

**When to Use SR-TE:**
- **Low-latency paths** for latency-sensitive apps (gaming, trading)
- **Disjoint paths** for primary/backup (diverse routing)
- **Bandwidth reservation** (soft bandwidth, unlike RSVP hard reservation)
- **Traffic steering** away from congested links
- **5G transport slicing** (per-slice color-based paths)

**When NOT to Use:**
- Default internet transit (shortest path is fine)
- Networks with no TE requirements (why add complexity?)
- Small networks (<10 routers) where manual TE is overkill

**Color Allocation Strategy:**
- Reserve color ranges for specific purposes:
  - **0: INVALID** (RFC 9012 reserves color 0, do not use)
  - 1-99: Reserved/internal
  - 100-199: Low latency
  - 200-299: High bandwidth
  - 300-399: Best effort disjoint
  - 400+: Customer-specific
- Document color meanings (avoid "color 100 is prod but 101 is also prod but different")
- Use YANG models to enforce color policy across network

**Binding SID:**
- A **Binding SID (BSID)** represents an SR-TE policy with a single SID
- Allows **hierarchical TE**: outer policy uses BSID, BSID expands to inner segment list
- Useful for recursive paths or policy stitching
- **Label allocation**: BSID uses dynamic label range (outside SRGB) or explicit static label
  - IOS-XR: Auto-allocated from dynamic range (default) or `binding-sid mpls label <value>`
  - Junos: Requires explicit `binding-sid-rules` configuration
- Example: Policy A (color 100, BSID 30000) → Policy B references BSID 30000 in its segment list
- **Signaled via PCEP/BGP**: BSID is a local label on the head-end. Remote routers learn about BSIDs via PCEP (controller-mediated) or static configuration, not IGP advertisement

**Vendor Differences:**
- **Cisco IOS-XR**: Full SR-TE support since XR 6.1.2, PCE integration mature
- **Juniper Junos**: SR-TE support since 17.2, `spring-traffic-engineering` config stanza
- **Nokia SR-OS**: SR-TE via `configure router segment-routing traffic-engineering`
- **Arista EOS**: Limited SR-TE support (basic SR-MPLS forwarding, no native PCE client)
- **PCE options**: Cisco NSO/WAE, Juniper Paragon Pathfinder, open-source ODL/ONOS

**Gotchas:**
- **MTU**: Each SID in the stack adds 4 bytes (label). A 10-SID stack = 40 bytes overhead. Ensure MTU is configured for worst case.
- **Loop prevention**: If segment list has cycles (bad config), packets loop. Validate paths before deployment.
- **Fallback behavior**: If no policy matches, traffic falls back to IGP. Test this to avoid surprise blackholes.
- **PHP vs Explicit-Null**: Policies must align with penultimate hop behavior (Section 6.1). PHP pops the last SID; explicit-null sends label 0.
- **SR-TE + RSVP-TE coexistence**: If both exist to the same destination, SR-TE typically wins (platform-dependent). Check `show route <prefix>` to see which is installed.
- **ODN queue limits**: Most platforms queue up to 100 pending ODN requests (IOS-XR default). Beyond this, new requests are dropped. Monitor with `show segment-routing traffic-eng pcc ipv4 peer`.

## Configuration

### Cisco IOS-XR

**On-Box Policy (Manual Segment List):**
```cisco
segment-routing
 traffic-eng
  !
  segment-list name path-via-P1-P3
   index 10 mpls label 16002  ! P1 Prefix-SID
   index 20 mpls label 16004  ! P3 Prefix-SID
  !
  policy PE1-to-PE2-lowlat
   color 100 end-point ipv4 10.0.0.5
   candidate-paths
    preference 100
     explicit segment-list path-via-P1-P3
    !
   !
  !
 !
!
```

**PCE-Based Policy (ODN Enabled):**
```cisco
! PCEP client config (with authentication)
segment-routing
 traffic-eng
  pcc
   source-address ipv4 10.0.0.1
   pce address ipv4 192.0.2.100
    precedence 10
    password encrypted <hash>
   !
  !
  on-demand color 100
   dynamic
    pcep
    !
   !
  !
 !
!

! BGP-LS config (required for PCE topology learning)
router isis CORE
 distribute link-state instance-id 100
!
router bgp 65000
 address-family link-state link-state
  neighbor 192.0.2.100 activate
 !
!
```
**Key Point:** BGP-LS requires TWO pieces:
1. **IGP export** (`distribute link-state` in IS-IS/OSPF) — makes IGP export topology into BGP-LS
2. **BGP neighbor** to PCE — sends BGP-LS NLRI to PCE

Without #1, BGP-LS is empty. Without #2, PCE learns nothing.

**BGP Color Steering (Services Using Policy):**
```cisco
! Define color extended community
extcommunity-set color color-100
  100
end-set
!
route-policy SET-COLOR-100
  set extcommunity color color-100
end-policy
!
router bgp 65000
 address-family vpnv4 unicast
  vrf CUSTOMER-A
   route-target import 65000:100
   route-target export 65000:100
  !
 !
 vrf CUSTOMER-A
  address-family ipv4 unicast
   redistribute connected route-policy SET-COLOR-100
  !
 !
!
```

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| **max-segment-list-depth** | Platform-dependent (3-12) | 10 | Balance expressiveness vs MTU overhead |
| **preference** | N/A | 100 (explicit), 200 (PCE dynamic) | PCE paths should override manual fallback (higher = higher priority) |
| **pcep precedence** | N/A | Primary PCE=10, backup PCE=20 | **Lower number = higher priority** (opposite of preference!) |
| **delegation-cleanup-timeout** | 60s (Junos) | 120s | Avoid premature policy removal on PCE flap |
| **pce-initiated** | Disabled | Enabled (production) | Allow PCE to push new policies without manual config |
| **dynamic-label-range** | 24000-1048575 (XR), 1000000-1048575 (Junos) | **Verify no overlap with LDP/RSVP** | Conflicts cause label allocation failures |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Policy installed but traffic not using it | No color community on BGP route | Add color community in export policy |
| Policy shows `Operational: down` | Segment list includes unreachable SID | Verify all SIDs are in LFIB (`show mpls forwarding`) |
| ODN not triggering | No PCEP session to PCE | Check `show pcep session`, verify PCE IP reachable |
| PCEP session flapping | Keepalive timeout too aggressive | Increase keepalive (default 30s) to 60s |
| Traffic loops | Segment list has duplicate SIDs or cycles | Review segment list, ensure forward progress |
| MTU issues | Too many SIDs in stack | Reduce segment list depth or increase MTU |
| Policy not installed after PCEP push | Preference conflict with local policy | Lower local policy preference or delete it |
| Binding SID not resolving | BSID not signaled to controller | Check SR-TE policy config and PCEP session on headend |

### Methodology

**Policy Not Working (Traffic Still Using IGP Path):**

1. **Verify policy operational state:**
   ```
   show segment-routing traffic-eng policy color <color>
   ```
   - Must show `Operational: up`
   - If down, check segment list SIDs

2. **Verify BGP route has color community:**
   ```
   show bgp vrf <vrf> ipv4 unicast <prefix> detail
   ```
   - Look for `Extended Community: Color:0:<color>`
   - If missing, check export policy on remote PE

3. **Verify policy forwarding:**
   ```
   show segment-routing traffic-eng forwarding policy color <color>
   ```
   - Should show outbound interface and label stack
   - If empty, policy is not programmed in FIB

4. **Check for ECMP splitting:**
   - If there's an IGP path AND SR-TE policy, traffic may load-balance
   - Some platforms prefer IGP (check vendor docs)

5. **Verify endpoint matches next-hop:**
   - Policy endpoint must exactly match BGP next-hop
   - Common error: policy endpoint is loopback0, BGP next-hop is loopback1

6. **Validate before deployment:**
   - IOS-XR: `commit confirm 5` (auto-rollback after 5 min if not confirmed)
   - Junos: `commit check` + `commit confirmed 5`
   - Test segment lists with `ping mpls sr-policy` before pushing to production

**PCEP Session Issues:**

1. **Check session state:**
   ```
   show segment-routing traffic-eng pcc ipv4 peer
   ```
   - Must show `State: Up`

2. **Check reachability:**
   ```
   ping 192.0.2.100 source 10.0.0.1
   ```
   - PCE must be reachable from source address

3. **Check PCE logs:**
   - Look for authentication failures (if using PCEP MD5)
   - Check for capability mismatch (stateful, PCE-initiated)

4. **Enable PCEP debugging:**
   ```
   debug segment-routing traffic-eng pcep all
   ```
   - Watch for PCEP Open, Keepalive, PCRpt, PCUpd messages

### War Stories

**Scenario: ODN Cascade Failure**

**Setup:** Large SP network with 200 PE routers, centralized PCE managing 10,000 SR-TE policies. ODN enabled for all L3VPN services. PCE and routers use color 100 for low-latency, color 200 for high-bandwidth.

**Failure:** Network maintenance requires shutting down core router P3 for 30 minutes. P3 is on the path for ~2,000 ODN policies. When P3 goes down:
1. 2,000 policies go `Operational: down` simultaneously
2. 2,000 PE routers send PCEP PCUpd messages to PCE requesting new paths
3. PCE CPU spikes to 100% trying to recompute 2,000 paths
4. PCE crashes and restarts (out of memory)
5. During PCE restart (~90 seconds), all new ODN requests queue
6. PE routers hit ODN queue limit (100 pending), start dropping new requests
7. New L3VPN services fail to activate (no policy created)
8. Existing services fall back to IGP (fine, but defeats TE)

**Root Cause:** PCE not sized for simultaneous bulk recomputation. No rate limiting on PCEP PCUpd.

**Lessons:**
1. **Rate-limit ODN requests** at PE: max 10 per second per PE
2. **PCE redundancy**: Deploy HA PCE pair, each handles 50% of policies
3. **Policy pre-computation**: PCE maintains backup paths for all colors (no compute during failure)
4. **Graceful degradation**: If ODN fails, use IGP path (better than blackhole)
5. **Change control**: For planned maintenance, pre-compute backup paths and push to PEs before change window

**Fix Applied:**
- Upgraded PCE from 16 GB RAM to 64 GB RAM
- Enabled PCE state sync between primary/secondary (stateful PCE redundancy)
- Configured per-PE ODN rate limiting (10 req/sec)
- Added monitoring: alert if PCE CPU >60% or pending PCUpd >100

**Color Community Format:**
- Type: `0x030b` (Opaque Extended Community, Color sub-type)
- Value: `color:0:<32-bit-number>`
- Example: `color:0:100` = decimal 100

**Policy Priority:**
- Highest **preference** wins (per candidate path)
- Typical values: Explicit=100, PCE dynamic=200, ODN=300
- **Higher number = higher priority** (300 > 200 > 100)
- If you want PCE paths to override manual config, give PCE higher preference
