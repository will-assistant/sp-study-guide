# 6.4 SRv6 Fundamentals

## Overview

SRv6 (Segment Routing over IPv6) is the next evolution of Segment Routing — same source routing principles as SR-MPLS, but using IPv6 extension headers instead of MPLS labels. The promise: all the benefits of SR (stateless core, source routing, traffic engineering) without requiring an MPLS data plane.

For greenfield IPv6 networks or SPs looking to simplify their stack, SRv6 is compelling. For existing MPLS deployments, SR-MPLS is usually the better choice (hardware support, operational familiarity, no IPv6 migration required).

This section covers SRv6 architecture, SID structure, the Segment Routing Header (SRH), basic network programming functions, and IOS-XR/Junos configuration.

## Theory: SRv6 Architecture

### The Problem SRv6 Solves

SR-MPLS works great, but it requires:
1. **MPLS data plane** — Every device in the path must support MPLS forwarding
2. **Label management** — SRGB coordination, label allocation, label stack depth limits
3. **Hardware support** — Not all platforms support deep label stacks or SR-MPLS features

SRv6 sidesteps these issues:
- **Pure IPv6** — No MPLS required; uses IPv6 extension headers
- **128-bit SIDs** — Vastly larger address space than 20-bit MPLS labels
- **Simplified operations** — One protocol stack (IPv6), not two (IPv6 + MPLS)

**Trade-off:**
SRv6 has higher overhead (IPv6 header + SRH extension header vs MPLS label stack) and requires IPv6-capable hardware. It's a bet on IPv6 as the long-term future.

**Critical requirement:** SRv6 REQUIRES a native IPv6 underlay. You cannot run SRv6 over an IPv4-only core (unlike SR-MPLS, which works on any IP transport). If your network is IPv4-only today, migrating to SRv6 means migrating to dual-stack or IPv6-only first.

---

### SRv6 SID Structure

An SRv6 **SID** is a 128-bit IPv6 address with three logical components:

```
┌─────────────────────┬───────────────┬───────────────┐
│      Locator        │   Function    │   Argument    │
│   (network prefix)  │   (behavior)  │  (optional)   │
└─────────────────────┴───────────────┴───────────────┘
```

**Locator** (typically /48 or /64):
- Identifies the node or service (e.g., `fc00:0:1::/48` = R1's SRv6 block)
- Routable via the IGP (advertised as a standard IPv6 prefix)

**Function** (16 bits, typically):
- Defines the SID's behavior (END, END.X, END.DT4, etc.)
- This is the "network programming" part

**Argument** (optional):
- Additional data for the function (e.g., VRF table ID, interface index)

**Example SID:**
```
fc00:0:1:e000::
  fc00:0:1::/48  → Locator (R1's SRv6 block)
  e000           → Function (END = basic forwarding)
  ::             → No argument
```

**Comparison to SR-MPLS:**
| Aspect | SR-MPLS | SRv6 |
|--------|---------|------|
| SID size | 20-bit label | 128-bit IPv6 address |
| SID allocation | SRGB (16000–23999 typical) | IPv6 prefix block (fc00::/48 typical) |
| Forwarding | MPLS label swap | IPv6 routing + SRH processing |
| Overhead | 4 bytes per label | 40-byte IPv6 header + 8+ bytes per SID in SRH |
| Hardware support | Widespread (ASR9k, MX, 7750) | Emerging (NCS5500, ASR9k-X64, PTX) |

### IPv6 Segment Routing Header (SRH)

The **SRH** is an IPv6 extension header (type 43, per RFC 8754) that carries the segment list — the ordered list of SIDs the packet should traverse.

**Packet structure:**
```
┌──────────────────────┐
│ IPv6 Base Header     │  (40 bytes)
├──────────────────────┤
│ Segment Routing Hdr  │  (8 bytes + N×16 bytes)
│  - Next Header       │
│  - Hdr Ext Len       │
│  - Routing Type = 4  │
│  - Segments Left     │
│  - Last Entry        │
│  - Flags, Tag        │
│  - Segment List[0]   │  (128-bit SID)
│  - Segment List[1]   │
│  - ...               │
│  - Segment List[N]   │
├──────────────────────┤
│ Upper Layer Header   │  (TCP, UDP, etc.)
│ Payload              │
└──────────────────────┘
```

**Key fields:**
- **Segments Left**: How many SIDs remain in the path (decremented at each SRv6 node)
- **Last Entry**: Index of the final SID in the segment list
- **Segment List**: Ordered array of 128-bit SIDs (indexed from 0)

**Forwarding behavior:**
1. Node receives packet with SRH
2. Checks `Segments Left` (SL):
   - **If SL > 0:**
     - Decrement `Segments Left` by 1
     - Copy `Segment List[new SL]` to IPv6 Destination Address
     - Forward packet based on new DA
   - **If SL == 0:**
     - Execute the endpoint function specified in the current SID
     - For END: just forward (no SRH update)
     - For END.DT4/DT6: decapsulate and route in VRF
     - For END.X: forward to specific next-hop

**Example:**
```
Path: R1 → R3 → R5 → R7
Segment List = [fc00:0:7::, fc00:0:5::, fc00:0:3::]  ← REVERSE order (destination first per RFC 8754)
  Segments Left = 2 (Last Entry index)
  IPv6 DA = Segment List[2] = fc00:0:3::

R3 (first hop):
  Match fc00:0:3:: → SRv6 END function
  Decrement Segments Left (2 → 1)
  IPv6 DA ← Segment List[1] = fc00:0:5::
  Forward toward fc00:0:5::

R5 (second hop):
  Match fc00:0:5:: → SRv6 END function
  Decrement Segments Left (1 → 0)
  IPv6 DA ← Segment List[0] = fc00:0:7::
  Forward toward fc00:0:7::

R7 (final hop):
  Match fc00:0:7:: → SRv6 END.DT4 function
  Segments Left = 0 → execute endpoint behavior (decapsulate, route in VRF)
  SRH removed during decapsulation (inner packet forwarded)
```

**Note:** Segment lists are ordered REVERSE to the forwarding path (RFC 8754). If you want traffic to go R1→R3→R5→R7, the segment list is [R7, R5, R3].

**SRH handling at final destination:**
- For **END** behavior: SRH remains in packet (forwarded as-is)
- For **END.DT4/DT6/DX4/DX6** (decap behaviors): SRH is removed, inner packet extracted and forwarded

### SRv6 Network Programming Functions

SRv6 defines a set of **endpoint behaviors** (functions) that determine how a node processes a SID. These are the building blocks of SRv6 forwarding.

**RFC 8986 defines the core functions:**

| Function | Behavior | Use Case |
|----------|----------|----------|
| **END** | Basic endpoint — decrement SL, update DA, forward | Transit node (like Prefix-SID in SR-MPLS) |
| **END.X** | Endpoint with Layer-3 cross-connect — forward to specific next-hop | Strict forwarding (like Adj-SID in SR-MPLS) |
| **END.DT4** | Decapsulation and IPv4 table lookup | L3VPN egress (IPv4 VRF route lookup) |
| **END.DT6** | Decapsulation and IPv6 table lookup | L3VPN egress (IPv6 VRF route lookup) |
| **END.DT46** | Decapsulation and dual-stack table lookup | L3VPN egress (IPv4 or IPv6 VRF) |
| **END.DX4** | Decapsulation and IPv4 cross-connect | L3VPN egress with specific next-hop |
| **END.DX6** | Decapsulation and IPv6 cross-connect | L3VPN egress with specific next-hop |
| **END.B6.ENCAPS** | Binding SID — encapsulate in outer IPv6 header | SR Policy binding (like BSID in SR-MPLS) |

**Example: END function**
```
R3 receives packet:
  IPv6 DA = fc00:0:3:e000::  (R3's END SID)
  SRH Segments Left = 2

R3 processing:
  1. Match fc00:0:3:e000:: in local SID table → behavior = END
  2. Decrement Segments Left (2 → 1)
  3. Copy Segment List[1] to IPv6 DA
  4. Forward based on IPv6 DA
```

**Example: END.DT4 function**
```
R7 receives packet:
  IPv6 DA = fc00:0:7:d400::  (R7's END.DT4 SID for VRF "cust-a")
  SRH Segments Left = 0
  Inner payload = IPv4 packet (dest 10.100.1.5)

R7 processing:
  1. Match fc00:0:7:d400:: → behavior = END.DT4, arg = VRF "cust-a"
  2. Remove IPv6 + SRH headers (decapsulation)
  3. Look up 10.100.1.5 in VRF "cust-a" routing table
  4. Forward inner IPv4 packet
```

### SRv6 vs SR-MPLS: When to Use Which

| Criteria | SR-MPLS | SRv6 |
|----------|---------|------|
| **Existing MPLS network** | ✅ Best choice | ⚠️ Requires parallel IPv6 deployment |
| **Greenfield IPv6 network** | ⚠️ Requires MPLS addition | ✅ Native fit |
| **Hardware support** | ✅ Mature (ASR9k, MX, NCS, PTX, 7750) | ⚠️ Emerging (NCS5500, ASR9k-X64, PTX10k, limited MX) |
| **Label/SID space** | ⚠️ Limited (20-bit labels, SRGB conflicts possible) | ✅ Vast (128-bit SIDs, no collision risk) |
| **Packet overhead** | ✅ Low (4 bytes/label) | ⚠️ High (40-byte IPv6 hdr + 8+ bytes/SID) |
| **MTU impact** | ✅ Minimal | ⚠️ Significant (need jumbo frames or fragmentation) |
| **Operational complexity** | ✅ Well-understood, mature tooling | ⚠️ Newer, fewer engineers with experience |
| **Interworking with legacy** | ✅ LDP/RSVP coexistence well-defined | ⚠️ Limited (requires SR-MPLS <-> SRv6 gateway) |
| **Vendor interoperability** | ✅ Cisco/Juniper/Nokia/Huawei all support SR-MPLS | ⚠️ Limited (fewer vendors, less mature implementations) |
| **Future-proofing** | ⚠️ Assumes MPLS longevity | ✅ Aligns with IPv6-only future |

**Bottom line:**
- **2026 reality:** SR-MPLS is production-ready, widely deployed, safe choice for most SPs
- **Long-term bet:** SRv6 is the eventual destination, but you're an early adopter if you deploy it today

## Configuration

**⚠️ Platform Warning:** These configs are vendor-specific. Do NOT copy IOS-XR config to a Junos router (or vice versa). Syntax errors will cause commit failures; platform mismatch may cause undefined behavior.

### IOS-XR: SRv6 Basics

**Enable SRv6 globally:**
```cisco
segment-routing
 srv6
  encapsulation
   source-address fc00:0:1::1
  !
  locators
   locator MAIN
    micro-segment behavior unode psp-usd
    prefix fc00:0:1::/48
   !
  !
 !
!
```

**Configure IGP to advertise SRv6 locator:**

**IS-IS:**
```cisco
router isis CORE
 net 49.0001.0000.0000.0001.00
 address-family ipv6 unicast
  segment-routing srv6
   locator MAIN
  !
 !
!
```
*Note: Unlike SR-MPLS, SRv6 does NOT use `prefix-sid index` — the locator itself is advertised automatically in IS-IS SRv6 TLVs.*

**OSPFv3** (SRv6 requires OSPFv3, not OSPFv2):
```cisco
router ospfv3 1
 router-id 1.1.1.1
 address-family ipv6 unicast
  segment-routing srv6
   locator MAIN
  !
 !
!
```

**END.DT4 SID for L3VPN (VRF egress):**
```cisco
vrf cust-a
 address-family ipv4 unicast
  segment-routing srv6
   locator MAIN
   alloc mode per-vrf
  !
 !
!
```

**SR Policy with SRv6 (explicit path):**
```cisco
segment-routing
 traffic-eng
  srv6
   source-address fc00:0:1::1
  !
  policy R1-to-R7
   srv6
    locator MAIN binding-sid dynamic behavior ub6-insert-reduced
   !
   color 100 end-point fc00:0:7::1
   candidate-paths
    preference 10
     explicit segment-list R1-R3-R5-R7
      index 10 sid fc00:0:3:e000::
      index 20 sid fc00:0:5:e000::
      index 30 sid fc00:0:7:e000::
     !
    !
   !
  !
 !
!
```

**Key IOS-XR knobs:**
| Parameter | Default | Notes |
|-----------|---------|-------|
| `micro-segment behavior unode psp-usd` | Disabled | Enables **uSID** (micro-segment ID = compressed SIDs, multiple logical SIDs in one 128-bit address) with PSP (Penultimate Segment Pop) and USD (Ultimate Segment Decapsulation) — see Module 6.5 |
| `alloc mode per-vrf` | per-ce | Allocate one END.DT SID per VRF (not per CE) |
| `binding-sid dynamic behavior ub6-insert-reduced` | — | Binding SID behavior: **ub6-insert-reduced** = reduced SRH (omits first segment from SRH if it matches DA); **ub6-insert-encaps** = full encapsulation (all segments in SRH). Reduced saves 16 bytes per policy. |

### Junos: SRv6 Basics

**Enable SRv6 globally:**
```junos
set routing-options source-packet-routing srv6 locator MAIN fc00:0:1::/48
set routing-options source-packet-routing srv6 locator MAIN end-dt46-sid
```

**Configure IS-IS to advertise SRv6 locator:**
```junos
set protocols isis source-packet-routing srv6 locator MAIN
set protocols isis interface lo0.0 ipv6-adjacency-segment advertise-sids
```

**END.DT4 SID for L3VPN (VRF egress):**
```junos
set routing-instances cust-a instance-type vrf
set routing-instances cust-a routing-options source-packet-routing srv6 locator MAIN end-dt4-sid
```

**SR Policy with SRv6 (PCEP or local config):**
```junos
set protocols source-packet-routing segment-list R1-R3-R5-R7 srv6
set protocols source-packet-routing segment-list R1-R3-R5-R7 hop 1 srv6-sid fc00:0:3:e000::
set protocols source-packet-routing segment-list R1-R3-R5-R7 hop 2 srv6-sid fc00:0:5:e000::
set protocols source-packet-routing segment-list R1-R3-R5-R7 hop 3 srv6-sid fc00:0:7:e000::

set protocols source-packet-routing source-routing-path R1-to-R7 to fc00:0:7::1
set protocols source-packet-routing source-routing-path R1-to-R7 color 100
set protocols source-packet-routing source-routing-path R1-to-R7 primary R1-R3-R5-R7
```

**Key Junos knobs:**
| Parameter | Default | Notes |
|-----------|---------|-------|
| `end-dt46-sid` | — | Dual-stack VRF decap (IPv4 + IPv6) |
| `end-dt4-sid` | — | IPv4-only VRF decap |
| `end-dt6-sid` | — | IPv6-only VRF decap |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| SRv6 SID not in `show srv6 sid` | Locator not configured or not active | Check `segment-routing srv6 locators` config; verify IGP is advertising locator |
| Packets dropped at SRv6 node | SID not in local SID table | Verify SID is allocated (`show srv6 sid`); check for typo in segment list |
| END.DT4 decap fails | VRF not bound to SRv6 locator OR malformed packet (no inner payload) | Configure `vrf <name> segment-routing srv6 locator <loc> alloc mode per-vrf`; check packet capture for inner IP header |
| END.DT4 drops packets silently | Malformed SRv6 packet (no inner IPv4) | Router drops packet, may send ICMP Parameter Problem (check `show logging`); verify ingress encapsulation |
| SRH header malformed | MTU exceeded (packet fragmented) | Increase interface MTU by 80+ bytes OR enable MSS clamping; remember IPv6 fragmentation is END-TO-END (source must fragment, not routers) |
| Traffic not using SR Policy | Color not matching BGP community | Verify `extcommunity-set color` and route-policy applying it |
| Locator not advertised in IS-IS | IGP config missing `segment-routing srv6 locator` | Add `router isis <tag> address-family ipv6 unicast segment-routing srv6 locator <name>` |

### Troubleshooting Methodology

**1. Verify SRv6 is enabled globally:**
```
IOS-XR: show run segment-routing srv6
Junos:  show configuration routing-options source-packet-routing srv6
```

**2. Check local SID table:**
```
IOS-XR: show segment-routing srv6 sid
Junos:  show spring srv6 locator detail
```
- Verify expected SIDs are present (END, END.DT4, etc.)
- Check SID ownership (IS-IS, BGP, static)

**3. Verify locator is advertised in IGP:**
```
IOS-XR: show isis database verbose | include SRv6
Junos:  show isis database extensive | match SRv6
```
- Look for SRv6 Locator TLV in LSP
- Verify locator prefix matches configuration

**4. Check IPv6 routing for remote locators:**
```
IOS-XR: show route fc00:0:<remote-node>::/48
Junos:  show route fc00:0:<remote-node>::/48
```
- Ensure IGP installed route to remote SRv6 locator
- Verify next-hop is correct

**5. Verify CEF/FIB entry for specific SID:**
```
IOS-XR: show cef fc00:0:3:e000:: detail
Junos:  show route forwarding-table destination fc00:0:3:e000:: extensive
```
- Check forwarding next-hop matches expected path
- For END.X SIDs, verify outgoing interface

**6. For VPN services (END.DT4/DT6), check VRF bindings:**
```
IOS-XR: show bgp vpnv4 unicast vrf <name>
Junos:  show route table <vrf>.inet.0 protocol bgp
```
- Verify VRF routes have correct next-hop (END.DT SID on remote PE)
- Check for `encap=srv6` in route attributes

**7. For SR Policy, verify policy is installed:**
```
IOS-XR: show segment-routing traffic-eng policy color 100
Junos:  show spring-traffic-engineering lsp color 100
```
- Check policy state (UP/DOWN)
- Verify segment list matches intended path
- Check for PCEP session if using dynamic computation

**8. MTU check (SRv6 overhead is significant):**
```
IOS-XR: show interfaces <name> | include MTU
Junos:  show interfaces <name> | match MTU
```
- IPv6 base header = 40 bytes
- SRH = 8 bytes + (16 bytes × number of SIDs)
- Example: 3-SID path = 40 + 8 + 48 = 96 bytes overhead
- If original MTU = 1500, payload max = 1404 bytes
- **Best practice:** Set interface MTU to 1600+ for SRv6

**9. Packet capture (if all else fails):**
```
IOS-XR: monitor capture ipv6 start
Junos:  monitor traffic interface <name> extensive
```
- Look for SRH extension header (Next Header = 43)
- Verify Segments Left decrements correctly
- Check for ICMP Packet Too Big (fragmentation issue)
- **IPv6 fragmentation note:** Unlike IPv4, IPv6 routers do NOT fragment packets. If MTU exceeded, router sends ICMP Packet Too Big back to source. Source must fragment (or reduce payload). This is why MTU planning is CRITICAL for SRv6.

## Design Considerations

### When to Use SRv6

**Good fit:**
- Greenfield IPv6 network (no MPLS legacy)
- Long-term IPv6-only strategy
- Need for massive SID space (128-bit vs 20-bit labels)
- Willing to be an early adopter (accept risks of emerging tech)

**Bad fit:**
- Existing MPLS network with SR-MPLS already deployed
- Hardware doesn't support SRv6 (check platform support matrices)
- Need maximum performance (SRv6 overhead > SR-MPLS overhead)
- Operational team not comfortable with IPv6 troubleshooting

---

### SRv6 Locator Design

**Locator size:**
- **/48** is typical (allows 65,536 /64 function blocks)
- **/64** is possible but limits growth (only 65,536 SIDs in the entire locator)
- **/40** or larger is sometimes used for very large deployments
- **Minimum:** Platform-dependent (IOS-XR supports /32 to /96; Junos /48 to /64 typical)
- **Best practice:** Allocate /48 per node, subdivide into /64 blocks per function type

**Example allocation:**
```
fc00:0:1::/48      → R1's SRv6 block
  fc00:0:1:e000::/64  → END functions
  fc00:0:1:d400::/64  → END.DT4 functions (VRFs)
  fc00:0:1:d600::/64  → END.DT6 functions (VRFs)
  fc00:0:1:b000::/64  → Binding SIDs (SR Policies)
```

**SID allocation conflict risk:**
Unlike SR-MPLS (where SRGB coordination prevents SID conflicts), SRv6 has no built-in protection. If two nodes accidentally allocate the same locator prefix, traffic will be blackholed. **Best practice:** Use IPAM (IP Address Management) or centralized automation to assign locators, not manual configuration.

**Locator allocation:**
- Use **ULA space (fd00::/8)** for private networks (RFC 4193; fc00::/8 requires registry assignment that doesn't exist in practice)
- Use **GUA space (2001::/3)** for public/Internet-facing SRv6
- **Do NOT use link-local (fe80::/10)** — not routable
- **Do NOT use fc00::/8** — technically requires central registry per RFC 4193, but no registry exists; use fd00::/8 instead

---

### MTU Planning

SRv6 has significant overhead:
- **IPv6 base header:** 40 bytes
- **SRH:** 8 bytes + (16 bytes per SID)

**Example calculations (full 128-bit SIDs, NOT compressed uSIDs):**
| Segment List Size | SRH Size | Total Overhead | Max Payload (1500 MTU) |
|-------------------|----------|----------------|------------------------|
| 1 SID | 8 + 16 = 24 | 40 + 24 = 64 | 1436 |
| 3 SIDs | 8 + 48 = 56 | 40 + 56 = 96 | 1404 |
| 6 SIDs | 8 + 96 = 104 | 40 + 104 = 144 | 1356 |

*Note: With uSID compression (Module 6.5), overhead is much lower — multiple logical SIDs can fit in one 128-bit address.*

**Recommendations:**
- **Option 1:** Increase interface MTU to 1600+ (eliminates fragmentation for typical 3-SID paths)
- **Option 2:** Enable MSS clamping for TCP (reduces payload to fit in 1500-byte MTU)
- **Option 3:** Use uSID compression (Module 6.5) to reduce SID count

**IOS-XR MTU config:**
```cisco
interface TenGigE0/0/0/0
 mtu 9000
 ipv6 mtu 9000
!
```

**Junos MTU config:**
```junos
set interfaces ge-0/0/0 mtu 9000
set interfaces ge-0/0/0 unit 0 family inet6 mtu 9000
```

- **Nokia:** 7750 SR-1 (limited, FP4/FP5 only)
- **Huawei:** NE9000, NetEngine 8000

**NOT supported (as of 2026):**
- Cisco ASR9000 (legacy line cards)
- Cisco NCS540 (XR6 platform)
- Juniper MX960/480/240 (Trio chipset)
- Most campus/enterprise routers

**Before deploying SRv6:**
1. Check vendor feature navigator or TAC (support matrices change frequently)
2. Verify platform + line card + IOS-XR/Junos version support SRv6
3. Test in lab — not all advertised features work reliably
4. **Critical:** Some platforms advertise SRv6 support but only for specific functions (e.g., END + END.DT4 only, no END.X or complex behaviors)
5. **Performance warning:** SRv6 without hardware (ASIC) offload is CPU-intensive. SID table lookups, SRH processing, and decapsulation can saturate CPU on high-throughput links (10G+). Always verify hardware datasheet confirms SRv6 acceleration before production deployment.

### Interworking with SR-MPLS

If you have an existing SR-MPLS core and want to add SRv6 edges (or vice versa), you need a **gateway node** that translates between the two data planes.

**Gateway functions (RFC 9252):**
- **T.Encaps** — SR-MPLS to SRv6 (encapsulate MPLS packet in IPv6 + SRH)
- **T.Decaps** — SRv6 to SR-MPLS (decapsulate IPv6, expose MPLS label stack)

**Example use case:**
- Core network: SR-MPLS (existing deployment)
- Edge network: SRv6 (new IPv6-only DCs)
- Gateway: R5 runs both SR-MPLS and SRv6, translates between them

**IOS-XR gateway config (simplified):**
```cisco
segment-routing
 srv6
  locators
   locator GATEWAY
    prefix fc00:0:5::/48
   !
  !
 !
!

segment-routing
 traffic-eng
  policy MPLS-to-SRv6
   color 200 end-point fc00:0:7::1
   candidate-paths
    preference 10
     explicit segment-list hybrid
      index 10 mpls label 16003   ! SR-MPLS segment
      index 20 srv6 sid fc00:0:7:e000::  ! SRv6 segment
     !
    !
   !
  !
 !
!
```

This is advanced — most SPs avoid interworking by picking one data plane (SR-MPLS or SRv6) and sticking with it.

## War Story: SRv6 MTU Black Hole

**Composite scenario based on common SRv6 deployment issues**  
**Date:** October 2024  
**Network:** Large European ISP, greenfield IPv6 SRv6 core  
**Impact:** ~15% packet loss on customer traffic for 8 hours  

**Setup:**
- Core network: SRv6, all interfaces MTU 1500 (default)
- SR Policies using 6-SID paths (144 bytes overhead)
- Customer traffic: mix of TCP and UDP

**What happened:**
1. SRv6 policy installed with 6-SID segment list (40 + 8 + 96 = 144 bytes overhead)
2. Customer sent 1500-byte UDP packets → total 1644 bytes on the wire
3. First-hop router fragmented the packet (IPv6 fragmentation)
4. Middle router had firewall rule blocking IPv6 fragments (security policy)
5. Packets silently dropped, no ICMP Packet Too Big sent back (firewall config issue)
6. UDP apps saw ~15% loss (only large packets affected), TCP was fine (MSS negotiation limited payload)

**Root cause:**
- MTU not accounted for during SRv6 deployment
- Firewall team didn't know SRv6 was going live (no change coordination)
- No end-to-end testing with realistic packet sizes

**Fix:**
- Immediate: Reduced segment list to 3 SIDs (96-byte overhead → fits in 1500 MTU for most packets)
- Short-term: Increased core interface MTU to 9000 (jumbo frames)
- Long-term: Enabled uSID compression (Module 6.5) to reduce SID count
- Process: Added SRv6 MTU check to pre-deployment validation

**Lesson:**
SRv6 overhead is NOT like MPLS overhead. A 4-label SR-MPLS stack is 16 bytes; a 4-SID SRv6 path is 104 bytes. You MUST plan MTU accordingly or you WILL have a bad day.

## Summary

SRv6 is elegant in theory: all the benefits of Segment Routing, pure IPv6, no MPLS required. In practice (as of 2026), it's still emerging:
- Limited hardware support
- Higher overhead than SR-MPLS
- Operational complexity (MTU planning, IPv6 troubleshooting)

**For most SPs:** SR-MPLS is the better choice today.  
**For greenfield IPv6 networks:** SRv6 is worth serious consideration, but test thoroughly.  
**For everyone:** Keep an eye on SRv6 — it's where the industry is heading long-term.
