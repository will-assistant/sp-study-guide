# 6.1 SR-MPLS Fundamentals

## Overview

Segment Routing (SR) is the first clean-sheet rethinking of MPLS control planes in 20+ years. Instead of running separate protocols for label distribution (LDP), traffic engineering (RSVP-TE), and fast reroute (LFA/FRR), SR collapses all of those functions into a single elegant architecture driven by the IGP you already have.

The core idea: every node in the network advertises a label (called a **Segment ID** or **SID**) for itself via IS-IS or OSPF extensions. To build a path, you just stack those labels. No signaling protocol needed. No per-tunnel state. Just the IGP and a label stack.

This section builds the foundation: what SIDs are, how they're allocated and advertised, how packets get forwarded, and how this all maps to IOS-XR, IOS-XE, and Junos.

## Theory: Segment Routing Architecture

### The Problem SR Solves

Traditional MPLS has three fundamental problems:

1. **Control plane sprawl**: You need LDP for basic LSPs, RSVP-TE for traffic engineering, and LFA/FRR for fast reroute. Each protocol has its own failure modes, scaling limits, and operational complexity.

2. **Per-tunnel state**: RSVP-TE maintains soft state for every tunnel on every hop. In a large network with thousands of tunnels, that's millions of state entries to refresh every 30 seconds.

3. **Tight coupling**: LDP ties label distribution to the IGP's shortest path. If you want to steer traffic somewhere else, you need RSVP-TE. If RSVP-TE fails, you fall back to LDP (or worse, IP). There's no middle ground.

SR fixes all three:
- Single control plane (the IGP)
- Stateless core (state only at ingress)
- Flexible path steering (source routing)

### SR-MPLS vs SRv6

Segment Routing has two data planes:

- **SR-MPLS**: Uses MPLS labels. Runs on existing hardware. This is what most SPs deploy today.
- **SRv6**: Uses IPv6 extension headers. No MPLS needed. Still emerging; limited hardware support.

This section covers SR-MPLS. SRv6 gets its own section (6.4).

## Segment IDs (SIDs)

### Prefix-SID (Node-SID)

A **Prefix-SID** is a label that identifies a specific node in the network. Every router advertises a Prefix-SID for its loopback (or any other prefix you want globally routable).

**Key properties:**
- **Globally significant**: All routers in the SR domain use the same label for a given prefix
- **IGP-distributed**: Advertised via IS-IS or OSPF TLVs
- **ECMP-aware**: If there are multiple equal-cost paths to the destination, packets are load-balanced

**Example:**
```
R1 loopback: 10.1.1.1/32 ‚Üí Prefix-SID 16001
R2 loopback: 10.2.2.2/32 ‚Üí Prefix-SID 16002
R3 loopback: 10.3.3.3/32 ‚Üí Prefix-SID 16003
```

When R1 wants to send traffic to R3, it pushes label **16003** and forwards. Every router along the path knows "label 16003 = shortest path to R3."

### Adjacency-SID (Adj-SID)

An **Adjacency-SID** is a label that identifies a specific link (not a destination). It's used to force traffic onto a particular next-hop, bypassing ECMP or IGP shortest path.

**Key properties:**
- **Locally significant**: Only the advertising router knows what the label means
- **Dynamically allocated**: Created automatically for every adjacency
- **Strict forwarding**: Traffic MUST go out the specified link (no ECMP)

**Example:**
```
R2 has two neighbors:
  R2 ‚Üí R3 (interface Gi0/0/1) ‚Üí Adj-SID 24001
  R2 ‚Üí R4 (interface Gi0/0/2) ‚Üí Adj-SID 24002
```

If you push label **24001** on R2, the packet goes out Gi0/0/1 toward R3, even if the IGP's best path is via R4.

**Adj-SID Allocation:**
- IOS-XR/IOS-XE: Dynamically allocated from the label space **outside** the SRGB (typically 15000-15999 or higher ranges)
- Junos: Dynamically allocated from a separate dynamic label range

Adjacency-SIDs are advertised via IS-IS TLV 22 (Extended IS Reachability) Sub-TLV 31 or OSPF Opaque LSA.

### SRGB: Segment Routing Global Block

The **SRGB** is the range of labels reserved for Prefix-SIDs. It **should be consistent across all routers** in the SR domain for operational simplicity.

**Default SRGB:**
- IOS-XR: 16000‚Äì23999 (8,000 labels)
- Junos: 16000‚Äì23999 (8,000 labels, changed to match Cisco for interop in Junos 18.1+)

**Can different routers have different SRGBs?**
Technically **yes** ‚Äî each router advertises its own SRGB via IGP, and the ingress PE computes the correct label per-hop as `target_node_SRGB_base + SID_index`. The architecture supports heterogeneous SRGBs (RFC 8402). However, this is **strongly discouraged** because:
1. It complicates troubleshooting (label 16100 on R1 ‚âÝ the same SID as 16100 on R2)
2. Mismatched SRGB sizes risk SID indices exceeding the range on some routers
3. Expanded SRGBs may overlap with dynamic label ranges, causing label collisions

**Best practice:**
Configure the same SRGB everywhere. Size it with 2-3x headroom for future growth. If you need to expand it, do a network-wide change during a maintenance window.

## How Prefix-SIDs Work

### SID Index vs Absolute Label

There are two ways to assign a Prefix-SID:

1. **SID Index**: You configure an **index** (e.g., 1, 2, 3‚Ä¶), and each router calculates the label as:
   ```
   Label = SRGB_base + SID_index
   ```
   Example:
   ```
   R1: Loopback 10.1.1.1/32, SID index 1
   SRGB: 16000‚Äì23999
   Label = 16000 + 1 = 16001
   ```

2. **Absolute Label**: You configure the exact label value (e.g., 16001). The router uses it as-is.

**Best practice:** Use SID index. It's more flexible and easier to renumber.

### IGP Advertisement

Prefix-SIDs are advertised via new TLVs in IS-IS and OSPF:

**IS-IS:**
- TLV 135: Extended IP Reachability
  - Sub-TLV 3: Prefix-SID

**OSPF:**
- Opaque LSA Type 10 (Area-local)
- TLV 1: Extended Prefix Range
  - Sub-TLV 2: Prefix-SID

Every router in the SR domain receives these advertisements and populates:
- **Routing table**: Prefix ‚Üí next-hop
- **Label table (LFIB)**: Incoming label ‚Üí outgoing label + next-hop

## SR-MPLS Forwarding

### Packet Flow Example

Topology:
```
R1 ---- R2 ---- R3 ---- R4
```

Prefix-SIDs:
```
R1: 10.1.1.1/32 ‚Üí 16001
R2: 10.2.2.2/32 ‚Üí 16002
R3: 10.3.3.3/32 ‚Üí 16003
R4: 10.4.4.4/32 ‚Üí 16004
```

**Scenario:** R1 sends a packet to R4 (10.4.4.4).

**Step 1 (R1):**
- Lookup 10.4.4.4 in routing table ‚Üí next-hop R2
- Push label **16004** (Prefix-SID for R4)
- Forward to R2

**Step 2 (R2):**
- Incoming label: 16004
- LFIB lookup: 16004 ‚Üí swap to 16004, next-hop R3
- Forward to R3

**Step 3 (R3):**
- Incoming label: 16004
- LFIB lookup: 16004 ‚Üí swap to 16004, next-hop R4 (penultimate hop)
- **PHP (Penultimate Hop Popping)**: Pop label, forward IP packet to R4

**Step 4 (R4):**
- Receives IP packet (no label)
- Destination is local loopback ‚Üí deliver

### PHP vs No-PHP

By default, SR-MPLS uses **PHP** (just like LDP): the second-to-last router pops the label.

Why? Saves the egress router from doing a label lookup.

You can disable PHP by configuring **explicit-null** on the advertising router:
- IPv4: Label 0 (same as LDP)
- IPv6: Label 2

With explicit-null, R3 swaps label 16004 ‚Üí label 0, and R4 receives a labeled packet.

**How to enable explicit-null:**

**IOS-XR:**
```cisco
router isis CORE
 interface Loopback0
  address-family ipv4 unicast
   prefix-sid index 4 explicit-null
```

**Junos:**
```junos
protocols isis {
    source-packet-routing {
        node-segment ipv4-index 4 explicit-null;
    }
}
```

**When to use explicit-null:**
- QoS: Preserve MPLS EXP bits to the egress
- Troubleshooting: Easier to see labeled traffic in packet captures

## Label Stack: Building Paths with Multiple SIDs

### Why Stack Labels?

A single Prefix-SID gives you the IGP shortest path. To steer traffic along a different path, you **stack multiple SIDs**.

**Example:**
Topology with parallel paths:
```
       R2
      /  \
R1 --+    +-- R4
      \  /
       R3
```

Force traffic from R1 to R4 via the R1‚ÜíR2‚ÜíR3‚ÜíR4 path (instead of R1‚ÜíR2‚ÜíR4 or R1‚ÜíR3‚ÜíR4):

```
R1 pushes TWO labels:
  - Top: 16003 (Prefix-SID for R3)
  - Bottom: 16004 (Prefix-SID for R4)
```

**Forwarding:**
1. R1 ‚Üí R2: Top label is 16003 ‚Üí forward toward R3 (IGP shortest path)
2. R2 ‚Üí R3: Top label is 16003 ‚Üí forward toward R3
3. R3: Top label is 16003, R3 is the destination ‚Üí **pop top label** (PHP)
4. R3 ‚Üí R4: New top label is 16004 ‚Üí forward toward R4
5. R4: Receives IP packet (after penultimate hop pop)

This is **source routing**: the ingress router (R1) decides the entire path by stacking the right SIDs.

**Label stack depth limits:**
Hardware has label stack processing limits:
- Legacy ASICs: 3-6 labels
- Modern ASICs (Jericho2, Tomahawk3): 8-12 labels
- Software routers: Unlimited (performance-limited)

If you exceed the hardware limit, packets are **silently dropped** (no ICMP error). Always check your platform's datasheet before deploying deep label stacks.

### Mixing Prefix-SIDs and Adj-SIDs

For even more precise control, mix Prefix-SIDs (for destinations) and Adj-SIDs (for specific links).

**Example:**
Force R1 ‚Üí R2 ‚Üí (specific link) ‚Üí R3 ‚Üí R4:

```
Label stack (top to bottom):
  1. Adj-SID 24001 (R2's link to R3)
  2. Prefix-SID 16003 (R3)
  3. Prefix-SID 16004 (R4)
```

**Why this matters:**
- Prefix-SIDs leverage ECMP (good for load balancing)
- Adj-SIDs bypass ECMP (good for deterministic paths or link coloring)

## Configuration: IOS-XR (Primary)

### Enable SR in IS-IS (IOS-XR)

```cisco
!! SR-MPLS with IS-IS ‚Äî IOS-XR (ASR 9000, NCS 5500)
router isis CORE
 is-type level-2-only
 net 49.0001.0000.0000.0001.00
 address-family ipv4 unicast
  metric-style wide
  segment-routing mpls
 !
 interface Loopback0
  address-family ipv4 unicast
   prefix-sid index 1
  !
 !
 interface GigabitEthernet0/0/0/0
  circuit-type level-2-only
  point-to-point
  address-family ipv4 unicast
  !
 !
!
```

**Key commands:**
- `segment-routing mpls` under `address-family`: Enables SR label distribution via IS-IS
- `prefix-sid index N` under the interface: Assigns a Prefix-SID index to the loopback (replaces the IOS-XE `connected-prefix-sid-map` approach)

### Enable SR in OSPF (IOS-XR)

```cisco
!! SR-MPLS with OSPF ‚Äî IOS-XR
router ospf 1
 router-id 10.1.1.1
 segment-routing mpls
 area 0
  interface Loopback0
   prefix-sid index 1
  !
  interface GigabitEthernet0/0/0/0
   network point-to-point
  !
 !
!
```

**Note:** OSPF SR requires OSPFv2 Extensions (RFC 7684). Adjacency-SIDs are automatically allocated.

### SRGB Configuration (IOS-XR)

```cisco
!! Explicit SRGB (must match across all routers)
segment-routing
 global-block 16000 23999
!
```

> **Also on IOS-XE:** IOS-XE uses a global `segment-routing mpls` block with `connected-prefix-sid-map` to map loopback prefixes to SID indices, rather than IOS-XR's per-interface `prefix-sid index N` model. IOS-XE also enables SR under the IGP with `segment-routing mpls` and `segment-routing prefix-sid-map advertise-local`.

## Configuration: Junos

### Enable Segment Routing

```junos
protocols {
    mpls {
        interface all;
    }
    isis {
        source-packet-routing {
            srgb start-label 16000 index-range 8000;
            node-segment ipv4-index 1;
        }
        level 2 wide-metrics-only;
        interface lo0.0 {
            passive;
        }
        interface ge-0/0/0.0;
        interface ge-0/0/1.0;
    }
}
interfaces {
    lo0 {
        unit 0 {
            family inet {
                address 10.1.1.1/32;
            }
            family iso {
                address 49.0001.0000.0000.0001.00;
            }
        }
    }
}
```

**Explanation:**
- `source-packet-routing`: Enables SR (Junos calls it "SPRING")
- `srgb start-label 16000 index-range 8000`: Defines SRGB (16000‚Äì23999)
- `node-segment ipv4-index 1`: Assigns SID index 1 to loopback 10.1.1.1/32

### OSPF SR (Junos)

```junos
protocols {
    mpls {
        interface all;
    }
    ospf {
        source-packet-routing {
            srgb start-label 16000 index-range 8000;
            node-segment ipv4-index 1;
        }
        area 0.0.0.0 {
            interface lo0.0 {
                passive;
            }
            interface ge-0/0/0.0;
        }
    }
}
```

## Troubleshooting

### Problem: Prefix-SID Not Showing in IGP

**Symptom:**
```cisco
RP/0/RSP0/CPU0:R1# show isis segment-routing prefix-sid-map active
% No prefix-SID entries found
```

**Check:**
1. Is `segment-routing mpls` enabled under the IGP address-family?
   ```cisco
   router isis CORE
    address-family ipv4 unicast
     segment-routing mpls
   ```

2. Is the prefix-SID index assigned on the loopback interface?
   ```cisco
   router isis CORE
    interface Loopback0
     address-family ipv4 unicast
      prefix-sid index 1
   ```

3. Is the interface running the IGP?
   ```cisco
   router isis CORE
    interface Loopback0
     address-family ipv4 unicast
   ```

### Problem: LFIB Not Populated

**Symptom:**
```cisco
RP/0/RSP0/CPU0:R1# show mpls forwarding labels 16000 23999
% No entries found
```

**Check:**
1. Is the SRGB configured?
   ```cisco
   segment-routing
    global-block 16000 23999
   ```

2. Are Prefix-SIDs being received from neighbors?
   ```cisco
   show isis database detail | include Prefix
   ```

3. Is SR enabled under the IGP address-family? (IOS-XR doesn't need interface-level `mpls ip` for SR)
   ```cisco
   router isis CORE
    address-family ipv4 unicast
     segment-routing mpls
   ```

4. Is the IGP adjacency up?
   ```cisco
   show isis neighbors
   ```

### Problem: SRGB Mismatch Between Routers

**Symptom:**
R1 (SRGB 16000‚Äì23999) sends label 16003 for prefix 10.3.3.3/32.
R2 (SRGB 20000‚Äì27999) calculates label 20003 for the same prefix.
R2 doesn't have a forwarding entry for label 16003 ‚Üí traffic is dropped.

**Fix:**
Standardize the SRGB across all routers:

**IOS-XR:**
```cisco
segment-routing
 global-block 16000 23999
```

**Junos:**
```junos
protocols isis {
    source-packet-routing {
        srgb start-label 16000 index-range 8000;
    }
}
```

## Design Considerations

### SID Index Planning

**Small network (<100 nodes):** Use simple sequential indices:
```
R1: index 1
R2: index 2
R3: index 3
...
```

**Large network (100+ nodes):** Encode meaning into the index:
```
POP code + node number:
  Denver POP: 1xx (101, 102, 103...)
  Phoenix POP: 2xx (201, 202, 203...)
```

**Multi-area/multi-domain:** Reserve ranges per area:
```
Area 0: 1‚Äì999
Area 1: 1000‚Äì1999
Area 2: 2000‚Äì2999
```

### SRGB Sizing

**Default:**
- IOS-XR: 8,000 labels (16000‚Äì23999)
- Junos: 100,000 labels (800000‚Äì899999)

**Rule of thumb:**
SRGB size ‚â• number of nodes in the SR domain.

**Example:**
500-node network ‚Üí SRGB must support at least 500 indices. Default IOS-XR SRGB is fine.

**Why expand the SRGB?**
- Anycast SIDs (multiple nodes share the same SID)
- Prefix-SIDs for non-loopback prefixes (e.g., service endpoints)
- Future growth

**How to expand the SRGB safely:**
1. **During a maintenance window**, expand the SRGB on ALL routers simultaneously
2. Use automation (Ansible playbook) to push the change network-wide
3. Verify SRGB consistency before assigning new SID indices

**Example: Expand from 16000-23999 to 16000-31999:**
```cisco
!! IOS-XR
segment-routing
 global-block 16000 31999
```
```junos
/* Junos */
protocols isis {
    source-packet-routing {
        srgb start-label 16000 index-range 16000;
    }
}
```

**Warning:** Expanding SRGB on one router without changing others will cause **silent label mismatch** (traffic drop). All routers must have identical SRGB.

### Interoperability: Cisco ‚Üî Juniper

SR-MPLS is vendor-neutral (RFC 8402, RFC 8660, RFC 8665). A Cisco router will correctly process Prefix-SIDs advertised by a Juniper router **as long as the SRGB matches**.

**Gotcha:** Older Junos versions (pre-18.1) had a default SRGB of 800000‚Äì899999, which didn't match Cisco's 16000‚Äì23999. Modern Junos (18.1+) defaults to 16000‚Äì23999 for interoperability.

**Solution:** Explicitly configure the SRGB on all routers to guarantee consistency. Most networks use 16000‚Äì23999 (Cisco default, now also Junos default).

## War Story: The Great SID Index Collision

**Network:** Regional ISP, 80 routers, migrating from LDP to SR-MPLS.

**Problem:**
Two teams were rolling out SR in parallel:
- **Core team:** Assigned SID index 1‚Äì40 to core routers
- **Access team:** Assigned SID index 1‚Äì50 to access routers

Nobody noticed until a core router (R10, SID index 10) and an access router (R10-ACCESS, SID index 10) both advertised the same Prefix-SID (16010).

**Result:**
The IGP selected one based on router-ID tie-breaking. Half the network thought "label 16010 = core R10," the other half thought "label 16010 = access R10-ACCESS." Traffic black-holed.

**Fix:**
Emergency SID index renumbering:
- Core: 1001‚Äì1080
- Access: 2001‚Äì2100

Script-generated configs pushed via Ansible in a 4-hour maintenance window.

**Lesson:** **Centralize SID index allocation.** Use an IPAM system (NetBox, phpIPAM) or a dedicated SR planning tool. Never let teams self-assign SID indices. Automation (Ansible, Salt, Nornir) is essential for mass renumbering.

## Summary

| Concept | Definition |
|---------|------------|
| Prefix-SID | Globally unique label for a prefix (usually loopback). IGP shortest path. |
| Adjacency-SID | Locally unique label for a specific link. Forces traffic onto that link. |
| SRGB | Label range reserved for Prefix-SIDs. Must be consistent network-wide. |
| SID Index | Offset into the SRGB. Label = SRGB_base + index. |
| PHP | Penultimate Hop Popping. Second-to-last router pops the top label. |
| Label Stack | Multiple SIDs stacked to define a path (source routing). |

**Key takeaways:**
1. SR-MPLS uses the IGP to distribute labels (no LDP, no RSVP-TE signaling)
2. Prefix-SIDs provide global reachability; Adj-SIDs provide link-level steering
3. SRGB should be consistent across all routers (heterogeneous is technically supported but operationally risky)
4. Label stacking enables source-routed paths (covered in 6.2: SR-TE Policies)
