# 6.3 — TI-LFA (Topology-Independent Loop-Free Alternate)

## Overview
TI-LFA (Topology-Independent Loop-Free Alternate) is the SR-MPLS implementation of IP Fast Reroute (FRR). It provides **sub-50ms convergence** by pre-computing backup paths for every destination and installing them in the FIB. When a link or node fails, the router immediately switches to the backup path without waiting for IGP reconvergence. Unlike LFA (RFC 5286) or Remote LFA (RLFA), TI-LFA works in **any topology** — no restrictions on connectivity or metrics.

## Theory

### Concepts

**What is IP Fast Reroute (FRR)?**
- **Problem**: IGP convergence takes 1-5 seconds (even with tuning). During this time, traffic is black-holed.
- **Solution**: Pre-compute a backup path for every destination. If the primary path fails, switch to backup immediately.
- **Goal**: Sub-50ms failover (imperceptible to most applications)

**LFA Evolution:**
1. **Basic LFA (RFC 5286)**: Use a loop-free alternate neighbor as backup
   - **Limitation**: Only works if a loop-free neighbor exists (topology-dependent, ~60-70% coverage)
2. **Remote LFA (RLFA, RFC 7490)**: Tunnel to a distant node using LDP LSP
   - **Limitation**: Requires LDP, relies on LDP convergence, still topology-dependent (~85-90% coverage)
3. **TI-LFA (RFC 9514)**: Use SR label stack to steer around failure
   - **Advantage**: Works in any topology, no tunneling protocol needed, 100% coverage

**TI-LFA Mechanics:**
- For every destination, compute a backup path that avoids the primary next-hop link/node
- Express the backup path as a **segment list** (Prefix-SIDs + Adj-SIDs)
- Install both primary and backup in FIB
- On failure detection (BFD, link down), switch to backup instantly
- While using backup, IGP reconverges in background

**Post-Convergence Path:**
- TI-LFA backup is temporary (only used during failure)
- After IGP reconverges, traffic switches to new primary path (computed by SPF)
- Backup is then recomputed for the new topology

**Protection Types:**
- **Link protection**: Backup path avoids the failed link (protects against link failure)
- **Node protection**: Backup path avoids the failed node (protects against router failure)
- TI-LFA prefers node protection when possible (stronger guarantee)

### Architecture

**Where TI-LFA Fits:**
- **Edge/Core**: Typically deployed everywhere in the SR-MPLS network
- **Coverage**: ~100% of prefixes in well-connected topologies (unlike basic LFA which covers 60-70%)
  - **Edge case**: Single-homed nodes or stub links may have no backup (no alternate path exists)
  - **Typical production**: 98-100% coverage in properly designed networks
- **Overhead**: Minimal — backup paths stored in FIB, no extra control plane

**Failure Detection:**
- **BFD (Bidirectional Forwarding Detection)**: Detects failures in 10-50ms
  - Sub-second detection recommended (50ms is common)
- **Link-down events**: Physical layer failure (immediate)
- Once failure is detected, FIB switches to backup path within 1-2ms (hardware switchover)

**TI-LFA Path Computation:**
1. **Run SPF** from local router (primary path)
2. **Run reverse SPF** from each destination (view from destination's perspective)
3. **Identify P-space** (nodes reachable without using protected link)
4. **Identify Q-space** (nodes that can reach destination without using protected link)
5. **Find PQ-node** (node in both P-space and Q-space)
6. **Build segment list**: Local→PQ-node→Destination
   - If direct path to PQ-node avoids failure: use Prefix-SID(PQ-node) + Prefix-SID(Destination)
   - If not: use Adj-SID to steer around failure, then Prefix-SID(PQ-node), Prefix-SID(Destination)

**Example (Topology: R1—R2—R5, R1—R3—R4—R5):**
- **Primary path** from R1 to R5: R1→R2→R5 (metric 20)
- **Protect link**: R1→R2
- **P-space** (reachable without R1→R2): {R1, R3, R4}
- **Q-space** (can reach R5 without R1→R2): {R3, R4, R5}
- **PQ-node**: R3 (in both sets)
- **Backup segment list**: Prefix-SID(R3) + Prefix-SID(R5)
  - R1 pushes label stack: [16003, 16005]
  - Packet goes R1→R3→(R3 pops 16003, forwards on 16005)→R4→R5

### Design Considerations

**When to Use TI-LFA:**
- **Always**, if you're deploying SR-MPLS (it's the killer feature)
- Replaces RSVP-TE FRR (no per-tunnel state required)
- Required for carrier-grade SLAs (5-nines availability needs sub-50ms failover)

**When NOT to Use:**
- Network without SR-MPLS (use basic LFA or RLFA instead)
- Testing/lab environments where convergence speed doesn't matter

**BFD Tuning:**
- **Interval**: 50ms tx/rx (common for production)
  - Lower = faster detection, higher CPU cost
  - 10ms is possible on high-end platforms (NCS 5500, PTX, MX)
- **Multiplier**: 3 (allows 2 missed packets before declaring failure)
- **Hardware offload**: Use BFD offload if available (reduces CPU impact)

**Label Stack Depth:**
- TI-LFA backup paths can use 2-4 labels depending on topology
  - **Best case** (2 labels): Prefix-SID(PQ-node) + Prefix-SID(Destination)
  - **Worst case** (4 labels): Adj-SID(avoid failed link) + Adj-SID(reach PQ) + Prefix-SID(PQ-node) + Prefix-SID(Destination)
  - **Typical** (2-3 labels): Most production networks see 2-3 labels for TI-LFA backups
- Ensure platform supports required depth (most modern platforms support 8+ labels)
- **MTU impact**: 4 labels = 16 bytes overhead (check interface MTU, add 20-30 bytes margin)

**Vendor Differences:**
- **Cisco IOS-XR**: TI-LFA enabled per-protocol (IS-IS, OSPF), supports link + node protection
- **Juniper Junos**: Requires explicit config (`source-packet-routing { ti-lfa }`), node protection default
- **Nokia SR-OS**: TI-LFA via `fast-reroute { ti-lfa }` under IGP
- **Arista EOS**: Limited support (basic SR, no native TI-LFA as of 2024)

**Gotchas:**
- **ECMP and TI-LFA**: If primary path is ECMP (multiple equal-cost paths), TI-LFA protects each individual next-hop. Failure of one ECMP member → traffic shifts to other members (no backup needed). Failure of all members → backup path used.
- **BFD session scale**: Each link needs a BFD session. 100 links = 100 BFD sessions. Monitor CPU usage.
- **Convergence + Protection**: TI-LFA protects during convergence, not instead of convergence. IGP still reconverges in background (tune timers to 1-2 seconds).

## Configuration

### Cisco IOS-XR

**Enable TI-LFA (IS-IS):**
```cisco
! SR-MPLS must be enabled first (prerequisite)
segment-routing
 global-block 16000 23999
!
router isis CORE
 is-type level-2-only
 net 49.0001.0000.0000.0001.00
 address-family ipv4 unicast
  metric-style wide
  segment-routing mpls
  fast-reroute per-prefix
  fast-reroute per-prefix ti-lfa
 !
 interface GigabitEthernet0/0/0/0
  point-to-point
  address-family ipv4 unicast
   fast-reroute per-prefix
   fast-reroute per-prefix ti-lfa
  !
 !
!
```

**Enable TI-LFA (OSPF):**
```cisco
router ospf CORE
 area 0
  fast-reroute per-prefix
  fast-reroute per-prefix ti-lfa enable
 !
 interface GigabitEthernet0/0/0/0
  fast-reroute per-prefix
  fast-reroute per-prefix ti-lfa
 !
!
```

**BFD Integration:**
```cisco
! Enable BFD globally
bfd
 echo disable
 multipath include location 0/0/CPU0
!

! Enable BFD on ISIS interfaces
router isis CORE
 interface GigabitEthernet0/0/0/0
  bfd minimum-interval 50
  bfd multiplier 3
  bfd fast-detect ipv4
 !
!
```

**Verification Commands:**
```cisco
show isis fast-reroute summary
show isis fast-reroute 10.0.0.5/32 detail
show bfd session
show cef 10.0.0.5/32 detail
```

### Juniper Junos

**Enable TI-LFA (IS-IS):**
```junos
protocols {
    isis {
        source-packet-routing {
            ti-lfa {
                node-protection;
            }
        }
        level 2 {
            wide-metrics-only;
        }
        interface ge-0/0/0 {
            point-to-point;
        }
    }
}
```
**Note:** Junos also supports `post-convergence-lfa` (optimizes path after IGP reconvergence), but it's separate from TI-LFA. For basic TI-LFA, the above config is sufficient.

**Enable TI-LFA (OSPF):**
```junos
protocols {
    ospf {
        area 0.0.0.0 {
            interface ge-0/0/0 {
                interface-type p2p;
            }
        }
        source-packet-routing {
            ti-lfa {
                node-protection;
            }
        }
    }
}
```

**BFD Integration:**
```junos
protocols {
    isis {
        interface ge-0/0/0 {
            point-to-point;
            bfd-liveness-detection {
                minimum-interval 50;
                multiplier 3;
            }
        }
    }
}
```

**Verification Commands:**
```junos
show isis backup coverage
show isis route 10.0.0.5 detail
show bfd session
show route 10.0.0.5 active-path detail
```

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| **ti-lfa** | Disabled | **Enabled** | Provides 100% coverage FRR |
| **node-protection** | Link protection | **Node protection** | Stronger guarantee (survives router failure) |
| **bfd minimum-interval** | 300ms | **50ms** | Faster failure detection |
| **bfd multiplier** | 3 | **3** | Balance between false positives and speed |
| **bfd echo** | Enabled (Junos), Disabled (IOS-XR) | **Platform-dependent** | IOS-XR: echo mode reduces CPU (recommended). Junos: echo rarely used. Check vendor docs. |
| **fast-reroute per-prefix** | Disabled | **Enabled** | TI-LFA requires per-prefix FRR |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Backup coverage <100% | TI-LFA not enabled on all interfaces | Enable `fast-reroute per-prefix ti-lfa` on all IGP interfaces |
| Node protection 0% | Topology doesn't support node protection | Verify there are alternate paths; may need link protection |
| BFD session down | Incorrect interval/multiplier mismatch | Ensure both sides have matching BFD config |
| Failure takes >50ms | BFD interval too high | Lower BFD interval to 50ms or less |
| CPU high after enabling BFD | Too many BFD sessions, no hardware offload | Enable BFD offload or increase interval to 100ms |
| Backup path not used during failure | FIB not programmed | Check `show cef <prefix> detail` for backup next-hop |
| Label stack too deep | Complex topology requires many hops | Simplify topology or increase MTU |

### Methodology

**TI-LFA Not Working (Traffic Blackholed During Failure):**

1. **Verify TI-LFA is enabled:**
   ```
   show isis fast-reroute summary
   ```
   - Must show "FRR Type: Per-prefix TI-LFA"
   - Coverage should be ~100%

2. **Check specific prefix has backup:**
   ```
   show isis fast-reroute <prefix> detail
   ```
   - Must show "Backup path (TI-LFA)" with repair list
   - If missing, check SR-MPLS config (SIDs must be advertised)

3. **Verify BFD session is up:**
   ```
   show bfd session
   ```
   - State must be "Up"
   - If down, check interface config and reachability

4. **Verify FIB is programmed:**
   ```
   show cef <prefix> detail
   ```
   - Should show both primary and backup next-hop
   - If backup missing, check platform supports TI-LFA (some line cards don't)

5. **Test failure manually:**
   ```
   shutdown interface <primary-interface>
   ```
   - Traffic should switch to backup within 50ms
   - Use continuous ping to measure failover time

**BFD Not Detecting Failures:**

1. **Check BFD config on both sides:**
   - Intervals must be compatible (lower of the two is used)
   - Multiplier must allow for packet loss (3 is typical)

2. **Verify BFD packets are being sent/received:**
   ```
   debug bfd packet
   ```
   - Watch for "Tx" and "Rx" messages
   - If no Rx, check for firewall/ACL blocking UDP 3784

3. **Check platform limitations:**
   - Some platforms limit BFD sessions (e.g., 100 per line card)
   - Check `show bfd summary` for scale limits

### War Stories

**Scenario: BFD False Positives During Upgrade**

**Setup:** Large SP network, 500 routers, TI-LFA + BFD (50ms interval) deployed everywhere. Network is stable, ~15,000 BFD sessions network-wide. Planned IOS-XR upgrade from 6.5.3 to 7.3.2 on 50 core routers (rolling upgrade, 5 routers at a time).

**Failure:** During the upgrade of the first batch (5 routers), BFD sessions to neighbors start flapping. Each flap triggers TI-LFA failover, causing micro-bursts of traffic on backup paths. This happens 20-30 times per router during the 10-minute upgrade window. Some sessions experience 200-500ms packet loss during each flap (exceeds 50ms SLA).

**Root Cause:** IOS-XR 7.3.2 changed BFD session handling during process restarts. During upgrade, BFD control plane restarts, sessions go down for ~2 seconds. BFD hardware offload was NOT enabled, so all sessions relied on CPU. During CPU spike (upgrade process), BFD keepalives were delayed → false failure detection.

**Impact:**
- ~500 BFD flaps across 5 routers during 10-minute window
- TI-LFA triggered ~500 backup path activations
- Some backup paths were congested (not pre-validated)
- 10 customer tickets (VoIP quality degradation)

**Lessons:**
1. **Enable BFD hardware offload** — CPU-based BFD fails during high load
2. **Increase BFD interval during maintenance** — Temporarily change to 300ms before upgrade, revert after
3. **Use graceful restart for BFD** — IOS-XR 7.3.2+ supports BFD NSR (Non-Stop Routing); sessions survive process restart
4. **Pre-validate backup paths** — Run `traceroute` on backup paths to ensure they're not congested
5. **Stagger upgrades** — 5 simultaneous upgrades was too aggressive; reduce to 2 at a time

**Fix Applied:**
- Enabled BFD hardware offload on all platforms that support it (NCS 5500, ASR 9000)
- Changed upgrade procedure: increase BFD interval to 300ms → upgrade → revert to 50ms
- Enabled BFD NSR (Non-Stop Routing) where available
- Added BFD session monitoring: alert if >10 flaps/hour per router

## Quick Reference

**Key RFCs:**
- RFC 9514 — TI-LFA (Topology-Independent Loop-Free Alternate)
- RFC 5286 — Basic LFA
- RFC 5880 — BFD (Bidirectional Forwarding Detection)
- RFC 5881 — BFD for IPv4/IPv6 (Single Hop)
- RFC 7490 — Remote LFA (RLFA)

**Cert Relevance:**
- **IE-SP:** FRR and LFA are tested (TI-LFA is emerging)
- **IE-SP:** TI-LFA is core topic (tested since 2021)

**Failover Time Breakdown:**
- **Failure detection**: 50ms (BFD) or 0ms (link down)
- **FIB switchover**: 1-2ms (hardware)
- **IGP reconvergence**: 1-2 seconds (background, doesn't block traffic)
- **Total user-visible impact**: ~50ms (one packet loss)

**TI-LFA vs Other FRR:**
- **Basic LFA**: 60-70% coverage, no tunneling, topology-dependent
- **Remote LFA**: 85-90% coverage, LDP tunneling, topology-dependent
- **TI-LFA**: 100% coverage, SR label stack, topology-independent
