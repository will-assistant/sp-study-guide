# 12.2 — Design Case Study: Data Center Interconnect with EVPN

> **Level:** Internet Expert / IE-SP  
> **Prerequisites:** Module 8 (L2VPN & EVPN), Module 3 (BGP Deep Dive), Module 6 (Segment Routing)

---

## Overview

> Data Center Interconnect is one of the highest-stakes design problems in modern networking. Get it wrong and you've built a failure domain that spans two data centers instead of one. Get it right and you can run active-active clusters across buildings hundreds of kilometers apart with sub-second failover and zero L2 loop risk.

**Scenario: GlobalTrade Financial Services**

GlobalTrade is a financial services firm running high-frequency trading infrastructure across three sites:

- **DC-EAST** — Secaucus, NJ (primary east coast trading floor, 800 bare-metal servers, 40Gbase-T ToR, 100G spine)
- **DC-WEST** — Ashburn, VA (primary west coast-facing workloads, 600 servers, same fabric)
- **DC-DR** — Dallas, TX (disaster recovery + dev/staging, 400 servers, asynchronous replication target)

Distance: DC-EAST ↔ DC-WEST: 280km (dark fiber via colocation providers). DC-EAST ↔ DC-DR: 2,100km (leased wavelength). RTT: East↔West ~2.8ms; East↔DR ~21ms.

**Business requirements:**
- vMotion between DC-EAST and DC-WEST without application disruption (GlobalTrade's internal SLA requires <5ms RTT for this — their HFT workloads are latency-sensitive; requires L2 extension)
- Active-active trading cluster across both primary DCs — same IP addresses, same VLANs, split at the application tier
- L3 routing for inter-VRF traffic (trading VRF, risk VRF, management VRF) across all three sites
- Sub-second failover if a DC loses its upstream spine layer
- DR site receives replicated production data and L3 reachability (no L2 stretch) — can be brought live within 15 minutes RTO

**What you'll design:** A complete DCI architecture using EVPN, from underlay choices through BGW placement to failure recovery procedures.

## Current State: Why STP-Based DCI Is Failing

GlobalTrade's current DCI uses 100G dark fiber between sites with a "supercore" layer running RSTP. Two Cisco Nexus 9000s in each DC act as the DCI border nodes, connected by a pair of 100G dark fiber pairs. VLAN pruning is done manually by the network team.

### Problem 1: Layer 2 Domain = Failure Domain

```
Current topology:
DC-EAST                    DC-WEST
[Spine1] [Spine2]          [Spine1] [Spine2]
    |         |                |         |
[Border1] [Border2] -------- [Border1] [Border2]
         (RSTP root)         (blocked)
```

RSTP is blocking one of the four inter-DC links at all times — 50% of bandwidth wasted for loop prevention. Worse, a spanning tree topology change (TCN) in DC-EAST can flush MAC tables in DC-WEST, causing 30-60 seconds of flooding across a 280km link. During a TCN event, a 10Gbps trading VLAN becomes 10Gbps of broadcast.

**Observed failure:** Three times in the past year, a failed power supply caused a TCN that triggered MAC flush propagation, resulting in 45-second traffic blackouts. At $40M/minute trading velocity, this is existential.

### Problem 2: No Loop Prevention Without STP = No L2 Redundancy

When GlobalTrade tried to activate the second link on each DC Border pair (for active-active bandwidth), they hit a broadcast storm within 8 minutes. The network team has been running active-standby on the DCI links since the incident. Half the fiber capacity sits dark.

### Problem 3: Manual VLAN Management Across Sites

```
Current state:
- 48 VLANs stretched across DC-EAST and DC-WEST
- 12 VLANs should NOT be stretched (legacy systems with local broadcast dependencies)
- No automated policy — engineers manually add/remove VLANs from trunk lists
- Last audit found 7 VLANs stretched that should have been local-only
```

A misconfigured VLAN stretch can expose unnecessary east-west broadcast domains across sites — a material operational and security policy risk for regulated financial environments.

## EVPN DCI Architecture Design

### Design Goal: Eliminate STP from the DCI Path

EVPN replaces spanning tree with BGP MAC/IP route distribution. Instead of flooding BUM (Broadcast, Unknown unicast, Multicast) traffic to learn MACs, VTEPs (VXLAN Tunnel Endpoints) or PE nodes pre-populate their MAC tables from BGP Type-2 route advertisements. The result:

- No TCN propagation — MAC flush stays local to the site that experienced the topology change
- No blocked links — all DCI links active simultaneously, ECMP across all paths
- Deterministic L2 loop prevention via split-horizon enforcement at each site's BGW
- Per-VLAN (per-EVI) control — stretch exactly the VLANs that need stretching, nothing else

### Component Architecture

```
DC-EAST                                DC-WEST
┌─────────────────────┐               ┌─────────────────────┐
│  ToR (VTEP)  ToR    │               │  ToR (VTEP)  ToR    │
│     │           │   │               │     │           │   │
│  Leaf-1      Leaf-2 │               │  Leaf-1      Leaf-2 │
│        \    /       │               │        \    /       │
│         Spine       │               │         Spine       │
│        /    \       │               │        /    \       │
│     BGW-1  BGW-2    │               │     BGW-1  BGW-2    │
└──────┬──────┬───────┘               └───────┬──────┬──────┘
       │      │                               │      │
       │   DCI Fabric (MPLS or IP/VXLAN)      │      │
       └──────┬────────────────────────────────┘      │
              └────────────────────────────────────────┘
              
BGW = Border Gateway (EVPN multi-site gateway role)
```

### Three Underlay Options for the DCI Fabric

Before designing the EVPN control plane, you must choose the DCI transport. Three options exist:

#### Option A: MPLS/RSVP or SR-MPLS Backbone (recommended for carriers, large enterprises)

GlobalTrade connects to a carrier MPLS backbone between Secaucus and Ashburn. There are two distinct service models and they must not be conflated:

- **Managed L2 handoff (EoMPLS/VPWS):** Customer sees a point-to-point L2 circuit and does **not** exchange EVPN Type-2/5 with the carrier PE.
- **Carrier-managed EVPN interconnect:** EVPN route exchange/policy is explicitly part of the service contract and platform support model.

**Pros:** Carrier SLA, QoS guarantees, no need to own fiber  
**Cons:** Carrier dependency, service-model constraints (managed L2 vs EVPN handoff), higher recurring cost

#### Option B: VXLAN over Routed IP (dark fiber + BGP underlay)

GlobalTrade owns dark fiber. Underlay is simple routed IP — eBGP or OSPF for reachability. VXLAN is the overlay encapsulation. EVPN distributes MAC/IP and provides the control plane. BGW-to-BGW tunnels are VXLAN UDP (port 4789).

**Pros:** Vendor-agnostic, owns the fiber, can run 400G LAG for throughput, ECMP across all paths naturally  
**Cons:** IP MTU management (VXLAN adds ~50 bytes of overhead; a common operational target is 9100 or 9216 underlay MTU to preserve 9000-byte tenant frames with headroom), no inherent QoS without DSCP marking + CoS at fiber layer

#### Option C: EVPN over SR-MPLS DCI Fabric (enterprise private backbone)

GlobalTrade runs its own SR-MPLS backbone between sites. BGW nodes are SR-capable. EVPN L2VPN runs over SR tunnels — the BGW pushes SR label stack to reach remote BGW. This is the Cisco DCI-MPLS or Juniper EVPN-DCI-MPLS model.

**Pros:** Traffic engineering capability, fast-reroute (TI-LFA) for DCI link protection, deterministic bandwidth allocation per tenant  
**Cons:** Requires SR expertise, more complex than VXLAN/IP, significant investment if no existing SR backbone

**GlobalTrade Decision:** Option B (VXLAN/IP over dark fiber) between primary sites. Option A (carrier MPLS wavelength) for the DR site in Dallas due to 2,100km span.

## EVPN Multi-Site BGW Design

The Border Gateway (BGW) is the critical element in multi-site EVPN. Understanding what the BGW does — and what it doesn't do — determines the entire design.

### What the BGW Does

1. **Re-originates Type-2 routes:** A MAC learned at DC-EAST ToR is advertised as a local Type-2 into the DC-EAST EVPN fabric. The DC-EAST BGW re-originates that Type-2 toward DC-WEST, replacing the originating VTEP with its own BGW IP. DC-WEST sees the BGW as the next hop — not the original ToR.

2. **Stitches EVIs across sites:** EVI (EVPN Instance) 100 in DC-EAST maps to EVI 100 in DC-WEST. The BGW maintains the mapping and translates Route Target + VNI/label as needed.

3. **Implements split-horizon:** Traffic received from the DCI fabric is NOT flooded back into the DCI fabric. This prevents inter-site loops without spanning tree. Each site appears as a single "site-id" to the BGW — traffic from site-2 arriving at site-1's BGW is never re-advertised back to site-2.

   **Operational guardrails (must validate in lab):**
   - site-id must be **unique per site** and **consistent on all BGWs in that site**
   - BGWs must reject/refuse reflected remote-origin traffic based on site-of-origin policy
   - telemetry should alarm on unexpected remote-origin loop indicators (rapid MAC move storms, duplicate remote origin patterns)

4. **Advertises policy-shaped Type-5 routes for DCI L3 reachability:** For inter-site routing, BGWs advertise only the prefixes needed by policy. Controlled aggregation may be used (for example `10.100.0.0/16 → DC-EAST BGW`) but should preserve required specifics where partial-failure visibility matters.

5. **Maintains anycast gateway for inter-subnet routing:** GlobalTrade uses a distributed anycast gateway model. The same SVI IP/MAC exists on every leaf in DC-EAST for local inter-subnet routing. The BGW redistributes inter-VRF routes as Type-5 to other sites.

### BGW Placement: Two Models

#### Model 1: Spine-Layer BGW (BGW is also a spine)

The spine switches also function as BGW nodes. Pros: fewer devices, simpler cabling. Cons: a DCI event (BGW upgrade, DCI link failure) affects spine-layer forwarding in the DC fabric. Not recommended for trading environments where DC fabric and DCI should have independent failure domains.

#### Model 2: Dedicated BGW Tier (recommended for GlobalTrade)

Dedicated BGW nodes sit between the spine layer and the DCI. The DC fabric (spines, leaves, ToRs) is completely isolated from DCI complexity. A BGW failure or upgrade doesn't touch the intra-DC fabric.

```
GlobalTrade Chosen Design:

DC-EAST Fabric           DCI Transport         DC-WEST Fabric
[ToR] → [Leaf] → [Spine] → [BGW-1a] ─────── [BGW-1b] ← [Spine] ← [Leaf] ← [ToR]
                           [BGW-2a] ─────── [BGW-2b]
                             (two dedicated BGW pairs per site, four total DCI paths)
```

BGW-1a ↔ BGW-1b: 100G VXLAN/IP tunnel, ECMP via two separate dark fiber paths  
BGW-2a ↔ BGW-2b: Same, different fiber conduits (geographically diverse)

Total DCI bandwidth: 4 × 100G = 400Gbps usable (all paths active, ECMP)

### BGW Redundancy: Active-Active vs Active-Standby

With two BGWs per site, both must handle traffic simultaneously (active-active). This requires:

**Anycast BGW:** Both BGW-1a and BGW-2a in DC-EAST present a common multi-site BGW identity to remote sites (platform-specific implementation, often via anycast loopback and multi-site site-id constructs). Do not assume this automatically guarantees balanced ECMP — validate that remote BGWs install multiple eligible EVPN paths and that underlay recursion supports multipath.

**Implication for loop prevention:** In EVPN multi-site, inter-site split-horizon is enforced using site-id/site-of-origin semantics on BGWs. ESI/DF election is used for true multihomed Ethernet segments (for example host/CE-facing segments), not generic BGW-to-BGW DCI overlay links.

## BGP Design for DCI

### Route Reflector Architecture

GlobalTrade needs BGP iBGP RRs at each site for the intra-site EVPN fabric. For the multi-site DCI, the BGW nodes peer with remote BGW nodes using eBGP (different AS per site) or iBGP with the Site-ID attribute.

**Chosen model:** Each site is a separate BGP AS.

```
DC-EAST:  AS 65001
DC-WEST:  AS 65002  
DC-DR:    AS 65003
```

Intra-site: iBGP with RR pair (spine layer acts as RR for leaf VTEP nodes)  
Inter-site: eBGP between BGW nodes, exchanging EVPN NLRIs

### EVPN Route Types Used

| Route Type | Purpose | Used By |
|------------|---------|---------|
| Type-1 (Ethernet Auto-Discovery) | A-D signaling per ES/EVI (aliasing/mass-withdraw behavior depends on platform) | Multihomed EVPN segments where ES/EVI A-D procedures are used |
| Type-2 (MAC/IP Advertisement) | Distributes MAC + ARP/ND bindings | Leaf VTEPs (intra-site); BGW re-originates toward remote sites |
| Type-3 (Inclusive Multicast) | BUM traffic delivery tree | Per-EVI, per-site; ingress replication list |
| Type-4 (Ethernet Segment Route) | Advertises ES membership and participates in DF election for multihomed segments | Multihomed EVPN segments |
| Type-5 (IP Prefix Route) | L3 routing — summarized or policy-filtered prefixes across DCI | BGW generates for inter-site L3 |

### What the BGW Filters

Not everything learned intra-site should cross the DCI. GlobalTrade's policy:

```
Inbound from intra-site fabric (importing into BGW):
  - Accept Type-2 routes for VLANs in the stretch list (48 VLANs)
  - Reject Type-2 routes for non-stretch VLANs (local-only VLANs, legacy systems)
  - Accept Type-5 routes for VRFs: trading-vrf, risk-vrf, mgmt-vrf
  - Reject Type-5 routes: dev-vrf (dev traffic must never reach production DR)

Outbound toward DCI (advertising to remote BGWs):
  - Type-2 re-originated with BGW as next-hop
  - Type-5 policy-filtered with controlled aggregation where operationally safe (for example: 10.100.0.0/16 trading, 10.101.0.0/16 risk, 10.200.0.0/16 mgmt)
  - Preserve required more-specifics where partial-failure visibility is needed; do not blindly suppress all /32s
```

## BUM Traffic Handling

BUM (Broadcast, Unknown unicast, Multicast) is the hardest problem in L2 DCI. EVPN gives three mechanisms:

### Option 1: Ingress Replication (Head-End Replication)

The originating VTEP (or BGW) sends a separate copy of each BUM packet to every remote VTEP/BGW. No multicast required in the underlay.

```
DC-EAST BGW sends ARP broadcast for VLAN 100 (primary sites only):
  → Copy 1 to DC-WEST BGW-1b (VXLAN unicast UDP)
  → Copy 2 to DC-WEST BGW-2b (VXLAN unicast UDP)

(DC-DR is L3-only in this design and is not included in L2 BUM replication.)
```

**Pros:** No underlay multicast dependency, works over any IP fabric  
**Cons:** Replication burden on originating BGW increases linearly with site count. At 48 stretched VLANs × ARP storms = BGW CPU impact.

**GlobalTrade uses ingress replication** because their dark fiber network doesn't run IP multicast and the carrier MPLS wavelength is point-to-point.

### Option 2: Underlay Multicast (PMSI — Provider Multicast Service Interface)

Each EVI maps to a multicast group in the underlay. BUM traffic is sent to the group — the underlay multicast tree replicates to all members.

**Pros:** Efficient for many sites (replication happens in network, not at source)  
**Cons:** Requires underlay PIM + multicast routing, complex troubleshooting

### Option 3: Selective Multicast (sMBB, mLDP) — MPLS Backbones Only

For MPLS backbones, P-tunnels (mLDP or RSVP-P2MP trees) per EVI. The PE/BGW sends BUM once; the MPLS core replicates along the P-tree. This is the most efficient for large-scale SP EVPN deployments but requires full MPLS multicast support.

### ARP Suppression — Critical for Trading Networks

Without ARP suppression, every ARP request floods to all remote sites. At GlobalTrade's scale:

```
Per-site endpoint ARP baseline:
  800 endpoints × 1 ARP every 60 seconds ≈ 13.3 ARPs/second (all endpoints combined)

Worst-case synthetic upper bound (unrealistic but useful for stress framing):
  If all 48 stretched VLANs each experienced that full baseline simultaneously:
  13.3 × 48 ≈ 640 ARPs/second

More realistic planning model:
  ARP rate ≈ (active endpoints per VLAN ÷ ARP interval) summed across stretched VLANs

Storm case during failover:
  transient spikes can be orders of magnitude above steady-state; test in lab with replayed failure events

Without suppression: high broadcast ARP volume can traverse DCI
With EVPN ARP suppression: unresolved/first-seen lookups may traverse DCI, while steady-state broadcast ARP is greatly reduced
```

**How EVPN ARP suppression works:**

When a VM at DC-EAST registers its IP (10.100.1.50) → MAC (00:50:56:aa:bb:cc) binding, the local leaf advertises a Type-2 route with the MAC and IP. The DC-WEST BGW installs this as an ARP entry in its local ARP table for VLAN 100. When a VM at DC-WEST ARPs for 10.100.1.50, the local leaf responds on behalf of the remote VM — the ARP never crosses the DCI.

**Configuration on Cisco Nexus (VXLAN/EVPN):**

```
! Enable ARP suppression for an L2 VNI
vlan 100
  vn-segment 10100

interface nve1
  member vni 10100
    ingress-replication protocol bgp
    suppress-arp

! L3 VNI uses associate-vrf (separate from L2 VNI)
interface nve1
  member vni 20001 associate-vrf
```

**Configuration on Juniper QFX (EVPN/VXLAN):**

```
# ARP/ND proxy in EVPN routing instances (Junos 19.1+)
# Syntax varies by Junos release and platform — verify against target version
set routing-instances EVPN-TRADING instance-type evpn
set routing-instances EVPN-TRADING protocols evpn proxy-macip-advertisement
# On some QFX platforms with integrated routing and bridging (IRB):
# ARP proxy is implicit when the EVPN instance has an IRB interface —
# the IRB responds to ARPs for remote MAC/IP bindings learned via Type-2
```

## L3 DCI — Inter-VRF Routing Across Sites

L2 stretch handles same-VLAN traffic. L3 routing across VRFs across sites is a different problem.

### GlobalTrade VRF Layout

```
trading-vrf:  10.100.0.0/16  (HFT systems, sub-ms latency, strict QoS)
risk-vrf:     10.101.0.0/16  (risk engines, moderate latency tolerance)
mgmt-vrf:     10.200.0.0/16  (management plane, out-of-band)
```

These VRFs must be reachable across all three sites, but with policy:
- trading-vrf ↔ risk-vrf: allowed (risk engine queries trading position data)
- trading-vrf ↔ mgmt-vrf: management access only (SSH, SNMP)
- Any VRF ↔ dev-vrf: REJECTED at BGW — production isolation is contractual

### Type-5 Route Distribution for L3 DCI

The BGW summarizes each site's VRF prefixes as EVPN Type-5 IP prefix routes:

```
DC-EAST BGW advertises (Type-5):
  10.100.0.0/16  RT 65001:100  → trading-vrf, next-hop BGW-anycast
  10.101.0.0/16  RT 65001:101  → risk-vrf, next-hop BGW-anycast
  10.200.0.0/16  RT 65001:200  → mgmt-vrf, next-hop BGW-anycast

DC-WEST BGW imports (RT-based policy):
  10.100.0.0/16 → installed in trading-vrf with next-hop toward DC-EAST BGW
  10.101.0.0/16 → installed in risk-vrf
  10.200.0.0/16 → installed in mgmt-vrf
```

**Key design detail:** In common symmetric IRB designs, Type-5 forwarding is typically resolved via the EVPN next-hop (egress VTEP/BGW) and underlay reachability. The GW-IP field is optional and platform/use-case dependent (for example, overlay-index recursion in RFC 9136 workflows), not a universal anycast-gateway pointer. Packets destined for DC-EAST's trading subnet are encapsulated and sent to the resolved DC-EAST BGW/VTEP next-hop, then forwarded locally in trading-vrf.

### Inter-VRF Routing: Route Leaking vs. Separate VPN

For trading-vrf ↔ risk-vrf communication, two options:

**Option A: Route Leaking at BGW** — Import trading-vrf prefixes into risk-vrf (RT manipulation). Traffic from risk-vrf hits the local BGW, which has a leaked route for trading-vrf. No extra hops.

**Option B: Firewall-Based Policy (recommended for GlobalTrade)** — Trading ↔ risk inter-VRF traffic routes through a dedicated policy enforcement point. This is an internal security/compliance policy choice at GlobalTrade: all cross-VRF communication is logged and inspected. The firewall has legs in both VRFs.

```
DC-EAST design for inter-VRF:
trading-vrf ─────────────────────── risk-vrf
     │                                   │
     └──── [Palo Alto Firewall] ──────────┘
              (VRF-aware, policy enforcement)
              
Each site has a local firewall. Cross-site inter-VRF traffic goes:
  risk-vrf (DC-WEST) → BGW → VXLAN → BGW (DC-EAST) → local FW → trading-vrf (DC-EAST)
```

## Failure Scenarios and Recovery

### Failure 1: BGW-1a Fails (Single BGW at DC-EAST)

```
Before failure:
  Traffic from DC-WEST: ECMP across BGW-1a and BGW-2a at DC-EAST
  
After BGW-1a failure:
  Remote peers detect adjacency loss (BFD/BGP), then withdraw paths learned via BGW-1a
  Convergence uses platform EVPN multihoming behavior (Type-4 and Type-1 procedures where applicable)
  Convergence time target: BFD detects failure in ~150ms (3×50ms) + control-plane reconvergence
  
During convergence: some hashed flows may briefly hit the failed path
Mitigation: BFD on underlay routed adjacencies and/or BGP sessions (platform-dependent), plus validated fast-failover behavior in staging/lab
```

**Protection mechanism:** Inter-site loop prevention is enforced by multi-site site-id/site-of-origin split-horizon behavior on BGWs. ESI/DF procedures remain relevant for actual multihomed Ethernet segments; do not model generic BGW-to-BGW DCI overlay links as one shared ES unless the platform specifically implements that pattern.

### Failure 2: All DCI Links Between DC-EAST and DC-WEST Fail (Site Isolation)

This is the "island" scenario — DC-EAST and DC-WEST lose connectivity entirely.

```
Impact assessment:
  L2 stretch: VMs stretched across sites can no longer communicate. ARP entries cached locally
              for up to ARP timeout (GlobalTrade: 1800s). Existing flows may continue briefly
              if endpoints have cached MACs, but new ARP requests will not be answered.
  L3 routing: Type-5 routes from remote site are withdrawn. Traffic to remote subnets black holes.
  Active-active cluster: Application-layer split-brain. Trading cluster must be designed for this.
```

**GlobalTrade's response procedure:**
1. BFD detects link failure: 3×50ms = 150ms
2. BGW withdraws all remote-site Type-2 and Type-5 routes within 500ms
3. Application monitoring detects split-brain (ZooKeeper quorum, application-layer heartbeat) within 2 seconds
4. Automated DR trigger: DC-EAST becomes the authoritative site (pre-decided by run-book). DC-WEST enters read-only mode for trading.
5. External DNS TTL 30s: trading endpoint routes to DC-EAST within 30 seconds

**Network layer cannot solve application split-brain.** The EVPN design ensures clean withdrawal and no traffic black holes after BGP convergence. But which site "wins" during partition must be decided at the application layer before the incident occurs.

### Failure 3: BGW Failure Isolates DR Site

If DC-DR loses connectivity, trading continues without DR replication. The NOC receives an alert within 30 seconds (BGP session down via BFD). The DR site's Type-5 routes are withdrawn from production sites. Replication resumes when connectivity is restored; there is no automated failover to DR because RTO/RPO for DR is 15 minutes (not sub-second).

### Failure 4: MAC Flap / Duplicate MAC (VM Moved Without Notification)

vMotion moves a VM from DC-EAST to DC-WEST without renegotiating EVPN. Both the old (DC-EAST) and new (DC-WEST) leaf VTEPs advertise the same MAC.

EVPN handles this via **seq-number** in the Type-2 route. The new location advertises a higher sequence number. BGP path selection prefers the higher sequence number. Within one BGP update cycle (~100ms convergence), traffic converges to DC-WEST. The old DC-EAST entry ages out.

**Risk:** During the ~100ms window, packets destined for the VM may be sent to DC-EAST (old location). The VM is no longer there. Packets are dropped. This is the fundamental limitation of L2 stretch for live migration — there is always a brief period of loss.

**GlobalTrade mitigation:** vCenter integrates with the EVPN fabric via a controller (Cisco ACI or VMware NSX-T overlay) that sends explicit EVPN MAC withdrawals before vMotion starts. This reduces the loss window to near-zero.

## DR Site Design (Dallas, TX)

The DR site presents a special challenge: it's 2,100km away over a carrier MPLS wavelength, not owned dark fiber. RTT of 21ms makes live vMotion impractical for GlobalTrade's HFT workloads — GlobalTrade's internal SLA prohibits live migration over links with >5ms RTT due to application sensitivity (VMware itself supports vMotion at higher RTTs but with increased pause times). DR is warm standby only — replicated data, not live VMs.

### DR EVPN Approach: Selective Stretch (L3 Only)

GlobalTrade does NOT stretch L2 VLANs to the DR site. Rationale:
- 21ms RTT would cause ARP suppression cache misses and significant L2 traffic amplification
- DR is never expected to run live trading — L2 stretch provides no value
- Only L3 VRF routes (Type-5) are distributed to DR so that when DR is activated, the correct routing infrastructure is in place

```
DC-DR receives from production:
  Type-5: 10.100.0.0/16 (trading), 10.101.0.0/16 (risk) — for failover routing only
  Type-2: NOT distributed (no L2 stretch to DR)
  
DC-DR advertises to production:
  Type-5: 10.150.0.0/16 (DR-only management subnet)
  Type-2: NOT distributed
```

### DR Activation Procedure

When DR must come live (production sites both unavailable):

1. NOC confirms production unavailable (automated + manual confirmation to prevent false DR activation)
2. DNS change: trading endpoints → DR site IPs (pre-staged in Route 53 with 30s TTL)
3. BGW at DR site begins advertising DR prefixes with full production community tags
4. DR trading VMs start from last-known-good snapshot (RPO = last replication, typically <15 minutes)
5. Trading resumes at DR capacity (400 servers vs 1,400 production)

## Sizing and Scale

### MAC Table Scale

```
GlobalTrade per-site scale (endpoint estimate):
  800 servers × 8 VMs average × 1 vNIC = 6,400 endpoint MACs per site

With L2 stretch (two primary sites):
  Remote site MACs learned via EVPN Type-2: ~6,400 additional
  BGW MAC table planning target: ~12,800 entries (local + remote)
  Leaf/BGW distribution of remote endpoint state is platform-dependent — size leaf and BGW tables for stretched-segment scale, not local-only assumptions
  
Headroom example: Nexus 9300 MAC table 128K entries. 12.8K / 128K = 10% utilized (before growth headroom).
```

### ARP/ND Table Scale

```
Each IP-to-MAC binding = one ARP entry
6,400 endpoints per site × 2 sites = ~12,800 ARP entries at BGW (planning estimate)
Nexus 9300 ARP table example: 64K entries. 12.8K / 64K = 20% utilized before growth.
```

### VNI (VXLAN Network Identifier) Allocation

```
VXLAN VNI range: 1 – 16,777,215 (24-bit)
GlobalTrade allocation:
  L2 VNIs (per stretched VLAN): 10000 – 10999 (1000 slots, 48 in use)
  L3 VNIs (per VRF): 20000 – 20099 (100 slots, 3 in use)
  Reserved for growth: remainder
```

### BGP Session Scale at BGW

```
DC-EAST BGW peers:
  Intra-site: iBGP with 2× spine RRs = 2 sessions
  Inter-site: eBGP with 4× remote BGWs (2 per DC-WEST + 2 per DC-DR) = 4 sessions
  Total per BGW: 6 BGP sessions
  
BGP table at BGW (illustrative):
  Type-2 routes: ~12,800 MAC/IP entries (local + remote across two primary sites)
  Type-5 routes: policy-filtered prefixes (aggregates + required specifics)
  Type-1/3/4: ~100+ control routes depending on ES/site scale
  Total: dominated by Type-2 scale, with growth tied to stretched endpoints and policy
  
This is very manageable. The BGP table only grows with stretched VLANs and VMs — not with the full routing table.
```

## Design Review Checklist

Before signing off on a DCI EVPN design, verify these:

**Control Plane**
- [ ] BGW Re-origination: Are Type-2 routes re-originated (next-hop changed to BGW) rather than reflected? (Reflection would expose intra-site VTEP IPs across the DCI — breaks return path)
- [ ] RT policy at BGW: Are non-stretched VLANs filtered at the BGW before being advertised to DCI?
- [ ] Type-5 policy shaping: Are exported prefixes intentionally scoped (aggregates + required specifics) based on failure-domain visibility and FIB limits, rather than blanket /32 suppression?
- [ ] eBGP vs iBGP for DCI: Is AS separation correct? If same AS, are Next-Hop-Unchanged or route-map next-hop-self policies in place?

**Data Plane**
- [ ] MTU: Is end-to-end underlay MTU sized for VXLAN overhead (~50 bytes for IPv4 encapsulation) plus operational headroom (commonly 9100/9216 by platform)?
- [ ] ARP suppression: Enabled on all stretched EVIs? Verify with `show mac address-table evpn | count`
- [ ] BUM handling: Is ingress-replication membership complete for all intended remote BGWs/sites (and excluded for L3-only DR)?
- [ ] Split-horizon: Is inter-site loop prevention enabled via platform multi-site site-id/site-of-origin semantics?

**Redundancy**
- [ ] BGW anycast/site identity: Are both BGWs at each site presenting consistent multi-site identity and validating multipath installation from remote sites?
- [ ] ESI/DF scope: Is ESI/DF used only where true multihomed Ethernet segments exist (not assumed for generic DCI overlays)?
- [ ] BFD: Enabled on supported underlay adjacencies and/or BGP sessions with validated failover timers?
- [ ] BGP fast-fallover: Enabled for DCI eBGP sessions?

**Security & Compliance**
- [ ] VLAN whitelist: Only approved VLANs stretch across DCI — verified by route-policy at BGW import?
- [ ] VRF isolation: dev-vrf Route Target not exported to DCI by any policy?
- [ ] ARP proxy scope: ARP suppression does not proxy for VLANs that should not be stretched?

**Operations**
- [ ] Monitoring: EVPN route count per EVI monitored (alert if drops to zero — BGW policy error)
- [ ] Duplicate MAC detection configured with auto-recovery? (Prevents silent traffic black holes on VM migration)
- [ ] Change procedure documented: How is a new VLAN added to the stretch list? Who approves? (GlobalTrade: change board + security review for compliance)

## Internet Expert / Internet Expert Review Questions

**Q1: Traffic Engineering**

A stretched VLAN at GlobalTrade has 2Gbps of traffic symmetrically between DC-EAST and DC-WEST. BGW-1a and BGW-2a are both active. Describe exactly how traffic is load-balanced across the two BGWs at DC-EAST. What mechanism prevents one BGW from receiving all traffic destined for DC-EAST from DC-WEST?

> **Discussion:** In multi-site EVPN, remote BGWs should install multiple eligible paths toward the destination site (subject to platform best-path and multipath policies). Load distribution then depends on both EVPN path installation and underlay ECMP recursion toward reachable BGW/VTEP next-hops. The hash is flow-based, so individual flows remain ordered while aggregate traffic spreads across paths.
>
> Do not assume that a shared "anycast BGW" identity alone guarantees perfect balancing. Validate operationally: number of installed EVPN paths, FIB recursion next-hops, and actual traffic distribution under load. Enable required multipath knobs per platform where applicable.

---

**Q2: ARP Suppression Failure Mode**

A developer complains that a VM at DC-WEST cannot reach a new VM that was just deployed at DC-EAST. Ping works from DC-EAST to other DC-WEST VMs. The new VM's IP is 10.100.5.200/24. It came up 30 seconds ago. What is the most likely cause and how do you diagnose it?

> **Discussion:** Most likely cause: DC-WEST does not yet have a valid Type-2 MAC/IP advertisement for 10.100.5.200 from DC-EAST. Without that binding, ARP suppression cannot answer locally and reachability may fail or fall back to limited flooding behavior depending on platform policy.
>
> Diagnosis (in order):
> 1. **Origin check (DC-EAST leaf):** verify MAC/IP binding exists and Type-2 is originated.
> 2. **Policy check (DC-EAST BGW):** verify Type-2 is exported to remote site (RT/policy not filtering it).
> 3. **Import check (DC-WEST BGW/leaf):** verify Type-2 is received and installed in the correct EVI/VRF context.
> 4. **Data-plane check:** verify local ARP/ND cache and MAC table programming reflect the imported Type-2.
>
> Fix: trigger endpoint learning on the new VM (for example ping default gateway), then confirm Type-2 origination/export/import chain end-to-end.

---

**Q3: Split-Brain Design**

GlobalTrade's NOC asks you to design the split-brain handling for the "all DCI links down" scenario. What should the network layer do automatically, and what must be left to the application layer? What is the most dangerous mistake an engineer could make in trying to solve split-brain at the network layer?

> **Discussion:** The network layer should automatically and deterministically withdraw all remote-site EVPN routes when DCI connectivity is lost. This is correct — traffic to dead paths should be dropped cleanly, not black-holed silently. BGP route withdrawal upon BFD failure does this correctly.
>
> What the network layer must NOT do: it must not try to "pick a winner" by suppressing one site's IP addresses or injecting a summary to make one site unreachable. Any automated suppression that touches the production routing table of live sites risks causing an outage at the suppressed site that would otherwise be fully operational (the local traffic is fine — only DCI is down).
>
> The most dangerous mistake is configuring an automated script that, upon detecting DCI failure, withdraws routes from one site globally — including to the site's own local users and internet edge. This turns a "degraded DCI" event into a full site outage. The script must scope its withdrawal to only the DCI fabric, not the production routing table.
>
> The application layer must handle quorum: which site runs the trading cluster master, which is the follower. Pre-agreed run-books and tested DR procedures, not network automation, resolve split-brain for stateful applications.

---

**Q4: Type-5 Route Design**

Your DC-EAST BGW is currently advertising 8,200 individual /32 host routes to DC-WEST via EVPN Type-5. The DC-WEST network team complains that their BGW's FIB is nearly full. How do you fix this? What are the trade-offs of summarization?

> **Discussion:** The problem is route granularity: 8,200 /32 Type-5s may be unnecessary for inter-site policy and can pressure FIB scale. Fix with policy-driven prefix advertisement: export only required prefixes and aggregate where operationally safe. Instead of blanket /32 export, advertise controlled aggregates plus any required specifics for failure-domain visibility.
>
> Configuration approach: use EVPN export policy/route-maps (platform-specific) to shape Type-5 advertisements. Avoid relying on a single generic `aggregate-address ... summary-only` recipe without lab validation in your specific EVPN implementation.
>
> Trade-offs: Aggressive summarization reduces control-plane scale but hides partial failures. If 10.100.5.0/24 is impaired while 10.100.0.0/16 remains advertised, remote sites still send traffic that may fail at the destination site. Mitigate with selective specifics, health-based withdrawals, and operational telemetry.

**Q5: Platform Selection**

You must choose BGW hardware for a deployment with 100 stretched VLANs, 50,000 total MAC entries (local + remote), active-active eBGP DCI at 400Gbps aggregate, and 5 remote sites. What hardware requirements does this drive? What platform capabilities are non-negotiable?

> **Discussion:** Requirements:
> - MAC/ARP table: 50,000 entries → minimum 64K hardware entries (most modern data center spines handle 256K+)
> - BGP EVPN table: 50,000 Type-2 + Type-5 routes → manageable for any modern BGP implementation
> - VTEP scale: 5 remote sites × 2 BGW each = 10 remote VTEPs → trivial
> - Throughput: 400Gbps aggregate → requires 4× 100G ports or 2× 400G ports with line-rate forwarding; no deep packet inspection at line rate
> - Non-negotiable capabilities: VXLAN hardware termination at line rate (software VXLAN cannot sustain 400Gbps), multisite BGW mode (re-origination, split-horizon), BFD for sub-second failure detection, ECMP across all DCI links (minimum 4-path ECMP in hardware), ARP suppression in hardware (not control-plane ARP proxy which limits to ~1K ARP/sec on some platforms)
>
> Example qualifying platforms: Cisco Nexus 9300-GX (400G, 256K MACs, hardware VXLAN), Juniper QFX10008 (chassis form factor for 400Gbps+ with EVPN multi-site support), Arista 7800 series. BGW is not a job for older platforms designed for L2 aggregation — it requires hardware-accelerated VXLAN encap/decap at full line rate.

## Summary: Key Design Decisions

| Decision | GlobalTrade Choice | Rationale |
|----------|-------------------|-----------|
| DCI underlay | VXLAN/IP (dark fiber primary, MPLS for DR) | Own the fiber for latency control; carrier for long-haul |
| BGW placement | Dedicated BGW tier (not spine) | Independent failure domain for DCI and DC fabric |
| BGW redundancy | Active-active, anycast loopback | Full bandwidth utilization, sub-second failover |
| BUM handling | Ingress replication | No underlay multicast dependency at current 3-site scope; re-evaluate multicast-assisted options as site count grows |
| L2 stretch scope | 48 VLANs, explicit allowlist at BGW | Compliance isolation, reduce failure domain |
| DR connectivity | L3 Type-5 only, no L2 stretch | RTT too high for L2; DR is warm standby, not live |
| Inter-VRF routing | Firewall-based, not route leak | Internal security/compliance policy: all cross-VRF traffic logged and inspected |
| ARP suppression | Enabled on all stretched EVIs | Eliminates ARP broadcast amplification across 280km DCI |
| Split-brain handling | Network withdraws routes; application decides quorum | Network cannot safely pick a winner; application run-book owns DR activation |
