# 12.3 — Design Case Study: 5G Mobile Backhaul / Transport Network

> **Level:** Internet Expert / IE-SP  
> **Prerequisites:** Module 6 (Segment Routing), Module 10 (Slicing, FlexE, 5G Xhaul)

---

## Overview

> Mobile backhaul is where everything in this guide converges: SR-MPLS transport, timing distribution, network slicing, QoS, and fast protection — all under the strictest latency and reliability constraints in carrier networking. This case study designs a complete 5G transport network from cell tower to core, for a regional operator upgrading from 4G to 5G.

**Scenario: Prairie Wireless**

Prairie Wireless is a regional mobile operator in the US Midwest serving 2.8 million subscribers across Kansas, Nebraska, and Iowa. They operate 1,200 cell sites today (all 4G LTE with monolithic eNBs) and need to deploy 5G NR across their top 400 sites within 18 months, with full network coverage within 3 years.

**Current state:**
- 1,200 macro cell sites with CPRI fronthaul over dedicated dark fiber (point-to-point wavelengths)
- Backhaul: Carrier Ethernet pseudowires (VPWS) over a flat MPLS/LDP backbone
- 14 aggregation sites, 4 regional hub sites, 2 mobile core locations (Omaha, Kansas City)
- Timing: GPS at every cell site (1,200 GPS receivers — maintenance nightmare)
- No SR, no network slicing, no eCPRI capability
- Transport network designed for 4G peak: ~150 Mbps per cell site average

**Business drivers:**
- 5G eMBB launch (sub-6 GHz, 100 MHz carriers, 64T64R mMIMO at tier-1 sites)
- Enterprise 5G slicing contracts signed for two customers: a precision agriculture IoT provider (mIoT) and a regional hospital network (URLLC for remote surgery)
- C-RAN centralization: consolidate O-DU processing at 14 aggregation sites (cost savings on site power/cooling)
- Eliminate per-site GPS: move to PTP/SyncE network-based timing (reduce OPEX by ~$800K/year)

**What you'll design:** A complete xhaul transport network — fronthaul, midhaul, backhaul — with SR-MPLS transport, PTP/SyncE timing, network slicing, and QoS differentiation for URLLC, eMBB, and mIoT traffic classes.

## Current State: Why the 4G Transport Network Can't Carry 5G

### Problem 1: CPRI Fronthaul Doesn't Scale to mMIMO

```
Current: Point-to-point CPRI over dedicated wavelengths per cell site
4G (4T4R, 20 MHz FDD): ~2.5 Gbps CPRI per sector × 3 sectors = ~7.5 Gbps/site
Handled by: 10G dedicated wavelength per site — one fiber pair, one λ

5G target (64T64R mMIMO, 100 MHz TDD):
  CPRI Option 8: ~157 Gbps per sector × 3 sectors = ~471 Gbps/site
  Problem: No affordable wavelength solution at 471 Gbps per tower

eCPRI Option 7-2x: ~10-25 Gbps per sector × 3 sectors = ~30-75 Gbps/site
  Feasible: 4×25GE eCPRI links per site (100 Gbps aggregate)
```

The shift from CPRI to eCPRI is not optional — it's a physics constraint. Prairie Wireless must deploy packet-based eCPRI fronthaul for all 5G sites. (See Module 10.3 for the CPRI vs eCPRI bandwidth analysis and Option 7-2x split details.)

### Problem 2: Flat LDP Backbone Has No Traffic Engineering

```
Current: 18 P routers in flat OSPF area 0, LDP for label distribution
All 1,200 cell sites backhaul through the same MPLS core — no path differentiation
A remote surgery session and a firmware update share the same best-effort forwarding

5G requirement: URLLC needs < 1ms transport latency, 99.999% reliability
Current backbone: average transport latency 8-15ms, no fast reroute, no SLA differentiation
```

LDP follows IGP shortest path — you get one path, no TE, no per-slice steering. RSVP-TE was evaluated and rejected due to signaling scale (1,200 sites × multiple LSPs = state explosion on P routers). SR-MPLS eliminates both problems: stateless core forwarding with SR-TE policies for per-slice path selection.

### Problem 3: GPS at Every Site Is Unsustainable

```
Current: 1,200 GPS receivers, one per cell site
Annual maintenance: ~$670/site (GPS antenna replacement, cable weathering, survey calibration)
Total: ~$800K/year just for timing infrastructure
Failure mode: GPS antenna cable degrades → timing drift → call drops → NOC dispatches truck roll

4G tolerance: ±1.5 µs phase alignment (FDD — frequency sync is primary requirement)
5G requirement: ±130 ns phase alignment for TDD massive MIMO (beamforming coherence)
  GPS alone often can't maintain ±130 ns at cell site (multipath, antenna placement, cable delay)
```

Network-based timing (PTP with SyncE) from centrally placed GNSS grandmaster clocks provides better accuracy, lower OPEX, and eliminates 1,200 points of GPS failure. This requires every transport switch to be a PTP boundary clock — a fundamental change to the network architecture.

## Network Topology Design

### Four-Tier Converged Xhaul Architecture

Prairie Wireless adopts a converged xhaul topology where fronthaul, midhaul, and backhaul share the same physical infrastructure with logical isolation (see Module 10.3 for the converged xhaul concept).

```
Tier 0: Cell Site        Tier 1: Pre-Aggregation    Tier 2: Aggregation     Tier 3: Core
┌──────────┐             ┌──────────┐               ┌──────────┐           ┌──────────┐
│  O-RU    │  eCPRI      │  Pre-Agg │   Midhaul     │  Agg Hub │ Backhaul │  Mobile  │
│  (tower) │────25GE────│  Switch  │───100GE──────│  Router  │──400GE──│  Core    │
│          │             │  (leaf)  │               │  (O-CU)  │          │ (Omaha/  │
│  O-RU    │────25GE────│          │               │          │          │  KC)     │
│  O-RU    │────25GE────│          │               │          │          │          │
└──────────┘             └──────────┘               └──────────┘           └──────────┘
                          ↑ O-DU co-located          ↑ O-CU co-located
                          ↑ PTP Boundary Clock       ↑ PTP Boundary Clock
                          ↑ SyncE node               ↑ PTP Grandmaster (some)

Cell Sites: 1,200 (400 5G Phase 1)    Pre-Agg: 120 sites    Agg Hubs: 14    Core: 2
Fiber: 25GE cell→pre-agg             100GE rings            100G/400G mesh   400G
       (dedicated or PON)            (pre-agg→agg)          (agg→core)       (core↔core)
```

### Tier 0 — Cell Site

Each 5G cell site has 3 sectors, each with a 64T64R mMIMO O-RU panel.

```
Per-site connectivity:
  3× O-RU panels → 4× 25GE eCPRI uplinks (total 100 Gbps per site)
  eCPRI traffic: Option 7-2x split, Ethernet-encapsulated
  Additional: 1× 10GE for site management, legacy 4G backhaul during migration

Cell site switch: Layer 2/3 capable, PTP-aware (transparent clock minimum)
  Example platforms: Nokia 7220 IXR-D1, Juniper ACX710, Cisco NCS 540 (small form factor)
  Role: aggregate eCPRI flows from O-RUs, apply CoS marking, pass PTP
```

### Tier 1 — Pre-Aggregation (O-DU Hosting)

120 pre-aggregation sites, each serving ~10 cell sites. This is where C-RAN O-DU processing is centralized.

```
Pre-agg site equipment:
  - O-DU servers (COTS x86 with FPGA/GPU acceleration)
  - Pre-agg switch/router: 48×25GE (cell-facing) + 4×100GE (agg-facing)
  - PTP boundary clock (ITU-T G.8275.1 profile)
  - SyncE on all interfaces

Fiber topology: star from cell sites to pre-agg (dedicated fiber or passive WDM)
  Maximum fiber distance cell→pre-agg: 20 km (100 µs one-way latency budget, see below)
```

**Fronthaul latency budget (cell site to O-DU):**

```
Total budget: ~100-150 µs one-way (3GPP / O-RAN Alliance target)

Breakdown:
  Fiber propagation: 5 µs/km × 20 km max    = 100 µs
  Switch serialization (25GE): ~5 µs           =   5 µs
  eCPRI encap/decap overhead:                  =   5 µs
  Queuing (must be near-zero):                 =   0-5 µs
  ─────────────────────────────────────────────────────
  Total:                                       = 110-115 µs (within budget at 20 km)

At 15 km average:  75 + 5 + 5 + 5 = 90 µs  ← comfortable margin
At 25 km:          125 + 5 + 5 + 5 = 140 µs ← barely within budget, risky
```

This drives a hard design rule: **no cell site may be more than 20 km from its pre-aggregation site.** Sites beyond this distance must run local O-DU (D-RAN mode) with midhaul/backhaul only.

### Tier 2 — Aggregation Hub (O-CU Hosting)

14 aggregation hubs, each serving ~8-10 pre-aggregation sites. O-CU-CP and O-CU-UP are deployed here.

```
Agg hub equipment:
  - O-CU servers (COTS, less latency-sensitive than O-DU)
  - Aggregation router: Juniper MX304, Cisco NCS 5504, Nokia 7750 SR-1
  - PTP grandmaster clock (4 of 14 sites) or PTP boundary clock (remaining 10)
  - 100GE ring connectivity between pre-agg sites
  - 100G/400G uplinks to core

Inter-agg topology: partial mesh with ring protection
  - Western ring: 6 agg sites (NE, KS west)
  - Eastern ring: 8 agg sites (KS east, IA)
  - Cross-connect: 2× 100GE between rings at Topeka hub
```

### Tier 3 — Mobile Core

Two core sites: Omaha (primary) and Kansas City (DR + active load sharing).

```
Core equipment:
  - 5G core functions: UPF, AMF, SMF (containerized on Kubernetes)
  - Core routers: Juniper MX10008 or Cisco ASR 9906 (400GE capable)
  - PTP grandmaster clocks (GNSS-fed, redundant)
  - Peering: internet transit, IXP, enterprise VPN hand-off

Core interconnect: 2× 400GE diverse fiber paths (Omaha ↔ Kansas City, ~300 km)
```

## SR-MPLS Transport Design

### Why SR-MPLS (Not SRv6)

Prairie Wireless chose SR-MPLS over SRv6 for this deployment:

- **Hardware:** Pre-agg and cell-site switches are small-form-factor platforms. Many support SR-MPLS natively but have limited or no SRv6 dataplane support (SRv6 requires IPv6 header chain processing that stresses low-end ASICs).
- **MTU:** SRv6 SRH adds ~40+ bytes per SID. On a fronthaul path with tight MTU budgets (eCPRI frames are latency-sensitive and should not be fragmented), SR-MPLS 4-byte labels are far more efficient.
- **Operational readiness:** Prairie Wireless engineering team has MPLS expertise. SRv6 introduces a parallel IPv6 forwarding plane they're not ready to operate.
- **Interop:** The existing 4G LDP backbone must coexist during migration. SR-MPLS coexists with LDP via the same LFIB; SRv6 is a different forwarding plane entirely.

(See Module 6.6 for SR migration strategies including LDP-to-SR coexistence.)

### IS-IS Underlay with SR-MPLS

```
IS-IS design:
  Level structure: L1/L2 hierarchy
    - Level-1 areas: one per aggregation hub region (14 L1 areas)
    - Level-2 backbone: agg hubs + core routers (16 L2 nodes)
    - Pre-agg and cell-site switches: L1-only (topology contained within region)

  Metric policy:
    25GE: metric 400
    100GE: metric 100
    400GE: metric 10
    (Inverse bandwidth — lower metric = preferred path)

  BFD: 3×50ms on all inter-site links (150ms detection)
  SPF timers: initial 50ms, secondary 200ms, max 5000ms
```

**SRGB and Node SID Allocation:**

```
SRGB: 16000–23999 (consistent across all SR-capable nodes)
SRLB: 15000–15999 (adjacency SIDs, local significance)

SID allocation scheme (by tier):
  Core routers:        16001–16010
  Agg hub routers:     16101–16199  (14 agg hubs, room for growth)
  Pre-agg switches:    16200–16499  (120 pre-agg sites)
  Cell site switches:  16500–17999  (1,200 cell sites — only SR-capable 5G sites)

Anycast SIDs:
  Omaha core (both routers):    16901
  Kansas City core (both):      16902
  Per-agg-hub anycast:          16801–16814 (both pre-agg-facing routers share SID)
```

Anycast SIDs are critical for redundancy: a pre-agg switch reaching for "the Omaha core" pushes SID 16901 — traffic goes to whichever Omaha core router is topologically closest. If one fails, TI-LFA reroutes to the survivor without changing the SID stack.

### TI-LFA Protection

```
Coverage target: 100% of backbone links (agg↔agg, agg↔core)
Pre-agg ring links: TI-LFA with node protection where topology permits
Cell site access links: protected via dual-homing (two uplinks to different pre-agg switches)

Configuration — IOS-XR (agg hub router):
router isis PRAIRIE
 interface HundredGigE0/0/0/0
  address-family ipv4 unicast
   fast-reroute per-prefix
   fast-reroute per-prefix ti-lfa
  !
 !

Configuration — Junos (agg hub router):
protocols {
    isis {
        interface et-0/0/0 {
            level 2 {
                post-convergence-lfa {
                    node-protection;
                }
            }
        }
    }
}
```

### Flex-Algo for Transport Slicing

Prairie Wireless uses three Flex-Algos to create isolated transport planes for different 5G traffic classes (see Module 10.1 for the slice-to-transport mapping concept):

```
Flex-Algo 128: "Low-Latency" — URLLC transport
  Metric: minimum delay (uses IS-IS delay measurement or static delay config)
  Constraint: exclude links with delay > 2ms (affinity "high-delay")
  Use: remote surgery, industrial automation

Flex-Algo 129: "High-Bandwidth" — eMBB transport
  Metric: IGP metric (bandwidth-proportional)
  Constraint: exclude links marked "capacity-constrained" (affinity)
  Use: consumer broadband, video streaming

Flex-Algo 130: "Ultra-Reliable" — mIoT + critical signaling
  Metric: IGP metric
  Constraint: include only links marked "dual-fiber-protected" (affinity)
  Use: IoT sensor networks, 5G control plane (N2/N3)
```

**Flex-Algo definition — IOS-XR (on all IS-IS nodes):**

```
router isis PRAIRIE
 flex-algo 128
  metric-type delay
  affinity exclude-any high-delay
  advertise-definition
 !
 flex-algo 129
  metric-type igp
  affinity exclude-any capacity-constrained
  advertise-definition
 !
 flex-algo 130
  metric-type igp
  affinity include-all dual-fiber-protected
  advertise-definition
 !
 interface HundredGigE0/0/0/0
  affinity flex-algo
   color dual-fiber-protected
  !
  address-family ipv4 unicast
   prefix-sid algorithm 128 index 301
   prefix-sid algorithm 129 index 401
   prefix-sid algorithm 130 index 501
```

**Flex-Algo definition — Junos:**

```

    isis {
        source-packet-routing {
            flex-algorithm 128 {

                    metric-type delay;
                    affinity {
                        exclude high-delay;
                    }
                }
                prefix-sid {

                }

        }
    }
}
segment-routing {
    affinity-map {
        high-delay bit-position 0;
        capacity-constrained bit-position 1;
        dual-fiber-protected bit-position 2;
    }
}

## eCPRI Fronthaul Design

### Packet-Based Fronthaul Architecture

Prairie Wireless deploys eCPRI (Option 7-2x split) for all 5G fronthaul. eCPRI frames are standard Ethernet — they can be switched, prioritized, and transported over the same infrastructure as midhaul and backhaul traffic.

```
eCPRI frame format:
┌──────────────┬──────────────┬──────────────┬──────────────┐
│  Ethernet    │  eCPRI       │  eCPRI       │  IQ Data /   │
│  Header      │  Common Hdr  │  Message     │  Payload     │
│  (14 bytes)  │  (4 bytes)   │  Type        │  (variable)  │
└──────────────┴──────────────┴──────────────┴──────────────┘

Transport options:
  a) Native Ethernet (L2) — eCPRI over VLAN-tagged Ethernet (Prairie's choice for fronthaul)
  b) Over UDP/IP — eCPRI encapsulated in UDP (for routed networks)
  c) Over GTP-U — rarely used, future 3GPP option
```

**Prairie Wireless fronthaul design:**

```
O-RU → Cell site switch → Pre-agg switch → O-DU server

  O-RU uplink: 25GE fiber, VLAN-tagged per antenna panel/sector
    Sector 1: VLAN 1001 (eCPRI plane A + plane B)
    Sector 2: VLAN 1002
    Sector 3: VLAN 1003
    Management: VLAN 100

  Cell site switch: L2 forwarding, strict-priority QoS for eCPRI VLANs
  Pre-agg switch: terminates eCPRI VLANs, hands off to local O-DU via internal fabric

QoS at cell site switch:
  Queue 7 (strict priority): eCPRI traffic (DSCP 46 / PCP 7)
  Queue 6 (strict priority): PTP timing (DSCP 48 / PCP 6)
  Queue 5 (shaped):          S1/N3 backhaul (DSCP 34)
  Queue 0-4 (WFQ):           management, OAM, best-effort
```

### FlexE for Fronthaul Bandwidth Isolation

At pre-aggregation sites where fronthaul and backhaul share the same 100GE uplinks toward aggregation hubs, Prairie Wireless uses FlexE channelization to provide hard bandwidth isolation (see Module 10.2 for FlexE architecture).

```
Pre-agg uplink: 100GE physical port to aggregation hub

FlexE channelization:
  FlexE Client 1: 50 Gbps — Fronthaul (eCPRI) — guaranteed, isolated
  FlexE Client 2: 30 Gbps — Midhaul (F1-C/F1-U) — guaranteed
  FlexE Client 3: 20 Gbps — Backhaul (N3 user plane) + management

Why FlexE here (not just QoS)?
  QoS provides scheduling priority but not bandwidth isolation.
  A backhaul traffic burst can increase queuing delay for fronthaul frames.
  With ~5 µs queuing budget in the fronthaul latency model, even a single
  deep-buffer event from backhaul traffic could violate the 150 µs budget.
  FlexE physically separates the calendar slots — backhaul traffic literally
  cannot consume fronthaul bandwidth, regardless of burst behavior.
```

**FlexE configuration (conceptual — IOS-XR, verify against platform/release):**

```
! Platform support required: NCS 5500/NCS 540 with FlexE-capable line cards
! Exact CLI varies by IOS-XR release — treat as implementation intent

controller FlexE0/0/0/0
 flexe-group 1
  phy 0
   100GE
  !
 !
 client 1
  calendar-slots 0-49    ! 50 slots × 1G = 50 Gbps
  description "Fronthaul-eCPRI"
 !
 client 2
  calendar-slots 50-79   ! 30 slots = 30 Gbps
  description "Midhaul-F1"
 !
 client 3
  calendar-slots 80-99   ! 20 slots = 20 Gbps
  description "Backhaul-N3"
 !
```

## PTP/SyncE Timing Distribution

### Why Network-Based Timing

5G TDD requires phase synchronization between all O-RUs in a coverage area. Without it, uplink/downlink slot boundaries across neighboring cells collide, causing massive interference. The requirement: **±130 ns phase alignment** between any two O-RUs with overlapping coverage (3GPP TS 38.104).

GPS at every site achieves this but at enormous operational cost. Network-based timing distributes synchronization from a small number of GNSS grandmaster clocks through the transport network using PTP (IEEE 1588v2) and SyncE.

### Timing Architecture

```
                          GNSS
                           │
                    ┌──────┴──────┐
                    │ Grandmaster │  (PTP GM + SyncE source)
                    │ Clock       │  ITU-T G.8272 Class B (±20 ns)
                    └──────┬──────┘
                           │ PTP + SyncE
                    ┌──────┴──────┐
                    │ Agg Hub     │  PTP Boundary Clock (G.8275.1)
                    │ Router      │  SyncE EEC Level 1 (G.8262)
                    └──────┬──────┘
                           │ PTP + SyncE
                    ┌──────┴──────┐
                    │ Pre-Agg     │  PTP Boundary Clock (G.8275.1)
                    │ Switch      │  SyncE EEC Level 1
                    └──────┬──────┘
                           │ PTP + SyncE
                    ┌──────┴──────┐
                    │ Cell Site   │  PTP Slave Clock → O-RU
                    │ Switch      │  SyncE EEC Level 1
                    └──────┴──────┘
                           │
                        O-RU (consumes PTP for TDD frame timing)

Error budget (G.8275.1 full on-path support):
  GM accuracy:              ±20 ns  (Class B)
  Per boundary clock hop:   ±5 ns   (each BC adds noise)
  Maximum hops GM→slave:    ~20     (G.8275.1 profile)
  Prairie Wireless hops:    3-4     (GM → agg → pre-agg → cell → O-RU)
  Accumulated error:        ±20 + (4 × 5) = ±40 ns ← well within ±130 ns
```

### Grandmaster Clock Placement

```
Prairie Wireless deploys 6 GNSS grandmaster clocks:
  Omaha Core:      2× GM (primary, redundant — GNSS diversity: GPS + Galileo)
  Kansas City Core: 2× GM (primary, redundant)
  Topeka Agg Hub:  1× GM (western ring — reduces hop count for western sites)
  Des Moines Agg:  1× GM (eastern ring)

Selection logic:
  Each BC selects best GM using BMCA (Best Master Clock Algorithm, IEEE 1588)
  Priority: local GM > adjacent GM > remote GM
  Holdover: if all GMs fail, BC enters holdover (OCXO oscillator, ~10 µs/day drift)
    Prairie Wireless OCXO holdover budget: maintain ±130 ns for ~1 hour
    (adequate for GNSS antenna repair or short outage)
```

### End-to-End Slice Path Example

A remote surgery session from a hospital connected to cell site KS-042:

```
Step 1: UE → O-RU (radio, not transport-visible)
Step 2: O-RU → O-DU (eCPRI fronthaul, VLAN 1001, strict priority queue)
        Path: Cell site KS-042 → Pre-agg site KS-PA-07 (25GE, FlexE client 1)
        Latency: ~90 µs

Step 3: O-DU → O-CU (F1-U midhaul, MPLS, Flex-Algo 128)
        Path: Pre-agg KS-PA-07 → Agg hub Topeka (100GE ring, TI-LFA protected)
        Latency: ~500 µs

Step 4: O-CU → UPF (N3 backhaul, MPLS, Flex-Algo 128, SR-TE URLLC policy)
        Path: Agg hub Topeka → Core Omaha (100GE, low-delay constrained path)
        Latency: ~800 µs

Total transport latency: ~1.4 ms (within URLLC < 5ms air-to-core budget)
  Note: 3GPP URLLC E2E target is application-dependent (1-10ms); transport
  portion is typically budgeted at 1-3ms depending on the use case.
```

## QoS Design

### Per-Hop Behavior Model

Prairie Wireless implements a 6-class QoS model across all transport tiers:

```
┌──────┬──────────────────┬───────┬─────────────┬───────────────────────────────┐
│Queue │ Traffic Class     │ DSCP  │ Scheduling  │ Bandwidth                     │
├──────┼──────────────────┼───────┼─────────────┼───────────────────────────────┤
│  7   │ PTP/SyncE timing │ CS7   │ Strict Prio │ < 1% (policed, small volume)  │
│  6   │ eCPRI fronthaul  │ EF    │ Strict Prio │ up to 60% (FlexE-backed)      │
│  5   │ URLLC user plane │ EF    │ Strict Prio │ 10% guaranteed                │
│  4   │ eMBB user plane  │ AF31  │ WFQ         │ 20% guaranteed, burst to 50%  │
│  3   │ mIoT user plane  │ AF11  │ WFQ         │ 5% guaranteed                 │
│  2   │ Signaling/OAM    │ CS6   │ WFQ         │ 5% guaranteed                 │
│  1   │ Management       │ CS2   │ WFQ         │ 5% guaranteed                 │
│  0   │ Best-effort      │ BE    │ WFQ         │ remainder                     │
└──────┴──────────────────┴───────┴─────────────┴───────────────────────────────┘

Note: eCPRI (Queue 6) and URLLC (Queue 5) both use EF marking but are distinguished
by ingress interface classification at pre-agg switches. On fronthaul-only segments,
eCPRI owns the strict-priority class exclusively.
```

## Redundancy and Protection

### Link Protection: TI-LFA + Dual-Homing

```
Tier 3 (Core):     Dual 400GE diverse paths, TI-LFA node protection
                    Anycast SID 16901/16902 — core router failure is transparent

Tier 2 (Agg rings): 100GE ring topology, TI-LFA link + node protection
                     Ring failure: traffic reroutes around ring in < 50ms
                     Ring double-failure: SR-TE policy falls back to alternate ring via cross-connect

Tier 1 (Pre-agg):  Dual-homed to two agg hub routers (100GE each)
                    Primary/backup via IS-IS metric differential
                    Failure: BFD detects in 150ms, traffic shifts to backup uplink

Tier 0 (Cell site): Dual 25GE uplinks to two different pre-agg switches (where fiber available)
                     Single-homed sites (rural): single 25GE — no transport redundancy
                     (Acceptable: rural mMIMO sites have lower traffic; SLA is relaxed)
```

### Node Protection: Anycast SIDs

Anycast SIDs provide seamless node-level redundancy without path recalculation:

```
Example: Omaha core routers CR-OMA-1 (16001) and CR-OMA-2 (16002)
Both also advertise anycast SID 16901

Traffic from any agg hub to "Omaha core":
  Label stack: [..., 16901]
  IS-IS sees two equal-cost paths to SID 16901 → ECMP load balancing
  
CR-OMA-1 fails:
  IS-IS withdraws CR-OMA-1's advertisement of prefix for SID 16901
  TI-LFA pre-computed backup immediately shifts to CR-OMA-2
  No SR-TE policy change needed — SID 16901 still valid, points to CR-OMA-2
  Convergence: < 50ms (BFD + TI-LFA)
```

### Timing Protection

```
PTP grandmaster failure:
  BMCA selects next-best grandmaster (e.g., if Omaha GM fails, KC GM takes over)
  BC holdover bridges the gap (~seconds to reconverge PTP)
  SyncE provides continuous frequency — no frequency disruption during PTP reconvergence

All-GM failure (catastrophic):
  BCs enter holdover on local OCXO
  Holdover accuracy: ±130 ns maintained for ~1 hour (high-quality OCXO)
  Alarm: immediate P1 alert to NOC
  Response: portable GNSS grandmaster deployed (2-hour response SLA)
```

## Capacity Planning and Growth Model

### Current vs. 5G Bandwidth Requirements

```
4G LTE per site (current):
  Backhaul only: ~150 Mbps average, ~500 Mbps peak
  No fronthaul or midhaul on transport network (CPRI on dedicated wavelengths)

5G NR per site (target, 64T64R mMIMO, 100 MHz):
  Fronthaul (eCPRI): ~50 Gbps sustained (3 sectors × ~17 Gbps average)
  Midhaul (F1-U/F1-C): ~5 Gbps peak
  Backhaul (N3): ~3 Gbps peak (user-plane throughput after RAN processing)
  Total per site: ~58 Gbps peak transport demand

Growth factor over 4G: 58 Gbps / 0.5 Gbps ≈ 116x per site
```

### Aggregation Scaling

```
Pre-agg site (serves 10 cell sites):
  Fronthaul aggregate: 10 × 50 Gbps = 500 Gbps (but statistically multiplexed)
  Statistical mux factor: ~0.6 (not all sectors peak simultaneously)
  Effective fronthaul: ~300 Gbps
  Uplink to agg hub: 4× 100GE (400 Gbps) with FlexE channelization
    300 Gbps fronthaul + 50 Gbps midhaul + 30 Gbps backhaul = 380 Gbps

Agg hub (serves 10 pre-agg sites):
  Aggregate from pre-agg: 10 × 380 Gbps = 3.8 Tbps (before stat mux)
  Midhaul + backhaul stat mux: ~0.4 (O-DU processing absorbs fronthaul locally)
  Effective agg-to-core: ~200 Gbps midhaul/backhaul per agg hub
  Uplink to core: 2× 400GE (800 Gbps capacity)
  Utilization: 200/800 = 25% — headroom for 3-year growth
```

### 3-Year Growth Model

```

Pre-agg→Agg:

## Internet Expert / Internet Expert Design Review Questions

**Q1: Fronthaul Latency**

Prairie Wireless wants to extend C-RAN centralization to a cell site that is 28 km from the nearest pre-aggregation site. The fiber route is 32 km due to road-following conduit. Calculate the one-way fronthaul latency and explain why this site cannot use centralized O-DU. What is the recommended design for this site?

> **Discussion:** Fiber propagation: 32 km × 5 µs/km = 160 µs. Add switching (5 µs) + eCPRI overhead (5 µs) = 170 µs — exceeds the 150 µs fronthaul budget. This site must deploy a local O-DU at the cell site (D-RAN mode). The transport network carries only midhaul (F1-U/F1-C from local O-DU to O-CU at the agg hub) and backhaul. Midhaul latency budget is ~1-3 ms, so 170 µs propagation is easily within budget. The cell site switch configuration changes: eCPRI traffic terminates locally at the O-DU, and only midhaul/backhaul traffic traverses the transport network. FlexE channelization at this site's pre-agg uplink should be re-profiled (no fronthaul allocation needed, more bandwidth for midhaul/backhaul).

**Q2: Timing Failure**

The Omaha GNSS grandmaster loses satellite lock due to a GPS constellation issue affecting the central US. The Kansas City GM is unaffected. Describe the timing reconvergence sequence and calculate how long Prairie Wireless has before TDD phase alignment degrades beyond ±130 ns.

> **Discussion:** Sequence: (1) Omaha GM detects loss of satellite lock, sets PTP clockClass to degraded (e.g., class 52 → 187 or platform-specific holdover class). (2) BMCA runs on all BCs — within seconds, BCs that were using Omaha GM as best master re-evaluate. Kansas City GM (still class 6, locked) wins BMCA selection. (3) BCs in the western ring reconverge PTP hierarchy toward KC GM — additional hop count but still within G.8275.1 budget (≤ 20 hops). (4) During BMCA reconvergence (seconds to tens of seconds), BCs use holdover from SyncE (frequency) and local OCXO (phase). OCXO holdover at ±1 µs/day drift: to stay within ±130 ns, maximum holdover time is roughly 130 ns / (1 µs / 86400 s) ≈ 11.2 seconds of uncorrected phase drift. However, SyncE maintains frequency — phase drift depends on the frequency offset quality. With EEC Level 1 SyncE and a good OCXO, practical holdover is minutes to hours for ±130 ns. The 11.2-second calculation assumes worst-case free-running OCXO with no frequency assist — SyncE significantly extends this. Bottom line: PTP BMCA reconvergence to KC GM (seconds) completes well before phase degrades, assuming the transport path to KC is intact.

**Q3: Slice Isolation Failure**

A network engineer accidentally removes the `affinity exclude-any high-delay` constraint from Flex-Algo 128 on two of the 14 aggregation hub routers. What happens to URLLC traffic? How would you detect this misconfiguration before it causes a patient safety incident?

> **Discussion:** With the affinity constraint removed on two nodes, those nodes compute Flex-Algo 128 topology differently — they include high-delay links that other nodes exclude. This creates an inconsistent Flex-Algo view: traffic entering the network at a correctly configured node avoids high-delay links, but if the SR-TE path transits a misconfigured node, that node may forward URLLC traffic onto a high-delay path. Worst case: URLLC packets traverse a satellite backhaul or long-haul link with 15+ ms delay, violating the < 1ms transport SLA. Detection: (1) Automation: Flex-Algo definitions should be deployed via NETCONF templates with pre-commit validation — any node deviating from the template triggers a config compliance alert. (2) Telemetry: monitor per-Flex-Algo reachability (`show segment-routing flex-algo 128`) — if the reachable node set changes unexpectedly, alert. (3) Active probing: SR-TE OAM (MPLS echo with Flex-Algo SID stack) measures actual path latency — alert if URLLC probe latency exceeds threshold. (4) Prevention: treat Flex-Algo definitions as immutable infrastructure — deploy from CI/CD pipeline, never via manual CLI.

**Q4: Capacity Planning**

Prairie Wireless signs a contract to provide 5G Fixed Wireless Access (FWA) in rural areas, adding 50 Mbps average per household to 10,000 homes across 200 cell sites. These sites share transport infrastructure with mobile 5G. Calculate the additional transport load and determine which tier becomes the bottleneck first.

> **Discussion:** Additional load: 10,000 homes × 50 Mbps = 500 Gbps aggregate FWA demand. Distributed across 200 sites: ~2.5 Gbps additional backhaul per site. This doesn't affect fronthaul (eCPRI bandwidth is radio-config-dependent, not user-traffic-dependent). Midhaul impact: O-DU→O-CU F1-U traffic increases by ~2.5 Gbps per site. Backhaul impact: N3 user plane increases by ~2.5 Gbps per site. Pre-agg level: 10 sites × 2.5 Gbps = 25 Gbps additional per pre-agg. Current FlexE backhaul allocation: 20 Gbps per pre-agg uplink — **this is the bottleneck**. FWA traffic alone would exceed the backhaul FlexE channel. Fix: re-profile FlexE allocation (reduce fronthaul share if statistical mux allows, or add uplink capacity). Agg hub level: 10 pre-agg × 25 Gbps = 250 Gbps additional per agg hub — current 200 Gbps utilization + 250 Gbps = 450 Gbps, still within 800 Gbps capacity but approaching 56% utilization. Year 1 upgrade timeline accelerates.

**Q5: Migration Strategy**

Prairie Wireless must keep all 1,200 4G sites operational during the 18-month 5G rollout. The existing LDP backbone must coexist with the new SR-MPLS transport. Describe the migration sequence and identify the highest-risk step.

> **Discussion:** Migration sequence: (1) Deploy IS-IS alongside existing OSPF (ships-in-the-night, both IGPs active). IS-IS carries SR prefix-SIDs; OSPF continues to support LDP. (2) Enable SR-MPLS on all P routers (agg hubs, core) — SR labels are allocated but LDP labels remain active. Configure `segment-routing mpls sr-prefer` so SR labels are preferred when available. (3) Verify: every PE can reach every other PE via both SR and LDP labels (`show mpls forwarding` — both label sources present). (4) Migrate services one tier at a time: core first, then agg, then pre-agg. Each tier: verify SR forwarding, then disable LDP on those nodes. (5) Decommission OSPF after all nodes run IS-IS + SR-MPLS exclusively.
>
> Highest-risk step: Step 2 — enabling SR-MPLS globally. If the SRGB range overlaps with LDP's dynamic label range, label collisions cause forwarding black holes. Before enabling SR: verify `show mpls label range` on every router, ensure SRGB (16000-23999) does not overlap with LDP dynamic range (typically starts much lower). Also verify that IS-IS SR prefix-SID advertisements are consistent — a single node advertising a conflicting SID index creates network-wide forwarding ambiguity. Mitigation: stage SR enablement in a maintenance window, verify LFIB consistency at each step, have a rollback plan (disable SR prefix-SIDs, revert to LDP-only).

## Key Design Decisions Summary

| Decision | Prairie Wireless Choice | Rejected Alternative | Rationale |
|----------|------------------------|---------------------|-----------|
| Transport protocol | SR-MPLS | SRv6 | Hardware support on small platforms, MTU efficiency, operational readiness |
| Fronthaul | eCPRI Option 7-2x (Ethernet) | CPRI (TDM) | 10:1 bandwidth reduction, packet switching, statistical multiplexing |
| RAN architecture | C-RAN (centralized O-DU at pre-agg) | D-RAN | Cost savings on site power/cooling, enables CoMP, easier O-DU lifecycle |
| Timing | PTP/SyncE (G.8275.1) | Per-site GPS | $800K/year OPEX savings, better accuracy (±40 ns vs ±100+ ns GPS) |
| Fronthaul isolation | FlexE channelization | QoS-only | Hard bandwidth guarantee, zero queuing interference from backhaul bursts |
| Slice transport | Flex-Algo (3 algorithms) | VRF-per-slice | Topology-aware path computation, delay-optimized URLLC paths |
| Core redundancy | Anycast SIDs | VRRP/HSRP | Stateless, topology-independent, sub-50ms with TI-LFA |
| IGP | IS-IS L1/L2 | OSPF | Single process for IPv4/IPv6, natural hierarchy, SR-MPLS native support |
| Fast reroute | TI-LFA | LFA/RLFA | Any-topology coverage, pre-computed backup with bounded label stack |
| QoS model | 6-class with strict-priority eCPRI/URLLC | Flat best-effort | Deterministic latency for fronthaul and URLLC contractual SLAs |
