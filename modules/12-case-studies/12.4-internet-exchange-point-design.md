# 12.4 — Design Case Study: Internet Exchange Point (IXP) Design

> **Level:** Internet Expert / IE-SP  
> **Prerequisites:** Module 2 (IS-IS/OSPF), Module 3 (BGP), Module 4 (Internet Peering), Module 8 (EVPN)

---

## Overview

> An IXP is a shared Layer 2 fabric where networks interconnect and exchange traffic directly, bypassing transit providers. Designing one from scratch requires Ethernet switching expertise, BGP route server architecture, peering policy enforcement, RPKI/IRR filtering, and physical infrastructure planning — all disciplines that IE-SP candidates must synthesize. This case study builds a greenfield IXP from site selection to first peering session.

**Scenario: Basin Internet Exchange (BasinIX)**

Basin City is a fast-growing metro area (population 1.4 million) in the western United States. The city hosts 18 local ISPs, 3 regional carriers, 2 content delivery networks, and a growing cluster of cloud on-ramps — but there is no local IXP. All inter-network traffic between Basin City networks currently trombones 600 miles to the nearest major IXP. A round trip that should be 2ms locally adds 12ms of latency and transit cost.

**Current state:**
- 18 local ISPs pay transit providers to reach each other — even for local traffic
- A video call between two Basin City residents traverses a backbone 600 miles away and back
- Local CDN cache nodes exist but can't serve local eyeball networks directly (no local peering)
- Regional carriers pass through Basin City fiber but have no local interconnection point
- Monthly transit costs for local ISPs average $8,000–$25,000 each — much of it is local traffic hairpinning

**Business case:**
- Founding consortium: 6 local ISPs + 1 regional carrier commit to initial membership
- Target: 25 members within 12 months, 60 within 3 years
- Estimated local traffic exchange: 40 Gbps at launch, 200 Gbps at year 3
- Non-profit model: membership fees cover infrastructure costs, no per-Mbps billing
- Two colocation facilities in Basin City with dark fiber between them (dual-site from day one)

**What you'll design:** A complete IXP — Layer 2 peering fabric, route server architecture, peering policies, RPKI/IRR filtering, physical infrastructure, security controls, and growth model.

**Physical design decisions:**

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Number of sites | 2 (active-active) | Single-site failure must not kill the exchange |
| Fiber between sites | 2 diverse paths, 4 pairs each | Physical path diversity; northern and southern conduit routing |
| Meet-me room model | Carrier-neutral, shared patch panels | Any member can cross-connect without bilateral negotiation with the facility |
| IXP cage | Locked cage per site, IXP staff only | Members terminate in meet-me room; IXP runs patch cables into its cage |
| Cross-connect type | Single-mode fiber (SMF), LC/UPC | Standard for 1G/10G/100G optics; no multimode in production |
| Power | Dual A+B feeds per rack, 30A 208V | N+1 power; UPS + generator backed by facility SLA |

### Optics and Port Speeds

BasinIX offers three port speeds at launch:

| Port Speed | Optic | Reach | Monthly Port Fee |
|------------|-------|-------|------------------|
| 1 GE | 1000BASE-LX (SFP) | 10 km SMF | $250 |
| 10 GE | 10GBASE-LR (SFP+) | 10 km SMF | $500 |
| 100 GE | 100GBASE-LR4 (QSFP28) | 10 km SMF | $2,000 |

Members supply their own optics on their router side. BasinIX supplies matched optics on the IXP switch side. All cross-connects are SMF — no copper, no multimode.

## Layer 2 Peering Fabric Design

### Design Choice: Traditional VLAN vs EVPN-VXLAN

Two architectures are viable for an IXP peering LAN:

**Option A — Traditional VLAN-Based Fabric (Selected for BasinIX)**

```
Simple: One VLAN = one broadcast domain = the peering LAN
All member ports are access ports on VLAN 100
Switches at both sites are interconnected via 802.1Q trunks
No overlay, no control plane complexity
```

**Option B — EVPN-VXLAN Fabric**

```
Modern: EVPN learns MACs via BGP, VXLAN encapsulates L2 frames
Scales beyond traditional spanning tree limits
Enables per-member VNI isolation if needed
Adds operational complexity: BGP sessions between switches, VTEP management
```

**Why BasinIX chooses Option A:**

| Factor | VLAN-Based | EVPN-VXLAN |
|--------|-----------|------------|
| Member count at launch | 7 (trivial for flat L2) | Overkill |
| Operational complexity | Minimal — any network engineer can troubleshoot | Requires EVPN expertise on IXP ops team |
| Spanning tree risk | Managed via RSTP + BPDU guard + loop guard | Eliminated (no STP in VXLAN fabric) |
| Scale ceiling | ~500 MAC addresses (sufficient to 200+ members) | Thousands of MACs across multiple VNIs |
| Vendor flexibility | Any managed switch vendor | Requires EVPN-capable switches (limits options) |
| Migration path | Can migrate to EVPN-VXLAN later if needed | Already there |

**When to choose EVPN-VXLAN:** IXPs exceeding 200 members, multiple peering VLANs, or those offering private VLANs / VPLS services benefit from EVPN-VXLAN. Large IXPs (DE-CIX, AMS-IX, LINX) have migrated to EVPN-VXLAN fabrics for scale reasons. BasinIX plans this as a Phase 2 migration if membership exceeds 150.

### Ethernet Fabric: Switch Topology

BasinIX deploys four switches — two per site — in a full mesh ring:

```
          DC-A                              DC-B
    ┌────────────┐                    ┌────────────┐
    │            │                    │            │
    │   SW-A1 ──────── 100GE ────────── SW-B1    │
    │    │  ╲    │                    │   ╱  │    │
    │    │   ╲   │                    │  ╱   │    │
    │  LAG    ╲  │                    │ ╱  LAG    │
    │    │     ╲ │                    │╱    │     │
    │   SW-A2 ──────── 100GE ────────── SW-B2    │
    │            │                    │            │
    └────────────┘                    └────────────┘
         │  │                              │  │
      Member ports                      Member ports
      (1G/10G/100G)                     (1G/10G/100G)

    Intra-site: 100GE LAG between SW-A1↔SW-A2, SW-B1↔SW-B2
    Inter-site: 100GE links on diverse fiber paths
    All four switches carry VLAN 100 (peering LAN)
```

**Spanning tree avoidance strategy:**

IXPs and spanning tree are a dangerous combination. A single misconfigured member port sending BPDUs can take down the peering LAN.

```
! BasinIX switch configuration (Junos QFX5120 example)
! ─── RSTP with aggressive protection ───

protocols {
    rstp {
        bridge-priority 4096;          /* IXP switches are always root */
        interface ge-0/0/0-47 {        /* All member-facing ports */
            edge;                       /* Portfast equivalent */
            bpdu-block;                 /* Drop + disable port on BPDU receipt */
        }
        interface et-0/0/48-55 {       /* Inter-switch links */
            /* No edge, no bpdu-block — these are legitimate trunk ports */
        }
    }
}
```

**Key protections on member-facing ports:**

| Protection | Purpose | Action on Violation |
|------------|---------|-------------------|
| BPDU guard | Prevent member from participating in STP | Disable port |
| Loop guard | Detect unidirectional link failures on trunks | Block port |
| Storm control | Rate-limit broadcast/multicast/unknown unicast | Police to 1% of port speed |
| MAC limit | Prevent MAC flooding attacks | Limit to 1 MAC per port (or per LAG) |
| DHCP snooping | Prevent rogue DHCP servers on peering LAN | Drop DHCP server packets from member ports |

### Member Connection Model

Members connect to BasinIX via one of three models:

**Model 1: Single-homed (1 port, 1 site)**
```
Member Router ──── cross-connect ──── SW-A1 (or SW-B1)
Simple, cheap, no redundancy
Suitable for: small ISPs, initial connections
```

**Model 2: Dual-homed same site (LAG, 1 site)**
```
Member Router ──── 2× cross-connect ──── SW-A1 + SW-A2 (MC-LAG)
Link redundancy within one site
Suitable for: mid-size members wanting link protection
```

**Model 3: Dual-homed multi-site (1 port per site)**
```
Member Router-1 @ DC-A ──── SW-A1
Member Router-2 @ DC-B ──── SW-B1
Full site redundancy (member runs own HA)
Suitable for: large members, content networks, transit providers
```

## Peering LAN Addressing

### IPv4 + IPv6 Address Plan

BasinIX operates a single peering LAN with dual-stack addressing:

| Property | IPv4 | IPv6 |
|----------|------|------|
| Peering LAN prefix | 198.51.100.0/24 | 2001:db8:basx::/64 |
| Usable addresses | .1 – .254 (254 hosts) | /64 (effectively unlimited) |
| Route server 1 | 198.51.100.1 | 2001:db8:basx::1 |
| Route server 2 | 198.51.100.2 | 2001:db8:basx::2 |
| Reserved (infra) | .1 – .10 | ::1 – ::a |
| Member allocation | .11 – .254 | ::b – ::fe (also accept SLAAC/EUI-64) |
| VLAN ID | 100 | 100 (same VLAN, dual-stack) |

**Address assignment policy:**
- Each member gets one IPv4 and one IPv6 address per physical port
- Addresses are assigned sequentially from the member pool
- Members MUST configure the assigned address only — no additional IPs on the peering interface
- /24 IPv4 provides capacity for 244 member ports (sufficient for 3+ year growth plan)

**Why /24 IPv4 and /64 IPv6:**
- /24 is the minimum routeable prefix (important if the peering LAN prefix appears in routing artifacts)
- /64 is standard for Ethernet segments and required for SLAAC (though BasinIX uses static assignment)

## Route Server Architecture

### Why Route Servers

Without route servers, N members require up to N×(N-1)/2 bilateral BGP sessions. With 25 members, that's 300 sessions to configure. Route servers provide multilateral peering: each member peers with the route servers, and the route servers re-advertise everyone's routes to everyone else (subject to policy).

```
Without route servers (25 members):     With route servers (25 members):
  300 bilateral BGP sessions needed       25 BGP sessions to RS1
  Each member configures 24 peers         25 BGP sessions to RS2
  Adding member #26 = 25 new sessions     Adding member #26 = 2 new sessions

  Member-A ─── BGP ─── Member-B          Member-A ─── BGP ─── RS1
  Member-A ─── BGP ─── Member-C          Member-A ─── BGP ─── RS2
  Member-A ─── BGP ─── Member-D              RS1 ─── BGP ─── Member-B
  Member-B ─── BGP ─── Member-C              RS1 ─── BGP ─── Member-C
  ...300 sessions...                         ...50 sessions total...
```

**Route servers do NOT forward traffic.** They are control-plane only. Data-plane traffic flows directly between member routers across the peering LAN. The route server simply tells Member-A "Member-B has prefix 203.0.113.0/24 — and the next-hop is Member-B's peering LAN IP." BGP next-hop is preserved (not changed to the RS address).

### BIRD 2 Route Server Configuration

BasinIX runs two route servers on dedicated Linux servers (no routing on the IXP switches). BIRD 2 is chosen for its robust filtering language, RPKI support, and wide deployment at production IXPs (Euro-IX reports ~60% BIRD adoption among IXPs).

**RS1: 198.51.100.1 / 2001:db8:basx::1 (DC-A)**  
**RS2: 198.51.100.2 / 2001:db8:basx::2 (DC-B)**

Both route servers run identical policy but operate independently — members should peer with both for redundancy.

```
# /etc/bird/bird.conf — BasinIX RS1 (BIRD 2.15)
# ─────────────────────────────────────────────

log syslog all;
router id 198.51.100.1;

# ─── Kernel and device protocols ───
protocol device {
    scan time 10;
}

# Do NOT export routes to kernel — RS is control-plane only
protocol kernel {
    ipv4 { export none; import none; };
}

protocol kernel {
    ipv6 { export none; import none; };
}

# ─── RPKI (RTR protocol to local validator) ───
roa4 table r4;
roa6 table r6;

protocol rpki rpki_validator {
    roa4 { table r4; };
    roa6 { table r6; };
    remote "127.0.0.1" port 3323;     # Routinator or Fort running locally
    retry keep 90;
    refresh keep 3600;
    expire keep 7200;
}

# ─── Prefix size filters ───
function reject_small_prefixes_v4()
{
    if net.len > 24 then reject;       # No IPv4 longer than /24
    if net.len < 8 then reject;        # No IPv4 shorter than /8
}

function reject_small_prefixes_v6()
{
    if net.len > 48 then reject;       # No IPv6 longer than /48
    if net.len < 16 then reject;       # No IPv6 shorter than /16
}

# ─── Bogon and martian rejection ───
function reject_bogons_v4()
{
    if net ~ [
        0.0.0.0/8+,           # "This" network
        10.0.0.0/8+,          # RFC 1918
        100.64.0.0/10+,       # Shared address space
        127.0.0.0/8+,         # Loopback
        169.254.0.0/16+,      # Link-local
        172.16.0.0/12+,       # RFC 1918
        192.0.0.0/24+,        # IETF protocol assignments
        192.0.2.0/24+,        # TEST-NET-1
        192.168.0.0/16+,      # RFC 1918
        198.18.0.0/15+,       # Benchmarking
        198.51.100.0/24+,     # TEST-NET-2
        203.0.113.0/24+,      # TEST-NET-3
        224.0.0.0/4+,         # Multicast
        240.0.0.0/4+          # Reserved
    ] then reject;
}

function reject_bogons_v6()
{
    if net ~ [
        ::/8+,                # Loopback, unspecified, v4-mapped, etc.
        0100::/64+,           # Discard-only (RFC 6666)
        2001:2::/48+,         # Benchmarking
        2001:db8::/32+,       # Documentation
        fc00::/7+,            # Unique local
        fe80::/10+,           # Link-local
        ff00::/8+             # Multicast
    ] then reject;
}

# ─── RPKI validation function ───
function apply_rpki_v4()
{
    if roa_check(r4, net, bgp_path.last) = ROA_INVALID then reject;
    if roa_check(r4, net, bgp_path.last) = ROA_VALID then {
        bgp_large_community.add((65530, 1, 1));   # Tag RPKI valid
    }
    if roa_check(r4, net, bgp_path.last) = ROA_UNKNOWN then {
        bgp_large_community.add((65530, 1, 2));   # Tag RPKI unknown
    }
}

function apply_rpki_v6()
{
    if roa_check(r6, net, bgp_path.last) = ROA_INVALID then reject;
    if roa_check(r6, net, bgp_path.last) = ROA_VALID then {
        bgp_large_community.add((65530, 1, 1));
    }
    if roa_check(r6, net, bgp_path.last) = ROA_UNKNOWN then {
        bgp_large_community.add((65530, 1, 2));
    }
}

# ─── IRR prefix-list filter (generated by bgpq4, refreshed hourly via cron) ───
# Example for member AS64501 — one include file per member ASN
# Generated by: bgpq4 -4 -l AS64501_v4 -A AS64501 > /etc/bird/members/AS64501_v4.conf
# Generated by: bgpq4 -6 -l AS64501_v6 -A AS64501 > /etc/bird/members/AS64501_v6.conf

include "/etc/bird/members/*.conf";

# ─── Blackhole community handling ───
function handle_blackhole()
{
    # Accept /32 IPv4 or /128 IPv6 with blackhole community
    if (65535, 666) ~ bgp_community then {
        if (net.type = NET_IP4 && net.len = 32) ||
           (net.type = NET_IP6 && net.len = 128) then {
            bgp_local_pref = 0;
            dest = RTD_BLACKHOLE;
            accept;
        }
    }
}

# ─── Per-member import filter template ───
# Each member BGP session uses a tailored filter referencing their IRR prefix-list

filter import_AS64501_v4 {
    reject_bogons_v4();
    reject_small_prefixes_v4();
    handle_blackhole();
    if net !~ AS64501_v4 then reject;   # IRR prefix-list filter
    apply_rpki_v4();
    accept;
}

filter import_AS64501_v6 {
    reject_bogons_v6();
    reject_small_prefixes_v6();
    handle_blackhole();
    if net !~ AS64501_v6 then reject;
    apply_rpki_v6();
    accept;
}

# ─── Export filter (same for all members) ───
filter export_default_v4 {
    # Strip IXP-internal communities before export if needed
    # Preserve RPKI tags for member visibility
    reject_bogons_v4();
    accept;
}

filter export_default_v6 {
    reject_bogons_v6();
    accept;
}

# ─── BGP template for RS peering sessions ───
template bgp rs_member {
    local as 65530;                     # IXP RS ASN (private, not in AS path)
    rs client;                          # Route server mode: don't add own ASN to path
    passive on;                         # Members initiate sessions to RS
    hold time 90;
    keepalive time 30;
    graceful restart on;
    interpret communities on;
    enable extended messages on;
    ipv4 {
        add paths tx;                   # ADD-PATH: advertise multiple paths per prefix
        import keep filtered;           # Keep filtered routes for debugging
        next hop self off;              # CRITICAL: preserve member's next-hop
    };
    ipv6 {
        add paths tx;
        import keep filtered;
        next hop self off;
    };
}

# ─── Member sessions ───
# AS64501: Basin City Broadband (founding member)
protocol bgp rs_AS64501_v4 from rs_member {
    neighbor 198.51.100.11 as 64501;
    ipv4 {
        import filter import_AS64501_v4;
        export filter export_default_v4;
    };
}

protocol bgp rs_AS64501_v6 from rs_member {
    neighbor 2001:db8:basx::b as 64501;
    ipv6 {
        import filter import_AS64501_v6;
        export filter export_default_v6;
    };
}

# ... repeat for each member AS ...
```

**Key route server design decisions:**

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Software | BIRD 2 | Dominant IXP RS platform; robust filtering, RPKI, ADD-PATH |
| RS ASN | 65530 (private) | RS doesn't insert itself in AS path (`rs client` mode) |
| Transparent mode | `rs client` + `next hop self off` | Members see each other's ASN and next-hop directly |
| ADD-PATH | Enabled (tx) | Members receive multiple paths per prefix — enables member-side best-path selection |
| RPKI | ROA_INVALID → reject | Strict RPKI enforcement; unknown prefixes tagged but accepted |
| IRR filtering | bgpq4-generated prefix-lists per member | Hourly cron regeneration; each member filtered to their IRR objects |
| Max-prefix | Per-member limit | Prevents a single member from flooding the RS table |
| Graceful restart | Enabled | RS restart doesn't tear down member forwarding |
| Deployment | 2 independent RS instances | RS1 at DC-A, RS2 at DC-B; no shared state, no split-brain risk |

### RPKI and IRR Filtering Pipeline

The filtering pipeline is the most critical security function of the route server. Every route received from a member passes through this chain:

```
Incoming route from member
    │
    ▼
┌─────────────────┐
│ 1. Bogon filter  │──→ RFC 1918, documentation, multicast → REJECT
└────────┬────────┘
         ▼
┌─────────────────┐
│ 2. Prefix length │──→ IPv4 >/24 or </8 → REJECT
│    filter        │──→ IPv6 >/48 or </16 → REJECT
└────────┬────────┘
         ▼
┌─────────────────┐
│ 3. Blackhole     │──→ Community 65535:666 + /32 or /128 → ACCEPT (blackhole)
│    check         │
└────────┬────────┘
         ▼
┌─────────────────┐
│ 4. IRR filter    │──→ Prefix NOT in member's IRR objects → REJECT
│   (bgpq4)       │    Generated from RADB/RIPE/ARIN via bgpq4
└────────┬────────┘
         ▼
┌─────────────────┐
│ 5. RPKI check    │──→ ROA_INVALID → REJECT
│   (Routinator)   │──→ ROA_VALID → tag community, ACCEPT
│                  │──→ ROA_UNKNOWN → tag community, ACCEPT
└────────┬────────┘
         ▼
    Route accepted into RS RIB
    Re-advertised to all other members
```

**IRR prefix-list generation (cron job, runs hourly):**

```bash
#!/bin/bash
# /usr/local/bin/update-irr-filters.sh
# Regenerates IRR-based prefix-lists for all BasinIX members

MEMBERS="/etc/bird/members"
BIRD_CONF="/etc/bird"

# Member list: ASN and their AS-SET (or ASN if no set registered)
declare -A MEMBER_SETS=(
    [64501]="AS64501"
    [64502]="AS-BASINFIBER"
    [64503]="AS64503:AS-CUSTOMERS"
    [13335]="AS-CLOUDFLARE"
    [15169]="AS-GOOGLE"
    [32934]="AS-FACEBOOK"
)

for ASN in "${!MEMBER_SETS[@]}"; do
    AS_SET="${MEMBER_SETS[$ASN]}"

    # IPv4 prefix-list
    bgpq4 -4 -A -l "AS${ASN}_v4" "${AS_SET}" > "${MEMBERS}/AS${ASN}_v4.conf.tmp"
    mv "${MEMBERS}/AS${ASN}_v4.conf.tmp" "${MEMBERS}/AS${ASN}_v4.conf"

    # IPv6 prefix-list
    bgpq4 -6 -A -l "AS${ASN}_v6" "${AS_SET}" > "${MEMBERS}/AS${ASN}_v6.conf.tmp"
    mv "${MEMBERS}/AS${ASN}_v6.conf.tmp" "${MEMBERS}/AS${ASN}_v6.conf"
done

# Reload BIRD to pick up new prefix-lists
birdc configure check && birdc configure
logger "BasinIX IRR filters updated: ${#MEMBER_SETS[@]} members"
```

**RPKI validator (Routinator, running locally on each RS):**

```toml
# /etc/routinator/routinator.conf
```

## BGP Peering Policies

### Multilateral vs Bilateral Peering

BasinIX supports both peering models:

**Multilateral peering (via route servers):**
- Member peers with RS1 and RS2
- RS re-advertises all accepted routes to all members
- Default policy: open peering (all members receive all routes)
- Opt-out available: members can use BGP communities to restrict advertisement

**Bilateral peering (direct BGP sessions):**
- Members configure direct BGP sessions between their routers across the peering LAN
- No route server involvement — member-to-member control and data plane
- Required for: members wanting private peering policies, custom communities, or BFD timers
- Common for: large content networks that prefer full control over peer relationships

**Community-based selective peering on the route server:**

BasinIX implements RFC 8326-style graceful shutdown and selective advertisement communities:

| Community | Meaning |
|-----------|---------|
| 65530:0:peer-ASN | Do NOT advertise my routes to peer-ASN |
| 65530:1:peer-ASN | Advertise my routes ONLY to peer-ASN |
| 65530:0:0 | Do NOT advertise my routes to anyone (withdraw from RS) |
| 65535:666 | Blackhole this prefix (RTBH) |
| 65535:65281 | NO_EXPORT — don't export beyond peering LAN |
| 65530:1:1 (large) | RPKI Valid tag (informational, set by RS) |
| 65530:1:2 (large) | RPKI Unknown tag (informational, set by RS) |

## Blackholing Service (RTBH)

BasinIX provides a Remote Triggered Blackhole service, allowing members to drop traffic to their own prefixes under DDoS attack.

**How it works:**

1. Member is under DDoS attack targeting 203.0.113.100
2. Member announces 203.0.113.100/32 to the route server with community 65535:666
3. Route server accepts the /32 (normally /32 would be filtered, but blackhole exception applies)
4. Route server re-advertises 203.0.113.100/32 with next-hop set to the discard address to all members
5. All members who honor the blackhole community drop traffic to that /32 at their edge

**Safeguards:**
- Only /32 (IPv4) and /128 (IPv6) accepted with blackhole community
- The /32 must fall within the member's registered IRR prefix space (prevents blackholing someone else's prefix)
- Blackhole routes get LOCAL_PREF 0 (lowest priority — won't override legitimate routes in normal operation)

```
    pass
  endif
end-policy

  203.0.113.100/32
end-set
```

## ARP/ND Sponge and BCP Recommendations

### The ARP Problem at IXPs

On a peering LAN with 60+ members, every member's router sends ARP requests for every other member's IP. When a member disconnects (port down, maintenance, decommission), their IP remains in other routers' ARP caches — and when the cache expires, ARP requests for the dead IP broadcast across the entire peering LAN. With 244 possible IPs in a /24, abandoned addresses create constant ARP broadcast noise.

**ARP sponge:**

An ARP sponge is a daemon that watches the peering LAN for ARP requests that go unanswered. After a threshold of unanswered requests for the same IP, the sponge begins responding to those ARP requests with its own MAC address — absorbing the traffic into a blackhole. When the real owner comes back online (sends a gratuitous ARP), the sponge releases that IP.

```
# arpsponge deployment on BasinIX management server
# Connected to VLAN 100 via a trunk port
arpsponge --interface eth0.100 \
          --network 198.51.100.0/24 \
          --queuedepth 5 \
          --rate 30 \
          --sponge-network \
          --log-mask all \
          --daemon
```

### BCP Recommendations (Euro-IX / MANRS)

BasinIX follows these IXP best current practices:

| BCP | Implementation |
|-----|---------------|
| No IP forwarding on switches | IXP switches are pure L2 — no IP addresses on VLAN 100 |
| No proxy ARP | Disabled on all switch SVIs and member ports |
| MAC address filtering | One MAC per member port (enforced at switch) |
| No DHCP on peering LAN | Static addressing only; DHCP snooping drops server frames |
| Peering LAN not routed | /24 is not advertised into any IGP or BGP |
| Member must filter | Members should implement ingress bogon filtering + max-prefix |
| PeeringDB registration | All members must maintain accurate PeeringDB records |
| IPv6 ND RA suppression | Router advertisements suppressed on all switch ports |
| No spanning tree from members | BPDU guard on all member ports |
| MANRS compliance | IXP implements RPKI validation, IRR filtering, anti-spoofing |

## Security Design

### Switch-Level Security

```
! Junos QFX5120 — member port security template
! Applied to every member-facing interface

interfaces {
    xe-0/0/5 {
        description "Member: AS64501 Basin City Broadband";
        unit 0 {
            family ethernet-switching {
                interface-mode access;
                vlan {
                    members PEERING;
                }
            }
        }
    }
}

/* MAC limiting: 1 MAC per port */
ethernet-switching-options {
    secure-access-port {
        interface xe-0/0/5 {
            mac-limit 1 action drop;
        }
    }
}

/* Storm control: limit broadcast/multicast to 1% of port speed */
forwarding-options {
    storm-control-profiles IXP-STORM {
        all {
            bandwidth-percentage 1;
        }
    }
}

/* Apply storm control to member ports */
interfaces {
    xe-0/0/5 {
        unit 0 {
            family ethernet-switching {
                storm-control IXP-STORM;
            }
        }
    }
}

/* BPDU guard — shut port on BPDU receipt */
protocols {
    rstp {
        interface xe-0/0/5 {
            edge;
            bpdu-block;
        }
    }
}

/* RA guard — suppress IPv6 router advertisements from members */
forwarding-options {
    dhcp-relay {
        /* No DHCP relay — just snooping to block rogue servers */
    }
}
```

### Defense-in-Depth Summary

| Layer | Threat | Control |
|-------|--------|---------|
| Physical | Unauthorized cross-connect | Locked IXP cage; member access to meet-me room only |
| L2 | MAC flooding | MAC limit 1 per port; MAC table size alarm |
| L2 | Broadcast storm | Storm control 1% bandwidth; automatic port shutdown |
| L2 | STP manipulation | BPDU guard on all member ports |
| L2 | VLAN hopping | Access ports only; no trunking to members; DTP disabled |
| L3 | Rogue DHCP | DHCP snooping; static IP assignment only |
| L3 | ARP spoofing | Static ARP entries for RS; ARP inspection on switch |
| L3 | IPv6 RA spoofing | RA guard on all member ports |
| BGP | Prefix hijack | IRR prefix-list filtering + RPKI ROA validation |
| BGP | Route leak | AS-path filtering; max-prefix limits per member |
| BGP | Table explosion | Max-prefix teardown at 90% threshold |
| DDoS | Volumetric attack on member | RTBH blackholing service (community 65535:666) |

## Design Review Questions

These questions test architectural reasoning — the kind of analysis expected in IE-SP lab design scenarios.

**Q1: A member accidentally configures a second IP address on their peering LAN interface. What breaks, and which IXP controls prevent or mitigate this?**

> *The second IP consumes an address from the peering LAN — potentially conflicting with another member's assigned address. The MAC limit (1 per port) won't catch this since both IPs share the same MAC. The ARP sponge won't help because the IP is actively in use. The IRR/RPKI filtering on the route server is irrelevant (this is a Layer 3 peering LAN issue, not a BGP announcement issue). The mitigation is: (a) the member agreement contractually prohibits additional IPs, (b) ARP inspection with static bindings (IP-MAC-port) on the switch would detect and block the rogue IP's ARP responses, and (c) monitoring should alert on unexpected ARP entries. This is why static DHCP/ARP bindings are a best practice even on IXP peering LANs.*

**Q2: BasinIX route server RS1 crashes and takes 10 minutes to recover. What is the impact on member-to-member traffic forwarding, and why?**

> *Zero impact on data-plane forwarding. Route servers are control-plane only — they never forward packets. Members' routers already have routes learned from RS1 in their RIB with the correct next-hop (the other member's peering LAN IP). With graceful restart enabled, member routers retain stale routes from RS1 during the outage. RS2 independently holds the same routes, providing full redundancy. The only risk is: if a member withdrew a prefix during the 10-minute window, other members wouldn't learn about the withdrawal from RS1 until it recovers (but RS2 would propagate it). The key insight: dual RS means no single RS is ever critical to forwarding.*

**Q3: A new CDN member with 50,000 prefixes joins BasinIX. What RS configuration changes are needed, and what risk does this create for other members?**

> *The RS needs: (a) a new BGP session configured with the CDN's ASN, (b) IRR prefix-list generated via bgpq4 for the CDN's AS-SET, (c) a per-member max-prefix limit set appropriately (e.g., 60,000 with 90% warning). The risk: 50,000 prefixes re-advertised to all other members increases their RIB size. Small members on low-memory routers might hit their own max-prefix limits and tear down their RS sessions. Mitigation: notify all members before onboarding a large prefix holder, recommend members set generous max-prefix limits, and consider the ADD-PATH impact (multiple paths per prefix multiplied by 50K prefixes). The RS hardware itself (CPU/memory) must be sized for the aggregate RIB — this is why production IXP route servers typically run on servers with 32GB+ RAM.*

**Q4: During a DDoS attack, member AS64501 announces a /32 blackhole via the route server. Explain the end-to-end flow: what happens at the RS, what do other members receive, and how does traffic actually get dropped?**

> *AS64501 announces 203.0.113.100/32 with community 65535:666 to both RS1 and RS2. The RS import filter detects the blackhole community, verifies the /32 falls within AS64501's IRR-registered space, sets LOCAL_PREF to 0, and accepts it. The RS then re-advertises the /32 to all other members. Each receiving member's router installs the /32 — but the traffic drop mechanism depends on the member's own configuration. Members who implement RTBH (mapping community 65535:666 to a null0/discard next-hop) will drop traffic locally before it crosses the IXP fabric. Members who don't implement RTBH will still forward traffic toward AS64501 — but it will be blackholed at AS64501's router (the /32 is more specific than the /24, so AS64501's own router can install a static discard route). The blackhole is most effective when receiving members honor the community.*

**Q5: BasinIX is evaluating migration from its VLAN-based fabric to EVPN-VXLAN. What are the operational risks of this migration, and how would you execute it without disrupting member peering sessions?**

> *The primary risks are: (a) MAC learning behavior changes (BGP-learned MACs vs data-plane learning), which could cause brief traffic blackholing during cutover, (b) MTU changes — VXLAN adds 50 bytes of encapsulation, so inter-switch links need 9050+ byte MTU while the peering LAN MTU presented to members stays at 1500 (or 9000 if jumbo), and (c) control-plane complexity — EVPN BGP sessions between switches add a failure domain that didn't exist before. Migration approach: deploy new EVPN-VXLAN switches in parallel, migrate members one port at a time (dual-home during transition), verify MAC learning and forwarding per member before cutting over, and keep the old switches as rollback until all members are migrated. The entire migration should be transparent to members — their IP/MAC/BGP config doesn't change, only the underlying switch fabric. Test BUM (Broadcast, Unknown unicast, Multicast) handling carefully: EVPN's ARP suppression and ingress replication behavior differs from traditional flooding.*

## Summary

BasinIX demonstrates the core architectural principles of IXP design:

1. **Physical redundancy** — Dual sites, diverse fiber, no single point of infrastructure failure
2. **Simple Layer 2** — Start with VLANs, migrate to EVPN-VXLAN when scale demands it
3. **Route server as filtering engine** — RPKI + IRR + bogon filtering protects all members
4. **Transparent RS operation** — `rs client` mode, next-hop preservation, ADD-PATH for full visibility
5. **Security by default** — MAC limits, BPDU guard, storm control, ARP inspection on every port
6. **Blackholing as a service** — Community-triggered RTBH gives members DDoS mitigation at the fabric level
7. **Financial sustainability** — Non-profit model with fees sized to cover infrastructure and 1 FTE

The IXP is one of the simplest designs in service provider networking — it's fundamentally a shared Ethernet switch with a BGP route server — but the operational discipline required to run it securely and reliably at scale is what separates a production exchange from a lab exercise.
