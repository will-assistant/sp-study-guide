## Overview

> Design case studies are how the IE-SP lab tests whether you can synthesize protocol knowledge into defensible architecture decisions. This case study walks a full ISP backbone design from requirements to implementation — the same exercise that happens in real SP pre-sales and capacity engineering reviews.

**Scenario: ClearPath Networks**

ClearPath is a regional US carrier with 12 PoPs (NY, LA, CHI, DAL, MIA, SEA, DEN, PHX, BOS, ATL, DET, SLC) and two international PoPs (LON, TYO). They carry 200K enterprise routes, 1.2M internet routes, and run 47 L3VPN customers. Their current network runs OSPF + LDP in a flat design, with full-mesh IBGP and no SR. Traffic engineering is done manually by moving static routes. They want to modernize.

**What you'll design:** A complete backbone modernization — IGP, MPLS, BGP, SR-TE, L3VPN, peering, and automation.

## Current State: Why the Existing Design Is Breaking

Before designing the replacement, understand what's failing and why.

### Problem 1: OSPF Flat Design

```
Current state: Single OSPF area 0 — 14 PoPs × 6 routers each = 84 routers flooding LSAs
LSA churn: Any interface flap triggers topology recalculation across all 84 nodes
SPF convergence: ~800ms with OSPF — slow for backbone links
Problem: No type-1/2 LSA summarization possible inside area 0 — every prefix floods everywhere
```

**The IS-IS argument:** IS-IS Level-1/Level-2 hierarchy naturally segments flooding. Level-1 areas contain intra-PoP topology. Level-2 carries inter-PoP reachability only. An interface flap in NYC stays within Level-1 for intra-PoP convergence; only L2 reachability matters backbone-wide. With 84 routers in flat OSPF, every SPF runs 84-node Dijkstra. With IS-IS hierarchy, most calculations run only within a Level-1 area (6 nodes per PoP).

### Problem 2: LDP + Full Mesh IBGP

```
Current: full-mesh IBGP — n(n-1)/2 sessions
14 PoPs × 6 routers = 84 routers needing full mesh = 3,486 IBGP sessions
With internet full table: 84 × 1.2M routes = 100.8M route table entries across the network
Each BGP speaker must re-advertise to all peers — no summarization, no delegation
```

**The route reflector argument:** RRs allow hub-and-spoke IBGP. Clients peer only with RR — O(n) sessions instead of O(n²). With two RRs per cluster (redundancy), 14 PoPs = 28 RR sessions per router maximum. At ClearPath's scale this reduces session count from 3,486 to ~200 total. Critical: RR placement matters — a poorly placed RR becomes a single point of failure for BGP convergence.

### Problem 3: LDP + Manual Traffic Engineering

```
Current: RSVP-TE was evaluated, rejected due to signaling complexity
LDP label allocation: one FEC per prefix — huge LFIB at scale
"Traffic engineering" = change metric on interface, wait for SPF, monitor
Result: 3 outages in 18 months caused by human error during manual TE adjustments
```

**The SR argument:** SR-MPLS eliminates signaling protocols entirely. SRGB (Segment Routing Global Block) assigns stable node SIDs from a global pool — one SID per router, not one per prefix. The LFIB size collapses. SR-TE policies encode explicit paths without RSVP state. A head-end can steer traffic onto any explicit path by pushing a label stack — without touching intermediate routers. Zero protocol change needed at P routers.

## Topology Design

### Physical Topology Choices

**Core vs Edge vs PE Separation (3-tier model)**

```
           ┌──────────────────────────────┐
           │       P Routers (Core)        │
           │   (label-forwarding only)     │
           └──────────┬───────────────────┘
                       │
           ┌──────────┴───────────────────┐
           │      PE Routers (Edge)        │
           │   (VRFs, customer facing)     │
           └──────────┬───────────────────┘
                       │
           ┌──────────┴───────────────────┐
           │       CPE / IXP Peers         │
           └──────────────────────────────┘
```

**Why separate P and PE?**

P routers need maximum forwarding performance — they look at nothing but the MPLS label. They carry no VRF tables, no full internet table, no BGP. PE routers carry customer VRFs, BGP full table for internet PE, and MP-BGP for VPNv4/VPNv6. Mixing roles forces a compromise: either your P routers need PE-class memory, or your PE routers become a bottleneck at label switching speed.

At ClearPath's scale: 2 P routers per PoP (pure forwarding), 2-4 PE routers per PoP (customer-facing). P routers don't run BGP. PE routers don't need 400G backplane — they need fast control plane and large RIB.

### PoP Interconnect Design

```
Full mesh between P routers at 14 PoPs? No. That's 91 circuits.

ClearPath chose: Hub-and-spoke ring with diagonal shortcuts

Ring: NY → BOS → ATL → MIA → DAL → DEN → PHX → LA → SEA → SLC → CHI → DET → NY
Diagonals: NY-CHI, LA-DAL, SEA-DEN (traffic engineering shortcuts for coast-to-coast)
International: NY-LON, LA-TYO
```

**Design Rationale:** A pure ring maximizes circuit efficiency but creates long AS-hop paths for coast-to-coast traffic. Diagonals cut the average path length from 7 hops to 3-4. The ring provides resilience — any single link failure has an alternate path around the ring. SR-TE uses the diagonals for preferred paths; in failure, traffic shifts automatically without reconfiguration.

**Link capacity:** 100G ring segments, 400G on the NY-CHI and LA-DEN diagonals (highest traffic corridors). International: 100G NY-LON, 100G LA-TYO with dark fiber lease option to 400G.

## Protocol Design

### IS-IS Design

**Level structure:**
```
Level-2 backbone: All 14 PoPs + 2 international PoPs
Level-1 areas: Each PoP is a separate L1 area (e.g., area 49.0001 for NY)
             P and PE routers at each PoP are L1/L2 (both levels)
             Any intra-PoP access routers run L1-only
```

**Metric style:** Wide metrics (style=wide) — required for SR-MPLS (prefix-sid + TI-LFA need wide metrics). Do not mix narrow and wide. If any router runs narrow, TI-LFA breaks.

**Interface configuration discipline:**
```
All P-P backbone links: network point-to-point (no DIS election overhead)
Any broadcast segment that sneaks into the core: verify hello-interval and holdtime
SPF timers: initial 50ms, incremental 200ms, max 5000ms (exponential backoff)
BFD: 3×33ms on all P-P links — detects link failure before IS-IS hello expires
```

**Prefix advertisement policy:**
```
PE loopbacks: advertised into L2 IS-IS — required for MPLS transport
P loopbacks: advertised into L2 — required for node SIDs
Customer prefixes: NOT in IS-IS — stay in VRF/BGP
Internet prefixes: NOT in IS-IS — stay in internet VRF BGP
```

**Why not OSPF?** IS-IS handles IPv4 + IPv6 in a single adjacency (multi-topology TLV). OSPF runs OSPFv2 for IPv4 and OSPFv3 for IPv6 — two separate routing processes, two separate databases, two separate SPFs. At 84 routers that doubles the control plane overhead. IS-IS also has no equivalent of the OSPF type-7 LSA ASBR complexity that bites people at area boundaries.

### SR-MPLS Design

**SRGB (Segment Routing Global Block):**
```
Range: 16000–23999 (8000 labels)
Standard: IANA recommended SRGB is 16000–23999
Rule: Every router in the network uses the SAME SRGB range — this is the whole point
```

**Node SID allocation (from SRGB):**
```
NY P-1:    16001    LA P-1:    16009    LON P-1:   16013
NY P-2:    16002    LA P-2:    16010    TYO P-1:   16014
NY PE-1:   16003    LA PE-1:   16011    (and so on)
...
```

**Why a flat SID allocation?** You could allocate by PoP (e.g., NY = 16000-16009, CHI = 16010-16019). This looks organized but creates operational headaches: you need to know which PoP a SID belongs to before you can trace a path. Flat sequential allocation with a registry is simpler — look up any SID in one table.

**Adjacency SIDs:** Auto-allocated by each router from the local SRLB (Segment Routing Local Block, typically 15000-15999). Adjacency SIDs are local-significance — NY P-1 to NY P-2 link gets a local adj-SID. Used in SR-TE for explicit hop-by-hop paths. Allocate SRLB range per router in your SID registry.

**TI-LFA Configuration:**
```
All backbone interfaces: ti-lfa enable
Post-convergence path: backup SID stack limited to 3 labels (keeps LFIB manageable)
Microloop avoidance: enabled with 3-second delay (prevents transient microloops during SPF)
```

Critical: TI-LFA requires IS-IS metric consistency. If any link uses metric 1 instead of the standard metric (100 for 10G, 10 for 100G, 1 for 400G), TI-LFA may compute incorrect loop-free alternates. Standardize metrics — document them.

### BGP Design

**Route Reflector Placement:**
```
RR-1: NY (geographically central to east coast, closest to internet exchanges)
RR-2: CHI (center of US, resilient against east coast outages)
RR-3: LA (west coast, resilient against both east coast + central outages)

Cluster design: All three are in a single cluster (cluster-id shared)
Each PE peers with all 3 RRs — session loss to one RR is non-event
P routers: NO BGP (pure forwarding)
```

**Why 3 RRs?** Two RRs in the same PoP fail together if the PoP loses power. Three geographically distributed RRs ensure that no single failure (PoP outage, fiber cut) leaves any PE without a functioning RR. Each RR maintains a full view of the network — BGP ADD-PATH enabled so RRs advertise multiple paths to PEs.

**Address families on RRs:**
```
IPv4 unicast: internet full table for internet PEs only
VPNv4: all customer L3VPN routes
VPNv6: IPv6 L3VPN routes
IPv6 unicast: internet IPv6 table
L2VPN EVPN: if EVPN services deployed (future)
```

**Community design:**
```
Standard communities (numeric values — operator chooses scheme):
  65000:100 — learned from customer (full routes, can advertise to peers/upstreams)
  65000:200 — learned from peer (can advertise to customers, not to other peers)  
  65000:300 — learned from upstream transit (can advertise to customers only)
  65000:999 — no-export (keep within AS, do not advertise to EBGP peers)

Large communities (RFC 8092) for richer policy:
  65000:1:CUST_ID  — per-customer tagging for traffic accounting
  65000:2:PEER_ASN — track which peer originated a route
```

**Route policy discipline:** Every EBGP session has explicit import and export policies. No implicit allow. Import policy filters: bogons (RFC 5735), too-long prefixes (/25 and longer from peers), your own prefixes (anti-leak), routes without valid AS path. Export policy: strip communities that shouldn't leave the AS, apply prepend for depreferring certain paths.

### L3VPN Design

**RD/RT allocation scheme:**
```
Route Distinguisher: ASN:customer_id (e.g., 65000:1001 for customer 1001)
  RD must be unique per VRF per PE — use ASN:CUSTID prevents collision across PEs
  
Import/Export RT:
  Simple any-to-any VPN: 65000:CUSTID import and export
  Hub-and-spoke VPN: Hub exports 65000:CUSTID-h, Spoke imports 65000:CUSTID-h
                     Spoke exports 65000:CUSTID-s, Hub imports 65000:CUSTID-s
  Extranet: Additional import of partner RT (65000:PARTNER_CUSTID)
```

**VRF scale per PE:**
```
Average PE at ClearPath: 8 customers, 3 VRFs each = 24 VRFs
Maximum VRF scale (Juniper MX): 32,000 VRFs per chassis — not a constraint
Memory constraint is route table: 8 customers × 200 routes avg = 1,600 customer routes
Plus internet VRF for internet PE: 1.2M routes — this is the constraint
Internet PE and L3VPN PE: separate boxes if internet-full-table PEs
```

**CE-PE routing:**
```
Most customers: eBGP (clean policy, no IGP redistribution needed)
Small sites: Static routes (simple, no customer BGP required)
Avoid: eBGP with private ASN without AS-override or allow-AS-in — creates leaks
Check: Maximum-prefix on every CE-PE session (prevents accidental full-table push from CE)
```

## Internet Peering Design

### Peering Points

```
Primary IXPs:
- NYIIX (New York Internet Exchange) — NY PoP, 100G
- Any2 (West Coast) — LA PoP, 100G  
- DE-CIX Chicago — CHI PoP, 100G

Private peering (bilateral):
- Major CDNs (Akamai, Cloudflare, Fastly, Amazon CloudFront): direct fiber at IXP colos
- Reason: CDN traffic volume justifies private session to avoid IXP fabric congestion

Transit upstreams:
- Two transit providers (never same parent company) for full-table internet routes
- NY + LA as transit ingress points (closest to IXP density)
- Prefer: AS-path length policy — prefer customer > peer > transit for outbound
```

### Route Security (RPKI)

```
ROA validation: Enabled on all EBGP sessions
Invalid routes: REJECT (not drop-local-pref) — never accept invalid ROAs
Unknown routes: ACCEPT with lower local-pref (prefer valid > unknown > invalid)
BGPsec: Not yet — operational complexity not worth it until major transit providers deploy

IRR filtering:
  All customer BGP sessions: filter on IRR-registered prefixes only
  Peer sessions: use RPKI as primary, IRR as secondary
  Transit: rely on upstream's filtering + your own ROA validation
```

### Outbound Traffic Engineering with SR-TE

```
Problem: Default BGP outbound path follows hot-potato routing (exit at nearest PoP)
Result: LA-bound traffic from NY exits at NY even if NY's LA-transit link is congested

SR-TE solution:
- Telemetry: gNMI streaming interface utilization from all P routers (30s interval)
- Controller: compares utilization to 80% threshold per link
- Policy: SR-TE policy on NY PE steers transit traffic to LA via less-loaded path
  (SID stack: NY-P-SID → CHI-P-SID → LA-PE-SID)
- Fallback: If SR policy withdrawn, BGP selects next-best path automatically
```

## Convergence and Protection Analysis

### Failure Scenario 1: P Router Failure (Core Node)

```
Pre-failure path: NY-PE → NY-P1 → CHI-P → LA-P → LA-PE
Failure: CHI-P router goes offline

Timeline:
T=0ms    BFD session to CHI-P detects loss (3×33ms = ~100ms)
T=100ms  IS-IS P router deletes adjacency, begins SPF
T=120ms  SPF completes (small topology, fast SPF)
T=120ms  TI-LFA pre-computed backup path activates (NY-P1 → DEN-P → LA-P)
T=300ms  BGP path selection updates for affected prefixes
T=300ms  SR-TE policies recalculate via controller

TI-LFA means: traffic reroutes at T=100ms, before SPF even completes.
Total outage for established flows: ~100ms
New flows: < 300ms
```

**What TI-LFA computes pre-failure:**

For each protected link, TI-LFA finds a backup path that is guaranteed loop-free. It encodes this path as a label stack pushed at the point of failure — no intermediate routers need to know about the failure. The backup path is computed using the P-space/Q-space algorithm: find which nodes can reach the destination without going through the failed link (Q-space), and which nodes can be reached from the source without going through the failed link (P-space). The intersection is the repair path.

### Failure Scenario 2: PE Router Failure (Edge Node)

```
Pre-failure: Customer A's NY traffic → NY-PE1
Failure: NY-PE1 crashes

Impact: Customer A's NY VRF is gone — traffic to NY-PE1 is blackholed

Mitigation options:
1. Dual-homed CE: CE has EBGP to both NY-PE1 and NY-PE2
   → Customer traffic shifts to NY-PE2 within BGP convergence time (~300ms)
   → Requires customer CPE to support dual BGP sessions
   
2. EVPN multi-homing for L2 customers:
   → All-Active ES (Ethernet Segment) — traffic hashes across both PEs
   → Mass withdrawal on PE failure cleans up remote PEs in one message
   
For ClearPath: All enterprise customers dual-homed by contract requirement
Mitigation: IBGP hold-down timers tuned to 3s (not default 90s)
```

### Failure Scenario 3: International Link Failure (NY-LON 100G fiber cut)

```
Pre-failure: European traffic from US customers transits NY → LON
Failure: NY-LON subsea fiber cut

Backup path: via TYO (LA → TYO → LON, ~280ms RTT instead of 80ms)

IS-IS metric design for international links:
  NY-LON: metric 100 (high — prefer LA-TYO as backup, not primary)
  LA-TYO: metric 150 (even higher — but available as emergency backup)
  
Traffic engineering override: SR-TE can force traffic onto LA-TYO for specific VPNs
  even before NY-LON recovers, if latency SLA requires Atlantic routing

SLA implication: Backbone outage NY-LON shifts RTT from 80ms to 280ms
Customer notification: Any SLA breach triggers automated ticket (telemetry alert)
```

## Automation Strategy

### Day-0: Provisioning

```
New PoP addition requires:
1. SRGB SID allocation (from registry)
2. IS-IS L1 area creation
3. BGP sessions to 3 RRs
4. L3VPN template applied (standard VRF config)
5. BFD + TI-LFA on all backbone links

Automation: Ansible playbook (collect-all.yml pattern)
  - Variables: inventory YAML with PoP parameters
  - Templates: Jinja2 Junos/IOS-XR configs per device role
  - Validation: post-deploy NETCONF check (IS-IS adjacency up, BGP sessions up)
  - Time: target < 2 hours for new PoP activation
```

### Day-1: Customer Provisioning

```
L3VPN new customer:
1. Allocate RD/RT from customer registry (ASN:CUSTID)
2. Create VRF on PE routers serving that customer
3. Configure CE-PE BGP (with max-prefix)
4. Advertise route-target to RRs
5. Verify: ping across VPN, traceroute, no route leaks

Automation: NETCONF-based provisioning via controller API
  Target: < 15 min from ticket submission to service active
```

### Day-2: Telemetry and Operations

```
gNMI streaming (dial-out to Prometheus):
- Interface counters: in/out octets, errors, drops (every 30s)
- BGP session state: neighbor up/down, prefix counts (every 60s)
- IS-IS adjacency: up/down events (ON_CHANGE)
- SR-TE policy state: active/inactive, installed paths (every 60s)

Alert thresholds:
- Interface utilization > 80% for > 5 min → automated SR-TE steering + alert
- BGP session down → PagerDuty P1 alert
- IS-IS adjacency down → immediate alert (pre-convergence telemetry fires)
- Prefix count deviation > 5% from baseline → possible route leak alert
```

## Design Review Checklist

Before signing off on a backbone design, verify:

**IGP:**
- [ ] Wide metrics enabled on all IS-IS nodes (required for SR/TI-LFA)
- [ ] SPF timers standardized network-wide (exponential backoff)
- [ ] BFD on all backbone links (3×33ms or tighter)
- [ ] All loopbacks in IS-IS (required for MPLS transport)
- [ ] Customer/internet prefixes NOT in IGP (only loopbacks and backbone links)

**SR-MPLS:**
- [ ] SRGB consistent across all routers
- [ ] SID registry maintained (who owns which SID)
- [ ] TI-LFA pre-computed on all backbone interfaces
- [ ] Microloop avoidance timer configured
- [ ] Post-convergence path label stack depth limited (≤ 3 labels)

**BGP:**
- [ ] No full-mesh IBGP (RRs or confederations)
- [ ] RR redundancy: minimum 2 per cluster, geographically separated
- [ ] ADD-PATH enabled on RRs (multi-path visibility)
- [ ] All EBGP sessions have explicit import/export policy (no implicit permit)
- [ ] Maximum-prefix on all PE-CE sessions
- [ ] RPKI validation enabled on all EBGP sessions
- [ ] Communities documented and applied consistently

**L3VPN:**
- [ ] RD uniqueness guaranteed (ASN:CUSTID scheme)
- [ ] RT import/export documented per customer
- [ ] Maximum-prefix on CE-PE BGP
- [ ] Dual-homing plan for every customer

**Operations:**
- [ ] gNMI telemetry streaming configured and tested
- [ ] Alert thresholds documented and validated
- [ ] Runbook: P router failure, PE router failure, BGP session down
- [ ] Automation: new customer provisioning < 15 min (tested)
- [ ] Change management: pre-commit validation + `commit confirmed`

## IE-SP Design Review Questions

1. **ClearPath currently uses LDP for MPLS. The migration plan proposes running SR-MPLS and LDP in parallel for 6 months. What are the risks of this approach, and what mechanism prevents SR and LDP from allocating conflicting labels for the same prefixes?**

   *Discussion:* SR-MPLS and LDP co-exist via separate label ranges (SRGB vs dynamic LDP allocation). The risk is label advertisement overlap — if the SRGB range (16000-23999) overlaps with LDP's dynamic range, label conflicts cause forwarding black holes. The solution is to configure the SRGB before deploying LDP, ensure LDP excludes the SRGB range, and run `show mpls forwarding` verification at each step. IS-IS prefix-sids are advertised in IS-IS; LDP labels are distributed via LDP Discovery. A router can receive both and install both — the forwarding preference is configurable (SR preferred by default).

2. **You place route reflectors in NY, CHI, and LA. A fiber cut isolates CHI from the rest of the network. What happens to BGP convergence, and are any customers impacted?**

   *Discussion:* CHI-RR loses IBGP sessions when isolated. PE routers in CHI still have sessions to NY-RR and LA-RR. Other PEs lose the CHI-RR session but retain NY-RR and LA-RR — no BGP impact outside CHI. CHI PEs lose one of three RR sessions — reduced redundancy but BGP remains stable. Customers in CHI PoP are impacted only if their PE has no reachable RR, which only happens if all three RRs fail simultaneously. This is the argument for 3 geographically diverse RRs.

3. **A customer reports that traffic from their NYC site to their LA site is taking an unusual path (NY → ATL → DAL → LA instead of NY → CHI → LA). The path should prefer the NY-CHI-LA corridor. Diagnose the possible causes.**

   *Discussion:* Possible causes: (a) IS-IS metric on NY-CHI link modified (increased), making ATL-DAL cheaper; (b) SR-TE policy steering traffic onto an alternate path; (c) TI-LFA activated a backup path due to NY-CHI link issue; (d) BFD false positive caused IS-IS to delete the NY-CHI adjacency temporarily. Diagnosis: `show isis topology`, `show route`, `show mpls forwarding`, check SR-TE policy state, check BFD session history. Key question: is this happening at IS-IS level (check SPF results) or SR-TE level (check active policies)?

4. **ClearPath wants to offer a latency-sensitive service (financial trading, real-time video) with a guaranteed < 5ms NY to Chicago RTT. The current IGP path is optimal but the customer wants SLA guarantees. Design the solution.**

   *Discussion:* Guaranteed < 5ms NY-CHI requires: (a) dedicated SR-TE policy on the direct NY-CHI fiber path (not via ring), (b) telemetry monitoring for latency not just utilization, (c) admission control — only traffic with appropriate DSCP marking or community gets steered onto the policy. The SR-TE policy uses explicit SIDs: [NY-P1-SID, CHI-P1-SID] — a 2-label stack forcing the direct path. Fallback (if link fails): traffic shifts to best-effort BGP path (SLA violated, customer notified). Customer accepts: SLA has force-majeure exclusion for fiber cuts. Implement: SR-TE via PCE, DSCP-based steering at PE.

5. **During a planned maintenance window, you need to drain traffic from NY-P1 (a core P router) without blackholing any flows. Walk through the exact sequence of steps.**

   *Discussion:* (a) Raise IS-IS metric on all NY-P1 links to maximum (65535 for wide metrics) — this makes NY-P1 unattractive without taking it down. (b) Wait for SPF convergence (< 1s) and traffic to shift to alternate paths — verify via telemetry (interface counters should drop near zero). (c) Verify no SR-TE policies explicitly route through NY-P1 — if any do, update them first. (d) Graceful shutdown: `isis graceful-shutdown` or equivalent — sends IS-IS GR notification to neighbors, suppressing overloaded router from being used in new SPF calculations. (e) Verify BGP paths don't use NY-P1 as next-hop resolver. (f) Take the router offline. Reverse: bring up with high metric, verify adjacencies, restore original metrics.

## Key Design Takeaways

| Decision | Chosen | Rejected | Why |
|----------|--------|----------|-----|
| IGP | IS-IS L1/L2 | OSPF | Multi-topology, hierarchy, single process for IPv4+IPv6 |
| MPLS signaling | SR-MPLS | LDP + RSVP-TE | No signaling protocol, smaller LFIB, TE without RSVP |
| FRR | TI-LFA | RLFA/IPFRR | Loop-free without topology constraints, any topology |
| BGP topology | RR (3 geographically distributed) | Full mesh | O(n) vs O(n²) sessions, fault isolation |
| TE policy | SR-TE via PCEP | RSVP-TE | Stateless at P routers, simpler operations |
| Peering security | RPKI + IRR | Trust-based | Route hijack prevention (BGP is insecure by default) |
| Provisioning | NETCONF + Ansible | CLI + change tickets | Repeatability, audit trail, < 15 min SLA |
