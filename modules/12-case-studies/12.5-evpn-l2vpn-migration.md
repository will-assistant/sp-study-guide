# 12.5 — Design Case Study: SP EVPN L2VPN Migration

> **Level:** Internet Expert / IE-SP  
> **Prerequisites:** Module 5 (L2VPN), Module 7 (EVPN), Module 12.2 (DCI with EVPN)

---

## Overview

> L2VPN migration is the single most common large-scale network transformation facing service providers today. Every SP that deployed VPWS and VPLS over the last 15 years now faces the same problem: thousands of legacy L2VPN services that must move to EVPN without customer-impacting outages. This case study walks through the full migration — planning, interworking, cutover, and validation — for a realistic Tier-2 SP environment.

**Scenario: TransitLink Communications**

TransitLink is a Tier-2 service provider operating across 28 PoPs in the southeastern US. Their L2VPN portfolio serves 1,400 enterprise customers across financial services, healthcare, and manufacturing verticals. The legacy infrastructure:

- **~5,000 VPWS pseudowires** — point-to-point Ethernet circuits (targeted LDP signaled)
- **~200 VPLS instances** — multipoint bridged domains (BGP-signaled, RFC 4761)
- **PE fleet:** 84 Cisco ASR 9000 (IOS-XR 7.x) and 42 Juniper MX480/MX960 (Junos 22.x)
- **Transport:** SR-MPLS backbone (IS-IS, SRGB 16000–23999), migrated from LDP 18 months ago
- **Multi-homing:** MC-LAG on ~300 dual-homed CE sites (active/standby)
- **SLA commitments:** 99.99% availability, <50ms failover, contractual penalties for outages

**What you'll design:** A phased migration from legacy L2VPN to EVPN — VPWS to EVPN-VPWS (RFC 8214), VPLS to EVPN-MPLS, MC-LAG to EVPN all-active multi-homing — with seamless interworking during the migration window and rollback procedures at every stage.

## Why Migrate: The Case Against Staying on Legacy L2VPN

Before committing engineering resources and risk budget, you need a defensible business case. Here's what's breaking.

### Problem 1: VPLS MAC Learning and BUM Flooding

```
Current state: 200 VPLS instances, average 15 PEs per instance
MAC learning: data-plane learned — every unknown unicast floods to all PEs
BUM traffic: broadcast/unknown-unicast/multicast floods via ingress replication
  → 200 instances × 15 PEs × average 50 MACs = ~150K MAC entries learned via flooding
  → Each BUM frame replicated 14 times (to every other PE in the instance)
Result: 30% of backbone bandwidth on some links is BUM replication traffic
```

**The EVPN argument:** EVPN advertises MAC addresses via BGP Type-2 routes. MACs are learned in the control plane — no data-plane flooding required for known unicast. ARP suppression intercepts ARP requests at the ingress PE and responds locally from the EVPN route table, eliminating most broadcast traffic. BUM handling shifts to ingress replication lists built from Type-3 Inclusive Multicast routes — predictable, controllable, and monitorable via BGP.

### Problem 2: No Native Multi-Homing

```
Current: MC-LAG on 300 dual-homed sites
MC-LAG requires: inter-chassis link (ICL), ICCP session, dedicated config per pair
Active/standby only: one PE forwards, other is hot standby
Failover: ICCP detects failure → standby takes over → 200-500ms convergence
Problem: half the PE capacity wasted (standby PE receives no traffic)
Problem: ICL bandwidth consumed by state sync, not revenue traffic
Problem: MC-LAG is pair-locked — cannot do 3-way or N-way redundancy
```

**The EVPN argument:** EVPN Ethernet Segment (ES) multi-homing supports all-active forwarding across 2+ PEs with no inter-chassis link required. DF election (Type-4 routes) determines BUM forwarding responsibility per VLAN. All PEs in the ES forward known unicast simultaneously — full bandwidth utilization. Aliasing (Type-1 per-EVI routes) enables remote PEs to load-balance across all ES members. Failover is BGP withdrawal speed — sub-second with BFD-triggered fast withdraw. Mass withdrawal (Type-1 per-ES routes) handles PE failure for all services in a single BGP update.

### Problem 3: Signaling Scale Limits

```
VPWS: 5,000 pseudowires × 2 endpoints = 10,000 targeted LDP sessions
  → Each PW requires point-to-point LDP session between PE pair
  → Adding a new PE requires establishing sessions to every peer PE
VPLS: BGP-signaled (RFC 4761) — better than LDP-signaled, but:
  → Still full-mesh within each VPN instance
  → No route-target constrain on many legacy implementations
  → BGP table: 200 instances × 15 PEs × AD routes = ~3,000 BGP L2VPN routes
```

**The EVPN argument:** EVPN uses a single BGP address family (AFI 25, SAFI 70) for all L2VPN services — VPWS and multipoint. Route reflectors handle distribution. Route-target constrain (RFC 4684) limits what each PE receives. One BGP session per PE to RR replaces thousands of targeted LDP sessions for VPWS. The signaling plane collapses from O(n²) to O(n).

### Problem 4: No Host Mobility

```
Current: when a VM or workload moves between sites, MAC appears on new PE
VPLS: old PE still has stale MAC entry → traffic black-holes until aging timer expires
  → Default aging: 300 seconds (5 minutes of potential black-holing)
  → Manual clear required for immediate recovery
EVPN: MAC Mobility extended community (sequence number) in Type-2 routes
  → New PE advertises MAC with incremented sequence → old PE withdraws → convergence in seconds
```

### Problem 5: No Integrated L2+L3

```
Current: L2VPN (VPWS/VPLS) and L3VPN (VRF) are separate service constructs
  → Customer needs L2 extension + L3 routing = two overlays, two configs, two troubleshooting paths
EVPN: Type-2 routes carry both MAC and IP → integrated IRB (Integrated Routing and Bridging)
  → Symmetric or asymmetric IRB in single EVPN instance
  → Anycast gateway with shared MAC across PEs → seamless L2/L3
```

## Migration Strategy Decision

Three approaches. Only one is viable at TransitLink's scale.

### Option 1: Big Bang Cutover

```
Approach: Schedule maintenance window, convert all services simultaneously
Pros: Clean break, no interworking complexity
Cons: 5,200 services in one window = impossible
  → Single failure affects all customers simultaneously
  → No rollback path once committed
  → Contractual SLA violations guaranteed
  → Staff: would require 40+ engineers simultaneously
Verdict: REJECTED — unacceptable risk at any scale above 50 services
```

### Option 2: Parallel Overlay (Chosen)

```
Approach: Build EVPN overlay alongside legacy, migrate services one at a time
  → Both legacy and EVPN coexist on same PE routers
  → Each service migrated independently during per-customer maintenance windows
  → Rollback = revert individual service to legacy config
Pros: Granular control, per-service rollback, proven at scale
Cons: Temporary double resource consumption (both overlays active)
  → BGP table grows during transition (legacy + EVPN routes)
  → Operational complexity: staff must manage both technologies
Verdict: PRIMARY STRATEGY — used for majority of services
```

### Option 3: Seamless VPWS-to-EVPN Interworking (RFC 9572)

```
Approach: EVPN PEs and legacy PEs coexist in the same VPN instance
  → EVPN PE advertises both EVPN routes and legacy LDP/BGP L2VPN routes
  → Traffic forwarded between EVPN and legacy PEs via interworking function
  → Gradual PE-by-PE migration without per-service cutovers
Pros: No per-service maintenance window needed
  → Each PE migrated independently — service follows automatically
Cons: Interworking complexity, vendor support varies
  → Not all platforms support seamless mode for all service types
  → Debugging interworking issues requires deep protocol knowledge
Verdict: SECONDARY STRATEGY — used for large VPLS instances where
  per-service cutover is impractical (50+ PE instances)
```

**TransitLink's hybrid approach:** Parallel overlay for VPWS (simple, low-risk per-service cutover). Seamless interworking for the 20 largest VPLS instances (50+ PEs each). Parallel overlay for remaining VPLS. MC-LAG to EVPN multi-homing as final phase.

## Phase 0: Pre-Migration Infrastructure

Before touching any service, prepare the EVPN control plane.

### Step 0.1: BGP EVPN Address Family Activation

EVPN requires BGP AFI 25 / SAFI 70 on all PEs and route reflectors.

**IOS-XR (ASR 9000):**

```
router bgp 65000
 address-family l2vpn evpn
 !
 neighbor-group RR-CLIENTS
  address-family l2vpn evpn
   route-reflector-client
  !
 !
 neighbor 10.0.0.1
  use neighbor-group RR-CLIENTS
  address-family l2vpn evpn
 !
```

**Junos (MX):**

```
protocols bgp {
    group EVPN-RR {
        type internal;
        family evpn signaling;
        neighbor 10.0.0.1;
        neighbor 10.0.0.2;
    }
}
```

**Rollback:** Deactivating `address-family l2vpn evpn` removes all EVPN routes. No impact to legacy L2VPN services since they use separate address families (L2VPN VPLS for BGP-signaled VPLS, or targeted LDP for VPWS).

### Step 0.2: Route-Target Constrain (RFC 4684)

Essential to prevent every PE from receiving every EVPN route.

**IOS-XR:**

```
router bgp 65000
 address-family l2vpn evpn
  retain route-target all    ! on RRs only — retain for re-advertisement
 !
 neighbor-group PE-PEERS
  address-family l2vpn evpn
   route-policy RT-CONSTRAIN in
  !
```

**Junos:**

```
protocols bgp {
    group EVPN-RR {
        family evpn signaling;
    }
    group INTERNAL {
        family route-target;   /* RFC 4684 */
    }
}
```

### Step 0.3: Control-Plane Impact Assessment

**BGP table growth projection:**

```
Current L2VPN BGP routes:
  VPLS (RFC 4761):     ~3,000 AD routes
  VPWS (targeted LDP): 0 BGP routes (LDP signaled)

Projected EVPN routes after full migration:
  EVPN-VPWS:
    Type-1 (per-EVI AD): 5,000 (one per PW endpoint)
    Type-3 (IMET):       0 (VPWS doesn't use Type-3)
    Subtotal:             5,000 routes

  EVPN multipoint (replacing VPLS):
    Type-1 (per-EVI AD): 200 instances × 15 PEs = 3,000
    Type-2 (MAC/IP):     200 instances × avg 50 MACs = 10,000
    Type-3 (IMET):       200 instances × 15 PEs = 3,000
    Subtotal:             16,000 routes

  Multi-homing:
    Type-1 (per-ES):     300 ES × 2 PEs = 600
    Type-4 (ES):         300 ES × 2 PEs = 600
    Subtotal:             1,200 routes

  Total EVPN:             ~22,200 routes
  During migration peak:  ~25,200 (legacy + EVPN coexist)

RR memory impact:  ~25K routes × ~500 bytes/route = ~12.5 MB
  → Negligible on modern RR platforms (BGP internet table is 1M+ routes)
PE memory impact:  With RT-constrain, each PE receives only relevant routes
  → Average PE: ~2,000 EVPN routes (manageable)
```

### Step 0.4: EVPN Route Distinguisher and Route Target Design

```
RD convention:  <loopback>:<service-id>
  → Ensures globally unique RD per PE per service
  → Example: 10.0.0.10:1001 for PE 10.0.0.10, service ID 1001

RT convention:  65000:<service-id>
  → Same RT as legacy VPLS instance where possible (simplifies interworking)
  → New EVPN-only services: allocate from reserved range 65000:50000–65000:59999

VPWS service IDs:  Reuse existing VPWS circuit IDs (1–5000)
VPLS/EVPN IDs:     Reuse existing VPLS instance IDs (10001–10200)
```

## Phase 1: VPWS to EVPN-VPWS Migration (RFC 8214)

VPWS is the simplest migration — point-to-point services with exactly two endpoints. Migrate these first to build operational confidence.

### EVPN-VPWS Architecture

EVPN-VPWS uses Type-1 (Ethernet Auto-Discovery) routes to signal pseudowire endpoints. Each endpoint advertises a per-EVI Type-1 route containing:

- **Ethernet Tag ID:** Identifies the specific pseudowire within the EVI
- **MPLS label:** Locally allocated, carried in the route
- **Route Target:** Maps to the VPWS service instance

No Type-2 or Type-3 routes — EVPN-VPWS is purely point-to-point signaling.

### Legacy VPWS Configuration (Before)

**IOS-XR — Targeted LDP VPWS:**

```
l2vpn
 xconnect group CUST-A
  p2p CUST-A-PW-1001
   interface GigabitEthernet0/0/0/1.100
   neighbor 10.0.0.20 pw-id 1001
   !
  !
 !
```

**Junos — Targeted LDP VPWS:**

```
protocols l2circuit {
    neighbor 10.0.0.20 {
        interface ge-0/0/1.100 {
            virtual-circuit-id 1001;
            encapsulation-type ethernet-vlan;
        }
    }
}
```

### EVPN-VPWS Configuration (After)

**IOS-XR — EVPN-VPWS:**

```
evpn
 evi 1001
  bgp
   route-target import 65000:1001
   route-target export 65000:1001
  !
  advertise-mac
  !
 !
!
l2vpn
 xconnect group CUST-A
  p2p CUST-A-EVPN-1001
   interface GigabitEthernet0/0/0/1.100
   neighbor evpn evi 1001 target 1001 source 1001
   !
  !
 !
```

**Junos — EVPN-VPWS:**

```
routing-instances {
    CUST-A-VPWS-1001 {
        instance-type evpn-vpws;
        protocols evpn {
            interface ge-0/0/1.100 {
                vpws-service-id {
                    local 1001;
                    remote 1001;
                }
            }
        }
        interface ge-0/0/1.100;
        route-distinguisher 10.0.0.10:1001;
        vrf-target target:65000:1001;
    }
}
```

### Per-Service Migration Runbook: VPWS

**Pre-cutover (during business hours):**

1. Stage EVPN-VPWS config on both PEs (do not commit yet)
2. Verify EVPN BGP session is established: `show bgp l2vpn evpn summary`
3. Confirm no EVPN route conflicts for the target EVI
4. Notify customer of maintenance window (per SLA terms)
5. Prepare rollback config (legacy VPWS config saved to file)

**Cutover sequence (maintenance window, ~5 minutes per service):**

```
Step 1: On PE-A, remove legacy VPWS config
  IOS-XR:  no l2vpn xconnect group CUST-A p2p CUST-A-PW-1001
  Junos:   deactivate protocols l2circuit neighbor 10.0.0.20 interface ge-0/0/1.100

Step 2: On PE-A, commit EVPN-VPWS config
  IOS-XR:  commit
  Junos:   commit confirmed 5   ← auto-rollback in 5 minutes if not confirmed

Step 3: Verify PE-A advertises EVPN Type-1 route
  IOS-XR:  show bgp l2vpn evpn rd 10.0.0.10:1001
  Junos:   show route table CUST-A-VPWS-1001.evpn.0

Step 4: Repeat Steps 1-3 on PE-B
  → At this point both endpoints are EVPN-VPWS

Step 5: Verify end-to-end connectivity
  → Ping from CE-A to CE-B
  → Verify MAC learning (if applicable)
  → Check counters: packets in/out incrementing on both ACs

Step 6: Confirm commit on both PEs
  Junos:   commit   ← finalizes the commit confirmed
```

**Rollback (if Step 5 fails):**

```
On Junos: Do nothing — commit confirmed auto-reverts after 5 minutes
On IOS-XR:
  Step R1: Remove EVPN-VPWS config
    no evpn evi 1001
    no l2vpn xconnect group CUST-A p2p CUST-A-EVPN-1001
  Step R2: Re-apply legacy VPWS config from saved file
    l2vpn xconnect group CUST-A p2p CUST-A-PW-1001 ...
  Step R3: Verify legacy PW comes up
    show l2vpn xconnect group CUST-A detail
```

**Outage window per service:** 30 seconds to 2 minutes (time between removing legacy and EVPN coming up). Both endpoints must be cut over — during the gap (one legacy, one EVPN), the pseudowire is down.

> **Optimization:** For critical circuits, use EVPN-VPWS with dual-homing to create a make-before-break path. Stand up the EVPN-VPWS on a secondary AC, verify it works, then swing the primary AC. This reduces outage to sub-second but requires a spare interface/sub-interface.

### VPWS Migration Capacity Plan

```
Services to migrate:    5,000 VPWS pseudowires
Cutover time per PW:    ~5 minutes (including validation)
PW cutovers per window: 20 per night (conservative, 2 engineers)
Maintenance windows:    4 per week (Tue/Wed/Thu/Sat nights, 10pm–2am)
Duration:               5,000 / 20 / 4 = 62.5 weeks

Accelerated plan:       4 engineers, 40 PW per night
Duration:               5,000 / 40 / 4 = 31 weeks (~7 months)

Batch optimization:     Group PWs by PE pair — all PWs between PE-A↔PE-B
  migrated in same window. Average 8 PWs per PE pair.
  → 625 PE-pair batches / 40 per night / 4 nights = ~16 weeks (4 months)
```

## Phase 2: VPLS to EVPN Multipoint Migration

VPLS migration is more complex — multipoint domains with MAC learning, BUM replication, and potentially dozens of PEs per instance.

### EVPN-MPLS Replacing VPLS

EVPN uses three route types for multipoint L2:

| Route Type | Function | Replaces in VPLS |
|-----------|----------|------------------|
| Type-1 (AD per-EVI) | Auto-discovery of PEs in the EVI | BGP VPLS AD route (RFC 4761) |
| Type-2 (MAC/IP) | Control-plane MAC learning | Data-plane MAC learning (flooding) |
| Type-3 (IMET) | BUM replication tree membership | Implicit full-mesh replication |

**Key behavioral changes from VPLS:**

- **MAC learning:** Moves from data plane to control plane. Unknown unicast flooding drops to near zero once all MACs are learned via BGP.
- **BUM handling:** Type-3 IMET routes build explicit ingress replication lists. Each PE knows exactly which peers need BUM traffic. No change in replication behavior, but it's now visible and controllable.
- **Split-horizon:** EVPN uses ESI-based split-horizon (Type-1 per-ES routes) instead of VPLS split-horizon groups. Remote PEs don't send BUM back to the originating ES.
- **Designated Forwarder (DF):** In multi-homed scenarios, Type-4 ES routes elect a DF per VLAN per ES. Only the DF forwards BUM into the ES — prevents duplication. This replaces the MC-LAG active/standby model.

### Strategy A: Parallel Overlay (Small/Medium VPLS Instances)

For VPLS instances with ≤20 PEs, use parallel overlay — stand up EVPN alongside legacy, migrate all PEs, then decommission legacy.

### Per-Service Migration Runbook: VPLS (Parallel Overlay)

**Pre-cutover:**

1. Identify all PEs participating in the VPLS instance
2. Stage EVPN config on ALL PEs (do not activate)
3. Schedule a single maintenance window for all PEs in this instance
4. Verify no MAC address conflicts or duplicate RD/RT assignments

**Cutover sequence (all PEs must be migrated in the same window):**

```
Step 1: On ALL PEs simultaneously (scripted via NETCONF):
  Remove legacy VPLS binding from bridge domain
  Add EVPN EVI binding to bridge domain
  
  IOS-XR per PE:
    l2vpn bridge group CUST-B bridge-domain CUST-B-BD-10001
      no vfi CUST-B-VFI-10001
      evi 10001
    commit

  Junos per PE:
    deactivate routing-instances CUST-B-VPLS-10001
    activate routing-instances CUST-B-EVPN-10001
    commit confirmed 5

Step 2: Verify EVPN Type-3 IMET routes (one per PE in the EVI)
  IOS-XR: show bgp l2vpn evpn route-type 3 rd 10.0.0.10:10001
  Junos:  show route table CUST-B-EVPN-10001.evpn.0 match-prefix "3:*"

Step 3: Verify EVPN Type-2 MAC routes populate
  IOS-XR: show bgp l2vpn evpn route-type 2 rd 10.0.0.10:10001
  Junos:  show route table CUST-B-EVPN-10001.evpn.0 match-prefix "2:*"
  → MACs appear as traffic flows or ARP is exchanged
  → Initial convergence: 5-30 seconds for MAC population

Step 4: Verify end-to-end L2 connectivity
  → Ping between all CE sites in the VPLS domain
  → Verify MAC count matches expected (compare to pre-migration baseline)
  → Check BUM replication: Type-3 route count = number of active PEs

Step 5: Confirm commits on all Junos PEs
  → commit    (finalizes commit confirmed on each)

Step 6: Monitor for 30 minutes — watch for:
  → MAC flapping (show l2vpn bridge-domain mac-address flapping)
  → Unknown unicast flooding (should decrease as MACs learn via BGP)
  → BUM traffic levels (should remain stable or decrease)
```

**Rollback:**

```
On Junos: If within 5-minute commit confirmed window, auto-reverts
On IOS-XR:
  l2vpn bridge group CUST-B bridge-domain CUST-B-BD-10001
    no evi 10001
    vfi CUST-B-VFI-10001
      vpn-id 10001
      autodiscovery bgp
        rd 10.0.0.10:10001
        route-target 65000:10001
        signaling-protocol bgp
          ve-id 10
  commit
```

**Critical constraint:** All PEs in the VPLS instance must be migrated in the same window. You cannot have some PEs on legacy VPLS and others on EVPN (parallel overlay) — they use different signaling and won't interoperate. This is why large VPLS instances use seamless interworking (Strategy B) instead.

### Strategy B: Seamless Interworking (Large VPLS Instances)

For the 20 largest VPLS instances (50+ PEs each), migrating all PEs in a single window is impractical. Seamless interworking allows EVPN PEs and legacy VPLS PEs to coexist in the same VPN instance during migration.

**How seamless interworking works:**

```
                VPLS PE-1          EVPN PE-2          VPLS PE-3
                    │                  │                   │
                    │◄── BGP VPLS ────►│◄── BGP VPLS ────►│
                    │     (RFC 4761)   │    (RFC 4761)     │
                    │                  │                   │
                    │                  │◄── BGP EVPN ─────►│ (future)
                    │                  │                   │
                    │    Interworking  │                   │
                    │    PE: speaks    │                   │
                    │    BOTH protocols│                   │
```

The interworking PE (EVPN PE-2 in the diagram) participates in both the legacy VPLS signaling and EVPN signaling simultaneously:

1. **Receives VPLS AD routes** from legacy PEs → establishes pseudowires to them
2. **Advertises EVPN Type-1/3 routes** to other EVPN PEs → builds EVPN overlay
3. **Bridges traffic** between VPLS pseudowires and EVPN fabric
4. **Translates MAC learning:** MACs learned via VPLS data plane are re-advertised as EVPN Type-2 routes. MACs learned via EVPN Type-2 are installed in the VPLS forwarding table.

**IOS-XR — Seamless interworking (EVPN PE):**

```
evpn
 evi 10001
  bgp
   route-target import 65000:10001
   route-target export 65000:10001
  !
  advertise-mac
  !
 !
!
l2vpn
 bridge group CUST-B
  bridge-domain CUST-B-BD-10001
   interface GigabitEthernet0/0/0/2.200
   !
   evi 10001
   !
   vfi CUST-B-VFI-10001
    vpn-id 10001
    autodiscovery bgp
     rd 10.0.0.10:10001
     route-target 65000:10001
     signaling-protocol bgp
      ve-id 10
     !
    !
   !
  !
 !
```

**Key detail:** Both `evi 10001` and `vfi CUST-B-VFI-10001` are bound to the same bridge domain. The PE bridges between them. Traffic from a legacy VPLS PE arrives on a VFI pseudowire, is bridged into the bridge domain, and can exit via the EVPN EVI toward EVPN PEs — and vice versa.

**Junos — Seamless interworking:**

```
routing-instances {
    CUST-B-10001 {
        instance-type evpn;
        protocols {
            evpn;
        }
        interface ge-0/0/2.200;
        route-distinguisher 10.0.0.10:10001;
        vrf-target target:65000:10001;
    }
}
```

> **Junos note:** On MX, EVPN-MPLS instances can interwork with VPLS when both use the same route target. Junos handles the interworking implicitly when EVPN and VPLS routes with the same RT are present — the EVPN instance processes both EVPN and VPLS NLRI. Verify platform and version support before deploying.

### Per-PE Migration Runbook: VPLS (Seamless Interworking)

With seamless interworking, you migrate PE-by-PE instead of per-service.

```
Migration order for VPLS instance 10001 (15 PEs):
  Wave 1: Migrate 2 PEs to EVPN+interworking (low-traffic PEs first)
  Wave 2: Migrate 4 more PEs (after 1-week soak)
  Wave 3: Migrate 5 more PEs
  Wave 4: Migrate final 4 PEs
  Wave 5: Remove legacy VPLS config from all PEs (interworking no longer needed)

Per-PE cutover:
  Step 1: Add EVPN EVI to existing bridge domain (alongside legacy VFI)
    → PE now speaks both VPLS and EVPN
    → No traffic impact — existing VPLS forwarding continues
    → EVPN routes begin advertising to EVPN RR

  Step 2: Verify interworking
    → Traffic between this PE and legacy PEs: via VPLS pseudowires (unchanged)
    → Traffic between this PE and other EVPN PEs: via EVPN
    → show l2vpn bridge-domain CUST-B-BD-10001 detail
      → Should see both VFI pseudowires and EVPN peers

  Step 3: After all PEs are EVPN-capable (Wave 4 complete):
    Remove VFI from all PEs — interworking no longer needed
    → Pure EVPN multipoint

Rollback (any wave):
  Remove EVPN EVI from bridge domain
    → PE reverts to pure VPLS operation
    → No impact to other PEs (they still see this PE via VPLS)
```

**Advantage of seamless interworking:** Rollback at any PE is non-disruptive. Removing the EVPN EVI from the bridge domain returns the PE to legacy VPLS — other PEs (whether EVPN or legacy) continue operating. The interworking PEs handle the translation.

## Phase 3: MC-LAG to EVPN Multi-Homing Migration

This is the most operationally sensitive phase. MC-LAG provides dual-homing today — breaking it incorrectly causes customer outages.

### Current MC-LAG Architecture

```
         CE (dual-homed)
        /              \
     LAG               LAG
      |                  |
   PE-A (active)     PE-B (standby)
      |                  |
      └──── ICL ─────────┘
           (ICCP)
```

- **ICCP** (Inter-Chassis Communication Protocol) synchronizes state between PE-A and PE-B
- **ICL** (Inter-Chassis Link) carries data when needed (e.g., BUM from active to standby for local delivery)
- **Active/standby:** CE sees single LAG, PE-A forwards all traffic, PE-B is hot standby
- **Failover:** PE-A failure → ICCP detects → PE-B becomes active (200-500ms)

### EVPN All-Active Multi-Homing Target

```
         CE (dual-homed)
        /              \
     LAG               LAG
      |                  |
   PE-A (active)     PE-B (active)    ← both forward!
      |                  |
      └── BGP EVPN ──────┘
          (no ICL needed)
```

- **Ethernet Segment (ES):** Both PEs configured with same ESI (10-byte identifier)
- **Type-4 ES routes:** PEs discover each other as members of the same ES, elect DF per VLAN
- **Type-1 per-ES routes:** Enable mass withdrawal on PE failure (all services, one BGP update)
- **Type-1 per-EVI routes:** Enable aliasing — remote PEs load-balance across both ES members
- **All-active:** Both PEs forward known unicast traffic simultaneously → double bandwidth
- **DF election:** Per VLAN, one PE is DF for BUM traffic → prevents duplication into ES

### ESI Assignment Strategy

```
ESI format: Type 0 (manually configured), 10 bytes
Convention: 00:00:00:00:00:00:00:XX:YY:ZZ
  XX = rack/site identifier (01-FF)
  YY = PE-pair identifier (01-FF)
  ZZ = interface/bundle identifier (01-FF)

Example for site 42, PE pair 1, bundle 1:
  ESI: 00:00:00:00:00:00:00:2A:01:01

LACP system-id: Must match across PE-A and PE-B
  → CE sees single LAG partner
  → Convention: 00:00:00:2A:01:01 (derived from ESI)
```

### EVPN Multi-Homing Configuration

**IOS-XR (PE-A):**

```
evpn
 interface Bundle-Ether42
  ethernet-segment
   identifier type 0 00.00.00.00.00.00.00.2a.01.01
   bgp route-target 002a.0101.0000
   load-balancing-mode per-flow
  !
 !
!
```

**IOS-XR (PE-B) — identical ESI:**

```
evpn
 interface Bundle-Ether42
  ethernet-segment
   identifier type 0 00.00.00.00.00.00.00.2a.01.01
   bgp route-target 002a.0101.0000
   load-balancing-mode per-flow
  !
 !
!
```

**Junos (PE-A):**

```
interfaces {
    ae42 {
        esi {
            00:00:00:00:00:00:00:2a:01:01;
            all-active;
        }
        aggregated-ether-options {
            lacp {
                system-id 00:00:00:2a:01:01;
                active;
            }
        }
    }
}
```

### Per-Site Migration Runbook: MC-LAG to EVPN Multi-Homing

**Prerequisites:**
- Phase 2 complete — VPLS/VPWS already migrated to EVPN on both PEs
- CE LAG configuration unchanged — CE is unaware of migration

```
Step 1: Pre-stage EVPN ES config on both PEs (do not commit)
  → Prepare ESI, LACP system-id, load-balancing mode

Step 2: On PE-B (standby), remove MC-LAG/ICCP config
  → PE-B standby interface goes down
  → All traffic now flows through PE-A (active) — no outage
  
  IOS-XR:
    no redundancy iccp group 42
    no l2vpn bridge group ... neighbor ... pw-class MC-LAG-PW
    commit

Step 3: On PE-B, apply EVPN ES config + commit
  → PE-B comes up as EVPN ES member
  → But PE-A is still MC-LAG active — PE-B won't forward yet (no ES peer)

Step 4: On PE-A (active), remove MC-LAG/ICCP config
  → Brief outage (CE LAG detects member down on PE-A side)
  
Step 5: On PE-A, apply EVPN ES config + commit
  → PE-A comes up as EVPN ES member
  → Type-4 ES route exchange → DF election
  → Both PEs negotiate all-active forwarding
  → CE LAG re-establishes on both members

Step 6: Verify multi-homing
  IOS-XR:
    show evpn ethernet-segment interface Bundle-Ether42 detail
    → Verify: ES state = Up, Forwarder = All-active
    → Verify: DF election complete, peer PE listed

  Junos:
    show evpn instance extensive
    → Verify: ESI, all-active, DF status per VLAN

Step 7: Verify traffic load-balancing
  → Monitor per-PE interface counters
  → Both PEs should show traffic (all-active)
  → Compare to pre-migration: total throughput should be similar, split across PEs

Step 8: Remove ICL (inter-chassis link)
  → Only after 1-week soak period
  → ICL is no longer needed with EVPN multi-homing
  → Reclaim ICL ports for revenue traffic
```

**Outage window:** Steps 4-5 cause a brief outage (CE LAG reconverges). Expected duration: 1-3 seconds (LACP fast timers) to 30 seconds (LACP slow timers).

> **Recommendation:** Ensure CE LAG uses LACP fast timers (1-second intervals) before migration. Coordinate with customer to verify LACP settings.

**Rollback:**

```
If Step 5 fails (PE-A EVPN ES doesn't come up):
  Step R1: Remove EVPN ES config from PE-A
  Step R2: Re-apply MC-LAG/ICCP config on PE-A
  Step R3: PE-A becomes MC-LAG active again
  Step R4: Remove EVPN ES config from PE-B
  Step R5: Re-apply MC-LAG/ICCP config on PE-B
  Step R6: Verify MC-LAG pair is operational
  → Full rollback to pre-migration state

If Step 6 shows partial failure (one PE not DF):
  → Check Type-4 route exchange: show bgp l2vpn evpn route-type 4
  → Common issue: ESI mismatch (typo), RT mismatch, or BGP session down
  → Fix config and re-verify before proceeding
```

## Risk Matrix

| Phase | Risk | Likelihood | Impact | Mitigation |
|-------|------|-----------|--------|------------|
| 0 - BGP activation | EVPN AFI activation causes BGP session flap | Low | High — all L2VPN services down if BGP resets | Test on lab/non-prod PE first; activate during maintenance window; use `soft-reconfiguration` |
| 0 - RT constrain | Missing RT constrain floods all EVPN routes to all PEs | Medium | Medium — memory/CPU spike on PEs | Verify RT constrain config before first EVI; monitor RIB size |
| 1 - VPWS cutover | Both endpoints not cut simultaneously → PW down | High | Low — single service, planned window | Script both-end cutover; pre-stage configs |
| 1 - VPWS cutover | EVPN-VPWS label not programmed in hardware | Low | High — service black-holes silently | Verify `show mpls forwarding` shows EVPN labels; test on non-prod PW first |
| 2 - VPLS parallel | Not all PEs migrated in same window | Medium | High — split-brain, some PEs on VPLS, others EVPN, no interop | Use automation (NETCONF) to cut all PEs simultaneously; fallback to seamless interworking |
| 2 - VPLS interworking | Interworking PE CPU overload (bridging + translating) | Medium | High — interworking PE becomes bottleneck | Monitor CPU during soak; limit interworking PEs to high-capacity platforms |
| 2 - VPLS MAC | MAC address table explosion during interworking | Low | Medium — MAC scale limit hit, flooding resumes | Set MAC limit per bridge domain; monitor MAC count vs baseline |
| 3 - MC-LAG removal | CE LAG doesn't reconverge after MC-LAG removal | Medium | High — customer outage | Verify LACP timers pre-migration; have console access to CE if possible |
| 3 - ESI mismatch | ESI typo → PEs don't recognize same ES | High | High — no multi-homing, single point of failure | Automate ESI assignment from database; verify Type-4 route exchange before proceeding |
| 3 - DF election | DF election fails → BUM duplication or black-hole | Low | High — duplicate frames to CE or no BUM delivery | Check Type-4 routes; verify DF status; test BUM forwarding explicitly |
| All | Software bug in EVPN implementation | Medium | High — unpredictable behavior | Run migration on well-tested software versions; check vendor bug databases; maintain rollback capability |
| All | BGP RR failure during migration | Low | Critical — all EVPN services down | Redundant RRs (minimum 2); verify RR health before each migration window |

## Design Review Questions

1. **TransitLink has a VPLS instance with 60 PEs. The project manager wants to migrate it using parallel overlay in a single maintenance window. Explain why this is risky and propose an alternative.**

   *Discussion:* Parallel overlay requires all 60 PEs to be cut over simultaneously — any single PE failure means partial VPLS/partial EVPN with no interoperability between them. If PE 47 of 60 fails to commit, you must roll back all 46 completed PEs. At 5 minutes per PE, the window is 5+ hours with no margin for troubleshooting. The alternative is seamless interworking: migrate PEs in waves of 5-10, with each wave soaking for a week. Interworking PEs bridge traffic between legacy VPLS PEs and EVPN PEs during the transition. Rollback is per-PE (remove EVPN EVI, legacy VPLS continues). Total migration time: 6-8 weeks for this instance, but with near-zero risk per cutover.

2. **During Phase 3, you remove MC-LAG from PE-A and apply EVPN ES config. The CE's LAG comes back up, but all traffic is forwarded by PE-A only — PE-B shows zero traffic. What's wrong?**

   *Discussion:* Several possibilities: (a) ESI mismatch — PE-A and PE-B have different ESI values, so they don't form an ES pair. Verify with `show evpn ethernet-segment` on both PEs, confirm ESI matches. (b) PE-B is still running MC-LAG config (Step 2 not completed) — it's in standby mode, not EVPN mode. (c) DF election completed but PE-B lost — check Type-4 routes on PE-B. (d) Remote PEs don't have aliasing (Type-1 per-EVI) routes from PE-B — check `show bgp l2vpn evpn route-type 1` for PE-B's routes on remote PEs. If aliasing routes are missing, remote PEs only know about PE-A and send all traffic there. (e) LACP system-id mismatch — CE sees two separate LAGs instead of one. Verify LACP system-id is identical on both PEs.

3. **You're halfway through Phase 2 VPLS migration using seamless interworking. A customer reports intermittent packet duplication on their VPLS instance. Some PEs are EVPN, some are legacy. Diagnose.**

   *Discussion:* Packet duplication in an interworking scenario typically means a forwarding loop exists between the VPLS and EVPN domains. Check: (a) Is the same MAC reachable via both a VPLS pseudowire and an EVPN route on an interworking PE? This can happen if split-horizon isn't correctly applied between VFI and EVI within the bridge domain. (b) Are two interworking PEs both bridging the same traffic between VPLS and EVPN? In a properly designed migration, interworking should have split-horizon — traffic received from the EVPN side should not be re-flooded to other EVPN PEs via the VPLS side. (c) Check BUM replication — the interworking PE should not replicate BUM to EVPN peers that already received it directly. Verify Type-3 IMET routes and ingress replication lists. Mitigation: reduce to a single interworking PE temporarily to eliminate the duplication path, then debug split-horizon rules.

4. **TransitLink's CFO asks: "We're 4 months into a 12-month migration. Can we run legacy and EVPN forever instead of completing the migration?" Argue for completing it.**

   *Discussion:* Running dual overlays permanently means: (a) Double operational complexity — every engineer must know both VPLS and EVPN troubleshooting. (b) Interworking PEs carry permanent translation overhead — CPU, memory, and potential bugs in interworking code paths that are less tested than pure EVPN. (c) No EVPN benefits for legacy services — MAC flooding, no host mobility, no all-active multi-homing on unmigrated services. (d) Vendor support: legacy L2VPN features receive fewer bug fixes and no new features. Eventually, new platform software may deprecate legacy VPLS. (e) ICL ports on 300 MC-LAG pairs remain consumed — ~600 ports of capacity locked up for inter-chassis links. (f) Training burden: new hires must learn both. Complete the migration: the remaining 8 months of work eliminates permanent operational tax.

5. **During Phase 1, you've migrated 2,000 of 5,000 VPWS circuits. A software bug is discovered in the EVPN-VPWS implementation on IOS-XR that causes intermittent label corruption under specific conditions. The vendor says a fix is 6 weeks away. What do you do?**

   *Discussion:* Immediate actions: (a) Stop all new migrations — no additional services move to EVPN until the fix is available. (b) Assess impact — does the bug affect all 2,000 migrated circuits or only specific conditions? If specific (e.g., certain label ranges, certain interface types), identify affected circuits. (c) For affected circuits: roll back to legacy VPWS. The rollback runbook exists and has been tested — this is exactly why rollback procedures are maintained. (d) For unaffected circuits: monitor closely, keep on EVPN. (e) When fix arrives: validate in lab, deploy to production PEs, re-migrate rolled-back circuits. (f) Timeline impact: 6-week pause + re-migration of rolled-back circuits adds ~2 months to project. Update stakeholders immediately. Key lesson: rollback capability is not optional — it's the fundamental risk mitigation that makes incremental migration viable.

## Key Design Takeaways

| Decision | Chosen | Rejected | Why |
|----------|--------|----------|-----|
| Migration strategy | Parallel overlay + seamless interworking | Big bang | Per-service rollback, manageable risk windows |
| VPWS migration | EVPN-VPWS (RFC 8214) | Keep legacy | Unified signaling, RR-based, eliminates targeted LDP |
| VPLS migration (small) | Parallel overlay | Seamless interworking | Simpler, cleaner cut, all PEs in one window |
| VPLS migration (large) | Seamless interworking | Parallel overlay | 50+ PE instances can't be cut simultaneously |
| Multi-homing | EVPN all-active ES | Keep MC-LAG | Double bandwidth, no ICL, N-way redundancy |
| Migration order | VPWS → VPLS → multi-homing | Any other order | Lowest risk first, build confidence, multi-homing last (most complex) |
| Commit safety | Junos `commit confirmed`, IOS-XR saved rollback | Unprotected commits | Auto-revert on failure is non-negotiable at this scale |
| Automation | NETCONF for batch cutovers | Manual CLI | 5,000+ services requires scripted execution and validation |
