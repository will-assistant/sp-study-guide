> **Level:** Internet Expert / IE-SP

## Overview

Segment Routing Traffic Engineering (SR-TE) can operate in two fundamental modes:

1. **Headend-computed** — each PE computes its own SR-TE candidate paths locally using CSPF or IS-IS/OSPF-TE algorithms. Simple, no external dependency.
2. **Controller-computed** — an external Path Computation Element (PCE) centrally computes paths and delegates segment lists to headend routers via PCEP. Enables global optimization, multi-layer visibility, and coordinated reroutes.

In production SP networks, the controller model dominates at scale. A PCE sees the complete topology (via BGP-LS), runs sophisticated algorithms (bandwidth calendaring, disjointness, latency optimization), and pushes optimized segment lists to hundreds of headends simultaneously.

**Key protocols:**
- **PCEP** (Path Computation Element Communication Protocol) — RFC 5440, extended by RFC 8231 (stateful), RFC 8281 (PCE-initiated), RFC 8664 (SR-TE)
- **BGP-LS** (BGP Link-State) — RFC 7752 — topology distribution to PCE
- **gRPC/gNMI** — telemetry feedback for bandwidth/utilization awareness

## PCE Architecture

### PCE Roles

```
                   ┌─────────────────────────────┐
                   │         PCE Server           │
                   │  (Cisco SR-PCE / Paragon /   │
                   │   Nokia WavePath / Huawei    │
                   │         Crosswork)           │
                   │                              │
                   │  ┌──────────┐ ┌───────────┐  │
                   │  │  TED     │ │  Path     │  │
                   │  │ (Topo)   │ │  Compute  │  │
                   │  └──────────┘ └───────────┘  │
                   └──────┬──────────────┬────────┘
                   PCEP   │              │  BGP-LS
                          │              │
           ┌──────────────┘              └──────────────┐
           │                                             │
    ┌──────▼──────┐                            ┌────────▼───────┐
    │  PCC (PE-1) │                            │  RR / Route    │
    │  Headend    │  ◄──── PCEP TCP 4189 ────► │  Reflector     │
    └─────────────┘                            └────────────────┘
```

**PCC (Path Computation Client):** Every headend router that delegates path computation to the PCE. Initiates PCEP TCP connection (port 4189) to the PCE.

**PCE (Path Computation Element):** Centralized server holding the Traffic Engineering Database (TED). Computes optimal segment lists based on constraints.

**TED (Traffic Engineering Database):** The PCE's view of the network topology — links, node SIDs, adjacency SIDs, TE metrics, bandwidth, SRLG, delay. Populated via BGP-LS from the IGP.

### Stateless vs Stateful PCE

| | Stateless PCE | Stateful PCE |
|---|---|---|
| RFC | RFC 5440 | RFC 8231 |
| State | No LSP state maintained | PCE tracks all LSP states |
| Mode | Request/Reply per computation | Continuous synchronization |
| Delegation | None | PCC delegates control to PCE |
| PCE-initiated | Not possible | RFC 8281 (PCE creates new LSPs) |
| Use case | Simple path query | SR-TE optimization at scale |

All production SR-TE deployments use **stateful PCE with delegation** — the PCE actively manages LSP segment lists and can update them without PCC request.

### PCE Delegation vs Non-Delegation

**Delegated LSP:** PCC hands control to PCE. The PCE can update the segment list at any time. PCC applies updates immediately. If PCEP session drops, PCC falls back to locally computed path (or removes the LSP, depending on config).

**Non-delegated LSP:** PCC retains control. PCE can only suggest updates via PCUpdate; PCC decides whether to apply.

**PCE-initiated LSP (RFC 8281):** PCE creates an SR policy that doesn't exist on the PCC. The PCC instantiates the SR policy based on PCE instruction. Used for centralized provisioning.

## BGP-LS: Feeding Topology to the PCE

BGP-LS (RFC 7752) exports IGP topology information into BGP so the PCE can build a global TED without running inside the IGP domain.

### How It Works

```
IS-IS/OSPF LSDB  ──►  BGP-LS Speaker (RR or PE)  ──►  PCE (BGP-LS Consumer)
                           (NLRI: Node, Link, Prefix TLVs)
```

BGP-LS uses AFI 16388 / SAFI 71. Each node, link, and prefix in the IGP is encoded as a BGP NLRI with rich TE attribute TLVs: TE metric, max BW, available BW, SRLG membership, admin groups, delay measurement, SID adjacencies, and node prefix SIDs.

**BGP-LS NLRI types:**
- **Node NLRI** — router capabilities, SRv6 capabilities, SR algorithm support
- **Link NLRI** — link TE attributes, adj-SID, LAN adj-SID
- **Prefix NLRI** — prefix SID, SR capabilities
- **TE Policy state** — SR policy state is reported via the BGP Tunnel Encapsulation Attribute (RFC 9012 / RFC 9256), NOT as a BGP-LS NLRI type. This is separate from BGP-LS topology distribution.

### IOS-XR: BGP-LS Configuration

```
! Enable BGP-LS on an RR or PE speaker
router bgp 65000
 address-family link-state link-state
 !
 neighbor 10.0.0.200       ! PCE loopback
  remote-as 65000
  update-source Loopback0
  address-family link-state link-state
  !
 !
!

! Enable IS-IS to redistribute topology into BGP-LS
router isis CORE
 distribute link-state level-2
 !
!
```

### Junos: BGP-LS Configuration

```
protocols {
    bgp {
        group TO-PCE {
            type internal;
            local-address 10.0.0.1;
            family traffic-engineering {
                unicast;
            }
            neighbor 10.0.0.200;   # PCE address
        }
    }
    isis {
        interface all;
        traffic-engineering {
            advertisement always;
        }
        export EXPORT-BGP-LS;
    }
}

policy-options {
    policy-statement EXPORT-BGP-LS {
        term ISIS-TE {
            from protocol isis;
            then accept;
        }
    }
}
```

> ⚠️ **BGP-LS scaling note:** In a large network with 500+ nodes, BGP-LS convergence can be slow — hundreds of NLRI updates on topology change. Use BGP-LS on route reflectors, not directly on every PE, to centralize the distribution point.

## PCEP Deep Dive

### Session Establishment

PCEP uses TCP port 4189. The **PCC always initiates** the connection to the PCE.

```
PCC ──► TCP SYN ──► PCE (4189)
PCC ◄── TCP SYN-ACK ◄── PCE
PCC ──► Open msg (capabilities, keepalive timer) ──► PCE
PCE ──► Open msg (capabilities) ──► PCC
PCC / PCE exchange Keepalive msgs
     ─── Session Established ───
PCC ──► PCRpt (state sync: all existing LSPs) ──► PCE
PCE ──► PCUpd (update segment lists) ──► PCC
```

**Key PCEP messages:**
| Message | Direction | Purpose |
|---------|-----------|---------|
| `Open` | Both | Session establishment, capability negotiation |
| `Keepalive` | Both | Session maintenance |
| `PCReq` | PCC → PCE | Path computation request (stateless mode) |
| `PCRep` | PCE → PCC | Path computation reply |
| `PCRpt` | PCC → PCE | State report (LSP state, segment list) |
| `PCUpd` | PCE → PCC | Update LSP (change segment list) |
| `PCInitiate` | PCE → PCC | Create/delete PCE-initiated LSP |
| `PCErr` | Both | Error notification |
| `PCNtf` | Both | Notification (bandwidth exceeded, etc.) |

### PCEP Capabilities Negotiation

During `Open`, PCC and PCE negotiate:
- **Stateful PCE** (RFC 8231): `LSP Update Capability (U)` flag
- **PCE-initiated** (RFC 8281): `LSP Instantiation Capability (I)` flag
- **SR-TE** (RFC 8664): SRPOLICY-PC-TLV, SR-ERO support
- **Segment list type**: MPLS label stack, SRv6 SID list
- **Maximum SID depth (MSD)**: Limits how deep a segment list the PCC can impose

### SR ERO (Explicit Route Object)

In PCEP, the computed path is returned as an ERO containing SR segments:

```
ERO subobject type:
  Type 36 (0x24) = SR-ERO subobject (RFC 8664)
    NT field (NAI Type, 4 bits) inside type-36 identifies address type:
      NT=1 = IPv4 node ID
      NT=3 = IPv4 adjacency
      NT=5 = IPv6 node ID
      NT=7 = IPv6 adjacency
      NT=0 = No NAI (SID only)

  Note: There is no separate Type 40 for IPv6. All SR-ERO subobjects use Type 36;
  the NAI Type (NT) field within the subobject distinguishes address families.

Example SR-ERO for path PE-1 → AGG-3 → PE-6:
  [Node SID 16003] [Adj SID 24001] [Node SID 16006]
```

## On-Demand Nexthop (ODN)

ODN is a critical SR-TE feature that automates SR policy provisioning — instead of pre-provisioning hundreds of per-VPN SR policies, the headend creates them **on demand** when a BGP route with a specific color extended community arrives.

### ODN Workflow

```
1. CE → PE-1: BGP route for 10.1.1.0/24
2. PE-2 (RR) → PE-1: BGP update with color 100 ext-community (type 0x030B, value 0x00000064)
                      and nexthop 10.0.0.50 (PE-EAST loopback)
3. PE-1: Sees color 100 + nexthop → looks up SR policy {color=100, endpoint=10.0.0.50}
4. PE-1: No such policy exists → triggers on-demand creation
5. PE-1: Creates SR policy, delegates path computation to PCE (or runs CSPF locally)
6. PCE: Returns segment list [16020][16050]
7. PE-1: Installs SR policy, steers 10.1.1.0/24 traffic via segment list
```

### IOS-XR ODN Configuration

```
! ODN on SR-TE headend
segment-routing
 traffic-eng
  on-demand color 100            ! Create SR policies on-demand for color 100
   dynamic
    pcep                         ! Delegate to PCE
    !
    metric
     type latency                ! Optimize for latency (alternative: igp, te, hopcount)
    !
   !
  !
  on-demand color 200            ! VoIP — ultra-low latency
   dynamic
    pcep
    metric
     type latency
    !
    constraints
     bandwidth 10000             ! 10 Gbps minimum available BW
    !
   !
  !
 !
!
```

### BGP Color Extended Community (Junos)

```
! On CE or PE, tag routes with color community to steer into SR-TE policy
policy-options {
    community COLOR-100 members color:100:0;
    community COLOR-200 members color:200:0;
    
    policy-statement TAG-CLOUD-ROUTES {
        term cloud-traffic {

                prefix-list CLOUD-PREFIXES;
            }

            }
        }
    }
}

protocols {
    bgp {
        group TO-RR {
            export TAG-CLOUD-ROUTES;
        }
    }
}
```

### Junos ODN Configuration

In Junos, color-based SR-TE steering (ODN equivalent) is configured via resolution RIBs and the `egress-policy` knob under `routing-options segment-routing traffic-engineering`. The PCC creates SR-TE LSPs on demand when a BGP route carrying the matching Color Extended Community is received.

```

    segment-routing {
        traffic-engineering {
            egress-policy {
                color 100;    # Enable ODN for color 100 SR-TE policies

Full syntax (including constraints like latency objectives) is configured on the SR-TE policy template level, not shown here. Verify exact knob names in your Junos platform/version — MX, PTX, and ACX may differ. Consult Juniper TechLibrary for `traffic-engineering egress-policy` under your specific release.

## Binding SID (BSID)

The Binding SID is a local label (or SRv6 SID) that represents an entire SR policy. It enables:

1. **Policy stitching** — another device can forward into the SR policy by pushing only the BSID, without knowing the full segment list
2. **Scale** — instead of distributing full segment lists in BGP, distribute just the BSID
3. **Midpoint transparency** — intermediate nodes don't need to understand the segment list

```
PE-1 SR Policy {color=100, endpoint=PE-6}:
  Binding SID: 900100
  Active segment list: [16020] [16050]

When PE-0 wants to steer traffic into PE-1's SR policy:
  Pushes: [BSID:900100]
  PE-1 receives and swaps to: [16020] [16050]
```

### IOS-XR BSID Configuration

```
segment-routing
 traffic-eng
  policy LATENCY-PATH
   binding-sid mpls 900100       ! Static BSID allocation
   color 100 end-point ipv4 10.0.0.6
   ...
```

> ⚠️ **Dynamic BSID**: If no static BSID is configured, IOS-XR dynamically allocates from the SRGB dynamic range. Static allocation is recommended for stitching use cases where other devices reference the BSID.

## Disjoint Path Computation

One of the most powerful PCE features: computing **geographically or topologically disjoint** path pairs for protection.

**Use cases:**
- Primary + backup SR-TE policy pair with zero SRLG overlap
- Active/standby VPN paths over diverse fiber routes
- Bandwidth reservation with node-disjoint guarantee

### IOS-XR Disjoint Constraints

```
segment-routing
 traffic-eng
  policy ACTIVE-PATH
   color 100 end-point ipv4 10.0.0.6
   candidate-paths
    preference 100
     dynamic
      pcep
      !
      constraints
       disjoint-path group-id 1 type node    ! Node-disjoint, group 1
      !
     !
    !
  !
  policy BACKUP-PATH
   color 200 end-point ipv4 10.0.0.6
   candidate-paths
    preference 100
     dynamic
      pcep
      !
      constraints
       disjoint-path group-id 1 type node sub-id 1  ! Same group, sub-id 1 = secondary
      !
    !
  !
 !
!
```

The PCE computes both paths simultaneously, ensuring they share no common nodes (node-disjoint) or links (link-disjoint) or SRLGs (SRLG-disjoint).

## PCEP High Availability

### Redundant PCE

The PCC maintains PCEP sessions to multiple PCE instances. Each has a **precedence value** — lower precedence = preferred.

```
! IOS-XR: Redundant PCE
segment-routing
 traffic-eng
  pcc
   pce address ipv4 10.0.0.200
    precedence 10                  ! Primary
   !
   pce address ipv4 10.0.0.201
    precedence 20                  ! Secondary
   !
  !
 !
!
```

**Failover behavior:**
1. Primary PCE session drops (TCP FIN or deadtimer expiry)
2. PCC immediately activates secondary PCE
3. PCC sends PCRpt to new PCE to re-synchronize LSP state
4. New PCE computes updated paths (may be identical or re-optimized)
5. Traffic continues uninterrupted — SR dataplane doesn't depend on PCEP

### Delegation Cleanup Timer

If PCEP session drops and no secondary PCE available:

```
! IOS-XR: Fallback behavior when no PCE available
segment-routing
 traffic-eng
  pcc
   timers
    delegation-timeout 60          ! Wait 60s before taking back delegation
    state-timeout 180              ! After 180s with no PCE, PCC recomputes locally
   !
  !
 !
!
```

## Design Considerations

### When to Use PCE vs Headend CSPF

| Criterion | PCE | Headend CSPF |
|-----------|-----|--------------|
| Network size | >200 nodes | <100 nodes |
| Multi-domain | Required | Per-domain only |
| Disjoint paths | Centralized optimization | Possible but complex |
| Bandwidth awareness | Global visibility | Local only |
| Latency optimization | Real-time telemetry feed | TE metric only |
| Operational complexity | Higher (controller infra) | Lower |
| Failure blast radius | PCE outage = fallback paths | Per-device, no single PoF |

**Rule of thumb:** Use PCE when you need coordinated optimization across the entire network. Use headend CSPF for simple topologies or when controller dependencies are unacceptable.

### PCE Placement

1. **Co-located with RR:** Common — same high-availability node, single BGP-LS feed
2. **Dedicated controller VM:** Preferred for production — isolated failure domain, upgrade independently
3. **Anycast PCE:** Multiple PCE instances with same anycast IP — PCC connects to nearest, provides automatic redundancy and geographic load distribution

### Binding SID Allocation Strategy

```
! Segment Routing label blocks:
segment-routing
 global-block
  16000 23999                ! SRGB: for node/prefix SIDs (shared globally)
 !
 local-block
  15000 15999                ! SRLB (SR Local Block): for adjacency SIDs
                             ! NOT for Binding SIDs — common misconception
 !
!

! Binding SIDs are allocated from the dynamic label range unless statically
! configured per-policy. For explicit BSID allocation, configure on the policy:
!

## War Story: PCE Reoptimization Loop

**Problem:** A carrier deployed Paragon Pathfinder with auto-bandwidth enabled. After a major link failure, Paragon rerouted 40 SR policies away from the failed segment. This created congestion on the backup paths. Paragon detected the congestion via telemetry and immediately rerouted again to less-loaded paths. The new paths also congested. Paragon kept rerouting — within 30 seconds, 40 policies were cycling through paths every 5 seconds. Traffic dropped significantly.

**Root cause:** The telemetry sampling interval (10 seconds) was faster than the ECMP convergence time. By the time Paragon measured congestion and rerouted, the previous reroute hadn't yet been reflected in measurements. Classic control loop oscillation.

**Fix:**
1. Increased telemetry sampling from 10s → 60s
2. Added reoptimization dampening: minimum 5 minutes between consecutive reroutes per policy
3. Implemented a "reoptimization hold-down" — no reroute within 10 minutes of a topology change
4. Added BW headroom: only trigger reroute when utilization exceeds 85%, not 70%

**Lesson:** Telemetry-driven controllers need carefully tuned feedback loops. Fast telemetry + fast reoptimization = oscillation. Treat it like a control systems problem (stability margins, not just correctness).
