# 11.2 — Streaming Telemetry

## Overview

> Traditional SNMP polling asks devices "how are you?" every 5 minutes and hopes nothing happened in between. Streaming telemetry flips the model: devices push data continuously at configurable intervals (typically 10–30 seconds, sub-second possible), enabling near-real-time visibility into interface counters, BGP state, optics, and resource utilization. For SP networks carrying millions of flows across hundreds of nodes, this shift from pull to push is the difference between reacting to outages and preventing them. This section covers the architecture, protocols, collection pipelines, and operational patterns for production telemetry at scale.


## Key Concepts
> **After this section you will understand:**
> - how model-driven interfaces standardize configuration and state
> - how streaming telemetry pipelines improve observability and alerting
> - integration patterns between controllers and SR/TE policy workflows
> - safe automation practices for validation, rollback, and change control

## Theory

### The Polling Problem

SNMP polling has served network monitoring for three decades, but it hits hard limits at SP scale:

| Limitation | Impact at Scale |
|-----------|----------------|
| Fixed poll interval (typically 5 min) | 300-second blind spots — microbursts, flaps, and transient loops invisible |
| CPU cost per poll cycle | 1,000 OIDs × 500 devices × 12/hr = 6M SNMP walks/hr on collector |
| MIB limitations | No YANG-modeled state (SR policies, EVPN routes, FlexAlgo) |
| UDP transport | Connectionless; manager retries help but no session-level reliability |
| Scalar responses | One OID = one value; table walks are sequential |
| Device CPU impact | Each SNMP walk triggers CLI-equivalent internal queries |

Streaming telemetry addresses every one of these:

| Capability | Streaming Telemetry |
|-----------|-------------------|
| Update frequency | Configurable per-path: 1s, 5s, 30s, on-change |
| Transport | gRPC (HTTP/2, TLS, multiplexed streams) |
| Data model | YANG paths — same models used for config |
| Encoding | Protocol Buffers (compact, schema-aware) or JSON |
| Push model | Device pushes; collector just listens |
| Efficiency | Only changed values sent (on-change mode) |

### Dial-In vs. Dial-Out

Two architectural models exist for establishing the telemetry session. Both run over gRPC transport, but the TCP connection direction differs — and critically, **Cisco's MDT dial-out is NOT gNMI**: it uses Cisco's own `telemetry.proto` schema, not `gnmi.proto`. gNMI Subscribe is always dial-in (collector initiates). This distinction matters for exam preparation and tool compatibility:

```
Dial-Out (Device-Initiated):
┌──────────┐                    ┌──────────────┐
│  Router   │───gRPC connect──→│  Collector    │
│           │═══data stream═══→│  (listening)  │
└──────────┘                    └──────────────┘
  Subscription configured         Accepts streams
  on-box (CLI/YANG)               on a listening port

Dial-In (Collector-Initiated):
┌──────────┐                    ┌──────────────┐
│  Router   │←──gRPC connect───│  Collector    │
│ (gRPC    │═══data stream═══→│  (initiating) │
│  server)  │                   └──────────────┘
└──────────┘
  gRPC server enabled            Subscription sent
  on device                      via gNMI Subscribe
```

#### When to Use Each

| Factor | Dial-Out | Dial-In |
|--------|----------|---------|
| Firewall friendliness | Better — device initiates outbound | Requires inbound access to device |
| Subscription management | On-box config (CLI/model) | Centralized in collector |
| Dynamic subscriptions | No — requires config change | Yes — collector can add/remove paths |
| Scale pattern | Device decides what to send | Collector decides what to request |
| Failure recovery | Device reconnects automatically | Collector must reconnect |
| Protocol | Vendor-specific (Cisco MDT, JTI) over gRPC | gNMI Subscribe (OpenConfig standard) |
| **Common SP pattern** | **Production bulk telemetry** | **Centrally-managed production or ad-hoc** |

> **Practical reality:** Many traditional SPs use **dial-out for production telemetry** (bulk counters, optics, BGP state) and **dial-in for ad-hoc queries**. However, hyperscalers and modern SPs (Google pioneered this with gNMI) often use **dial-in at scale** with centralized subscription management. The two models complement each other — choose based on your operational model and firewall topology.

### gNMI Subscribe — Deep Dive

gNMI Subscribe (defined in the gNMI specification, OpenConfig) is the primary dial-in telemetry protocol. Section 11.1 introduced the basics; here we go deep.

#### Subscription Modes

```protobuf
message SubscribeRequest {
  oneof request {
    SubscriptionList subscribe = 1;  // Initial subscription
    Poll poll = 3;                   // On-demand poll trigger
  }
}

message SubscriptionList {
  Path prefix = 1;                   // Common path prefix
  repeated Subscription subscription = 2;
  // ...
  enum Mode {
    STREAM = 0;    // Continuous updates
    ONCE = 1;      // One-shot dump, then close
    POLL = 2;      // Updates only when Poll message sent
  }
}

// Per-subscription mode (distinct from SubscriptionList.Mode above)
enum SubscriptionMode {
  TARGET_DEFINED = 0;  // Device chooses SAMPLE or ON_CHANGE per leaf
  ON_CHANGE      = 1;  // Push only when value changes
  SAMPLE         = 2;  // Periodic push at fixed interval
}

message Subscription {
  Path path = 1;
  SubscriptionMode mode = 2;        // See enum above
  uint64 sample_interval = 3;       // Nanoseconds between samples
  bool suppress_redundant = 4;      // Don't send if value unchanged
  uint64 heartbeat_interval = 5;    // Max time between updates even if unchanged
}
```

#### Three Stream Modes

**SAMPLE** — Periodic push at fixed intervals:
```
Time:    |--5s--|--5s--|--5s--|--5s--|
Updates: [val]  [val]  [val]  [val]
```
- Best for: counters (interface bytes, CPU, memory, optics power)
- Interval range: 1s–3600s typical; sub-second possible but CPU-expensive
- `suppress_redundant`: if `true`, skips update when value hasn't changed
- `heartbeat_interval`: guarantees at least one update per interval even with suppression

**ON_CHANGE** — Push only when value changes:
```
Time:    |----------|--|----|----------------|
Updates: [val]      [val][val]               [heartbeat]
```
- Best for: state (BGP neighbor status, interface oper-status, IS-IS adjacency)
- Initial sync: sends full snapshot, then only deltas
- `heartbeat_interval`: critical — without it, no way to distinguish "unchanged" from "dead session"
- **Not suitable for counters** — counters always change, defeating the purpose

**TARGET_DEFINED** — Device chooses SAMPLE or ON_CHANGE per leaf:
```
Device decides:
  /interfaces/interface/state/counters → SAMPLE (counters always change)
  /interfaces/interface/state/oper-status → ON_CHANGE (state is discrete)
```
- Simplest for operator — subscribe to a subtree, let the device optimize
- Vendor-dependent behavior — test to verify what you actually get
- Most OpenConfig implementations default to: counters=SAMPLE, state=ON_CHANGE

#### The Sync Response

After establishing a subscription, the device sends an initial dump of all matching data, followed by a `sync_response = true` marker:

```
Client sends: SubscribeRequest { subscribe { path: /interfaces mode: STREAM } }

Server sends:
  Update { /interfaces/interface[name=GigE0/0/0/0]/state/counters/in-octets = 1234567 }
  Update { /interfaces/interface[name=GigE0/0/0/0]/state/oper-status = UP }
  Update { /interfaces/interface[name=GigE0/0/0/1]/state/counters/in-octets = 7654321 }
  Update { /interfaces/interface[name=GigE0/0/0/1]/state/oper-status = DOWN }
  ...
  SubscribeResponse { sync_response = true }   ← "Initial sync complete"

  // Now only periodic SAMPLE or ON_CHANGE updates follow
```

> **Implementation note:** The sync phase can be expensive on large routers. Subscribing to `/` (root) on a device with 10,000 interfaces will dump millions of leaves before sync completes. Always scope subscriptions to specific subtrees.

```

# Memory by process (not in OC)
/junos/system/linecard/cpu/memory/usage

# FPC/PFE utilization
/junos/system/linecard/packet/usage
```

> **Best practice:** Use OpenConfig paths wherever possible — they work across IOS-XR, Junos, and EOS. Fall back to vendor-native only for platform-specific internals (NPU counters, FPC utilization, process memory). Document every native path with a comment explaining *why* the OC path was insufficient.

## Platform Configuration

### IOS-XR: Dial-Out Telemetry

IOS-XR has mature dial-out support via its Model-Driven Telemetry (MDT) framework:

```
!! IOS-XR 7.x+ Dial-Out Configuration
!! Sensor group → Subscription → Destination group

! Step 1: Define what to collect
telemetry model-driven
 sensor-group INTERFACE-COUNTERS
  sensor-path openconfig-interfaces:interfaces/interface/state/counters
 !
 sensor-group BGP-NEIGHBORS
  sensor-path openconfig-bgp:bgp/neighbors/neighbor/state
 !
 sensor-group OPTICS
  sensor-path openconfig-platform:components/component/transceiver/physical-channels/channel/state
 !

 ! Step 2: Define where to send it
 destination-group COLLECTOR-PRIMARY
  address-family ipv4 10.0.100.10 port 57400
   encoding self-describing-gpb    !! Options: gpb, self-describing-gpb, json
   protocol grpc no-tls            !! Use 'protocol grpc tls' in production
  !
 !

 ! Step 3: Bind sensors to destinations with intervals
 subscription INTF-30S
  sensor-group-id INTERFACE-COUNTERS sample-interval 30000    !! 30s in ms
  destination-id COLLECTOR-PRIMARY
 !
 subscription BGP-ONCHANGE
  sensor-group-id BGP-NEIGHBORS sample-interval 0             !! 0 = on-change
  destination-id COLLECTOR-PRIMARY
 !
 subscription OPTICS-60S
  sensor-group-id OPTICS sample-interval 60000                !! 60s
  destination-id COLLECTOR-PRIMARY
 !
```

**Key IOS-XR details:**
- `sample-interval 0` = on-change (not all paths support it — device will fall back to minimum sample)
- `self-describing-gpb` includes field names in each message (larger but debuggable); `gpb` uses compiled proto (compact but requires schema)
- TLS: use `protocol grpc tls` with `tls-hostname` and certificate profile for production
- Maximum sensor paths per subscription: platform-dependent (typically 50–100)
- CPU impact: monitor `show telemetry model-driven subscription` for `total-not-sent` (queue overflow)

### IOS-XR: Dial-In (gNMI)

```
!! Enable gRPC server for dial-in gNMI
grpc
 port 57400
 no-tls                              !! Use TLS in production
 address-family dual                  !! Listen on IPv4 and IPv6
 max-request-per-user 32
 max-request-total 128
 max-streams 128                      !! Max concurrent subscriptions
 max-streams-per-user 32
!
```

Then use `gnmic` from the collector side:

```bash
# Subscribe to interface counters, 10s sample
gnmic -a 10.0.0.1:57400 --insecure \
  subscribe \
  --path "/openconfig-interfaces:interfaces/interface/state/counters" \
  --stream-mode sample \
  --sample-interval 10s \
  --encoding proto

# Subscribe to BGP state, on-change
gnmic -a 10.0.0.1:57400 --insecure \
  subscribe \
  --path "/openconfig-bgp:bgp/neighbors/neighbor/state/session-state" \
  --stream-mode on-change \
  --heartbeat-interval 60s

# Multiple paths in one subscription
gnmic -a 10.0.0.1:57400 --insecure \
  subscribe \
  --path "/interfaces/interface/state/counters" \
  --path "/components/component/transceiver/physical-channels/channel/state" \
  --stream-mode sample \
  --sample-interval 30s
```

### Junos: Dial-Out Telemetry (JTI)

Junos Telemetry Interface (JTI) supports native Junos resource paths for dial-out export. This is the legacy approach — still functional on MX/PTX but being superseded by gNMI dial-in for OpenConfig paths:

```
!! Junos Dial-Out via JTI (legacy — uses native resource paths only)
set services analytics streaming-server COLLECTOR-1 remote-address 10.0.100.10
set services analytics streaming-server COLLECTOR-1 remote-port 50051

!! Export profile — defines encoding and transport
set services analytics export-profile GRPC-EXPORT local-address 10.0.0.1
set services analytics export-profile GRPC-EXPORT reporting-rate 30
set services analytics export-profile GRPC-EXPORT transport grpc

!! Sensor definition — what to export
!! NOTE: services analytics uses NATIVE Junos resource paths only.
!! For OpenConfig paths, use gNMI dial-in (gnmic subscribe) instead.
set services analytics sensor INTF-COUNTERS server-name COLLECTOR-1
set services analytics sensor INTF-COUNTERS export-name GRPC-EXPORT
set services analytics sensor INTF-COUNTERS resource /junos/system/linecard/interface/

!! Native Junos routing state (not OpenConfig — OC paths require gNMI dial-in)
set services analytics sensor ROUTING-STATE server-name COLLECTOR-1
set services analytics sensor ROUTING-STATE export-name GRPC-EXPORT
set services analytics sensor ROUTING-STATE resource /junos/system/linecard/firewall/
```

> **Junos caveat:** The `reporting-rate` is in **seconds** (not milliseconds like IOS-XR). Minimum rate varies by platform — MX Series supports 1s for most counters, PTX may have higher minimums for some native paths. The `services analytics sensor` framework only supports **native Junos resource paths** (`/junos/...`), not OpenConfig YANG paths. For OpenConfig telemetry on Junos, use gNMI dial-in (below). Always verify with `show agent sensors` after committing.

#### Junos: Dial-In (gNMI via OpenConfig)

```
!! Enable gNMI server
!! Prerequisite: Junos OpenConfig package must be installed (separate .tgz)
!! Verify with: show version | match openconfig
set system services extension-service request-response grpc clear-text port 32767
!! WARNING: skip-authentication disables all auth — lab/dev only, NEVER production
!! set system services extension-service request-response grpc skip-authentication
!! Production: TLS + certificate auth is mandatory
set system services extension-service request-response grpc ssl port 32767
set system services extension-service request-response grpc ssl local-certificate <cert-name>
```

```bash
# gnmic dial-in to Junos
gnmic -a 10.0.0.2:32767 --insecure \
  subscribe \
  --path "/interfaces/interface/state/counters" \
  --stream-mode sample \
  --sample-interval 15s
```

> **Junos gNMI port note:** The gNMI service on Junos shares the gRPC extension-service framework. Unlike IOS-XR (which uses `grpc port 57400` for both gNMI and MDT), Junos uses the extension-service port (default 32767 for clear-text, separate port for TLS). Ensure you're connecting to the correct port — they are NOT interchangeable.

## Collection Pipeline Architecture

### Reference Architecture

A production telemetry pipeline has four stages:

```
┌─────────────────────────────────────────────────────────────────┐
│                      TELEMETRY PIPELINE                         │
├──────────┬──────────────┬───────────────┬──────────────────────┤
│ COLLECT  │  TRANSPORT   │   STORE       │  VISUALIZE/ALERT     │
│          │              │               │                      │
│ gnmic    │  Kafka       │  Prometheus   │  Grafana             │
│ telegraf │  (buffering, │  InfluxDB     │  (dashboards)        │
│ pipeline │   fan-out)   │  ClickHouse   │                      │
│          │              │  TimescaleDB  │  Alertmanager        │
│ gRPC     │  NATS        │               │  PagerDuty           │
│ receiver │              │  Parquet/S3   │  (alerting)          │
│          │              │  (archive)    │                      │
└──────────┴──────────────┴───────────────┴──────────────────────┘
  Devices push              Broker decouples    TSDB for real-time
  or collectors             producers from      Object store for
  dial-in                   consumers           long-term
```

### Component Selection Guide

| Component | Good For | Not Great For | SP Scale Notes |
|-----------|----------|---------------|----------------|
| **gnmic** | gNMI collection, flexible output (Kafka, Prometheus, file) | Non-gNMI sources | Can cluster for HA; single binary, Go-based |
| **Telegraf** | Multi-protocol input (SNMP, gNMI, syslog, Exec) | Complex transformations | Heavy plugin ecosystem; Cisco MDT input plugin |
| **Kafka** | High-throughput buffering, fan-out to multiple consumers | Simple deployments (<10 devices) | 3-node minimum for production; topic-per-sensor-group |
| **Prometheus** | Metrics + alerting, pull-based scrape model | Raw telemetry ingestion (needs adapter) | Use `remote_write` from gnmic/telegraf; 15s scrape default |
| **InfluxDB** | Time-series native, Flux query language | Very high cardinality (>1M series) | OSS v2 or Cloud; retention policies per measurement |
| **ClickHouse** | Massive-scale analytics, columnar compression | Simple dashboards (overkill) | Best for >100 devices, historical analysis |
| **Grafana** | Dashboards, multi-datasource, alerting | Raw data processing | Pairs with any TSDB; SP-specific community dashboards available |

### gnmic — The Swiss Army Knife

`gnmic` (OpenConfig project) is the recommended gNMI collector for SP telemetry:

```yaml
# gnmic.yml — Production collector configuration
# Targets: all PE routers
targets:
  pe01-sjc.example.com:57400:
    username: telemetry
    password: "${GNMIC_PASSWORD}"       # Environment variable
    tls-ca: /etc/gnmic/ca.pem
    tls-cert: /etc/gnmic/client.pem
    tls-key: /etc/gnmic/client-key.pem
    subscriptions:
      - interface-counters
      - bgp-state
      - optics

  pe02-sjc.example.com:57400:
    username: telemetry
    password: "${GNMIC_PASSWORD}"
    tls-ca: /etc/gnmic/ca.pem
    subscriptions:
      - interface-counters
      - bgp-state

# Subscription definitions
subscriptions:
  interface-counters:
    paths:
      - /interfaces/interface/state/counters
      - /interfaces/interface/state/oper-status
    stream-mode: sample
    sample-interval: 30s
    encoding: proto

  bgp-state:
    paths:
      - /network-instances/network-instance/protocols/protocol/bgp/neighbors/neighbor/state
    stream-mode: on-change
    heartbeat-interval: 120s
    encoding: proto

  optics:
    paths:
      - /components/component/transceiver/physical-channels/channel/state
    stream-mode: sample
    sample-interval: 60s
    encoding: proto

# Output to Kafka
outputs:
  kafka-primary:
    type: kafka
    address: kafka-1.example.com:9092,kafka-2.example.com:9092,kafka-3.example.com:9092
    topic: "telemetry.${subscription-name}"    # Topic per subscription
    encoding: proto                             # or json for debugging
    max-retry: 3
    timeout: 5s
    # TLS for Kafka
    tls-ca: /etc/gnmic/kafka-ca.pem

  # Direct Prometheus export for quick dashboarding
  prometheus-output:
    type: prometheus
    listen: ":9804"
    path: /metrics
    expiration: 120s           # Drop metrics not updated in 120s
    # Note: gnmic automatically adds 'source' tag with the target address.
    # Use gnmic's event-processors for custom label manipulation if needed.
    # See gnmic docs for current processor schema — it evolves between versions.
```

```bash
# Run gnmic with config
gnmic --config gnmic.yml subscribe

# Verify subscriptions
gnmic --config gnmic.yml subscribe --dry-run

# Clustered mode (HA — multiple gnmic instances share targets)
gnmic --config gnmic.yml subscribe \
  --cluster-name telemetry-collectors \
  --instance-name collector-01 \
  --targets-watch-timer 30s
```

## Scale Considerations

### How Much Data?

Estimating telemetry volume for capacity planning:

```
Per interface, 30s sample:
  ~20 counters × 8 bytes (raw) = 160 bytes raw values
  + proto field tags + path encoding + timestamp + gRPC framing ≈ 600–1000 bytes/message
  2 updates/min × 60 min = 120 updates/hr per interface

Per PE router (200 interfaces):
  120 × 200 = 24,000 updates/hr
  @ ~800 bytes/update = ~19 MB/hr (proto encoding with overhead)

Per network (500 PEs):
  24,000 × 500 = 12M updates/hr ≈ 9.3 GB/hr for interface counters alone

Add BGP (on-change, ~1,000 neighbors):
  ~100 state changes/hr × 500 bytes = 50 KB/hr (bursty during events)

Add optics (60s, 2,000 transceivers):
  2,000 × 60 updates/hr × 600 bytes = ~72 MB/hr

Total steady-state for 500-PE network:
  ~10 GB/hr ≈ 240 GB/day ≈ 7 TB/month (compressed: ~1 TB/month)
```

> **Reality check:** These numbers are manageable for any modern pipeline. The challenge isn't volume — it's **cardinality** (unique time series). 500 routers × 200 interfaces × 20 metrics = 2M unique series. Prometheus handles this; InfluxDB handles this. Problems start at 10M+ series or sub-second intervals on thousands of interfaces.

### High Availability Patterns

```
Pattern 1: Active/Active Collectors (Recommended)
┌──────────┐     ┌───────────────┐     ┌─────────┐
│  Router   │────→│ gnmic cluster │────→│  Kafka   │
│  (dial-   │────→│ (3 instances, │     │ (3-node) │
│   out to  │     │  target       │     └─────────┘
│   both)   │     │  sharding)    │         │
└──────────┘     └───────────────┘     ┌───┴───┐
                                       │Consumer│
                                       │(idempotent)│
                                       └───────┘

- gnmic cluster mode: targets distributed across instances
- If one collector dies, others absorb its targets
- Kafka provides durability — no data loss during failover
- Consumer must handle duplicates (at-least-once delivery)

Pattern 2: Dual Dial-Out (Simple HA)
┌──────────┐     ┌──────────────┐     ┌─────────┐
│  Router   │────→│ Collector A  │────→│  TSDB   │
│  (sends   │     └──────────────┘     │ (dedup) │
│   to both │     ┌──────────────┐     │         │
│   dest.)  │────→│ Collector B  │────→│         │
└──────────┘     └──────────────┘     └─────────┘

- Configure two destination-groups on IOS-XR
- Both collectors receive identical streams
- TSDB or consumer handles deduplication
- Simple but doubles bandwidth and storage
```

### Cardinality Management

High cardinality kills time-series databases. Strategies:

1. **Filter at collection:** Only subscribe to interfaces you care about (use gNMI path wildcards with key filters)
2. **Aggregate at the broker:** Kafka Streams or Flink to pre-aggregate per-device totals
3. **Drop labels:** Remove high-cardinality labels (individual queue names, per-flow counters) before TSDB
4. **Tiered retention:** 10s resolution for 7 days, 5m aggregates for 90 days, 1hr aggregates for 1 year
5. **Partition by sensor type:** Interface counters in one TSDB, optics in another — different cardinality profiles

## Alerting on Telemetry

### From Polling Alerts to Streaming Alerts

The shift from polling to streaming changes alerting fundamentally:

| Polling Alert | Streaming Alert |
|--------------|----------------|
| "Interface down" detected at next poll (5 min) | "Interface down" detected in <1s via on-change |
| "High utilization" based on 5-min average | "Microburst" detected from 10s samples |
| "BGP neighbor down" — SNMP trap (unreliable UDP) | "BGP neighbor down" — on-change push (reliable gRPC) |
| Alert fires, then next check is 5 min later | Continuous stream — alert resolves immediately when state recovers |

### Prometheus Alerting Rules (Example)

> **Note:** Metric names below are illustrative. Actual names depend on your collector's export format — gnmic uses path-derived names (e.g., `openconfig_interfaces_interface_state_counters_in_octets`), Telegraf uses measurement + field names. Adapt to match your pipeline's actual metric naming.

```yaml
# prometheus-alerts.yml
groups:
  - name: network-telemetry
    rules:
      # Interface down for more than 30 seconds
      # NOTE: OC oper-status is a string enum (UP/DOWN/TESTING), not IANA integer.
      # gnmic exports it as a label value, so match as label filter, not numeric.
      # Actual metric name and label scheme depends on your gnmic/Telegraf config.
      - alert: InterfaceDown
        expr: |
          openconfig_interfaces_interface_state_oper_status{oper_status="DOWN"} == 1
        for: 30s
        labels:
          severity: critical
        annotations:
          summary: "Interface {{ $labels.name }} down on {{ $labels.source }}"
          description: "Interface has been operationally down for >30s"

      # Interface utilization >85% sustained 5 minutes
      # NOTE: interface_speed_bps must be explicitly collected or statically labeled.
      # OpenConfig path: /interfaces/interface/ethernet/state/port-speed (string, e.g. "SPEED_100GB")
      # Alternatively, use a static recording rule to define known link speeds by interface.
      - alert: HighUtilization
        expr: |
          rate(openconfig_interfaces_interface_state_counters_in_octets[5m]) * 8
          /
          on(name, source) group_left() interface_speed_bps_static
          > 0.85
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "{{ $labels.name }} on {{ $labels.source }} at {{ $value | humanizePercentage }} utilization"

      # Optics power degradation (input power below threshold)
      # NOTE: OpenConfig stores optical power in units of 0.01 dBm (integer).
      # -25 dBm = -2500 in raw OC value. Threshold depends on your exporter's unit handling.
      # gnmic may pass through raw integers; verify your pipeline's scaling before deploying.
      - alert: OpticsRxLow
        expr: transceiver_input_power_dbm < -2500    # -25.00 dBm in 0.01 dBm units
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "Low Rx power ({{ $value | humanize }}×0.01dBm) on {{ $labels.component }} / {{ $labels.source }}"

      # BGP session flapping (>3 transitions in 10 minutes)
      # for: 1m avoids false-fires on single evaluation spikes
      - alert: BGPFlapping
        expr: increase(bgp_established_transitions[10m]) > 3
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "BGP neighbor {{ $labels.neighbor_address }} flapping on {{ $labels.source }}"
```

## Troubleshooting Telemetry

### Common Issues and Fixes

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Subscription state = `NOT_ACTIVE` | Sensor path invalid or unsupported | Verify with `show telemetry model-driven sensor-group <name> internal` (XR); check YANG model support |
| Data stops flowing | gRPC session timeout or device CPU overload | Check `total-not-sent` counter; reduce subscription count or increase interval |
| `sync_response` never arrives | Too many leaves in subscription scope | Narrow the path — subscribe to specific interfaces instead of `/interfaces` |
| Duplicate data in TSDB | Dual dial-out without dedup | Add dedup at consumer or use gnmic cluster mode |
| High collector CPU | Proto decoding overhead with many streams | Use compiled proto (not self-describing); batch processing |
| Missing on-change updates | Path doesn't support on-change mode | Device silently falls back to minimum sample interval; verify actual behavior |
| gRPC connection refused | Wrong port, TLS mismatch, or gRPC not enabled | Verify `show grpc status` (XR); check port and TLS config match |
| Stale metrics in Prometheus | Heartbeat interval too long or not set | Set `heartbeat_interval` on on-change subscriptions; set `expiration` in gnmic Prometheus output |

### Telemetry Health Dashboard

Essential Grafana panels for monitoring the telemetry system itself:

1. **Updates/second per device** — detects dead subscriptions
2. **Kafka consumer lag** — pipeline bottleneck indicator
3. **TSDB ingestion rate** — capacity planning
4. **gRPC error rate** — connection health
5. **Collector CPU/memory** — scaling trigger
6. **Time since last update per subscription** — staleness detector

## War Story: The Phantom Microburst

A Tier 1 SP migrated from 5-minute SNMP polling to 10-second streaming telemetry on their peering edge. Within the first week, they discovered that 15% of their peering links were experiencing repeated 2–3 second microbursts hitting 95%+ utilization — events completely invisible to 5-minute averages that showed comfortable 40% utilization.

The fix wasn't adding capacity. Analysis of the burst patterns revealed they correlated exactly with a major CDN's cache-fill schedule — every 8 minutes, a coordinated burst of cache refreshes from the CDN's origin servers. The SP worked with the CDN to stagger cache fills across a 60-second window instead of a 3-second burst. Peak utilization dropped from 95% to 65%.

**Lesson:** The telemetry didn't reveal a capacity problem — it revealed an *interaction* problem. Five-minute averages aren't just imprecise; they hide an entirely different class of issues. You can't fix what you can't see, and you can't see what you don't measure at the right granularity.

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| Polling vs. streaming | Streaming eliminates blind spots; higher granularity with more efficient per-data-point delivery |
| Dial-out vs. dial-in | Dial-out for production bulk; dial-in for ad-hoc; use both |
| gNMI Subscribe modes | SAMPLE for counters, ON_CHANGE for state, TARGET_DEFINED for convenience |
| Sensor paths | OpenConfig first, vendor-native for platform internals only |
| Pipeline architecture | Collect (gnmic/telegraf) → Buffer (Kafka) → Store (TSDB) → Alert (Grafana/Alertmanager) |
| Scale | 500-PE network ≈ 10 GB/hr (with proto overhead); cardinality is the real limit |
| HA | gnmic clustering + Kafka for zero-loss failover |
| Alerting | Streaming enables sub-second detection; microbursts, flaps, optics degradation |
| Troubleshooting | Monitor the telemetry system itself — dead subscriptions are silent failures |
| Protocol clarification | IOS-XR MDT dial-out uses Cisco `telemetry.proto`, NOT gNMI; gNMI Subscribe is always dial-in |

## IETF YANG-Push vs. gNMI

A note for IE-SP candidates: the IETF standardized its own subscription mechanism in parallel with OpenConfig's gNMI:

| Aspect | gNMI Subscribe | IETF YANG-Push (RFC 8641) |
|--------|----------------|--------------------------|
| Origin | OpenConfig (Google-led) | IETF (NETCONF Working Group) |
| Transport | gRPC / HTTP2 | NETCONF (SSH) or RESTCONF (HTTP) |
| Encoding | Protocol Buffers or JSON | XML or JSON |
| Deployment | Widely deployed (IOS-XR, Junos, EOS, NX-OS) | Limited — mainly academic/early adoption |
| Modes | SAMPLE, ON_CHANGE, TARGET_DEFINED | Periodic, On-Change |
| Auth/TLS | gRPC-native | SSH (NETCONF) / TLS (RESTCONF) |

In practice: **gNMI dominates production SP deployments.** YANG-Push is referenced in the RFCs and may appear on IETF-heavy exams as the standards track mechanism. Know that both exist, gNMI is what you'll configure, YANG-Push is what the IETF considers "correct."
