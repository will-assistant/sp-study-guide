# 11.1 — Model-Driven Networking

## Overview

> Model-driven networking replaces CLI scraping with structured, programmatic device management. YANG defines the data schema, NETCONF/RESTCONF/gRPC provide the transport, and the combination gives operators machine-parseable configuration and state across multi-vendor SP networks. This section covers the protocols, data models, and operational patterns that underpin every modern automation workflow — from simple config pushes to closed-loop telemetry-driven SR-TE.


## Key Concepts
> **After this section you will understand:**
> - how model-driven interfaces standardize configuration and state
> - how streaming telemetry pipelines improve observability and alerting
> - integration patterns between controllers and SR/TE policy workflows
> - safe automation practices for validation, rollback, and change control

## Theory

### The Problem with CLI

Service provider networks grew up on CLI — human-readable, vendor-specific, and fundamentally unstructured. The cost of CLI-driven operations scales poorly:

| Approach | Parse Complexity | Multi-Vendor | Transactional | Rollback |
|----------|-----------------|--------------|---------------|----------|
| CLI + screen scraping | High (regex, expect) | Per-vendor parsers | No (partial apply) | Manual |
| SNMP SET | Low (MIB OIDs) | MIB support varies | No | No |
| NETCONF/YANG | None (structured XML) | Vendor + standard models | Yes (candidate commit) | Native |
| gNMI/YANG | None (structured protobuf) | Vendor + standard models | Yes | Native |

The shift to model-driven management isn't optional — it's the prerequisite for every automation pattern in this module.

### YANG — The Data Modeling Language

YANG (RFC 7950) defines the **schema** for network device configuration and operational state. It is to NETCONF what MIBs were to SNMP — but dramatically more expressive.

#### Core Constructs

```
module ietf-interfaces {
  // NOTE: Simplified fragment — real module has import statements for
  // ietf-yang-types, iana-if-type, etc. Use pyang to validate full modules.
  namespace "urn:ietf:params:xml:ns:yang:ietf-interfaces";
  prefix if;

  container interfaces {           // Single instance container
    list interface {               // Ordered/unordered list of entries
      key "name";                  // Unique key field
      leaf name { type string; }   // Scalar value
      leaf type { type identityref { base iana-if-type:interface-type; } }
      leaf enabled { type boolean; default true; }
      
      container statistics {       // Operational state (config false)
        config false;
        leaf in-octets { type yang:counter64; }
        leaf out-octets { type yang:counter64; }
      }
    }
  }
}
```

**Key YANG concepts for SP automation:**

| Construct | Purpose | Example |
|-----------|---------|---------|
| `container` | Grouping node (no data of its own) | `interfaces`, `routing` |
| `list` | Keyed collection of entries | `interface`, `neighbor`, `route` |
| `leaf` | Single scalar value | `name`, `mtu`, `enabled` |
| `leaf-list` | Array of scalars | `name-server` (multiple DNS servers) |
| `choice/case` | Mutually exclusive branches | `static-route` vs `connected` |
| `augment` | Extend another module's tree | Vendor adding features to IETF model |
| `deviation` | Declare non-compliance | "We don't support leaf X" |
| `grouping/uses` | Reusable template | Common address-family parameters |
| `identity/identityref` | Extensible enum | Interface types, AFI/SAFI |
| `typedef` | Custom type with constraints | `ipv4-address`, `as-number` |

#### Model Ecosystem

Three model families coexist in every SP network:

1. **IETF standard models** — `ietf-interfaces` (RFC 8343), `ietf-routing` (RFC 8349), `ietf-bgp` (RFC 9657). Lowest common denominator, multi-vendor.
2. **OpenConfig models** — `openconfig-network-instance`, `openconfig-bgp`, `openconfig-interfaces`. Operator-driven (Google, AT&T, Microsoft). More opinionated than IETF. Widely adopted.
3. **Vendor-native models** — `Cisco-IOS-XR-*`, `junos-conf-*`, `nokia-conf-*`. Full feature coverage but vendor-locked.

**Practical guidance:** Start with OpenConfig for multi-vendor use cases (BGP, interfaces, LLDP). Fall back to native models for vendor-specific features (Cisco FlexAlgo, Junos CoS). Never mix model families in the same RPC for the same feature.

#### YANG Model Discovery

Before automating, discover what models the device supports:

```xml
<!-- Sent automatically during <hello> handshake -->

```

```bash
# Junos: YANG module list
```


### NETCONF — The Configuration Protocol

NETCONF (RFC 6241, updated by RFC 6242 for chunked framing) provides transactional, connection-oriented configuration management over SSH.

#### Protocol Stack

```
┌──────────────────────┐
│  Content (YANG data)  │   XML-encoded config/state
├──────────────────────┤
│  Operations           │   <get>, <get-config>, <edit-config>, <copy-config>,
│                       │   <lock>, <unlock>, <commit>, <discard-changes>
├──────────────────────┤
│  Messages (RPC)       │   <rpc>, <rpc-reply>, <notification>
├──────────────────────┤
│  Transport (SSH)      │   RFC 6242, subsystem=netconf, port 830
└──────────────────────┘
```

#### Datastores

NETCONF operates on datastores — not files, not running memory, but abstract data stores:

| Datastore | Purpose | Supported By |
|-----------|---------|--------------|
| `running` | Currently active configuration | All |
| `candidate` | Staging area for uncommitted changes | IOS-XR, Junos, SR-OS |
| `startup` | Configuration loaded at boot | IOS-XE, some NOS |
| `operational` | Read-only state + applied config (RFC 8342) | IOS-XR 7.x+, Junos 21+, SR-OS |

The **candidate datastore** is critical for SP operations — it enables:
- **Confirmed commit** — Auto-rollback if not confirmed within timeout (prevents lockout)
- **Validate before commit** — Syntax + semantic checks without applying
- **Atomic apply** — All-or-nothing change application

#### Core Operations

**Edit-Config with candidate commit (the SP workflow):**

```xml
<!-- Step 1: Lock candidate to prevent concurrent edits -->
<rpc message-id="1" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <lock><target><candidate/></target></lock>
</rpc>

<!-- Step 2: Edit candidate datastore -->
<rpc message-id="2" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <edit-config>
    <target><candidate/></target>
    <default-operation>merge</default-operation>
    <config>
      <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces">
        <interface>
          <name>GigabitEthernet0/0/0/1</name>
          <description>PE1-to-PE2 backbone link</description>
          <enabled>true</enabled>
        </interface>
      </interfaces>
    </config>
  </edit-config>
</rpc>

<!-- Step 3: Validate before applying -->
<rpc message-id="3" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <validate><source><candidate/></source></validate>
</rpc>

<!-- Step 4: Confirmed commit (auto-rollback in 120s if not confirmed) -->
<rpc message-id="4" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <commit>
    <confirmed/>
    <confirm-timeout>120</confirm-timeout>
  </commit>
</rpc>

<!-- Step 5: Confirm the commit (after verifying connectivity) -->
<rpc message-id="5" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <commit/>
</rpc>

<!-- Step 6: Unlock -->
<rpc message-id="6" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <unlock><target><candidate/></target></unlock>
</rpc>
```

**Operation semantics:**

| Operation | `merge` (default) | `replace` | `create` | `delete` | `remove` |
|-----------|-------------------|-----------|----------|----------|----------|
| Exists? → | Update fields | Overwrite entirely | Error | Delete (error if missing) | Delete (no error if missing) |
| Missing? → | Create | Create | Create | Error | No-op |

**`replace` vs `merge` — the #1 automation footgun:** `merge` adds/updates but never removes. If you delete a BGP neighbor from your source-of-truth but use `merge`, the neighbor stays on the device. Use `replace` at the container level for declarative config management.

#### Subtree Filtering vs XPath

```xml
<!-- Subtree filter: Get specific interface -->
<rpc message-id="7" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <get>
    <filter type="subtree">
      <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces">
        <interface>
          <name>GigabitEthernet0/0/0/1</name>
        </interface>
      </interfaces>
    </filter>
  </get>
</rpc>

<!-- XPath filter: Get all enabled interfaces -->
<rpc message-id="8" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <get>
    <filter type="xpath"
            select="/if:interfaces/if:interface[if:enabled='true']"
            xmlns:if="urn:ietf:params:xml:ns:yang:ietf-interfaces"/>
  </get>
</rpc>
```

XPath filters are more expressive but not universally supported. Junos supports both; IOS-XR supports subtree and limited XPath. Always test filter support on target platform.

### RESTCONF — HTTP for Network Config

RESTCONF (RFC 8040) maps YANG models to a RESTful HTTP interface. Same data models, different transport.

#### URL Structure

```
https://{host}/restconf/data/{yang-module}:{container}/{list}={key}
```

Examples:
```
GET  /restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet0%2F0%2F0%2F1
PUT  /restconf/data/ietf-interfaces:interfaces/interface=Loopback0
POST /restconf/data/ietf-interfaces:interfaces
DELETE /restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet0%2F0%2F0%2F99
```

#### HTTP Verb Mapping

| HTTP Verb | NETCONF Equivalent | Behavior |
|-----------|-------------------|----------|
| GET | `<get>`, `<get-config>` | Read data |
| POST | `<edit-config>` (create) | Create new resource (error if exists) |
| PUT | `<edit-config>` (replace) | Create or replace entire resource |
| PATCH | `<edit-config>` (merge) | Merge/update existing resource |
| DELETE | `<edit-config>` (delete) | Remove resource |

#### RESTCONF Example

```bash
# Get interface config (JSON)
curl -k -u admin:password \
  -H "Accept: application/yang-data+json" \
  https://router1/restconf/data/ietf-interfaces:interfaces/interface=Loopback0

# Response:
{
  "ietf-interfaces:interface": {
    "name": "Loopback0",
    "type": "iana-if-type:softwareLoopback",
    "enabled": true,
    "ietf-ip:ipv4": {
      "address": [
        {
          "ip": "10.0.0.1",
          "prefix-length": 32
        }
      ]
    }
  }
}

# Update description (PATCH = merge)
curl -k -u admin:password \
  -X PATCH \
  -H "Content-Type: application/yang-data+json" \
  -d '{"ietf-interfaces:interface": {"name": "Loopback0", "description": "Router-ID"}}' \
  https://router1/restconf/data/ietf-interfaces:interfaces/interface=Loopback0
```

#### NETCONF vs RESTCONF

| Dimension | NETCONF | RESTCONF |
|-----------|---------|----------|
| Transport | SSH (port 830) | HTTPS (port 443) |
| Encoding | XML | JSON or XML |
| Connection | Persistent (session) | Stateless (per-request) |
| Candidate datastore | Yes | Implementation-dependent |
| Confirmed commit | Yes | Not in RFC 8040 (vendor extensions exist) |
| Notifications | Yes (RFC 5277) | SSE (RFC 8040 §6) |
| Firewall-friendly | Often blocked (SSH) | Yes (HTTPS) |
| Best for | Bulk config, transactional changes | Lightweight reads, integration APIs |

**SP guidance:** Use NETCONF for configuration management (candidate + confirmed commit is non-negotiable for production changes). Use RESTCONF for operational queries, monitoring dashboards, and integration with web-based tools.

### gRPC and gNMI — The Modern Transport

gRPC (Google Remote Procedure Call) provides high-performance, bidirectional streaming over HTTP/2. gNMI (gRPC Network Management Interface) is the network-specific interface built on gRPC.

#### gNMI Operations

| Operation | Purpose | Direction |
|-----------|---------|-----------|
| `Capabilities` | Discover supported models and encodings | Client → Server |
| `Get` | Read config or state at a path | Client → Server |
| `Set` | Write config (update/replace/delete) | Client → Server |
| `Subscribe` | Stream telemetry updates | Bidirectional |

#### gNMI Paths

gNMI uses a path format derived from YANG tree structure. The paths below use **OpenConfig** models (note the `/config/` and `/state/` containers — this is an OC convention, not present in native/IETF models):

```
/interfaces/interface[name=GigabitEthernet0/0/0/1]/state/counters/in-octets
/network-instances/network-instance[name=default]/protocols/protocol[identifier=BGP][name=bgp]/bgp/neighbors/neighbor[neighbor-address=10.0.0.2]/state
```

#### gNMI Set (Configuration)

```protobuf
// gNMI SetRequest — equivalent to NETCONF edit-config
SetRequest {
  update {
    path: /interfaces/interface[name=Loopback0]/config/description
    val {
      json_val: "\"Router-ID\""
    }
  }
  replace {
    path: /network-instances/network-instance[name=default]/protocols/protocol[identifier=BGP][name=bgp]/bgp/neighbors/neighbor[neighbor-address=10.0.12.2]/config
    val {
      json_ietf_val: '{"peer-as": 65002, "description": "PE2"}'
    }
  }
  delete {
    path: /interfaces/interface[name=GigabitEthernet0/0/0/99]
  }
}
```

#### gNMI Subscribe (Telemetry)

This is where gNMI excels over NETCONF — streaming telemetry at scale:

```
// Subscribe to interface counters every 10 seconds
SubscribeRequest {
  subscribe {
    subscription {
      path: /interfaces/interface/state/counters
      mode: SAMPLE
      sample_interval: 10000000000  // nanoseconds (10s)
    }
    subscription {
      path: /network-instances/network-instance/protocols/protocol/bgp/neighbors/neighbor/state
      mode: ON_CHANGE
    }
    mode: STREAM
    encoding: JSON_IETF
  }
}
```

**Subscription modes:**

| Mode | Behavior | Use Case |
|------|----------|----------|
| `SAMPLE` | Periodic push at fixed interval | Counters, utilization (every 10-30s) |
| `ON_CHANGE` | Push only when value changes | BGP state, interface up/down, alarms |
| `TARGET_DEFINED` | Device chooses sample or on-change per leaf | General monitoring (let device optimize) |
| `ONCE` | One-shot read (like Get but over Subscribe) | Inventory collection, polling |

#### Protocol Comparison Summary

| Feature | NETCONF | RESTCONF | gNMI |
|---------|---------|----------|------|
| Encoding | XML | JSON/XML | Protobuf/JSON |
| Transport | SSH | HTTPS | gRPC (HTTP/2 + TLS) |
| Streaming | Notifications (push) | SSE | Full bidirectional |
| Performance | Good | Good | Best (binary, multiplexed) |
| Tooling | ncclient, Ansible | curl, Python requests | gnmic, gNMIc, pygnmi |
| Candidate datastore | Yes | Varies | Implementation-specific |
| Maturity | Oldest (2006) | Medium (2017) | Newest (2018) |
| SP adoption | Universal | Moderate | Growing fast |


### Junos — Enabling NETCONF and gRPC

```junos
# Enable NETCONF over SSH
set system services netconf ssh port 830
set system services netconf rfc-compliant
set system services netconf yang-compliant

# Enable gRPC (for gNMI) — Junos 21.1+
# Choose ONE: clear-text (lab) or SSL (production) — not both on same port
set system services extension-service request-response grpc ssl port 50051
set system services extension-service request-response grpc ssl local-certificate grpc-cert
set system services extension-service notification allow-clients address 0.0.0.0/0

# Load OpenConfig models (Junos ships both native and OC)
# Syntax varies by version — on some Junos, OC is enabled via packages
set system schema openconfig

# RESTCONF (Junos 21.2+)
set system services rest http port 8080
set system services rest https port 8443
set system services rest https server-certificate rest-cert

# Session limits
set system services netconf ssh connection-limit 16
set system services netconf ssh rate-limit 8
```

### Key Knobs

| Parameter | IOS-XR Default | Junos Default | Recommended | Why |
|-----------|---------------|---------------|-------------|-----|
| NETCONF port | 830 | 830 | 830 | Standard, firewall teams expect it |
| gRPC port | 57400 | 50051 (varies) | 57400 | IOS-XR convention, becoming industry standard |
| Max NETCONF sessions | 50 | 16 | 16-32 | Prevent automation runaway; each session holds memory |
| Idle timeout | None | None | 30 min | Clean up abandoned automation sessions |
| TLS for gRPC | Disabled | Disabled | **Enabled** | gNMI carries config data — encrypt it |
| Confirmed commit timeout | 600s (XR) | 10 min (Junos) | 120-300s | Long enough to verify, short enough to auto-rollback |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| NETCONF connection refused | SSH subsystem not configured, port 830 blocked | Verify `netconf-yang agent ssh`, check ACL/firewall |
| `<rpc-error>` on edit-config | YANG path mismatch, wrong namespace, missing key | Use `pyang` to validate path, check model revision |
| Commit fails with conflict | Another session holds candidate lock | Check `show netconf-yang clients`, clear stale session |
| gNMI subscribe returns empty | Wrong path, model not supported | Run `gnmic capabilities` first, verify path with `gnmic get` |
| RESTCONF 404 | URL path encoding wrong, module not loaded | Verify model name/revision, check URL escaping (slashes → `%2F`) |
| Timeout on large get-config | Full config tree is enormous on SP routers | Use subtree/XPath filter, never request entire `/` unfiltered |
| "Exceeded maximum sessions" | Automation leaked sessions (no disconnect) | Implement session cleanup, set idle-timeout, audit code |
| Confirmed commit auto-rollback | Confirm RPC not sent within timeout | Increase timeout, add health check before confirm |

### Methodology

1. **Verify transport** — Can you SSH to port 830? `nc -zv router 830`. Can you reach gRPC port? `grpcurl router:57400 list`
2. **Check capabilities** — NETCONF hello exchange, gNMI CapabilityRequest. If the model isn't advertised, it's not supported.
3. **Validate YANG path** — Use `pyang -f tree <model>.yang` to visualize the tree. Most path errors are namespace or key mismatches.
4. **Test with minimal RPC** — `<get>` with a small filter before attempting `<edit-config>`.
5. **Check commit state** — Is candidate dirty? Is another session holding a lock? On Junos: `show configuration | compare`.
6. **Examine device logs** — IOS-XR: `show logging | include netconf`. Junos: `show log messages | match mgd`.
7. **Packet capture** — Last resort: `tcpdump -i any port 830 -w netconf.pcap`. NETCONF XML is readable in Wireshark.

### War Stories

> **The Phantom Lock**: A large SP was deploying NETCONF-based automation for 200+ PE routers. Everything worked in the lab, but in production, random routers would reject edit-config with "candidate locked by another session." Investigation revealed their Python automation framework didn't call `<close-session>` on exception paths — if a validation error occurred mid-workflow, the SSH connection was dropped without unlocking candidate. NETCONF sessions have no inherent idle timeout on many platforms (IOS-XR's default is unlimited), so the lock persisted until process restart. Fix: explicit `<unlock>` in finally blocks, platform idle-timeout set to 30 minutes, and a cron job to audit stale NETCONF sessions and issue `<kill-session>` for any session idle > 15 minutes. The Junos equivalent — `request system logout pid` — was needed for three Juniper MX routers that also held stale locks.

### Example: ncclient (Python)

```python
from ncclient import manager

# Connect to IOS-XR PE router
with manager.connect(
    host="10.0.0.1",
    port=830,
    username="admin",
    password="secret",
    hostkey_verify=False,       # Lab only — verify in production
    device_params={"name": "iosxr"},
    timeout=30,
) as m:
    # Get BGP neighbor state
    filter_xml = """
    <bgp xmlns="http://openconfig.net/yang/bgp">
      <neighbors>
        <neighbor>
          <neighbor-address>10.0.12.2</neighbor-address>
          <state/>
        </neighbor>
      </neighbors>
    </bgp>
    """
    result = m.get(filter=("subtree", filter_xml))
    print(result.xml)

    # Edit config with confirmed commit
    # NOTE: ncclient adds <config> wrapper — do NOT include it in the XML
    config_xml = """
      <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces">
        <interface>
          <name>Loopback99</name>
          <description>Automation test</description>
          <enabled>true</enabled>
        </interface>
      </interfaces>
    """
    m.lock("candidate")
    try:
        m.edit_config(target="candidate", config=config_xml)
        m.validate("candidate")
        m.commit(confirmed=True, confirm_timeout=120)  # Auto-rollback in 120s
        # ... verify connectivity ...
        m.commit()  # Confirm (cancels auto-rollback)
    except Exception:
        m.discard_changes()
        raise
    finally:
        m.unlock("candidate")
```

## Architecture Patterns

### Pattern 1: Source-of-Truth Driven (Declarative)

```
┌─────────────────┐     ┌───────────────────┐     ┌──────────────┐
│  Source of Truth │────▶│  Config Generator  │────▶│  Device      │
│  (NetBox, Git)  │     │  (Jinja2/Python)   │     │  (NETCONF    │
│                 │◀────│                    │     │   replace)   │
│  Intended state │     │  Renders YANG XML  │     │              │
└─────────────────┘     └───────────────────┘     └──────────────┘
```

- **Replace** operation at container level ensures device matches source-of-truth exactly
- Drift detected by comparing `<get-config>` output against intended state
- Production example: Google's network automation, Meta's FBOSS

### Pattern 2: Event-Driven (Closed-Loop)

```
┌──────────┐   gNMI    ┌──────────────┐  Decides   ┌──────────────┐
│  Device   │─────────▶│  Collector/  │───────────▶│  Controller  │
│  (sensor) │  stream  │  Processor   │            │  (PCE, ODL)  │
│           │◀─────────│  (Kafka,     │◀───────────│              │
│           │  gNMI    │   InfluxDB)  │  Pushes    │  SR-TE path  │
│           │  set     └──────────────┘  config    │  recompute   │
└──────────┘                                       └──────────────┘
```

- Telemetry triggers automated response (e.g., link utilization > 80% → reroute SR-TE policy)
- Covered in depth in section 11.3 (SR-TE Controller Integration)

### Pattern 3: Hybrid (Most Common in SP)

Most real SP networks use a hybrid:
- **NETCONF** for config changes (transactional, auditable)
- **gNMI** for telemetry collection (streaming, efficient)
- **RESTCONF** for lightweight operational queries (dashboards, NOC tools)
- **CLI** as escape hatch (break-glass, one-off debugging)

## Review Questions

1. **Scenario:** Your automation framework pushes BGP neighbor configs to 50 PE routers using NETCONF `<edit-config>` with `merge` operation. A decommissioned customer's BGP session is removed from your source-of-truth, but the session persists on the routers. Why? What operation should you use instead, and what are the risks of switching?

2. **Design:** You're designing the management plane for a greenfield SP backbone (IOS-XR and Junos mixed). You need configuration management, real-time telemetry for traffic engineering, and a NOC dashboard. Which protocol (NETCONF/RESTCONF/gNMI) would you select for each use case, and why?

3. **Troubleshooting:** A NETCONF session to a PE router hangs on `<commit>` for over 60 seconds and eventually times out. The candidate datastore has valid XML (passes `<validate>`). What are the likely causes, and how would you diagnose the issue?

4. **Comparison:** Compare `ON_CHANGE` and `SAMPLE` gNMI subscription modes for monitoring BGP neighbor state on a route reflector with 500 peers. Which mode is appropriate, what's the resource impact of each, and what failure mode does each have?

5. **Architecture:** A Tier-1 SP wants to implement "zero-touch provisioning plus confirmed commit" for 2,000 PE routers. They want every config change to auto-rollback if it causes a connectivity loss. Describe the workflow using NETCONF operations, including error handling for cases where the confirming NETCONF session itself is disrupted by the change.
