# 11.4 CI/CD for Network Config

> **Level:** Internet Expert / IE-SP  
> **Prerequisites:** 11.1 (Model-Driven Networking), 11.2 (Streaming Telemetry), 11.3 (SR-TE Controller Integration)

---

## Overview

In SP networks, config mistakes are rarely isolated. One bad route-policy can blackhole a region; one wrong prefix-limit can reset thousands of BGP sessions. CI/CD for network config replaces fragile, ticket-driven CLI changes with reproducible pipelines: lint → validate → simulate → stage → deploy → verify → rollback.

For service providers, CI/CD is not "DevOps cosplay." It's blast-radius control.

## Why CI/CD Matters in SP

### Traditional Change Model (Pain)
- Engineer edits CLI manually on production device
- Validation is mostly "show" commands after the fact
- Rollback depends on memory, screenshots, and panic
- Knowledge lives in people, not in versioned artifacts

### CI/CD Model (Control)
- Desired state is versioned in Git
- Every change goes through automated checks before touching production
- Deploys are staged (lab → canary → ring-based rollout)
- Post-deploy health gates can auto-abort or trigger rollback workflows (when rollback is feasible for that change type)

## CI/CD Architecture for Service Providers

```text
Engineer PR
   |
   v
GitHub/GitLab
   |
   +--> CI Stage 1: Syntax + schema lint (YANG, Jinja2, policy checks)
   +--> CI Stage 2: Semantic validation (Batfish, policy unit tests)
   +--> CI Stage 3: Render + diff + peer approval gates
   |
   v
CD Controller (Argo/NSO/Ansible/AWX)
   |
   +--> Pre-check: device reachability, lock, backup snapshot
   +--> Deploy ring 0 (lab/canary)
   +--> Verify SLO/telemetry gates
   +--> Deploy ring 1, ring 2...
   |
   v
Observability inputs (telemetry transport: gNMI/SNMP; logs/events; protocol/service KPIs: BFD/BGP/IS-IS/SR/L3VPN health)
   |
   +--> Trigger rollback workflow if health checks fail (with explicit caveats for non-reversible/stateful changes)
```

### Core Components

1. **Source of Truth (SoT)**
   - Git repo with declarative intent (not ad-hoc CLI text blobs)
   - Inventory, roles, policy objects, templates, test cases

2. **Validation Engine**
   - Syntax: yamllint, pyang, templating checks
   - Semantics: policy tests + route behavior simulation

3. **Orchestrator**
   - NSO, Ansible/AWX, Nornir, or custom pipeline runners
   - Must support idempotence, retries, and partial-failure handling

4. **Observability + Gates**
   - Hard pass/fail criteria after deploy (BGP session count, route churn, loss, latency)
   - No "looks fine to me" promotion

## Pipeline Stages (Practical)

### Stage 0 — Commit Hygiene

- Conventional commit message required
- Change ticket / incident / maintenance reference mandatory
- Files must pass formatting + lint before pipeline starts

### Stage 1 — Static Validation

- Template render check for all targeted devices
- YANG-based payload validation for NETCONF/RESTCONF changes
- Duplicate object detection (prefix-sets, communities, policy names)
- Reserved resource validation (labels, RTs, colors, SR policy names)

### Stage 2 — Behavioral Validation

- Unit-test route policies with known input/output routes
- Simulate control-plane impact (where supported) using Batfish-like checks; unsupported features require targeted lab/synthetic validation
- Assert invariants:
  - No default route leak from customer VRFs
  - No transit between forbidden AS domains
  - No route-policy removes critical BGP communities

### Stage 3 — Human Gate

- Mandatory peer review
- Mandatory change rationale in PR description:
  - What is changing?
  - Why now?
  - Blast radius?
  - Rollback trigger + plan?

### Stage 4 — Pre-Deploy Checks

- Acquire config lock using platform/API-specific mechanism (for example NETCONF lock/candidate lock where supported)
- Create platform-native rollback point/snapshot per device
- Config backup pulled
- Reachability and API session checks
- Dry-run diff (`show | compare`, `commit check` on Junos, and tooling-based candidate diff/validation on XR)
- **Do not treat `commit confirmed` as dry-run** — it is a live change safety net

### Stage 5 — Deploy (Rings)

- Ring 0: 1 canary node per role/region
- Ring 1: 10–20% of fleet
- Ring 2+: full rollout only if SLO gates green

### Stage 6 — Post-Deploy Verification

- Protocol state checks (BGP/ISIS/LDP/SR/BFD)
- Service checks (L3VPN reachability, latency thresholds)
- Telemetry drift checks vs baseline

### Stage 7 — Auto-Rollback

- Triggered by health gate failures
- Use pre-saved checkpoint and confirmed commit timeouts
- Incident record auto-created with evidence bundle

## Data Model Strategy

### Intent Model (Recommended)

Store intent objects like:
- `peer-groups`
- `routing-policies`
- `service-templates`
- `region/ring metadata`

Avoid storing only raw vendor CLI snippets. CLI-only repos quickly become untestable spaghetti.

### Example Intent Object

```yaml
service: l3vpn
customer: ACME
vrf: ACME-INET
rd: 65000:1101
rt_import: [65000:1101]
rt_export: [65000:1101]
bgp_policy_in: CUST-IN-SANITIZE
bgp_policy_out: CUST-OUT-ALLOWLIST
max_prefix: 50000
rings:
  - canary
  - west-core
  - east-core
```

## Platform-Aware Deployment Safety

### IOS-XR Guardrails

- On IOS-XR, use commit workflows supported by your release/transport (CLI/NETCONF), including validated/confirmed commit patterns where available
- Prefer atomic policy replacement blocks over line-by-line drift edits
- Preserve route-policy names to reduce control-plane churn
- For risky changes, use timed confirmed commit + watchdog automation

```cisco
! Deploy safety pattern (live change, not dry-run)
configure
 router bgp 65000
  neighbor-group EDGE-PEERS
   description EDGE-PEERS-CI-UPDATE
!
commit confirmed 300
! Pipeline runs post-checks within 5 minutes
! If checks fail, pipeline should trigger explicit rollback immediately.
! Timeout-based auto-revert is safety net only, not primary rollback method.
```

### Junos Guardrails

- Always use candidate mode + `commit check`
- Use `commit confirmed <minutes>` for production changes
- Group related config in `groups`/`apply-groups` for predictable rollbacks

```junos
configure
set policy-options policy-statement CUST-IN-SANITIZE term BLOCK-DEFAULT from route-filter 0.0.0.0/0 exact
set policy-options policy-statement CUST-IN-SANITIZE term BLOCK-DEFAULT then reject
set policy-options policy-statement CUST-IN-SANITIZE term ALLOW then accept
commit check
commit confirmed 5
# CI verifies BGP sessions + prefix acceptance
# After gates pass, run plain `commit` to confirm/finalize
```

## Telemetry Gates (Promote/Abort)

Promotion criteria (example):
- BGP established peers meet role-weighted thresholds (for example 100% on critical RR/ASBR sets, thresholded on large edge populations)
- IS-IS adjacency flaps <= threshold
- Service-path loss/jitter/latency probes remain within guardrails (not interface counters alone)
- SR policy active-path count unchanged for protected services
- 95th percentile latency delta < 3 ms for affected service classes

If any gate fails:
- Freeze rollout
- Auto-rollback changed ring
- Open incident with attached diffs + telemetry snapshots

## Rollback Patterns

### 1) Native Confirmed Commit Timeout
- Strong first-line safety net
- Not a substitute for orchestrated rollback across all affected nodes/services

### 2) Checkpoint-Based Revert
- Save checkpoint before deploy
- Revert with explicit command if gates fail

### 3) Git Revert + Pipeline Redeploy
- For deterministic intent repos
- Fastest path for policy mistakes discovered after rollout

## Common Failure Modes

| Symptom | Likely Cause | Fix |
|---|---|---|
| Canary passes, ring-1 fails | Device role mismatch not encoded in template | Add role guards + capability matrix in render stage |
| BGP sessions flap after policy update | Policy replacement changed term ordering | Add policy unit tests for ordering + explicit term IDs |
| Rollback fails on subset of nodes | Drifted manual config on live nodes | Enforce no-CLI-drift policy + nightly drift detection |
| Commit confirmed expires unexpectedly | Pipeline exceeded maintenance window | Increase timeout or split deployment rings |
| Service impact despite green protocol checks | Missing service-layer probes | Add synthetic traffic checks per service class |

## Reference Pipeline (GitHub Actions Style)

```yaml
name: network-ci
on:
  pull_request:
    types: [opened, synchronize, reopened, closed]

jobs:
  validate:
    if: github.event.action != 'closed'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: make lint
      - run: make render
      - run: make policy-test
      - run: make simulation

  deploy-canary:
    if: github.event.action == 'closed' && github.event.pull_request.merged == true
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh --ring canary
      - run: ./verify.sh --ring canary
```

> Note: This is a conceptual flow. Real deployments must include auth vaulting, maintenance windows, explicit approvals, and rollback hooks.

## Design Principles (What Actually Works)

1. **Test policies like code**
   - Most outages are policy logic errors, not transport failures.

2. **Canary by role, not by random node**
   - One edge PE and one core RR can represent very different risk.

3. **No human-only rollback plans**
   - If rollback requires someone remembering commands at 3 AM, you don't have rollback.

4. **Treat observability as part of CD**
   - Deploy without gates is just automated risk.

5. **Drift is the silent killer**
   - CI/CD assumes declared state is truth; manual drift breaks that assumption.

## War Story: The "Green Pipeline" Outage

A provider introduced CI/CD for BGP policy updates. Pipeline was pristine: lint passed, render passed, canary passed. Ring-1 rollout dropped 18% of customer routes in one region.

Root cause: policy term order changed during a template refactor. Unit tests covered prefix matches but not term precedence side-effects. Canary PE had a small customer set and didn't expose the conflict; ring-1 had large enterprise customers with overlapping communities.

Fixes applied:
1. Added policy precedence tests (first-match behavior)
2. Added role-based canary selection (high-diversity customer edge first)
3. Added service-level synthetic route probes before promotion

Lesson: "Pipeline green" is not proof of safety. It means your tests passed — and your tests may be incomplete.
