## 1.1 SP Network Architecture Overview

### What a Service Provider Network Actually Is

A service provider network exists to carry other people's traffic. Not a subtle distinction — it changes everything about how you design, operate, and think about the network.

An enterprise network connects *your* users to *your* applications. You control the traffic. You know (roughly) who's doing what. The network's job is to be invisible.

A service provider network connects *other companies'* users to *other companies'* applications. You don't control the traffic. You often don't know what's flowing. The network's job is to be a revenue-generating service that works so reliably that customers never think about it — and to do that for hundreds or thousands of customers simultaneously without any one of them affecting the others.

This is the foundational fact from which everything else in this guide derives.

### The SP Business Model

Service providers sell connectivity. The products vary — dedicated internet access, MPLS VPN, Ethernet private line, cloud interconnect, mobile data transport — but the underlying model is consistent:

1. **Customers pay for guaranteed performance**: bandwidth, latency, jitter, availability. These become contractual SLAs with financial penalties for violation.
2. **Revenue is recurring**: monthly charges tied to service uptime. Every outage minute has a direct financial cost.
3. **Multi-tenancy is the design constraint**: thousands of customers share physical infrastructure. They must be completely isolated from each other in the data plane, and the control plane must not leak routing information between them.

The key word in that last point is *thousands*. A typical Tier-2 SP might have:
- 800–3,000 enterprise VPN customers
- 50,000–200,000 broadband residential customers  
- 10–100 transit peers and CDN customers
- 50–300 IX members

All of these customers get services from the same physical backbone. The protocols you'll learn in this guide — VRFs, MPLS label stacks, BGP communities, VNIs — are how that sharing is made possible without contamination.

### Scale: The Numbers That Define SP Design

The scale difference between SP and enterprise isn't linear — it's categorical. Consider what "routing table" means at different scales:

| Environment | IPv4 BGP Table | VRFs | Physical Nodes |
|---|---|---|---|
| Enterprise (large) | 10–50K internal | 0–5 | 50–500 |
| Regional ISP | 500K+ full BGP | 500–5,000 | 200–2,000 |
| Tier-1 / National | ~1M full BGP | 5,000–50,000 | 2,000–20,000+ |

At regional ISP scale, you cannot run OSPF with 900,000 routes in the global table — it would flood the entire backbone with every BGP prefix change. This is why SP networks use MPLS: the backbone routers don't need to know the BGP routes. They just swap labels. BGP routes live only on the PE routers that connect customers.

The convergence numbers also change:
- Enterprise acceptable outage: minutes (maybe seconds for critical links)
- SP acceptable outage: **sub-50ms** for protected paths; **sub-200ms** for full reconvergence
- IE-SP exam expectation: you can tune any tier of the network to hit these targets

### The Three-Tier Topology

Almost every SP backbone — regardless of size — decomposes into three tiers:

```
                    ┌──────────────────────────────────┐
        CORE        │  P (Provider) Routers            │
                    │  High-speed label switching only  │
                    │  No customer-facing interfaces    │
                    │  Typically no customer BGP        │
                    │  (just IGP + MPLS labels)         │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────┴───────────────────┐
        EDGE        │  PE (Provider Edge) Routers       │
                    │  Customer VRFs live here          │
                    │  BGP for VPNv4/VPNv6/EVPN        │
                    │  Connects to P via labeled paths  │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────┴───────────────────┐
        ACCESS      │  CE (Customer Edge) / CPE        │
                    │  Customer-managed or SP-managed   │
                    │  PE-CE protocol: BGP, OSPF, or    │
                    │  static depending on product      │
                    └──────────────────────────────────┘
```

**P (Provider) routers** are the backbone. They see MPLS labels, not IP prefixes. They switch packets at line rate without inspecting them. They need to know: (a) how to reach other P and PE routers via the IGP, and (b) what labels map to what next-hops. That's it. A P router with 10K IGP routes can forward packets for 50K customers without any customer routing knowledge. (Note: in Inter-AS Option C designs, P routers may also run BGP labeled-unicast (BGP-LU) for cross-AS MPLS reachability. Some operators also run iBGP on P routers for operational visibility, but this is not required for basic forwarding.)

**PE (Provider Edge) routers** are where the complexity lives. Each PE:
- Maintains one VRF per customer (complete routing table isolation)
- Runs BGP with route reflectors to exchange VPN routes
- Pushes MPLS labels onto customer packets as they enter the backbone
- Pops labels and routes to the correct VRF as packets exit

**CE (Customer Edge)** is the demarcation point. The CE might be customer-managed or SP-managed (as a CPE). The PE-CE protocol — BGP, OSPF, static — depends on the service product and customer sophistication.

This guide is organized by protocol, but the protocols are not independent. Here's how they fit together in a real backbone:

## 1.2 Design Principles

### Three Principles That Govern Every Decision

SP network design comes down to three principles that permeate every choice — every protocol selection, every hierarchy level, every timer value, every redundancy model. They are:

1. **Scalability** — Can this design handle 10× the current load?
2. **Convergence** — How quickly does the network recover from failure?
3. **Fault isolation** — Does a failure affect only what it should?

These three are not independent. Optimizing one often creates tension with another. Understanding those tradeoffs — and knowing which principle dominates in which context — is what separates SP networking expertise from protocol memorization.

### Scalability

**The scalability problem in SP networks:** you have a large number of devices, a large number of routes, and a large number of customers. Each of these dimensions must be controlled independently, and they must not multiply against each other.

**Route table scalability**: 
- BGP routes (~1M IPv4 prefixes, growing) live only on PEs and route reflectors
- P routers need only IGP routes (typically 5K–20K for even a large national backbone, with summarization keeping the number bounded)
- IGP summarization at area boundaries keeps flooding domains bounded
- MPLS labels scale independently of the routing table — a PE needs one /32 label per remote PE, not per VPN route

The mathematical insight: the *transport label space* on a backbone with N PEs needs O(N) labels (one per remote PE loopback), not O(routes × N). PEs also allocate VPN labels — per-prefix or per-VRF depending on label allocation mode — but those labels are local to the PE, not carried by every P router. This separation is why MPLS scales where IP-only backbones cannot.

**Protocol scalability**:
- Industry experience suggests IS-IS handles 2,000+ routers in a single level with less operational friction than OSPF — IS-IS's TLV-based extensibility and simpler flooding model (no OSPF LSA type complexity) reduce edge cases at scale
- BGP route reflectors collapse an O(N²) iBGP mesh to O(N) sessions
- SR-MPLS eliminates per-tunnel signaling state on transit routers — no RSVP Path/Resv messages in the backbone; headend state still exists for SR-TE policies, but transit P routers carry no per-tunnel state

**Customer scalability**:
- VRFs provide complete routing isolation — 10,000 customers don't affect each other
- RT constrain (BGP ORF) keeps VPNv4 routes from flooding to PEs that don't need them
- EVPN Route Type 3 reduces BUM flooding compared to VPLS

**Design rule**: whenever you're evaluating a technology choice, ask "what happens at 10× scale?" If the answer involves an O(N²) relationship — full iBGP mesh, spanning-tree convergence, ARP flooding domain — the design will break at scale. The SP choice almost always involves hierarchy, summarization, or a protocol that avoids the O(N²) relationship entirely.

### Convergence

**Sub-50ms is the target** for a protected SP backbone. This is not aspirational — it's a contractual requirement in most carrier SLAs. Here's how it's achieved:

**Layer 1/2 failure detection:**
- BFD (Bidirectional Forwarding Detection) — detects link failures in 10–150ms depending on platform and configuration
- Without BFD, IGP hello dead intervals are 30–40 seconds
- BFD timer: `min-interval Xms multiplier 3` = 3×X detection. Hardware-offloaded BFD on specific linecards (NCS 5500, PTX, 7750 SR) supports 3.3–10ms intervals; software BFD floors at 50–100ms on most platforms
- Typical SP deployment: 10ms×3=30ms or 50ms×3=150ms detection

**IGP reconvergence:**
- IS-IS/OSPF fast-hello (1-second intervals) for near-instant detection without BFD
- SPF throttle tuning: initial 50ms, backoff 200ms, max 5s
- Incremental SPF (iSPF) — only recompute affected subtree, not full topology

**Pre-computed FRR (Fast Reroute):**
- RSVP-TE FRR (legacy): pre-signaled bypass tunnels, sub-50ms switchover
- TI-LFA (modern): SR-MPLS computes backup path using P/Q space algorithm; no RSVP state needed; sub-50ms on any topology

**Convergence budget** — work backwards from the 50ms target:
```
BFD detection:              ~30ms  (10ms interval × 3 with HW-offload)
FRR path switchover:        ~10ms  (pre-computed backup installed in FIB)
Propagation + processing:   ~10ms  (hardware forwarding latency)
─────────────────────────────────
Total:                      ~50ms
```

This is a simplified pedagogical model. Real SP designs document detection time + switchover time + propagation delay. The key insight: BFD + TI-LFA is the combination that consistently achieves sub-50ms on a multi-hop MPLS backbone. Without hardware-offloaded BFD (software BFD at 50ms×3 = 150ms), the target becomes sub-200ms instead.

**Convergence after the backup path:** The 50ms window is the FRR path. Full IGP reconvergence — new primary LSPs, updated forwarding entries everywhere — takes 200ms–2s depending on topology size. Both matter: the FRR path is what keeps the SLA during the outage; the full reconvergence determines when the backup path is cleared and resources are restored.

### Fault Isolation

**The problem**: a large SP backbone is a shared medium. A fault in one domain should not cascade to affect others. Two categories of fault isolation matter:

**Control plane isolation:**
- IS-IS level boundaries prevent flooding from propagating across the backbone
- BGP VRF isolation: a PE's VRF routes are not visible to other VRFs or the global table
- Route reflector clients do not peer directly with each other; they rely on RRs to distribute routes between them
- RPKI ROV at the edge prevents origin AS hijacking; PE-CE prefix filtering (max-prefix limits, prefix lists) protects against a misbehaving CE injecting unauthorized routes

**Data plane isolation:**
- MPLS VPN label stacks ensure customer traffic cannot cross VRF boundaries
- EVPN site-of-origin (SoO) prevents routing loops at multi-homed sites
- Ingress filtering (BCP38) prevents source IP spoofing in the SP core

**Failure domain design**: the goal is that any single failure should affect only the minimal set of customers and infrastructure. This informs:
- Diverse physical paths (not same conduit, not same fiber, not same DWDM system)
- Redundant PE attachment (dual-PE, dual-homed CE)
- Geographic P router placement (ring topology with N-2 protection possible)
- Separate IGP instances for underlay vs overlay (SR-MPLS underlay, EVPN overlay)

**The failure isolation cascade test**: for any proposed design, ask "if this device fails, what is the blast radius?" Then ask "if two adjacent devices fail simultaneously?" If the blast radius of a two-device failure takes down an entire metro region, the design needs more redundancy or better hierarchy.

### QoS and Service Differentiation

One principle that permeates every SP design but deserves its own callout: **Quality of Service (QoS)**. SP networks carry traffic for customers paying different prices for different service levels. The network must differentiate:

- **Traffic classification**: DSCP marking at ingress (PE-CE boundary) assigns each packet to a forwarding class
- **MPLS Traffic Class (TC)**: the 3-bit TC field in the MPLS header carries QoS information across the backbone
- **Scheduling and queuing**: each P and PE router implements strict-priority (for voice/real-time) and weighted-fair queuing (for data classes)
- **Policing and shaping**: bandwidth enforcement at customer interfaces ensures SLA compliance

A typical SP QoS model defines 4–8 forwarding classes (e.g., Network Control, Real-Time, Business Critical, Best Effort, Scavenger). This model is applied consistently at every hop — inconsistent QoS policy across the backbone breaks SLA guarantees.

QoS is not covered in a standalone module in this guide, but it is referenced in the context of traffic engineering (Module 5), 5G transport (Module 10, which includes a detailed QoS model), and design case studies (Module 12).

### Design Tradeoffs and the Day-2 Framing

New SP engineers often optimize for the initial design — the "Day 1" configuration. Experienced engineers design for Day 2: the operations, the troubleshooting, the ticket at 3 AM.

**Day-2 design principles:**

**Consistency over cleverness**: a uniform IS-IS metric scheme you can explain in 30 seconds beats an elaborately tuned policy that only the original designer understands. When the backbone is on fire at 2 AM, simple is correct.

**Observability by design**: if you can't measure it, you can't operate it. Every tunnel, every VRF, every BGP session should have telemetry. gNMI streaming (Module 11) exists precisely because SNMP polling doesn't scale for SP operations.

**Conservative defaults in production**: `commit confirmed` before any major change. Hot standby configurations for RRs. Maintenance mode (IS-IS overload, BGP graceful-shutdown) before touching a live device.

**Documentation as config**: every design decision — why IS-IS L1/L2 rather than flat, why this SRGB allocation, why this RD/RT scheme — should be in a document that survives the engineer who made it. This matters more in SP than enterprise because the blast radius of institutional knowledge loss is higher.

## 1.3 SP vs Enterprise — Fundamental Differences

This section exists because many SP engineers come from enterprise backgrounds, and the mental model transfer is not always clean. The protocols overlap, the terminology overlaps, but the design philosophy is deeply different.

### Protocol Selection: Why SP Chooses Differently

**IS-IS vs OSPF in the backbone:**

Enterprise networks default to OSPF. SP networks predominantly use IS-IS. The reasons:

| Factor | IS-IS Advantage | OSPF Behavior |
|---|---|---|
| Topology independence | Runs over any L2, including CLNS-only links | Requires IP-reachable neighbors |
| Convergence at scale | Cleaner with 2,000+ nodes, fewer flooding nuances | Well-characterized up to ~500 nodes |
| BGP interaction | Traditional SP choice; tooling ecosystem | Works equally well |
| Segment routing | SR-MPLS, SRv6 both use IS-IS extensions natively in most implementations | Also supported, but IS-IS extensions deployed first |
| Vendor default | Juniper PE defaults, many NTT/AT&T configs | Cisco campus default |

The honest answer: in a greenfield SP deployment, both work. IS-IS is the overwhelming choice because the industry conventions, tools, and interop experience are there.

**BGP as the SP control plane:**

In enterprise networking, BGP is the external routing protocol — it handles internet edge and maybe some campus separation. In SP networking, BGP is the primary mechanism for:

- Distributing VPN routes (MP-BGP, VPNv4/v6 address families)
- EVPN control plane for L2VPN and DCI
- Route reflectors replacing iBGP full mesh (O(N²) → O(N) sessions)
- BGP-LS for exporting topology to SDN controllers
- Flowspec for programmatic traffic steering

An SP BGP design must handle:
- BGP tables of ~1M prefixes per address family
- 5,000–50,000 VRF route imports per PE
- RR scale: a single RR reflecting routes for 300+ PEs
- Session stability under flap damping and route suppression

This is categorically different from enterprise BGP, which rarely involves more than a few hundred prefixes or a few dozen sessions per router.

**MPLS as a requirement:**

Enterprise networks can function without MPLS. Most do. SP networks cannot — MPLS is the mechanism that makes the P router separation possible. Without MPLS:

- P routers would need full BGP VPN routes (millions of entries)
- Traffic separation would require VRF-aware forwarding at every hop
- Traffic engineering would require source routing or policy routing at every hop

MPLS decouples the routing knowledge from the forwarding knowledge. P routers only need to know how to forward labels; the routing intelligence lives at the edges.

**Convergence requirements:**

| Metric | Enterprise | SP |
|---|---|---|
| Acceptable downtime (planned) | Minutes to hours | Sub-30 minutes with change control |
| Acceptable downtime (failure) | 2–5 minutes often acceptable | Sub-50ms for protected paths |
| SLA penalty trigger | Usually none | Financial penalties in SLA |
| Uptime target | 99.9% ("three nines") | 99.999% ("five nines") |
| BFD | Nice to have | Mandatory on all production links |
| FRR/TI-LFA | Optional | Mandatory |

Five nines means 5.26 minutes of downtime per year. You can't achieve that without BFD and pre-computed fast-reroute.

### Multi-Tenancy: The SP Design Constraint

Enterprise networks handle multi-tenancy as an edge case (guest Wi-Fi, PCI segmentation). SP networks are fundamentally multi-tenant — it's the business model.

**VRF as the isolation primitive:**

Every customer L3VPN service lives in a separate VRF on the PE. The VRF:
- Has its own routing table (no route leakage to other VRFs or the global table by default)
- Has its own forwarding table
- Has its own BGP import/export policy
- Uses Route Distinguisher (RD) to make customer routes globally unique in the VPNv4/v6 address space

A PE with 500 enterprise VPN customers has 500 VRFs. Each VRF looks like a separate router to the customer — they see their own routes, their own prefixes, their own peers.

**Route Target (RT) as the VPN membership signal:**

The RD makes routes unique; the RT determines which VRFs import the route. This separation is powerful:
- Hub-and-spoke VPN: hub imports all spoke RTs; spokes only import hub RT
- Extranet: shared services VRF exports with a shared RT; customer VRFs import it
- Any-to-any: all sites in the VPN share the same RT

See Module 7 for full L3VPN design detail.

**EVPN for L2 multi-tenancy:**

Where L3VPN provides IP isolation, EVPN provides Ethernet isolation. A customer who needs layer-2 adjacency between sites — shared VLANs across a WAN — gets EVPN VPWS or EVPN multipoint. From the customer's perspective, their switches are directly connected. From the SP's perspective, the customer's L2 domains are fully isolated from every other customer.

See Module 8 for L2VPN and EVPN design.

### Operational Model

**NOC vs help desk:**

Enterprise IT has a help desk that escalates to network engineers. SP has a Network Operations Center (NOC) — a 24×7 operations team that watches dashboards, handles alarms, and executes runbooks. The design must support NOC operations:

- Every link, every BGP session, every service path is monitored
- Alarms have severity levels and defined escalation paths
- MTTR (Mean Time to Repair) is measured; typical Tier-2 SP target is 4 hours for major incidents
- Change control windows exist; every major change has a back-out procedure

**Fault detection → isolation → repair:**

The operations workflow:
1. **Detect** (automated): gNMI telemetry, SNMP traps, BFD notifications
2. **Isolate** (NOC): correlate alarms, identify scope, engage field if physical
3. **Contain** (NOC): route traffic around the fault using maintenance commands
4. **Repair** (field + NOC): fix the physical or logical fault
5. **Verify** (NOC): confirm full service restoration before closing ticket

The design must make step 2 fast. If a fiber cut triggers 3,000 correlated alarms, the NOC can't isolate the root cause quickly. Hierarchical alarm correlation, event suppression, and well-designed monitoring are operational requirements, not nice-to-haves.

**Revenue-generating vs cost-center framing:**

In enterprise, the network is an operating expense — it enables the business but doesn't directly generate revenue. In SP, the network *is* the product. This changes how investment decisions are made:
- A $2M router that reduces MTTR by 30 minutes per year might pay for itself in SLA credits avoided
- Automation investments (Module 11) are justified by reduced OPEX, not just engineering elegance
- Capacity planning is tied directly to revenue forecasts

## 1.4 Reference Topologies

### How to Read a Topology Diagram

Before introducing the reference topologies, a few conventions used throughout this guide:

- **P** = Provider core router (no customer interfaces, label-only forwarding)
- **PE** = Provider Edge router (customer-facing, VRFs, BGP VPN)
- **RR** = Route Reflector (may be co-located on a PE or dedicated)
- **CPE/CE** = Customer Premises/Edge equipment
- **PoP** = Point of Presence (a physical datacenter location with P/PE equipment)

Topology diagrams in this guide show the *logical* topology — IP/MPLS connections — not the physical topology. The physical topology may involve DWDM amplifiers, OTN muxponders, and fiber paths that are invisible at the IP layer.

### Topology 1: Regional ISP Backbone (~300–800 Nodes)

A regional ISP serves a geographic area — a single metropolitan region, a state, or a small country. Typical scale: 300–800 routers, 50K–500K BGP prefixes, 500–5,000 L3VPN customers.

```
                    ┌─────────────────────────────────┐
                    │        BACKBONE RING            │
                    │                                 │
                    │   P1 ────────────── P2          │
                    │   │                  │          │
                    │   │     P3──P4       │          │
                    │   │     │    │       │          │
                    │   P5 ──P6   P7──P8  │          │
                    │   │                  │          │
                    │   P9 ────────────── P10         │
                    │                                 │
                    └──────────┬──────────────────────┘
                               │ iBGP (RR)
                    ┌──────────┴──────────────────────┐
                    │    METRO AGGREGATION            │
                    │                                 │
                    │  PE-METRO-1    PE-METRO-2       │
                    │       │              │          │
                    │  PE-METRO-3    PE-METRO-4       │
                    │       │              │          │
                    └──────────────────────────────────┘
                               │
                    Customer CE devices
```

**IS-IS design:**
- Single IS-IS Level 2 for the backbone P routers
- Optional L1 areas for each metro aggregation zone (keeps SPF flooding bounded)
- Wide metrics; SR-MPLS prefix-SIDs on all P and PE loopbacks

**BGP design:**
- 2 Route Reflectors (geographically distributed for redundancy)
- All PEs are RR clients; RRs reflect VPNv4/VPNv6/EVPN routes
- Internet edge: 2–4 eBGP upstream transit peers + IXP peering

**Redundancy:**
- P ring provides two diverse paths between any PE pair
- PE metro routers are dual-attached to the P ring
- Customer CEs are typically single-homed (cost) with diverse backups for premium customers

**Typical equipment:**
- P routers: Cisco ASR 9000, Juniper PTX, Nokia 7750 (high throughput, moderate route scale)
- PE routers: Cisco ASR 9000/NCS 5500, Juniper MX (high route scale, VRF capacity)

### Topology 2: National Carrier (2,000–20,000 Nodes)

A national carrier spans a country with multiple major cities and hundreds of regional PoPs. Scale: 2,000–20,000 routers, full BGP table (~~1M prefixes), 5,000–50,000 L3VPN customers.

```
                 NATIONAL BACKBONE (Core PoPs)
                 
         NYC──────────────────────────────LAX
          │\                              /│
          │  \                          /  │
          │    CHI────────────────DFW     │
          │     │\              /│        │
          │     │  \          /  │        │
          │     │   ATL──MIA   │        │
         BOS    │              │         SEA
                │              │
               DC             PHX
               
         (each node = major backbone PoP with P routers)

                    ↕ iBGP via 3 geographically distributed RRs
                    
         METRO AGGREGATION (per city)
         
         NYC-PE-01   NYC-PE-02   NYC-PE-03
              ↕            ↕           ↕
         NYC-AGG-01  NYC-AGG-02  NYC-AGG-03
              ↕            ↕           ↕
         NYC-ACCESS-01 ... NYC-ACCESS-N
         
         (replicated per major city, simplified per regional city)
```

**IS-IS design:**
- IS-IS Level 2 for the national backbone (P + backbone-facing PE interfaces)
- IS-IS Level 1 per metro area (AGG + access rings within each city)
- Level 1/2 boundary at metro AGG routers
- Area summarization at L1→L2 boundaries reduces backbone flooding

**BGP design:**
- 3–4 RRs with geographic diversity (NY, CHI, LA or similar)
- Unique cluster-ID per RR (ensures clients receive routes from all RRs independently; shared cluster-ID causes RRs to discard each other's reflected routes)
- ADD-PATH for ECMP and backup path distribution
- BGP-CT (Color-aware tunnels) or Flex-Algo for service-differentiated transport

**Internet peering:**
- Major IXPs in each core city (NYIIX, Any2, DE-CIX, etc.)
- Private peering with top CDNs (Netflix, Google, Akamai, Meta)
- 2–4 upstream transit providers for full BGP table
- RPKI ROV at all eBGP peers: `route-policy RPKI-INVALID drop if bgp-origin-as-validity invalid`

**Redundancy:**
- Full mesh between core PoPs (not just a ring) — any single PoP loss < 25ms reconvergence
- Metro AGG dual-homed to at least 2 core PoPs
- SR-TE with ODN (On-Demand Nexthop) for latency-sensitive services
- TI-LFA for sub-50ms FRR everywhere

### Topology 3: Tier-1 / Global Backbone

Tier-1 networks (AT&T, NTT, Deutsche Telekom, Telia) have no upstream transit peers — they carry the global internet themselves. Scale: 20,000+ nodes, full global BGP table, hundreds of IXP connections, all 6 populated continents.

The design principles are the same as Topology 2, but with additional complexity:

**Cross-continent MPLS:**
- Trans-Atlantic and Trans-Pacific submarine cable systems
- OTN/DWDM wavelength circuits leased or owned
- Optical path diversity mandatory (different cable systems for redundancy)
- MPLS LSPs span oceans: SRGB must accommodate all PoPs globally. Default SRGB is small (IOS-XR: 16000–23999 = 8K labels; Junos: 16000–16999 = 1K labels). Tier-1 networks expand to 16000–65535+ to cover 20K+ nodes with multiple SID planes (Flex-Algo). Plan the SRGB allocation scheme at design time.

**Multi-AS design:**
- Very large carriers operate multiple ASNs by region or subsidiary
- BGP confederations or route-leaking between ASNs
- Inter-AS MPLS: Option A (VRF-to-VRF at ASBR borders), Option B (eBGP redistribution of labeled VPN routes between ASBRs with next-hop-self and relabeling), or Option C (iBGP multihop for VPN + eBGP labeled-unicast for transport, most scalable for Tier-1)

**IXP strategy:**
- Connected to 50+ IXPs globally
- Route server selection: prefer full prefix acceptance from RS + bilateral policy
- BGP communities for traffic engineering across IXP (local pref adjustment for preferred entry)
- MANRS compliance: IRR filtering + RPKI on all peering sessions

**Operations at Tier-1 scale:**
- NOC: 24×7 with 100+ operators across multiple time zones
- Automation: zero-touch provisioning mandatory (thousands of new devices per year)
- Change management: peer review + automated pre-check + staged rollout
- Incident management: SEV1 bridge within 5 minutes, executive notification within 15

## Review Questions

These questions are IE-SP caliber — scenario-based, requiring you to apply multiple concepts.

**Q1.** A regional ISP has 800 routers in a single IS-IS Level 2 domain with 45,000 internal prefixes. During a backbone P-router software upgrade, the NOC observes BGP VPN reconvergence taking 4–6 minutes for some customer VRFs. IS-IS reconvergence itself completes in 400ms. What is the most likely cause, and what design change would address it?

> **Discussion:** BGP reconvergence is separated from IGP reconvergence. The IGP reconverges in 400ms (fast), but BGP VPN routes need to be individually reconverged when the BGP next-hop changes. Without PIC (Prefix Independent Convergence), each of the thousands of VPN prefixes pointing to the affected next-hop must be individually re-resolved — with 45K prefixes and RR best-path recalculation, this takes minutes. Solutions: (a) enable ADD-PATH so PIC can pre-install backup VPN routes (the backup next-hop is already in the FIB — switchover is immediate), (b) scale-up or add redundant RRs to reduce recalculation time, (c) use BGP GR (Graceful Restart) on the router being upgraded so BGP sessions don't drop in the first place. The IS-IS reconvergence being fast rules out IGP as the bottleneck — this is purely a BGP design issue. Note: on Junos, BGP route resolution is event-driven rather than timer-based; on IOS-XR, the BGP scanner interval (default 60s) adds additional delay.

**Q2.** You are designing a national carrier backbone for a new SP. The sales team is committing 99.999% uptime SLAs to enterprise customers. What is your failure recovery strategy, and specifically how do you justify the choice between RSVP-TE FRR and TI-LFA?

> **Discussion:** 99.999% = 5.26 minutes/year. You must achieve sub-50ms recovery for all backbone failures. Both RSVP-TE FRR and TI-LFA can meet the 50ms target. The choice: RSVP-TE FRR requires pre-signaled bypass LSPs. In a full-mesh PE-to-PE design, each P router transits O(N²) LSPs (where N = number of PEs), plus bypass tunnels for FRR. While modern platforms (ASR 9000, MX series) handle 50K–100K+ RSVP-TE LSPs, the operational complexity of managing this state — Path/Resv refresh, make-before-break, bandwidth bookkeeping — becomes the real scaling constraint. TI-LFA uses SR-MPLS with no backbone soft-state — backup paths computed locally per-prefix, no signaling overhead, linear scaling. For a greenfield national carrier, TI-LFA + SR-MPLS is the correct choice: less state, simpler operations, equivalent convergence, better scale. RSVP-FRR is retained only for legacy MPLS-TE deployments or specific use cases requiring bandwidth guarantees on protected paths.

**Q3.** A customer reports intermittent traffic blackholing in their L3VPN. The PE serving the customer has the VPN routes in the VRF. The remote PE also has the routes. Traffic flows from Site A to Site B but not from Site B to Site A. What is your diagnostic approach?

> **Discussion:** Asymmetric blackhole in L3VPN points to an asymmetry in one of: (a) the MPLS label path (Site B's PE can't reach Site A's PE label), (b) the VPNv4 route on Site B's PE has a BGP next-hop that isn't MPLS-reachable, or (c) there's a policy discrepancy (Route Target not imported on one side, or prefix filtered at CE). Diagnostic steps on Site B's PE: verify the VPNv4 route exists (Junos: `show route table bgp.l3vpn.0`; IOS-XR: `show bgp vpnv4 unicast`); check if it's installed in the VRF (Junos: `show route table VRF-NAME`; IOS-XR: `show route vrf VRF-NAME`); verify the transport LSP to Site A's PE loopback exists (Junos: `show mpls lsp`; IOS-XR: `show mpls forwarding`); trace the MPLS path (Junos: `traceroute mpls segment-routing`; IOS-XR: `traceroute mpls ipv4 [site-A-PE-loopback]`). The asymmetry often means the issue is only in the B→A direction at the MPLS transport level — a broken IGP route to Site A's PE loopback on the backbone, or a missing/withdrawn VPN label on one direction.

**Q4.** Your NOC receives an alarm: a backbone P router in the New York core has gone offline. Within 100ms, all backbone traffic around the failed router has reconverged via TI-LFA backup paths. Two minutes later, 15% of BGP VPN routes still show as unreachable on some PEs. What is happening and how do you resolve it?

> **Discussion:** TI-LFA handles the fast-path (sub-50ms switchover to backup). But the 15% of VPN routes still showing unreachable after 2 minutes indicates a BGP-layer issue. The most likely cause: some PE loopbacks whose shortest MPLS path traversed the failed P router now have a new (longer) path after IGP reconvergence. VPN routes whose BGP next-hop points to those PE loopbacks need the PE to resolve the new MPLS path. Without PIC, each affected VPN prefix must be individually reconverged — with thousands of prefixes, this takes minutes. Check: `show bgp vpnv4 unicast` for routes with unreachable or changed next-hops; verify IGP reachability to all PE loopbacks; confirm MPLS labels are installed for the new paths. Resolution: enable PIC + ADD-PATH so backup VPN paths are pre-installed. For planned maintenance, use IS-IS overload bit to drain traffic before the P router goes offline.

**Q5.** You are reviewing a proposed SP backbone design that uses OSPF for the IGP instead of IS-IS. The designer argues this is fine because both protocols support SR-MPLS and TI-LFA. What is your response?

> **Discussion:** The designer is technically correct that OSPF supports SR-MPLS and TI-LFA. The choice is not about protocol capability. Arguments for IS-IS over OSPF in SP context: (1) IS-IS has a richer operational history in SP environments — vendor implementations are more mature for SP-specific extensions like SR, BGP-LS, and large-scale deployment; (2) IS-IS L1/L2 hierarchy is cleaner for multi-area SP design than OSPF's multi-area (no stub area ambiguity, no Type 3/4/5 LSA flooding complexity); (3) IS-IS runs over CLNS which historically made it more flexible for non-IP links; (4) the industry conventions, the IE-SP exam expectations, and the interoperability testing were built around IS-IS for SP. If the SP is running a purely homogeneous Cisco IOS-XR network and the designer prefers OSPF, it's defensible. In a multi-vendor environment or greenfield national carrier, IS-IS is the correct choice.

## Key Terms Reference

| Term | Definition |
|---|---|
| **P router** | Provider core router; forwards labels only, no customer routes |
| **PE router** | Provider edge router; customer VRFs, MP-BGP, MPLS label push/pop |
| **CE router** | Customer edge; demarcation between customer and SP |
| **VRF** | Virtual Routing and Forwarding; per-customer isolated routing table |
| **RD** | Route Distinguisher; makes VPN prefixes globally unique in VPNv4/v6 |
| **RT** | Route Target; BGP extended community controlling VRF import/export |
| **SRGB** | Segment Routing Global Block; label range for SR-MPLS prefix SIDs |
| **TI-LFA** | Topology-Independent Loop-Free Alternates; SR-MPLS fast reroute |
| **BFD** | Bidirectional Forwarding Detection; fast link failure detection (3–10ms) |
| **RR** | Route Reflector; BGP scaling mechanism to avoid full iBGP mesh |
| **GR** | Graceful Restart; BGP mechanism to maintain forwarding during control plane restart |
| **FRR** | Fast Reroute; generic term for pre-computed backup path switchover |
| **NOC** | Network Operations Center; 24×7 SP operations team |
| **SLA** | Service Level Agreement; contractual performance guarantees with penalties |
| **PoP** | Point of Presence; a physical location with SP infrastructure |
| **MTTR** | Mean Time to Repair; average time to restore service after failure |
