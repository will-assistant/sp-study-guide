# 8.3 — EVPN-MPLS vs EVPN-VXLAN

## Overview

> EVPN is the control plane. The data plane is your choice — MPLS or VXLAN. This isn't a religious war; it's an architectural decision with real consequences. Service providers overwhelmingly run EVPN-MPLS because they already have an MPLS core and need deterministic transport with TE, QoS, and sub-50ms FRR. Data centers overwhelmingly run EVPN-VXLAN because they want IP-only underlay, ECMP everywhere, and couldn't care less about MPLS. The interesting question isn't which is "better" — it's what happens at the boundary, and why a growing number of operators run both. This section dissects the two data plane options, their trade-offs, and how to interconnect them when your DC fabric meets your WAN backbone.

## Theory

### The Data Plane Question

EVPN (RFC 7432) was designed data-plane-agnostic from day one. The BGP control plane advertises MAC/IP bindings, ESI membership, and multicast group assignments — none of which depend on how the actual Ethernet frames get tunneled between PEs. The original RFC uses MPLS labels. RFC 8365 formalized the VXLAN (and NVGRE/MPLS-in-GRE) overlay options.

Think of it like this: EVPN is the brain. MPLS and VXLAN are two different spinal cords connecting it to the body. Same intelligence, different wiring.

### EVPN-MPLS

#### How It Works

EVPN-MPLS uses the same MPLS data plane you already know from L3VPN and VPLS. Each PE allocates a local MPLS label per EVI (or per MAC, or per ES — depends on the label allocation model). The EVPN BGP update carries this label in the NLRI. Remote PEs encapsulate Ethernet frames with:

```
┌──────────────────────────────────────────────────────┐
│ Outer: Transport Label (IGP/LDP/SR-MPLS)            │
├──────────────────────────────────────────────────────┤
│ Inner: EVPN Service Label (from BGP NLRI)            │
├──────────────────────────────────────────────────────┤
│ Original Ethernet Frame (with or without VLAN tag)   │
└──────────────────────────────────────────────────────┘
```

The transport label gets the packet to the right PE (via LDP, RSVP-TE, or SR-MPLS). The service label identifies the EVI/bridge domain on that PE. Exactly the same two-label stack model used for L3VPN — swap the VRF for a bridge domain and you're done.

#### Label Allocation Models

This is where vendors diverge, and it matters for scale:

| Model | Label Granularity | Scale Impact | Use Case |
|-------|-------------------|-------------|----------|
| **Per-EVI** | One label per EVPN instance | Fewest labels; MAC lookup after decap | Default on most platforms; best for SP |
| **Per-MAC** | One label per MAC address | Massive label consumption; direct forwarding | Rarely used; theoretical optimization |
| **Per-ES/EVI** | One label per ES per EVI | Enables split-horizon without source MAC check | Required for all-active multi-homing on some platforms |

**Per-EVI** is the standard. The PE decapsulates the MPLS label, identifies the bridge domain, then does a MAC lookup to find the output interface. Simple, scalable, and what you'll configure 99% of the time.

**Per-ES/EVI** matters for multi-homing. When PE1 and PE2 are both all-active for the same ES, and PE1 receives a BUM frame from the CE, it floods to PE2 via the core. PE2 needs to know this frame came from a peer in the same ES so it doesn't send it back to the CE (split-horizon). A per-ES/EVI label lets PE2 make that determination without inspecting the source MAC.

#### Transport Options

EVPN-MPLS inherits all MPLS transport capabilities:

- **LDP:** Simple, auto-discovers, but no TE. Legacy but ubiquitous.
- **RSVP-TE:** Full traffic engineering, FRR, bandwidth reservations. Being sunset by SR.
- **SR-MPLS:** Best of both — TE via SR policies, TI-LFA for FRR, no protocol state per path. The future for most SP networks.
- **SRv6:** Uses IPv6 as transport instead of MPLS labels. EVPN over SRv6 is technically a third option, not MPLS, but the control plane is identical. See Module 6.

### EVPN-VXLAN

#### VXLAN Primer

If you've lived in the SP world, VXLAN might feel foreign. It's a DC overlay protocol (RFC 7348) that tunnels Layer 2 frames inside UDP/IP:

```
┌──────────────────────────────────────────────────────┐
│ Outer Ethernet Header (next-hop MAC)                 │
├──────────────────────────────────────────────────────┤
│ Outer IP Header (src=VTEP1, dst=VTEP2)              │
├──────────────────────────────────────────────────────┤
│ Outer UDP Header (src=hash, dst=4789)                │
├──────────────────────────────────────────────────────┤
│ VXLAN Header (VNI — 24-bit, ~16M segments)           │
├──────────────────────────────────────────────────────┤
│ Original Ethernet Frame                              │
└──────────────────────────────────────────────────────┘
```

**Key differences from MPLS encapsulation:**

- **IP underlay** — no label-switched paths, no MPLS infrastructure. Plain IP routing (usually OSPF or eBGP in a Clos fabric).
- **UDP source port hashing** — the outer UDP source port is a hash of the inner frame (5-tuple or src/dst MAC). This is how ECMP works — each flow gets a different UDP source port, and the IP fabric load-balances based on the 5-tuple of the outer header.
- **VNI = service identifier** — the 24-bit VXLAN Network Identifier replaces the MPLS service label. 16 million possible segments, and VNIs are **globally significant** (same VNI = same service everywhere). MPLS labels are **locally significant** (each node assigns its own labels independently) — so while there are ~1M labels per 20-bit space, that's per-node, not network-wide. Different architectural model.
- **VTEP (VXLAN Tunnel Endpoint)** — the equivalent of a PE in MPLS terminology. Where encap/decap happens.
- **UDP port 4789** — IANA standard per RFC 7348. Note: Linux kernel historically used **UDP 8472** before standardization. A mismatch between 4789 and 8472 causes silent black-holing — one of those "why does nothing work" traps that eats hours.

#### EVPN as VXLAN Control Plane

Original VXLAN (RFC 7348) used flood-and-learn — the same dumb approach VPLS used. The whole point of pairing EVPN with VXLAN (RFC 8365) is to replace that with BGP-based MAC/IP learning:

- **MAC advertisement:** Type 2 routes carry the VNI instead of an MPLS label
- **BUM handling:** Type 3 (IMET) routes use ingress replication lists (each VTEP listed by IP) instead of P2MP LSP or mLDP
- **ARP suppression:** Same as EVPN-MPLS — PE/VTEP answers ARP from its BGP-learned MAC/IP table
- **Multi-homing:** Same ESI mechanics, same DF election, same EAD routes

The control plane is **identical**. Both EVPN-MPLS and EVPN-VXLAN use PMSI Tunnel Type 6 (Ingress Replication) in the IMET (Type 3) route. The difference is in the **tunnel identifier** (MPLS label vs. VTEP IP address) and the **BGP Encapsulation Extended Community** (RFC 8365 §5.1), which tells the receiving PE whether to expect MPLS or VXLAN encapsulation.

### Head-to-Head Comparison

This is the table that matters. Print it, laminate it, tape it to your monitor.

| Criterion | EVPN-MPLS | EVPN-VXLAN |
|-----------|-----------|------------|
| **Underlay** | MPLS (LDP, RSVP-TE, SR-MPLS) | IP (OSPF, BGP, IS-IS) |
| **Encap overhead** | 8 bytes (2× 4-byte labels) | 36 bytes L3 (IP+UDP+VXLAN); 50 bytes on wire (+outer Ethernet) |
| **MTU impact** | Minimal (8 bytes + original) | Significant (36-byte L3 overhead; jumbo frames recommended) |
| **Service ID space** | 20-bit label (~1M) per node, locally significant | 24-bit VNI (~16M) globally significant |
| **ECMP** | Entropy label (RFC 6790) or FAT PW | Native — UDP src port hash; IP fabric does 5-tuple ECMP |
| **Traffic Engineering** | Full TE (RSVP-TE, SR-TE, Flex-Algo) | None inherent; relies on underlay ECMP |
| **QoS** | MPLS EXP bits (3-bit, 8 classes); well-understood SP QoS models | DSCP in outer IP header; re-marking by intermediate hops varies |
| **FRR** | Sub-50ms (TI-LFA, RSVP-TE FRR) | IP convergence (typically 1-5s without BFD tuning) |
| **OAM** | MPLS OAM (LSP ping, traceroute, BFD-for-MPLS) | IP tools (ping, traceroute); limited L2 OAM |
| **Multi-homing** | Full (ESI, DF election, aliasing) | Full (identical control plane) |
| **Hardware** | SP routers (ASR 9000, MX, 7750 SR) | DC switches (Nexus, QFX, 7220) |
| **Operational complexity** | Higher (MPLS + IGP + BGP + TE) | Lower (IP fabric + BGP) |
| **Typical deployment** | WAN, metro, backbone | Data center fabric, campus |
| **Interworking** | Needs EVPN GW for VXLAN DCs | Needs EVPN GW for MPLS WAN |

#### Encapsulation Overhead: Why It Matters

MPLS adds ~8 bytes (two 4-byte labels). VXLAN adds overhead at two levels: **36 bytes at L3** (20 IP + 8 UDP + 8 VXLAN header) and **50 bytes on the wire** (add 14-byte outer Ethernet header). The MTU that matters is the L3 MTU — a 1500-byte customer frame becomes a 1536-byte IP packet on the underlay, which must fit within the underlay's IP MTU. Vendor documentation often quotes "50 bytes" (the wire-level number), but for MTU calculations, think in terms of the 36-byte IP overhead.

In a DC with jumbo frames (9000+ MTU), nobody cares either way. On a WAN with 1500-byte L3 MTU links, that 36-byte tax means customer frames above ~1464 bytes get fragmented or dropped — and that's before you consider the inner Ethernet header. This is why SPs don't use VXLAN across the WAN.

#### ECMP: The DC Advantage

VXLAN's killer feature is effortless ECMP. The outer UDP source port is a flow-based hash, and every IP router in the path does standard 5-tuple load balancing. No entropy labels, no FAT pseudowires, no special config — it Just Works™.

MPLS ECMP requires either:
- **Entropy labels** (RFC 6790) — adds a hash label below the transport label; requires hardware support on every transit LSR
- **FAT pseudowires** (RFC 6391) — adds a flow label; same hardware dependency
- Deep packet inspection of the payload — vendor-specific, inconsistent, and terrible for labeled packets

In a DC Clos fabric with 128 equal-cost paths, VXLAN's ECMP model is simply better. In an SP backbone with 2-4 ECMP paths and SR-TE policies, MPLS ECMP is fine.

### Integrated Routing and Bridging (IRB) Differences

Both EVPN-MPLS and EVPN-VXLAN support IRB — combining L2 bridging and L3 routing in the same EVPN instance. The control plane (Type 2 with IP, Type 5 for IP prefixes) is identical. The implementation differs:

#### Symmetric vs. Asymmetric IRB

| IRB Model | How It Works | Best For |
|-----------|-------------|----------|
| **Asymmetric** | Ingress PE does L2→L3→L2 lookup (routes AND bridges); egress PE just bridges. Requires all VNIs/BDs on all PEs. | Small DC fabrics with few subnets |
| **Symmetric** | Ingress PE routes into a transit VNI/label (L3VPN style); egress PE routes out. Each PE only needs local VNIs. | Large scale; only way that works for EVPN-MPLS SP networks |

**EVPN-MPLS almost always uses symmetric IRB.** The PE routes the frame into a VRF (Type 5 route, IP prefix), encapsulates with a VRF label, and the remote PE routes it out. This is identical to L3VPN — which is the whole point. One control plane, two services.

**EVPN-VXLAN supports both**, but symmetric IRB with a transit L3VNI (VNI for the tenant VRF) is the standard in modern DC fabrics (Cisco VXLAN EVPN with "vrf overlay" design).

### When to Use Which

**Use EVPN-MPLS when:**
- You have an existing MPLS backbone (why rip it out?)
- You need traffic engineering (bandwidth guarantees, path constraints)
- You need deterministic sub-50ms FRR (TI-LFA, node-protect)
- You need strict QoS with MPLS EXP (voice, video, SLA-bound services)
- You're providing managed L2VPN services to customers (VPWS/VPLS replacement)
- Your WAN MTU is 1500 and you can't afford VXLAN overhead
- You're a service provider. Period.

**Use EVPN-VXLAN when:**
- You're building a new DC fabric (greenfield, IP-only underlay)
- You need massive L2/L3 multi-tenancy (16M VNIs > 1M labels)
- You want native ECMP without entropy labels
- You're running leaf-spine Clos with commodity switches
- Your team knows IP but not MPLS (operational simplicity wins)
- You're a cloud provider or enterprise DC team

**Use both when:**
- Your DC fabric is VXLAN and your WAN is MPLS (this is most large operators)
- DCI (Data Center Interconnect) — you need a gateway between the two worlds
- You're migrating from VPLS to EVPN and your DCs are already VXLAN

## Configuration

### EVPN-MPLS (IOS-XR)

```cisco
!! EVPN-MPLS configuration on IOS-XR PE
!! Uses SR-MPLS transport, per-EVI label allocation

! Enable EVPN globally
evpn
 evi 100
  bgp
   route-distinguisher auto
   route-target import 65000:100
   route-target export 65000:100
  !
  advertise-mac
  !
 !
!

! Bridge domain bound to EVPN EVI
l2vpn
 bridge group CUST-A
  bridge-domain BD-100
   interface GigabitEthernet0/0/0/1.100
   !
   evi 100
   !
  !
 !
!

! CE-facing sub-interface
interface GigabitEthernet0/0/0/1.100 l2transport
 encapsulation dot1q 100
!

! BGP EVPN address family
router bgp 65000
 address-family l2vpn evpn
 !
 neighbor 10.0.0.1
  remote-as 65000
  update-source Loopback0
  address-family l2vpn evpn
   route-reflector-client  ! if this PE is a RR
  !
 !
!
```

**Key points:**
- `evi 100` with `route-distinguisher auto` auto-derives RD from loopback + EVI ID (or set explicitly)
- `advertise-mac` enables MAC advertisement (Type 2 routes) — off by default on IOS-XR
- The bridge domain binds the CE-facing interface to the EVI
- Transport is whatever IGP + label distribution you've already configured (LDP, SR-MPLS, etc.)
- No tunnel configuration needed — EVPN-MPLS reuses the existing MPLS infrastructure

### EVPN-MPLS (Junos)

```junos
## EVPN-MPLS configuration on Junos PE
## Uses LDP or RSVP transport

# EVPN instance (routing instance type evpn for MPLS)
routing-instances {
    CUST-A {
        instance-type evpn;
        vlan-id 100;
        interface ge-0/0/1.100;
        route-distinguisher 10.0.0.2:100;
        vrf-target target:65000:100;
        protocols {
            evpn;
        }
    }
}

# CE-facing interface
interfaces {
    ge-0/0/1 {
        unit 100 {
            encapsulation vlan-bridge;
            vlan-id 100;
        }
    }
}

# BGP EVPN family
protocols {
    bgp {
        group EVPN-RR {
            type internal;
            local-address 10.0.0.2;
            family evpn {
                signaling;
            }
            neighbor 10.0.0.1;
        }
    }
}
```

**Junos note:** Instance type `evpn` is for MPLS data plane. Instance type `virtual-switch` is for VLAN-aware bundling with MPLS. Don't confuse with `mac-vrf` which is for EVPN-VXLAN on some platforms (though Junos uses `evpn` for both in newer releases — check your version).

### EVPN-VXLAN (IOS-XR)

IOS-XR is primarily an SP platform. EVPN-VXLAN config on IOS-XR exists (NCS 5500/8000 series) but it's less common than NX-OS for VXLAN. On IOS-XR, VXLAN VNI mapping is done under the `l2vpn bridge-domain evi` stanza — there is no separate `interface nve` with `member vni` like NX-OS:

```cisco
!! EVPN-VXLAN on IOS-XR (NCS 5500 / 8000 series)

evpn
 evi 100
  bgp
   route-distinguisher auto
   route-target import 65000:100
   route-target export 65000:100
  !
  advertise-mac
  !
 !
!

l2vpn
 bridge group CUST-A
  bridge-domain BD-100
   interface GigabitEthernet0/0/0/1.100
   !
   evi 100
   !
   vni 10100
   !
  !
 !
!

! NVE source interface (VTEP IP)
interface nve1
 source-interface Loopback0
 overlay-encapsulation vxlan
!
```

**Key difference from MPLS config:** The `vni 10100` under the bridge domain maps it to a VXLAN VNI, and `interface nve1` (no space) creates the VXLAN tunnel endpoint. Note: IOS-XR's VXLAN config model differs significantly from NX-OS — don't copy NX-OS syntax (`member vni`, `ingress-replication bgp`) into IOS-XR configs.

### EVPN-VXLAN (Junos — QFX/DC Focus)

```junos
## EVPN-VXLAN on Junos (QFX series leaf switch)
## Typical DC leaf-spine deployment

# VXLAN tunnel interface
interfaces {
    lo0 {
        unit 0 {
            family inet {
                address 10.0.0.3/32;
            }
        }
    }
}

# Overlay routing instance
routing-instances {
    EVPN-VXLAN-100 {
        instance-type mac-vrf;
        protocols {
            evpn {
                encapsulation vxlan;
                default-gateway do-not-advertise;
            }
        }
        vtep-source-interface lo0.0;
        service-type vlan-aware;
        interface ge-0/0/1.100;
        route-distinguisher 10.0.0.3:100;
        vrf-target target:65000:100;
        vlans {
            VLAN100 {
                vlan-id 100;
                vxlan {
                    vni 10100;
                    ingress-node-replication;
                }
            }
        }
    }
}

# BGP EVPN peering (typically eBGP to spine in DC)
protocols {
    bgp {
        group EVPN-SPINE {
            type external;
            multihop;
            local-address 10.0.0.3;
            family evpn {
                signaling;
            }
            peer-as 65100;
            neighbor 10.0.1.1;
            neighbor 10.0.1.2;
        }
    }
}
```

**Junos VXLAN notes:**
- `instance-type mac-vrf` is the VXLAN-native instance type (vs. `evpn` for MPLS)
- `encapsulation vxlan` explicitly selects VXLAN data plane
- `vtep-source-interface` sets the VTEP IP (like NVE source on IOS-XR/NX-OS)
- `ingress-node-replication` = ingress replication for BUM (no multicast underlay needed)
- DC fabrics typically use eBGP (each leaf/spine has its own ASN) vs. iBGP with RR in SP

### Key Knobs

| Parameter | EVPN-MPLS | EVPN-VXLAN | Notes |
|-----------|-----------|------------|-------|
| Service ID | MPLS label (auto-allocated) | VNI (manually mapped or auto) | VNI must match across all VTEPs for same service |
| Label/VNI allocation | Per-EVI (default), per-MAC, per-ES | Per-VNI (always) | VXLAN is always per-VNI by design |
| BUM replication | Ingress replication (default) or mLDP P2MP | Ingress replication (default) or multicast underlay | Ingress replication is standard for both |
| MTU | No special requirement | Underlay MTU ≥ original + 50 bytes | Set jumbo frames (9216) on all underlay interfaces |
| VTEP/PE source IP | Loopback (for BGP next-hop) | Loopback (for VTEP IP) | Must be reachable in the underlay; often the same loopback |
| ARP suppression | Supported | Supported | Enable to reduce BUM flooding significantly |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| EVPN routes received but no MAC entries in bridge domain | Encap mismatch — one PE advertising MPLS, other expecting VXLAN | Verify `show evpn evi detail` — Encap must match on all PEs for same EVI |
| VXLAN traffic black-holed | Underlay MTU too small; VXLAN frames fragmented or dropped | Set underlay MTU ≥ 9216 on all links; verify with `ping size 9000 df-bit` |
| Remote MACs learned but traffic fails (MPLS) | Transport label not programmed; LDP/SR adjacency down | Check `show mpls forwarding` for valid label to remote PE loopback |
| Remote MACs learned but traffic fails (VXLAN) | VTEP source IP unreachable; NVE interface down | Verify `show nve peers`; ping remote VTEP IP from source loopback |
| Partial MAC learning — some MACs missing | Route target mismatch between PEs | Verify RT import/export matches on all PEs for the same EVI |
| BUM flooding not working (VXLAN) | Ingress replication list incomplete; Type 3 routes not exchanged | `show evpn evi inclusive-multicast` — verify all VTEPs listed |
| BUM flooding not working (MPLS) | Ingress replication list missing PEs; P2MP LSP not up | `show evpn evi inclusive-multicast` — verify all PEs listed with correct labels |
| ECMP not load-balancing (MPLS) | No entropy label; transit LSRs hashing on top label only | Enable `entropy-label` or FAT PW; verify transit LSRs support it |
| ECMP not load-balancing (VXLAN) | Underlay not hashing on UDP source port | Verify underlay routers include UDP src port in ECMP hash (`port-channel load-balance` or equivalent) |

### Methodology

1. **Identify the data plane** — `show evpn evi detail` → Encap: MPLS or VXLAN? Start here. Everything else branches.

2. **Control plane sanity** — `show bgp l2vpn evpn summary` → are all EVPN peers established? Are routes being received?

3. **Route type check** — `show bgp l2vpn evpn route-type 2` → are MAC/IP routes present for the remote site? Check RD, RT, next-hop.

4. **Encap consistency** — every PE in the same EVI must use the same encapsulation. You cannot mix MPLS and VXLAN within a single EVI without a gateway.

5. **Transport verification:**
   - MPLS: `show mpls forwarding` — valid label path to remote PE?
   - VXLAN: `ping <remote VTEP IP> source <local VTEP IP>` — reachable?

6. **MTU check (VXLAN critical):**
   ```
   ping <remote VTEP IP> size 8950 df-bit source <loopback>
   ```
   If this fails, your underlay MTU is too small. VXLAN adds 50 bytes — a 1500-byte customer frame becomes 1550 on the wire.

7. **Data plane forwarding** — inject a known frame, watch for it with capture/mirror on the remote PE. If control plane is clean and forwarding fails, it's usually MTU, ACLs, or hardware programming.

### War Stories

> **The VXLAN MTU Cliff**
>
> An operator migrated a DC fabric from VPLS to EVPN-VXLAN. Tests passed beautifully — ping worked, SSH worked, web traffic worked. Then they migrated the backup servers. NFS broke. iSCSI broke. Database replication broke.
>
> Every protocol that sent jumbo frames or even near-1500-byte frames was failing silently. The underlying issue: the leaf-spine fabric had been configured with 1500 MTU on inter-switch links. VPLS over MPLS added 8 bytes (two labels) — 1508 worked fine because every switch silently forwarded slightly oversized frames. VXLAN added 50 bytes — 1550 was over the threshold where switches would either fragment (breaking performance) or drop with DF-bit set (breaking connectivity).
>
> The fix was trivial: set all fabric links to 9216 MTU. The lesson was expensive: EVPN migration isn't just a control plane swap. If you change the data plane encapsulation, you change the MTU math. And unlike MPLS, where the overhead is small enough to hide, VXLAN's 50-byte tax is the first thing that bites you.
>
> The red flag they missed: nobody tested with `ping size 1500 df-bit` through the VXLAN fabric. That one command would have caught it in five minutes.

## Interworking: The Gateway Problem

### DCI with Mixed Data Planes

The real world isn't "MPLS vs. VXLAN" — it's "MPLS AND VXLAN." Your WAN is MPLS. Your DCs are VXLAN. You need a gateway.

```
┌──────────────┐        EVPN-MPLS WAN        ┌──────────────┐
│   DC-West    │←──────────────────────────→ │   DC-East    │
│  EVPN-VXLAN  │     ┌──────────────┐        │  EVPN-VXLAN  │
│              │     │   GW Router   │        │              │
│  ┌────────┐  │     │ MPLS ↔ VXLAN │        │  ┌────────┐  │
│  │  Leaf  │──┼─────┤  re-encap    ├────────┼──│  Leaf  │  │
│  └────────┘  │     │              │        │  └────────┘  │
└──────────────┘     └──────────────┘        └──────────────┘
```

**The gateway router** (sometimes called a "border leaf" or "DCI gateway") participates in both EVPN-MPLS (WAN-facing) and EVPN-VXLAN (DC-facing). It:

1. Receives Type 2 MAC/IP routes from the DC fabric (VXLAN encap)
2. Re-advertises them into the WAN with MPLS encap (and vice versa)
3. Performs data plane re-encapsulation: strips VXLAN headers, adds MPLS labels (and vice versa)

**Platforms that do this natively:**
- Cisco ASR 9000 / NCS 5500 (IOS-XR) — can terminate both MPLS and VXLAN EVIs and stitch them
- Juniper MX series — supports EVPN-MPLS and EVPN-VXLAN simultaneously
- Nokia 7750 SR — EVPN-MPLS native, VXLAN gateway support

**Configuration pattern (IOS-XR gateway):**

```cisco
!! DCI Gateway — stitching EVPN-VXLAN (DC) to EVPN-MPLS (WAN)
!! Two EVIs (different encapsulations) in ONE bridge domain

! DC-facing EVPN (VXLAN encap)
evpn
 evi 100
  bgp
   route-distinguisher auto
   route-target import 65000:100
   route-target export 65000:100
  !
  advertise-mac
 !
 ! WAN-facing EVPN (MPLS encap)
 evi 200
  bgp
   route-distinguisher auto
   route-target import 65000:200
   route-target export 65000:200
  !
  advertise-mac
 !
!

l2vpn
 bridge group DCI
  bridge-domain BD-100
   ! DC-facing (VXLAN)
   evi 100
   !
   vni 10100
   !
   ! WAN-facing (MPLS) — same bridge domain, different EVI/encap
   evi 200
   !
  !
 !
!

! NVE for VXLAN side
interface nve1
 source-interface Loopback0
 overlay-encapsulation vxlan
!
```

**The trick:** Two different EVIs (100 for VXLAN, 200 for MPLS) land in the same bridge domain. The gateway learns MACs from the DC (via EVI 100/VXLAN) and re-originates them toward the WAN (via EVI 200/MPLS), and vice versa. From the bridge domain's perspective, it's just two access paths — one tunneled via VXLAN, one via MPLS. Note the **different RTs** — the DC uses 65000:100, the WAN uses 65000:200 — so each side only imports routes from its own domain, and the gateway stitches them through the shared bridge domain.

**Gotchas:**
- **RT stitching:** The DC-facing and WAN-facing EVIs may use different RTs. The gateway imports one and exports the other.
- **Next-hop handling:** The gateway becomes the next-hop for re-originated routes. Remote PEs see the gateway's loopback, not the originating leaf.
- **MAC scale:** The gateway sees every MAC from both sides. Size your MAC table accordingly.
- **BUM replication doubling:** BUM frames from the DC get replicated to all WAN PEs, and vice versa. The gateway is the replication point — it needs the bandwidth.

### SRv6 as a Unifier

There's a third path emerging: run EVPN over SRv6 end-to-end. Since SRv6 uses IPv6 as transport, it works across both DC and WAN without MPLS infrastructure. The VPN SID replaces both the MPLS service label and the VXLAN VNI. No gateway needed — one encapsulation everywhere.

This is still early (2026), but it's where Cisco and Juniper are pushing. See Section 6.4 for SRv6 details.
