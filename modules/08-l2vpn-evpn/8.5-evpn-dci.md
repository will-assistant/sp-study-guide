# 8.5 — EVPN for Data Center Interconnect (DCI)

## Overview

> Data Center Interconnect is where EVPN stops being a single-site protocol and becomes a network architecture. Every operator eventually needs to stretch services between sites — disaster recovery, workload mobility, distributed applications, regulatory compliance — and every operator eventually discovers that naively stretching Layer 2 across a WAN is how you turn a single-site outage into a multi-site catastrophe. EVPN-based DCI solves this by providing control-plane intelligence at the boundary: MAC mobility tracking, ARP suppression to reduce cross-WAN flooding, multi-homed gateway redundancy, and — critically — the ability to do inter-subnet routing at the optimal site rather than tromboning traffic across a 50ms latency WAN link. This section covers the gateway models, topology patterns, configuration, and operational realities of EVPN DCI. If you've been building toward this since Module 4, this is where it pays off.

## Theory

### Why DCI Is Hard

Stretching Layer 2 across sites sounds simple. Two bridge domains, a tunnel, done. Here's what actually happens:

1. **BUM flooding scales with sites** — Every unknown unicast, broadcast, and multicast frame now crosses the WAN. ARP storms in one DC become ARP storms in all DCs.
2. **MAC table explosion** — Every border gateway must hold MACs from every site. At 100K MACs per DC across 4 sites, that's 400K entries on a device that also does WAN routing.
3. **Failure blast radius** — A spanning-tree loop in Site A takes down Sites B, C, and D. Layer 2 failures propagate at wire speed.
4. **Traffic tromboning** — Host in Site A talks to its default gateway in Site B because the gateway hasn't moved with the workload. Traffic crosses the WAN twice for every packet.
5. **Split-brain** — WAN link goes down. Both sites think they're authoritative. Both issue GARPs. When the link comes back, MAC flapping ensues.

EVPN addresses each of these with specific mechanisms:

| Problem | EVPN Solution | Mechanism |
|---------|--------------|-----------|
| BUM flooding | ARP/ND suppression + ingress replication lists | Proxy-ARP from local RT-2 cache; selective BUM replication per site |
| MAC table explosion | RT-constrain (RFC 4684) | Border gateways only import RTs they need |
| Failure blast radius | Per-site EVPN domains with gateway re-origination | Faults don't propagate BGP state across domain boundaries |
| Traffic tromboning | Distributed anycast gateway (IRB) | Same gateway IP/MAC active at every site |
| Split-brain | MAC mobility sequence numbers + site-of-origin | Higher sequence number wins; SoO prevents re-advertisement loops |

### DCI Topology Models

There are three fundamental approaches, and your choice cascades into every design decision that follows.

#### Model 1: Unified EVPN Fabric (Single Domain)

```
  Site A                              Site B
┌──────────┐     DCI (MPLS/VXLAN)   ┌──────────┐
│  Leaf-1  │◄──────────────────────►│  Leaf-3  │
│  Leaf-2  │◄──────────────────────►│  Leaf-4  │
│  Spine-1 │                        │  Spine-2 │
└──────────┘                        └──────────┘
         Single BGP EVPN domain
```

All PEs participate in one EVPN control plane. Every leaf at every site sees every MAC via BGP. The DCI link is just another fabric link.

**Pros:**
- Simplest configuration — no gateway translation
- Optimal forwarding — any leaf can route to any MAC directly
- Single EVPN instance for each service

**Cons:**
- Full mesh of BGP sessions (or RR per-site with inter-site RR peering)
- MAC tables on every leaf hold MACs from all sites
- No fault isolation — BGP churn in Site A hits Site B
- Doesn't scale beyond 2-3 sites with moderate MAC counts
- WAN must carry all EVPN routes (RT-1, RT-2, RT-3, RT-4, RT-5)

**When to use:** Small deployments (2 sites, <50K total MACs), same administrative domain, low-latency DCI (dark fiber, <5ms RTT).

#### Model 2: EVPN DCI Gateway (Multi-Domain)

```
  Site A                                      Site B
┌──────────┐                                ┌──────────┐
│ Leaf-1/2 │     Site A       Site B        │ Leaf-3/4 │
│          │    Border        Border        │          │
│ Spine-1  │◄──►GW-A1/A2 ════ GW-B1/B2◄───►│ Spine-2  │
└──────────┘    (re-orig)  DCI (re-orig)    └──────────┘
     EVPN          │      MPLS/SR      │         EVPN
   Domain A        └──────────────────┘       Domain B
                    EVPN Domain DCI
```

Each site runs its own EVPN domain. Dedicated border gateways (BGWs) sit at the boundary, participating in both the local site EVPN and the DCI EVPN. They **re-originate** routes between domains — stripping site-local information and injecting DCI-appropriate attributes.

**Pros:**
- Fault isolation — local BGP churn stays local
- Scale — leafs only see local MACs + summarized remote routes
- Independent upgrade/maintenance per site
- Flexibility — sites can run different data planes (VXLAN intra-site, MPLS inter-site)

**Cons:**
- Border gateways are critical path — must be multi-homed (ES between BGWs)
- Route re-origination adds complexity (RT rewrite, label reallocation)
- Slightly suboptimal forwarding (extra hop through BGW)
- BGW must hold MACs from both domains

**When to use:** Most production DCI deployments. 2-8 sites. Mixed fabrics. Separate operational teams per site.

#### Model 3: EVPN Multi-Site (RFC 9014)

```
  Site A                                       Site B
┌──────────┐                                 ┌──────────┐
│ Leaf-1/2 │    BGW-A         BGW-B          │ Leaf-3/4 │
│ Spine-1  │◄──►(VIP) ═══════(VIP)◄─────────►│ Spine-2  │
└──────────┘      │    DCI Overlay    │      └──────────┘
                  └───────────────────┘
          Uses VIP (Virtual IP) for BGW anycast
          Type-5 routes for inter-subnet DCI
          MAC summarization at BGW
```

An evolution of Model 2 formalized in RFC 9014 (originally from the VXLAN EVPN world). BGWs advertise themselves using a **virtual IP (VIP)** — all BGWs at a site share the same VIP. The DCI overlay uses this VIP as the tunnel endpoint. This means:

- Remote sites don't need to know how many BGWs exist at a site — they tunnel to the VIP
- BGW failover is seamless — other BGWs at the same site continue with the same VIP
- MAC summarization is natural — the BGW can advertise Type-5 (IP prefix) routes toward the WAN instead of individual Type-2 (MAC/IP) routes

**Pros:**
- All Model 2 benefits plus:
- BGW scale-out — add more BGWs without reconfiguring remote sites
- MAC summarization reduces DCI control plane by 10-100x
- VIP-based tunneling simplifies remote PE configuration
- Natively supports asymmetric IRB at the BGW

**Cons:**
- More complex initial setup (VIP allocation, VTEP anycast)
- Type-5 summarization loses per-MAC granularity (affects traffic engineering)
- Vendor support varies (strongest on NX-OS, growing on IOS-XR and Junos)
- RFC 9014 is relatively new — operational tooling still catching up

**When to use:** Large-scale DCI (4+ sites, 100K+ MACs), VXLAN-based fabrics, need for MAC summarization.

### DCI Gateway Functions

The border gateway is the linchpin of DCI. Regardless of model, it performs these functions:

#### Route Re-origination

The BGW receives EVPN routes from the local site and re-advertises them toward the DCI/WAN with modified attributes:

| Attribute | Local Site | DCI/WAN |
|-----------|-----------|---------|
| Next-hop | Original PE address | BGW address (or VIP) |
| MPLS label | Local PE's label | New label allocated by BGW |
| Route target | Site-local RT | DCI RT (may differ) |
| RD | Original PE's RD | BGW's RD |
| ESI | Preserved if same ES spans sites | Typically reset (ESI is site-local) |

This re-origination creates a **clean boundary** — the WAN never sees intra-site topology. Remote sites don't know (or care) how many leafs exist at Site A.

#### Data Plane Stitching

When the local fabric runs VXLAN and the DCI runs MPLS (or vice versa), the BGW performs encapsulation translation:

```
Inbound from local fabric:
┌──────────────────────────┐
│ Outer IP + UDP + VXLAN   │ ← Site-local VXLAN
├──────────────────────────┤
│ Original Ethernet Frame  │
└──────────────────────────┘
         │
         ▼ BGW decaps VXLAN, encaps MPLS
┌──────────────────────────┐
│ Transport Label (SR/LDP) │ ← DCI MPLS tunnel
├──────────────────────────┤
│ EVPN Service Label       │
├──────────────────────────┤
│ Original Ethernet Frame  │
└──────────────────────────┘
```

This stitching happens at line rate on modern hardware (Memory-to-memory rewrite, no CPU punt).

#### Distributed Anycast Gateway

The BGW can host a distributed anycast gateway — the same virtual IP and virtual MAC appear at every site. When a host in Site A sends traffic to its default gateway:

1. The local BGW (or leaf, in Model 1) performs L3 lookup in the VRF
2. If the destination is in the same subnet at Site B → encapsulate and send as L2 over DCI
3. If the destination is in a different subnet → route in the VRF, forward to the appropriate next-hop

This eliminates tromboning. The host's default gateway is always local. Traffic only crosses the WAN when the destination is actually remote.

**Anycast Gateway Configuration:**

Every site's gateway interface uses:
- Same IP address (e.g., 10.0.1.1/24)
- Same virtual MAC (e.g., 00:00:5e:00:01:01 or a configured static MAC)
- Different RD (per-PE, as always)
- Same RT (so all sites import the same routes)

The virtual MAC is critical — if each site used its real MAC as the gateway, hosts that move between sites would need to re-ARP. With a consistent virtual MAC, the host's ARP cache is valid at any site.

#### ARP/ND Suppression at the BGW

The BGW maintains a local ARP/ND cache built from EVPN RT-2 routes (which include IP in the MAC/IP advertisement). When a host in Site A ARPs for a host in Site B:

1. ARP request hits the local BGW
2. BGW checks its EVPN-learned ARP cache
3. If the binding exists → BGW responds directly with a proxy ARP reply
4. ARP request **never crosses the WAN**

This single mechanism can reduce DCI BUM traffic by 80-95% in typical enterprise/cloud environments.

### Inter-Subnet DCI with Type-5 Routes

Type-5 (IP Prefix) routes are the key to scalable DCI. Instead of advertising every individual MAC/IP across the WAN, the BGW can advertise IP prefixes:

```
Site A leafs advertise to BGW-A:
  RT-2: MAC aa:bb:cc:dd:ee:01, IP 10.0.1.10 (host route)
  RT-2: MAC aa:bb:cc:dd:ee:02, IP 10.0.1.11 (host route)
  RT-2: MAC aa:bb:cc:dd:ee:03, IP 10.0.1.12 (host route)
  ... (10,000 more)

BGW-A advertises across DCI to BGW-B:
  RT-5: 10.0.1.0/24 (one prefix, covers all 10,000+ hosts)
```

**10,000 RT-2 routes become 1 RT-5 route.** That's the power of summarization.

#### When Type-5 Works (and When It Doesn't)

| Scenario | Type-5 Suitable? | Why |
|----------|-----------------|-----|
| Inter-subnet routing between sites | ✅ Yes | Destination is in a different subnet — IP prefix is sufficient |
| L2 stretch (same subnet, both sites) | ❌ No | Need MAC for L2 forwarding — RT-2 required |
| VM mobility (live migration) | ❌ No | Specific MAC must move between sites — RT-2 with mobility seq |
| Disaster recovery (cold standby) | ✅ Yes | Standby site only needs IP reachability, not L2 adjacency |
| Firewall service chaining | ⚠️ Depends | If firewall expects L2 adjacency → no. If routed → yes |

The practical design: **Use RT-5 across the DCI for inter-subnet traffic, and RT-2 only for services that genuinely require L2 stretch.** Most operators find that 90% of their DCI traffic can be handled by Type-5 once they stop reflexively stretching VLANs.

### MAC Mobility and Site-of-Origin

When a workload moves between sites (VM migration, DR failover), the EVPN MAC Mobility extended community tracks the move:

1. Host H1 (MAC M1) is at Site A. Site A's PE advertises RT-2 for M1 with sequence number 0.
2. H1 migrates to Site B. Site B's PE detects M1 locally, advertises RT-2 with **sequence number 1**.
3. All PEs receive both advertisements. **Higher sequence number wins.** Site A's old route is implicitly withdrawn.
4. Site A's PE sees the higher-seq remote advertisement, withdraws its local route.

**Site-of-Origin (SoO):**

The BGW stamps a **Site-of-Origin** extended community on routes re-originated toward the DCI:

```
BGW-A re-originates routes with SoO: 65000:100  (Site A)
BGW-B re-originates routes with SoO: 65000:200  (Site B)
```

When BGW-A receives a route from the DCI with SoO 65000:100, it **drops it** — this route originated from Site A, went to Site B, and is now being reflected back. Without SoO, you get routing loops. With SoO, routes flow cleanly.

**SoO + MAC Mobility interaction:**

When H1 moves from Site A to Site B:
1. BGW-B advertises H1 with SoO 65000:200, seq 1
2. BGW-A receives it, SoO is 65000:200 (not 100), so it's accepted
3. BGW-A withdraws its own Site A route for H1
4. Remote sites (C, D) now route H1 traffic to Site B

If the WAN link fails during a move, both sites may claim the MAC. When the link recovers, the higher sequence number resolves the conflict. If sequence numbers are equal (simultaneous GARPs), the route from the PE with the **numerically higher IP** wins (RFC 7432 §15.1 tiebreaker).

### BUM Traffic Handling Across DCI

BUM traffic over the DCI is the single biggest performance concern. Strategies:

#### Ingress Replication (Head-End Replication)

Each BGW maintains a list of remote BGWs and sends a copy of each BUM frame to each one:

```
BUM frame at Site A:
  → Copy 1: to BGW-B (Site B)
  → Copy 2: to BGW-C (Site C)
  → Copy 3: to BGW-D (Site D)
```

**Bandwidth impact:** A single broadcast frame generates N-1 copies across the WAN (where N = number of sites). At 4 sites with 10Gbps WAN links, a broadcast storm in one site can saturate 30Gbps of DCI bandwidth.

**Mitigation:** ARP suppression (eliminates 80-95% of broadcast), storm control on BGW DCI-facing interfaces, and EVPN per-EVI BUM rate limiting.

#### Multicast Underlay (P-tunnel)

For large-scale deployments (8+ sites), use multicast in the WAN core:

```
BUM frame at Site A:
  → BGW-A encapsulates into multicast group 239.1.1.1
  → WAN core replicates via PIM-SSM
  → Each remote BGW receives one copy
```

The EVPN PMSI (Provider Multicast Service Interface) Tunnel attribute in the RT-3 (Inclusive Multicast) route signals which multicast group to use per EVI.

**Trade-off:** Multicast underlay is more WAN-efficient (one copy per link, not per site) but requires PIM in the WAN core — which many operators have spent years trying to eliminate.

#### Selective BUM Replication

Only replicate BUM frames to sites that have active hosts in the relevant EVI. If Site C has no hosts in VLAN 100, don't send VLAN 100 BUM there. EVPN knows this from the RT-3 (Inclusive Multicast) routes — only sites that advertise an RT-3 for a given EVI are included in the replication list.

### Convergence and Failure Scenarios

#### BGW Failure

Single BGW fails at Site A (Site A has BGW-A1 and BGW-A2 in multi-homing):

1. BGW-A1 goes down. ES routes (RT-1, RT-4) are withdrawn.
2. Remote BGWs remove BGW-A1 from ECMP next-hop group.
3. All traffic to Site A now goes via BGW-A2.
4. **Convergence time:** BGP withdrawal propagation (~1-3s) or BFD-triggered fast failover (~50-200ms with BFD on the DCI tunnel).

#### DCI Link Failure

The WAN link between sites goes down:

1. BGW detects link loss (physical down or BFD timeout).
2. BGW withdraws all DCI-facing EVPN routes.
3. Remote sites remove the affected BGW from their tables.
4. If redundant DCI paths exist (via Site C), traffic reconverges through alternate site.
5. If no alternate path, sites are isolated — local traffic continues, inter-site traffic drops.

**Critical design point:** Always have at least two independent DCI paths. Two diverse fiber routes, or a primary DCI + backup over an internet VPN.

#### Split-Brain

Both DCI links fail. Sites A and B are isolated. Both continue to operate locally. When connectivity restores:

1. BGP sessions re-establish.
2. MAC routes are exchanged. Conflicting MACs (same MAC learned locally at both sites during the split) are resolved via MAC Mobility sequence numbers.
3. The PE with the higher sequence number wins.
4. GARPs are generated for moved/conflicting MACs.
5. **Recovery time:** BGP reconvergence + GARP propagation (~5-30s depending on MAC count).

**Prevention:** Out-of-band heartbeat link between sites (management network) to detect split-brain and suppress conflicting advertisements from the standby site.

## Configuration

### Model 2: EVPN DCI Gateway (IOS-XR)

This is the most common production DCI topology. Site A runs EVPN-VXLAN internally, DCI runs EVPN-MPLS over SR.

#### BGW-A1 Configuration (IOS-XR 7.x)

```cisco
!! --- Loopback for BGP peering and MPLS transport ---
interface Loopback0
 ipv4 address 10.255.0.1/32
 description BGW-A1 Router-ID
!
!! --- DCI-facing interface ---
interface HundredGigE0/0/0/0
 description DCI to BGW-B1
 ipv4 address 10.100.0.0/31
 load-interval 30
!
!! --- SR-MPLS transport for DCI ---
router isis DCI
 is-type level-2-only
 net 49.0001.0102.5500.0001.00
 address-family ipv4 unicast
  metric-style wide
  segment-routing mpls
 !
 interface Loopback0
  passive
  address-family ipv4 unicast
   prefix-sid index 1
  !
 !
 interface HundredGigE0/0/0/0
  point-to-point
  address-family ipv4 unicast
   metric 10
  !
 !
!
!! --- BGP: DCI EVPN peering with remote BGW ---
router bgp 65000
 bgp router-id 10.255.0.1
 !
 !! Site-local EVPN (peering with local spines/RR)
 neighbor 10.254.0.1
  remote-as 65001
  description Site-A-RR
  update-source Loopback0
  address-family l2vpn evpn
   route-policy SITE-A-IMPORT in
   route-policy SITE-A-EXPORT out
  !
 !
 !! DCI EVPN (peering with remote BGWs)
 neighbor 10.255.0.2
  remote-as 65000
  description BGW-B1
  update-source Loopback0
  address-family l2vpn evpn
   route-policy DCI-IMPORT in
   route-policy DCI-EXPORT out
   next-hop-self
  !
 !
!
!! --- Route Policies for DCI ---
!! --- SoO community sets (must be defined before route-policy) ---
extcommunity-set soo SOO-SITE-A
  65000:100
end-set
!
extcommunity-set soo SOO-SITE-B
  65000:200
end-set
!
route-policy DCI-EXPORT
  !! Re-originate local routes with DCI attributes
  if evpn-route-type is 2 then
    set next-hop self
    set extcommunity soo SOO-SITE-A additive
    pass
  elseif evpn-route-type is 5 then
    set next-hop self
    set extcommunity soo SOO-SITE-A additive
    pass
  elseif evpn-route-type is 1 then
    set next-hop self
    pass
  elseif evpn-route-type is 3 then
    !! Inclusive Multicast — needed for BUM replication across DCI
    set next-hop self
    pass
  else
    drop
  endif
end-policy
!
route-policy DCI-IMPORT
  !! Filter routes that originated from this site (loop prevention)
  if extcommunity soo matches-any SOO-SITE-A then
    drop
  else
    pass
  endif
end-policy
!
route-policy SITE-A-EXPORT
  !! Re-originate DCI-learned routes into local site
  if extcommunity soo matches-any SOO-SITE-B then
    set next-hop self
    pass
  else
    pass
  endif
end-policy
!
route-policy SITE-A-IMPORT
  !! Accept all local site routes
  pass
end-policy
!
!! --- EVPN configuration ---
evpn
 evi 100
  bgp
   route-target import 100:100
   route-target export 100:100
  !
  advertise-mac
  !
 !
 evi 200
  bgp
   route-target import 200:200
   route-target export 200:200
  !
  advertise-mac
  !
 !
!
!! --- Distributed Anycast Gateway (IRB) ---
!! Same IP + same MAC on every BGW at every site = anycast gateway
!! EVPN advertises this IP/MAC from each PE — hosts always reach a local GW
interface BVI100
 host-routing
 ipv4 address 10.0.1.1/24
 mac-address 0000.5e00.0101
 description Anycast-GW-VLAN100
!
interface BVI200
 host-routing
 ipv4 address 10.0.2.1/24
 mac-address 0000.5e00.0201
 description Anycast-GW-VLAN200
!
!! --- VRF for inter-subnet routing ---
vrf TENANT-A
 address-family ipv4 unicast
  import route-target
   100:1000
  !
  export route-target
   100:1000
  !
 !
!
!! --- Bridge domains ---
l2vpn
 bridge group DCI
  bridge-domain BD100
   interface Bundle-Ether1.100
   !
   routed interface BVI100
   !
   evi 100
   !
  !
  bridge-domain BD200
   interface Bundle-Ether1.200
   !
   routed interface BVI200
   !
   evi 200
   !
  !
 !
!
!! --- BGW Multi-Homing (ES between BGW-A1 and BGW-A2) ---
evpn
 interface Bundle-Ether1
  ethernet-segment
   identifier type 0 0011.2233.4455.6677.8800
   bgp route-target 0011.2233.4455
  !
  lacp system mac 0011.2233.4455
 !
!
!! --- ARP/ND Proxy (IOS-XR 7.5+, NCS 5500/ASR 9000) ---
!! Configured under the BVI interface, not under EVI
interface BVI100
 arp evpn-proxy mode full-suppression
!
interface BVI200
 arp evpn-proxy mode full-suppression
!
!! Modes:
!!  full-suppression  — proxy replies for all known bindings, suppress flood
!!  partial-suppression — proxy replies but still floods to unknown targets
!! --- BFD on DCI BGP session ---
router bgp 65000
 neighbor 10.255.0.2
  bfd fast-detect
  bfd minimum-interval 300
  bfd multiplier 3
 !
!
```

#### Type-5 Summarization at the BGW (IOS-XR)

```cisco
!! --- Advertise subnet summaries instead of host routes across DCI ---
!! This goes in the VRF address family
router bgp 65000
 vrf TENANT-A
  address-family ipv4 unicast
   !! Suppress individual /32 host routes across DCI
   !! Advertise aggregate instead
   aggregate-address 10.0.1.0/24 summary-only
   aggregate-address 10.0.2.0/24 summary-only
  !
  !! EVPN Type-5 advertisement
  address-family l2vpn evpn
   advertise ipv4 unicast
   !! Controls whether RT-5 is generated from VRF routes
  !
 !
!
```

### Model 2: EVPN DCI Gateway (Junos)

#### BGW-A1 Configuration (Junos 23.x)

```junos
## --- Loopback ---
interfaces {
    lo0 {
        unit 0 {
            family inet {
                address 10.255.0.1/32;
            }
        }
    }
}

## --- DCI-facing interface ---
interfaces {
    et-0/0/0 {
        description "DCI to BGW-B1";
        unit 0 {
            family inet {
                address 10.100.0.0/31;
            }
            family mpls;
        }
    }
}

## --- Routing options (AS number required for iBGP) ---
routing-options {
    autonomous-system 65000;
    router-id 10.255.0.1;
}

## --- MPLS on DCI interfaces ---
protocols {
    mpls {
        interface et-0/0/0.0;
        interface lo0.0;
    }
}

## --- IS-IS with SR for DCI transport ---
protocols {
    isis {
        level 1 disable;
        level 2 wide-metrics-only;
        interface et-0/0/0.0 {
            point-to-point;
            level 2 metric 10;
        }
        interface lo0.0 {
            passive;
        }
        source-packet-routing {
            srgb start-label 16000 index-range 8000;
            node-segment {
                ipv4-index 1;
            }
        }
    }
}

## --- BGP with EVPN for DCI ---
protocols {
    bgp {
        group DCI-EVPN {
            type internal;
            local-address 10.255.0.1;
            family evpn {
                signaling;
            }
            neighbor 10.255.0.2 {
                description "BGW-B1";
                export DCI-EXPORT;
                import DCI-IMPORT;
            }
        }
        group SITE-A-EVPN {
            type internal;
            local-address 10.255.0.1;
            family evpn {
                signaling;
            }
            neighbor 10.254.0.1 {
                description "Site-A-RR";
            }
        }
    }
}

## --- Route policies ---
policy-options {
    policy-statement DCI-EXPORT {
        term re-originate {
            from {
                family evpn;
                nlri-route-type [2 5];
            }
            then {
                next-hop self;
                community add SOO-SITE-A;
                accept;
            }
        }
        term ead-routes {
            from {
                family evpn;
                nlri-route-type 1;
            }
            then {
                next-hop self;
                accept;
            }
        }
        term imet-routes {
            from {
                family evpn;
                nlri-route-type 3;  /* Inclusive Multicast — BUM replication */
            }
            then {
                next-hop self;
                accept;
            }
        }
        term default {
            then reject;
        }
    }
    policy-statement DCI-IMPORT {
        term reject-own-site {
```

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| ARP suppression | Disabled | **Enabled** | Reduces DCI BUM traffic 80-95% |
| BFD on DCI BGP | Disabled | **Enabled, 300ms×3** | Sub-second BGW failure detection |
| MAC mobility seq | Auto (0) | Auto | Don't manually set — let the protocol handle it |
| SoO community | None | **Mandatory** | Loop prevention — without it, DCI routes bounce between sites |
| RT-constrain (RFC 4684) | Often disabled | **Enabled** | BGWs only import relevant RTs — major scale improvement |
| Type-5 summarization | Per-host (/32) | **Aggregate** for inter-subnet DCI | 10-100x route reduction across WAN |
| Storm control on DCI | None | **Rate-limit BUM** | Protect WAN bandwidth from broadcast storms |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| MAC flapping between sites | Missing SoO community | Configure SoO on DCI export policy; verify DCI import drops own SoO |
| Asymmetric routing (traffic enters Site A, exits Site B) | Anycast gateway MAC mismatch | Verify identical virtual MAC on all sites; check `mac-address` on BVI/IRB |
| DCI routes not propagated | RT mismatch between sites | Verify RT import/export matches on both BGWs; check `show bgp l2vpn evpn` for filtered routes |
| ARP not suppressed | Suppression not enabled or no RT-2 with IP | Enable ARP suppression; verify RT-2 routes include IP field (not all hosts send GARP) |
| Slow convergence on BGW failure | BFD not configured on DCI | Enable BFD on DCI BGP session; tune interval to 300ms×3 |
| Type-5 routes not generated | Missing `advertise ipv4 unicast` under L2VPN EVPN AF | Add the advertisement config under VRF EVPN AF; verify BVI/IRB routes are in VRF |
| Duplicate packets to multi-homed CE | DF election not converged | Check ES state on both BGWs; verify ESI matches; check RT-4 (ES route) exchange |
| WAN bandwidth saturated by BUM | ARP suppression off + large broadcast domain | Enable ARP suppression; implement storm control; consider reducing L2 stretch scope |
| Split-brain MAC conflicts after WAN recovery | Sequence numbers not incrementing | Verify MAC mobility is enabled; check for static MAC entries (don't participate in mobility) |
| Type-5 and Type-2 both present across DCI | Summary-only not configured | Add `summary-only` to aggregates; verify suppress-map if using selective suppression |

### Methodology

1. **Verify BGP EVPN sessions** — Is the DCI peering established? Check `show bgp l2vpn evpn summary` for the remote BGW.
2. **Check SoO** — Are routes being tagged with Site-of-Origin? Missing SoO is the #1 cause of DCI routing loops.
3. **Verify route re-origination** — Does the remote site see your routes with BGW's next-hop (not the original leaf's next-hop)?
4. **Check MAC tables** — Are remote MACs showing up with the DCI BGW as next-hop? If they show the remote leaf directly, re-origination isn't working.
5. **Test ARP suppression** — Send an ARP from Site A for a host at Site B. Does it cross the DCI? Check suppression counters.
6. **Verify anycast gateway** — Ping the gateway IP from hosts at both sites. Both should respond. Check that the MAC in the ARP reply is the virtual MAC, not a per-site real MAC.
7. **Test failover** — Shut one BGW's DCI link. Measure convergence time. Verify traffic shifts to the surviving BGW. Bring it back — verify no MAC flapping.
8. **Check BUM impact** — Monitor DCI interface counters for broadcast/multicast rates. If they're high, ARP suppression isn't working or you have a chatty protocol flooding across the DCI.

### War Stories

> **The Midnight MAC Flap Storm**
>
> A financial services operator ran EVPN DCI between two data centers — 60km apart on dark fiber, 0.3ms RTT. Should have been a textbook deployment. At 2 AM on a Tuesday, their monitoring exploded: 50,000 MAC flap events in 10 minutes, both sites reporting conflicting MACs, and the BGWs' CPU hit 95% processing BGP updates.
>
> Root cause: A junior engineer had copied the BGW config from an internal wiki. The wiki article was for a single-site deployment and didn't include SoO communities. Without SoO, the loop prevention logic was absent. A VM migration (routine, happened 20 times a day) triggered the first MAC move. Site A's BGW re-originated the move to Site B. Site B's BGW, seeing a "new" route, re-originated it back to Site A. Site A saw a "newer" route (higher seq number), re-originated again. Each re-origination incremented the sequence number. Within seconds, the MAC mobility sequence hit thousands, and every BGP UPDATE triggered more UPDATEs.
>
> The fix was a named SoO set and two policy changes per BGW:
> ```
> extcommunity-set soo SOO-SITE-A        ← define the set
>   65000:100
> end-set
> set extcommunity soo SOO-SITE-A        ← on DCI export
> if extcommunity soo matches-any SOO-SITE-A then drop  ← on DCI import
> ```
>
> **Lesson:** SoO is not optional in DCI. It's load-bearing. Without it, any MAC move becomes an exponential amplification event. They added SoO to their golden config template that night and added a pre-deployment validation check that greps for SoO in every DCI config. The junior engineer went on to become a very SoO-conscious senior engineer.

## Design Considerations

### Choosing Your DCI Model

| Criteria | Model 1 (Unified) | Model 2 (Gateway) | Model 3 (Multi-Site RFC 9014) |
|----------|-------------------|-------------------|-------------------------------|
| Number of sites | 2-3 | 2-8 | 4+ |
| MAC scale | <50K total | <500K per site | >500K per site |
| Fault isolation | None | Strong | Strong |
| Operational complexity | Low | Medium | High |
| WAN control plane load | High (all routes) | Medium (re-originated) | Low (Type-5 summary) |
| Data plane flexibility | Same everywhere | Mixed (VXLAN+MPLS) | Mixed |
| Vendor maturity | Universal | Universal | Varies (strongest NX-OS) |
| Recommendation | Dev/test, small SP | **Most production DCI** | Hyperscale, multi-DC cloud |

### Capacity Planning

**DCI bandwidth sizing:**
- **Baseline:** Sum of inter-site traffic flows (use NetFlow/sFlow from existing WAN links)
- **BUM overhead:** With ARP suppression, add 5-10% for residual BUM. Without, add 20-40%.
- **Failover headroom:** If one DCI link fails, the remaining link must handle 100% of traffic. Design for 2×steady-state.
- **Growth:** EVPN DCI tends to attract more workloads once deployed ("now that we CAN stretch, let's stretch everything"). Plan 3x.

**BGW sizing:**
- MAC table: Sum of both sites' MACs per EVI (BGW holds both)
- RIB/FIB: All local EVPN routes + all DCI EVPN routes + VRF routes
- CPU: BGP update processing during convergence events (test with mass withdrawal)
- Memory: RT-2 routes consume ~200-500 bytes each in BGP RIB. At 100K MACs × 500 bytes = 50MB per EVI. Plan accordingly.

### What NOT to Stretch

Not every workload belongs on a stretched L2 domain. Use this framework:

| Workload Type | Stretch L2? | Better Alternative |
|---------------|-------------|-------------------|
| VM live migration | Yes (requires L2) | — |
| Active-active database cluster | Yes (low latency required) | Only if RTT < 5ms |
| Web tier (stateless) | **No** | DNS-based GSLB, anycast |
| Backup/DR (cold standby) | **No** | L3 routing + Type-5, restore from backup |
| Legacy apps that "need L2" | **Maybe** | Investigate if a proxy/NAT eliminates the requirement |
| Storage replication | **No** | Dedicated storage network, async replication |

**The golden rule:** If you can solve it at L3, solve it at L3. L2 stretch is a precision tool for specific use cases, not a default architecture.
