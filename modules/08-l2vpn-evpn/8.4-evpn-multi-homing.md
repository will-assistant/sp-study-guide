# 8.4 — EVPN Multi-Homing

## Overview

> Multi-homing is where EVPN earns its reputation. VPLS gave you active/standby with STP — one link idle, one working, and a convergence event that made everyone hold their breath. EVPN multi-homing delivers all-active forwarding across multiple PEs with sub-second failover, no STP, no wasted bandwidth, and a control plane that actually knows what's happening. It's the single biggest reason operators migrated from VPLS. The mechanics — Ethernet Segments, DF election, aliasing, mass withdrawal, split-horizon filtering — are the most complex part of EVPN, and the most heavily tested on IE-SP and IE-SP exams. If you understand this section, you understand EVPN.

## Theory

### The Multi-Homing Problem

A CE device connects to two (or more) PEs via a Link Aggregation Group (LAG). From the CE's perspective, it's one logical link. From the SP backbone's perspective, it's two separate attachment points that must be coordinated. Without coordination, you get:

1. **Duplicate frames** — Both PEs forward the same BUM frame to the CE
2. **MAC flapping** — Remote PEs see the same MAC from two different PEs
3. **Loops** — BUM traffic bounces between PEs through the CE
4. **Black holes** — One PE goes down, remote PEs still send traffic to it

VPLS "solved" this with STP or MC-LAG (vendor-specific, operationally painful). EVPN solves it with three BGP-based mechanisms that require zero data-plane hacks:

| Mechanism | Route Type | What It Does |
|-----------|-----------|-------------|
| **DF Election** | RT-4 (ES Route) | Elects one PE per VLAN to forward BUM traffic to the CE — prevents duplicates |
| **Aliasing** | RT-1 (EAD per-EVI) | Tells remote PEs that *all* MH PEs can forward traffic for a given ES — enables ECMP |
| **Mass Withdrawal** | RT-1 (EAD per-ES) | When a PE loses its ES link, one withdrawal triggers fast failover for *all* MACs on that segment |

Plus two supporting mechanisms:

| Mechanism | How | What It Does |
|-----------|-----|-------------|
| **Split-Horizon Filtering** | ESI label in MPLS / ESI in VXLAN | Prevents BUM traffic from being looped back through the multi-homed CE |
| **MAC Syncing** | RT-2 with ESI field | Ensures all MH PEs know all MACs learned locally on any PE in the ES |

### Ethernet Segment (ES)

The **Ethernet Segment Identifier (ESI)** is the cornerstone of EVPN multi-homing. It's a 10-byte value that uniquely identifies a multi-homed link bundle. Every PE connecting to the same CE via the same LAG must be configured with the *same* ESI.

**ESI Format (10 bytes):**

```
 Byte 0       Bytes 1-9
┌──────┬──────────────────────┐
│ Type │   Value (9 bytes)    │
└──────┴──────────────────────┘
```

**ESI Types:**

| Type | Name | Derivation | Use Case |
|------|------|-----------|----------|
| 0 | Arbitrary | Manually configured | Most common in SP deployments — full control |
| 1 | LACP-based | Auto-derived from LACP system MAC + port key | When CE runs LACP — auto-generates ESI |
| 2 | STP-based | Bridge priority + MAC | Legacy STP interop (rare) |
| 3 | MAC-based | System MAC + local discriminator | CE uses a consistent system MAC |
| 4 | Router ID-based | Router ID + local discriminator | When CE is a router (not a switch) |
| 5 | AS-based | AS number + local discriminator | BGP-based CE identification |

**Special values:**
- **ESI 0 (all zeros):** Single-homed — no multi-homing procedures apply
- **MAX-ESI (all 0xFF):** Reserved

> **Operational note:** Type 0 (manual) dominates in production because operators want explicit control over which PEs form a redundancy group. Auto-derived ESIs (Types 1-5) sound elegant but can cause mismatches when CE configs change unexpectedly. The 30 seconds you save on config isn't worth the 3 AM call when LACP renegotiation generates a new ESI.

### DF Election — Who Forwards BUM?

When multiple PEs connect to the same ES, only **one PE per VLAN** should forward BUM (Broadcast, Unknown unicast, Multicast) traffic toward the CE. Otherwise, the CE receives duplicate frames. The PE that forwards BUM for a given VLAN is the **Designated Forwarder (DF)**.

**How DF election works:**

1. Each PE advertises an **ES Route (Route Type 4)** containing:
   - ESI value
   - Originating PE IP (next-hop)
   - ES-Import extended community (auto-derived from ESI bytes 1-6)

2. Each PE collects all RT-4 routes for the same ESI → discovers all PEs in the redundancy group

3. PEs run the DF election algorithm to assign Ethernet Tags (VLANs) to PEs

> **Ethernet Tag vs VLAN:** RFC 7432 specifies DF election per **Ethernet Tag**, not per VLAN. For VLAN-based service (one VLAN per EVI), the Ethernet Tag equals the VLAN ID. For VLAN-aware bundle service (multiple VLANs per EVI), each VLAN has its own Ethernet Tag and gets its own DF. For VLAN-bundle service (all VLANs mapped to one EVI with Ethernet Tag 0), there is **one DF for the entire ES** — no per-VLAN distribution. This distinction matters for exam scenarios.

**DF Election Algorithms:**

| Algorithm | RFC | Method | Pros | Cons |
|-----------|-----|--------|------|------|
| **Default (mod-based)** | RFC 7432 | `VLAN mod N` (N = number of PEs) | Simple, deterministic | Uneven with prime VLAN counts; all VLANs shift on PE add/remove |
| **HRW (Highest Random Weight)** | RFC 8584 | Hash(VLAN, PE_IP) → highest wins | Minimal disruption on PE change; better distribution | Slightly more CPU |
| **Preference-based** | RFC 8584 | Admin weight per PE | Full manual control | Operational overhead |

**Example — mod-based with 3 PEs and 6 VLANs:**

```
PE1 (IP: 10.0.0.1), PE2 (IP: 10.0.0.2), PE3 (IP: 10.0.0.3)
PEs sorted by IP: [PE1, PE2, PE3]  → index 0, 1, 2

VLAN 10:  10 mod 3 = 1 → PE2 is DF
VLAN 20:  20 mod 3 = 2 → PE3 is DF
VLAN 30:  30 mod 3 = 0 → PE1 is DF
VLAN 40:  40 mod 3 = 1 → PE2 is DF
VLAN 50:  50 mod 3 = 2 → PE3 is DF
VLAN 60:  60 mod 3 = 0 → PE1 is DF
```

Each PE is DF for 2 VLANs — perfectly balanced. But add a 4th PE and every assignment shifts. HRW avoids this churn.

**DF vs non-DF behavior:**

| Traffic Direction | DF | Non-DF |
|------------------|-----|--------|
| BUM → CE (from backbone) | Forwards | **Drops** |
| Unicast → CE | Forwards (if it has the MAC) | Forwards (if it has the MAC) |
| Any traffic from CE → backbone | Forwards | Forwards |
| BUM from CE → backbone | Forwards (originates into EVPN) | Forwards (originates into EVPN) |

> **Critical distinction:** DF election only controls **BUM traffic toward the CE**. Unicast traffic in both directions flows through *all* PEs in the ES. This is what makes all-active actually all-active — every PE forwards unicast; only BUM is gated.

### All-Active vs Single-Active Multi-Homing

EVPN supports two forwarding modes per Ethernet Segment:

#### All-Active

```
            ┌──────────────────────┐
            │    Remote PE (PE3)   │
            │  Sees: MAC-A via     │
            │  PE1 AND PE2 (ECMP)  │
            └──────────┬───────────┘
                       │
              ┌────────┴────────┐
              │                 │
        ┌─────┴─────┐   ┌──────┴────┐
        │   PE1     │   │   PE2     │
        │  (DF V10) │   │ (non-DF)  │
        │  ESI: X   │   │  ESI: X   │
        └─────┬─────┘   └─────┬─────┘
              │                │
              └───────┬────────┘
                      │ LAG (LACP)
                ┌─────┴─────┐
                │    CE1    │
                │  MAC-A    │
                └───────────┘
```

- **Both PEs forward unicast traffic** to/from the CE
- **One PE is DF** for BUM per VLAN (prevents duplicate delivery)
- **CE must run LACP** — it sees one logical link; it load-balances across members
- **Aliasing (RT-1 per-EVI)** tells remote PEs to ECMP across both PEs
- **Split-horizon** prevents loops (PE1 won't re-forward BUM received from PE2 via the CE)
- **Best for:** Bandwidth utilization, sub-second failover, modern deployments

#### Single-Active

```
        ┌───────────┐   ┌───────────┐
        │   PE1     │   │   PE2     │
        │ (ACTIVE)  │   │ (STANDBY) │
        │  ESI: X   │   │  ESI: X   │
        └─────┬─────┘   └─────┬─────┘
              │                │ (blocked)
              │                │
        ┌─────┴────────────────┴─────┐
        │           CE1              │
        │   (two separate links)     │
        └────────────────────────────┘
```

- **Only one PE forwards** all traffic (active); others are standby
- **CE does NOT need LACP** — it can use separate physical links (or even different link types)
- **No aliasing needed** — only one PE advertises MAC routes
- **Failover** relies on RT-1 EAD withdrawal + standby PE activating
- **Best for:** Legacy CEs that can't do LACP, or when CE is a host with active/standby NICs

**Comparison:**

| Aspect | All-Active | Single-Active |
|--------|-----------|---------------|
| CE requirement | Must support LACP (or static LAG) | No LAG needed — independent links OK |
| Bandwidth | Full utilization (ECMP across PEs) | 50% wasted (standby link idle) |
| BUM handling | DF election per VLAN | Active PE handles all BUM |
| Unicast handling | Both PEs forward (aliasing) | Only active PE forwards |
| Failover speed | Sub-second (mass withdrawal + LACP) | 1-3 seconds (EAD withdrawal + activation) |
| Split-horizon | Required (ESI label) | Not needed (only one PE active) |
| Complexity | Higher (DF + aliasing + split-horizon) | Lower |
| Use case | Modern L2 services, DCI | Legacy CE, host multi-homing |

### Aliasing — Making ECMP Work

The problem: A CE's MAC address is learned *locally* on whichever PE the CE's LACP hashing sent the frame to. If MAC-A is learned on PE1, remote PEs only know about PE1. PE2 is a valid path (same ES, same LAG), but remote PEs don't know that.

**Aliasing** solves this. Each PE in an all-active ES advertises an **EAD per-EVI route (Route Type 1)** that says: "I am attached to this ESI in this EVI. You can send me traffic for any MAC on this segment."

When a remote PE (PE3) receives:
1. **RT-2 (MAC/IP):** MAC-A is on PE1 via ESI-X
2. **RT-1 per-EVI:** PE1 is part of ESI-X in EVI 100
3. **RT-1 per-EVI:** PE2 is also part of ESI-X in EVI 100

PE3's logic: "MAC-A is on ESI-X. Both PE1 and PE2 are in ESI-X for this EVI. I can ECMP across both."

**Without aliasing:** Traffic to MAC-A goes only through PE1. PE2's link to the CE sits partially idle.

**With aliasing:** Traffic to MAC-A is ECMP'd across PE1 and PE2. Full bandwidth utilization.

> **Key detail:** Aliasing works at the *ES* level, not the MAC level. Remote PEs don't need per-MAC ECMP state. They see "this MAC is on ESI-X" and "these PEs serve ESI-X" — the ECMP next-hop list is per-ES. This is critical for scale: 10,000 MACs on an ES with 2 PEs = 10,000 MAC entries but only 1 ECMP group.

### Mass Withdrawal — Fast Failover

Traditional MAC withdrawal in VPLS: each MAC is withdrawn individually. 10,000 MACs on a failed link = 10,000 BGP withdrawals. At scale, this takes seconds to minutes.

EVPN mass withdrawal: one **EAD per-ES route (Route Type 1)** withdrawal triggers immediate failover for **all MACs** on that Ethernet Segment.

**How it works:**

1. PE1 detects its link to CE1 has failed (physical down or LACP timeout)
2. PE1 withdraws its **EAD per-ES route** for ESI-X (one BGP UPDATE)
3. All remote PEs receive this single withdrawal
4. Remote PEs immediately:
   - Remove PE1 from the ECMP next-hop set for ESI-X
   - Re-resolve all MACs on ESI-X → now only PE2
   - No need to wait for individual MAC withdrawals
5. Convergence time: **one BGP UPDATE propagation** (~100-500ms with BFD on BGP sessions)

**Contrast with VPLS:**

| Scenario: 10,000 MACs, PE failure | VPLS | EVPN |
|-----------------------------------|------|------|
| BGP withdrawals needed | 10,000 | **1** (EAD per-ES) |
| Convergence time | 5-30 seconds | **< 1 second** |
| CPU spike on remote PEs | Severe (processing 10K updates) | Minimal (1 update, bulk re-resolve) |
| Impact on other services | MAC table churn affects everything | Scoped to one ES |

> **The EAD per-ES route is the most operationally important route in EVPN.** It's the difference between "PE failed, 30 seconds of packet loss across 500 services" and "PE failed, 200ms blip on affected segment only."

### Split-Horizon Filtering — Preventing Loops

In an all-active ES, both PEs forward BUM traffic from the backbone toward the CE via the LAG. The CE, treating it as one logical link, may forward that BUM frame back out another LAG member — which lands on the *other* PE. Without filtering, that PE would re-inject the frame into the backbone. Loop.

**EVPN split-horizon** prevents this using the **ESI label**:

**MPLS data plane:**

1. PE1 receives a BUM frame from a remote PE (or originates one from a different local ES)
2. PE1 floods to all PEs, encapsulating with **two labels**: outer = transport, inner = EVPN service label + **ESI label**
3. PE2 receives the frame, pops the transport label, sees the ESI label
4. PE2 checks: "Does this ESI label match one of my local Ethernet Segments?"
5. **If yes:** PE2 **does NOT forward** the frame out the local ES interface — this prevents the loop
6. **If no:** PE2 forwards normally (it's from a different segment)

**VXLAN data plane:**
- No label stack available. VXLAN uses **local bias** for split-horizon: when a PE receives a BUM frame from a remote PE, it checks whether the destination ES is locally attached *and* whether the remote PE is also a member of that ES. If so, the PE suppresses forwarding to that local ES — the originating PE's ES peer already delivered the frame locally. This is fundamentally different from MPLS ESI label: it's a local ingress decision, not a label-based egress filter.

```
Split-Horizon in Action:

  BUM Frame from Remote PE3:
  
  PE3 ──[EVPN BUM]──► PE1: Forwards to CE via ESI-X ✓
                  └──► PE2: Forwards to CE via ESI-X ✓ (but DF check first)
                        BUT: PE2 adds ESI label when re-flooding
  
  CE sends frame back via LAG:
  
  CE ──[LAG hash]──► PE1: Learns locally, floods to backbone ✓
                └──► PE2: Learns locally, floods to backbone ✓
                      BUT: PE1/PE2 mark frame with ESI-X label
  
  PE2 receives from PE1 (via backbone):
  PE2 sees ESI label = ESI-X = my local ES → DROP (split-horizon) ✓
```

### MAC Synchronization

In all-active mode, LACP hashing determines which PE learns a given MAC locally. If MAC-A is learned on PE1, PE2 doesn't know about it from the data plane. But PE2 needs to know — if a remote PE sends unicast traffic for MAC-A to PE2 (via aliasing/ECMP), PE2 must be able to forward it to the CE.

**Solution:** PE1 advertises MAC-A via BGP (Route Type 2) with the ESI field populated. PE2 receives this RT-2, sees the ESI matches its local ES, and installs MAC-A as a **"remote-learned-via-ES"** entry pointing to the local CE-facing interface.

This is often called **MAC syncing** — all PEs in an all-active ES end up with the full MAC table for that segment, regardless of which PE learned each MAC from the data plane.

### Putting It All Together — Packet Walk

**Scenario:** CE1 dual-homed to PE1+PE2 (ESI-X, all-active). CE2 single-homed to PE3. CE1 sends a frame destined for CE2's MAC-B.

**Ingress (CE1 → backbone):**
1. CE1 sends frame with src=MAC-A, dst=MAC-B
2. LACP hashing on CE1 selects PE1
3. PE1 receives on ESI-X interface, looks up MAC-B → known via RT-2 from PE3
4. PE1 encapsulates: MPLS label stack = [transport][EVPN service label]
5. Frame arrives at PE3, PE3 pops labels, forwards to CE2

**Ingress (CE2 → CE1, with aliasing):**
1. CE2 sends frame with src=MAC-B, dst=MAC-A
2. PE3 receives, looks up MAC-A → learned via RT-2 from PE1, but aliasing (RT-1 per-EVI from PE1 and PE2) means ECMP next-hops = {PE1, PE2}
3. PE3 ECMP hashes → selects PE2
4. PE2 receives, pops labels. MAC-A is in PE2's table (via MAC sync). Forwards to CE1 via ESI-X LAG.

**BUM (ARP from CE1, unknown destination):**
1. CE1 sends ARP broadcast, LACP hashes to PE1
2. PE1 is DF for this VLAN on ESI-X → floods to all remote PEs via ingress replication
3. PE1 adds ESI label for ESI-X in the encapsulation
4. PE2 receives (via backbone): sees ESI label = ESI-X (local ES) → **split-horizon drop** — does NOT flood to CE1 again
5. PE3 receives: no local ESI-X → forwards to CE2 ✓
6. If PE2 had *also* received the ARP from CE1 via the LAG (LACP duplicate): PE2 is non-DF → drops the frame before flooding to backbone (DF filtering)

### Route Type Interactions for Multi-Homing

| Route Type | Per-ES or Per-EVI | Multi-Homing Role |
|-----------|------------------|------------------|
| **RT-1 EAD per-ES** | Per-ES | Mass withdrawal — one withdraw = all MACs failed over |
| **RT-1 EAD per-EVI** | Per-EVI | Aliasing — tells remotes to ECMP across all PEs in ES |
| **RT-2 MAC/IP** | Per-EVI | MAC sync — populates tables on peer MH PEs via ESI field |
| **RT-4 ES Route** | Per-ES | DF election — discovers ES peers, runs election algorithm |

> **RT-1 is overloaded** — it serves two completely different purposes depending on whether it's per-ES (Ethernet Tag = 0, mass withdrawal) or per-EVI (Ethernet Tag = EVI, aliasing). This confuses everyone the first time. The key: per-ES has Ethernet Tag 0 and is about *reachability*. Per-EVI has a non-zero Ethernet Tag and is about *forwarding optimization*.

## Configuration

### IOS-XR (All-Active Multi-Homing)

```cisco
!! -- Interface Configuration --
!! LAG facing multi-homed CE
interface Bundle-Ether10
 description "LAG to CE1 (dual-homed)"
 lacp system mac 0011.2233.4455     !! MUST match on all PEs — CE sees one LACP peer
 lacp period short                  !! Fast LACP timers (1 sec)
!

!! -- Sub-interface for L2 transport --
interface Bundle-Ether10.100 l2transport
 encapsulation dot1q 100
!

!! -- Ethernet Segment Configuration (global EVPN stanza) --
evpn
 interface Bundle-Ether10
  ethernet-segment
   identifier type 0 0011.2233.4455.6677.8800    !! Must match on all PEs in this ES
   bgp route-target 0011.2233.4455               !! ES-Import RT — optional (auto-derived from ESI bytes 1-6 if omitted)
   load-balancing-mode all-active                 !! All-active forwarding
  !
 !

!! -- EVPN EVI Configuration --
evpn
 evi 100
  bgp
   route-target import 100:100
   route-target export 100:100
  !
  advertise-mac
  !
 !
!

!! -- Bridge Domain --
l2vpn
 bridge group BG1
  bridge-domain BD100
   interface Bundle-Ether10.100
   !
   evi 100
   !
  !
 !
!

!! -- BGP EVPN Address Family --
router bgp 65000
 address-family l2vpn evpn
 !
 neighbor 10.0.0.99
  remote-as 65000
  update-source Loopback0
  address-family l2vpn evpn
   route-reflector-client    !! If this PE is behind an RR
  !
 !
!
```

**Key knobs for IOS-XR multi-homing:**

```cisco
!! -- DF Election Algorithm (optional, default = mod-based) --
evpn
 interface Bundle-Ether10
  ethernet-segment
   identifier type 0 0011.2233.4455.6677.8800
   service-carving hrw          !! Use HRW instead of mod-based (recommended)
  !
 !
!

!! -- Single-Active (instead of all-active) --
evpn
 interface Bundle-Ether10
  ethernet-segment
   identifier type 0 0011.2233.4455.6677.8800
   load-balancing-mode single-active
  !
 !
!

!! -- Timers --
evpn
 timers
  peering 3                   !! Seconds to wait for ES peer discovery before DF election
  recovery 30                 !! Seconds to wait after ES comes up before becoming DF (prevents thrashing)
 !
!
```

### Junos (All-Active Multi-Homing)

```junos
## -- Aggregated Ethernet (LAG) interface --
interfaces {
    ae0 {
        description "LAG to CE1 (dual-homed)";
        esi {
            00:11:22:33:44:55:66:77:88:00;    ## Must match on all PEs in this ES
            all-active;                         ## All-active mode (default is single-active)
        }
        aggregated-ether-options {
            lacp {
                active;
                periodic fast;                  ## Fast LACP timers (1 sec)
                system-id 00:00:00:11:22:33;   ## Same system-id on both PEs for LACP
                admin-key 1;                    ## Same admin-key on both PEs
            }
        }
        unit 0 {
            family ethernet-switching {
                interface-mode trunk;
                vlan {
                    members [ v100 v200 ];
                }
            }
        }
    }
}

## -- EVPN Instance --
routing-instances {
    EVPN-100 {
        instance-type virtual-switch;
        interface ae0.0;
        route-distinguisher 10.0.0.1:100;
        vrf-target target:100:100;
        protocols {
            evpn {
                encapsulation mpls;
                extended-vlan-list 100;
                df-election-type {
                    preference {                ## Or use "mod" for default
                        value 200;              ## Higher = preferred DF
                    }
                }
            }
        }
        bridge-domains {
            BD100 {
                vlan-id 100;
            }
        }
    }
}

## -- BGP EVPN --
protocols {
    bgp {
        group EVPN-RR {
            type internal;
            local-address 10.0.0.1;
            family evpn {
                signaling;
            }
            neighbor 10.0.0.99;
        }
    }
}
```

**Junos single-active:**

```junos
interfaces {
    ae0 {
        esi {
            00:11:22:33:44:55:66:77:88:00;
            single-active;                      ## Only one PE forwards at a time
        }
    }
}
```

**Junos DF election options:**

```junos
routing-instances {
    EVPN-100 {
        protocols {
            evpn {
                ## Default: mod-based (no explicit config needed)
                ## To use preference-based:
                df-election-type {
                    preference {
                        value 200;              ## Higher value = preferred DF
                    }
                }
            }
        }
    }
}
```

> **LACP system-id gotcha:** In all-active mode, both PEs must present the **same LACP system-id** (system MAC) and **admin-key** to the CE. The CE thinks it's talking to one device. If these don't match, the CE sees two separate LAGs and LACP won't form a bundle. On **IOS-XR**, you must explicitly configure `lacp system mac` on the Bundle-Ether interface — it is **not** auto-derived from the ESI. On **Junos**, you must explicitly configure `system-id` under `aggregated-ether-options > lacp`. Both platforms require this as manual, matching config on all PEs in the ES.

### Key Knobs

| Parameter | IOS-XR Default | Junos Default | Recommended | Why |
|-----------|---------------|--------------|-------------|-----|
| DF election | mod-based | mod-based | **HRW** (IOS-XR `service-carving hrw`) | Minimal disruption when PEs join/leave |
| MH mode | single-active | single-active | **all-active** (unless CE can't do LACP) | Full bandwidth utilization |
| LACP period | 30s (slow) | 30s (slow) | **Fast (1s)** | Faster local failure detection |
| Peering timer | 3s | 3s | 3s | Time to discover ES peers before DF election |
| Recovery timer | 30s | varies | 20-30s | Prevents DF thrashing on flapping links |
| MAC sync | Enabled (automatic) | Enabled (automatic) | Keep enabled | Required for all-active forwarding |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| ES shows no peers | RT-4 not being received — BGP EVPN session down, or ES-Import RT mismatch | Verify BGP EVPN peering, check ESI matches exactly on both PEs |
| CE sees two separate links (not one LAG) | LACP system-id or admin-key mismatch between PEs | Align LACP system-id and admin-key on both PEs (Junos: explicit config; IOS-XR: auto from ESI) |
| All DF on one PE | DF election not running — likely only one PE in ES, or mod-based with unfortunate VLAN distribution | Verify both PEs in ES; consider HRW election |
| Duplicate BUM frames at CE | DF election broken — both PEs think they're DF, or split-horizon not working | Check `show evpn ethernet-segment carving` — should have one DF per VLAN; verify ESI label in data plane |
| No ECMP for MH MACs at remote PE | Aliasing not working — RT-1 per-EVI missing or not imported | Verify RT-1 per-EVI routes are advertised and imported; check RT match |
| Slow failover (seconds, not sub-second) | Mass withdrawal not working — RT-1 per-ES not being withdrawn, or BGP propagation slow | Verify EAD per-ES route exists pre-failure; add BFD to BGP sessions for fast detection |
| MAC flapping between PEs | ESI not configured (MACs learned as single-homed on both PEs) or ESI mismatch | Verify ESI is identical on both PEs — even one byte off = separate ESes |
| Traffic black hole after PE failover | Recovery timer too long — standby PE won't forward until timer expires | Tune recovery timer (default 30s may be too long for some environments) |

### Methodology

1. **Verify ES formation:** `show evpn ethernet-segment` — both PEs visible? ES state Up?
2. **Verify LACP:** `show lacp` — CE sees single LAG? System-id matches across PEs?
3. **Verify DF election:** `show evpn ethernet-segment carving` — VLANs distributed?
4. **Verify RT-1 routes:** `show bgp l2vpn evpn route-type 1` — per-ES AND per-EVI present?
5. **Verify RT-4 routes:** `show bgp l2vpn evpn route-type 4` — ES peers discovered?
6. **Verify MAC sync:** Check MAC table on non-learning PE — does it have MACs learned by peer?
7. **Verify aliasing at remote PE:** Check ECMP next-hop count for MH MAC — should be 2 (or N for N PEs)
8. **Test failover:** Shut one PE's link → verify sub-second convergence via ICMP loss
9. **Test BUM:** Send broadcast from CE → verify received once (not duplicated) at remote CE

### War Stories

> **The Phantom ESI Mismatch**
>
> A Tier 1 SP deployed EVPN all-active multi-homing across 200 PE pairs for enterprise L2VPN services. Everything tested perfectly in the lab. Week one in production: intermittent MAC flapping on ~15% of services, random enough to make the NOC question their sanity.
>
> The ESIs were configured using Type 0 (manual), and someone had written a Jinja2 template to generate configs. The template used this:
>
> ```
> identifier type 0 {{ site_id }}.{{ rack_id }}.{{ port_id }}.00.00.00.00
> ```
>
> The problem: `site_id` was a 4-digit number, but the ESI field expected two-digit hex groups. Site 1234 generated `12.34.xx...` on PE1, but the PE2 config was generated from a different inventory source that zero-padded differently: `12.34.0x...`. One byte off = two different Ethernet Segments. Each PE thought it was single-homed.
>
> The fix: Standardized ESI generation from a single source of truth (IPAM), added a pre-deployment validator that compared ESI configs across PE pairs, and added telemetry alerting on "ES with only one peer" (which should be zero for a dual-homed design).
>
> **Lesson:** Automate ESI assignment from one source. Never let two different systems generate the same ESI independently. And always alert when an ES has fewer peers than your design calls for.

## Lab Exercise

- **Topology:** Dual-homed CE (CE1) connected to PE1 and PE2 via LAG; single-homed CE2 on PE3; PE1-PE3 connected via MPLS core with IS-IS and SR-MPLS
- **Objective:**
  1. Configure ESI on PE1 and PE2 (all-active, same ESI)
  2. Verify DF election — confirm VLANs distributed across PEs
  3. Verify aliasing — confirm remote PE3 has ECMP to PE1+PE2 for CE1's MACs
  4. Test failover — shut PE1's link, measure convergence time
  5. Verify split-horizon — send broadcast from PE3, confirm CE1 receives one copy only
- **Validation:**
  - `show evpn ethernet-segment` shows both peers
  - `show bgp l2vpn evpn route-type 1` shows per-ES and per-EVI routes
  - ICMP from CE2 to CE1 shows < 1 second loss during PE1 shutdown
  - tcpdump on CE1 shows exactly one copy of broadcast from CE2
- **Stretch Goal:** Convert to single-active, observe behavior differences. Then add a third PE to the ES and verify DF election with 3-way.
