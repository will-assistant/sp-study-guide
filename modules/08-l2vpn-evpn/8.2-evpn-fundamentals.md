# 8.2 — EVPN Fundamentals

## Overview

> EVPN (Ethernet VPN, RFC 7432) is the control plane that finally makes L2VPN services sane. Where VPLS learned MACs in the data plane (flood-and-learn, like a dumb switch), EVPN advertises MAC and IP addresses through BGP — giving operators the same visibility and control over Layer 2 services that they've had over Layer 3 for decades. It fixes every major VPLS complaint: no more full-mesh pseudowires, no more STP over the backbone, proper multi-homing with all-active forwarding, ARP/ND suppression to kill broadcast storms, and aliasing so traffic uses all available paths. EVPN isn't just "better VPLS" — it's a unified framework that handles L2VPN, L3VPN, and DCI use cases with one control plane. If you're building anything new in 2026, it's EVPN or you're wrong.


## Key Concepts
> **After this section you will understand:**
> - how EVPN advertises MAC/IP reachability and multihoming state
> - differences between legacy L2VPN models and EVPN-based control planes
> - design tradeoffs between EVPN-MPLS and EVPN-VXLAN deployments
> - operational checks for split-horizon, aliasing, and DF behavior

## Theory

### Why EVPN Exists

VPLS works. Millions of services run on it. But it has fundamental architectural problems that can't be patched:

| VPLS Problem | Why It Hurts | EVPN Solution |
|-------------|-------------|---------------|
| Flood-and-learn MAC learning | No control plane visibility into MACs; can't filter, can't policy, can't optimize | BGP advertises MACs — full control plane awareness |
| Full-mesh PWs (N×(N-1)/2) | O(N²) signaling and state for N PEs | BGP RR eliminates full mesh — O(N) state |
| STP required for multi-homing | Active/standby only; wasted bandwidth | All-active multi-homing with DF election |
| BUM flooding everywhere | Every BUM frame hits every PE | Ingress replication with known-unicast optimization; ARP suppression |
| No IP awareness | Can't do ARP suppression, can't do integrated routing | MAC+IP advertisement enables proxy ARP/ND and IRB |
| No MAC mobility | Silent moves cause stale entries and black holes | MAC Mobility extended community tracks moves |
| Separate L2/L3 control planes | VPLS for L2, L3VPN for L3 — two services, two configs | Unified EVPN with IRB handles both |

That table is the entire sales pitch. EVPN takes the data plane simplicity of VPLS (Ethernet frames over MPLS or VXLAN) and replaces the flood-and-learn control plane with BGP. Same forwarding, infinitely better control.

### EVPN Architecture

```
                    ┌─────────────────────────────────┐
                    │         BGP Route Reflector      │
                    │   (EVPN AFI 25 / SAFI 70)       │
                    └──────┬──────────────┬────────────┘
                           │              │
                    BGP EVPN sessions     │
                           │              │
                    ┌──────┴──────┐ ┌─────┴──────┐
                    │    PE1      │ │    PE2      │
                    │ EVI / BD    │ │ EVI / BD    │
                    │ MAC/IP table│ │ MAC/IP table│
                    └──────┬──────┘ └─────┬──────┘
                           │              │
                    ┌──────┴──────┐ ┌─────┴──────┐
                    │    CE1      │ │    CE2      │
                    │  (Site A)   │ │  (Site B)   │
                    └─────────────┘ └─────────────┘
```

**Key components:**

- **EVI (EVPN Instance):** The logical service — equivalent to a VPLS instance. Each EVI has a unique RD and one or more import/export RTs.
- **Bridge Domain (BD):** The Layer 2 broadcast domain on a PE. An EVI can contain one or more BDs (for VLAN-aware bundling).
- **MAC-VRF:** The forwarding table for an EVI — stores locally learned and BGP-received MAC/IP bindings.
- **ESI (Ethernet Segment Identifier):** A 10-byte value identifying a multi-homed link. Two PEs connecting to the same CE with the same ESI = multi-homing group.
- **EAD (Ethernet Auto-Discovery):** Route Type 1 — advertises ESI membership and enables mass withdrawal, aliasing, and split-horizon filtering.

### EVPN Route Types

This is the core of EVPN. Five route types, each solving a specific problem. Know these cold.

#### Route Type 1 — Ethernet Auto-Discovery (EAD)

**Purpose:** Multi-homing magic. Advertised per-EVI and per-ES (Ethernet Segment).

**Two flavors:**
- **Per-ES EAD (ESI, RD, ETI=0):** "I'm attached to this Ethernet Segment." Used for split-horizon filtering and fast mass withdrawal.
- **Per-EVI EAD (ESI, RD, ETI=EVI tag):** "This EVI is reachable via this ES." Enables aliasing — remote PEs can load-balance to all PEs in the ES.

**When you see it:** Always present when ESI ≠ 0 (i.e., any multi-homed scenario). Single-homed CEs (ESI=0) don't generate Type 1.

**NLRI format:**
```
+---------------------------------------+
| RD (8 bytes)                          |
+---------------------------------------+
| ESI (10 bytes)                        |
+---------------------------------------+
| Ethernet Tag ID (4 bytes)             |
+---------------------------------------+
| MPLS Label (3 bytes)*                 |
+---------------------------------------+
```
*\* 20-bit label in 3-byte field per standard MPLS encoding. For per-EVI EAD, this is the aliasing label.*

**Key behavior:**
- Per-ES EAD withdrawal = **mass withdrawal**. If PE1 loses its link to a dual-homed CE, it withdraws the per-ES EAD route. Every remote PE immediately removes PE1 as a next-hop for ALL MACs on that ES. No per-MAC withdrawal needed — one route, all MACs. This is orders of magnitude faster than VPLS MAC flush.
- Per-EVI EAD carries a label that enables **aliasing**: remote PEs send traffic to any PE in the ES, even if only one PE learned the MAC locally. The receiving PE forwards it across the ES link.

#### Route Type 2 — MAC/IP Advertisement

**Purpose:** The workhorse. Advertises MAC addresses (and optionally their bound IP addresses) learned from CEs.

**This is what makes EVPN fundamentally different from VPLS.** Instead of flooding a frame and letting every PE learn the source MAC from the data plane, the PE that learns a MAC advertises it in BGP. Every other PE in the EVI installs the MAC→next-hop binding *before* any traffic arrives.

**NLRI format:**
```
+---------------------------------------+
| RD (8 bytes)                          |
+---------------------------------------+
| ESI (10 bytes)                        |
+---------------------------------------+
| Ethernet Tag ID (4 bytes)             |
+---------------------------------------+
| MAC Address Length (1 byte)           |
+---------------------------------------+
| MAC Address (6 bytes)                 |
+---------------------------------------+
| IP Address Length (1 byte)            |
+---------------------------------------+
| IP Address (0, 4, or 16 bytes)       |
+---------------------------------------+
| MPLS Label 1 (3 bytes)*              |
+---------------------------------------+
| MPLS Label 2 (3 bytes*, optional)    |
+---------------------------------------+
```

**Key fields:**
- **MAC Address:** The learned CE MAC.
- **IP Address:** Optional — the ARP/ND-resolved IP bound to this MAC. This enables ARP suppression and integrated routing.
- **ESI:** Which Ethernet Segment this MAC was learned on (ESI=0 for single-homed).
- **MPLS Label 1:** The EVPN service label (identifies the EVI/BD on the egress PE). *3 bytes but only 20 bits are the label (standard MPLS encoding — top 20 bits label, bottom 4 bits flags/TC/S/TTL packed per RFC 3032).*
- **MPLS Label 2:** Optional L3VPN label for integrated routing and bridging (IRB). Same 20-bit encoding.

**MAC+IP vs MAC-only:**
- MAC-only (IP length=0): Pure L2 service. Remote PEs install MAC forwarding entry.
- MAC+IP: Enables ARP/ND suppression. When a remote host ARPs for this IP, the local PE can respond with a proxy ARP instead of flooding the request.

**Extended communities carried:**
- **MAC Mobility:** Sequence number tracking MAC moves between PEs. Higher sequence wins. Prevents loops during silent moves.
- **ESI Label:** For split-horizon — frames from an ES should not be sent back to the same ES.
- **Default Gateway:** Marks the MAC as a gateway MAC (for anycast gateway scenarios).

#### Route Type 3 — Inclusive Multicast Ethernet Tag (IMET)

**Purpose:** BUM (Broadcast, Unknown unicast, Multicast) traffic delivery. Sets up the flooding tree.

**Every PE in an EVI advertises a Type 3 route** saying "I exist in this EVI, and here's how to send me BUM traffic." Remote PEs use these to build their ingress replication lists (or to join multicast trees if using P2MP LSPs).

**NLRI format:**
```
+---------------------------------------+
| RD (8 bytes)                          |
+---------------------------------------+
| Ethernet Tag ID (4 bytes)             |
+---------------------------------------+
| IP Address Length (1 byte)            |
+---------------------------------------+
| Originating Router's IP Address       |
+---------------------------------------+
```

**PMSI Tunnel attribute:**
This is carried as a BGP attribute (not in the NLRI). Specifies the BUM delivery mechanism:
- **Ingress Replication (type 6):** PE sends a copy of each BUM frame to every other PE. Simple, no state in the core, but O(N) replication at the ingress PE.
- **P2MP mLDP (type 2):** Multicast LDP tree. Core routers replicate — reduces PE load but adds P-network state.
- **P2MP RSVP-TE (type 1):** TE-based multicast tree. Traffic-engineered BUM delivery.
- **PIM-based (types 3-5):** Various PIM-based options. Less common in SP EVPN.

**In practice:** Ingress replication dominates for EVPN-MPLS in SP networks (few PEs per service, BUM traffic is small). Data center EVPN-VXLAN also uses ingress replication almost universally. P2MP only makes sense for very large broadcast domains or multicast-heavy services.

#### Route Type 4 — Ethernet Segment (ES)

**Purpose:** Designated Forwarder (DF) election for multi-homed Ethernet Segments.

When multiple PEs connect to the same CE (same ESI), they need to agree on which PE forwards BUM traffic to that CE. Without DF election, every PE would send a copy → duplicate frames to the CE.

**NLRI format:**
```
+---------------------------------------+
| RD (8 bytes)                          |
+---------------------------------------+
| ESI (10 bytes)                        |
+---------------------------------------+
| IP Address Length (1 byte)            |
+---------------------------------------+
| Originating Router's IP Address       |
+---------------------------------------+
```

**DF election process:**
1. All PEs connected to the same ESI exchange Type 4 routes
2. Routes carry the originating PE's IP address
3. PEs sort the list of participating PEs and run the election algorithm
4. **Default algorithm (RFC 7432):** `VLAN mod N` — each VLAN is assigned to a PE round-robin. Simple, deterministic, but uneven distribution with few VLANs.
5. **Preference-based (RFC 8584):** PEs advertise a preference weight. Highest wins. Better for active/standby with explicit primary.
6. **HRW / Highest Random Weight (RFC 8584):** Hash-based. Much better distribution than mod-based. **Recommended for new deployments.**

**BUM forwarding with DF:**
- Only the DF for a given VLAN/BD forwards BUM traffic to the CE
- Non-DF PEs on the same ES silently discard BUM toward that CE
- For unicast: all PEs can forward (all-active), or only one (single-active) — determined by ES configuration, not DF election

#### Route Type 5 — IP Prefix

**Purpose:** L3VPN-style prefix advertisement within the EVPN framework. This is what makes EVPN a unified L2+L3 control plane.

**Why not just use L3VPN (VPNv4/v6)?** You can. But Type 5 lets you advertise IP prefixes in the same EVPN address family, with the same RD/RT structure, through the same BGP sessions. One control plane for L2 MAC reachability and L3 IP reachability. Particularly powerful for data center fabrics with distributed gateways, but also used in SP environments for integrated routing and bridging (IRB) services.

**NLRI format:**
```
+---------------------------------------+
| RD (8 bytes)                          |
+---------------------------------------+
| ESI (10 bytes)                        |
+---------------------------------------+
| Ethernet Tag ID (4 bytes)             |
+---------------------------------------+
| IP Prefix Length (1 byte)             |
+---------------------------------------+
| IP Prefix (4 or 16 bytes)            |
+---------------------------------------+
| GW IP Address (4 or 16 bytes)         |
+---------------------------------------+
| MPLS Label (3 bytes)                  |
+---------------------------------------+
```

**Key fields:**
- **IP Prefix:** The advertised subnet (e.g., 10.0.0.0/24 or 2001:db8::/48)
- **GW IP Address:** Next-hop for recursive resolution. Often 0.0.0.0 (resolved via overlay).
- **ESI:** Usually 0 for prefix routes (not tied to a specific segment)
- **MPLS Label:** L3 VPN label for the prefix (same role as VPNv4 label)

**Type 5 vs VPNv4:**
| Aspect | Type 5 | VPNv4/VPNv6 |
|--------|--------|-------------|
| Address family | L2VPN EVPN (AFI 25, SAFI 70) | VPNv4 (AFI 1, SAFI 128) |
| Coexists with L2 | Yes — same session, same RTs | Separate AFI/SAFI |
| ESI awareness | Yes — can pin prefix to an ES | No |
| GW IP field | Yes — enables recursive resolution | No (uses BGP NH) |
| Deployment | Newer, growing in SP | Mature, ubiquitous |

**In practice:** Most SP networks still use VPNv4/v6 for pure L3VPN and reserve EVPN Type 5 for scenarios requiring integrated L2+L3. Data center fabrics use Type 5 heavily (every IP-VRF route is a Type 5).

### Route Type Summary

| Type | Name | Purpose | Multi-homing? | When Generated |
|------|------|---------|---------------|----------------|
| 1 | Ethernet Auto-Discovery | Fast withdrawal, aliasing, split-horizon | Yes (ESI≠0 only) | PE attached to multi-homed ES |
| 2 | MAC/IP Advertisement | MAC (and optional IP) reachability | Carries ESI | CE MAC learned (local or remote) |
| 3 | Inclusive Multicast | BUM delivery tree setup | No | PE joins an EVI |
| 4 | Ethernet Segment | DF election | Yes (defines the ES) | PE attached to multi-homed ES |
| 5 | IP Prefix | L3 prefix reachability | Optional ESI | IP route in EVPN IP-VRF |

### BGP Address Family

EVPN uses **AFI 25 (L2VPN), SAFI 70 (EVPN)**. All five route types share this single address family. This is negotiated during BGP session establishment via capabilities.

```
BGP Capability: Multi-protocol Extensions
  AFI: L2VPN (25)
  SAFI: EVPN (70)
```

**RD and RT usage:** Identical to L3VPN. Each EVI gets a unique RD (for BGP best-path uniqueness) and one or more RTs (for import/export filtering).

- **RD format:** Typically Type 1 (IP:number) — e.g., `10.0.0.1:100` for EVI 100 on PE 10.0.0.1
- **RT format:** Type 0 (ASN:number) or Type 1 (IP:number) — e.g., `65000:100`

### Extended Communities

EVPN introduces several new BGP extended communities:

| Extended Community | Type | Purpose |
|-------------------|------|---------|
| MAC Mobility | 0x06, 0x00 | Sequence number for MAC moves. Higher seq = newer location. Sticky bit = static MAC (never moves). |
| ESI Label | 0x06, 0x01 | Carries label for split-horizon. Single-active flag indicates ES mode. |
| ES-Import | 0x06, 0x02 | Route target derived from ESI — ensures Type 1/4 routes reach only PEs sharing the same ES. |
| Default Gateway | 0x03, 0x0D | Marks a MAC as a gateway (enables anycast gateway optimization). **Note: Opaque type, not EVPN type.** |
| EVPN Layer 2 Attributes | 0x06, 0x04 | MTU, control word flag, backup PE flag. |

### Control Plane Operation: MAC Learning

**Local learning (same as traditional bridging):**
1. CE sends a frame to PE
2. PE learns source MAC on the local attachment circuit (AC)
3. PE creates a Type 2 route with the MAC (and IP if ARP/ND snooped)
4. PE advertises the Type 2 route to all EVPN peers via BGP

**Remote learning (this is the EVPN magic):**
1. PE receives Type 2 route from BGP
2. PE installs MAC → remote-PE-next-hop in the MAC-VRF
3. When a local host sends traffic to that MAC, PE encapsulates and forwards directly to the remote PE
4. **No flooding required for known unicast**

**MAC aging:**
- Locally learned MACs age normally (default 300s, configurable)
- BGP-learned MACs do **not** age — they're removed only when the originating PE withdraws the route
- This asymmetry is intentional: BGP is the source of truth, not data plane activity

**MAC move detection:**
1. MAC appears on PE2 that was previously advertised by PE1
2. PE2 advertises Type 2 with MAC Mobility sequence incremented (+1 from PE1's sequence)
3. PE1 receives the update, sees higher sequence, withdraws its own route
4. All PEs update: MAC → PE2
5. **Loop detection:** If sequence exceeds a threshold (rapid bouncing), the MAC is flagged as "duplicate" and may be frozen

### Data Plane: BUM Handling

When a PE receives a BUM frame from a CE:

1. **Broadcast/Unknown-unicast:** PE looks up the ingress replication list (built from Type 3 routes) and sends a copy to each remote PE
2. **ARP/ND (with suppression enabled):** PE checks its local EVPN table. If the target IP has a Type 2 route with MAC+IP, PE responds with a proxy ARP/ND reply **without flooding**
3. **Multicast:** Same as broadcast — ingress replication to all PEs in the EVI (unless multicast-specific optimization like IGMP snooping + selective replication is configured)

**ARP suppression is a game-changer.** In a 500-host VPLS, every ARP request floods to every PE. With EVPN ARP suppression, the local PE handles >90% of ARP requests locally. BUM traffic drops by an order of magnitude.

### Service Types

EVPN supports three service interface models:

**VLAN-Based Service (one VLAN per EVI):**
```
EVI 100 ←→ VLAN 100
EVI 200 ←→ VLAN 200
```
- Each VLAN maps to its own EVPN instance
- Simple, clean isolation
- Most common in SP networks

**VLAN-Bundle Service (multiple VLANs per EVI):**
```
EVI 100 ←→ VLAN 100, 101, 102
```
- Multiple VLANs share one EVI (one RD, one set of RTs)
- MAC addresses are unique across all VLANs (no per-VLAN MAC scoping)
- Lower control plane overhead, but less isolation

**VLAN-Aware Bundle Service (multiple VLANs per EVI, with VLAN scoping):**
```
EVI 100 ←→ BD 100 (VLAN 100), BD 101 (VLAN 101), BD 102 (VLAN 102)
```
- Multiple VLANs in one EVI, but each gets its own Bridge Domain
- Ethernet Tag ID in the NLRI differentiates VLANs
- MAC uniqueness is per-BD, not per-EVI
- Best of both worlds: operational simplicity with proper isolation

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| ARP suppression | Disabled | **Enabled** | Reduces BUM flooding by 80-90% in most deployments |
| MAC aging (local) | 300s | 300-1800s | Match to CE MAC aging; too low = excessive BGP churn |
| MAC mobility threshold | 5 moves/s | 5 moves/s | Detects loops; lower = faster detection but more false positives |
| MAC duplication freeze time | 30s (IOS-XR) | 30-60s | How long to freeze a duplicate MAC. Too short = keeps bouncing |
| BUM replication | Ingress replication | Ingress replication | P2MP only worth it for very large BUM-heavy services |
| DF election | mod-based | **HRW (RFC 8584)** | Much better VLAN-to-PE distribution than default mod |
| Advertise default gateway | Disabled | **Enabled** (if IRB) | Required for anycast gateway and ARP optimization |
| Unknown-unicast suppression | Disabled | Consider enabling | Drops unknown unicast instead of flooding — safe only if all MACs are in BGP |
| send-community both | Off | **Required** | EVPN relies entirely on extended communities; without this, nothing works |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| No EVPN routes received | BGP session not negotiating L2VPN EVPN AFI/SAFI | Verify `family evpn signaling` (Junos) or `address-family l2vpn evpn` (IOS-XR). Check `send-community both`. |
| Routes received but no MAC entries | RT mismatch — routes not imported | Verify export/import RT match. Check `show bgp l2vpn evpn` for routes with no import. |
| MAC learned locally but not advertised | `advertise-mac` not configured (IOS-XR) | Add `advertise-mac` under the EVI. Junos enables this by default. |
| Remote MAC installed but traffic black-holed | Transport LSP broken (no MPLS path to remote PE) | Verify underlay — `traceroute mpls` or `show mpls forwarding`. EVPN labels need a working transport tunnel. |
| Duplicate MAC detected (frozen) | MAC moving between PEs too fast — loop or misconfigured ESI | Check if two PEs have the same ESI but different link connections. `show evpn evi mac [mac] detail` shows move history. |
| BUM flooding not working | Type 3 IMET routes missing | Verify EVI is up on all PEs. Check BD is not in "shutdown" state. |
| ARP suppression not working | No IP in Type 2 routes | Verify PE is doing ARP/ND snooping. IP must be learned (Type 2 MAC+IP) before suppression works. |
| DF election stuck | Type 4 ES routes not exchanged | Verify ESI is configured identically on both PEs. ES-Import RT must match (derived from ESI). |

### Methodology

1. **Verify BGP EVPN session** — Is the session up? Are routes being exchanged? (`show bgp l2vpn evpn summary`)
2. **Check route types** — Are Type 2 (MAC), Type 3 (IMET) present? For multi-homing: Type 1 (EAD) and Type 4 (ES)?
3. **Verify RT import/export** — Routes exist in BGP but not in the EVI table? RT mismatch. Compare `show bgp l2vpn evpn route-type 2` with `show evpn evi mac`.
4. **Check bridge domain state** — Is the BD up? Are attachment circuits up? (`show l2vpn bridge-domain`)
5. **Verify transport** — EVPN is an overlay. The underlay MPLS (or VXLAN) path must be working. Trace the transport tunnel.
6. **Check MAC table** — Are MACs learning locally? Are remote MACs being installed? Is any MAC stuck in "duplicate" state?
7. **Test BUM path** — Ping from CE. Does ARP resolve? If not, BUM replication may be broken. Check IMET routes.
8. **Check ESI (if multi-homed)** — Is the ESI consistent? Are Type 1/4 routes being exchanged? Is DF election complete?
9. **Escalate to packet capture** — If all control plane looks correct but traffic fails, capture on the PE egress to verify encapsulation (label stack, control word, etc.)

### War Stories

> **The Case of the Disappearing MACs**
>
> A regional SP deployed EVPN-MPLS to replace aging VPLS for a banking customer with 12 sites. Everything worked perfectly for two weeks. Then, randomly, a site would lose connectivity for exactly 30 seconds before recovering. The pattern was maddeningly inconsistent — different sites, different times of day, no correlation with traffic load.
>
> The smoking gun was in `show evpn evi mac detail`: MAC Mobility sequence numbers were incrementing on the gateway MAC. The banking customer had deployed a pair of firewalls at their HQ in active/passive mode. The passive firewall was sending periodic GARP (Gratuitous ARP) packets on its monitoring interface — which was trunked to the SP PE. Every GARP from the passive firewall triggered a MAC move event: PE2 would advertise the gateway MAC with sequence +1, PE1 would withdraw, all sites would reconverge to PE2, then the active firewall would send traffic and PE1 would reclaim the MAC.
>
> After enough bounces, the MAC hit the duplication threshold and was frozen for 30 seconds — the exact outage window the customer reported.
>
> **Fix:** Added a MAC ACL on the PE to filter GARPs from the passive firewall's monitoring interface. The customer also fixed their firewall config to suppress GARPs on the monitoring port. The real lesson: **MAC Mobility is the first place to look when EVPN connectivity is intermittent.** Run `show evpn evi mac [gw-mac] detail` and check the sequence number history.

## Lab Exercise

- **Topology:** 3-PE triangle (PE1, PE2, PE3) with Route Reflector, one CE per PE, single EVI
- **Objective:** Configure EVPN-MPLS with VLAN-based service, verify MAC advertisement via BGP, test ARP suppression
- **Validation:**
  1. All PEs have Type 3 IMET routes from each other PE
  2. CE-to-CE ping works (MAC learned via Type 2, not flooding)
  3. ARP suppression: ARP request from CE1 for CE2's IP is answered by local PE without flooding
  4. MAC move: Disconnect CE2 from PE2, reconnect to PE3 — verify MAC Mobility sequence increments and traffic converges
