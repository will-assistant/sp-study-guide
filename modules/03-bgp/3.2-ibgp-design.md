# 3.2 — iBGP Design: Route Reflectors & Confederations

## Overview
> BGP's full-mesh requirement for iBGP is the single biggest scalability problem in the protocol. With *n* routers, you need n(n-1)/2 sessions. 10 routers = 45 sessions. 50 routers = 1,225 sessions. 200 routers = 19,900 sessions. That's not just operationally painful — it's a convergence nightmare. Route Reflectors (RFC 4456) and Confederations (RFC 5065) solve this, and every SP on the planet uses one or both. Route Reflectors win in almost every modern design.


## Key Concepts
> **After this section you will understand:**
> - how BGP sessions form and exchange reachability at provider scale
> - how next-hop, path attributes, and policy drive best-path decisions
> - design tradeoffs for route reflectors, peering models, and scaling
> - how to troubleshoot control-plane versus policy-induced routing issues

## Theory

### The Full-Mesh Problem

iBGP has a fundamental rule: **a route learned from an iBGP peer is never advertised to another iBGP peer**. This prevents loops (since iBGP doesn't modify AS_PATH), but it means every router must hear every route directly from the originator — hence, full mesh.

```
Full mesh: n(n-1)/2 sessions
  10 routers →      45 sessions
  50 routers →   1,225 sessions
 100 routers →   4,950 sessions
 500 routers → 124,750 sessions  ← insane
```

### Route Reflectors (RFC 4456)

The dominant solution. A Route Reflector (RR) acts as a "hub" — clients peer with the RR instead of each other.

#### How It Works

```
                    ┌─────┐
          ┌─────────│ RR  │─────────┐
          │         └──┬──┘         │
          │            │            │
       ┌──┴──┐     ┌──┴──┐     ┌──┴──┐
       │ PE1 │     │ PE2 │     │ PE3 │
       │(cli)│     │(cli)│     │(cli)│
       └─────┘     └─────┘     └─────┘
```

**Reflection rules:**
| Route Learned From | Reflected To |
|--------------------|----|
| eBGP peer | All clients + all non-client iBGP peers |
| Client | All other clients + all non-client iBGP peers |
| Non-client iBGP peer | All clients only (NOT to other non-clients) |

**Loop prevention attributes:**
- **ORIGINATOR_ID** (type 9): Set to the router-id of the route originator. If a router sees its own router-id in ORIGINATOR_ID, it drops the route.
- **CLUSTER_LIST** (type 10): Appended by each RR (with its cluster-id). If a router sees its own cluster-id in CLUSTER_LIST, it drops the route.

#### Hierarchical Route Reflectors

For very large networks (500+ PEs), a single RR tier doesn't scale. Use hierarchical RR:

```
                    ┌──────┐
              ┌─────│ TRR  │─────┐       ← Top-level RR (Tier 2)
              │     └──────┘     │
           ┌──┴──┐           ┌──┴──┐
           │ RR1 │           │ RR2 │     ← Regional RRs (Tier 1)
           └──┬──┘           └──┬──┘
         ┌───┼───┐         ┌───┼───┐
        PE1 PE2 PE3       PE4 PE5 PE6    ← Clients
```

**Design rules for hierarchical RR:**
1. Each regional RR is a client of the top-level RR
2. PEs are clients of their regional RR only
3. ORIGINATOR_ID and CLUSTER_LIST still prevent loops across tiers
4. Top-level RRs should be redundant (at least 2)

#### RR Placement: The Big Decision

**Option A: On-path RR (RR is also a P router)**
- Pros: No extra devices, RR has natural topology awareness
- Cons: Forwarding + control plane load on same box, RR failure = forwarding impact

**Option B: Dedicated off-path RR (RR does NOT forward traffic)**
- Pros: Purpose-built for control plane, can use high-memory/low-bandwidth hardware, no forwarding impact if RR fails
- Cons: Extra devices, potential suboptimal path selection (see below)

**SP best practice: Dedicated RRs. Always.** A route reflector carrying 2M VPNv4 routes + 500K IPv4 routes + EVPN needs memory and CPU for control plane, not fast forwarding silicon. Put them on Linux-based or VM-based platforms (Juniper vRR, Cisco IOS-XRv, GoBGP, BIRD).

#### The Optimal Path Problem

This is the RR gotcha that bites people on the Internet Expert lab:

**Scenario**: PE1 and PE2 both advertise the same prefix to RR. PE1 has IGP cost 10 to the RR. PE2 has IGP cost 100 to the RR. The RR picks PE1 (lower IGP cost). But PE3, which is a client, might be IGP-closer to PE2. The RR already chose PE1 as best and only reflects PE1's route. PE3 never sees PE2's route and takes a suboptimal path.

**Solutions:**
1. **ADD-PATH (RFC 7911)** — RR sends multiple paths per prefix. PE3 can run its own best-path selection with all candidates. **This is the correct modern solution.**
2. **BGP Optimal Route Reflection (RFC 9107)** — RR calculates best path *per client* based on client's IGP cost. Computationally expensive.
3. **Multiple RRs in different locations** — Each PE hears from the topologically closest RR. Helps but doesn't fully solve it.

### Confederations (RFC 5065)

Split your AS into sub-ASes. Within each sub-AS, full mesh or RR. Between sub-ASes, eBGP-like sessions.

```
                   ┌── AS 65000 ──────────────────────┐
                   │                                   │
                   │  ┌─Sub-AS 65001─┐  ┌─Sub-AS 65002─┐  │
                   │  │  R1 ── R2    │──│  R3 ── R4    │  │
                   │  │  (iBGP mesh) │  │  (iBGP mesh) │  │
                   │  └──────────────┘  └──────────────┘  │
                   │                                   │
                   └───────────────────────────────────┘
                         Appears as AS 65000 to outside
```

**Key behaviors:**
- Between sub-ASes: eBGP rules apply (next-hop changes, AS_PATH modified)
- The sub-AS numbers are stripped from AS_PATH before advertising externally
- LOCAL_PREF, MED, and NEXT_HOP are preserved across confederation boundaries (unlike true eBGP)
- `CONFED_SEQUENCE` and `CONFED_SET` added to AS_PATH (don't count for path length in best-path)

**When to use confederations:**
- Very large SP formed by merging multiple networks (each sub-AS was a separate network)
- Administrative separation with shared external identity
- When you want eBGP-like policy control between network regions

**When NOT to use:**
- Pretty much every other case. Route reflectors are simpler, more flexible, better tooled, and better understood by operations teams.

### RR vs. Confederation Decision

| Criteria | Route Reflectors | Confederations |
|----------|-----------------|----------------|
| Operational complexity | Low | High |
| Configuration changes | Add client, done | New sub-AS, eBGP config, policy |
| Path diversity | ADD-PATH solves it | Naturally better (eBGP behavior) |
| Industry adoption | ~95% of SPs | ~5% (legacy, merged networks) |
| Vendor support | Excellent | Good but fewer knobs |
| Troubleshooting difficulty | Medium | High (AS_PATH gets weird) |
| Scaling ceiling | Hierarchical RR = unlimited | Sub-AS count manageable |

**Verdict**: Route Reflectors with ADD-PATH. Unless you've just merged three ISPs and each one insists on their own routing policy, skip confederations.

## Configuration

### IOS-XE: Route Reflector

```cisco
! Dedicated Route Reflector - IOS-XE
router bgp 65000
 bgp router-id 10.0.0.100
 bgp log-neighbor-changes
 no bgp default ipv4-unicast
 bgp cluster-id 10.0.0.100
 !
 ! RR clients (PE routers)
 neighbor RR-CLIENTS peer-group
 neighbor RR-CLIENTS remote-as 65000
 neighbor RR-CLIENTS update-source Loopback0
 neighbor RR-CLIENTS password RR_CLIENT_KEY
 !
 neighbor 10.0.0.1 peer-group RR-CLIENTS
 neighbor 10.0.0.2 peer-group RR-CLIENTS
 neighbor 10.0.0.3 peer-group RR-CLIENTS
 neighbor 10.0.0.4 peer-group RR-CLIENTS
 !
 ! Redundant RR peer (non-client iBGP)
 neighbor 10.0.0.101 remote-as 65000
 neighbor 10.0.0.101 update-source Loopback0
 neighbor 10.0.0.101 description Redundant-RR2
 !
 address-family ipv4 unicast
  ! Reflect routes to clients
  neighbor RR-CLIENTS route-reflector-client
  neighbor RR-CLIENTS send-community both
  ! Note: next-hop-self on RR is design-dependent
  ! Generally OFF — let originating PE set NH. Enable only for inter-cluster scenarios.
  !
  ! ADD-PATH: send additional paths to clients
  neighbor RR-CLIENTS advertise additional-paths best 3
  bgp additional-paths select all
  bgp additional-paths send receive
  !
  neighbor 10.0.0.1 activate
  neighbor 10.0.0.2 activate
  neighbor 10.0.0.3 activate
  neighbor 10.0.0.4 activate
  neighbor 10.0.0.101 activate
 exit-address-family
 !
 address-family vpnv4 unicast
  neighbor RR-CLIENTS route-reflector-client
  neighbor RR-CLIENTS send-community extended
  !
  neighbor 10.0.0.1 activate
  neighbor 10.0.0.2 activate
  neighbor 10.0.0.3 activate
  neighbor 10.0.0.4 activate
  neighbor 10.0.0.101 activate
 exit-address-family
 !
 address-family l2vpn evpn
  neighbor RR-CLIENTS route-reflector-client
  neighbor RR-CLIENTS send-community extended
  !
  neighbor 10.0.0.1 activate
  neighbor 10.0.0.2 activate
  neighbor 10.0.0.3 activate
  neighbor 10.0.0.4 activate
  neighbor 10.0.0.101 activate
 exit-address-family
```

### Junos: Route Reflector

```junos
# Dedicated Route Reflector - Junos
protocols {
    bgp {
        log-updown;
        group RR-CLIENTS {
            type internal;
            local-address 10.0.0.100;
            cluster 10.0.0.100;
            hold-time 30;
            family inet {
                unicast {
                    add-path {
                        send {
                            path-count 3;
                        }
                        receive;
                    }
                }
            }
            family inet-vpn {
                unicast;
            }
            family evpn {
                signaling;
            }
            # All PE clients
            neighbor 10.0.0.1 {
                description "PE1";
                authentication-key "$9$ENCRYPTED";
            }
            neighbor 10.0.0.2 {
                description "PE2";
                authentication-key "$9$ENCRYPTED";
            }
            neighbor 10.0.0.3 {
                description "PE3";
                authentication-key "$9$ENCRYPTED";
            }
            neighbor 10.0.0.4 {
                description "PE4";
                authentication-key "$9$ENCRYPTED";
            }
        }
        group RR-MESH {
            type internal;
            local-address 10.0.0.100;
            description "Redundant RR peer";
            neighbor 10.0.0.101 {
                family inet {
                    unicast;
                }
                family inet-vpn {
                    unicast;
                }
                family evpn {
                    signaling;
                }
            }
        }
    }
}

routing-options {
    router-id 10.0.0.100;
    autonomous-system 65000;
}
```

### IOS-XE: Confederation

```cisco
! Confederation member - IOS-XE
router bgp 65001                            ! Sub-AS number
 bgp router-id 10.0.0.1
 bgp confederation identifier 65000        ! External AS seen by outside world
 bgp confederation peers 65002 65003       ! Other sub-ASes
 !
 ! iBGP within sub-AS
 neighbor 10.0.0.2 remote-as 65001
 neighbor 10.0.0.2 update-source Loopback0
 !
 ! Confederation eBGP to other sub-AS
 neighbor 10.1.0.1 remote-as 65002
 neighbor 10.1.0.1 update-source Loopback0
 !
 address-family ipv4 unicast
  neighbor 10.0.0.2 activate
  neighbor 10.0.0.2 next-hop-self
  neighbor 10.1.0.1 activate
  neighbor 10.1.0.1 next-hop-self-all
 exit-address-family
```

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| `cluster-id` | Router-ID | Explicit (same on redundant RRs) | Must match for redundant RRs sharing clients |
| `route-reflector-client` | — | Per neighbor/group under each AF | Enables reflection |
| ADD-PATH send count | — | 2-6 paths | More = better path diversity, more memory |
| `bgp additional-paths select all` | — | Enable on RR | RR must select additional paths to send |
| RR count per cluster | — | 2 minimum | Redundancy; clients peer with both |
| `next-hop-self` on RR | Off | Depends on design | Generally OFF (let PE set its own NH), but useful for inter-cluster |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Route not reflected to all clients | AF not configured as `route-reflector-client` | Verify under each address-family |
| Route loop detected (route dropped) | ORIGINATOR_ID matches local router-id | Check for duplicate router-IDs |
| Suboptimal path selection | RR selected non-optimal path for client | Enable ADD-PATH or use multiple RRs |
| RR failover causes convergence storm | All clients lose all routes simultaneously | Use redundant RRs (2+), stagger peer timers, enable GR/LLGR |
| Confederation AS_PATH looks wrong | Sub-AS numbers visible in debug but stripped externally | Normal behavior — verify with external peer |

### War Story: The Cluster-ID That Wasn't

An SP deployed two redundant route reflectors, RR1 and RR2. Both were clients of each other (for redundancy). But the engineer forgot to set `cluster-id` explicitly — each RR used its own router-id as cluster-id. 

Result: When PE1 sent a route to RR1, RR1 reflected it with CLUSTER_LIST [10.0.0.100]. When RR2 received it, it reflected it again with CLUSTER_LIST [10.0.0.101, 10.0.0.100]. Every route had two copies in every client's Adj-RIB-In. Memory usage doubled. Convergence slowed.

**Fix**: Set the same `cluster-id` on both RRs. Now when RR2 receives a route with CLUSTER_LIST containing the shared cluster-id, it recognizes it as a duplicate and doesn't re-reflect.

**Lesson**: Redundant RRs serving the same set of clients MUST share a cluster-id.
