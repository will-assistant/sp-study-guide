# 3.1 — BGP Fundamentals at SP Scale

## Overview
> BGP is the protocol that holds the internet together. Full stop. In a service provider, BGP does double duty: it's your external routing protocol for exchanging prefixes with peers, transit providers, and customers — and it's your internal service signaling layer for VPNs, EVPN, traffic engineering, and flowspec. If IS-IS is the plumbing, BGP is the nervous system. Every SP engineer must understand BGP not just at a configuration level but at a *behavioral* level — how it converges, how it fails, and how it scales to millions of routes.


## Key Concepts
> **After this section you will understand:**
> - how BGP sessions form and exchange reachability at provider scale
> - how next-hop, path attributes, and policy drive best-path decisions
> - design tradeoffs for route reflectors, peering models, and scaling
> - how to troubleshoot control-plane versus policy-induced routing issues

## Theory

### BGP in SP Context: Why It's Different

When you learn BGP in enterprise, you learn "advertise my networks to my ISP." In an SP, BGP is everything:

| Role | What BGP Does | Address Family |
|------|--------------|----------------|
| Internet routing | Full/partial/default table exchange | IPv4/IPv6 unicast |
| L3VPN signaling | VPNv4/v6 prefix distribution | VPNv4/VPNv6 unicast |
| EVPN | MAC/IP advertisement, multi-homing | L2VPN EVPN |
| Traffic engineering | SR policy distribution | SR-TE (link-state) |
| Flowspec | DDoS mitigation rules | IPv4/IPv6 flowspec |
| Multicast | MVPN signaling | MCAST-VPN |
| Labeled unicast | Inter-AS MPLS (Option B/C) | IPv4/IPv6 labeled-unicast |

A single BGP process on a PE router might carry 10+ address families. This is the BGP multiprotocol extension (RFC 4760) in action — and it's why BGP, not any IGP, is the true workhorse protocol in SP.

### BGP Finite State Machine

Every IE-SP candidate must know this cold:

```
Idle → Connect → OpenSent → OpenConfirm → Established
  ↑       ↓          ↓           ↓
  └───────┴──────────┴───────────┘ (any error → Idle)
         Active ←→ Connect
```

| State | What's Happening | Stuck Here? Check... |
|-------|-----------------|---------------------|
| **Idle** | No connection attempt | Peer config, admin shutdown, route-map deny on neighbor |
| **Connect** | TCP SYN sent | Reachability (IGP), ACL/firewall, TCP/179 blocked |
| **Active** | TCP retry (connect failed) | Same as Connect — this is the "trying again" state |
| **OpenSent** | TCP up, OPEN sent | AS mismatch, router-id conflict, capability mismatch |
| **OpenConfirm** | OPEN received, waiting for KEEPALIVE | Rarely stuck — auth failure, hold-time negotiation |
| **Established** | Peer is up, routes exchanging | You're good (until you're not) |

**Key insight**: If a peer oscillates between `Active` and `Connect`, it's a transport problem (reachability, firewall, TTL). If it gets to `OpenSent` and drops, it's a BGP-level mismatch (AS, capabilities, authentication).

### Path Attributes Deep Dive

BGP's decision process is attribute-driven. Understanding attribute propagation is non-negotiable at IE-SP level.

#### Well-Known Mandatory
| Attribute | Type Code | Description | Scope |
|-----------|-----------|-------------|-------|
| ORIGIN | 1 | How the route was injected (IGP/EGP/Incomplete) | Transitive |
| AS_PATH | 2 | Ordered list of ASes traversed | Transitive |
| NEXT_HOP | 3 | Next-hop IP address | Transitive |

#### Well-Known Discretionary
| Attribute | Type Code | Description | Scope |
|-----------|-----------|-------------|-------|
| LOCAL_PREF | 5 | iBGP preference (higher = better) | Non-transitive (iBGP only) |
| ATOMIC_AGGREGATE | 6 | Route was aggregated, detail lost | Transitive |

#### Optional Transitive
| Attribute | Type Code | Description |
|-----------|-----------|-------------|
| AGGREGATOR | 7 | AS + router-id of aggregator |
| COMMUNITY | 8 | 32-bit tags (RFC 1997) |
| EXTENDED COMMUNITY | 16 | 64-bit tags (RT, SoO for VPNs) |
| LARGE COMMUNITY | 32 | 96-bit tags (RFC 8092) |

#### Optional Non-Transitive
| Attribute | Type Code | Description |
|-----------|-----------|-------------|
| MED | 4 | Multi-Exit Discriminator (lower = better) |
| ORIGINATOR_ID | 9 | Loop prevention for route reflection |
| CLUSTER_LIST | 10 | Loop prevention for route reflection |
| AIGP | 26 | Accumulated IGP metric (RFC 7311) |

### Best Path Selection Algorithm

The BGP decision process. Memorize the order, understand the *why* behind each step:

```
1.  Highest Weight (Cisco-proprietary, local only)
2.  Highest LOCAL_PREF
3.  Locally originated (network/redistribute/aggregate)
4.  Shortest AS_PATH
5.  Lowest ORIGIN (IGP < EGP < Incomplete)
6.  Lowest MED (only compared within same neighbor AS, by default)
7.  eBGP over iBGP
8.  Lowest IGP metric to NEXT_HOP
9.  Oldest route (stability)
10. Lowest Router ID of advertising peer
11. Lowest peer IP address (tiebreaker)
```

**SP-specific considerations:**

- **Step 2 (LOCAL_PREF)** — This is your primary traffic engineering knob for inbound iBGP preference. Customer routes: 150. Peer routes: 100. Transit routes: 80. Simple, effective, done.
- **Step 4 (AS_PATH)** — Prepending works here. But prepending more than 3x is pointless — most peers don't look past 3 prepends. Also: `remove-private-as` at eBGP boundaries.
- **Step 6 (MED)** — The most misunderstood attribute. Only compared between routes from the *same neighbor AS* by default. Enable `always-compare-med` or `bgp bestpath med always` at your peril — it breaks determinism when MEDs come from different origins.
- **Step 8 (IGP cost)** — This is hot-potato routing. Traffic exits at the closest egress point. Crucial for understanding transit costs and traffic flow in multi-homed SPs.

### BGP Message Types

| Message | Function | Key Fields |
|---------|----------|------------|
| OPEN | Session establishment | Version, AS, Hold Time, Router ID, Capabilities |
| UPDATE | Route advertisement/withdrawal | NLRI, Path Attributes, Withdrawn Routes |
| KEEPALIVE | Session maintenance | Empty (19-byte header only) |
| NOTIFICATION | Error signaling (session teardown) | Error Code, Subcode, Data |
| ROUTE-REFRESH | Request re-advertisement (RFC 2918) | AFI, SAFI |

**OPEN Capabilities** (what to watch for):
- 4-byte ASN (RFC 6793) — mandatory for any AS > 65535
- Multiprotocol extensions (RFC 4760) — AFI/SAFI negotiation
- Route Refresh (RFC 2918)
- Graceful Restart (RFC 4724)
- ADD-PATH (RFC 7911) — send/receive multiple paths per prefix
- Long-Lived Graceful Restart (RFC 9494)
- Enhanced Route Refresh (RFC 7313)

### Multiprotocol BGP (MP-BGP)

The key architectural innovation that makes BGP the universal service signaling protocol.

**Address Family Identifier (AFI) / Subsequent AFI (SAFI):**

| AFI | SAFI | Address Family | Use Case |
|-----|------|---------------|----------|
| 1 | 1 | IPv4 Unicast | Internet routing |
| 1 | 2 | IPv4 Multicast | RPF for PIM |
| 1 | 4 | IPv4 Labeled Unicast | Inter-AS MPLS (Option C) |
| 1 | 128 | VPNv4 Unicast | L3VPN |
| 2 | 1 | IPv6 Unicast | Internet routing |
| 2 | 4 | IPv6 Labeled Unicast | 6PE, Inter-AS |
| 2 | 128 | VPNv6 Unicast | L3VPN for IPv6 |
| 25 | 65 | L2VPN EVPN | EVPN services |
| 1 | 73 | SR-TE Policy | Segment routing TE |
| 1 | 133 | IPv4 Flowspec | DDoS mitigation |
| 16388 | 71 | BGP-LS | Topology export to controllers |

Each address family maintains its own RIB (Adj-RIB-In, Loc-RIB, Adj-RIB-Out per AFI/SAFI). This is critical for understanding scale — a route reflector carrying VPNv4 + EVPN + IPv4 unicast has *three separate RIBs* to manage.

### Scale Numbers Every SP Engineer Should Know

| Metric | Typical Value | Notes |
|--------|--------------|-------|
| Full IPv4 table | ~950K prefixes (2024) | Growing ~50K/year |
| Full IPv6 table | ~220K prefixes (2024) | Growing faster |
| BGP convergence (full table) | 5-15 minutes | From cold start |
| BGP convergence (single prefix) | 1-30 seconds | Depends on timers, dampening |
| Max prefixes per VRF (practical) | 10K-100K | Depends on PE memory |
| Route Reflector capacity | 5-10M paths | Dedicated RR (no forwarding) |
| BGP sessions per router | 500-2000+ | Platform-dependent |
| Memory per 1M IPv4 routes | ~500MB-1GB | Varies by implementation |

### 4-Byte ASN

The original 2-byte ASN space (0-65535) is exhausted. All new assignments are 4-byte (RFC 6793).

**Ranges:**
| Range | Use |
|-------|-----|
| 0 | Reserved |
| 1-64495 | Public 2-byte |
| 64496-64511 | Documentation (RFC 5398) |
| 23456 | AS_TRANS (RFC 6793) |
| 64512-65534 | Private 2-byte |
| 65535 | Reserved |
| 65536-4199999999 | Public 4-byte |
| 4200000000-4294967294 | Private 4-byte |
| 4294967295 | Reserved |

**AS_TRANS (23456)**: When a 4-byte-capable speaker sends an UPDATE through a 2-byte-only speaker, the real ASN is encoded in `AS4_PATH` and `AS4_AGGREGATOR` attributes, while the AS_PATH gets `23456` as a placeholder. The receiving 4-byte speaker reconstructs the real path.

**Operational gotcha**: If you see `23456` in an AS_PATH in 2026, something is very wrong. All modern gear supports 4-byte ASN.

## Configuration

### IOS-XR: Basic BGP with MP-BGP

```cisco
!! Basic SP BGP configuration - IOS-XR (ASR 9000, NCS 5500)
router bgp 65000
 bgp router-id 10.0.0.1
 bgp log neighbor changes detail
 !
 ! Timers: 10s keepalive, 30s hold (aggressive for SP)
 timers bgp 10 30
 !
 ! iBGP peer to route reflector
 neighbor 10.0.0.100
  remote-as 65000
  update-source Loopback0
  password encrypted SECURE_KEY_HERE
  address-family ipv4 unicast
   send-community-ebgp
   next-hop-self                       !! Standard for iBGP
  !
  address-family vpnv4 unicast
   send-extended-community-ebgp
  !
  address-family ipv6 unicast
   send-community-ebgp
   next-hop-self
  !
 !
 ! eBGP peer to transit provider
 neighbor 198.51.100.1
  remote-as 64999
  description Transit-ProviderA
  ebgp-multihop 2                      !! If not directly connected
  password encrypted TRANSIT_KEY_HERE
  address-family ipv4 unicast
   send-community-ebgp
   route-policy TRANSIT-IN-POLICY in
   route-policy TRANSIT-OUT-POLICY out
   maximum-prefix 1000000 85
  !
 !
```

> **Also on IOS-XE:** IOS-XE uses flat neighbor statements with `address-family` / `exit-address-family` blocks. Key difference: IOS-XE requires `no bgp default ipv4-unicast` to prevent auto-activation of IPv4 unicast on all peers — IOS-XR doesn't auto-activate, so this isn't needed. IOS-XE also uses `send-community both` instead of IOS-XR's `send-community-ebgp` / `send-extended-community-ebgp`.

### Junos: Basic BGP with MP-BGP

```junos
# Basic SP BGP configuration - Junos
protocols {
    bgp {
        log-updown;
        # Global BGP settings
        group IBGP-RR {
            type internal;
            local-address 10.0.0.2;
            # Aggressive timers for SP
            hold-time 30;
            # iBGP to route reflector
            neighbor 10.0.0.100 {
                description "Route-Reflector-1";
                authentication-key "$9$ENCRYPTED";
            }
            # Address families
            family inet {
                unicast;
            }
            family inet6 {
                unicast;
            }
            family inet-vpn {
                unicast;
            }
            family evpn {
                signaling;
            }
            # Export next-hop-self for iBGP
            export NHS;
        }
        group EBGP-TRANSIT {
            type external;
            peer-as 64999;
            neighbor 198.51.100.1 {
                description "Transit-ProviderA";
                authentication-key "$9$ENCRYPTED";
                family inet {
                    unicast {
                        prefix-limit {
                            maximum 1000000;
                            teardown 85 idle-timeout 30;
                        }
                    }
                }
                import TRANSIT-IN-POLICY;
                export OUR-PREFIXES-OUT;
            }
        }
    }
}

routing-options {
    router-id 10.0.0.2;
    autonomous-system 65000;
}

# Next-hop-self policy
policy-options {
    policy-statement NHS {
        term SET-NHS {
            then {
                next-hop self;
            }
        }
    }
}
```

### Key Knobs

| Parameter | IOS-XR Default | Junos Default | Recommended (SP) | Why |
|-----------|---------------|---------------|-------------------|-----|
| IPv4 unicast auto-activate | Off (explicit AFs) | Off (explicit AFs) | Default is correct | IOS-XR requires explicit AF activation per neighbor (IOS-XE needs `no bgp default ipv4-unicast`) |
| Keepalive / Hold | 60s / 180s | 30s / 90s | 10s / 30s (iBGP), 30s / 90s (eBGP) | Faster failure detection; BFD preferred |
| `send-community both` | Off | Standard only | Always both/extended | Required for VPN RT, policy communities |
| `next-hop-self` | Off | Off | On for iBGP | iBGP peers can't resolve eBGP next-hops via IGP |
| `maximum-prefix` | None | None | Set per peer/AF | Protects against route leaks and fat fingers |
| `bgp bestpath as-path multipath-relax` | Off | `multipath multiple-as` | Enable for ECMP | Allows multipath across different neighbor ASes |
| `bgp graceful-restart` | Off | On | Enable (with LLGR) | Reduces churn during control plane restarts |

### What Good Looks Like

**Healthy iBGP peer (IOS-XR):**
```
BGP neighbor is 10.0.0.100,  remote AS 65000, internal link
  BGP version 4, remote router ID 10.0.0.100
  BGP state = Established, up for 3w2d
  Neighbor under address family IPv4 Unicast:
    Prefixes Current:  850234     (Coverage: full table)
    Session: Active
  Neighbor under address family VPNv4 Unicast:
    Prefixes Current:  42817
```

**Red flags to watch:**
- Prefix count drops suddenly → route leak, deaggregation, or upstream issue
- Frequent state changes → flapping (check `show bgp neighbor X flap-statistics`)
- `MaxPrefixExceeded` → peer sending more than expected (route leak?)
- `NOTIFICATION sent/received` → check error codes
- Established but 0 prefixes → filter problem, AF mismatch, or `no bgp default ipv4-unicast` missing activation

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Peer stuck in Active | TCP/179 blocked, wrong source address, no route to peer | Check ACLs, verify `update-source`, trace route to peer |
| Peer stuck in OpenSent | AS mismatch, capability mismatch | Check `remote-as`, verify AF config on both sides |
| Established but no routes | AF not activated, filter blocking all, `no bgp default ipv4-unicast` | Check `show bgp neighbor X af-flags`, verify `activate` under AF |
| Routes received but not installed | Next-hop unreachable, route filtered by import policy | Check `show bgp X bestpath`, verify `next-hop-self` on iBGP |
| Route oscillation | MED comparison across ASes, non-deterministic bestpath | Check `always-compare-med` setting, enable `bgp deterministic-med` |
| Memory exhaustion | Full table + VPN routes exceed platform capacity | Check `show process memory`, consider partial views, upgrade |

### Methodology
1. **Check peer state**: `show bgp summary` — is the peer Established?
2. **Check prefix counts**: Expected vs actual — sudden changes?
3. **Check specific prefix**: `show bgp X.X.X.X/YY` — is it in the table? What's the best path?
4. **Check filters**: `show bgp neighbor X received-routes` vs `show bgp neighbor X routes` — policy dropping it?
5. **Check next-hop**: Is the BGP next-hop reachable via IGP? `show ip route <next-hop>`
6. **Check path attributes**: Is LOCAL_PREF/MED/AS_PATH what you expect?
7. **Debug carefully**: `debug bgp updates` is EXTREMELY verbose on a production router. Use `debug bgp updates X.X.X.X` (neighbor-specific) and only for specific prefixes if possible.

### War Story: The MED That Ate Christmas

A Tier-2 SP peered with two upstream transit providers at different PoPs. Both upstreams advertised the same prefixes with MEDs. Provider A: MED 100 at West Coast. Provider B: MED 50 at East Coast.

The SP had `always-compare-med` enabled, so BGP preferred Provider B (lower MED) for *all* traffic — even traffic originating at the West Coast PoP. Result: traffic from West Coast customers hairpinned across the backbone to the East Coast egress, crossed Provider B's network to the West Coast, then back to the destination. Latency doubled. Bandwidth cost exploded.

**Fix**: Remove `always-compare-med`. MEDs should only be compared between routes from the *same* neighbor AS. Use LOCAL_PREF for inter-provider preference, not MED.

**Lesson**: MED is a *suggestion* from your neighbor about which of *their* entry points to use. It was never designed for inter-AS comparison.
