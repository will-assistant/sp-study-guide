# 3.3 — eBGP Peering: IX, Transit & Customers

## Overview
> eBGP peering is where the money is. Your peering strategy — who you peer with, where, and under what terms — directly impacts transit costs, performance, and resilience. An SP's peering fabric includes transit providers (you pay them), peers (settlement-free), customers (they pay you), and Internet Exchange (IX) points. Getting this right is half network engineering, half business negotiation, and half security paranoia. Yes, that's three halves. Welcome to SP.


## Key Concepts
> **After this section you will understand:**
> - how BGP sessions form and exchange reachability at provider scale
> - how next-hop, path attributes, and policy drive best-path decisions
> - design tradeoffs for route reflectors, peering models, and scaling
> - how to troubleshoot control-plane versus policy-induced routing issues

## Theory

### Peering Relationships

| Relationship | Traffic Flow | Routing | Revenue |
|-------------|-------------|---------|---------|
| **Transit** | You → them → internet | They send you full table (or default) | You pay them |
| **Peer** | You ↔ them (their customers + yours) | Exchange customer routes only | Free (settlement-free) |
| **Customer** | They → you → internet | They send you their prefixes | They pay you |
| **IX route server** | You ↔ many peers via RS | RS reflects all peer routes | IX membership fee |

**The business hierarchy** (RFC 7454 calls this "routing policy"):
```
Customer routes:  LOCAL_PREF 150  (always prefer — they pay you)
Peer routes:      LOCAL_PREF 100  (neutral)
Transit routes:   LOCAL_PREF 80   (last resort — you pay them)
```

This simple LOCAL_PREF scheme ensures: customer traffic stays local (revenue), peer traffic stays free, and transit is only used when no alternative exists.

### Private Peering vs. IX Peering

**Private (PNI — Private Network Interconnect):**
- Dedicated cross-connect or dark fiber between two networks
- Typical: 10G, 100G links
- Full control over capacity, monitoring, and failure detection
- Used for high-traffic peers (>1Gbps sustained)

**IX (Internet Exchange Point):**
- Shared L2 fabric (Ethernet switch)
- Single port, many peers (hundreds at large IXPs)
- Route server simplifies peering (one BGP session → routes from all RS peers)
- Used for long-tail peers (many networks, modest traffic each)
- Major IXPs: DE-CIX Frankfurt, AMS-IX, LINX, Equinix IX, etc.

### IX Route Servers

Instead of establishing bilateral BGP sessions with every peer at an IX (50+ sessions), you peer with the IX route server:

```
                    ┌───────────┐
            ┌───────│ Route     │───────┐
            │       │ Server    │       │
            │       └─────┬─────┘       │
            │             │             │
         ┌──┴──┐      ┌──┴──┐      ┌──┴──┐
         │ AS1 │      │ AS2 │      │ AS3 │
         └─────┘      └─────┘      └─────┘
```

**Route Server behavior:**
- Transparent: Does NOT insert its own ASN into AS_PATH
- Respects RPKI ROV (most modern RSes)
- Supports communities for selective peering (e.g., "don't announce to AS64999")
- Two RSes for redundancy (peer with both)
- Filtering: Uses IRR and RPKI for prefix validation

**IX community conventions (example):**
| Community | Meaning |
|-----------|---------|
| `RS:0:PEER_ASN` | Don't announce to specific peer |
| `RS:PEER_ASN:PEER_ASN` | Announce only to specific peer |
| `RS:0:0` | Don't announce to anyone (withdraw from RS) |

### Prefix Filtering: The Non-Negotiable

Prefix filtering on eBGP sessions is the single most important security measure in BGP. Not filtering = accepting route leaks, hijacks, and bogons.

#### What to Filter INBOUND (from peers/transit):

1. **Bogons** (RFC 5737, RFC 6890):
   ```
   0.0.0.0/8          # "This" network
   10.0.0.0/8         # RFC 1918
   100.64.0.0/10      # Shared address space (CGNAT)
   127.0.0.0/8        # Loopback
   169.254.0.0/16     # Link-local
   172.16.0.0/12      # RFC 1918
   192.0.2.0/24       # Documentation (TEST-NET-1)
   192.168.0.0/16     # RFC 1918
   198.18.0.0/15      # Benchmarking
   198.51.100.0/24    # Documentation (TEST-NET-2)
   203.0.113.0/24     # Documentation (TEST-NET-3)
   224.0.0.0/4        # Multicast
   240.0.0.0/4        # Reserved
   ```

2. **Your own prefixes** — If a peer advertises your address space back to you, that's a leak/hijack.

3. **Too-specific prefixes** — Reject anything more specific than /24 for IPv4, /48 for IPv6 (DFZ convention).

4. **Too-long AS_PATH** — Reject paths > 50 ASes (almost certainly garbage).

5. **RPKI Invalid** — Reject RPKI-invalid routes. Accept valid, accept unknown (for now).

6. **Maximum prefix limits** — Set per peer. If they suddenly send 10x expected prefixes, tear down the session.

#### What to Filter OUTBOUND (to peers/transit):

1. **Only your own + customer prefixes** — Never advertise peer or transit routes to other peers/transit (valley-free routing).
2. **Aggregated prefixes** — Advertise aggregates, not disaggregated specifics.
3. **No private ASNs** — Strip with `remove-private-as`.
4. **No bogons from your space** — Double-check you're not leaking internal infrastructure.

### RPKI (Resource Public Key Infrastructure)

RPKI provides cryptographic proof that an AS is authorized to originate a prefix. ROAs (Route Origin Authorizations) bind prefix → origin AS.

**ROV (Route Origin Validation) states:**
| State | Meaning | Action |
|-------|---------|--------|
| Valid | ROA exists, AS matches | Accept (prefer) |
| Invalid | ROA exists, AS does NOT match | **Reject** |
| Unknown (NotFound) | No ROA exists | Accept (for now) |

**Deployment status (2024-2025):** ~50% of IPv4 prefixes covered by ROAs. All Tier-1 providers do ROV. If you're not doing RPKI validation in 2026, you're negligent.

### BGP Security: Authentication & Protection

#### TCP MD5 (RFC 2385) — Legacy but Ubiquitous
```cisco
neighbor X.X.X.X password SECRET
```
- Adds MD5 hash to every TCP segment
- Prevents RST injection and session hijack
- **Limitation**: No key rotation without session teardown, weak hash

#### TCP-AO (RFC 5925) — The Replacement
- Multiple simultaneous keys with key IDs
- HMAC-SHA1, HMAC-SHA256, etc.
- Graceful key rotation
- **Reality**: Support still patchy in 2026. MD5 is still the pragmatic choice for most peers.

#### GTSM (Generalized TTL Security Mechanism — RFC 5082)
```cisco
neighbor X.X.X.X ttl-security hops 1
```
- Sets outgoing TTL to 255, requires incoming TTL ≥ 254
- Prevents remote spoofing attacks (attacker must be directly connected)
- **Use on all directly connected eBGP peers. No exceptions.**

### Multihop eBGP

Sometimes eBGP peers aren't directly connected (loopback peering for redundancy):

```
         ┌─────┐  link1  ┌─────┐
         │ AS1 │─────────│ AS2 │
         │     │─────────│     │
         └─────┘  link2  └─────┘
```

Peer on loopbacks (not link addresses) so the session survives a single link failure:

```cisco
neighbor 10.0.0.2 remote-as 64999
neighbor 10.0.0.2 ebgp-multihop 2
neighbor 10.0.0.2 update-source Loopback0
! Need static routes or IGP to reach peer loopback
ip route 10.0.0.2/32 198.51.100.2
ip route 10.0.0.2/32 198.51.100.6    ! Backup via link2
```

**Caution**: `ebgp-multihop` disables GTSM. You can't use both. For directly-connected peers, always prefer GTSM.

### Graceful Shutdown (RFC 8326)

When decommissioning a peering link, don't just shut it down — that causes a full convergence event. Instead:

```cisco
! On the router being decommissioned:
route-map GSHUT-OUT permit 10
 set community gshut additive
!
neighbor X.X.X.X route-map GSHUT-OUT out
! Then clear the session to re-advertise
clear bgp X.X.X.X soft out
```

The `GRACEFUL_SHUTDOWN` well-known community (65535:0) signals to all peers: "I'm going away, prefer other paths." Peers that honor this community set LOCAL_PREF to 0 on receipt, causing traffic to shift before the link goes down.

## Configuration

### IOS-XE: Complete eBGP Setup

```cisco
! eBGP peering configuration - IOS-XE
!
! === PREFIX LISTS ===
!
! Bogon filter
ip prefix-list BOGONS seq 5 permit 0.0.0.0/8 le 32
ip prefix-list BOGONS seq 10 permit 10.0.0.0/8 le 32
ip prefix-list BOGONS seq 15 permit 100.64.0.0/10 le 32
ip prefix-list BOGONS seq 20 permit 127.0.0.0/8 le 32
ip prefix-list BOGONS seq 25 permit 169.254.0.0/16 le 32
ip prefix-list BOGONS seq 30 permit 172.16.0.0/12 le 32
ip prefix-list BOGONS seq 35 permit 192.0.2.0/24 le 32
ip prefix-list BOGONS seq 40 permit 192.168.0.0/16 le 32
ip prefix-list BOGONS seq 45 permit 198.18.0.0/15 le 32
ip prefix-list BOGONS seq 50 permit 198.51.100.0/24 le 32
ip prefix-list BOGONS seq 55 permit 203.0.113.0/24 le 32
ip prefix-list BOGONS seq 60 permit 224.0.0.0/4 le 32
ip prefix-list BOGONS seq 65 permit 240.0.0.0/4 le 32
!
! Our prefixes (reject if received from peer)
ip prefix-list OUR-SPACE seq 5 permit 203.0.113.0/22 le 24
!
! Maximum specificity filter
ip prefix-list MAX-SPECIFIC seq 5 deny 0.0.0.0/0 ge 25
ip prefix-list MAX-SPECIFIC seq 10 permit 0.0.0.0/0 le 24
!
! === ROUTE MAPS ===
!
! Transit inbound policy
route-map TRANSIT-IN deny 10
 description "Reject bogons"
 match ip address prefix-list BOGONS
route-map TRANSIT-IN deny 20
 description "Reject our own space"
 match ip address prefix-list OUR-SPACE
route-map TRANSIT-IN permit 100
 description "Accept everything else"
 set local-preference 80
 set community 65000:80 additive
!
! Peer inbound policy
route-map PEER-IN deny 10
 match ip address prefix-list BOGONS
route-map PEER-IN deny 20
 match ip address prefix-list OUR-SPACE
route-map PEER-IN permit 100
 set local-preference 100
 set community 65000:100 additive
!
! Customer inbound policy
route-map CUSTOMER-IN deny 10
 match ip address prefix-list BOGONS
route-map CUSTOMER-IN permit 100
 set local-preference 150
 set community 65000:150 additive
!
! Outbound to peers — only customer routes
route-map PEER-OUT permit 10
 match community CUSTOMER-ROUTES
route-map PEER-OUT deny 100
!
! Graceful shutdown handling
route-map GSHUT-HANDLER permit 10
 match community GSHUT
 set local-preference 0
route-map GSHUT-HANDLER permit 100
!
! Community lists
ip community-list standard CUSTOMER-ROUTES permit 65000:150
ip community-list standard GSHUT permit graceful-shutdown
!
! === BGP ===
router bgp 65000
 !
 ! Transit provider
 neighbor 198.51.100.1 remote-as 3356
 neighbor 198.51.100.1 description "Lumen-Transit"
 neighbor 198.51.100.1 password TRANSIT_KEY
 neighbor 198.51.100.1 ttl-security hops 1
 !
 ! Peer (PNI)
 neighbor 198.51.100.5 remote-as 13335
 neighbor 198.51.100.5 description "Cloudflare-PNI"
 neighbor 198.51.100.5 password PEER_KEY
 neighbor 198.51.100.5 ttl-security hops 1
 !
 ! Customer
 neighbor 198.51.100.9 remote-as 65100
 neighbor 198.51.100.9 description "Customer-Acme"
 neighbor 198.51.100.9 password CUST_KEY
 neighbor 198.51.100.9 ttl-security hops 1
 !
 ! IX Route Server (AMS-IX example)
 neighbor 80.249.208.1 remote-as 6777
 neighbor 80.249.208.1 description "AMS-IX-RS1"
 neighbor 80.249.208.1 ebgp-multihop 1
 !
 address-family ipv4 unicast
  ! Transit
  neighbor 198.51.100.1 activate
  neighbor 198.51.100.1 send-community both
  neighbor 198.51.100.1 route-map TRANSIT-IN in
  neighbor 198.51.100.1 route-map PEER-OUT out
  neighbor 198.51.100.1 prefix-list MAX-SPECIFIC in
  neighbor 198.51.100.1 maximum-prefix 1000000 85 restart 30
  !
  ! Peer
  neighbor 198.51.100.5 activate
  neighbor 198.51.100.5 send-community both
  neighbor 198.51.100.5 route-map PEER-IN in
  neighbor 198.51.100.5 route-map PEER-OUT out
  neighbor 198.51.100.5 prefix-list MAX-SPECIFIC in
  neighbor 198.51.100.5 maximum-prefix 200000 85 warning-only
  !
  ! Customer
  neighbor 198.51.100.9 activate
  neighbor 198.51.100.9 send-community both
  neighbor 198.51.100.9 route-map CUSTOMER-IN in
  neighbor 198.51.100.9 maximum-prefix 100 80 restart 15
  !
  ! IX RS
  neighbor 80.249.208.1 activate
  neighbor 80.249.208.1 send-community both
  neighbor 80.249.208.1 route-map PEER-IN in
  neighbor 80.249.208.1 route-map PEER-OUT out
  neighbor 80.249.208.1 maximum-prefix 500000 85 warning-only
 exit-address-family
```

### Junos: Complete eBGP Setup

```junos
# eBGP peering configuration - Junos
policy-options {
    # Bogon list
    prefix-list BOGONS {
        0.0.0.0/8;
        10.0.0.0/8;
        100.64.0.0/10;
        127.0.0.0/8;
        169.254.0.0/16;
        172.16.0.0/12;
        192.0.2.0/24;
        192.168.0.0/16;
        198.18.0.0/15;
        198.51.100.0/24;
        203.0.113.0/24;
        224.0.0.0/4;
        240.0.0.0/4;
    }
    prefix-list OUR-SPACE {
        203.0.113.0/22 upto /24;
    }
    
    community CUSTOMER-ROUTES members 65000:150;
    community PEER-ROUTES members 65000:100;
    community TRANSIT-ROUTES members 65000:80;
    community GSHUT members graceful-shutdown;
    
    policy-statement TRANSIT-IN {
        term REJECT-BOGONS {
            from prefix-list BOGONS;
            then reject;
        }
        term REJECT-OUR-SPACE {
            from prefix-list OUR-SPACE;
            then reject;
        }
        term REJECT-LONG-PREFIX {
            from route-filter 0.0.0.0/0 prefix-length-range /25-/32;
            then reject;
        }
        term ACCEPT {
            then {
                local-preference 80;
                community add TRANSIT-ROUTES;
                accept;
            }
        }
    }
    
    policy-statement PEER-IN {
        term REJECT-BOGONS {
            from prefix-list BOGONS;
            then reject;
        }
        term REJECT-OUR-SPACE {
            from prefix-list OUR-SPACE;
            then reject;
        }
        term ACCEPT {
            then {
                local-preference 100;
                community add PEER-ROUTES;
                accept;
            }
        }
    }
    
    policy-statement CUSTOMER-IN {
        term REJECT-BOGONS {
            from prefix-list BOGONS;
            then reject;
        }
        term ACCEPT {
            then {
                local-preference 150;
                community add CUSTOMER-ROUTES;
                accept;
            }
        }
    }
    
    policy-statement PEER-OUT {
        term CUSTOMER-ONLY {
            from community CUSTOMER-ROUTES;
            then accept;
        }
        term DENY-REST {
            then reject;
        }
    }
}

protocols {
    bgp {
        log-updown;
        group TRANSIT {
            type external;
            import TRANSIT-IN;
            export PEER-OUT;
            peer-as 3356;
            neighbor 198.51.100.1 {
                description "Lumen-Transit";
                authentication-key "$9$ENCRYPTED";
                family inet {
                    unicast {
                        prefix-limit {
                            maximum 1000000;
                            teardown 85 idle-timeout 30;
                        }
                    }
                }
            }
        }
        group PEER-PNI {
            type external;
            import PEER-IN;
            export PEER-OUT;
            peer-as 13335;
            neighbor 198.51.100.5 {
                description "Cloudflare-PNI";
                authentication-key "$9$ENCRYPTED";
                family inet {
                    unicast {
                        prefix-limit {
                            maximum 200000;
                            teardown 85 idle-timeout 30;
                        }
                    }
                }
            }
        }
        group CUSTOMER {
            type external;
            import CUSTOMER-IN;
            peer-as 65100;
            neighbor 198.51.100.9 {
                description "Customer-Acme";
                authentication-key "$9$ENCRYPTED";
                family inet {
                    unicast {
                        prefix-limit {
                            maximum 100;
                            teardown 80 idle-timeout 15;
                        }
                    }
                }
            }
        }
        group IX-AMS {
            type external;
            import PEER-IN;
            export PEER-OUT;
            multihop {
                ttl 1;
            }
            peer-as 6777;
            neighbor 80.249.208.1 {
                description "AMS-IX-RS1";
                family inet {
                    unicast {
                        prefix-limit {
                            maximum 500000;
                            teardown 85;
                        }
                    }
                }
            }
        }
    }
}
```

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| `ttl-security hops 1` | Off | On (direct peers) | GTSM — prevents remote session hijack |
| `maximum-prefix` | None | Always set | Protects against route leaks |
| `remove-private-as` | Off | On (customer-facing) | Don't leak customer internal ASNs |
| `prefix-limit teardown idle-timeout` | — | 15-30 min | Auto-recover after prefix storm subsides |
| `send-community both` | Varies | Always | Required for policy communities, RT for VPN |
| Graceful shutdown community | — | Honor inbound | Smooth maintenance windows |

### What Good Looks Like

**Healthy peering summary:**
```
Neighbor     AS    MsgRcvd  MsgSent   Up/Down  PfxRcd   PfxAcc
198.51.100.1 3356  1842567  15234     4w2d     892341   891203  ← Transit (full table)
198.51.100.5 13335 234567   12890     2w5d     45678    45201   ← Peer (their routes)
198.51.100.9 65100 1234     9876      6w1d     12       12      ← Customer (their /24s)
80.249.208.1 6777  567890   14567     1w3d     312456   310234  ← IX RS (many peers)
```

**Red flags:**
- PfxRcd >> expected → route leak (check `maximum-prefix` fired?)
- PfxRcd << expected → peer is filtering, or upstream issue
- PfxRcd ≠ PfxAcc (big gap) → your filters are dropping lots of routes (review filter)
- Frequent Up/Down → flapping (BFD or link issue)

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| IX route server peer down | MTU issue on IX fabric (jumbo frames) | Match IX MTU (usually 1500 or 9000) |
| Peer receives no routes from you | Outbound filter too restrictive, no `activate`, community mismatch | Check `show bgp neighbor X advertised-routes` |
| Customer routes not propagating to transit | Missing `redistribute` or community-based outbound filter | Verify community tagging on customer inbound |
| RPKI invalid routes still accepted | Validator not configured, or policy not rejecting invalids | Check validator session, verify policy |
| Traffic asymmetry after adding new peer | New peer sends routes with better LOCAL_PREF/AS_PATH | Expected behavior; tune policy if problematic |

### War Story: The IX That Became a Transit

A mid-size SP joined a new IX and peered with the route server. They configured `PEER-IN` policy but forgot to set the outbound filter to customer-only. Their default export policy sent the *full table* to the IX route server.

Every other AS on the IX suddenly saw this SP as a transit provider. Traffic from hundreds of networks started flowing through them. Bandwidth usage spiked 400%. Their transit bills exploded because they were now carrying traffic between networks that had nothing to do with them.

The IX abuse team noticed within 4 hours and sent a strongly-worded email. The fix was one `route-map` on the outbound.

**Lesson**: Always verify outbound filters on IX peering. And test with `show bgp neighbor X advertised-routes` before enabling the session.
