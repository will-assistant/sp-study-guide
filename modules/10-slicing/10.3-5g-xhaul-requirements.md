# 10.3 — 5G Xhaul Requirements

## Overview
> "Xhaul" is the umbrella term for the three transport segments connecting 5G radio sites to the mobile core: fronthaul, midhaul, and backhaul. Each has radically different latency, bandwidth, and synchronization requirements — and your SP backbone has to carry all three simultaneously. This section covers the functional splits that create these segments, the transport requirements of each, the evolution from CPRI to eCPRI, and how the converged xhaul architecture maps onto the IP/MPLS/optical infrastructure you've been building throughout this guide.


## Key Concepts

> - how service requirements map to deterministic transport slices
> - how FlexE and resource partitioning enforce bandwidth isolation
> - latency, jitter, and synchronization targets for 5G xhaul services
> - operational guardrails for multi-tenant SLA assurance

## Theory

### The Disaggregated gNB

In 4G, the base station (eNB) was a monolith: radio, baseband, and network functions in one box at the cell site. 5G blew that apart. The 5G base station (gNB) is decomposed into three functional elements that can be physically separated:

```
4G (Monolithic)                    5G (Disaggregated)
┌──────────────┐                  ┌───────┐   ┌───────┐   ┌───────┐
│              │                  │  O-RU  │   │  O-DU  │   │  O-CU  │
│     eNB      │        →        │ Radio  │───│ Base-  │───│ Higher │
│  (all-in-one)│                  │ Unit   │   │ band   │   │ Layer  │
│              │                  │(cell   │   │(real-  │   │(RRC,   │
└──────────────┘                  │ site)  │   │ time)  │   │ PDCP)  │
                                  └───────┘   └───────┘   └───────┘
                                      ↑           ↑           ↑
                                  Fronthaul   Midhaul    Backhaul
                                  (O-RU↔O-DU) (O-DU↔O-CU) (O-CU↔Core)
```

**O-RU (Open Radio Unit):** The antenna and RF front-end at the cell site. Converts between RF and digital domain. Always at the tower.

**O-DU (Open Distributed Unit):** Real-time L1/L2 baseband processing — HARQ, scheduling, beamforming control. Can be at the cell site (D-RAN) or centralized at an edge site (C-RAN). This is where the split decision matters most.

**O-CU (Open Centralized Unit):** Higher-layer processing — RRC (control plane), PDCP (user plane). Further split into:
- **O-CU-CP:** Control plane (RRC, signaling). Connected to O-DU via F1-C interface.
- **O-CU-UP:** User plane (PDCP, GTP-U). Connected to O-DU via F1-U interface.
- **E1 interface:** Connects CU-CP to CU-UP (3GPP TS 38.401).

### The Three Xhaul Segments

Each physical separation between gNB components creates a transport segment with distinct requirements:

```
┌────────┐  Fronthaul  ┌────────┐  Midhaul   ┌────────┐  Backhaul  ┌────────┐
│  O-RU  │─────────────│  O-DU  │────────────│  O-CU  │────────────│  5G    │
│(tower) │  eCPRI/CPRI │(edge   │  F1-C/F1-U │(aggr.  │  N2/N3    │  Core  │
│        │  L2, strict │ site)  │  L3, mod.  │ site)  │  L3, std  │(UPF,   │
│        │  timing     │        │  latency   │        │  routing   │ AMF)   │
└────────┘             └────────┘            └────────┘            └────────┘
│←── ~100-150µs ──→│←─── 1-3ms ───→│←──── 10-300ms ────→│
│←─ 10-25 Gbps/RU ─→│←── ~1-5 Gbps ──→│←──── ~1-5 Gbps ────→│
```

### Fronthaul: The Brutal Segment

Fronthaul connects O-RU to O-DU and carries either raw I/Q samples (CPRI) or partially processed radio data (eCPRI). This is by far the most demanding transport segment.

**Why fronthaul is hard:**
- **Latency budget: ~100-150µs one-way** (Juniper JVD specifies <150µs RU-to-DU). This includes serialization, propagation, and switching — leaving almost no room for queuing.
- **Bandwidth scales with radio config, not user traffic.** A single 100 MHz 5G carrier with 4T4R MIMO generates ~10 Gbps of CPRI. Massive MIMO (64T64R) on the same carrier? ~157 Gbps. The pipe is always full, whether one user or a thousand.
- **Synchronization: sub-microsecond phase alignment** via IEEE 1588v2 (PTP) with ITU-T G.8275.1 (full on-path support) or G.8275.2 (partial timing support). SyncE required for frequency synchronization.
- **Packet loss tolerance: effectively zero.** Lost fronthaul frames mean corrupted radio frames — no TCP retransmit is going to save you.

#### CPRI vs eCPRI: The Generational Shift

| Feature | CPRI | eCPRI |
|---------|------|-------|
| **Transport** | TDM serial (dedicated fiber/wavelength) | Packet-based (Ethernet/IP/UDP) |
| **Data carried** | Raw I/Q samples (Option 8 split) | Partially processed (Option 7-2x split) |
| **Bandwidth** | Fixed, massive (~30x user data) | Variable, ~10x less than CPRI |
| **Topology** | Point-to-point only | P2P, P2MP, mesh — standard switching |
| **QoS/routing** | None (dedicated circuit) | Full Ethernet QoS, ECMP possible |
| **Synchronization** | Embedded in line rate | PTP/SyncE over packet network |
| **Vendor interop** | Effectively proprietary | O-RAN Alliance standardized |
| **Line rates** | 614.4M to 24.3 Gbps (Option 10) | 10GE, 25GE standard Ethernet |

**The key insight:** CPRI's Option 8 split puts the entire PHY in the DU — the fronthaul carries raw antenna samples. eCPRI's Option 7-2x moves the low-PHY (FFT/iFFT, beamforming) to the RU, sending frequency-domain I/Q instead of time-domain samples. This dramatically reduces bandwidth (roughly 10:1) while keeping enough centralization for features like CoMP and joint scheduling.

**Approximate fronthaul bandwidth per site (single 100 MHz NR carrier):**

| Configuration | CPRI (approx.) | eCPRI 7-2x (approx.) |
|--------------|----------------|----------------------|
| 4T4R | ~10 Gbps | ~2-3 Gbps |
| 8T8R | ~20 Gbps | ~4-5 Gbps |
| 32T32R (mMIMO) | ~80 Gbps | ~8-12 Gbps |
| 64T64R (mMIMO) | ~160 Gbps | ~10-25 Gbps |

> ⚠️ **These are planning estimates, not exact calculations.** CPRI uses fixed line rates (Option 7 = 9.8 Gbps, Option 10 = 24.3 Gbps) — actual deployments bond multiple CPRI links. The "per-antenna" scaling shown is approximate. eCPRI rates depend on numerology (subcarrier spacing), compression (O-RAN block floating point), number of PRBs, number of beamformed layers, and vendor implementation. For mMIMO, eCPRI is dramatically lower because beamforming happens in the RU — only beamformed spatial streams (typically 8-16 layers) traverse the fronthaul, not all 64 raw antenna streams. Use the upper end of ranges for capacity planning.

#### 3GPP Functional Split Options

3GPP TR 38.801 defines eight split options between the RAN protocol layers. Only a few matter for transport:

```
                   RRC    PDCP    RLC    MAC    High-PHY  Low-PHY   RF
                    │      │       │      │       │         │        │
Option:     1 ──┘  2 ──┘  3 ──┘  4 ──┘  5 ──┘  6 ──┘   7 ──┘   8 ──┘
                                                        ↑         ↑
                                                    O-RAN      CPRI
                                                    7-2x     (legacy)
            ←──── Higher-layer splits ─────→ ←── Lower-layer splits ──→
            Lower BW, higher latency OK        Higher BW, ultra-low latency
            More processing at edge            More centralization possible
```

**Split 2 (PDCP/RLC boundary):** The midhaul split. CU handles PDCP, DU handles RLC/MAC/PHY. Moderate bandwidth (~equal to user throughput), relaxed latency (~1-3ms). This is the F1 interface.

**Split 7-2x (intra-PHY):** The O-RAN fronthaul split. RU handles low-PHY (FFT, beamforming, PRACH extraction), DU handles high-PHY (LDPC coding, modulation mapping, resource element mapping). Tight latency (<150µs), high bandwidth but manageable.

**Split 8 (PHY/RF boundary):** Legacy CPRI. RU is just an antenna + ADC/DAC. All processing centralized. Maximum bandwidth consumption, tightest latency. Being phased out.

### Midhaul: The Forgotten Middle Child

Midhaul connects O-DU to O-CU over the F1 interface. It only exists in disaggregated deployments where the DU and CU are in different locations.

**Requirements:**
- **Latency: 1-3ms one-way** (per 3GPP TR 38.801 and O-RAN WG9 guidelines). More relaxed than fronthaul because HARQ timing is handled within the DU.
- **Bandwidth: roughly equal to user-plane throughput** plus control-plane signaling. Similar to backhaul — approximately 1-5 Gbps per site depending on capacity.
- **Transport: standard L3 IP/MPLS.** F1-C uses SCTP, F1-U uses GTP-U/UDP/IP. Standard routing works fine.
- **Synchronization: PTP still needed** but with relaxed accuracy requirements. G.8275.2 (partial timing) acceptable in most deployments.

**Why midhaul is often collapsed:**
Many operators combine midhaul and backhaul into a single transport domain because:
1. Both use L3/IP transport
2. Both have similar bandwidth profiles
3. Latency requirements, while tighter than backhaul, are achievable with standard MPLS networks
4. Separate O-DU and O-CU sites add real estate and power costs

In Juniper's 5G xHaul JVD, midhaul and backhaul share the aggregation/core ring topology, using L3VPN with BGP-CT color mapping for service differentiation.

### Backhaul: The Familiar Territory

Backhaul connects O-CU (or the entire gNB in D-RAN) to the 5G core (UPF, AMF, SMF) over N2 (control) and N3 (user) interfaces.

**Requirements:**
- **Latency: 10-300ms** depending on service. eMBB is forgiving; URLLC services may need <10ms end-to-end (but most of that budget is consumed by the RAN, not backhaul).
- **Bandwidth: 1-5 Gbps per macro site** in dense urban, scaling with traffic growth. Fiber-connected sites are standard; microwave still used in rural/emerging markets.
- **Transport: mature L3VPN/EVPN over SR-MPLS.** This is what you've been building since Module 6. Separate VPNs for N2 (control) and N3 (user) per 3GPP recommendation.
- **Synchronization: PTP for 5G** (G.8275.1 or G.8275.2), SyncE for frequency. 4G LTE sites using 1588v2 or GPS.

**Backhaul is the "solved problem"** — except for scale. A national carrier with 50,000+ cell sites, each needing 1-5 Gbps, means the aggregation and core network must handle petabits of capacity. This is where Module 9's coherent optics and DWDM really earn their keep.

### The Xhaul Requirements Summary

| Requirement | Fronthaul | Midhaul | Backhaul |
|-------------|-----------|---------|----------|
| **One-way latency** | <100-150µs | 1-3ms | 10-300ms |
| **Bandwidth/site** | 10-25 Gbps (eCPRI) | 1-5 Gbps | 1-5 Gbps |
| **Jitter tolerance** | <±5µs | <±10µs | <±100µs |
| **Packet loss** | <10⁻⁶ (effectively zero) | <10⁻⁴ | <10⁻³ |
| **Sync accuracy** | ±1.5µs phase (G.8275.1) | ±1.5µs phase | ±1.5µs phase |
| **Frequency sync** | SyncE (±50 ppb locked) | SyncE | SyncE or GPS |
| **Transport type** | L2 Ethernet (eCPRI) | L3 IP/MPLS | L3 IP/MPLS |
| **Topology** | Spine-leaf, P2P fiber | Ring, mesh | Ring, mesh, hub-spoke |
| **Protocol** | eCPRI/CPRI | F1 (SCTP + GTP-U) | N2 (SCTP) + N3 (GTP-U) |
| **Distance** | <20 km (fiber latency) | <100 km | No hard limit |
| **Standards** | O-RAN WG4, IEEE 1914.1 | 3GPP TS 38.401, TR 38.801 | 3GPP TS 23.501 |

> ⚠️ **Latency and jitter values represent typical planning targets** from O-RAN Alliance, IEEE 1914, and vendor design guides (Juniper JVD, Cisco Converged xHaul). Specific deployments may have tighter or relaxed requirements depending on RAN vendor, functional split choice, and service mix.

## Architecture

### Converged Xhaul Topology

The industry trend is a converged xhaul architecture where all three segments share physical infrastructure but are logically sliced. The O-RAN Alliance (WG9) and Juniper's validated design define four transport tiers:

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    CONVERGED XHAUL TRANSPORT                           │
│                                                                         │
│  ┌──────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────┐ │
│  │  ACCESS   │    │    PRE-AGG    │    │ AGGREGATION  │    │   CORE   │ │
│  │          │    │              │    │              │    │          │ │
│  │ Cell-site │───│  Hub site    │───│  Metro/Aggr  │───│  Core    │ │
│  │ routers   │   │  routers     │   │  routers     │   │  routers │ │
│  │          │    │              │    │              │    │          │ │
│  │ ACX710   │    │  ACX5448     │    │  MX204/      │    │  PTX/   │ │
│  │ ACX5448  │    │  (HSR)       │    │  MX10003     │    │  MX10K  │ │
│  └──────────┘    └──────────────┘    └──────────────┘    └──────────┘ │
│                                                                         │
│  ←── Fronthaul ──→←──── Midhaul + Backhaul ───────────────────────→   │
│  Spine-leaf, L2     Ring/mesh topology, L3VPN + SR-MPLS               │
│  eCPRI, PTP strict  BGP-CT color mapping, Flex-Algo slicing           │
│                                                                         │
│  IS-IS L1 domain    │←──────── IS-IS L2 domain ──────────────────→│   │
│                     BGP-LU stitching at L1/L2 borders                  │
└─────────────────────────────────────────────────────────────────────────┘
```

**Access (Cell-site routers):** Terminate fronthaul (eCPRI from O-RU), perform L2 switching and PTP distribution. Also serve as PE for backhaul VPNs in D-RAN deployments.

**Pre-aggregation (Hub-site routers):** Aggregate access rings. Collocate O-DU in C-RAN deployments. L1/L2 IS-IS boundary. First point of L3 services. Dual-homed for redundancy.

**Aggregation:** Metro aggregation rings. MX-class routers. L3VPN/EVPN termination for midhaul and backhaul services. SR-TE policies for SLA differentiation.

**Core:** National backbone routers. PTX/MX10K. Connect aggregation regions to 5G core data centers. BGP-LU for inter-domain MPLS. High-capacity coherent optics.

### Transport Slicing with Flex-Algo + BGP-CT

The converged xhaul uses the slicing tools from Sections 10.1-10.2 to differentiate traffic classes:

```
Service Layer (Overlay)        Transport Layer (Underlay)
┌──────────────────────┐       ┌──────────────────────┐
│  Gold VPN (URLLC)    │ ────→ │  Flex-Algo 128 (Red) │
│  - N2 control plane  │       │  - Min-latency metric│
│  - URLLC user plane  │       │  - Strict SLA paths  │
├──────────────────────┤       ├──────────────────────┤
│  Bronze VPN (eMBB)   │ ────→ │  Flex-Algo 129 (Blue)│
│  - eMBB user plane   │       │  - TE metric (cost)  │
│  - Best-effort data  │       │  - Bandwidth-opt path│
├──────────────────────┤       ├──────────────────────┤
│  Default VPN (mgmt)  │ ────→ │  Algo 0 (default)    │
│  - OAM, management   │       │  - IGP shortest path │
│  - Non-critical       │       │  - Fallback for all  │
└──────────────────────┘       └──────────────────────┘

BGP-CT maps overlay color → underlay Flex-Algo:
  Gold  → RT:color:128 → Algo 128 prefix-SIDs → low-latency path
  Bronze → RT:color:129 → Algo 129 prefix-SIDs → cost-optimized path
  Fallback → inet.3 / color-agnostic BGP-LU path
```

This is BGP Classful Transport (BGP-CT) — the mechanism from Section 10.1 put into practice. Each VPN prefix carries a transport class (via route target), which maps to a Flex-Algo topology in the underlay. If the colored path fails, traffic falls back to the default algorithm.

### Timing Architecture

5G requires sub-microsecond phase synchronization — GPS at every cell site is expensive and unreliable (urban canyon, indoor cells). The transport network must distribute timing:

```
              PRTC/T-GM                    PRTC/T-GM
              (GPS ref)                    (GPS ref)
                  │                            │
                  ▼                            ▼
┌──────────────────────────────────────────────────────┐
│                    PTP Domain                         │
│                                                       │
│  O-RU ──── T-TSC ──── T-BC ──── T-BC ──── T-GM     │
│ (slave)   (access)   (pre-agg)  (aggr)   (GPS ref) │
│                                                       │
│  Freq: SyncE chain (each node locks to upstream)     │
│  Phase: PTP G.8275.1 (full on-path, every hop)       │
│         or G.8275.2 (partial, fewer PTP-aware nodes) │
│                                                       │
│  Budget: ±1.5µs max |TE| at O-RU (ITU-T G.8271.1)  │
│  Per-hop: ~50-100ns at each T-BC/T-TSC node          │
│  Max hops: ~10-15 T-BC nodes (depends on budget)    │
└──────────────────────────────────────────────────────┘

PRTC = Primary Reference Time Clock
T-GM  = Telecom Grandmaster
T-BC  = Telecom Boundary Clock
T-TSC = Telecom Time Slave Clock (with SyncE)
```

**G.8275.1 (Full Timing Support):** Every node in the path is PTP-aware. Best accuracy, most expensive (requires PTP-capable hardware at every hop). Used for fronthaul.

**G.8275.2 (Partial Timing Support):** Only boundary clocks at key nodes; PTP transits non-PTP-aware switches using unicast. Lower accuracy, lower cost. Acceptable for midhaul/backhaul with relaxed requirements.

**SyncE** provides frequency synchronization via physical-layer clock recovery. Standard EEC (G.8262) has a free-run accuracy limit of ±4.6 ppm. Enhanced EEC (eEEC, G.8262.1) improves noise generation, noise tolerance, and transient response — critical for 5G fronthaul where PTP relies on a stable frequency reference. The locked SyncE output accuracy target is ±50 ppb (network limit), which applies to both EEC and eEEC when locked to an upstream source. Required alongside PTP — PTP alone can't achieve sub-µs phase accuracy without a stable frequency reference.

### Design Considerations

**When to use C-RAN vs D-RAN:**

| Factor | C-RAN (Centralized DU) | D-RAN (DU at Cell Site) |
|--------|----------------------|----------------------|
| **Fronthaul** | Required (eCPRI) | Not needed (local) |
| **Fiber availability** | Must have dark fiber to every site | Only needs backhaul fiber |
| **CoMP/joint scheduling** | Full support (DUs in same location) | Limited (inter-site coordination) |
| **Edge site cost** | Lower per-site (shared DU pool) | Higher per-site (DU at every tower) |
| **Latency** | Must meet <150µs fronthaul budget | No fronthaul latency concern |
| **When to use** | Dense urban, stadium, campus | Rural, remote, fiber-scarce |

**Fronthaul distance limits:**
Fiber latency alone is ~5µs/km. With a 100µs total budget and ~30-50µs for processing/switching, you have ~50-70µs for propagation — roughly **10-15 km maximum** for fronthaul fiber runs. This is a hard physics constraint.

**The 25GE question:**
eCPRI fronthaul standardized on 25GE interfaces (IEEE 802.3by). Why?
- A single 100 MHz mMIMO sector needs ~8-10 Gbps eCPRI
- Three sectors per site = ~25-30 Gbps
- 25GE × 3 or 100GE breakout covers a macro site
- 10GE is too small; 100GE at every cell site is overkill and expensive

## Configuration

### IOS-XR: Fronthaul CoS for eCPRI

```cisco
! --- Cell-site router: eCPRI traffic classification ---
! eCPRI uses Ethernet with ethertype 0xAEFE or IP/UDP
! Classify by DSCP or VLAN priority for strict scheduling

class-map match-any eCPRI-TRAFFIC
 match dscp 46        ! EF — fronthaul user plane
 match dscp 48        ! CS6 — fronthaul control
end-class-map

class-map match-any PTP-TRAFFIC
 match dscp 56        ! CS7 — timing critical
end-class-map

! --- Strict priority queuing: PTP > eCPRI > everything else ---
policy-map XHAUL-EGRESS
 class PTP-TRAFFIC
  priority level 1
  police rate percent 5   ! PTP is low bandwidth, high priority
 !
 class eCPRI-TRAFFIC
  priority level 2
  police rate percent 85  ! Bulk of fronthaul capacity
 !
 class class-default
  bandwidth remaining percent 100
 !
end-policy-map

! --- Apply to fronthaul-facing interface ---
interface TenGigE0/0/0/0
 description TO_O-RU_SECTOR1
 service-policy output XHAUL-EGRESS
 l2transport
 !
!

! --- PTP configuration (G.8275.1 full timing support) ---
ptp
 clock
  domain 24
  profile g.8275.1
 !
 profile FRONTHAUL-SLAVE
  transport ethernet
  port state slave-only
  sync interval -6             ! 2^(-6) = 64 per second
  announce interval -3         ! 2^(-3) = 8 per second
  delay-request interval -6    ! 2^(-6) = 64 per second
 !
!

interface TenGigE0/0/0/0
 ptp
  profile FRONTHAUL-SLAVE
 !
!

! --- SyncE ---
frequency synchronization
 quality itu-t option 1
!
interface TenGigE0/0/0/0
 frequency synchronization
  selection input
  priority 1
  wait-to-restore 0
 !
!
```

### Junos: Converged Xhaul with SR and PTP

```junos
# --- IS-IS with Flex-Algo for xhaul slicing ---
protocols {
    isis {
        interface ge-0/0/0.0 {
            level 1;                    # Access domain
            point-to-point;
        }
        interface ge-0/0/1.0 {
            level 2;                    # Aggregation domain
            point-to-point;
        }
        source-packet-routing {
            # Flex-Algo 128: low-latency (gold/URLLC)
            flex-algorithm 128 {
                definition {
                    metric-type min-delay;
                    priority 100;
                }
            }
            # Flex-Algo 129: TE-optimized (bronze/eMBB)
            flex-algorithm 129 {
                definition {
                    metric-type te-metric;
                    priority 100;
                }
            }
            node-segment {
                ipv4-index 1;               # Algo 0
                flex-algorithm 128 {
                    ipv4-index 128;          # Gold prefix-SID
                }
                flex-algorithm 129 {
                    ipv4-index 129;          # Bronze prefix-SID
                }
            }
        }
    }
}

# --- PTP for 5G timing (G.8275.1) ---
protocols {
    ptp {
        clock-mode boundary;
        domain 24;
        profile-type g.8275.1;
        priority1 128;
        priority2 128;
        slave {
            interface ge-0/0/2.0 {
                transport-type ethernet;
                delay-mechanism end-to-end;
                sync-interval -6;          # 64 per second
                delay-req-interval -6;
            }
        }
        master {
            interface ge-0/0/0.0 {
                transport-type ethernet;
                delay-mechanism end-to-end;
            }
        }
    }
}

# --- SyncE ---
chassis {
    synchronization {
        network-option option-1;
    }
}
interfaces {
    ge-0/0/2 {
        synchronous-ethernet;
    }
}
protocols {
    synchronization {
        source-list {
            interface ge-0/0/2.0 {
                quality-level prc;
                priority 1;
            }
        }
    }
}
```

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| PTP domain | 0 | 24 (G.8275.1) | Standard domain for telecom profile |
| PTP sync rate | varies | 64/sec (-6) | Fronthaul needs high sync rate for accuracy |
| SyncE quality | varies | Option 1 (SDH) or Option 2 (SONET) | Region-dependent; Option 1 for most of world |
| IS-IS level | L1L2 | L1 (access), L2 (agg+core) | Limits fronthaul blast radius |
| eCPRI DSCP | none | EF (46) | Ensures strict priority in converged network |
| PTP DSCP | none | CS7 (56) | Highest priority — timing cannot wait |

### What Good Looks Like

**PTP:**
- Clock state: `LOCKED` or `PHASE_ALIGNED` (not `ACQUIRING` or `FREERUN`)
- Offset from master: <100ns for fronthaul boundary clocks, <500ns for aggregation
- Path delay: stable, consistent with fiber distance (~5µs/km)
- No "clock quality degradation" alarms

**SyncE:**
- All fronthaul-facing interfaces: `Locked` to upstream source
- Quality level: `PRC` or `SSU-A` (not `DNU` or `SEC`)
- No `frequency offset exceeded` alarms

**eCPRI:**
- Zero drops in the eCPRI priority queue
- Latency <100µs (measured via in-band OAM or Y.1731)
- Jitter <±5µs peak-to-peak

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| PTP stuck in `ACQUIRING` | No reachable grandmaster; firewalled PTP (UDP 319/320) | Check PTP port states, verify L2 reachability to T-GM |
| Phase error >1.5µs at O-RU | Too many boundary clock hops, asymmetric paths | Reduce PTP hops, enable SyncE for freq assist, check for unequal fiber lengths |
| SyncE shows `DNU` quality | Upstream source lost GPS lock or chain broken | Trace SyncE chain from GM, check each node's lock state |
| eCPRI drops during congestion | eCPRI not in strict priority queue, or policer too low | Verify QoS policy: PTP and eCPRI must be priority queues, not weighted |
| Fronthaul latency >150µs | Too many switch hops, queuing delay from other traffic | Reduce to ≤2 hops; verify eCPRI is in highest-priority queue after PTP |
| O-RU deregistration / link failures | Timing lost — phase error causes radio sync failure | Always fix timing first; O-RU can't operate without phase alignment |
| BGP-CT fallback to default path | Flex-Algo partition — no Algo 128/129 path available | Check IS-IS Flex-Algo participation on all routers in path, verify FAD consistency |

### Methodology

1. **Start with timing.** 5G transport issues are timing issues until proven otherwise. Check PTP state → SyncE state → clock offset → path delay.
2. **Check the queues.** Any drops in PTP or eCPRI queues? Even one eCPRI drop can cause radio frame errors. If drops exist, the QoS policy is wrong.
3. **Verify the slice.** Is traffic using the correct Flex-Algo path? `traceroute sr-mpls` with the Algo-specific prefix-SID to validate forwarding plane matches control plane.
4. **Measure end-to-end.** Use Y.1731 or Two-Way Active Measurement Protocol (TWAMP) to verify per-segment latency against the budget table.
5. **Escalation:** If timing is locked, queues are clean, and latency is within budget — the problem is in the RAN, not transport. Push back to the radio team with your measurements.

### War Stories

> **The asymmetric fiber that killed PTP.**
>
> A Tier 1 operator deployed C-RAN with eCPRI fronthaul across a new metro ring. PTP was configured perfectly — G.8275.1 profile, boundary clocks at every hop, SyncE locked. But O-RUs kept reporting phase errors >3µs and dropping out of service intermittently.
>
> Root cause: the fiber ring used two different cable routes for the two directions (A→B on Route 1, B→A on Route 2). The physical fiber lengths differed by 4.2 km. PTP's delay measurement assumes symmetric paths — it computes one-way delay as half the round-trip time. With 4.2 km of asymmetry, that's ~21µs of uncompensated error. Well beyond the ±1.5µs budget.
>
> Fix: configured static delay asymmetry compensation on the boundary clocks (`ptp delay-asymmetry` in IOS-XR, `delay-asymmetry` in Junos). This required physically measuring or computing the fiber length difference for each span. Tedious, but that's the job when physics won't cooperate.
>
> **Lesson:** PTP path delay measurements assume symmetric propagation. Any time you have different physical routes for TX and RX fibers — separate conduits, aerial vs underground, different splice trays — measure the asymmetry and compensate. Or use co-routed fiber (same cable, same route) and save yourself the headache.
