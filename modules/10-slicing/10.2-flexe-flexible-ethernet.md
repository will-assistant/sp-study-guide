# 10.2 — FlexE: Flexible Ethernet

## Overview
> FlexE is the technology that makes "hard slicing" actually hard. While VPNs, QoS, and Flex-Algo give you varying degrees of logical isolation (Sections 10.1 soft and enhanced-soft models), FlexE carves bandwidth at the physical layer — TDM-style calendar slots on Ethernet PHYs — so that one interface genuinely becomes multiple isolated pipes. It's Ethernet pretending to be SONET, and it's disturbingly good at it. This section covers the OIF FlexE architecture, the three core operations (bonding, sub-rating, channelization), frame structure, vendor implementations, and where FlexE fits in your 5G transport slicing strategy.

## Key Concepts
> **After this section you will understand:**
> - how FlexE creates hard bandwidth isolation using calendar-slot partitioning
> - the difference between bonding, sub-rating, and channelization operations
> - where FlexE solves non-standard rate matching between packet and optical domains
> - implementation constraints that affect interoperability and deterministic behavior
> - how FlexE fits into 5G transport slicing designs

## Theory

### Why FlexE Exists

Standard Ethernet has a problem: fixed rates. 10G, 25G, 40G, 50G, 100G, 400G — these are the speeds IEEE 802.3 defines. If you need 75G to match a coherent optic's line rate, or 150G for a wholesale customer, or 800G from bonding two 400G ports — Ethernet says "no."

Before FlexE, you had two options:
1. **Link Aggregation (LAG)**: Bundle multiple interfaces. Achieves ~70-80% utilization due to hash-based load balancing. Non-deterministic latency. No per-flow isolation.
2. **Overprovision**: Use the next larger standard rate and waste capacity. A 75G coherent wavelength gets a 100G port; 25G sits idle.

Neither option works for 5G transport, where you need:
- **Arbitrary rates** to match optical line rates (coherent modulation → non-standard speeds)
- **Hard isolation** between slices (not just QoS priority — actual bandwidth reservation at PHY layer)
- **Deterministic latency** (TDM-style forwarding, not statistical multiplexing)
- **Efficient bonding** (100% link utilization, not 70-80% LAG efficiency)

FlexE solves all four. The Optical Internetworking Forum (OIF) published FlexE 1.0 in March 2016, with subsequent versions adding higher PHY rates and management features.

### FlexE Version History

| Version | Date | Key Additions |
|---------|------|---------------|

| FlexE 1.1 | June 2017 | Alignment refinements, FlexE-unaware transport mode |
| FlexE 2.0 | June 2018 | 200G and 400G PHY support, management channel, skew management, time/freq transport |
| FlexE 2.1 | July 2019 | 50G PHY support, 25G calendar slot granularity option |
| FlexE 2.2 | October 2021 | Maintenance release (corrections from 2.1 implementation guide) |
| FlexE 3.0 | May 2025 | 800G PHY support, 100G calendar slot granularity, payload type overhead |

**Key standard relationships:**
- **ITU-T G.8023** (2018, amended 2024): FlexE in OTN transport context
- **ITU-T G.8312** (2020): Metro transport interfaces using FlexE framing
- **IEEE 802.3**: Underlying PHY specifications reused by FlexE

### Architecture: The FlexE Shim

FlexE inserts a shim layer between the MAC and PCS layers of the Ethernet stack:

```
┌─────────────────────────────────────────┐
│            Ethernet MAC                  │
│    (standard frames, unmodified)         │
├─────────────────────────────────────────┤
│          FlexE Shim Layer                │  ← NEW
│  ┌─────────────────────────────────┐    │
│  │  Multiplexing / Demultiplexing  │    │
│  │  Calendar-based slot allocation │    │
│  │  Overhead insertion/extraction  │    │
│  └─────────────────────────────────┘    │
├─────────────────────────────────────────┤
│     Physical Coding Sublayer (PCS)       │
│     (64B/66B encoding, lane distribution)│
├─────────────────────────────────────────┤
│  Physical Medium Attachment (PMA)        │
├─────────────────────────────────────────┤
│  Physical Medium Dependent (PMD)         │
│  (optics, copper — unchanged)            │
└─────────────────────────────────────────┘
```

The shim operates within the PCS, reusing existing 64B/66B block encoding and lane structures. This is critical: **FlexE doesn't change the physical layer.** Same optics, same fiber, same SerDes. It's purely a logical partitioning mechanism above the electrical layer.

### Key Terminology

| Term | Definition |
|------|-----------|
| **FlexE Group** | One or more PHYs of the same rate bonded together (e.g., 4 × 100G = 400G group) |
| **FlexE Instance** | A 100G logical unit within a group. A 100G PHY = 1 instance. A 400G PHY = 4 instances (via internal block interleaving) |
| **FlexE Client** | A MAC-layer data stream mapped to calendar slots (the "logical interface" an operator configures) |
| **Calendar** | A TDM-style schedule that assigns 5G or 25G slots to clients across instances |
| **Calendar Slot** | The atomic unit of bandwidth allocation: 5 Gbps (default) or 25 Gbps |
| **FlexE Overhead** | Periodic control blocks carrying sync, calendar config, fault signaling, and management channel |

### The Three Operations

#### 1. Bonding (Aggregation)

Bonding combines multiple PHYs into a single higher-bandwidth logical pipe. Unlike LAG, FlexE bonding uses deterministic calendar-based distribution — not hash-based — achieving 100% link utilization.

```
                     FlexE Bonding
  ┌──────────────────────────────────────────┐
  │           FlexE Group (400G)              │
  │                                          │
  │   PHY 1       PHY 2       PHY 3    PHY 4 │
  │  (100G)      (100G)      (100G)   (100G) │
  │  [20 slots]  [20 slots]  [20 slots][20 slots]│
  │     │           │           │        │    │
  │     └───────────┴───────────┴────────┘    │
  │              Calendar (80 slots)           │
  │                                          │
  │  ┌──────────────────────────────────────┐│
  │  │     FlexE Client: 400G (80 slots)    ││
  │  └──────────────────────────────────────┘│
  └──────────────────────────────────────────┘
```

**Why this beats LAG:**
- **100% utilization** vs 70-80% for LAG (no elephant flow problem)
- **Deterministic latency** — TDM scheduling, not statistical multiplexing
- **Graceful degradation** — if one PHY fails, the client loses exactly that PHY's bandwidth; other traffic continues on remaining PHYs
- **No flow hashing** — calendar distributes blocks round-robin across all PHYs

**Use case:** Two 400G ports bonded to 800G for DCI, without waiting for 800GE optics to be available or affordable.

#### 2. Sub-Rating

Sub-rating creates a client at a non-standard Ethernet rate by using only a portion of the available calendar slots. Unused slots are marked as unavailable.

```
                     FlexE Sub-Rating
  ┌──────────────────────────────────────────┐
  │           100G PHY (20 slots)             │
  │                                          │
  │  Used: ████████████████░░░░              │
  │        (15 slots = 75G)  (5 unused)      │
  │                                          │
  │  ┌──────────────────┐                    │
  │  │ FlexE Client: 75G │                    │
  │  └──────────────────┘                    │
  └──────────────────────────────────────────┘
```

**Why this matters for transport:**
- Coherent optics often deliver non-standard rates depending on modulation format and reach
- A 300km DP-16QAM wavelength might deliver 300G — with sub-rating, you can match the router port exactly instead of provisioning 400G and wasting capacity
- **No bandwidth waste**: the router only sends what the wavelength can carry

**Use case:** 100G port connected to a DWDM transponder configured for 75G line rate (long-haul DP-QPSK). Sub-rate the FlexE client to 75G so the router doesn't try to push 100G into a 75G pipe.

#### 3. Channelization

Channelization creates multiple independent FlexE clients on a single PHY or group, each allocated a portion of the calendar slots. This is **hard slicing** — traffic between channels is isolated at the physical layer.

```
                   FlexE Channelization
  ┌──────────────────────────────────────────┐
  │           100G PHY (20 slots)             │
  │                                          │
  │  ████████  ██████  ████  ██              │
  │  (8 slots) (6 slots)(4 slots)(2 slots)   │
  │  = 40G     = 30G   = 20G   = 10G        │
  │                                          │
  │  ┌──────┐ ┌─────┐ ┌────┐ ┌───┐         │
  │  │Slice1│ │Slice2│ │Slice3│ │S4│         │
  │  │eMBB  │ │URLLC │ │mMTC │ │Mgmt│        │
  │  │ 40G  │ │ 30G  │ │ 20G │ │10G│         │
  │  └──────┘ └─────┘ └────┘ └───┘         │
  └──────────────────────────────────────────┘
```

**Why this is the killer feature:**
- Each channel is a **separate Ethernet interface** to the router — independent MAC, independent QoS, independent routing/VPN
- Traffic from Slice 1 **cannot** burst into Slice 2's bandwidth, even if Slice 2 is idle
- This is the "hard isolation" from Section 10.1's isolation spectrum — calendar-based TDM guarantees
- Latency within a channel is deterministic; no contention from other channels

**Use case:** Single 100G link between cell site and aggregation node carries separate eMBB (40G), URLLC (30G), mMTC (20G), and management (10G) channels with hard isolation.

### FlexE Frame Structure

The FlexE overhead is built on 64B/66B blocks, reusing the existing Ethernet PCS encoding:

```
FlexE Overhead Structure (per 100G Instance)
┌──────────────────────────────────────────────────┐
│  OVERHEAD BLOCK PERIOD (repeats 8× per frame):    │
│    Block 0:      FlexE OH block (ordered set)     │
│    Blocks 1-20:  Calendar round 1 (20 data slots) │
│    Blocks 21-40: Calendar round 2 (20 data slots) │
│    ...                                            │
│    Blocks 20441-20460: Calendar round 1023        │
│    = 1 OH + 20,460 data = 20,461 blocks/period    │
│                                                    │
│  OVERHEAD FRAME = 8 OH block periods:              │
│    8 × 20,461 = 163,688 blocks per frame           │
│                                                    │
│  MULTIFRAME = 32 overhead frames:                  │
│    Full calendar + PHY map distributed across       │
│    32 frames = 5,237,216 blocks                     │
└──────────────────────────────────────────────────┘
```

**Timing at 100G** (103.125 Gbps line rate, ~1.5625 billion 66-bit blocks/sec):
- One **overhead block period**: 20,461 blocks ≈ **13.1 µs**
- One **overhead frame** (8 periods): 163,688 blocks ≈ **104.8 µs** (OIF spec: ~104.77 µs)
- One **multiframe** (32 frames): 5,237,216 blocks ≈ **3.35 ms**

The 8 overhead blocks within a frame carry the calendar, PHY map, CRC, and management channel incrementally. It takes a full multiframe (32 frames, ~3.35 ms) to distribute the complete configuration to the far end.

**Overhead fields:**
| Field | Size | Purpose |
|-------|------|---------|
| FlexE Map | Variable | Identifies which PHYs are in the group |
| Calendar A/B | 20 entries/instance | Slot-to-client assignment (dual for hitless switching) |
| CRC-16 | 16 bits | Error protection for overhead |
| RPF (Remote PHY Fault) | 1 bit/PHY | Fault signaling per PHY |
| Management Channel | Variable | Optional LLDP, OAM, management traffic |
| Sync fields | Multiple | Multiframe alignment, skew compensation |

**Hitless calendar reconfiguration:**
FlexE supports two calendar configurations (A and B). To change bandwidth allocation:
1. Program the new calendar in the standby slot (B while A is active)
2. Signal the switch via overhead bits
3. Both endpoints switch simultaneously
4. No traffic disruption for unchanged client mappings

This is how you resize a slice without taking it down.

### Skew Management

When bonding multiple PHYs, blocks from different PHYs may arrive at slightly different times due to path differences (different fiber routes, different wavelengths with different propagation characteristics).

FlexE handles up to **10 µs** of inter-PHY skew through:
- Overhead-based alignment markers (multiframe sync across all PHYs)
- Receive-side deskew buffers that hold early-arriving blocks until the slowest PHY catches up
- Skew detection via overhead timestamps

**Design rule:** If bonding PHYs over separate fiber paths, verify that path length difference translates to < 10 µs differential delay. At ~5 µs/km in fiber, that's approximately **2 km** of path difference — not an issue within a building or campus, but relevant for metro bonding scenarios.

### FlexE Transport Modes

FlexE defines three modes for interworking with transport networks:

| Mode | Description | Use Case |
|------|------------|----------|
| **FlexE-Unaware** | Transport treats each PHY independently; no FlexE knowledge required. Tolerates high skew (each PHY is a separate wavelength) | Legacy DWDM transport; PHYs routed independently |
| **FlexE-Termination** | Transport terminates FlexE clients and maps them into OTN (e.g., ODUflex). Full client visibility at transport layer | OTN-based backbone; transport node does grooming |
| **FlexE-Aware** | Transport understands FlexE structure; compresses unused slots to match line rate. Preserves FlexE framing end-to-end | IP-over-DWDM with coherent pluggables; rate adaptation |

**FlexE-Unaware is the most common deployment** today because it requires zero transport upgrades. The FlexE endpoints (routers) handle everything; the DWDM network just carries standard Ethernet wavelengths.

## FlexE and Network Slicing

### Mapping Slice Types to FlexE

From Section 10.1, recall the three realization models:

| Model | Isolation Mechanism | FlexE Role |
|-------|-------------------|------------|
| **Soft** | VPN + QoS | None — no FlexE needed |
| **Enhanced Soft** | VPN + QoS + Flex-Algo | None — but FlexE can augment for critical links |
| **Hard** | FlexE + VPN + QoS | **Core component** — FlexE provides PHY-layer isolation |

In a hard slicing deployment:

```
                  Hard Slicing Stack
┌─────────────────────────────────────────────┐
│  Layer 4: Service Overlay (VPN/EVPN)         │  ← Per-slice VRF/EVI
├─────────────────────────────────────────────┤
│  Layer 3: Traffic Engineering (SR-TE/Flex-Algo)│  ← Per-slice path
├─────────────────────────────────────────────┤
│  Layer 2: QoS (scheduling/policing)          │  ← Per-slice SLA
├─────────────────────────────────────────────┤
│  Layer 1: FlexE (channelization)             │  ← Per-slice bandwidth
└─────────────────────────────────────────────┘
```

Each FlexE channel becomes an independent interface on the router. You then:
1. Apply QoS policies per channel (optional — the channel itself guarantees bandwidth)
2. Assign the channel to a Flex-Algo topology (for path isolation)
3. Bind VPN services to the channel

The result: a slice that's isolated at **every layer** from PHY through service overlay.

### FlexE vs Flex-Algo: When to Use Which

This is the question that generates the most debate in slice architecture meetings:

| Factor | Flex-Algo | FlexE |
|--------|-----------|-------|
| **Isolation level** | Topology-based (separate SPF, shared bandwidth) | Bandwidth-based (dedicated PHY capacity) |
| **Granularity** | Per-topology (coarse) | Per-interface, 5G increments (fine) |
| **Bandwidth guarantee** | Statistical — depends on traffic patterns | Deterministic — calendar-enforced |
| **Latency guarantee** | Best-effort within topology | Deterministic — TDM scheduling |
| **Complexity** | Low — IGP config only | High — per-link FlexE provisioning |
| **Cost** | Zero incremental (software feature) | Hardware-dependent (FlexE-capable NPUs) |
| **Scale** | Hundreds of nodes, easily | Per-link provisioning, complex at scale |
| **Failure behavior** | SPF reconverges in constrained topology | Channel bandwidth lost if PHY fails |

**The practical answer:** Use Flex-Algo for most slices (eMBB, mMTC). Use FlexE only for URLLC or wholesale isolation where deterministic bandwidth and latency guarantees are contractually required. Most operators deploy enhanced-soft slicing and reserve hard (FlexE) slicing for < 5% of traffic.

## Configuration

### Conceptual Configuration Model

FlexE configuration follows a three-step model on all platforms:

1. **Define the FlexE Group** — Assign physical ports to a group
2. **Create FlexE Clients** — Define logical interfaces with calendar slot assignments
3. **Use clients as interfaces** — Apply IP addressing, routing, QoS, VPN binding

> **⚠️ Important caveat on vendor configurations:** FlexE is a hardware-dependent feature with limited public documentation. The configurations below are **conceptual and representative** — they illustrate the logical configuration model (define group → create clients → use as interfaces) but **exact CLI syntax varies by platform, line card, and software release**. FlexE CLI documentation is often available only under NDA or in platform-specific release notes. Always verify against your specific platform's configuration guide before deploying. The conceptual model is consistent across vendors; the exact command hierarchy is not.

### IOS-XR (Cisco NCS 540 / NCS 5500 Series)

FlexE on IOS-XR uses the controller hierarchy for group and client management:

```cisco
! Step 1: Enable FlexE on the physical port
hw-module port-range 0 3 location 0/0/CPU0 mode flexe

! Step 2: Define FlexE Group (bond 4 × 100G ports)
controller FlexEGroup 0/0/0
 flexe-group id 1
 phy 0          ! Maps to HundredGigE0/0/0/0
 phy 1          ! Maps to HundredGigE0/0/0/1
 phy 2          ! Maps to HundredGigE0/0/0/2
 phy 3          ! Maps to HundredGigE0/0/0/3
!

! Step 3: Create FlexE Clients (channelization)
controller FlexEClient 0/0/0/0
 flexe-group 1
 client-id 1
 calendar-slots 0-7          ! Slots 0-7 = 40G (eMBB slice)
!
controller FlexEClient 0/0/0/1
 flexe-group 1
 client-id 2
 calendar-slots 8-13         ! Slots 8-13 = 30G (URLLC slice)
!
controller FlexEClient 0/0/0/2
 flexe-group 1
 client-id 3
 calendar-slots 14-17        ! Slots 14-17 = 20G (mMTC slice)
!
controller FlexEClient 0/0/0/3
 flexe-group 1
 client-id 4
 calendar-slots 18-19        ! Slots 18-19 = 10G (management slice)
!

! Step 4: Use FlexE Client interfaces like regular Ethernet
interface FlexEClient 0/0/0/0
 description eMBB-Slice-40G
 ipv4 address 10.0.12.1 255.255.255.252
 ipv6 address 2001:db8:12::1/126
!
interface FlexEClient 0/0/0/1
 description URLLC-Slice-30G
 ipv4 address 10.0.12.5 255.255.255.252
 ipv6 address 2001:db8:12::5/126
!

! Step 5: Bind to routing and VPN
router isis CORE
 interface FlexEClient 0/0/0/0
  circuit-type level-2-only
  address-family ipv4 unicast
   metric 100
  !
  address-family ipv6 unicast
   metric 100
  !
 !
 interface FlexEClient 0/0/0/1
  circuit-type level-2-only
  address-family ipv4 unicast
   metric 100
   ! Assign to Flex-Algo 129 (URLLC topology)
  !
 !
!

! Step 6: Bind URLLC client to Flex-Algo via affinity
segment-routing
 affinity-map
  name URLLC bit-position 0
 !
!
router isis CORE
 interface FlexEClient 0/0/0/1
  affinity flex-algo
   color URLLC
  !
 !
!
```

**Platform notes:**
- FlexE is supported on NCS 540 (specific variants with Memory Channel NPU) and NCS 5500 (J2-based line cards)
- `hw-module port-range ... mode flexe` is required before any FlexE configuration — this changes the port mode and requires a line card reload
- Calendar slot numbering is per-group (0 through (20 × num_instances - 1))
- Hitless calendar reconfiguration supported via `calendar-switch` command

### Junos (Juniper MX Series)

Juniper implements FlexE on MX Series with MPC11E line cards:

```junos
# Step 1: Configure FlexE Group
chassis {
    fpc 0 {
        pic 0 {
            flexe-group fg0 {
                # Add physical ports to the group

                member-interface et-0/0/1;
                member-interface et-0/0/2;

# Step 2: Create FlexE Clients
interfaces {
    # eMBB slice — 40G (slots 0-7)

                address 10.0.12.1/30;
            }
            family inet6 {
                address 2001:db8:12::1/126;

        }
    }
    # URLLC slice — 30G (slots 8-13)
    flexe-0/0/0/1 {
        flexe-client-id 2;
        flexe-group fg0;
        bandwidth 30g;

            family inet {
                address 10.0.12.5/30;
            }
            family inet6 {
                address 2001:db8:12::5/126;
            }
        }
    }
    # mMTC slice — 20G (slots 14-17)
    flexe-0/0/0/2 {
        flexe-client-id 3;
        flexe-group fg0;
        bandwidth 20g;
        unit 0 {
            family inet {
                address 10.0.12.9/30;

        }

```

**Platform notes:**
- MX Series requires MPC11E or later line cards for FlexE support

- FlexE group members must be same-rate PHYs (homogeneous groups only, per OIF spec)
- `show flexe group` and `show flexe client` for operational verification

Huawei's NE40E has the most mature FlexE implementation in deployed 5G transport networks, particularly in China:

**Platform notes:**

- Most widely deployed FlexE platform globally — China Mobile, China Telecom, and China Unicom 5G transport
- `display flexe-group` and `display interface FlexE` for operational state
- **Syntax varies significantly between VRP versions** — verify against your specific NE40E release

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| Calendar slot size | 5 Gbps | 5 Gbps | Finest standard granularity; 25G only useful for very high-rate clients |
| Group ID | — | Consistent end-to-end | Must match on both sides of the link |
| Client ID | — | Stable per slice | Changing client IDs disrupts traffic |
| Calendar type | A | A/B for hitless | Always configure both for live reconfiguration |
| Skew tolerance | 10 µs | Measure actual | If bonding over separate fiber paths, verify path skew |
| PHY fault propagation | Enabled | Enabled | Ensures fast detection of member link failures |

**Key things to verify:**
1. **Group state is UP** — all member PHYs operational
2. **Skew is well within 10 µs** — typically should be < 1 µs for same-chassis ports
3. **Overhead CRC errors = 0** — any errors indicate signal integrity or interop problems
4. **Allocated bandwidth = sum of all clients** — unallocated bandwidth is wasted
5. **Both calendars programmed** — even if using only A, B should be ready for hitless changes

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| FlexE Group stays DOWN | PHYs not in FlexE mode / mismatched group ID | Verify `hw-module mode flexe` and group IDs match on both ends |
| One PHY in group shows FAULT | Remote PHY Fault signaled by far end | Check far-end PHY state; likely optics or fiber issue on that PHY |
| High skew alarm | Bonded PHYs routed over different fiber paths with significant length difference | Measure path latency; ensure < 10 µs differential. Use same-route fiber if possible |
| Overhead CRC errors | Signal integrity issue or FlexE version mismatch | Check optic signal quality (OSNR, BER); verify same FlexE version on both ends |
| Client shows DOWN with group UP | Client ID mismatch or calendar conflict | Verify client IDs match end-to-end; check for overlapping slot assignments |
| Bandwidth mismatch after calendar change | Hitless switch not completed on both ends | Verify both A and B calendars programmed; check overhead for calendar switch acknowledgment |
| Packet drops on FlexE client | Traffic exceeds allocated channel bandwidth | This is **by design** — FlexE enforces the calendar. Police ingress or allocate more slots |
| FlexE not showing in `show interfaces` | Port not in FlexE mode | `hw-module port-range ... mode flexe` + line card reload required (IOS-XR) |

### Methodology

1. **Verify PHY layer first** — Are all member ports physically UP? Light levels OK?
2. **Check group state** — Is the FlexE group operational? All instances discovered?
3. **Verify overhead exchange** — Are overhead frames being sent/received without CRC errors?
4. **Confirm calendar match** — Do both ends agree on slot assignments? Same active calendar?
5. **Check client mapping** — Are client IDs consistent? Slot ranges non-overlapping?
6. **Monitor skew** — Is inter-PHY skew within tolerance?
7. **Verify end-to-end** — Can you ping across the FlexE client interface?
8. **Escalate if overhead sync fails** — Persistent overhead issues often indicate firmware bugs or interop problems between vendors

### War Stories

> **The Phantom Channel**
>
> A Tier 1 operator in Asia was deploying FlexE-based hard slicing for a 5G launch. Four slices over a 4 × 100G bonded group: eMBB (200G), URLLC (100G), mMTC (50G), and management (50G). The deployment went smoothly — all four slices came up, traffic flowed, SLAs met. The NOC high-fived. Then, three days before commercial launch, the URLLC slice started showing 2% packet loss during peak hours.
>
> The first instinct was "oversubscription" — but the URLLC channel was only running at 60% utilization. FlexE calendars don't lie; if you have 100G allocated, you get 100G. The counters showed drops happening at the far-end FlexE client, not the near-end. Signal quality was perfect.
>
> After 48 hours of troubleshooting, an engineer noticed that the calendar slot assignments on the near end used slots 20-39 for URLLC (instances 1-2), but the far end had slots 20-39 mapped to client ID 2 on instances 0-1. The group had four 100G PHYs, so four instances (0-3). The slot numbering was per-instance (0-19 each), but one end was configuring them as per-group (0-79 global) while the other used per-instance addressing.
>
> The root cause: two different engineers configured the two ends, and the Huawei CLI accepted both numbering schemes depending on software version. Slots 20-39 meant "instance 1, slots 0-19" on one end and "instance 0, slots 20-39" on the other (which doesn't exist on a 100G instance). The overhead frames showed the calendar as valid on both sides — because the calendar structure itself was consistent — but the actual data block mapping was misaligned.
>
> Fix: standardize on per-instance slot numbering, document it in the provisioning playbook, and add a pre-check that compares `display flexe-group calendar detail` output from both ends before any FlexE deployment goes live.
>
> **Lesson:** FlexE calendar slot numbering is the new VLAN mismatch. Both ends must agree not just on the calendar, but on the **addressing scheme for slots within the calendar.** This is a vendor interop minefield.

## Design Considerations

### When to Deploy FlexE

**Deploy FlexE when:**
- Contractual SLA requires **deterministic** bandwidth guarantees (not just priority queuing)
- Wholesale/multi-tenant isolation needs PHY-layer separation (regulatory or security requirement)
- Coherent optics deliver non-standard rates that waste bandwidth with fixed Ethernet ports
- Bonding lower-rate ports to higher aggregate is more cost-effective than upgrading to next standard rate
- 5G URLLC transport requires sub-millisecond jitter guarantees

**Don't deploy FlexE when:**
- QoS + Flex-Algo provides sufficient isolation (most eMBB/mMTC use cases)
- The additional provisioning complexity isn't justified (small networks, < 50 nodes)
- Hardware doesn't support it (many deployed routers lack FlexE-capable NPUs)
- Bandwidth is abundant and isolation is logical, not physical

### FlexE Deployment Scale (2025-2026 Reality)

FlexE deployment is concentrated in a few markets:
- **China**: Largest deployment globally. China Mobile, China Telecom, and China Unicom all use FlexE for 5G transport hard slicing. Primarily Huawei NE40E equipment.
- **Japan/Korea**: NTT DOCOMO and KT Corporation have selective FlexE deployments for enterprise URLLC services.
- **Europe**: Trial phase. Deutsche Telekom and Telefónica testing FlexE-based wholesale services.
- **North America**: Minimal deployment. Most operators use enhanced-soft slicing (VPN + QoS + Flex-Algo) and reserve FlexE for future enterprise 5G services.

The honest assessment: **most operators don't need FlexE today.** Enhanced-soft slicing handles 95%+ of slicing requirements. FlexE matters for the remaining 5% where "hard" isolation is non-negotiable.

### Bandwidth Planning with FlexE

FlexE granularity is 5 Gbps per slot. This has implications:

| Desired Rate | Slots Needed | Actual Rate | Waste |
|-------------|-------------|-------------|-------|
| 10 Gbps | 2 | 10 Gbps | 0% |
| 25 Gbps | 5 | 25 Gbps | 0% |
| 30 Gbps | 6 | 30 Gbps | 0% |
| 33 Gbps | 7 | 35 Gbps | 6% |
| 50 Gbps | 10 | 50 Gbps | 0% |
| 75 Gbps | 15 | 75 Gbps | 0% |
| 100 Gbps | 20 | 100 Gbps | 0% |

Since all client rates must be multiples of 5 Gbps, design your slice bandwidth allocations accordingly. The 5G granularity aligns well with typical slice requirements.

**FlexE 2.1 added 25G slot granularity** as an option for higher-rate groups where 5G granularity creates excessive calendar overhead. In a 4 × 400G group (1.6 Tbps), 5G slots mean 320 calendar entries — 25G slots reduce this to 64. **FlexE 3.0 added 100G slot granularity** for 800G PHY groups, further reducing calendar complexity at very high rates.

### Interaction with QoS

A common misconception: "FlexE replaces QoS." It doesn't.

FlexE provides **bandwidth isolation** — each channel gets its guaranteed pipe. But *within* a channel, you still need QoS for:
- **Multiple traffic classes within a slice** (voice, video, data within the eMBB channel)
- **Congestion management** when the channel is full
- **Marking/classification** for downstream treatment

The typical stack:
```
Physical Layer:   FlexE channel → guaranteed 40G pipe
QoS Layer:        Within 40G: 10G EF (voice), 15G AF (video), 15G BE (data)
Service Layer:    VPN binding per application
```

FlexE guarantees you won't be starved by other slices. QoS guarantees you won't be starved by other traffic *within your slice*.

- **ITU-T G.8312** (2020): Metro transport FlexE interfaces
- **IEEE 802.3** (various clauses): Underlying Ethernet PHY specifications

### Key Concepts
- FlexE shim between MAC and PCS — no physical layer changes required
- Calendar-based TDM: 5 Gbps slots (default) or 25 Gbps (FlexE 2.1+)
- 20 slots per 100G instance; 80 slots per 400G PHY (4 instances)
- Dual A/B calendars for hitless reconfiguration
- Max inter-PHY skew: 10 µs
- Three transport modes: unaware, termination, aware
