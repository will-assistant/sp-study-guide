# 10.1 — Network Slicing Concepts

## Overview
> Network slicing is the idea that one physical network can behave like many independent logical networks — each with its own performance guarantees, isolation boundaries, and lifecycle. It's the promise that sold 5G to every enterprise buyer, and the engineering reality that keeps transport architects up at night. This section covers the conceptual framework: what slicing means from 3GPP through IETF, how it maps to the transport network you've been building in Modules 1–9, and why it's harder than "just VPNs with SLAs."

## Theory

### What Is a Network Slice?

A network slice is an end-to-end logical network that provides specific network capabilities and characteristics. Three definitions matter:

**3GPP Definition (TS 23.501):**
> "A set of network function instances together with the required resources (e.g., compute, storage, networking) which form a complete instantiated logical network to meet certain network characteristics."

**IETF Definition (RFC 9543):**
> "A logical network topology connecting a number of endpoints using a set of shared or dedicated network resources that are used to satisfy specific service objectives."

**SP Engineer's Definition:**
> "A way to carve your backbone into isolated service planes — each with guaranteed bandwidth, latency, and jitter — without buying separate hardware."

The critical insight: 3GPP slicing is *end-to-end* (RAN + Core + Transport), but as a transport engineer, you own the *Transport Network (TN)* slice. Your slice must meet SLA guarantees for traffic between cell sites and core data centers, without knowing or caring about the radio or core network functions riding on top.

### The Three Domains of 5G Slicing

```
┌─────────────────────────────────────────────────────────────────────┐
│                     END-TO-END NETWORK SLICE                       │
│                                                                     │
│  ┌──────────────┐   ┌──────────────────┐   ┌──────────────────┐   │
│  │   RAN Slice   │   │  Transport Slice  │   │   Core Slice     │   │
│  │              │   │                  │   │                  │   │
│  │  gNB/CU/DU  │───│  IP/MPLS/Optical │───│  UPF/AMF/SMF    │   │
│  │  Radio       │   │  Backbone        │   │  Control + User  │   │
│  │  Resources   │   │  Resources       │   │  Plane Functions │   │
│  └──────────────┘   └──────────────────┘   └──────────────────┘   │
│                                                                     │
│  Owner: RAN vendor    Owner: SP Transport    Owner: Core vendor    │
│  (Ericsson/Nokia)     team (YOU)             (cloud-native CNF)    │
└─────────────────────────────────────────────────────────────────────┘
```

Each domain slices independently, coordinated by the 3GPP management system. As a transport engineer:
- You receive slice *requirements* (bandwidth, latency, jitter, isolation)
- You deliver slice *realization* (VPN + QoS + TE + possibly FlexE)
- You don't manage RAN scheduling or core function placement

### 3GPP Slice Types (S-NSSAI)

3GPP defines slices using the **Single Network Slice Selection Assistance Information (S-NSSAI)**, composed of:
- **SST (Slice/Service Type)** — 8-bit identifier for the slice category
- **SD (Slice Differentiator)** — 24-bit optional differentiator within an SST

| SST | Name | Description | Typical Transport SLA |
|-----|------|-------------|----------------------|
| 1 | eMBB | Enhanced Mobile Broadband | High bandwidth, moderate latency |
| 2 | URLLC | Ultra-Reliable Low-Latency | <1 ms one-way (radio+transport), 99.999% reliability |
| 3 | MIoT | Massive IoT | Low bandwidth, tolerant latency, massive scale |
| 4 | V2X | Vehicle-to-Everything | Low latency, high reliability, mobility |
| 5 | HMTC | High-Performance Machine-Type Communications | High reliability, medium latency (Rel-18+) |
| 6–127 | Reserved | Future standardized types | — |
| 128–255 | Operator-specific | Custom slice types | Per operator |

**Transport implications by slice type:**

- **eMBB**: Bandwidth is king. 10 Gbps+ per cell site during peak. SR-TE with bandwidth constraints, or Flex-Algo for topology isolation. Latency target: <10 ms RTT metro.
- **URLLC**: Latency and reliability dominate. The 3GPP end-to-end target is <1 ms one-way, but the **transport budget** is typically 100–500 μs for fronthaul and 1–5 ms for midhaul/backhaul (depending on RAN functional split). Requires pre-computed paths (TI-LFA), minimal hop counts, and possibly dedicated queuing (strict priority). FlexE hard isolation for critical deployments.
- **MIoT**: Scale matters more than speed. Millions of endpoints, kilobit flows. Efficient aggregation, minimal per-flow state. Best-effort with guaranteed minimum bandwidth.
- **V2X**: Combines URLLC reliability with mobility. Edge compute placement critical — UPF must be close to RAN.

### IETF Network Slice Framework (RFC 9543)

RFC 9543 (March 2024) defines the IETF's technology-agnostic framework for network slicing. Key concepts:

#### Architecture

```
┌─────────────────────────────────────────────────────────────────────┐
│  IETF Network Slice Customer                                        │
│  (3GPP management system, enterprise, or another NSC)               │
│                                                                     │
│              ▼ Network Slice Service Interface (NBI)                │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │          Network Slice Controller (NSC)                      │   │
│  │                                                             │   │
│  │  • Receives slice requests (SLO/SLE)                        │   │
│  │  • Maps to network resources                                │   │
│  │  • Manages lifecycle (create/modify/delete)                 │   │
│  │  • Monitors SLA compliance                                  │   │
│  │  • May use ACTN framework (RFC 8453)                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│              ▼ Southbound Interface (SBI)                           │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │          Network Infrastructure                              │   │
│  │                                                             │   │
│  │  Realized via: VPN + QoS + TE + Resource Reservation        │   │
│  │  Technologies: SR-TE, Flex-Algo, FlexE, MPLS, SRv6         │   │
│  └─────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────┘
```

#### Key Terminology

| Term | Definition |
|------|-----------|
| **IETF Network Slice** | A logical network with specific connectivity and performance characteristics |
| **Network Slice Service** | The service presented to the customer at the demarcation points |
| **SLO (Service Level Objective)** | Measurable performance targets (latency ≤5 ms, bandwidth ≥1 Gbps) |
| **SLE (Service Level Expectation)** | Desired but not strictly guaranteed attributes (e.g., preferred path diversity) |
| **NSC (Network Slice Controller)** | The entity that maps slice requests to network resources |
| **Connectivity Construct** | A set of endpoints and the connectivity between them within a slice |
| **Slice Demarcation Point** | Where the slice service is delivered to the customer (analogous to PE-CE interface) |

#### SLO Categories

RFC 9543 defines these measurable objectives for slices:

| Category | Examples | Transport Mechanism |
|----------|----------|-------------------|
| **Bandwidth** | Min guaranteed, max burst | QoS scheduling, admission control |
| **Latency** | One-way delay, RTT | Path computation, TE constraints |
| **Jitter** | Delay variation | Priority queuing, traffic shaping |
| **Availability** | 99.9%, 99.999% | FRR, dual-plane, disjoint paths |
| **Isolation** | Soft (QoS), hard (FlexE/TDM) | Resource partitioning |
| **Security** | Encryption, filtering | MACsec, IPsec, ACLs |

### Transport Network Slicing: How It Actually Works

Here's where the 3GPP slides meet the CLI. A transport network slice is realized using combinations of existing technologies — nothing magical, just disciplined application of what you already know:

#### Slicing Technology Stack

```
┌─────────────────────────────────────────────────────────────────────┐
│ Layer 4: Service Overlay                                            │
│   L3VPN (VPNv4/v6) or EVPN per slice                              │
│   RT-based isolation between tenants/slices                        │
├─────────────────────────────────────────────────────────────────────┤
│ Layer 3: Traffic Engineering                                        │
│   SR-TE Policies (color communities for slice steering)            │
│   Flex-Algo (topology-aware slicing — latency, affinity)           │
│   PCE (centralized path computation with slice constraints)        │
├─────────────────────────────────────────────────────────────────────┤
│ Layer 2: QoS & Scheduling                                          │
│   Per-slice queue mapping (TC → queue → scheduler)                 │
│   Hierarchical QoS (H-QoS) for per-slice bandwidth guarantees     │
│   WRED/ECN for congestion signaling within a slice                 │
├─────────────────────────────────────────────────────────────────────┤
│ Layer 1: Hard Isolation (optional)                                  │
│   FlexE for sub-port channelization                                │
│   Dedicated wavelengths/spectrum (optical layer)                   │
│   TDM (legacy, being replaced by FlexE)                            │
└─────────────────────────────────────────────────────────────────────┘
```

#### Realization Models

RFC 9889 (published November 2025) describes how existing IP/MPLS technologies map to 5G network slice requirements. Based on the realization patterns described in RFC 9889 and industry practice, three primary deployment models emerge:

**Model 1: VPN + QoS (Soft Slicing)**
- Each slice = separate L3VPN or EVPN instance
- QoS policy enforces bandwidth/latency SLOs per slice
- Shared infrastructure, statistical multiplexing
- **Best for**: eMBB, MIoT, cost-sensitive deployments
- **Limitation**: No hard bandwidth guarantee during extreme congestion

```
Slice A (eMBB)  ──┐
                   ├── Shared SR-MPLS/SRv6 Core ── QoS enforced at ingress PE
Slice B (MIoT)  ──┘
```

**Model 2: VPN + QoS + Flex-Algo (Enhanced Soft Slicing)**
- Each slice steered onto a dedicated Flex-Algo topology
- Different algorithms optimize for different metrics (latency vs bandwidth)
- Still shared physical infrastructure, but logically partitioned forwarding planes
- **Best for**: URLLC + eMBB coexistence, latency-sensitive slices
- **Limitation**: Flex-Algo doesn't enforce bandwidth — only path selection

```
Slice A (eMBB)   → Flex-Algo 128 (IGP metric, all links)
Slice B (URLLC)  → Flex-Algo 129 (min-delay, low-latency links only)
Slice C (MIoT)   → Flex-Algo 130 (IGP metric, exclude premium links)
```

**Model 3: FlexE + VPN + QoS (Hard Slicing)**
- FlexE creates deterministic sub-interfaces on physical Ethernet links
- Each FlexE group gets guaranteed, non-preemptable bandwidth
- VPN + QoS layers on top for service separation within each FlexE slice
- **Best for**: URLLC with regulatory isolation requirements, SLA-critical slices
- **Limitation**: Reduced statistical multiplexing, higher cost per bit

```
100GE Physical Link (FlexE Group — bonded PHYs)
├── FlexE Client 1 (60G) → URLLC slice → dedicated SR-TE paths
├── FlexE Client 2 (30G) → eMBB slice → shared SR-TE with QoS
└── FlexE Client 3 (10G) → MIoT slice → best-effort with floor
```

> **Terminology note:** A FlexE *Group* is the bonding of physical PHYs (e.g., 2×100GE bonded into 200G). FlexE *Clients* are the sub-rate channels carved from the group's aggregate bandwidth. Each slice maps to a FlexE Client, not a FlexE Group.

### Slice Isolation: The Spectrum

Not all isolation is created equal. Think of it as a spectrum from "honor system" to "physically separate":

| Level | Mechanism | Isolation Type | Overhead | Use Case |
|-------|-----------|---------------|----------|----------|
| 1 | VPN (RT/RD) | Control plane only | Minimal | Multi-tenant, no SLA |
| 2 | VPN + QoS | Control + data plane scheduling | Low | eMBB, managed services |
| 3 | VPN + Flex-Algo | Control + forwarding plane | Low-Medium | Latency-critical slices |
| 4 | VPN + SR-TE (dedicated) | Control + steering | Medium | Premium slices |
| 5 | FlexE + VPN + QoS | Hard bandwidth isolation | High | URLLC, regulatory |
| 6 | Dedicated wavelength/fiber | Physical isolation | Maximum | Government, military |

Most operators will use Levels 2–4 for 90% of slices. Level 5 (FlexE) is reserved for the slices that justify the cost. Level 6 is for when someone with stars on their shoulders is signing the contract.

### Slice Lifecycle Management

Slices aren't static — they have a lifecycle managed by the NSC:

```
                    ┌──────────┐
                    │  Request  │  Customer requests slice via NBI
                    └────┬─────┘  (SLO, SLE, topology, endpoints)
                         │
                    ┌────▼─────┐
                    │ Admission │  NSC checks: can the network support this?
                    │  Control  │  Resource availability, policy compliance
                    └────┬─────┘
                         │
                    ┌────▼─────┐
                    │  Create   │  NSC provisions: VPN + QoS + TE + FlexE
                    │           │  Configures PEs, activates paths
                    └────┬─────┘
                         │
                    ┌────▼─────┐
                    │  Monitor  │  Continuous SLA monitoring
                    │           │  Telemetry → NSC → alerts/remediation
                    └────┬─────┘
                         │
              ┌──────────┼──────────┐
              │          │          │
         ┌────▼───┐ ┌───▼────┐ ┌──▼──────┐
         │ Modify  │ │ Scale  │ │ Delete  │
         │ SLO/SLE │ │ Up/Down│ │ Tear    │
         └─────────┘ └────────┘ │ Down    │
                                └─────────┘
```

**Admission control** is the make-or-break function. The NSC must answer:
1. Do we have enough bandwidth on the required paths?
2. Can we meet the latency SLO given current network state?
3. Does this slice conflict with existing slices' guarantees?
4. Do we need to reoptimize existing slices to accommodate this one?

Without admission control, slicing is just VPNs with marketing attached.

### The S-NSSAI to Transport Mapping Problem

The hardest part of transport slicing isn't the technology — it's the *mapping*. 3GPP speaks in S-NSSAI. Your network speaks in VPNs, QoS classes, and TE policies. Something has to translate.

```
3GPP Domain                    Transport Domain
─────────────                  ─────────────────
S-NSSAI (1, 0x001)     →      VRF: slice-embb-001
  SST=1 (eMBB)                  RT: 65000:10001
  SD=0x001                       QoS: AF class, BW=10G guaranteed
                                 TE: Flex-Algo 128 or SR-TE color 100
                                 
S-NSSAI (2, 0x001)     →      VRF: slice-urllc-001
  SST=2 (URLLC)                  RT: 65000:20001
  SD=0x001                       QoS: EF class, strict priority
                                 TE: Flex-Algo 129 (min-delay)
                                 FlexE: Group 1 (dedicated 10G)
```

This mapping lives in the NSC and is typically implemented via YANG models and NETCONF/gRPC provisioning to the network elements.

**draft-ietf-teas-5g-network-slice-application** defines the formal mapping between S-NSSAI and IETF network slice attributes.

Network slicing doesn't replace what you know — it orchestrates it:

### Slice Realization: Model 2 (VPN + QoS + Flex-Algo)

This is the most common production deployment today. Three slices using Flex-Algo for topology separation and QoS for SLA enforcement.

#### IOS-XR (PE Router — NCS 5500/5700 or ASR 9000)

```cisco

!! Flex-Algo definitions (on all routers in the domain)
!! =============================================================

router isis CORE

  metric-type igp
  advertise-definition
 !
 ! Slice 2: URLLC — minimum delay, exclude high-latency links

  metric-type delay
  affinity exclude-any
   name high-latency
  !
  advertise-definition

  metric-type igp
  affinity exclude-any

  advertise-definition

!! =============================================================

segment-routing

            forwarding-class embb-queue scheduler embb-sched;
            forwarding-class miot-queue scheduler miot-sched;
            forwarding-class best-effort scheduler be-sched;
        }
    }
    schedulers {
        urllc-sched {
            priority strict-high;
            shaping-rate percent 15;
        }
        embb-sched {
            transmit-rate percent 60;   ## Align with IOS-XR: 60% of remaining
            buffer-size percent 40;
            drop-profile-map {
                loss-priority medium protocol any drop-profile embb-wred;
                loss-priority high protocol any drop-profile embb-wred-aggr;
            }
        }
        miot-sched {
            transmit-rate percent 20;   ## Align with IOS-XR: 20% of remaining
            buffer-size percent 20;
        }
        be-sched {
            transmit-rate remainder;
        }
    }
    interfaces {
        et-0/0/0 {
            scheduler-map slice-scheduler-map;
            unit 0 {
                classifiers {
                    dscp slice-classifier;
                }
            }
        }
    }
}
```

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| Flex-Algo range | 128-255 | 128-131 (start small) | Each algo is a full SPF computation — don't over-allocate |
| URLLC priority shaping | None | 10-20% of link | Prevents URLLC from starving other slices during burst |
| Delay measurement interval | 30s | 5-10s for URLLC paths | URLLC needs responsive delay-aware routing |
| Color community range | Any | Align with Flex-Algo ID | Color 128 = Algo 128 — simplifies operations |
| ODN vs static SR-TE | Static | ODN for scale | Manually defining policies per-PE per-slice doesn't scale |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Flex-Algo not participating on some routers | Router doesn't support the algo, or IS-IS topology partitioned | Check `show isis flex-algo` — ensure all routers advertise the definition; check IS-IS adjacency |
| SR-TE policy down for a slice color | No path available in that Flex-Algo topology | Verify Flex-Algo participation, check affinity exclusions aren't too aggressive |
| URLLC traffic taking high-latency path | Delay measurement not enabled on all links | Enable `performance-measurement` on every interface in the Flex-Algo 129 topology |
| Slice VRF has no routes | Color community not attached, or ODN template missing | Check `show bgp vpnv4 unicast` for color community; verify ODN color config |
| QoS drops in URLLC queue | Policer too aggressive, or traffic exceeds allocation | Increase policer rate; check for misclassified traffic entering URLLC queue |
| Inter-slice traffic leaking | Route-target misconfiguration or shared interface | Audit RT import/export; ensure CE interfaces are in correct VRF |
| NSC can't provision slices | NETCONF/gRPC connectivity, YANG model mismatch | Check NSC-to-PE connectivity, model version compatibility |
| Slice latency SLO violated | Path changed due to IGP reconvergence | Check TI-LFA coverage for Flex-Algo paths; consider static SR-TE for ultra-critical slices |

### Methodology

1. **Verify Flex-Algo health**: All routers participating? Consistent topology? `show isis flex-algo <id>` on every node.
2. **Check SR-TE policies**: ODN instantiated? Policy operational? `show segment-routing traffic-eng policy color <color>`.
3. **Validate VRF steering**: Traffic hitting the right policy? `show cef vrf <name> <prefix>` — look for the label stack.
4. **Inspect QoS counters**: Drops? Policer violations? `show policy-map interface` on every hop.
5. **Measure delay**: Per-hop delay sane? `show performance-measurement interfaces`.
6. **End-to-end test**: Inject marked traffic (correct DSCP), trace the path, measure latency/jitter at egress.
7. **Check telemetry pipeline**: NSC receiving metrics? Alerts firing for SLO violations?
8. **Audit admission control**: Did the NSC accept a slice the network can't support? Review NSC logs.

### War Stories

> **The Phantom Slice**
>
> A Tier-1 operator deployed three slices (eMBB, URLLC, MIoT) using Flex-Algo 128, 129, and 130. Everything looked perfect for two months. Then their URLLC customer — an autonomous vehicle fleet — reported intermittent 50ms latency spikes.
>
> Investigation revealed that Flex-Algo 129 (min-delay) was occasionally routing URLLC traffic through a 600km span that had been provisioned with the `high-latency` affinity *removed* during a maintenance window six weeks earlier. The technician cleared the affinity to test something, forgot to restore it, and the link quietly entered the URLLC topology.
>
> The fix was two lines of config. The lesson was deeper: Flex-Algo topology changes should go through the same change management as IGP metric changes. One forgotten affinity bit can silently violate an SLO for weeks before anyone measures it.
>
> They now run a nightly telemetry audit that compares Flex-Algo topology membership against a golden baseline. Any delta triggers a P1 alert.

## Design Considerations

### When to Use Network Slicing

**Use slicing when:**
- Multiple service types with fundamentally different SLAs share infrastructure
- Regulatory requirements demand demonstrable isolation (e.g., public safety, utilities)
- 5G transport requires per-S-NSSAI differentiation
- You need admission control to prevent over-subscription of premium SLAs

**Don't use slicing when:**
- A single QoS policy with proper scheduling is sufficient
- All tenants have the same SLA (just use VPN + QoS)
- The network is lightly loaded (slicing adds complexity for no benefit)
- You don't have the automation to manage slice lifecycle

### How Many Slices?

Start with 3–5. Seriously.

Every Flex-Algo runs a constrained SPF on every participating router (pruned to participating nodes/links, so smaller than a full IGP SPF — but still additional CPU and RIB/FIB state per prefix-SID × algorithm). Every slice VRF is state on every PE. Every QoS class is a queue on every interface. The operational complexity grows faster than the slice count.

| Slice Count | Complexity | Recommendation |
|-------------|-----------|----------------|
| 2-3 | Low | Most operators. eMBB + URLLC + best-effort. |
| 4-6 | Medium | Large operators with distinct wholesale customers |
| 7-10 | High | Only if per-slice Flex-Algo is NOT used (VPN+QoS only) |
| 10+ | Extreme | Avoid. Rethink the slicing model — aggregate similar SLAs |

### Flex-Algo vs SR-TE Policy for Slice Steering

| Criteria | Flex-Algo | SR-TE Policy |
|----------|-----------|-------------|
| Path computation | Distributed (each router runs SPF) | Centralized (PCE) or local |
| Topology awareness | Full IGP topology per algo | Per-policy path, not topology |
| Bandwidth guarantee | No (topology only, not scheduling) | Yes (with PCE bandwidth booking) |
| Scale | Few algos (3-5) across all nodes | Many policies per headend |
| Simplicity | High — define once, advertise everywhere | Medium — policy per PE pair |
| Best for | Latency-based slicing, topology partitioning | Bandwidth-guaranteed paths, complex constraints |

**Recommendation:** Use Flex-Algo for topology separation (latency vs. cost), layer SR-TE policies on top when you need bandwidth guarantees. They complement each other.

### Flex-Algo Partition and Fallback Behavior

**Critical design point:** If a Flex-Algo topology partitions (e.g., a link failure isolates a node from all Flex-Algo 129 paths), traffic destined for that node via algorithm 129 **black-holes**. There is no automatic fallback to algorithm 0 (default IGP). This is by design — falling back would violate the SLO the algorithm was designed to enforce.

Mitigation strategies:
- **Redundant topology**: Ensure every Flex-Algo topology has at least two diverse paths between all PE pairs
- **TI-LFA per algorithm**: Enable TI-LFA within each Flex-Algo for sub-50ms local repair
- **PCE re-optimization**: If using PCE, configure it to detect partition and signal alternative paths
- **Monitoring**: Alert on Flex-Algo participating node count changes — any drop may indicate partition risk
