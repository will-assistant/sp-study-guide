# 5.1 — Traffic Engineering Fundamentals

## Overview
> Traffic Engineering (TE) is the art and science of steering traffic away from the IGP shortest path to optimize network utilization, meet SLA constraints, and survive failures gracefully. In an SP backbone, TE is the difference between a network that wastes 60% of its capacity and one that runs at 85%+ utilization with sub-50ms failover. Every IE-SP and IE-SP candidate must understand TE mechanics cold — from CSPF computation to link coloring to bandwidth modeling.


## Key Concepts
> **After this section you will understand:**
> - how constrained paths are computed and signaled through the network
> - how bandwidth, affinity, and path constraints affect tunnel placement
> - tradeoffs between RSVP-TE and SR-based traffic engineering models
> - how to validate tunnel liveness, path intent, and failover behavior

## Theory

### Why TE Exists

IGP shortest-path routing has a fundamental limitation: it routes traffic based on a single additive metric with no awareness of link bandwidth, utilization, or administrative constraints. Consider a backbone with two parallel paths — a 100G direct link and a 2-hop path through 400G links. IGP picks the direct 100G (lower metric) even when it's saturated, while the 400G path sits nearly idle.

TE solves this by:
1. **Advertising link attributes** beyond basic metric (bandwidth, coloring, SRLGs)
2. **Computing constrained paths** that satisfy bandwidth/affinity requirements
3. **Establishing explicit-route tunnels** that override IGP forwarding
4. **Dynamically adapting** to traffic demands and topology changes

### TE Tunnel Lifecycle

A TE tunnel (LSP) goes through a well-defined lifecycle:

```
1. PATH COMPUTATION (Head-end or PCE)
   ├── Collect TED (Traffic Engineering Database)
   ├── Apply constraints (BW, affinity, explicit hops)
   └── Run CSPF → produce ERO (Explicit Route Object)

2. PATH SIGNALING (RSVP-TE)
   ├── Head-end sends PATH message with ERO
   ├── Each hop processes → allocates resources
   ├── Tail-end sends RESV message back
   └── Labels allocated hop-by-hop (downstream-on-demand)

3. FORWARDING STATE INSTALLATION
   ├── Each LSR installs label-swap entry in LFIB
   ├── Head-end installs tunnel as next-hop in routing
   └── Traffic begins flowing

4. MAINTENANCE
   ├── RSVP refresh (PATH/RESV every 30s default)
   ├── Hello keepalives (every 3s default, 3.5x timeout)
   ├── Periodic reoptimization (head-end re-runs CSPF)
   └── Auto-bandwidth adjustments (if enabled)

5. TEARDOWN
   ├── Administrative (config removal)
   ├── PathTear / ResvTear messages
   └── Preemption by higher-priority LSP
```

### The Traffic Engineering Database (TED)

The TED is a link-state database separate from the IGP LSDB, populated by TE extensions flooded via IS-IS or OSPF. Every TE-capable router maintains a TED containing:

| Attribute | Source TLV (IS-IS) | Source LSA (OSPF) | Description |
|-----------|-------------------|-------------------|-------------|
| Maximum link bandwidth | Sub-TLV 9 (RFC 5305) | Type 10 Opaque LSA (RFC 3630) | Physical capacity |
| Maximum reservable bandwidth | Sub-TLV 10 | Sub-TLV 6 | Admin-configured ceiling |
| Unreserved bandwidth (per priority) | Sub-TLV 11 | Sub-TLV 7 | 8 priority levels (0-7) |
| TE metric | Sub-TLV 18 | Sub-TLV 5 | Independent of IGP metric |
| Admin group (affinity) | Sub-TLV 3 | Sub-TLV 9 | 32-bit bitmask |
| Link local/remote ID | Sub-TLV 4/5 | Sub-TLV 11 | Unnumbered link support |
| SRLG | Sub-TLV 16 (RFC 4205) | Sub-TLV (RFC 4203) | Shared risk group |

### CSPF Algorithm

Constrained Shortest Path First (CSPF) is a modified Dijkstra that prunes links violating constraints before running SPF:

```
CSPF Algorithm:
1. PRUNE phase:
   a. Remove links with unreserved BW < requested BW (at setup priority)
   b. Remove links failing affinity check:
      - include-any: link.affinity & tunnel.include-any != 0
      - include-all: link.affinity & tunnel.include-all == tunnel.include-all
      - exclude-any: link.affinity & tunnel.exclude-any == 0
   c. Remove links in excluded SRLG list
   d. Remove explicitly excluded addresses

2. SPF phase:
   Run Dijkstra on pruned topology using TE metric (or IGP metric if TE metric not set)

3. TIE-BREAKING (vendor-specific order):
   a. Largest minimum bandwidth on path
   b. Fewest hops
   c. Lowest router-id (to break final ties)
```

**Critical exam point**: CSPF tie-breaking order varies between vendors. Cisco IOS-XE/XR: max-bandwidth → fewest-hops → lowest-address. Junos: random load-balancing by default (configurable).

### TE Metric vs IGP Metric

Two entirely separate metrics can coexist:

| Property | IGP Metric | TE Metric |
|----------|-----------|-----------|
| Used by | SPF for routing table | CSPF for TE path computation |
| Scope | All traffic by default | Only TE tunnel paths |
| Typical assignment | Inverse capacity (reference BW / link BW) | Latency, cost, admin preference |
| Default if unset | Interface default (1-10 depending on vendor/speed) | Equals IGP metric |

**Design pattern**: Use IGP metric for capacity-based routing, TE metric for latency-sensitive paths. A 100G terrestrial link and a 100G submarine link have the same IGP metric (same capacity) but wildly different TE metrics (5ms vs 80ms latency).

### IS-IS TE Extensions (RFC 5305)

IS-IS carries TE information in extended TLVs:

- **TLV 22** — Extended IS Reachability (replaces old TLV 2)
  - Sub-TLVs carry bandwidth, TE metric, affinity, SRLG
- **TLV 134** — TE Router ID (stable loopback address)
- **TLV 135** — Extended IP Reachability (prefix info)

IS-IS TE flooding behavior:
- TE sub-TLVs flood within the IS-IS level where they originate
- Level-1 TE info does NOT leak to Level-2 by default
- For inter-area TE, explicit route leaking or PCE is required

### OSPF TE Extensions (RFC 3630)

OSPF uses Type 10 Opaque LSAs (area-scoped) for TE:

- **Type 1 TLV** — Router Address (TE Router ID)
- **Type 2 TLV** — Link TLV with sub-TLVs for BW, metric, affinity
- Area-scoped means TE information stays within the OSPF area
- Inter-area TE requires Type 11 (AS-scoped) Opaque LSAs or PCE

### Link Coloring / Administrative Groups (Affinities)

Affinities are a 32-bit bitmask assigned to each link, used to include/exclude links from TE paths:

```
Bit 0  = "gold"       (premium fiber, diverse path)
Bit 1  = "silver"     (standard metro fiber)  
Bit 2  = "submarine"  (undersea cable segments)
Bit 3  = "microwave"  (wireless backhaul)
Bit 4  = "leased"     (third-party dark fiber)
Bit 5  = "low-latency" (sub-5ms links only)
...
Bit 31 = "exclude-maintenance" (links under planned work)
```

Tunnel constraint types:
- **affinity / include-any**: Path MUST traverse at least one link matching any set bit
- **affinity / include-all**: Path MUST traverse only links matching ALL set bits
- **affinity / exclude-any**: Path MUST NOT traverse any link matching any set bit

### Bandwidth Constraints — DS-TE

DiffServ-aware TE (DS-TE, RFC 4124) extends bandwidth reservations to multiple classes:

- **MAM (Maximum Allocation Model)**: Each class-type gets a hard bandwidth ceiling. Simple but wastes unreserved bandwidth.
- **RDM (Russian Dolls Model)**: Class-types nest — higher-priority can borrow from lower. Better utilization.

Eight priority levels (0=highest, 7=lowest):
- **Setup priority**: Priority used when establishing the LSP
- **Hold priority**: Priority used to defend against preemption
- Rule: hold ≤ setup (numerically), or you create thrashing (hold should be at least as strong as setup)
- Default: setup=7, hold=7 (lowest priority)

## Configuration

### IOS-XE Configuration

#### Enable TE on IS-IS
```cisco
! Enable MPLS TE globally
mpls traffic-eng tunnels

! IS-IS TE extensions
router isis CORE
 metric-style wide
 mpls traffic-eng router-id Loopback0
 mpls traffic-eng level-2

! Enable TE on interfaces
interface TenGigabitEthernet0/0/0
 mpls traffic-eng tunnels
 isis network point-to-point
 ip rsvp bandwidth 8000000 8000000
 !                 ^max-reservable ^max-per-flow (kbps)
 mpls traffic-eng administrative-weight 10
 mpls traffic-eng attribute-flags 0x00000001
 ! Bit 0 set = "gold"
```

#### Enable TE on OSPF
```cisco
mpls traffic-eng tunnels

router ospf 1
 mpls traffic-eng router-id Loopback0
 mpls traffic-eng area 0

interface TenGigabitEthernet0/0/0
 mpls traffic-eng tunnels
 ip rsvp bandwidth 8000000 8000000
 mpls traffic-eng administrative-weight 10
 mpls traffic-eng attribute-flags 0x00000003
```

#### Basic TE Tunnel
```cisco
interface Tunnel100
 ip unnumbered Loopback0
 tunnel mode mpls traffic-eng
 tunnel destination 10.0.0.5
 tunnel mpls traffic-eng bandwidth 1000000
 ! Request 1 Gbps
 tunnel mpls traffic-eng priority 4 4
 ! Setup priority 4, hold priority 4
 tunnel mpls traffic-eng affinity 0x00000001 mask 0x00000001
 ! MUST have bit 0 set (gold links only)
 tunnel mpls traffic-eng path-option 1 dynamic
 ! Let CSPF compute the path
 tunnel mpls traffic-eng path-option 2 explicit name BACKUP-PATH
 ! Fallback to explicit path
```

#### Explicit Path Definition
```cisco
ip explicit-path name BACKUP-PATH enable
 next-address 10.1.1.2
 next-address 10.1.2.2
 next-address 10.0.0.5
 ! Strict hops through specific router interfaces

ip explicit-path name LOOSE-VIA-P3 enable
 next-address loose 10.0.0.3
 next-address 10.0.0.5
 ! Loose hop — CSPF fills in between
```

#### TE Metric Configuration
```cisco
! Set TE metric independently of IGP metric
interface TenGigabitEthernet0/0/0
 mpls traffic-eng administrative-weight 50
 ! TE metric = 50 (IGP metric may be 10)
```

### Junos Configuration

#### Enable TE with IS-IS
```junos
# Enable MPLS and RSVP on interfaces
set interfaces ge-0/0/0 unit 0 family mpls
set protocols mpls interface ge-0/0/0.0
set protocols rsvp interface ge-0/0/0.0
set protocols rsvp interface ge-0/0/0.0 bandwidth 10g
# Sets max reservable bandwidth

# IS-IS TE extensions (automatic with wide-metrics)
set protocols isis interface ge-0/0/0.0 level 2 te-metric 10
set protocols isis traffic-engineering

# Enable CSPF
set protocols mpls label-switched-path to-PE5 to 10.0.0.5
set protocols mpls label-switched-path to-PE5 bandwidth 1g
set protocols mpls label-switched-path to-PE5 priority 4 4
# Setup-priority 4, hold-priority 4
```

#### TE Tunnel with Affinity
```junos
# Define admin groups (named, mapped to bit positions)
set protocols mpls admin-group gold 0
set protocols mpls admin-group silver 1
set protocols mpls admin-group submarine 2
set protocols mpls admin-group microwave 3

# Assign admin groups to interfaces
set protocols mpls interface ge-0/0/0.0 admin-group gold
set protocols mpls interface ge-0/0/1.0 admin-group silver

# LSP with affinity constraints
set protocols mpls label-switched-path to-PE5 to 10.0.0.5
set protocols mpls label-switched-path to-PE5 admin-group include-any gold
set protocols mpls label-switched-path to-PE5 admin-group exclude submarine
```

#### Explicit Path (ERO)
```junos
# Named path with strict/loose hops
set protocols mpls path VIA-P2-P3 10.1.1.2 strict
set protocols mpls path VIA-P2-P3 10.1.2.2 strict
set protocols mpls path VIA-P2-P3 10.0.0.5 strict

set protocols mpls label-switched-path to-PE5 primary VIA-P2-P3
set protocols mpls label-switched-path to-PE5 secondary VIA-P4
# Automatic failover to secondary path
```

#### TE Metric (Junos)
```junos
# TE metric is per-interface, per-level in IS-IS
set protocols isis interface ge-0/0/0.0 level 2 te-metric 50

# Or directly on MPLS interface
set protocols mpls interface ge-0/0/0.0 admin-group gold
# Admin groups are on MPLS stanza, te-metric on ISIS stanza
```

### Key Knobs

| Parameter | IOS-XE Default | Junos Default | Recommended | Why |
|-----------|---------------|---------------|-------------|-----|
| RSVP refresh interval | 30s | 30s | 30s (use summary refresh) | Lower = more overhead |
| RSVP hello interval | 3s | 3s (9s dead) | 3s hello / 9s dead | Balance detection speed vs CPU |
| Reoptimization timer | 3600s | 300s | 300-600s | Junos default is more aggressive |
| TE metric | = IGP metric | = IGP metric | Set explicitly for latency paths | Don't rely on IGP metric for TE |
| Max reservable BW | 0 (disabled) | 0 (disabled) | 75-80% of link capacity | Leave headroom for IGP/BFD |
| Setup priority | 7 | 7 | 0-3 for critical, 4-7 for best-effort | Lower number = higher priority |
| CSPF tie-breaking | max-bw → min-hops | random | Vendor default | Junos random distributes load |

```
      Bandwidth: 1Gbps
```

### Key Fields to Check
- **Oper state**: Must be "up" / "Up"
- **Path: valid**: CSPF found a path meeting constraints
- **Signalling: connected**: RSVP session established end-to-end
- **ERO matches RRO**: What was requested matches what was set up
- **Bandwidth**: Confirm reserved BW matches config
- **Tun_Instance**: Incrementing instance means tunnel has been re-signaled (reopt, failure recovery)

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Tunnel admin up, oper down | CSPF failure — no path meets constraints | Check `show mpls traffic-eng topology brief` for TED completeness; relax constraints |
| Tunnel up but "path not valid" | TED stale or incomplete | Verify IS-IS/OSPF TE extensions enabled; check `mpls traffic-eng tunnels` on all interfaces |
| Tunnel flapping | RSVP hello timeout | Check Layer 1/2 stability; tune hello timers; check for unidirectional failures |
| BW reservation rejected | Insufficient unreserved BW at required priority | Check `show mpls traffic-eng link-management bandwidth-allocation`; preempt lower priority or increase link BW cap |
| Affinity mismatch | Link admin-group bits don't match tunnel constraint | Map out bit assignments across all links; use `show mpls traffic-eng topology` to verify per-link affinity |
| Tunnel takes suboptimal path | TE metric not configured (falling back to IGP metric) | Explicitly set TE metric on all links; verify with `show ted database extensive` |
| TED empty or missing links | IS-IS `metric-style narrow` still in use, or TE not enabled on all interfaces | Must use `metric-style wide`; enable `mpls traffic-eng tunnels` on every P2P interface |
| Suboptimal after link recovery | Reoptimization timer hasn't fired yet | Manually reoptimize: `mpls traffic-eng reoptimize` (IOS-XE) or `clear mpls lsp optimize` (Junos) |

### Methodology

1. **Verify TED completeness**: Every link in the path must appear in TED with correct attributes
2. **Check CSPF inputs**: Bandwidth request, affinity, explicit hops — does a valid path exist?
3. **Trace RSVP signaling**: PATH message → each hop → RESV back. Where does it fail?
4. **Verify LFIB**: Label swap entries installed at each hop?
5. **Check forwarding**: Is traffic actually using the tunnel? (autoroute, static route, policy)
6. **Monitor BW accounting**: Reservation vs actual traffic — are you over-subscribing?

### War Story: The TE Metric That Wasn't

> A Tier-2 ISP deployed RSVP-TE tunnels for their core mesh, using dynamic paths with bandwidth constraints. Everything worked fine until they added latency-sensitive SR-TE policies using TE metrics. The TE metrics had never been explicitly configured — they defaulted to IGP metric values. A new submarine link got a low IGP metric (high capacity = low cost), so TE path computation happily routed latency-sensitive traffic across the Atlantic and back. The team spent 3 days chasing packet loss that turned out to be 120ms of added latency on VoIP traffic. **Fix**: Explicit TE metric policy — every link gets a TE metric reflecting actual propagation delay. They wrote a script to measure RTT and auto-generate TE metrics. Rule: **never assume TE metric = IGP metric is what you want in production**.

## Scaling Considerations

| Parameter | Practical Limit | Notes |
|-----------|----------------|-------|
| TE tunnels per head-end | 5,000-15,000 (platform dependent) | ASR 9000 supports ~10K; MX Series ~15K |
| RSVP sessions per router (transit) | 10,000-50,000 | Refresh reduction (RFC 2961) mandatory at scale |
| TED entries | Scales with network links | 10,000+ links common in large SPs |
| Reoptimization frequency | 300-3600s | More frequent = more CPU, more churn |
| BW update flooding rate | Threshold-based (±10% default) | Avoid constant flooding on busy links |

## Review Questions

1. **Scenario**: You configure a TE tunnel requesting 5 Gbps with `affinity 0x5 mask 0x5`. The tunnel fails to come up. TED shows all links have affinity `0x00000003`. Why does CSPF fail, and what's the fix?

2. **Design**: Your backbone has 400G core links and 100G edge links. You want TE tunnels to prefer the 400G path even when the 100G direct path has fewer hops. How would you configure metrics to achieve this without changing IGP routing?

3. **Troubleshooting**: A TE tunnel shows `Oper: up, Path: valid` but traffic isn't flowing through it. The destination prefix still uses the IGP next-hop. What's missing?

4. **Scaling**: You're deploying a full-mesh of TE tunnels across 200 PE routers. Calculate the number of tunnels and RSVP sessions. What mechanisms would you use to keep this manageable?

5. **Theory**: Explain why hold priority should always be numerically ≤ setup priority (hold stronger than setup). What happens if setup=3 and hold=5?
