# 5.4 — TE Deployment & Design

## Overview
> Theory without deployment is a whiteboard exercise. This section covers how real SP networks deploy traffic engineering: capacity planning with TE, failure modeling to validate tunnel placement, the practical mechanics of LDP-over-RSVP (the most common MPLS transport design of the past decade), Fast Reroute for sub-50ms protection, and the hard-won lessons of running TE at scale across hundreds of routers and thousands of tunnels. If you can design and deploy everything in this section, you can run TE in production.

> - how bandwidth, affinity, and path constraints affect tunnel placement
> - tradeoffs between RSVP-TE and SR-based traffic engineering models

## Theory

### Capacity Planning with TE

TE capacity planning is a closed-loop process:

```
Capacity Planning Loop:
1. MEASURE
   ├── Collect traffic matrix (SNMP, streaming telemetry, Netflow)
   ├── Per-tunnel utilization (auto-bandwidth samples)
   └── Per-link utilization (interface counters)

2. MODEL
   ├── Build topology model (nodes, links, capacities, metrics)
   ├── Import traffic matrix as demands
   ├── Simulate TE tunnel placement (CSPF or PCE algorithm)
   └── Compute per-link utilization under normal state

3. STRESS TEST
   ├── Simulate single link failure (SRLG-aware)
   ├── Simulate single node failure
   ├── Simulate dual failure (if SLA requires)
   └── Check: any link > threshold? (typically 80% under failure)

4. OPTIMIZE
   ├── Adjust tunnel paths (add/remove explicit hops)
   ├── Add capacity where failure-case utilization > threshold
   ├── Rebalance affinity assignments
   └── Validate with next iteration of MODEL

5. DEPLOY
   ├── Push config changes (ideally via automation/PCEP)
   ├── Monitor convergence and utilization
   └── Feed back to MEASURE
```

**Capacity planning tools**:
- Cisco WAE (WAN Automation Engine) — gold standard, expensive
- Juniper NorthStar / Paragon Pathfinder
- MATE (acquired by Cisco → WAE)
- Open-source: Net2Plan, OpenTRANS
- Custom: Many Tier-1s build their own using traffic matrices + linear programming

### Failure Modeling

The most critical part of TE design is answering: "What happens when things break?"

**Failure scenarios to model** (in order of likelihood):
1. **Single link failure**: Most common. Every tunnel must have a valid alternate path.
2. **Single node failure**: Less common but devastating. All tunnels transiting the node must reroute.
3. **SRLG failure**: Fiber cut takes out multiple links. This is where most TE designs break.
4. **Dual failure**: Two simultaneous independent failures. Required for some financial SLAs.
5. **Maintenance window**: Planned node/link removal. Model with RSVP graceful shutdown.

**The 80/50 Rule** (common SP design guideline):
- Under normal conditions: no link above 50% utilization
- Under single failure: no link above 80% utilization
- This leaves 20% headroom for micro-bursts and measurement lag

### LDP-over-RSVP (LDPoRSVP)

The most common MPLS transport architecture in SP networks for the past 15+ years:

```
LDPoRSVP Architecture:
                    RSVP-TE Tunnel (transport LSP)
  PE1 ═══════════════════════════════════════════ PE5
       └── LDP label (service LSP) rides inside ──┘
       
Label Stack (at PE1 egress):
  ┌──────────────────┐
  │ RSVP label (top) │ ← Swapped hop-by-hop by P routers
  ├──────────────────┤
  │ LDP label         │ ← Unchanged until PE5 (PHP may remove RSVP label)
  ├──────────────────┤
  │ VPN/service label │ ← Service-specific (L3VPN, VPLS, etc.)
  └──────────────────┘
```

**Why LDPoRSVP?**
- LDP provides reachability to all PE loopbacks (simple, scales well)
- RSVP-TE provides traffic-engineered transport between strategic PE pairs
- LDP sessions run over RSVP tunnels (LDP targeted sessions or IGP autoroute)
- Services (L3VPN, L2VPN) use LDP labels for PE-to-PE forwarding
- TE controls HOW traffic traverses the core; LDP provides WHERE traffic goes

**How it works mechanically:**

1. RSVP-TE tunnels established between PE pairs (or PE-to-P strategic points)
2. `autoroute announce` (IOS-XE) or `install` (Junos) makes IGP use the tunnel
3. LDP sees the tunnel endpoint as directly connected (IGP distance = 0 via tunnel)
4. LDP establishes targeted session over the tunnel
5. LDP labels are pushed under RSVP labels
6. Traffic: VPN label + LDP label + RSVP label (3-label stack)

### Fast Reroute (FRR) — RFC 4090

FRR provides sub-50ms protection for RSVP-TE tunnels by pre-computing and pre-signaling backup paths:

#### Facility Backup (Bypass Tunnels)

```
Normal Path:     PE1 ──→ P2 ──→ P3 ──→ PE5
                              ↑
                         Failure here!
                              
With FRR Bypass:  PE1 ──→ P2 ─╌╌→ P4 ──→ P3 ──→ PE5
                              └─ bypass tunnel ─┘
                              (pre-signaled, label-swapped)
```

**Link protection**: Bypass tunnel protects against the link between P2 and P3 failing. Tunnel goes around the failed link.

**Node protection**: Bypass tunnel protects against P3 (the next-next-hop) failing. Tunnel goes around P3 entirely.

```
Node Protection:
  Normal:          PE1 → P2 → [P3] → PE5
  Bypass:          PE1 → P2 → P4 → PE5
                              └──────┘
                         bypass tunnel avoids P3 entirely
                         (merge point MUST be downstream of P3)
```

**Label operations during FRR:**

```
Normal forwarding at P2:
  Incoming: [RSVP-label-A] | [LDP] | [VPN]
  Action: swap RSVP-label-A → RSVP-label-B, send to P3

After failure (FRR active) at P2:
  Incoming: [RSVP-label-A] | [LDP] | [VPN]
  Action: push bypass-label on top → [Bypass-label] | [RSVP-label-B] | [LDP] | [VPN]
          send to P4 (bypass tunnel first hop)

At P4 (bypass tunnel tail): 
  Pop bypass-label, forward [RSVP-label-B] | [LDP] | [VPN] to PE5
```

#### One-to-One Backup (Detour)

Each protected LSP gets its own backup path (not shared). More state but more granular control.

- **Facility backup**: Shared bypass tunnel protects ALL LSPs traversing the protected link/node. Efficient. Preferred at scale.
- **One-to-one backup**: Dedicated backup per LSP. More state. Used when bandwidth guarantees needed on backup path.

### SR-TE FRR: TI-LFA

SR-TE uses Topology-Independent Loop-Free Alternate (TI-LFA) instead of RSVP FRR:

```
TI-LFA Computation:
1. For each primary next-hop, compute post-convergence SPF
   (SPF with the failed link/node removed)
2. Find the post-convergence path
3. Encode the backup path as a segment list (repair segments)
4. Pre-install in FIB as backup next-hop

Advantage: No backup tunnel signaling. No RSVP. No state.
           Works for ANY topology. Coverage: typically 100%.
```

### TE at Scale — Architecture Patterns

#### Full-Mesh TE Tunnels
```
N PEs = N*(N-1) tunnels
100 PEs = 9,900 tunnels
200 PEs = 39,800 tunnels ← starts to hurt
```

Full-mesh is the simplest model but doesn't scale beyond ~200 PEs.

#### Hierarchical TE (Hub-and-Spoke)
```
Tier-1: Core P-to-P tunnels (few, large, managed)
Tier-2: PE-to-Core-P tunnels (autoroute into core)
Tier-3: LDP provides PE-to-PE reachability over Tier-1+2 tunnels
```

Reduces tunnel count dramatically. Most Tier-1 SPs use this model.

#### Forwarding Adjacency (FA)
TE tunnels advertised as links in IGP. Other routers see them as regular links and can compute SPF over them. Powerful but complex — recursive routing risks.

#### Autoroute vs Static Routing into Tunnels

| Method | Mechanism | Scope | Risk |
|--------|-----------|-------|------|
| `autoroute announce` | Inject tunnel into IGP SPF as shortcut | All prefixes behind tunnel endpoint | Over-steering if not careful |
| `autoroute destination` | Only tunnel endpoint prefix uses tunnel | Single prefix | Minimal — most conservative |
| Static route | `ip route 0.0.0.0/0 tunnel100` | Explicit | Manual but controlled |
| Policy-based | PBR or SR-TE color steering | Per-VPN or per-prefix | Granular |

## Configuration

### IOS-XE: LDP-over-RSVP

```cisco
! Step 1: RSVP-TE tunnel
interface Tunnel100
 ip unnumbered Loopback0
 tunnel mode mpls traffic-eng
 tunnel destination 10.0.0.5
 tunnel mpls traffic-eng bandwidth 5000000
 tunnel mpls traffic-eng priority 4 4
 tunnel mpls traffic-eng path-option 1 dynamic
 tunnel mpls traffic-eng autoroute announce
 ! ^^^ THIS is what makes LDPoRSVP work
 ! IGP sees PE5's loopback via this tunnel
 ! LDP session to PE5 established over the tunnel

! Step 2: Verify LDP follows the tunnel
! LDP targeted hello to PE5 will use the tunnel
! No additional LDP config needed — autoroute handles it

! Step 3: Confirm tunnel metrics
 tunnel mpls traffic-eng autoroute metric absolute 10
 ! Override IGP metric for this tunnel (optional)
 ! Or: tunnel mpls traffic-eng autoroute metric relative -5
```

### IOS-XE: Fast Reroute (Facility Backup)

```cisco
! On the protected TE tunnel (head-end)
interface Tunnel100
 tunnel mpls traffic-eng fast-reroute
 ! Enable FRR protection for this tunnel
 tunnel mpls traffic-eng fast-reroute node-protect
 ! Request node protection (not just link protection)

! On each P router — create bypass tunnels
! Option A: Let MPLS TE auto-create bypass tunnels
mpls traffic-eng auto-tunnel backup
 tunnel-num min 1000 max 1999
 ! Auto-created bypass tunnels use IDs 1000-1999

! Option B: Manual bypass tunnel
interface Tunnel500
 ip unnumbered Loopback0
 tunnel mode mpls traffic-eng
 tunnel destination 10.0.0.3
 ! Bypass merges at P3 (next-next-hop for node protection)
 tunnel mpls traffic-eng path-option 1 explicit name BYPASS-P3
 tunnel mpls traffic-eng priority 4 4

! Explicit path excluding protected node
ip explicit-path name BYPASS-P3 enable
 exclude-address 10.0.0.3
 ! Exclude the node being protected

! Enable FRR on the interface
interface TenGigabitEthernet0/0/0
 mpls traffic-eng backup-path Tunnel500
 ! This interface protected by Tunnel500
```

### IOS-XE: Autoroute Variants

```cisco
! Autoroute announce — all prefixes behind endpoint use tunnel
interface Tunnel100
 tunnel mpls traffic-eng autoroute announce

! Autoroute destination — ONLY endpoint prefix uses tunnel
interface Tunnel100
 tunnel mpls traffic-eng autoroute destination

! Forwarding adjacency — advertise tunnel as IGP link
interface Tunnel100
 tunnel mpls traffic-eng forwarding-adjacency
 ! Other routers see this tunnel as a link in their SPF
 ! WARNING: can cause routing loops if not carefully designed
```

### IOS-XE: RSVP Graceful Shutdown (Planned Maintenance)

```cisco
! Before maintenance — signal all tunnels to reroute
interface TenGigabitEthernet0/0/0
 mpls traffic-eng administrative-weight 4294967295
 ! Set TE metric to maximum — tunnels will reoptimize away

! Or use RSVP graceful shutdown
interface TenGigabitEthernet0/0/0
 ip rsvp bandwidth 0 0
 ! Set available bandwidth to 0 — new tunnels avoid, existing reroute
```

### Junos: LDP-over-RSVP

```junos
# Step 1: RSVP-TE LSP
set protocols mpls label-switched-path to-PE5 to 10.0.0.5
set protocols mpls label-switched-path to-PE5 bandwidth 5g
set protocols mpls label-switched-path to-PE5 priority 4 4
set protocols mpls label-switched-path to-PE5 install 10.0.0.5/32
# ^^^ Install PE5's loopback route via this LSP
# This is the Junos equivalent of "autoroute announce" for a specific prefix

# For broader autoroute behavior:
set protocols mpls label-switched-path to-PE5 install 0.0.0.0/0
# NOT recommended — use specific prefix or inet.3 default behavior

# Junos default: LSP routes go into inet.3
# LDP resolves BGP next-hops via inet.3 → picks up LSP automatically
# No "autoroute announce" needed in most Junos deployments!

# Explicit LDP-over-RSVP: targeted LDP session
set protocols ldp track-igp-metric
set protocols ldp interface lo0.0
# LDP resolves its transport via inet.3 which contains RSVP LSPs
```

### Junos: Fast Reroute (Facility Backup)

```junos
# Enable FRR on the LSP (head-end)
set protocols mpls label-switched-path to-PE5 fast-reroute

# Node-link protection (node protection preferred, link as fallback)
set protocols mpls label-switched-path to-PE5 node-link-protection

# On P routers — auto-create bypass tunnels
set protocols mpls interface ge-0/0/0.0 link-protection
# ^^^ Junos auto-creates bypass tunnels for this interface

# For node protection:
set protocols mpls interface ge-0/0/0.0 node-link-protection

# Verify bypass creation
# show mpls lsp bypass
# show rsvp interface detail (shows protection status)
```

### Junos: TE at Scale — Optimization

```junos
# RSVP refresh reduction (mandatory at scale)
set protocols rsvp interface ge-0/0/0.0 aggregate
set protocols rsvp interface ge-0/0/0.0 reliable
# 'aggregate' enables summary refresh bundling
# 'reliable' enables reliable messaging (ACK-based, fewer retransmits)

# RSVP hello tuning
set protocols rsvp interface ge-0/0/0.0 hello-interval 3
# Default 3s — leave as-is unless scaling dictates otherwise

# Reoptimization dampening
set protocols mpls optimize-timer 600
# Don't reoptimize more often than every 600 seconds

# LSP self-ping (verify data-plane before switching to new LSP during MBB)
set protocols mpls lsp-self-ping
# Sends a probe through the new LSP before cutting over
# Prevents traffic blackhole during MBB if data plane isn't ready
```

### Junos: Graceful Maintenance

```junos
# Graceful RSVP shutdown on an interface
set protocols rsvp interface ge-0/0/0.0 subscription 0
# Set bandwidth subscription to 0% — no new reservations

# Or use admin-group trick
set protocols mpls interface ge-0/0/0.0 admin-group maintenance
# Then ensure no LSP includes "maintenance" group
# All LSPs will reoptimize away
```

### Key Knobs

| Parameter | IOS-XE Default | Junos Default | Recommended | Why |
|-----------|---------------|---------------|-------------|-----|
| FRR type | Not enabled | Not enabled | Facility backup (node-protect) | Best scale + protection |
| Bypass tunnel creation | Manual | Auto (link-protection keyword) | Auto-bypass | Less operational overhead |
| Autoroute metric | IGP metric to tail-end | N/A (inet.3 resolution) | Relative -1 or absolute | Control tunnel preference |
| RSVP refresh reduction | Disabled | Disabled | Enable (aggregate + reliable in Junos) | Mandatory above 1K sessions |
| LSP self-ping | N/A | Disabled | Enable | Prevents MBB blackhole |
| Graceful shutdown | Manual metric change | subscription 0 | Use maintenance admin-group | Clean, reversible |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| FRR bypass not created | `link-protection` / `auto-tunnel backup` not configured on transit P routers | Enable on all core-facing interfaces |
| FRR shows "No protection" for a hop | No bypass path exists (topology doesn't allow it) | Add physical path diversity; check SRLG |
| LDP session down after tunnel up | Autoroute not configured; LDP can't resolve next-hop via tunnel | Add `autoroute announce` (IOS-XE) or verify inet.3 population (Junos) |
| Traffic blackhole during MBB | Data plane not ready when new LSP declared up | Enable `lsp-self-ping` (Junos); verify LFIB convergence |
| Tunnel not used despite autoroute | Tunnel metric higher than IGP metric to same destination | Set `autoroute metric relative -1` or `absolute` value |
| Routing loop with forwarding adjacency | FA tunnel advertised with metric lower than underlying links | Set FA metric carefully; avoid FA in complex topologies |
| FRR reroute takes >50ms | BFD not configured for fast failure detection | Enable BFD on all core interfaces (3x10ms or 3x50ms) |
| Post-failure utilization >80% | Failure modeling not done; backup paths overloaded | Re-run capacity model; add link capacity or adjust tunnel placement |

### Methodology

1. **LDPoRSVP verification**: Check label stack (3 labels: RSVP + LDP + VPN). If only 2 labels → LDP not tunneled.
2. **FRR verification**: Confirm bypass exists for every protected hop. `show mpls traffic-eng fast-reroute database` should show "Ready" for each.
3. **Failure simulation**: Shut an interface; measure convergence time and traffic loss. Target: <50ms with BFD + FRR.
4. **Capacity validation**: After failure sim, check all link utilizations. Any >80% = add capacity.

### War Story: The Full-Mesh That Ate the Control Plane

> A mid-size European SP decided to deploy a full mesh of RSVP-TE tunnels across their 150-PE network. Math: 150 × 149 = 22,350 tunnels. Each P router was transiting ~8,000 RSVP sessions. They enabled auto-bandwidth with 300-second intervals on all tunnels. Every 300 seconds, hundreds of tunnels would simultaneously re-signal via MBB, creating RSVP message storms that spiked CPU to 95% on their P routers (ASR 9006). RSVP hello timeouts triggered, declaring RSVP neighbors down, which triggered even MORE re-signaling. The cascading failure took down 40% of core traffic for 25 minutes. **Root causes**: (1) Full-mesh doesn't scale beyond ~100 PEs. (2) Auto-bandwidth intervals must be staggered, not synchronized. (3) RSVP refresh reduction wasn't enabled. **Fix**: Moved to hierarchical TE (PE→P tunnels only, ~450 total), enabled RSVP refresh reduction, staggered auto-bandwidth intervals with random jitter (Junos: `adjust-interval 86400 adjust-interval-random`), and added CPU protection (RSVP message-rate limiting). They've since migrated to SR-TE and don't miss RSVP at all.

## Design Decision Framework

### RSVP-TE vs SR-TE: When to Use What

| Criteria | RSVP-TE | SR-TE |
|----------|---------|-------|
| Brownfield with existing RSVP | Stay on RSVP, migrate gradually | Target state |
| Greenfield deployment | Don't. Use SR-TE. | Yes |
| Hard bandwidth admission control | RSVP (native) | External (PCE + controller) |
| >10K tunnels | Painful | Easy |
| Multi-vendor | Well standardized | Good (some interop gaps in PCEP/flex-algo) |
| Operational complexity tolerance | High (dedicated TE team) | Low |
| FRR mechanism | Facility bypass (good) | TI-LFA (better — 100% coverage) |

### TE Deployment Checklist

```
□ TE metric policy defined (latency-based for delay-sensitive, IGP-based for capacity)
□ Admin-group naming convention documented and consistent
□ SRLG assignments mapped from physical fiber/conduit surveys
□ RSVP refresh reduction enabled on all interfaces (if RSVP-TE)
□ Max-reservable bandwidth set to 75-80% of interface capacity
□ FRR enabled on all protected tunnels (node protection preferred)
□ Bypass tunnels verified for all transit hops
□ Auto-bandwidth configured with sane min/max and staggered intervals
□ BFD enabled on all core interfaces (3x10ms for <50ms detection)
□ Failure modeling completed for single-link, single-node, and SRLG scenarios
□ Graceful maintenance procedure documented and tested
□ PCEP sessions established and verified (if using PCE)
□ Monitoring: per-tunnel utilization, FRR activation events, reoptimization events
□ Capacity planning process scheduled (quarterly minimum)
```

## Scaling Considerations

| Parameter | Small SP (<50 PEs) | Medium SP (50-200 PEs) | Large SP (200+ PEs) |
|-----------|-------------------|----------------------|---------------------|
| Tunnel model | Full-mesh OK | Hierarchical recommended | Hierarchical mandatory or SR-TE |
| Tunnel count | <2,500 | 2,500-10,000 | >10,000 |
| TE technology | RSVP-TE or SR-TE | Either (migrate to SR-TE) | SR-TE strongly preferred |
| Path computation | Head-end CSPF | Head-end or PCE | PCE mandatory |
| FRR | Facility backup | Facility backup or TI-LFA | TI-LFA (SR-TE) |
| Auto-bandwidth | 1-24h intervals | 24h with overflow | 24h, PCE-managed |
| Capacity planning | Manual/spreadsheet | WAE/NorthStar | Custom tooling + WAE |

## Review Questions

1. **Design**: You have a 75-PE backbone. Calculate the tunnel count for full-mesh. Then design a hierarchical model using 8 core P routers and calculate the reduced tunnel count. What's the scaling improvement?

2. **Scenario**: After a fiber cut, FRR activates and traffic reroutes via bypass tunnels. You notice 200ms of packet loss before FRR kicks in. What's the most likely cause, and what do you add to reduce it to <50ms?

3. **Troubleshooting**: Your LDPoRSVP setup shows a 3-label stack (RSVP + LDP + VPN) on most prefixes, but some VPN prefixes only show a 2-label stack. What's different about those prefixes?

4. **Architecture**: Explain why forwarding adjacency can cause routing loops. Give a specific topology example where this occurs and how to prevent it.

5. **Migration**: You're migrating from RSVP-TE to SR-TE on a live 100-PE backbone. Describe a phased migration plan that maintains FRR protection at every stage. What's the coexistence model?
