# 5.2 — RSVP-TE Advanced

## Overview
> RSVP-TE is the battle-tested signaling protocol that has run the world's largest SP backbones for two decades. While Section 5.1 covered fundamentals, this section dives into the features that separate lab exercises from production deployments: auto-bandwidth that tracks demand, soft preemption that avoids traffic drops, inter-area/inter-AS TE for multi-domain networks, SRLG-aware path diversity, and the reoptimization mechanics that keep tunnels on optimal paths. If you can't design and troubleshoot these features, you can't run TE at scale.


## Key Concepts
> **After this section you will understand:**
> - how constrained paths are computed and signaled through the network
> - how bandwidth, affinity, and path constraints affect tunnel placement
> - tradeoffs between RSVP-TE and SR-based traffic engineering models
> - how to validate tunnel liveness, path intent, and failover behavior

## Theory

### Auto-Bandwidth

Static bandwidth reservations are a lie. Real traffic is bursty, seasonal, and unpredictable. Auto-bandwidth dynamically adjusts a tunnel's reserved bandwidth based on measured traffic:

```
Auto-Bandwidth Cycle:
1. Sample traffic rate on tunnel (every sample-interval, default 5 min)
2. At end of adjustment-interval (default 24h):
   a. Take maximum (or average, vendor-dependent) of samples
   b. If new BW differs from current by > adjustment-threshold:
      c. Re-signal tunnel with new bandwidth via make-before-break
3. Optional: min-bandwidth and max-bandwidth clamps
4. Optional: overflow/underflow for faster reaction
```

**Overflow detection**: If traffic exceeds current reservation by a threshold for N consecutive samples, trigger immediate re-signal without waiting for the full adjustment interval. Critical for traffic bursts.

**Underflow detection**: Opposite — if traffic drops significantly, reclaim bandwidth sooner.

**Make-before-break (MBB)**: Auto-bandwidth ALWAYS uses MBB to avoid traffic interruption. The new LSP is established before the old one is torn down. Both share the same Tunnel-ID but have different LSP-IDs.

### Soft Preemption

Traditional (hard) preemption tears down a lower-priority LSP immediately when a higher-priority LSP needs the bandwidth. Traffic drops. Soft preemption (RFC 5712) signals the head-end of the victim LSP to reroute before tearing it down:

```
Soft Preemption Flow:
1. Higher-priority LSP arrives at transit node, needs bandwidth
2. Transit node identifies lower-priority LSP to preempt
3. Instead of tearing down immediately:
   a. Mark victim LSP with "soft preempted" flag in RESV
   b. Send PathErr with "Reroute Request Soft" to victim head-end
4. Victim head-end has grace period (default 30s) to:
   a. Re-signal via make-before-break on alternate path
   b. If successful, old path torn down gracefully
5. If victim fails to reroute within timer → hard preemption occurs
```

**Key detail**: During soft preemption, the victim LSP continues forwarding traffic. The bandwidth is temporarily over-committed. This is acceptable because the alternative (hard preemption) drops traffic immediately.

### Inter-Area TE

RSVP-TE was originally intra-area only — CSPF can only compute paths within a single IGP area because the TED is area-scoped. Three approaches solve inter-area TE:

#### 1. Loose-Hop Expansion
```
Head-end specifies ABR as loose hop in ERO:
  ERO: [10.1.1.2(strict), ABR-1(loose), ABR-2(loose), 10.5.5.5(loose)]

Each ABR expands its loose hop using local CSPF:
  ABR-1 expands to strict hops within its area
  ABR-2 expands to strict hops within its area
```
- Simple but suboptimal — each ABR makes local decisions
- No end-to-end constraint satisfaction guarantee

#### 2. Per-Domain Path Computation (RFC 5152)
- Head-end computes to first ABR
- Each ABR computes to next ABR or tail-end
- Better visibility but still not globally optimal

#### 3. PCE-Based (RFC 5441 — BRPC)
- Backward Recursive PCE Computation
- PCE in each domain collaborates to find end-to-end optimal path
- Best path quality, most complex to deploy

### Inter-AS TE

Even harder than inter-area — crosses autonomous system boundaries:

| Option | Mechanism | Complexity | Path Quality |
|--------|-----------|------------|-------------|
| Contiguous LSP | Single RSVP session across AS boundaries | Requires inter-AS RSVP, trust | Optimal but operationally risky |
| Stitched LSP | Separate LSP per AS, stitched at ASBR | Moderate | Good, local optimization |
| Nested LSP | Hierarchical — outer LSP carries inner | Complex | Flexible encapsulation |

**Inter-AS TE is rare in practice.** Most SPs use BGP-based TE (communities, prepending) at AS boundaries and RSVP-TE within each AS. SR-TE with PCE (Section 5.3) is increasingly the answer for multi-domain TE.

### RSVP-TE Reoptimization

Tunnels can become suboptimal as network topology changes (links fail/recover, bandwidth freed). Reoptimization periodically re-runs CSPF to find a better path:

**Timer-based reoptimization:**
- Head-end runs CSPF every N seconds (default: 3600s IOS-XE, 300s Junos)
- If a better path exists, MBB to new path
- "Better" = lower metric OR same metric with more available bandwidth

**Event-driven reoptimization:**
- Triggered by topology change (link up/down, BW change)
- More responsive but more churn
- Dampening recommended to avoid oscillation

**Manual reoptimization:**
```
IOS-XE:  mpls traffic-eng reoptimize tunnel 100
Junos:   clear mpls lsp optimize name to-PE5
```

### SRLG (Shared Risk Link Group)

Links that share a common physical resource (fiber conduit, ROADM, building entry) share risk. If the conduit is cut, ALL links in the group fail simultaneously:

```
Example SRLG Assignment:
  Link PE1-P1 via Conduit-A: SRLG 100
  Link PE1-P2 via Conduit-A: SRLG 100  ← same conduit!
  Link PE2-P3 via Conduit-B: SRLG 200
  Link PE2-P4 via Conduit-B: SRLG 200
```

SRLG-aware path computation ensures primary and backup paths don't share any SRLG values. This is **critical** for FRR (Section 5.4) — a backup path through the same conduit is useless.

### Path Diversity

For critical services, you need two paths with zero shared resources:

1. **SRLG-diverse**: No shared risk groups
2. **Node-diverse**: No shared transit nodes
3. **Link-diverse**: No shared links (weakest — same node is OK)

Implementing diversity:
- **Exclude-based**: Compute primary, then exclude all primary's SRLGs/nodes/links and compute secondary
- **Simultaneous**: PCE computes both paths together (optimal but complex)
- **Cisco**: `tunnel mpls traffic-eng path-option 2 dynamic attributes exclude-list`
- **Junos**: `secondary` path with `standby` + `exclude` constraints

## Configuration

### IOS-XE: Auto-Bandwidth

```cisco
interface Tunnel100
 ip unnumbered Loopback0
 tunnel mode mpls traffic-eng
 tunnel destination 10.0.0.5
 tunnel mpls traffic-eng bandwidth 500000
 ! Initial bandwidth 500 Mbps
 tunnel mpls traffic-eng auto-bw
  frequency 86400
  ! Adjustment interval: 24 hours (seconds)
  min-bw 100000
  ! Never go below 100 Mbps
  max-bw 5000000
  ! Never exceed 5 Gbps
  adjustment-threshold 10 10
  ! Adjust if traffic changes by 10%, min 10% of current BW
  overflow-threshold 15 3
  ! If traffic exceeds reservation by 15% for 3 consecutive samples, adjust immediately
  underflow-threshold 20 5
  ! If traffic is 20% below reservation for 5 samples, adjust down
  collect-bw
  ! Start collecting samples (required to begin)
```

### IOS-XE: Soft Preemption

```cisco
! Global enable
mpls traffic-eng pce stateful-pcep
! Not required for soft preemption but commonly co-deployed

! On the tunnel
interface Tunnel100
 tunnel mpls traffic-eng priority 3 3
 tunnel mpls traffic-eng bandwidth 2000000

! Enable soft preemption globally
mpls traffic-eng soft-preemption

! Or per-tunnel
interface Tunnel100
 tunnel mpls traffic-eng soft-preemption
```

### IOS-XE: Inter-Area TE (Loose Hop)

```cisco
! Explicit path with loose hops at ABR boundaries
ip explicit-path name INTER-AREA-VIA-ABR1 enable
 next-address strict 10.1.1.2
 ! Strict hop to first P router in local area
 next-address loose 10.0.0.10
 ! Loose hop to ABR-1 (will be expanded by local CSPF)
 next-address loose 10.0.0.5
 ! Loose hop to destination (ABR or tail-end expands)

interface Tunnel200
 ip unnumbered Loopback0
 tunnel mode mpls traffic-eng
 tunnel destination 10.0.0.5
 tunnel mpls traffic-eng path-option 1 explicit name INTER-AREA-VIA-ABR1
```

### IOS-XE: SRLG Configuration

```cisco
! Define SRLG on interfaces
interface TenGigabitEthernet0/0/0
 mpls traffic-eng srlg 100
 mpls traffic-eng srlg 200
 ! This link is in SRLG 100 and 200

! On the tunnel — exclude specific SRLGs
interface Tunnel100
 tunnel mpls traffic-eng path-option 1 dynamic
 ! Primary — CSPF computes freely

interface Tunnel101
 tunnel mpls traffic-eng path-option 1 dynamic
 tunnel mpls traffic-eng path-selection srlg exclude preferred
 ! Prefer paths not sharing SRLGs with Tunnel100
 ! (requires path-protection or explicit SRLG exclude lists)
```

### Junos: Auto-Bandwidth

```junos
set protocols mpls label-switched-path to-PE5 to 10.0.0.5
set protocols mpls label-switched-path to-PE5 bandwidth 500m
set protocols mpls label-switched-path to-PE5 auto-bandwidth
set protocols mpls label-switched-path to-PE5 auto-bandwidth adjust-interval 86400
# 24-hour adjustment cycle
set protocols mpls label-switched-path to-PE5 auto-bandwidth minimum-bandwidth 100m
set protocols mpls label-switched-path to-PE5 auto-bandwidth maximum-bandwidth 5g
set protocols mpls label-switched-path to-PE5 auto-bandwidth adjust-threshold-percent 10
# Only adjust if delta > 10%
set protocols mpls label-switched-path to-PE5 auto-bandwidth overflow-limit 3
set protocols mpls label-switched-path to-PE5 auto-bandwidth overflow-threshold-percent 15
set protocols mpls label-switched-path to-PE5 auto-bandwidth underflow-limit 5
set protocols mpls label-switched-path to-PE5 auto-bandwidth underflow-threshold-percent 20
set protocols mpls label-switched-path to-PE5 auto-bandwidth monitor-bandwidth
```

### Junos: Soft Preemption

```junos
# Enable soft preemption on the LSP
set protocols mpls label-switched-path to-PE5 soft-preemption

# Set the cleanup timer (how long to wait before hard preemption)
set protocols mpls soft-preemption cleanup-timer 60
# 60 seconds grace period (default: 30s)
```

### Junos: SRLG Configuration

```junos
# Define SRLG values via fate-sharing (preferred Junos method):
set routing-options fate-sharing group CONDUIT-A members 10.1.1.0/30
set routing-options fate-sharing group CONDUIT-A members 10.1.2.0/30
set routing-options fate-sharing group CONDUIT-A cost 100

# LSP using SRLG-aware computation
set protocols mpls label-switched-path to-PE5 primary VIA-P2
set protocols mpls label-switched-path to-PE5 secondary VIA-P4 standby
# CSPF for secondary automatically excludes primary's SRLGs
# (Junos does this by default for secondary standby paths)
```

### Junos: Diverse Paths (Secondary Standby)

```junos
# Primary path
set protocols mpls path VIA-P2 10.1.1.2 strict
set protocols mpls path VIA-P2 10.0.0.5 strict

# Secondary standby — SRLG-diverse from primary
set protocols mpls label-switched-path to-PE5 primary VIA-P2
set protocols mpls label-switched-path to-PE5 secondary VIA-P4 standby
set protocols mpls label-switched-path to-PE5 secondary VIA-P4 exclude-srlg primary-path VIA-P2
# Explicitly exclude SRLGs of primary when computing secondary

# Or node-diverse:
set protocols mpls label-switched-path to-PE5 secondary VIA-P4 exclude-node 10.0.0.2
```

### Junos: Reoptimization

```junos
# Timer-based reoptimization
set protocols mpls optimize-timer 300
# Global: re-run CSPF every 300 seconds

# Per-LSP override
set protocols mpls label-switched-path to-PE5 optimize-timer 600

# Aggressive: optimize on topology change
set protocols mpls optimize-aggressive
# Re-run CSPF whenever TED changes (careful — can cause churn)
```

### Key Knobs

| Parameter | IOS-XE Default | Junos Default | Recommended | Why |
|-----------|---------------|---------------|-------------|-----|
| Auto-BW adjust interval | 86400s (24h) | 300s | 3600-86400s | 24h smooths out daily patterns |
| Auto-BW min-bw | 0 | 0 | Set to minimum SLA BW | Prevents tunnel from shrinking to nothing |
| Auto-BW max-bw | Link capacity | Link capacity | Set to realistic max | Prevents one tunnel consuming all BW |
| Soft preemption timer | 30s | 30s | 30-60s | Must be long enough for MBB to complete |
| Reoptimization timer | 3600s | 300s | 300-600s | Balance freshness vs stability |
| RSVP summary refresh | Disabled | Enabled | Enable always | Reduces refresh overhead by 10x |
| RSVP refresh reduction | Disabled | Enabled | Enable always | RFC 2961 — mandatory at scale |
| SRLG penalty | N/A | cost value | 10000+ | Must make SRLG-shared paths clearly worse |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Auto-BW stuck at initial value | `collect-bw` not configured (IOS-XE) or `monitor-bandwidth` missing (Junos) | Add the sampling enable command |
| Auto-BW oscillating wildly | Adjustment threshold too low, interval too short | Increase threshold to 15-20%, interval to 24h |
| Auto-BW spikes to max-bw | Burst traffic (DDoS, backup job) | Set max-bw cap; consider overflow dampening |
| Soft preemption fails — hard preempt occurs | Victim can't find alternate path within timer | Increase cleanup timer; ensure alternate capacity exists |
| Inter-area tunnel down | ABR can't expand loose hop (no TE info in next area) | Verify TE enabled in all areas; check ABR TED |
| SRLG-diverse path not found | All paths share an SRLG | Add physical diversity or relax SRLG constraint to "preferred" |
| Reoptimization churn | Timer too aggressive + topology instability | Dampen reopt: 600s minimum; use event-driven with hold-down |
| RSVP refresh storms at scale | Summary refresh not enabled | Enable `ip rsvp signalling refresh reduction` (IOS-XE) |
| MBB failure — double bandwidth reservation | Both old and new LSP allocated simultaneously | Ensure `shared-explicit` style (default); check BW pool has room for transient double-booking |

### Methodology

1. **Auto-bandwidth issues**: Check sampling state → verify counter collection → confirm adjustment interval timing → review threshold math
2. **Preemption issues**: Identify priority mismatch → check if soft preemption is configured on both victim and transit → verify alternate path exists → check timer expiry
3. **Inter-area issues**: Verify loose hop syntax → check ABR TE capability → trace RSVP PathErr messages → test each domain independently
4. **SRLG issues**: Dump full SRLG table → verify SRLG flooding in IS-IS/OSPF → check if SRLG constraint is "mandatory" or "preferred"

### War Story: The Auto-Bandwidth DDoS Amplifier

> A Tier-1 SP had auto-bandwidth on all core TE tunnels with 300-second adjust intervals, no max-bandwidth cap, and 5% overflow threshold. A volumetric DDoS attack hit a customer PE, sending 200 Gbps toward a single destination. Auto-bandwidth on the ingress tunnel ramped from 10 Gbps to the max link capacity in two overflow cycles (under 10 minutes). This starved all OTHER tunnels sharing those links, triggering a cascade of soft preemptions and re-routes across the core. The re-routes moved traffic to links that were ALSO being expanded by other auto-bandwidth tunnels chasing the DDoS traffic. Within 20 minutes, the entire core was in a reoptimization storm. **Fix**: (1) Always set `max-bw` to a sane value per tunnel class. (2) Overflow threshold should be 15%+ with count ≥ 3. (3) Auto-bandwidth adjust interval should be 1-24 hours, NOT 5 minutes. (4) DDoS mitigation (RTBH/Flowspec) must act faster than auto-bandwidth adjusts.

## Scaling Considerations

| Parameter | Guideline | Notes |
|-----------|----------|-------|
| Auto-bandwidth tunnels | Same as total tunnel limits | Every auto-BW tunnel adds polling overhead |
| Adjustment interval | 3600-86400s in production | Shorter intervals = more MBB events |
| RSVP refresh reduction | Mandatory above 1,000 sessions | Bundles refresh messages, 10x reduction |
| RSVP hello | 3s/9s dead for sub-10K sessions | Consider 9s/27s for >10K sessions |
| Summary refresh bundle size | 20-50 messages per bundle | Platform-dependent maximum |
| Soft preemption concurrency | Limited by MBB rate | 50-100 concurrent MBBs is practical max |
