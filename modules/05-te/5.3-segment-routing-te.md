# 5.3 — Segment Routing Traffic Engineering (SR-TE)

## Overview
> SR-TE is the modern replacement for RSVP-TE — achieving the same traffic steering goals without per-hop signaling state. Instead of maintaining thousands of RSVP sessions across the network, SR-TE encodes the path as a stack of segment IDs (SIDs) at the head-end. No state on transit nodes. No refresh messages. No soft-state timers. The control plane complexity moves from the network to the head-end (or a centralized PCE). This is the single most important architectural shift in SP traffic engineering since MPLS itself.


## Key Concepts
> **After this section you will understand:**
> - how constrained paths are computed and signaled through the network
> - how bandwidth, affinity, and path constraints affect tunnel placement
> - tradeoffs between RSVP-TE and SR-based traffic engineering models
> - how to validate tunnel liveness, path intent, and failover behavior

## Theory

### SR-TE vs RSVP-TE — Fundamental Comparison

| Property | RSVP-TE | SR-TE |
|----------|---------|-------|
| Per-hop state | Yes (PATH/RESV state at every transit) | No (stateless transit) |
| Signaling protocol | RSVP with TE extensions | None (or PCEP for centralized) |
| Path encoding | ERO in RSVP PATH message | Label stack at ingress |
| Bandwidth reservation | Native (RSVP FLOWSPEC) | Not native (needs external controller) |
| Scalability ceiling | ~10K-50K sessions per router | Limited only by label stack depth |
| FRR mechanism | Facility backup / one-to-one backup | TI-LFA (topology-independent) |
| Make-before-break | RSVP SE style | Inherent (just change label stack) |
| Deployment complexity | High (RSVP on every node) | Low (IGP extensions + head-end policy) |
| Multi-domain | Hard (inter-AS RSVP) | Easier (binding SID, PCE) |
| Bandwidth guarantees | Strong (admission control) | Weak (needs external measurement) |

**When RSVP-TE still wins**: Hard bandwidth guarantees with admission control. If you need the network to reject tunnel setup when bandwidth is unavailable, RSVP-TE is still the answer. SR-TE relies on external controllers for bandwidth awareness.

### SR-TE Policies (RFC 9256)

An SR-TE policy is the fundamental construct — it defines a path from head-end to endpoint using segment lists:

```
SR-TE Policy Structure:
  Policy:
    Head-end: 10.0.0.1 (PE1)
    Endpoint: 10.0.0.5 (PE5)
    Color: 100 (identifies the intent — e.g., low-latency)
    
    Candidate Paths (ordered by preference):
      Preference 200 (highest wins):
        Segment List 1: [16002, 16003, 16005]  ← via P2→P3→PE5
        Segment List 2: [16004, 16005]          ← via P4→PE5 (ECMP)
        Weight: 1, 1  (equal load-balancing between segment lists)
      
      Preference 100 (fallback):
        Segment List 1: [16006, 16005]  ← backup via P6→PE5
```

Key concepts:
- **Color**: 32-bit value that identifies the policy's intent (latency, gold, avoid-submarine, etc.)
- **Endpoint**: Tail-end router (identified by its prefix-SID address)
- **Candidate path**: A possible path, with preference for selection
- **Segment list**: Ordered list of SIDs encoding the explicit path
- **Binding SID (BSID)**: A local label allocated to the policy — any traffic hitting BSID follows the policy

### Segment Types in SR-TE

| Type | SID | Description | Use Case |
|------|-----|-------------|----------|
| Type A | Prefix-SID (node) | Routes to a specific node via IGP shortest path | Loose source routing — "go via node X" |
| Type B | Adjacency-SID | Uses a specific link | Strict source routing — "use this exact link" |
| Type C | Prefix-SID with algo | Routes via specific flex-algo topology | "Go to node X via low-latency topology" |
| Type F | BSP (Binding SID) | Steers into another SR policy | Policy chaining across domains |
| Type H | Prefix-SID of BGP peer | Inter-domain steering | BGP PeerNode SID |

### Binding SID (BSID)

The Binding SID is SR-TE's most powerful abstraction. It decouples the policy consumer from the policy implementation:

```
Example: Multi-Domain Path Stitching via BSID

Domain 1 (AS 65001):                Domain 2 (AS 65002):
  PE1 → P2 → ASBR1  ────────────  ASBR2 → P5 → PE6

Policy on PE1:
  Endpoint: PE6
  Segment List: [16002, 24001]
    16002 = Prefix-SID of ASBR1 (traverse Domain 1)
    24001 = BSID on ASBR1 (points to SR policy: ASBR1→PE6 through Domain 2)

ASBR1's SR Policy:
  BSID: 24001 (locally allocated)
  Segment List: [16502, 16506]
    16502 = Prefix-SID of P5 in Domain 2
    16506 = Prefix-SID of PE6 in Domain 2
```

The BSID acts as a "tunnel endpoint" without any tunnel state. PE1 doesn't need to know Domain 2's topology. ASBR1's local policy handles it.

### On-Demand Next-Hop (ODN)

ODN automatically instantiates SR-TE policies when a BGP next-hop is colored:

```
ODN Flow:
1. PE1 receives BGP route with color extended community:
   Prefix: 192.168.1.0/24
   Next-hop: 10.0.0.5
   Color: 100 (low-latency)

2. PE1 checks: is there an SR policy with (endpoint=10.0.0.5, color=100)?
   
3. If no → ODN template kicks in:
   a. Use ODN template for color 100
   b. Template says: "compute path using flex-algo 128" (or explicit constraints)
   c. Head-end CSPF (or PCE request) computes path
   d. SR policy auto-created with computed segment list
   
4. BGP route installed with SR policy as next-hop
```

ODN eliminates static SR-TE policy configuration. As new BGP destinations appear with color communities, policies are created automatically. This is how modern L3VPN-over-SR-TE works at scale.

### PCE / PCEP (RFC 5440, 8231, 8281)

The Path Computation Element separates brain from brawn:

```
PCE Architecture:
                    ┌─────────┐
                    │   PCE   │  (Cisco XTC, Juniper NorthStar, ODL)
                    │ Server  │
                    └────┬────┘
                    PCEP │ sessions
            ┌────────┬───┴───┬────────┐
           PCC      PCC    PCC      PCC
          (PE1)    (PE2)  (PE3)    (PE4)
```

PCE evolution:
1. **Stateless PCE** (RFC 5440): PCC asks, PCE computes, PCC signals. PCE forgets.
2. **Stateful PCE** (RFC 8231): PCE maintains LSP-DB synchronized from all PCCs. Sees all tunnels.
3. **PCE-Initiated** (RFC 8281): PCE can CREATE and DELETE LSPs on PCCs without being asked. Full control.

**Stateful + PCE-Initiated** is the production model. The PCE has global visibility of all SR policies, all bandwidth usage, and can optimize across the entire network.

PCEP message types:
- **PCReq / PCRep**: Path computation request/reply
- **PCRpt**: PCC reports LSP state to PCE
- **PCUpd**: PCE updates LSP on PCC
- **PCInitiate**: PCE creates/deletes LSP on PCC

### Flex-Algo (IGP Flexible Algorithm)

Flex-Algo creates multiple virtual topologies within a single IGP instance, each optimized for a different constraint:

```
Flex-Algo Definitions:
  Algorithm 128: Low-Latency
    Metric: min-unidirectional-delay
    Exclude: admin-group "submarine"
    
  Algorithm 129: High-Bandwidth  
    Metric: TE metric
    Include-any: admin-group "400G-links"
    
  Algorithm 130: Avoid-Country-X
    Metric: IGP metric
    Exclude: admin-group "country-x-links"
```

Each algorithm computes its own SPF tree using its own metric and constraints. Nodes advertise participation (FAD — Flex-Algo Definition) and compute prefix-SIDs per algorithm:

```
Node PE5:
  Algo 0 (default): Prefix-SID 16005 → SPF via IGP metric
  Algo 128: Prefix-SID 16128005 → SPF via delay metric, excluding submarine
  Algo 129: Prefix-SID 16129005 → SPF via TE metric, 400G only
```

An SR-TE policy just uses the right algorithm's prefix-SID:
- Low-latency to PE5: push label 16128005
- No CSPF needed — the IGP already computed the constrained shortest path

## Configuration

### IOS-XE: SR-TE Policy (Static)

```cisco
! Define segment-routing traffic-eng
segment-routing traffic-eng
 !
 ! SR-TE policy
 policy TO-PE5-LOW-LATENCY
  color 100 end-point 10.0.0.5
  !
  candidate-paths
   preference 200
    explicit segment-list VIA-P2-P3
   !
   preference 100
    explicit segment-list VIA-P4
   !
  !
 !
 ! Segment lists
 segment-list VIA-P2-P3
  index 10 mpls label 16002
  index 20 mpls label 16003
  index 30 mpls label 16005
 !
 segment-list VIA-P4
  index 10 mpls label 16004
  index 20 mpls label 16005
 !
```

```cisco

  isis network point-to-point
  ip router isis CORE
  segment-routing mpls

  !
 !
```

```cisco
  color 100 end-point 10.0.0.5
  binding-sid mpls 24100
  ! Allocate BSID 24100 — any traffic with this label follows this policy
```


### Key Knobs

| Parameter | IOS-XE Default | Junos Default | Recommended | Why |
|-----------|---------------|---------------|-------------|-----|
| PCEP session keepalive | 30s | 30s | 30s | Balance detection vs overhead |
| PCEP dead-timer | 120s | 120s | 120s (4x keepalive) | Standard ratio |
| Flex-algo range | 128-255 | 128-255 | Start at 128, sequential | Consistent across network |
| BSID allocation | Dynamic | Dynamic | Static for critical policies | Stability for inter-domain |

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| SR policy operationally down | SID in segment list not reachable (node down or SID not advertised) | Verify each SID exists: `show isis segment-routing prefix-sid-map` |
| BSID not in forwarding table | Policy not active or BSID conflict | Check policy status; ensure no duplicate BSID allocation |
| ODN policy not created | Color community not propagated in BGP or ODN template missing | Verify `send-community both` on BGP session; check ODN template config |
| PCEP session down | Firewall blocking TCP 4189, source address mismatch | Check ACLs; verify source address matches PCE peer config |
| Label stack too deep — packets dropped | Transit node can't process deep stacks (hardware limit) | Reduce hops in segment list; use BSIDs to compress; check MTU |
| Flex-algo path not computed | Not all nodes in path participate in the flex-algo | All transit nodes must advertise FAD for the algorithm |
| Traffic not following SR policy | BGP next-hop not resolving to SR policy (color mismatch) | Verify color community matches policy color; check `resolve-color` is enabled |
| PCE-initiated LSP not installed | PCC not configured for `lsp-provisioning` | Enable PCE-initiated on PCC: `pce lsp-provisioning` |

### Methodology

1. **Policy state**: Is the policy admin-up AND oper-up?
2. **Segment validation**: Can each SID in the segment list be resolved?
3. **BSID programming**: Is the BSID installed in the LFIB?
4. **Traffic steering**: Is the BGP/IGP route pointing to the SR policy?
5. **End-to-end data plane**: Traceroute with SR labels to verify actual path
6. **PCE health**: PCEP session up? LSP-DB synchronized?

### War Story: The Label Stack That Wouldn't Fit

> A large APAC carrier deployed SR-TE with PCE computing optimal low-latency paths across their continental backbone. PCE was computing paths with 8-10 node SIDs (prefix-SIDs) for the longest paths. Worked perfectly in the lab. In production, ~15% of traffic was being dropped on specific linecards that could only handle a 6-deep label stack (older generation silicon with limited TCAM depth for MPLS lookups). No alarms — the packets just got silently dropped at the transit node that couldn't parse beyond label 6. **Fix**: (1) Set max-sid-depth (MSD) advertisement in IS-IS so PCE knows each node's limit. (2) Use binding SIDs at intermediate points to compress the stack. (3) Replaced 4 linecards with newer silicon supporting 12-deep stacks. **Lesson**: Always verify MSD across your fleet before deploying SR-TE. It's not negotiable.

## Scaling Considerations

| Parameter | Practical Limit | Notes |
|-----------|----------------|-------|
| SR-TE policies per head-end | 10,000-50,000+ | No transit state — only head-end matters |
| Label stack depth (MSD) | 6-16 (hardware dependent) | Jericho2: 12, Memory: 6, Memory+: 10 |
| Flex-algo instances | 128 (algorithms 128-255) | Practical: 3-5 in production |
| PCE managed LSPs | 100,000+ (Cisco XTC) | PCE server sizing is the bottleneck |
| PCEP sessions per PCE | 5,000+ | Depends on PCE platform |
| Colors in use | Unlimited (32-bit) | Practical: 5-20 distinct intents |
| Segment lists per policy | 16-64 | For weighted ECMP |
