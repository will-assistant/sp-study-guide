# 2.1 — IS-IS Deep Dive

## Overview
> IS-IS is the IGP of choice for the vast majority of service provider backbones worldwide. Unlike OSPF, it runs directly over Layer 2 (no IP dependency), scales elegantly with multi-level hierarchy, and carries new address families via TLV extensibility without protocol rewrites. If you work in SP, you work in IS-IS. Period.


## Key Concepts
> **After this section you will understand:**
> - how SPF-based IGPs build topology and select best paths
> - which timers and fast-fail mechanisms control convergence behavior
> - how to validate IGP health with adjacency, LSDB, and route checks
> - common failure modes and how to isolate them quickly

## Theory

### Why IS-IS Over OSPF in SP?
1. **Protocol-independent by design** — Runs on CLNS (Layer 2), not IP. Adding IPv6, SR, or any new AF is just a new TLV, not a new protocol version.
2. **Simpler area design** — Two levels vs OSPF's stub/NSSA/totally-stubby madness. L1/L2 is clean.
3. **TLV extensibility** — The killer feature. IS-IS has been extended for TE, SR, IPv6, and more without breaking backward compatibility. OSPF needed OSPFv3 for IPv6.
4. **Stability at scale** — SPF calculation is bounded per level. Flooding is naturally contained.
5. **Industry inertia** — Every major SP runs IS-IS. Operational tooling, tribal knowledge, and vendor optimization all favor it.

### IS-IS Fundamentals

#### Addressing: NET (Network Entity Title)
```
49.0001.1921.6800.1001.00
│    │    │              │
│    │    └─ System ID   └─ SEL (always 00 for router)
│    └─ Area ID
└─ AFI (49 = private)
```
- **AFI 49**: Private addressing (most SP deployments)
- **Area ID**: Variable length (1-13 bytes). Keep it simple — `49.XXXX`
- **System ID**: 6 bytes, must be unique. Common convention: encode loopback IP
- **SEL**: Always `00` for a router (NET = NSAP with SEL=00)

#### Levels and Hierarchy
| Level | Scope | Analogy |
|-------|-------|---------|
| L1 | Intra-area | OSPF non-backbone area |
| L2 | Inter-area (backbone) | OSPF Area 0 |
| L1/L2 | Both (border router) | OSPF ABR |

**Key difference from OSPF**: There's no "Area 0" requirement. L2 forms the backbone by connecting L1/L2 routers. Any L1/L2 router is automatically a backbone member.

**SP reality**: Most SP cores run **L2-only** everywhere. L1/L2 hierarchy is used for large access/aggregation layers. Don't over-engineer levels where flat L2 works.

#### PDU Types
| PDU | Function | Direction |
|-----|----------|-----------|
| IIH (Hello) | Neighbor discovery, adjacency maintenance | P2P or broadcast |
| LSP | Link-state information (the database) | Flooded |
| CSNP | Complete sequence number PDU — DB sync | Periodic (broadcast) or on-demand |
| PSNP | Partial sequence number PDU — request/ack | As needed |

#### Adjacency Formation
1. Router sends IIH on interface
2. Neighbor receives, checks: area match (L1), authentication, MTU
3. Three-way handshake (RFC 5303): `Down → Init → Up`
4. Once Up, exchange LSPs via CSNP/PSNP

**P2P vs Broadcast**:
- P2P: No DIS election, direct LSP exchange. **Use this for all SP links** (even Ethernet — set `circuit-type point-to-point`).
- Broadcast: DIS elected (highest priority, then highest SNPA). DIS is NOT a DR — no backup, preemption happens. Generates pseudonode LSP.

### TLV Architecture

This is where IS-IS shines. Everything is a Type-Length-Value:

| TLV | Type | Purpose |
|-----|------|---------|
| 1 | Area Addresses | Area membership |
| 2 | IS Neighbors (old) | L1/L2 adjacencies |
| 22 | Extended IS Reachability | TE-capable neighbor info |
| 135 | Extended IP Reachability | IPv4 prefixes with TE metric |
| 236 | IPv6 Reachability | IPv6 prefixes |
| 229 | Multi-Topology | MT support |
| 242 | Router Capability | SR, flex-algo, etc. |

**Why this matters**: When Segment Routing was added, it was just new sub-TLVs under existing TLVs. No protocol version change. No new packet types. OSPF needed extensions across both v2 and v3.

### SPF Optimization

#### BFD (Bidirectional Forwarding Detection)
- Sub-second failure detection (50-300ms typical)
- Runs independently of IS-IS timers
- **Always deploy BFD on SP core links**

#### Timer Tuning
| Timer | Default | Aggressive | Notes |
|-------|---------|-----------|-------|
| Hello interval | 10s | 1-3s | Balance with BFD |
| Hello multiplier | 3 | 3 | Don't lower — false positives |
| SPF initial wait | 50ms | 50ms | Already fast |
| SPF secondary wait | 200ms | 200ms | Exponential backoff |
| SPF max wait | 5000ms | 5000ms | Cap the backoff |
| LSP gen interval | 5ms | 5ms | — |
| LSP refresh | 900s | 900s | Don't touch |
| LSP lifetime | 1200s | 1200s | Don't touch |

**RFC 8405** defines SPF back-off: initial wait → secondary wait → exponential backoff → max wait. This prevents SPF storms during convergence events.

#### Prefix Prioritization
- Critical prefixes (loopbacks, next-hops) get SPF'd first
- Ensures BGP next-hops converge before less-critical routes
- IOS-XR: `spf prefix-priority` under `router isis`
- Junos: `spf-options priority` under `protocols isis`

### Overload Bit
- Sets the OL bit in LSP header
- Tells other routers: "Don't transit me, but my directly-connected prefixes are still reachable"
- **Use cases**:
  - Maintenance windows (graceful drain)
  - Boot-up delay (wait for BGP to converge before attracting transit)
  - Memory/CPU protection

```
IOS-XR: set-overload-bit on-startup wait-for-bgp
Junos:  overload timeout 300
```

### Multi-Topology (MT)
- Run separate SPF computations for IPv4 and IPv6
- Allows ships-in-the-night routing — different topologies per AF
- TLV 229 carries MT information
- **When to use**: Dual-stack migrations where IPv4 and IPv6 topologies differ

## Configuration

### IOS-XR
```cisco
router isis CORE
 is-type level-2-only
 net 49.0001.0010.0000.0001.00
 address-family ipv4 unicast
  metric-style wide
  segment-routing mpls
  spf prefix-priority critical tag 1000
 !
 address-family ipv6 unicast
  metric-style wide
 !
 interface Loopback0
  passive
  address-family ipv4 unicast
   prefix-sid index 1
  !
 !
 interface GigabitEthernet0/0/0/0
  circuit-type level-2-only
  point-to-point
  hello-padding disable
  address-family ipv4 unicast
   metric 10
   bfd minimum-interval 100
   bfd multiplier 3
   bfd fast-detect
  !
 !
```

### Junos
```junos
protocols {
    isis {
        level 1 disable;
        interface ge-0/0/0.0 {
            level 2 metric 10;
            point-to-point;
            bfd-liveness-detection {
                minimum-interval 100;
                multiplier 3;
            }
        }
        interface lo0.0 {
            passive;
        }
        source-packet-routing {
            node-segment ipv4-index 1;
            srgb start-label 16000 index-range 8000;
        }
    }
}
routing-options {
    router-id 10.0.0.1;
}
```

### Key Knobs
| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| metric-style | narrow (old) | wide | Required for TE, SR. Non-negotiable. |
| circuit-type | L1/L2 | L2-only (core) | Reduce unnecessary L1 adjacencies |
| point-to-point | broadcast | P2P on all links | No DIS, faster convergence |
| hello-padding | enabled | disable (after adj up) | Save bandwidth on slow links |
| overload on-startup | disabled | wait-for-bgp | Prevent blackholing during boot |

| `show isis interface` | `show isis interface` | Per-interface IS-IS state |

### What Good Looks Like
- All adjacencies `Up` with expected hold time
- LSDB size consistent across all routers (same number of LSPs per level)
- SPF run count stable (not continuously incrementing)
- No `OL` (overload) bit set unexpectedly
- Sequence numbers incrementing normally (not wrapping)

## Troubleshooting

### Common Issues
| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Adjacency stuck in Init | MTU mismatch, one-way connectivity | Check `show isis interface` for MTU, verify L2 |
| Adjacency flapping | Unstable link, timer mismatch | Check BFD, interface errors, hello/hold timers |
| Missing routes | Area mismatch, metric overflow | Verify NET area, ensure wide metrics |
| SPF storms | Flapping link, LSP corruption | Check `show isis statistics`, find the churning LSP |
| LSP purge storms | System ID conflict | Every router MUST have unique system ID |

### Methodology
1. **Check adjacencies** — `show isis neighbors`. Any down?
2. **Check LSDB** — `show isis database`. Missing LSPs? Sequence number anomalies?
3. **Check SPF** — `show isis statistics`. How many runs? Recent triggers?
4. **Check the interface** — `show isis interface`. Correct level, metric, network type?
5. **Check the wire** — BFD state, interface counters, L2 connectivity
6. **Packet capture** — Wireshark filter: `isis`. Look at IIH, LSP, CSNP/PSNP

### War Stories
> **The Phantom DIS**: A large carrier had intermittent traffic blackholes on a metro Ethernet ring. Root cause: one link was configured as broadcast (default) instead of P2P. The DIS election was creating a pseudonode, and when the DIS flapped, all routers on the segment had to reconverge through the pseudonode LSP — adding 2-3 seconds of convergence time. Fix: `point-to-point` on every inter-router link. Always. Forever.
