# 2.4 — IS-IS vs OSPF: The Real-World Decision Framework

## Overview
> Every network engineer eventually faces the question: IS-IS or OSPF? In service provider networks, the answer is almost always IS-IS — but "almost always" isn't "always," and understanding *why* matters more than memorizing the answer. This section dismantles the myths, compares the protocols head-to-head on the metrics that actually matter at scale, and gives you a decision framework grounded in operational reality, not vendor marketing.


## Key Concepts
> **After this section you will understand:**
> - how SPF-based IGPs build topology and select best paths
> - which timers and fast-fail mechanisms control convergence behavior
> - how to validate IGP health with adjacency, LSDB, and route checks
> - common failure modes and how to isolate them quickly

## Theory

### The Myth vs Reality Check

| Claim | Reality |
|-------|---------|
| "IS-IS scales better than OSPF" | Mostly true at extreme scale (10K+ nodes), but OSPF handles 500+ routers fine with proper area design |
| "OSPF is easier to learn" | Only because most engineers encounter it first — IS-IS is arguably simpler once you drop the familiarity bias |
| "IS-IS is more stable" | True historically — IS-IS's TLV model means extensions don't break existing implementations |
| "OSPF has faster convergence" | False — convergence is a function of timers, BFD, and FRR, not the protocol itself |
| "IS-IS can't do NBMA" | True, but irrelevant — who's running Frame Relay in 2026? |
| "You need IS-IS for Segment Routing" | False — SR works with both. But IS-IS TLV extensions are cleaner |

### Head-to-Head Comparison

#### Protocol Fundamentals

| Feature | IS-IS | OSPF |
|---------|-------|------|
| **Runs on** | CLNS (L2) | IP (L3) |
| **Encapsulation** | Direct L2 frames (ethertype 0xFE) | IP protocol 89 |
| **Extensibility** | TLV-based (add new TLVs freely) | Fixed LSA types (new types = new code) |
| **Address family support** | Single topology or multi-topology, any AF via TLVs | OSPFv2 = IPv4 only; OSPFv3 = IPv6 (AF extensions exist but clunky) |
| **Area hierarchy** | 2 levels (L1 intra-area, L2 backbone) | Area 0 backbone + numbered areas + stub variants |
| **Flooding scope** | Per-level | Per-area + inter-area via ABR |
| **Authentication** | Per-level, per-interface | Per-area, per-interface, per-virtual-link |
| **MTU sensitivity** | Not MTU-dependent (no IP fragmentation) | MTU must match on adjacency (classic gotcha) |

#### Scalability Characteristics

| Metric | IS-IS | OSPF |
|--------|-------|------|
| **Max practical nodes (single area)** | 1,000+ | 300-500 (varies by vendor) |
| **Max practical areas** | 100+ | 50-100 |
| **LSDB size scaling** | Linear with nodes per level | Linear with nodes per area, but ABR summarization is more complex |
| **SPF computation** | Per-level (naturally bounded) | Per-area (comparable) |
| **Flooding storms** | Rare (TLV model, no type-5 equivalent explosion) | External LSA (Type-5) flooding can be problematic at scale |
| **Partial route computation (PRC)** | Native — leaf changes don't trigger full SPF | Possible but less standardized |

#### Convergence Comparison

| Component | IS-IS | OSPF |
|-----------|-------|------|
| **Detection** | BFD (identical for both) | BFD (identical for both) |
| **SPF delay defaults** | IOS-XR: 50ms initial, 200ms secondary, 5s max | IOS-XR: 50ms/200ms/5s (identical) |
| **LSP/LSA generation** | Configurable, same order of magnitude | Configurable, same order of magnitude |
| **Fast-reroute (LFA)** | Fully supported | Fully supported |
| **TI-LFA** | Fully supported | Fully supported |
| **Microloop avoidance** | Supported | Supported |

**Bottom line**: Convergence is protocol-agnostic at this point. The delta between IS-IS and OSPF convergence in a modern SP deployment is noise.

#### Segment Routing Support

| Feature | IS-IS | OSPF |
|---------|-------|------|
| **SR-MPLS (prefix SID, adj SID)** | Mature, widely deployed | Mature, widely deployed |
| **SRv6** | Native TLV support (RFC 9352) | OSPFv3 extensions exist but less mature |
| **Flex-Algo** | Full support | Full support |
| **SRv6 + multi-topology** | Clean integration | More complex |
| **Vendor support** | Universal tier-1 vendor support | Universal but SR implementations tested more with IS-IS |

#### Operational Characteristics

| Aspect | IS-IS | OSPF |
|--------|-------|------|
| **Troubleshooting complexity** | Moderate — NET addressing confuses newcomers, but protocol is predictable | Moderate — more familiar to most, but area/LSA interactions can surprise |
| **Configuration verbosity** | Minimal — one instance, add interfaces | Moderate — areas, stub config, network statements or interface-mode |
| **Graceful restart** | Supported | Supported |
| **NSF/NSR** | Supported on all major platforms | Supported on all major platforms |
| **Vendor consistency** | High — implementation is similar across IOS-XR, Junos, SR-OS | Moderate — NSSA behavior, type-7 to type-5 translation, forwarding address vary |
| **Monitoring/telemetry** | YANG models available | YANG models available |

### The Decision Framework

Use this flowchart when someone asks "which IGP?":

```
START
  │
  ├─ Greenfield SP/ISP backbone? ──────────────────────► IS-IS
  │
  ├─ Existing OSPF network, working fine, < 200 nodes?
  │   └─ No plans for SRv6? ──────────────────────────► Stay OSPF
  │   └─ SRv6 planned? ───────────────────────────────► Migrate to IS-IS (plan it)
  │
  ├─ Multi-vendor SP backbone? ────────────────────────► IS-IS
  │
  ├─ Enterprise campus/DC? ───────────────────────────► OSPF (or eBGP fabric)
  │
  ├─ Need multi-topology routing? ─────────────────────► IS-IS (native MT)
  │
  ├─ Team only knows OSPF? ───────────────────────────► IS-IS anyway.
  │   (seriously — it's not that hard, and they'll      Train them.
  │    need it eventually)
  │
  └─ NBMA legacy environment? ────────────────────────► OSPF (IS-IS can't do NBMA)
      (but also... why?)
```

### When OSPF Actually Wins

Let's be fair. OSPF is the right call in these specific scenarios:

1. **Enterprise campus**: Familiarity matters, OSPF tooling is ubiquitous, scale requirements are modest
2. **NBMA environments**: Frame Relay, DMVPN — IS-IS doesn't speak NBMA (though DMVPN can use other solutions)
3. **Small SP with no SR plans**: If you're a regional ISP with 30 routers and no Segment Routing roadmap, OSPF is fine
4. **Regulatory/compliance lock**: Some environments mandate specific protocols — don't fight the auditors
5. **Brownfield migration cost**: If migrating from OSPF to IS-IS would take 18 months of change windows and you're not gaining anything tangible... don't

### When IS-IS Always Wins

1. **Greenfield SP/ISP**: No debate. IS-IS. Period.
2. **SRv6 deployment**: IS-IS TLV extensions for SRv6 are cleaner, more tested, more vendor-consistent
3. **Scale > 500 nodes**: IS-IS's simpler hierarchy and PRC reduce operational headaches
4. **Multi-topology requirements**: IS-IS MT is native and clean
5. **Multi-vendor backbone**: IS-IS interop is more predictable across IOS-XR, Junos, SR-OS, EOS
6. **Long-term extensibility**: Every new SP technology gets IS-IS TLVs first (or simultaneously). Future-proofing matters.

### Migration Considerations: OSPF → IS-IS

If you decide to migrate, here's the playbook:

#### Strategy: Ships in the Night

Run both protocols simultaneously, migrate traffic gradually:

1. **Phase 1**: Enable IS-IS on all nodes alongside OSPF. IS-IS at higher AD/preference so it doesn't take over yet.
2. **Phase 2**: Validate IS-IS topology, adjacencies, and LSDB completeness.
3. **Phase 3**: Lower IS-IS AD below OSPF. Traffic shifts to IS-IS. OSPF still running as safety net.
4. **Phase 4**: Remove OSPF after burn-in period.

##### IOS-XR: Ships in the Night

```cisco
! Phase 1: IS-IS alongside OSPF, higher distance
router isis CORE
 address-family ipv4 unicast
  ! Default distance is 115, OSPF is 110 — IS-IS already loses
  ! No change needed for Phase 1
 !
 interface Loopback0
  passive
  address-family ipv4 unicast
 !
 interface GigabitEthernet0/0/0/0
  point-to-point
  address-family ipv4 unicast

! Phase 3: Make IS-IS preferred
router isis CORE
 address-family ipv4 unicast
  distance 105
```

##### Junos: Ships in the Night

```junos
# Phase 1: IS-IS alongside OSPF
protocols {
    isis {
        interface ge-0/0/0.0 {
            point-to-point;
        }
        interface lo0.0 {
            passive;
        }
        # Default preference 18 (L1) / 15 (L2) — already preferred over OSPF (10/150)
        # Wait — Junos IS-IS L2 preference (18) is HIGHER (worse) than OSPF internal (10)
        # So in Junos, IS-IS already loses to OSPF by default for internal routes!
        # Phase 1 works naturally in Junos
    }
}

# Phase 3: Make IS-IS preferred
protocols {
    isis {
        preference 9;  # Lower than OSPF's 10
    }
}
```

> ⚠️ **Junos preference gotcha**: Lower number = more preferred (opposite of Cisco's AD). Junos OSPF internal = 10, IS-IS L2 = 18. So IS-IS loses by default in Junos — which is actually convenient for ships-in-the-night migrations.

#### Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| Routing loops during migration | Never have asymmetric protocol selection — all nodes in a segment should prefer the same IGP |
| Incomplete IS-IS adjacencies | Validate before Phase 3: all expected adjacencies up, LSDB matches node count |
| BFD interaction | Ensure BFD is configured on IS-IS interfaces before Phase 3 |
| LDP/SR label switching | If running MPLS, ensure IS-IS is the IGP for LDP/SR before removing OSPF |
| Rollback | Keep OSPF running until IS-IS has been stable for 2+ weeks under production load |

## Design Patterns

### Pattern 1: SP Backbone (IS-IS)

```
Typical tier-1/tier-2 ISP:

  L2-only backbone (flat IS-IS domain)
  ├── All P routers: L2 only
  ├── PE routers: L1/L2 (if customer-facing areas exist) or L2 only
  ├── Metric style: wide
  ├── Overload on startup: 300s
  ├── SR-MPLS enabled
  └── TI-LFA for fast reroute
```

### Pattern 2: Large Enterprise (OSPF)

```
Multi-campus enterprise:

  Area 0 (backbone)
  ├── Core switches / routers
  ├── WAN connections
  ├── Inter-campus links
  │
  Area 10 (Campus A)
  ├── Totally stubby (or NSSA if external routes needed)
  │
  Area 20 (Campus B)
  └── Totally stubby
  
  Summary: ABR summary routes at area boundaries
```

### Pattern 3: Hybrid (Real World)

```
SP with acquired regional network:

  IS-IS L2 backbone (core)
  ├── New build, SR-MPLS
  │
  OSPF Area 0 (acquired regional)
  ├── Brownfield, working fine
  ├── Redistributed into IS-IS at border routers
  └── Migration planned over 12-18 months
```

## Verification

### Confirming Both Protocols Active (Ships in the Night)

| Check | IOS-XR | Junos |
|-------|--------|-------|
| IS-IS adjacencies | `show isis adjacency` | `show isis adjacency` |
| OSPF neighbors | `show ospf neighbor` | `show ospf neighbor` |
| Route preference | `show route <prefix> detail` | `show route <prefix> detail` |
| Active IGP for a prefix | `show route <prefix>` — check protocol code | `show route <prefix>` — check preference |

### Validating Migration Readiness

Before switching traffic from OSPF to IS-IS:

```
Checklist:
 □ IS-IS adjacency count matches expected (no missing neighbors)
 □ IS-IS LSDB node count matches network size
 □ Loopback routes reachable via IS-IS (check with higher AD first)
 □ BFD sessions up on IS-IS interfaces
 □ SR SIDs/labels programmed correctly under IS-IS
 □ No IS-IS overload bit set on any node
 □ Monitoring/alerting updated for IS-IS metrics
```

## Troubleshooting

### Common Migration Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| IS-IS adjacency not forming | NET not configured, P2P mismatch, or authentication mismatch | Check `show isis interface`, ensure P2P on both sides |
| Routes not shifting to IS-IS | AD/preference still favors OSPF | Verify distance/preference settings on ALL nodes |
| Traffic black hole after OSPF removal | IS-IS missing a prefix (interface not in IS-IS) | Audit all interfaces — compare OSPF and IS-IS interface lists |
| MPLS labels broken post-migration | LDP or SR still bound to OSPF | Ensure SR/LDP instance references IS-IS, not OSPF |
| Asymmetric routing during migration | Some nodes prefer IS-IS, others OSPF | Stage migration per-segment, never per-node |

### War Story: The 3 AM Migration That Went Sideways

> A tier-2 SP was migrating from OSPF to IS-IS across 400 nodes. They did Phase 3 (preference switch) region by region over four weeks. Week 3, they hit the Southeast region — 80 PEs. The cutover script worked perfectly on 79 of them. One PE had a stale config from a previous maintenance window that set OSPF distance to 5. That one PE kept preferring OSPF. Every prefix behind it was reachable, but asymmetric — traffic flowed in via IS-IS, returned via OSPF paths. Worked fine... until the OSPF removal in Phase 4 killed the return path. Traffic black hole, 47-minute outage, and a very unhappy NOC.
>
> **Lesson**: Before Phase 4 (OSPF removal), run a config audit across ALL nodes. Grep for `distance`, `preference`, any manual overrides. The one node you forget is the one that pages you.
