# 2.3 — IGP Convergence Tuning

## Overview
> Convergence isn't a single event — it's a pipeline. Failure detection → notification → SPF calculation → RIB update → FIB programming. Each stage has knobs. Tuning them wrong gives you either slow failover (revenue loss) or instability (also revenue loss). This section covers the convergence pipeline end-to-end for both IS-IS and OSPF, with real SP-grade timer values and the math behind them.


## Key Concepts
> **After this section you will understand:**
> - how SPF-based IGPs build topology and select best paths
> - which timers and fast-fail mechanisms control convergence behavior
> - how to validate IGP health with adjacency, LSDB, and route checks
> - common failure modes and how to isolate them quickly

## Theory

### The Convergence Pipeline

Every link failure traverses these stages in order:

```
┌─────────────┐   ┌─────────────┐   ┌──────────┐   ┌─────────┐   ┌─────────┐
│  Detection   │──▶│ Notification │──▶│   SPF    │──▶│   RIB   │──▶│   FIB   │
│  (BFD/IGP)   │   │ (LSP/LSA)   │   │  Compute │   │  Update │   │ Program │
└─────────────┘   └─────────────┘   └──────────┘   └─────────┘   └─────────┘
    ~150ms            ~10ms           ~50-200ms       ~10ms        ~50-500ms
```

**Total convergence = sum of all stages.** Typical SP target: **< 1 second** for any single failure. With optimization: **< 200ms** (approaches 50ms with FRR/TI-LFA).

### Stage 1: Failure Detection

#### BFD (Bidirectional Forwarding Detection)
The standard for sub-second failure detection. Runs in hardware on modern linecards.

| BFD Mode | Timer Range | Detection Time | Use Case |
|----------|------------|----------------|----------|
| Async (default) | 50ms–1s | 3× interval | Standard SP |
| Echo | 50ms–1s | 3× interval | Reduced control plane load |
| Micro-BFD (RFC 7130) | 3.3ms–1s | 3× interval | LAG member links |

**SP standard**: 50ms interval × 3 multiplier = **150ms detection**.

**Warning**: BFD at 50ms is aggressive. On congested control planes or software-based platforms, false positives *will* happen. Test before deploying. Some SPs use 300ms × 3 = 900ms as a safer default.

#### Interface Down Detection
Physical layer detection is fastest (~10ms for SONET/SDH, ~50ms for Ethernet). Modern optics report loss-of-light instantly. This is faster than BFD but only catches physical failures — not L3/forwarding plane issues.

#### IGP Hello Timers (Last Resort)
Default IS-IS: 10s hello / 30s hold. Default OSPF: 10s hello / 40s dead.
These are backup detection only. Never rely on IGP hellos for fast detection — that's BFD's job.

### Stage 2: LSP/LSA Generation & Flooding

Once a failure is detected, the router generates a new LSP (IS-IS) or LSA (OSPF) and floods it.

#### Throttling Parameters

| Parameter | IS-IS (IOS-XR) | OSPF (IOS-XR) | SP Recommended |
|-----------|----------------|---------------|----------------|
| Initial generation | `lsp-gen-interval initial` | `timers throttle lsa all initial` | 50ms |
| Second generation | `lsp-gen-interval secondary` | `timers throttle lsa all second` | 100ms |
| Max interval | `lsp-gen-interval maximum` | `timers throttle lsa all max` | 5000ms |

The exponential backoff works like this:
1. First LSP/LSA: sent after `initial` delay (50ms)
2. Second: sent after `secondary` delay (100ms)
3. Subsequent: delay doubles each time up to `maximum` (5000ms)
4. Timer resets to `initial` after quiet period

**Flooding optimization — IS-IS**:
```cisco
! IOS-XR: Reduce flooding by mesh-group or flooding reduction
router isis 1
  address-family ipv4 unicast
    ! Disable flooding on non-essential links
    interface GigabitEthernet0/0/0/2
      mesh-group blocked
```

**Flooding optimization — OSPF**:
```cisco
! IOS-XR: Prefix suppression (hide transit links)
router ospf 1
  prefix-suppression
```

### Stage 3: SPF Computation

#### SPF Throttling (Exponential Backoff)

Both IS-IS and OSPF use the same three-timer model:

| Timer | Function | SP Value |
|-------|----------|----------|
| `initial-wait` | Delay before first SPF after trigger | 50ms |
| `second-wait` | Delay before second SPF | 100ms |
| `max-wait` | Maximum delay (backoff ceiling) | 5000ms |

```
IS-IS (IOS-XR):
  router isis 1
    address-family ipv4 unicast
      spf-interval maximum-wait 5000 initial-wait 50 secondary-wait 100

IS-IS (Junos):
  protocols isis {
      spf-delay 50;
      overload-timeout 60;
  }

OSPF (IOS-XR):
  router ospf 1
    timers throttle spf 50 100 5000

OSPF (Junos):
  protocols ospf {
      spf-options {
          delay 50;
          holddown 100;
          rapid-runs 3;
      }
  }
```

#### Partial Route Computation (PRC) / Incremental SPF
When only a leaf prefix changes (not topology), a full SPF recomputation is wasteful. PRC/iSPF only recalculates affected prefixes:

```cisco
! IOS-XR IS-IS — PRC is automatic, but you can tune its timers separately
router isis 1
  address-family ipv4 unicast
    prc-interval maximum-wait 5000 initial-wait 50 secondary-wait 100
```

### Stage 4: RIB/FIB Update

After SPF computes new paths, the routing table and forwarding table must update.

- **RIB update**: Software operation, typically 10-50ms
- **FIB programming**: Hardware-dependent. 
  - Modern NPUs (Memory-based, e.g., Memory by Memory Silicon One): ~50ms for bulk updates
  - Memory-based TCAM (older platforms): 100-500ms
  - Software forwarding: up to seconds

**This stage is often the bottleneck.** You can tune SPF to 50ms, but if FIB programming takes 500ms, your total convergence is still >500ms.

### Putting It All Together — Convergence Budget

| Stage | Default | Tuned | With FRR |
|-------|---------|-------|----------|
| Detection (BFD) | 30s (IGP hello) | 150ms | 150ms |
| LSP/LSA generation | 5000ms | 50ms | 50ms |
| Flooding | 10-100ms | 10-100ms | N/A (precomputed) |
| SPF computation | 5000ms | 50ms | N/A (precomputed) |
| RIB/FIB update | 50-500ms | 50-500ms | ~0ms (pre-installed) |
| **Total** | **~35 seconds** | **~300-800ms** | **~150ms** |

### IP Fast Reroute (FRR) Mechanisms

FRR pre-computes and pre-installs backup paths. When BFD detects failure, traffic switches to backup *in the data plane* — no SPF needed.

#### LFA (Loop-Free Alternate) — RFC 5286
Simplest FRR. Uses a directly connected neighbor as backup if it provides a loop-free path.

**Coverage problem**: LFA can't always find a backup, especially in ring topologies. Typical coverage: 40-80% of prefixes.

#### Remote LFA (rLFA) — RFC 7490
Extends LFA by tunneling to a remote repair node via LDP. Covers most gaps LFA misses.

**Coverage**: 90-95% typically.

#### TI-LFA (Topology-Independent LFA) — RFC 9855
The gold standard. Uses Segment Routing to construct explicit backup paths via segment lists. **100% coverage** for any topology.

```cisco
! IOS-XR: TI-LFA with IS-IS + SR
router isis 1
  address-family ipv4 unicast
    segment-routing mpls
  interface GigabitEthernet0/0/0/0
    address-family ipv4 unicast
      fast-reroute per-prefix
      fast-reroute per-prefix ti-lfa
```

```junos
# Junos: TI-LFA with IS-IS + SR
protocols {
    isis {
        backup-spf-options {
            use-post-convergence-lfa;
        }
        interface ge-0/0/0.0 {
            node-link-protection;
        }
        source-packet-routing {
            srgb start-label 16000 index-range 8000;
        }
    }
}
```

**TI-LFA + BFD = sub-50ms failover for any single failure.** This is the endgame for SP convergence.

## Configuration

### Complete Convergence-Optimized IS-IS (IOS-XR)
```cisco
router isis CORE
 is-type level-2-only
 net 49.0001.0100.0000.0001.00
 log adjacency changes
 !
 address-family ipv4 unicast
  metric-style wide
  segment-routing mpls
  spf-interval maximum-wait 5000 initial-wait 50 secondary-wait 100
  prc-interval maximum-wait 5000 initial-wait 50 secondary-wait 100
  ! LSP generation throttle
  lsp-gen-interval maximum-wait 5000 initial-wait 50 secondary-wait 100
 !
 interface Loopback0
  passive
  address-family ipv4 unicast
   prefix-sid index 1
 !
 interface GigabitEthernet0/0/0/0
  point-to-point
  address-family ipv4 unicast
   fast-reroute per-prefix
   fast-reroute per-prefix ti-lfa
   metric 100
  !
  bfd minimum-interval 50
  bfd multiplier 3
  hello-interval 10
  hello-multiplier 3
```

### Complete Convergence-Optimized IS-IS (Junos)
```junos
protocols {
    isis {
        level 1 disable;
        lsp-lifetime 65535;
        spf-delay 50;
        overload-timeout 60;
        backup-spf-options {
            use-post-convergence-lfa;
            use-source-packet-routing;
        }
        source-packet-routing {
            srgb start-label 16000 index-range 8000;
            node-segment ipv4-index 1;
        }
        interface lo0.0 {
            passive;
        }
        interface ge-0/0/0.0 {
            point-to-point;
            level 2 metric 100;
            node-link-protection;
            bfd-liveness-detection {
                minimum-interval 50;
                multiplier 3;
            }
        }
    }
}
```

### Key Knobs

| Parameter | Conservative | Aggressive | Risk |
|-----------|-------------|------------|------|
| BFD interval | 300ms × 3 | 50ms × 3 | False positives on loaded control planes |
| SPF initial-wait | 200ms | 50ms | Minimal — SPF is lightweight |
| LSP/LSA gen initial | 100ms | 50ms | Minimal |
| Max-wait (all) | 10s | 5s | Lower ceiling = faster recovery from flap storms |
| Hello interval | 10s | 1s | Only if BFD not available — CPU cost |

## Verification & Monitoring

### Show Commands

| IOS-XR | Junos | What It Shows |
|--------|-------|---------------|
| `show isis spf-log` | `show isis spf log` | SPF run history with triggers |
| `show isis statistics` | `show isis statistics` | LSP counts, SPF counts |
| `show ospf statistics` | `show ospf statistics` | SPF run count and timing |
| `show bfd session` | `show bfd session` | BFD state, timers, packet counts |
| `show cef ipv4 <prefix>` | `show route forwarding-table <prefix>` | FIB entry with backup path |
| `show isis fast-reroute` | `show isis backup coverage` | TI-LFA/LFA coverage stats |

### What Good Looks Like
- SPF log shows runs triggered by real events, not constant churn
- BFD sessions stable with zero or near-zero packet loss
- TI-LFA coverage at **100%** for all protected prefixes
- FIB shows backup next-hop installed (repair path)
- Convergence test (shut/no-shut) completes in < 200ms with FRR

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| BFD flapping without link failure | Timer too aggressive for platform | Increase to 100ms or 300ms |
| SPF running every few seconds | Flapping link or LSA churn | Find source: `show isis/ospf spf-log` |
| TI-LFA shows < 100% coverage | Missing SR SIDs on some nodes | Verify all nodes have prefix-SIDs |
| Slow convergence despite tuned timers | FIB programming bottleneck | Check platform specs, consider FRR |
| Micro-loops during convergence | No ordered FIB update or FRR | Enable TI-LFA or ordered FIB |

### Methodology
1. **Measure baseline**: Shut a link, measure traffic loss duration
2. **Identify bottleneck**: Is it detection? SPF? FIB?
3. **Tune the bottleneck**: Apply appropriate timer changes
4. **Re-measure**: Confirm improvement
5. **Monitor for side effects**: Watch for false positives or instability

### War Stories

> **The BFD That Cried Wolf**
>
> A large SP deployed BFD at 50ms × 3 across their entire backbone. Worked great — until a new software release increased control plane CPU utilization by 15%. Suddenly, BFD sessions started flapping during peak traffic hours. Not because links were failing, but because the CPU couldn't process BFD keepalives fast enough. Each BFD flap triggered SPF, which triggered more CPU load, which caused more BFD flaps. Classic positive feedback loop.
>
> Fix: Moved BFD to hardware offload (supported on the platform but not enabled), and set a floor of 100ms for non-critical links. Core inter-POP links stayed at 50ms with hardware BFD. Access-facing links got 300ms.
>
> Lesson: **BFD timers are not one-size-fits-all.** Tiered BFD — aggressive on core, conservative on access — is the right pattern.
