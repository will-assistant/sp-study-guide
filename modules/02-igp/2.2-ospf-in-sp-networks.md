# 2.2 — OSPF in SP Networks

## Overview
> OSPF is the world's most deployed IGP — but in service provider backbones, it plays second fiddle to IS-IS. That doesn't mean you can ignore it. OSPF dominates enterprise aggregation, metro Ethernet access layers, and many regional/tier-2 SP networks. Understanding where OSPF fits (and where it breaks) in SP-scale environments is essential for anyone sitting a IE-SP or IE-SP exam. You'll encounter it. You need to own it.


## Key Concepts
> **After this section you will understand:**
> - how SPF-based IGPs build topology and select best paths
> - which timers and fast-fail mechanisms control convergence behavior
> - how to validate IGP health with adjacency, LSDB, and route checks
> - common failure modes and how to isolate them quickly

## Theory

### OSPF in the SP Context

#### Where OSPF Lives in SP Networks
OSPF rarely runs as the core backbone IGP in tier-1 providers. Where you *will* find it:

1. **Metro/access aggregation** — PE-to-CE or PE-to-aggregation in metro Ethernet rings
2. **Regional ISPs** — Smaller providers with hundreds (not thousands) of routers
3. **Data center underlay** — Especially with OSPFv3 for IPv6 fabrics
4. **Acquired networks** — Post-M&A, you inherit whatever the other guy ran
5. **Legacy cores** — Migrating from OSPF to IS-IS is painful enough that some just... don't

#### Why SPs Prefer IS-IS (Recap)
| Factor | OSPF | IS-IS |
|--------|------|-------|
| Runs on | IP (protocol 89) | CLNS (Layer 2) |
| IPv6 support | Requires OSPFv3 (separate process) | TLV extension (same process) |
| Extensibility | New LSA types, opaque LSAs | TLV — add anything, break nothing |
| Area design | Area 0 backbone requirement, stubs, NSSA | L1/L2, clean hierarchy |
| Segment Routing | Supported (but bolted on) | Native fit via TLVs |
| Vendor optimization | Good | Best (SP-focused investment) |

That said, OSPF is *not* technically inferior in all cases. For networks under ~500 routers with straightforward hierarchy, OSPF works fine.

### OSPF Packet Types

| Type | Name | Purpose |
|------|------|---------|
| 1 | Hello | Neighbor discovery, keepalive |
| 2 | DBD (Database Description) | LSDB summary exchange |
| 3 | LSR (Link State Request) | Request specific LSAs |
| 4 | LSU (Link State Update) | Carry LSAs |
| 5 | LSAck | Acknowledge LSAs |

All packets carry the OSPF header with Router ID, Area ID, authentication, and instance ID (OSPFv3).

### LSA Types — The Full Picture

This is where OSPF gets complex. Every LSA type exists because someone needed to solve a specific flooding/reachability problem. In SP networks, you must know all of them cold.

| Type | Name | Originated By | Flooding Scope | Purpose |
|------|------|--------------|----------------|---------|
| 1 | Router LSA | Every router | Intra-area | Links and costs for originating router |
| 2 | Network LSA | DR | Intra-area | Represents multi-access segment |
| 3 | Summary LSA | ABR | Inter-area | Prefixes from other areas |
| 4 | ASBR Summary | ABR | Inter-area | Reachability to ASBRs |
| 5 | AS External | ASBR | AS-wide | External routes (type 1/2 metrics) |
| 7 | NSSA External | ASBR in NSSA | NSSA area only | External routes in NSSA (converted to Type 5 at ABR) |
| 8 | Link LSA (v3) | Every router | Link-local | Link-local addr + options (OSPFv3) |
| 9 | Opaque (link) | Any | Link-local | Extended attributes (TE, SR) |
| 10 | Opaque (area) | Any | Intra-area | TE LSAs (RFC 3630), SR |
| 11 | Opaque (AS) | Any | AS-wide | Inter-area opaque info |

**Key insight**: Types 9/10/11 (Opaque LSAs) are how OSPF supports MPLS-TE and Segment Routing. Without them, OSPF can't play in modern SP networks.

### Area Design for Scale

#### The Backbone Problem
OSPF mandates Area 0 as the backbone. All inter-area traffic transits Area 0. This creates a single flooding domain at the core — the exact place where you want the *least* churn.

**SP-scale implications:**
- Every link-state change in Area 0 triggers SPF across all Area 0 routers
- ABRs must maintain the LSDB for Area 0 *plus* their attached areas
- Virtual links (the duct tape of OSPF) are fragile and operationally painful

#### Area Types

| Area Type | Allows External? | Allows Inter-area? | Use Case |
|-----------|-----------------|-------------------|----------|
| Standard | Yes (Type 5) | Yes (Type 3) | Default |
| Stub | No | Yes (Type 3) + default | Leaf areas, no external routes needed |
| Totally Stubby | No | No (default only) | Minimal routing table in leaf |
| NSSA | Via Type 7 (converted) | Yes (Type 3) | External injection without full external flooding |
| Totally NSSA | Via Type 7 (converted) | No (default only) | NSSA + totally stubby combined |

**SP design guidance:**
- Keep Area 0 lean — only P routers and PE uplinks
- Use NSSA for areas that originate external routes (e.g., connected/static redistribution at access PEs)
- Totally stubby/NSSA for pure leaf sites to minimize LSDB size
- **Never use virtual links in production** if you can avoid it. Redesign instead.

#### Summarization at ABRs
Unlike IS-IS (which has no inter-level summarization by default), OSPF *requires* you to think about summarization at every ABR:

```
ABR config (IOS-XR):
  area 10 range 10.10.0.0/16
  area 10 range 10.10.128.0/17 not-advertise   ← suppress specific

ABR config (Junos):
  area 0.0.0.10 {
      area-range 10.10.0.0/16;
      area-range 10.10.128.0/17 restrict;      ← suppress
  }
```

Good summarization at ABRs is the #1 way to keep OSPF stable at scale. Bad summarization is the #1 way to get paged at 3 AM.

### OSPF Convergence Mechanics

#### SPF Scheduling
Modern OSPF implementations use throttled SPF with exponential backoff:

| Parameter | Cisco Default | Junos Default | Aggressive SP |
|-----------|--------------|---------------|---------------|
| Initial delay | 5000 ms | 200 ms | 50 ms |
| Second delay | 10000 ms | 1000 ms | 100 ms |
| Max holdtime | 10000 ms | 5000 ms | 5000 ms |

```
IOS-XR:
  router ospf 1
    timers throttle spf 50 100 5000

Junos:
  protocols ospf {
      spf-options {
          delay 50;
          holddown 100;
          rapid-runs 3;
      }
  }
```

#### LSA Throttling
Controls how fast LSAs can be regenerated. Critical for dampening flapping links:

```
IOS-XR:
  router ospf 1
    timers throttle lsa all 50 100 5000

Junos:
  (LSA generation throttle is implicit in SPF options)
```

#### BFD Integration
For sub-second failure detection — essential in SP:

```
IOS-XR:
  router ospf 1
    area 0
      interface GigabitEthernet0/0/0/0
        bfd minimum-interval 50
        bfd multiplier 3
        bfd fast-detect

Junos:
  protocols ospf {
      area 0.0.0.0 {
          interface ge-0/0/0.0 {
              bfd-liveness-detection {
                  minimum-interval 50;
                  multiplier 3;
              }
          }
      }
  }
```

**BFD + OSPF = ~150ms failure detection** (3 × 50ms). Compare to OSPF dead timer default of 40 seconds. Night and day.

#### LFA and Remote LFA (Loop-Free Alternates)
IP Fast Reroute for OSPF — pre-computed backup paths:

```
IOS-XR:
  router ospf 1
    fast-reroute per-prefix
    fast-reroute per-prefix remote-lfa tunnel mpls-ldp

Junos:
  protocols ospf {
      backup-spf-options {
          remote-backup-calculation;
      }
      area 0.0.0.0 {
          interface ge-0/0/0.0 {
              node-link-protection;
          }
      }
  }
```

LFA gives you sub-50ms failover without MPLS-TE FRR. Remote LFA extends coverage when no direct LFA exists.

### OSPFv3 — The IPv6 Story

OSPFv3 (RFC 5340) was a full rewrite to support IPv6. Key differences from OSPFv2:

| Feature | OSPFv2 | OSPFv3 |
|---------|--------|--------|
| Addressing | IPv4 only | IPv6 native, dual-stack via AF (RFC 5838) |
| Runs over | IPv4 | IPv6 link-local |
| Authentication | In protocol (MD5/plaintext) | IPsec (external) |
| Identification | Router ID + IP | Router ID + Instance ID |
| LSA changes | Types 1-5, 7, opaque | New types 8 (Link), 9 (Intra-area Prefix) |
| Multi-instance | No | Yes (Instance ID field) |

**SP relevance**: OSPFv3 with Address Families (RFC 5838) can carry both IPv4 and IPv6 in a single process. This is how dual-stack OSPF deployments work in modern networks — but most SPs just use IS-IS with multi-topology instead.

## Configuration

### Basic Multi-Area OSPF (SP Style)

#### IOS-XR
```cisco
! P-router in Area 0
router ospf 1
 router-id 10.0.0.1
 log adjacency changes detail
 !
 ! SPF tuning for SP
 timers throttle spf 50 100 5000
 timers throttle lsa all 50 100 5000
 !
 ! LFA for IP FRR
 fast-reroute per-prefix
 !
 area 0
  interface Loopback0
   passive enable
  !
  interface GigabitEthernet0/0/0/0
   network point-to-point           ! Always P2P on SP links
   bfd minimum-interval 50
   bfd multiplier 3
   bfd fast-detect
   cost 100
  !
  interface GigabitEthernet0/0/0/1
   network point-to-point
   bfd minimum-interval 50
   bfd multiplier 3
   bfd fast-detect
   cost 100
```

```cisco
! ABR connecting Area 0 and Area 10 (NSSA)
router ospf 1
 router-id 10.0.0.5
 log adjacency changes detail
 timers throttle spf 50 100 5000
 timers throttle lsa all 50 100 5000
 !
 area 0
  interface Loopback0
   passive enable
  !
  interface GigabitEthernet0/0/0/0
   network point-to-point
   bfd minimum-interval 50
   bfd multiplier 3
   bfd fast-detect
 !
 area 10
  nssa                               ! NSSA area
  range 10.10.0.0/16                 ! Summarize into Area 0
  default-information-originate      ! Inject default into NSSA
  !
  interface GigabitEthernet0/0/0/1
   network point-to-point
   bfd minimum-interval 50
   bfd multiplier 3
   bfd fast-detect
```

#### Junos
```junos
# P-router in Area 0
protocols {
    ospf {
        spf-options {
            delay 50;
            holddown 100;
            rapid-runs 3;
        }
        area 0.0.0.0 {
            interface lo0.0 {
                passive;
            }
            interface ge-0/0/0.0 {
                interface-type p2p;
                metric 100;
                bfd-liveness-detection {
                    minimum-interval 50;
                    multiplier 3;
                }
            }
            interface ge-0/0/1.0 {
                interface-type p2p;
                metric 100;
                bfd-liveness-detection {
                    minimum-interval 50;
                    multiplier 3;
                }
            }
        }
    }
}
```

```junos
# ABR with NSSA area 10
protocols {
    ospf {
        area 0.0.0.0 {
            interface lo0.0 {
                passive;
            }
            interface ge-0/0/0.0 {
                interface-type p2p;
            }
        }
        area 0.0.0.10 {
            nssa {
                default-lsa default-metric 100;
                area-range 10.10.0.0/16;
            }
            interface ge-0/0/1.0 {
                interface-type p2p;
            }
        }
    }
}
```

### Key Knobs

| Parameter | Default (XR) | Default (Junos) | SP Recommended | Why |
|-----------|-------------|-----------------|----------------|-----|
| Hello interval | 10s | 10s | 10s (use BFD) | BFD handles fast detection; hello just maintains adjacency |
| Dead interval | 40s | 40s | 40s (use BFD) | Same — BFD is the fast failure detector |
| SPF initial delay | 5000ms | 200ms | 50ms | Faster convergence |
| SPF max holdtime | 10s | 5s | 5s | Bound backoff ceiling |
| Reference bandwidth | 100 Mbps | 100 Mbps | 1 Tbps | Differentiate 10G/100G/400G links |
| Interface type | Broadcast | Broadcast | point-to-point | Eliminates DR/BDR on P2P links (faster convergence) |
| Prefix suppression | Disabled | Disabled | Enabled | Reduce transit link LSAs |

**Critical**: Always set `auto-cost reference-bandwidth 1000000` (IOS-XR) or `reference-bandwidth 1t` (Junos) to differentiate modern link speeds. The 100 Mbps default makes 10G and 100G links the same cost. That's how you get asymmetric routing nightmares.

### Show Commands

| IOS-XR | Junos | What It Shows |
|--------|-------|---------------|
| `show ospf neighbor` | `show ospf neighbor` | Adjacency state, DR/BDR, timers |
| `show ospf database` | `show ospf database` | Full LSDB (all LSA types) |
| `show ospf database summary` | `show ospf database summary` | LSDB size per area |
| `show ospf interface` | `show ospf interface detail` | Interface state, timers, cost, area |
| `show ospf route` | `show ospf route` | OSPF RIB (pre-policy) |
| `show ospf border-routers` | `show ospf route abr` | ABR/ASBR reachability |
| `show ospf database opaque-area` | `show ospf database opaque-area` | TE/SR opaque LSAs |
| `show ospf statistics` | `show ospf statistics` | SPF run count, LSA counts |

### What Good Looks Like
- All expected neighbors in **FULL** state (or **FULL/DR** / **FULL/BDR** on multi-access)
- LSDB counts stable (not incrementing rapidly)
- SPF run count low and stable (not re-running every few seconds)
- BFD sessions **Up** on all OSPF interfaces
- No **EXSTART/EXCHANGE** stuck states (MTU mismatch indicator)
- `show ospf database summary` shows expected LSA counts per area

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Neighbors stuck in INIT | ACL blocking OSPF (protocol 89) or multicast | Check ACL, verify 224.0.0.5/6 permitted |
| Neighbors stuck in EXSTART | MTU mismatch on link | Match MTU or use `mtu-ignore` (bandaid) |
| Neighbors stuck in 2-WAY | Expected on multi-access (DROTHER) | Normal if not DR/BDR. Check if P2P is better |
| Flapping adjacencies | Unstable link, BFD too aggressive | Check interface errors, tune BFD timers |
| Missing routes from other area | ABR not in Area 0, or filtering | Check ABR connectivity to Area 0, check filters |
| Type 7→5 translation failure | ABR election issue in NSSA | Check ABR router-id priority, NSSA config |
| SPF running constantly | Flapping link in flooding domain | Find flapper: `show ospf database router adv-router X.X.X.X` |
| High CPU on SPF | LSDB too large, no summarization | Summarize at ABRs, stub areas, prefix suppression |

### Methodology
1. **Check neighbors**: `show ospf neighbor` — any not FULL?
2. **Check LSDB**: `show ospf database summary` — expected counts?
3. **Check specific LSAs**: `show ospf database router adv-router <rid>` — stale/flapping?
4. **Check SPF stats**: How often is SPF running? What's triggering it?
5. **Check BFD**: `show bfd session` — any down/flapping?
6. **Packet capture**: If all else fails, capture OSPF packets on the link

### War Stories

> **The MTU Mismatch That Wasn't**
>
> A regional ISP reported intermittent OSPF flaps on their ABR links. Neighbors would establish, exchange a few LSAs, then drop back to EXSTART. Classic MTU mismatch, right? Except both sides showed 9000 bytes.
>
> Turns out, the DWDM transport between sites was doing transparent L2 switching with a 16-byte overhead for OAM. The physical MTU was 9000, but the effective payload was 8984. OSPF's DBD packets would sometimes exceed this depending on LSDB size. The fix: set interface MTU to 8900 on both sides, or better yet, use `ip ospf mtu-ignore` and fix the transport config.
>
> Lesson: **Never trust the MTU your interface reports.** Test with large pings across the path. `ping size 8972 df-bit` is your friend.
