# 9.3 — OTN: Optical Transport Network

## Overview
> DWDM gives you wavelengths. OTN makes them manageable. Without OTN, your 80-channel DWDM system is just a really expensive bundle of point-to-point links with no visibility into what's happening between endpoints. OTN (ITU-T G.709) wraps client signals in a standardized digital envelope that provides per-wavelength performance monitoring, forward error correction, tandem connection monitoring across multi-operator paths, and deterministic multiplexing. It's SONET/SDH thinking applied to the optical age — the same obsessive OAM&P discipline, but designed for 100G/400G/800G and beyond. If DWDM is the highway, OTN is the traffic management system that tells you which lanes are degrading before trucks start crashing.


## Key Concepts
> **After this section you will understand:**
> - how packet and optical layers map services to physical transport
> - key constraints such as OSNR, reach, modulation, and wavelength planning
> - where OTN framing and grooming improve scale and resilience
> - how to coordinate packet and optical troubleshooting domains

## Theory

### Why OTN? The Problem It Solves

Before OTN, transporting diverse client signals over DWDM required protocol-specific transponders and gave operators zero visibility between endpoints. You had:

- **No standardized performance monitoring** per wavelength
- **No FEC** at the optical channel level (or proprietary-only FEC)
- **No fault isolation** across multi-operator/multi-domain paths
- **No multiplexing** — a 100G wavelength carrying a 10G client wasted 90G
- **No tandem connection monitoring** — if three carriers share a path, nobody knows whose segment is degrading

OTN solves all of this. Think of it as the **digital wrapper** — a protocol-agnostic envelope that encapsulates any client signal (Ethernet, SONET/SDH, Fibre Channel, ODUk) and adds the management plane that optical transport demands.

### OTN Layer Model

OTN defines a strict layered hierarchy, each with specific functions:

```
┌─────────────────────────────────────────────┐
│              Client Signals                  │
│  (Ethernet, SONET/SDH, FC, ODUk, FlexE)     │
└───────────────────┬─────────────────────────┘
                    │ Mapping
┌───────────────────▼─────────────────────────┐
│  OPU (Optical Payload Unit)                  │
│  Client adaptation, rate justification       │
└───────────────────┬─────────────────────────┘
                    │ + Path overhead
┌───────────────────▼─────────────────────────┐
│  ODU (Optical Data Unit)                     │
│  Path-level OAM, TCM, protection switching   │
└───────────────────┬─────────────────────────┘
                    │ + Section overhead + FEC
┌───────────────────▼─────────────────────────┐
│  OTU (Optical Transport Unit)                │
│  Section monitoring, FEC, frame alignment     │
└───────────────────┬─────────────────────────┘
                    │ Modulation onto wavelength
┌───────────────────▼─────────────────────────┐
│  OCh (Optical Channel)                       │
│  Single wavelength on the DWDM grid          │
└───────────────────┬─────────────────────────┘
                    │
┌───────────────────▼─────────────────────────┐
│  OMS (Optical Multiplex Section)             │
│  All wavelengths in a fiber (DWDM group)     │
└───────────────────┬─────────────────────────┘
                    │
┌───────────────────▼─────────────────────────┐
│  OTS (Optical Transport Section)             │
│  Fiber span between amplifiers/OXCs          │
└─────────────────────────────────────────────┘
```

**Key insight**: OPU/ODU/OTU are the **digital** layers (processed electronically). OCh/OMS/OTS are the **optical** layers (photonic domain). The boundary between them is the transponder/muxponder.

### The Signal Hierarchy

#### Classical OTN Rates (G.709)

| Signal | Line Rate | Payload Rate | Typical Client | FEC Overhead |
|--------|----------|-------------|----------------|-------------|
| OTU1 | 2.666 Gbps | 2.498 Gbps (ODU1) | STM-16/OC-48 (GbE via ODU0 mux) | ~7% RS(255,239) |
| OTU2 | 10.709 Gbps | 10.037 Gbps (ODU2) | 10GbE (LAN/WAN), STM-64/OC-192 | ~7% RS(255,239) |
| OTU2e | 11.096 Gbps | 10.399 Gbps (ODU2e) | 10GbE LAN PHY (native rate) | ~7% |
| OTU3 | 43.018 Gbps | 40.319 Gbps (ODU3) | 40GbE, STM-256/OC-768 | ~7% |
| OTU4 | 111.810 Gbps | 104.794 Gbps (ODU4) | 100GbE | ~7% |

**Why the odd rates?** OTN was designed to carry SONET/SDH natively. OTU1 carries STM-16 (2.488 Gbps) with byte-synchronous mapping. The extra bandwidth is overhead + FEC. When 10GbE came along, OTU2e was added because standard OTU2 payload (10.037 Gbps) couldn't carry 10GbE LAN PHY's full 10.3125 Gbps.

#### Sub-Rate ODU (Lower-Order)

For multiplexing smaller clients into larger containers:

| Signal | Payload Rate | Typical Client |
|--------|-------------|----------------|
| ODU0 | 1.24416 Gbps | GbE |
| ODU1 | 2.498 Gbps | STM-16, OC-48 |
| ODU2 | 10.037 Gbps | 10GbE, STM-64 |
| ODU2e | 10.399 Gbps | 10GbE LAN PHY |
| ODUflex(CBR) | Variable (n × 1.25 Gbps) | FC-1200, sub-rate Ethernet |
| ODUflex(GFP) | Variable | Packet clients (Ethernet, MPLS) via GFP |

#### Beyond 100G: OTUCn and FlexO (G.709.1, G.709.3)

At 400G and above, OTN had to evolve. A single OTU frame at 400G would require impossibly tight timing and monolithic hardware. The solution: **OTUCn** and **FlexO**.

**OTUCn (Optical Transport Unit - C for "beyond 100G", n = number of 100G slots)**:
- OTUC1 = 1 × ~100G (not used standalone — it's a building block)
- OTUC2 = 2 × ~100G → carries ODUCn server for 200GbE
- OTUC4 = 4 × ~100G → carries ODUCn server for 400GbE
- OTUC8 = 8 × ~100G → carries ODUCn server for 800GbE

**Key difference from classical OTN**: OTUCn has **no FEC** in the OTN frame itself. FEC is pushed down to the **FlexO** interface layer.

**FlexO (G.709.1/G.709.3)** — the physical interface framing for beyond-100G OTN:
- Provides FEC (typically concatenated FEC or oFEC) at the physical interface
- Allows bonding of multiple physical lanes (e.g., 4×100G FlexO instances = 400G FlexO group)
- Enables **interoperability** at 400G — different vendors' transponders can talk FlexO
- Think of it as the **PHY framing** that replaces OTUk's built-in FEC for high rates

```
Classical OTN (≤100G):
  Client → OPUk → ODUk → OTUk (with FEC) → wavelength

Beyond-100G OTN:
  Client → OPUCn → ODUCn → OTUCn (no FEC) → FlexO (with FEC) → wavelength(s)
```

### OTN Frame Structure

#### OTUk Frame (Classical, ≤100G)

The OTUk frame is a fixed structure of **4 rows × 4080 columns**, transmitted row by row:

```
        ←─── 3824 columns ───→ ←── 256 ──→
       ┌──────────┬─────────────┬──────────┐
Row 1  │  FAS/    │             │          │
       │  MFAS    │             │          │
       │  (7 col) │   OPUk      │   FEC    │
Row 2  │  OTUk OH │   Payload   │ RS(255,  │
       │  (3 col) │   (3808     │  239)    │
Row 3  │  ODUk OH │   columns)  │ (256     │
       │  (9 col) │             │  columns)│
Row 4  │          │             │          │
       └──────────┴─────────────┴──────────┘
        ← 16 col →
         Overhead

Total: 4 × 4080 = 16,320 bytes per frame
Frame period: 48.971 μs (OTU1), 12.191 μs (OTU2), 3.035 μs (OTU3), 1.168 μs (OTU4)
```

**Frame Alignment Signal (FAS)**: `0xF6F6F6282828` (6 bytes) — used for frame delineation. Same concept as SONET's A1/A2 bytes.

**Multi-Frame Alignment Signal (MFAS)**: 1 byte, counts 0–255, defines a 256-frame multi-frame. Critical for locating overhead fields that span multiple frames.

#### Overhead Breakdown

**OTU Overhead (Row 1, columns 8-14)**:
| Field | Bytes | Function |
|-------|-------|----------|
| SM (Section Monitoring) | 3 | BIP-8 error detection, BEI/BIAE, IAE, trail trace (TTI) |
| GCC0 (General Communication Channel) | 2 | Management communication between OTU termination points |
| Reserved | 2 | Future use |

**ODU Overhead (Rows 2-4, columns 1-14)**:
| Field | Bytes | Function |
|-------|-------|----------|
| PM (Path Monitoring) | 3 | BIP-8, BEI, trail trace (TTI), status |
| TCM1-6 | 3 each (18 total) | Tandem Connection Monitoring (up to 6 levels!) |
| GCC1, GCC2 | 2 each | General Communication Channels (in-band management) |
| APS/PCC | 4 | Automatic Protection Switching / Protection Communication Channel |
| FTFL | 1 | Fault Type and Fault Location |
| EXP | 2 | Experimental |
| TCM-ACT | 1 | TCM activation/deactivation |

**OPU Overhead (Rows 1-4, columns 15-16)**:
| Field | Bytes | Function |
|-------|-------|----------|
| PSI (Payload Structure Identifier) | 1 (256-frame multiframe) | Payload type, mapping type, multiplexing structure |
| JC (Justification Control) | 3 × variable | Rate adaptation (positive/negative justification) |
| NJO/PJO | variable | Justification opportunity bytes (frequency offset compensation) |

### Tandem Connection Monitoring (TCM)

This is OTN's killer feature for multi-operator networks. SONET/SDH had limited path monitoring — basically source-to-destination. OTN provides **six independent TCM levels**, each monitoring a different segment of the end-to-end path.

**Use case**: Customer A in New York connects to Customer B in London via three carriers:
```
Customer A ←→ [Carrier X, US] ←→ [Carrier Y, subsea] ←→ [Carrier Z, UK] ←→ Customer B
                 TCM1                    TCM2                   TCM3
              PM (end-to-end path monitoring)
```

Each carrier monitors their segment independently:
- **TCM1**: Carrier X monitors BIP-8 errors across their US network
- **TCM2**: Carrier Y monitors BIP-8 errors across the subsea segment
- **TCM3**: Carrier Z monitors BIP-8 errors across their UK network
- **PM**: End-to-end path monitoring from Customer A to Customer B

**Each TCM level provides**:
- BIP-8 error counting (bit-interleaved parity)
- BEI (Backward Error Indication) — tells the upstream about errors
- Trail Trace Identifier (TTI) — 64-byte string to verify connectivity
- Status bits (LCK, OCI, AIS)

**Why this matters**: When errors spike, you know *exactly* which operator's segment is degrading — no finger-pointing, no guessing. This is why wholesale/carrier-carrier transport contracts often mandate OTN with TCM.

### ODU Multiplexing

One of OTN's core strengths: deterministic, standards-based time-division multiplexing of lower-rate ODUs into higher-rate ODUs.

#### Multiplexing Map

```
ODU0 (1.25G) ──┐
                ├──→ ODU1 (2.5G): 2 × ODU0
ODU0 ───────────┘

ODU0 ──────────┐
ODU1 ──────────┤
ODUflex ───────┼──→ ODU2 (10G): 8 × ODU0, or 4 × ODU1, or mixed
               │
ODU0 ──────────┘

ODU0 ──────────┐
ODU1 ──────────┤
ODU2 ──────────┼──→ ODU3 (40G): 32 × ODU0, or 16 × ODU1, or 4 × ODU2, or mixed
ODUflex ───────┘

ODU0 ──────────┐
ODU1 ──────────┤
ODU2 ──────────┼──→ ODU4 (100G): 80 × ODU0, or 40 × ODU1, or 10 × ODU2,
ODU2e ─────────┤                  or 2 × ODU3 (31 TS each, 18 TS remaining), or mixed
ODU3 ──────────┤
ODUflex ───────┘
```

**Tributary Slot Assignment** (1.25G TS, per G.709 Amendment 3):
- ODU2 has **8 tributary slots** of 1.24416 Gbps each (~1.25G TS)
- ODU3 has **32 tributary slots** of 1.24416 Gbps each
- ODU4 has **80 tributary slots** of 1.24416 Gbps each

> **Note**: Earlier G.709 revisions also defined **2.5G tributary slots** (ODU3 = 16 TS, ODU4 = 40 TS). The 1.25G TS scheme is the current standard and provides finer granularity. Legacy equipment may use 2.5G TS — verify during interop planning.

> **Note**: ODU0 has no corresponding OTU0 — it exists only as a lower-order tributary for multiplexing into higher-rate ODUs.

An ODU1 (2.5G) consumes 2 tributary slots. An ODU2 consumes 8 tributary slots. The allocation is signaled in the OPU overhead (Tributary Slot Map / MSI - Multiplex Structure Identifier).

**ODUflex** allows arbitrary-rate multiplexing:
- **ODUflex(CBR)**: For constant-bit-rate clients that don't fit standard ODUk sizes. Rate = n × 1.24416 Gbps.
- **ODUflex(GFP)**: For packet clients mapped via Generic Framing Procedure. Rate can be resized hitlessly (bandwidth resizing — HAO, Hitless Adjustment of ODUflex).

> **Client Mapping Evolution**: G.709 originally defined AMP (Asynchronous Mapping Procedure) and BMP (Bit-synchronous Mapping Procedure) for client-to-OPU mapping. Modern OTN (G.709 2012+) uses **GMP (Generic Mapping Procedure)** as the primary mapping for both client signals and ODUj→ODUk multiplexing. GMP uses a simpler justification scheme and supports a wider range of client rates. If you see AMP/BMP on real gear, it's legacy.

#### HAO — Hitless Adjustment of ODUflex

This lets you resize an ODUflex connection **without dropping traffic**. The Link Connection Resize (LCR) protocol coordinates between endpoints to add or remove tributary slots from an active ODUflex connection.

**Process**:
1. Source requests resize via LCR protocol (carried in ODU overhead)
2. Intermediate nodes update cross-connect maps
3. Both ends adjust simultaneously at the negotiated multi-frame boundary
4. Zero packet loss if implemented correctly

**Real-world use**: A customer's bandwidth demand fluctuates between 5G and 15G. Instead of provisioning a fixed ODU2 (10G), you provision ODUflex and resize between 4 and 12 tributary slots based on demand. The catch: not all vendors implement HAO cleanly, and coordination across multiple nodes is where things get interesting.

### Forward Error Correction (FEC)

OTN's FEC is what allows optical signals to operate closer to the Shannon limit without regeneration.

#### Classical OTN FEC: Reed-Solomon RS(255,239)

- Adds **256 columns** per row (6.7% overhead)
- Corrects up to **8 symbol errors** per RS codeword (16 interleaved codewords per row)
- Provides ~6.0 dB **Net Coding Gain (NCG)** at 10⁻¹² output BER (range: 5.8–6.2 dB depending on conditions)
- Standardized, interoperable across all vendors

**Pre-FEC vs Post-FEC BER**:
- **Pre-FEC BER**: Error rate before FEC correction. This is what you monitor. Healthy range: < 10⁻³ to 10⁻⁴ depending on FEC type.
- **Post-FEC BER**: Error rate after FEC correction. Target: < 10⁻¹⁵ (error-free for practical purposes).
- **FEC limit/threshold**: The pre-FEC BER at which FEC can no longer correct all errors. For RS(255,239): ~1.8 × 10⁻⁴.

#### Enhanced FEC (EFEC) and Soft-Decision FEC (SD-FEC)

| FEC Type | NCG | Pre-FEC Threshold | Overhead | Notes |
|----------|-----|-------------------|----------|-------|
| RS(255,239) — G.709 standard | ~6.0 dB | ~1.8 × 10⁻⁴ | 6.7% | Interoperable |
| EFEC (various proprietary) | ~8–9 dB | ~1 × 10⁻³ | 7–25% | Vendor-specific |
| SD-FEC (soft-decision) | ~10–12 dB | ~2 × 10⁻² | 15–25% | Used with coherent; often proprietary |
| oFEC (open FEC, OIF) | ~11.1 dB | ~1.5 × 10⁻² | ~25% | 400ZR/ZR+ standard |
| cFEC (concatenated FEC, FlexO) | ~9.7 dB | ~4.5 × 10⁻³ | ~15% | FlexO standard option |

**SD-FEC revolution**: Classical RS FEC makes hard decisions (each bit is 0 or 1). SD-FEC receives *confidence levels* from the coherent DSP (e.g., "this bit is probably a 1 with 80% confidence") and uses that probabilistic information for dramatically better correction. This is why modern coherent optics can operate at pre-FEC BERs that would be catastrophic for hard-decision FEC.

**Key insight for IP engineers**: When your router's coherent optic shows pre-FEC BER of 1.5 × 10⁻², don't panic — if it's running SD-FEC with a threshold of 2 × 10⁻², you have margin. But if that number is *climbing*, your optical path is degrading and you need to investigate before it crosses the cliff.

### OTN Protection and Restoration

OTN provides several protection mechanisms at the ODU layer:

#### ODU SNC/I (Sub-Network Connection / Inherent Monitoring)

The most common OTN protection scheme. Uses ODU-level PM to detect failures.

- **1+1 Unidirectional**: Traffic sent on both working and protection paths; receiver selects the better one. No signaling delay — switchover in <50 ms.
- **1:1 Bidirectional**: Traffic on working path; protection path carries extra traffic (preemptible). Requires APS signaling via APS/PCC overhead.
- **1:N Shared**: N working paths share one protection path. Efficient but slower recovery.

#### ODU SNC/N (Non-Intrusive Monitoring)

Protection based on non-intrusive monitoring — doesn't terminate the ODU, just monitors it in transit. Used at intermediate nodes for segment protection.

#### Client Signal Fail (CSF)

When a client signal fails at the ingress, OTN propagates a **Client Signal Fail** indication through the ODU path to the far end. The far-end equipment can then signal the client (e.g., Ethernet remote fault) to trigger client-layer protection.

**Protection hierarchy**:
1. **Optical layer** (OMS/OTS): Optical path protection, <50 ms (wavelength restoration)
2. **OTN layer** (ODU): ODU SNC protection, <50 ms (digital cross-connect reroute)
3. **Client layer**: MPLS FRR, IP reroute (network-dependent)

Best practice: Let the lowest possible layer handle protection to minimize convergence time.

### GCC — General Communication Channels

OTN provides **three GCC channels** for in-band management:

| Channel | Location | Termination | Typical Use |
|---------|----------|-------------|-------------|
| GCC0 | OTU overhead | Section endpoints | OTU-level management, OSPF-T/IS-IS for control plane |
| GCC1 | ODU overhead | Path endpoints | Path-level management, EMS communication |
| GCC2 | ODU overhead | Path endpoints | GMPLS signaling (RSVP-TE), path computation |

**Why GCC matters**: Without GCC, you need a separate out-of-band management network (OSC - Optical Supervisory Channel, or dedicated management VLAN). GCC gives you in-band management that follows the OTN path — critical for automated provisioning and control plane signaling (GMPLS/ASON).

**GMPLS over GCC**: Many OTN implementations run OSPF-TE and RSVP-TE over GCC2 to create a distributed control plane for automatic ODU path provisioning and restoration. This is the ASON (Automatically Switched Optical Network) architecture in practice.

## Configuration

> **⚠️ Config Disclaimer**: OTN configuration varies significantly across platforms and software versions. The examples below are representative/illustrative — verify against current platform documentation before deploying. IOS-XR OTN syntax differs between NCS 2000, NCS 4000, and NCS 1004. Junos OTN syntax differs between PTX and MX with OTN MICs. Ciena uses SAOS/TL1 on 6500 and YANG-based CLI on Waveserver.

### IOS-XR (NCS 2000/NCS 4000/NCS 1004)

```cisco
! === OTN Controller Configuration ===

! Configure a 100G OTN wavelength with OTU4
controller Optics 0/0/0/0
 dwdm-carrier 100MHz-grid frequency 19310  ! 193.1 THz (C-band center)
!

! Configure OTU4 parameters
controller OTU4 0/0/0/0
 fec enhanced                               ! Enhanced FEC (configured at OTU level)
 gcc0                                       ! Enable GCC0 for section management
 performance-monitoring enable
 threshold pre-fec-ber 1e-3                 ! Alarm if pre-FEC BER exceeds threshold
 trail-trace-identifier
  tx-tti ascii "NYC-PE1-TO-LON-PE2"         ! 64-byte trail trace string
  expected-tti ascii "LON-PE2-TO-NYC-PE1"
 !
!

! Configure ODU4 path
controller ODU4 0/0/0/0
 gcc1                                       ! Enable GCC1 for path management
 gcc2                                       ! Enable GCC2 for GMPLS signaling
 performance-monitoring enable
 !
 ! Tandem Connection Monitoring - level 1
 tcm 1
  mode operational                          ! Enable TCM1 monitoring
  trail-trace-identifier
   tx-tti ascii "CARRIER-X-US-SEGMENT"
   expected-tti ascii "CARRIER-X-US-SEGMENT-FAR"
  !
  performance-monitoring enable
 !
 ! Tandem Connection Monitoring - level 2
 tcm 2
  mode transparent                          ! Pass through (monitored by another carrier)
 !
!

! Map 100GbE client into OTU4
! Note: On NCS 1004/2000, 100GbE → OTU4 client mapping is typically
! automatic when the client controller is associated with the OTN line controller.
! Explicit mapping configuration varies by platform and software version.
!

! === ODU Cross-Connect (Muxponder/OXC) ===
! Multiplex 10 × ODU2 into ODU4

controller ODU4 0/0/0/0
 odu-cross-connect 1
  from-controller ODU2 0/1/0/0              ! Client-side ODU2 #1
  tributary-slot 1-8                        ! Assign to TS 1-8 (each ODU2 = 8 TS)
 !
 odu-cross-connect 2
  from-controller ODU2 0/1/0/1              ! Client-side ODU2 #2
  tributary-slot 9-16                       ! Assign to TS 9-16
 !
 ! ... repeat for remaining ODU2 clients
!

! === ODU Protection (1+1) ===
controller ODU4 0/0/0/0
 protection 1plus1
  working-controller ODU4 0/0/0/0           ! Working path
  protection-controller ODU4 0/1/0/0        ! Protection path
  switching-mode unidirectional
  revertive                                 ! Revert to working after recovery
  wtr 300                                   ! Wait-to-restore: 300 seconds
 !
!

! === OTN Performance Monitoring Thresholds ===
controller OTU4 0/0/0/0
 pm 15-min otu threshold es 100             ! Errored Seconds threshold
 pm 15-min otu threshold ses 10             ! Severely Errored Seconds
 pm 15-min otu threshold bbe 1000           ! Background Block Errors
 pm 24-hour otu threshold uas 10            ! Unavailable Seconds
!
controller ODU4 0/0/0/0
 pm 15-min odu threshold es 100
 pm 15-min odu threshold ses 10
!
```

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| FEC type | RS(255,239) | Enhanced/SD-FEC for coherent | +3–6 dB coding gain extends reach |
| Pre-FEC BER threshold | Varies | 10⁻³ for SD-FEC, 10⁻⁴ for EFEC | Alarm before crossing FEC cliff |
| TTI string format | Empty | `<SITE>-<SLOT>-TO-<FAR-SITE>` | Essential for fault isolation |
| TCM levels | Disabled | Enable per operator segment | Multi-domain monitoring |
| GCC | Disabled | GCC0 + GCC2 | Section mgmt + GMPLS signaling |
| PM interval | 15-min + 24-hour | Both | ITU-T G.7710 standard intervals |
| Protection WTR | 300s | 300–600s | Avoid flapping; G.873.1 suggests 300s minimum |
| ODU2e vs ODU2 | ODU2 | ODU2e for 10GbE LAN PHY | ODU2 truncates 10GbE LAN rate |

### What Good Looks Like

**OTU4 healthy state**:
```
Controller State: Up
  FAS: In frame (locked)
  FEC: Enhanced FEC enabled
  Pre-FEC BER: 2.3e-05  ← Healthy (well below threshold)
  Post-FEC BER: 0.0e+00  ← Error-free post-correction
  FEC corrected bits (15-min): 45,230
  FEC uncorrectable blocks: 0  ← CRITICAL — must be zero
  SM-BIP8 errors: 0
  SM-TTI: Match
```

**ODU4 with TCM healthy**:
```
Controller State: Up
  PM-BIP8 errors (15-min): 0
  TCM1 Status: Operational
  TCM1-BIP8 errors: 0
  TCM1-TTI: Match
  TCM2 Status: Transparent (pass-through)
  Protection: Working active, no switch events
```

**Red flags**:
- `FEC uncorrectable blocks > 0` — FEC is overwhelmed; signal quality below FEC threshold
- `Pre-FEC BER climbing` — Optical path degrading (fiber issue, amplifier degradation, filter tightening)
- `SM-TTI: Mismatch` — Wrong fiber patched, or cross-connect misconfigured
- `TCMx-BIP8 errors increasing, PM-BIP8 = 0` — Errors being introduced in the TCMx segment but corrected before path endpoint
- `ODU-AIS` — Upstream failure; trace back to the source

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| OTU LOF (Loss of Frame) | No signal, wrong wavelength, fiber break | Check fiber, verify DWDM channel assignment, check optical power |
| OTU LOM (Loss of Multi-frame) | Frame locked but multi-frame not syncing | Typically follows LOF — same root cause. Check for signal quality |
| FEC uncorrectable blocks | Pre-FEC BER exceeds FEC threshold | Check OSNR, amplifier gains, fiber loss budget; reduce modulation order |
| TTI mismatch alarm | Wrong fiber patching or cross-connect error | Verify physical connectivity, cross-connect provisioning |
| ODU-AIS (Alarm Indication Signal) | Upstream OTU or server failure | Trace upstream — find the originating node |
| ODU-LCK (Locked) | Administrative lock on upstream ODU | Intentional maintenance; verify with upstream operator |
| ODU-OCI (Open Connection Indication) | No client mapped to this ODU | Verify client mapping, cross-connect provisioning |
| TCM-BEI errors on one level only | Degradation in that specific segment | Isolate to the TCM segment; check amplifiers/fibers in that span |
| ODUflex HAO resize failure | Node doesn't support HAO, or LCR protocol timeout | Verify HAO support on all nodes; check LCR signaling |
| PM threshold crossing alarm | ES/SES/UAS exceed configured thresholds | May be transient (fiber event) or sustained (degradation trend) |
| GCC channel down | GCC not enabled on both ends, or intermediate node stripping overhead | Enable GCC on all endpoints; verify transparent pass-through at intermediate OXC nodes |
| Protection switch flapping | WTR too short, intermittent fiber fault | Increase WTR timer; investigate root cause (bad splice, dirty connector) |

### Methodology

**OTN Fault Isolation Workflow**:

```
1. Check OTU layer first
   ├── LOF/LOM? → Physical layer issue (no signal, wrong λ)
   ├── FEC uncorrectable? → OSNR degradation → check amplifiers, fiber loss
   └── OTU layer clean? → Move to ODU layer
   
2. Check ODU layer
   ├── AIS? → Upstream failure → trace back node by node
   ├── LCK? → Maintenance lock → verify with upstream operator  
   ├── OCI? → No client mapped → check cross-connect provisioning
   └── ODU layer clean? → Check PM counters and TCM
   
3. Check TCM levels
   ├── Errors on TCM1 only? → Degradation in segment 1
   ├── Errors on TCM1+2? → Degradation before or within segment 2
   └── PM errors but all TCM clean? → Issue at path endpoints
   
4. Check PM trends
   ├── Sudden spike? → Fiber event (cut, bend, connector)
   ├── Gradual increase? → Component degradation (amplifier, connector aging)
   └── Periodic pattern? → Environmental (temperature, vibration)
   
5. Check protection state
   ├── Active on protection path? → Working path failed → diagnose working
   └── No switch? → Both paths OK or protection not configured
```

**FEC margin analysis**:
```
Current pre-FEC BER:  1.2 × 10⁻⁴
FEC threshold:        1.8 × 10⁻⁴  (RS(255,239))
Margin:               0.6 × 10⁻⁴   ← Getting tight!

Action: Monitor trend. If climbing, schedule maintenance.
        If stable, acceptable but low margin for temperature swings.
```

### War Stories

> **The Invisible TCM Error**
>
> A tier-1 carrier leased a 100G wavelength across three providers for a US coast-to-coast path. Everything looked clean — zero PM errors end-to-end. But customers on that path complained about intermittent packet loss, roughly once per hour, always for a few seconds.
>
> The problem: TCM was **provisioned but not activated** on the middle carrier's segment (mode transparent instead of operational). The middle carrier had a flaky amplifier that occasionally caused burst errors — their EDFA was thermally cycling and periodically dropping gain by 2 dB. The burst errors were short enough that PM 15-minute counters averaged them away (they showed up as a few BBE per 15-min interval, well below thresholds), and since TCM2 was transparent, nobody was monitoring that specific segment's BIP-8.
>
> Once TCM2 was switched to operational mode, they could see the BIP-8 error bursts correlating exactly with the amplifier's thermal cycles. The middle carrier replaced the EDFA, and the packet loss vanished.
>
> **Lesson**: TCM is useless if it's not **activated**. And 15-minute PM averaging can hide bursty errors that cause real traffic impact. For critical paths, also look at 1-second PM granularity (if your equipment supports it) or streaming telemetry at sub-second intervals.

## OTN in the Modern Network

### OTN vs. Packet-Only: The Eternal Debate

The industry has been arguing about this for 15 years, and it's not settled:

**Pro-OTN camp** (carriers, wholesale providers):
- Deterministic multiplexing for wholesale services (sell ODU2 services like you sold DS3s)
- TCM is non-negotiable for multi-operator paths
- Sub-50ms protection at the transport layer
- Hard QoS — an ODU2 is always 10G, no oversubscription possible
- Mature OAM that the operations team actually understands

**Anti-OTN camp** (webscalers, some modern SPs):
- OTN adds overhead (6.7-25% for FEC) and complexity
- Modern coherent optics have built-in FEC and PM — the DSP does what OTN does
- IPoDWDM eliminates the OTN layer entirely — router coherent pluggable direct to ROADM
- Packet statistical multiplexing is more efficient than TDM for bursty traffic
- EVPN/SR provides the flexibility that rigid ODU cross-connects can't match

**Reality**: Both are right, in context.
- **Wholesale/carrier-carrier transport**: OTN wins. You need TCM, you need hard-bandwidth services, your customers expect ODU2/ODU4 handoffs.
- **Single-operator backbone with own coherent optics**: OTN overhead may not be justified. IPoDWDM is simpler and cheaper.
- **Hybrid**: Many SPs run OTN on wholesale/multi-operator paths and IPoDWDM on their own backbone. This is probably the most common real-world architecture in 2025+.

### OTN + Coherent Pluggables (ZR/ZR+)

400ZR/ZR+ pluggables (from Section 9.2) raise an interesting question: if the pluggable has its own FEC and PM, where does OTN fit?

**Answer**: It depends on the deployment model.

1. **Router + ZR pluggable → ROADM**: No OTN. The coherent DSP in the pluggable handles FEC and provides PM via the router CLI. This is IPoDWDM.

2. **Router → Transponder → Line system**: OTN wrapping happens in the transponder. The transponder maps the client Ethernet into OTU4/OTUCn, adds TCM, FEC, and ODU multiplexing.

3. **Router + ZR pluggable → OTN switch → Line system**: The OTN switch (e.g., Ciena 6500) can receive the ZR signal and cross-connect it as an ODU within the OTN fabric. You get OTN benefits (TCM, protection, multiplexing) with router-based coherent optics.
