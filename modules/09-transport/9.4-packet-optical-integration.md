# 9.4 — Packet-Optical Integration

## Overview
> You've learned wavelengths (9.2), digital wrapping (9.3), and where they sit in the hierarchy (9.1). Now the question every SP architect eventually faces: how do packets and photons actually work together? Packet-optical integration is the convergence of IP/MPLS routers and optical transport — from the traditional "ships in the night" model where routers and DWDM systems pretend the other doesn't exist, to fully integrated architectures where a single router port lights a coherent wavelength directly onto fiber. This section covers the architectural models, control plane integration (GMPLS, OpenROADM, OpenConfig), the IPoDWDM revolution driven by 400ZR/ZR+ pluggables, multi-layer traffic engineering, and the operational reality of making Layer 0-3 behave as a single network. Getting this wrong means stranded capacity, slow restoration, and an OPEX death spiral. Getting it right means your network provisions wavelengths in minutes instead of weeks.


## Key Concepts
> **After this section you will understand:**
> - how packet and optical layers map services to physical transport
> - key constraints such as OSNR, reach, modulation, and wavelength planning
> - where OTN framing and grooming improve scale and resilience
> - how to coordinate packet and optical troubleshooting domains

## Theory

### The Integration Spectrum

Packet-optical integration exists on a spectrum, from complete separation to full convergence:

```
 Separation                                              Convergence
 ◄──────────────────────────────────────────────────────────────────►
 
 Ships in     Multi-layer    Unified        IPoDWDM      Converged
 the Night    Coordination   Control Plane  (ZR/ZR+)     Packet-Optical
 
 • Separate   • L0-L3 aware  • GMPLS/UNI   • Router     • Single
   management • Manual or    • Automated      lights       element
 • No cross-    semi-auto      provisioning   wavelength • Software-
   layer        correlation • Path          • No          defined
   visibility • OSS mediation  computation    transponder  optical
```

### Model 1: Ships in the Night (Traditional)

The dominant model for the last 20 years. Router team and optical team operate independently:

**Architecture:**
```
┌─────────────────────┐     ┌─────────────────────┐
│   IP/MPLS NMS/OSS   │     │  Optical NMS/OSS    │
│   (Crosswork, NSP)  │     │  (MCP, Ciena BP)    │
└─────────┬───────────┘     └─────────┬───────────┘
          │                           │
          │                           │
┌─────────▼───────────┐     ┌─────────▼───────────┐
│   Router Layer       │     │  Optical Layer       │
│   (IP/MPLS/SRv6)     │     │  (DWDM/OTN/ROADM)   │
│                      │     │                      │
│   ┌──────────────┐   │     │   ┌──────────────┐   │
│   │ ASR 9000     ├───┼─────┼──►│ Transponder  │   │
│   │ NCS 5500     │   │     │   │ (grey optic) │   │
│   │ MX Series    │   │     │   │              │   │
│   └──────────────┘   │     │   └──────┬───────┘   │
│                      │     │          │           │
└──────────────────────┘     │   ┌──────▼───────┐   │
                             │   │ Mux/ROADM    │   │
                             │   │ (colored λ)  │   │
                             │   └──────────────┘   │
                             └──────────────────────┘
```

**Characteristics:**
- Router uses grey (short-reach) optics; transponder converts to colored DWDM wavelength
- Separate management systems, separate teams, separate change windows
- Optical layer is a "black box" to the router — just another link
- No automatic correlation between router link failure and optical fault
- Provisioning requires tickets between teams (days to weeks)

**When it makes sense:**
- Multi-vendor optical and routing (different vendors per layer)
- Regulatory separation (wholesale optical, retail IP)
- Legacy infrastructure with sunk transponder investment
- Very large scale where organizational separation is intentional

**Limitations:**
- Stranded capacity: optical team provisions conservatively, router team doesn't know what's available
- Slow provisioning: cross-team coordination adds days
- Suboptimal restoration: router reroutes around a fiber cut, but optical protection was already handling it — double recovery, wasted capacity
- No multi-layer TE: can't optimize across packet and optical simultaneously

### Model 2: Multi-Layer Coordination

The middle ground. Layers remain separate but share information through an orchestration layer:

**Architecture:**
```
┌───────────────────────────────────────────────┐
│           Multi-Layer Orchestrator             │
│  (Cisco Crosswork, Nokia NSP, Ciena BP,       │
│   Juniper Paragon, open-source: OpenDaylight)  │
│                                                │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │ Topology  │  │ Path     │  │ Service  │    │
│  │ Discovery │  │ Compute  │  │ Provisn  │    │
│  └──────────┘  └──────────┘  └──────────┘    │
└────────┬───────────────────────────┬──────────┘
         │ BGP-LS / YANG             │ RESTCONF/
         │ LLDP / OSPF-TE            │ NETCONF
    ┌────▼────────┐           ┌──────▼──────────┐
    │ Router Layer │           │ Optical Layer    │
    │ (L2/L3)      │           │ (DWDM/OTN)      │
    └──────────────┘           └─────────────────┘
```

**Key enablers:**
- **BGP-LS** exports router topology to the orchestrator
- **YANG models** (OpenConfig, vendor-native) expose optical state
- **RESTCONF/NETCONF** for cross-layer provisioning
- **LLDP/adjacency discovery** correlates router ports to optical endpoints

**Operational flow:**
1. Orchestrator discovers both topologies
2. Capacity request comes in (e.g., "100G path from Chicago to Dallas")
3. Orchestrator checks optical inventory for available wavelength
4. Provisions optical path (ROADM cross-connects, transponder tuning)
5. Provisions router interface (IP address, IGP, MPLS)
6. End-to-end service validated automatically

**Benefits:**
- Provisioning drops from days to minutes
- Inventory visibility across layers prevents stranding
- Fault correlation: "Router sees link down because OTN alarm on span 3"
- Multi-layer path computation (route traffic over diverse optical paths)

### Model 3: Unified Control Plane (GMPLS)

GMPLS (Generalized MPLS) extends MPLS signaling to optical networks. Theoretically elegant. Practically... complicated.

**GMPLS Signaling Stack:**

| Protocol | Role | Packet Equivalent |
|----------|------|-------------------|
| OSPF-TE / IS-IS-TE | Topology & TE database | Same, extended with optical TLVs |
| RSVP-TE | Path signaling | Same, extended for lambda/fiber/TDM |
| LMP (Link Management Protocol) | Optical link discovery, verification | New — no packet equivalent |
| CR-LDP (optional) | Alternative signaling | Rarely deployed |

**Switching Capabilities (ISCDs):**

```
┌─────────────────────────────────────────────────┐
│  PSC (Packet Switch Capable)     — Routers      │
│  L2SC (Layer-2 Switch Capable)   — Switches     │
│  TDM (Time-Division Multiplex)   — OTN/SONET    │
│  LSC (Lambda Switch Capable)     — ROADMs       │
│  FSC (Fiber Switch Capable)      — OXC/patch    │
└─────────────────────────────────────────────────┘

Hierarchy: PSC < L2SC < TDM < LSC < FSC
Higher layers can request paths through lower layers
```

**UNI (User-Network Interface):**

The practical deployment model for GMPLS — routers request wavelengths from the optical network via a standardized interface:

```
    Router                    Optical Network
  ┌────────┐   UNI-C        ┌────────────────┐
  │        ├────────────────►│  UNI-N         │
  │ ASR 9K │  RSVP-TE       │  ┌──────────┐  │
  │        │  (wavelength    │  │  ROADM   │  │
  │        │   request)      │  │  NMS     │  │
  │        │◄────────────────│  │          │  │
  │        │  (confirm +     │  └──────────┘  │
  └────────┘   wavelength    └────────────────┘
               assignment)
```

**GMPLS Reality Check:**
- **Where it works**: Within a single vendor's optical domain (Ciena Waveserver + 6500, Nokia PSS + 1830, Infinera + ICE6)
- **Where it struggles**: Multi-vendor optical, multi-domain, brownfield deployments
- **Current status**: Largely superseded by SDN/RESTCONF-based orchestration in greenfield builds
- **Still relevant**: Some large carriers run GMPLS internally; understanding it is required for IE-SP exams

### Model 4: IPoDWDM — The Pluggable Revolution

This is where the industry is heading. The router becomes the transponder.

**Evolution:**

```
Generation 1 (2005-2015):           Generation 2 (2015-2022):
Discrete transponder                 Integrated photonics (CFP/CFP2)

Router ──grey──► Transponder ──λ──►  Router ──CFP2-DCO──────────────►
                                     (coherent pluggable in router)

Generation 3 (2022+):
400ZR/ZR+ (QSFP-DD, OSFP)

Router ──QSFP-DD ZR──────────────────────────────────────────────────►
(standard form factor, standard interop, standard management)
```

**400ZR (OIF Standard):**

| Parameter | Specification |
|-----------|---------------|
| Form Factor | QSFP-DD or OSFP |
| Data Rate | 400G (DP-16QAM) |
| Reach | ~120 km (unamplified), point-to-point |
| FEC | oFEC (concatenated, ~6.2 dB NCG) |
| Power | <15W per module |
| Wavelength Grid | 100 GHz ITU grid, C-band (48 channels) |
| Management | CMIS + OpenConfig |
| Interop | Multi-vendor by design (OIF interop tested) |

**OpenZR+ (Multi-Source Agreement):**

| Parameter | Specification |
|-----------|---------------|
| Form Factor | QSFP-DD or OSFP |
| Data Rate | 100G, 200G, 300G, 400G (tunable modulation) |
| Reach | Up to 2,000+ km (with amplification, at lower rates) |
| Modulation | DP-QPSK (100G) → DP-8QAM (200-300G) → DP-16QAM (400G) |
| Power | <25W per module |
| Wavelength Grid | Flex-grid (6.25 GHz granularity) |
| Management | CMIS + OpenConfig |
| Use Cases | Metro + regional + limited long-haul |

**IPoDWDM Architecture:**

```
                    No transponder shelf needed!
                    
┌──────────────┐    Dark fiber / amplified line    ┌──────────────┐
│  Router A    │◄─────────────────────────────────►│  Router B    │
│  NCS 5500    │    400ZR QSFP-DD                  │  NCS 5500    │
│              │    (coherent on router port)       │              │
│  ┌────────┐  │                                   │  ┌────────┐  │
│  │ ZR/ZR+ │  │    Amplified DWDM line system     │  │ ZR/ZR+ │  │
│  │ module  ├──┼──►[EDFA]──►[ROADM]──►[EDFA]──────┼──┤ module  │  │
│  └────────┘  │    (open line system only —        │  └────────┘  │
│              │     no transponders!)               │              │
└──────────────┘                                   └──────────────┘
```

**The Open Line System (OLS) Model:**

IPoDWDM enables a new disaggregated architecture:

1. **Router** with ZR/ZR+ pluggables provides the coherent signal
2. **Open Line System** provides only amplification and ROADM switching
3. **No transponder shelves** — eliminated entirely
4. **Management**: Router manages the coherent optics natively via OpenConfig

**When IPoDWDM makes sense vs. traditional:**

| Factor | Traditional (Transponder) | IPoDWDM (ZR/ZR+) |
|--------|--------------------------|-------------------|
| **Reach** | Unlimited (with regen) | 120 km (ZR), ~2000 km (ZR+, lower rate) |
| **Capacity per λ** | 400G-800G+ | 400G max today |
| **Fiber efficiency** | Better (optimized per span) | Good enough for many routes |
| **Cost** | Higher (transponder shelf) | 30-60% lower (no transponder) |
| **Power** | Higher (transponder + router port) | Lower (single device) |
| **Provisioning** | Slow (cross-team) | Fast (router team only) |
| **Multi-vendor** | Per-layer vendor lock | ZR interop tested |
| **Performance monitoring** | OTN TCM (excellent) | Router-based PM (good, improving) |
| **Long-haul/submarine** | Required | Not suitable |
| **Metro/DCI** | Overkill | Sweet spot |

### Multi-Layer Traffic Engineering

The real power of packet-optical integration: optimizing traffic across both layers simultaneously.

**Without multi-layer TE:**
- Router sees "links" — has no idea two links share the same fiber conduit
- Optical reroutes without telling the router — suboptimal paths persist
- Capacity additions require manual coordination

**With multi-layer TE:**

**Shared Risk Link Groups (SRLGs):**

The foundational concept. SRLGs encode shared physical resources into the IGP TE database:

```
                    Fiber conduit (SRLG 100)
                    ┌──────────────────────────┐
Router A ───link1───┤                          ├───link1─── Router C
         ───link2───┤  (same physical path!)   ├───link2───
                    └──────────────────────────┘

Without SRLG: CSPF may put primary on link1, backup on link2 → BOTH fail
With SRLG:    CSPF knows link1 and link2 share SRLG 100 → diverse backup
```

**SRLG Sources:**
- **Manual configuration** on router interfaces (traditional)
- **Optical domain advertisement** via BGP-LS or OSPF-TE inter-layer extensions
- **Orchestrator injection** from fiber plant database
- **GIS/fiber plant records** imported into planning tools

**Multi-Layer Path Computation:**

```
┌───────────────────────────────────────┐
│         Multi-Layer PCE               │
│                                       │
│  ┌─────────────┐  ┌───────────────┐  │
│  │ Packet TED   │  │ Optical TED   │  │
│  │ (BGP-LS)     │  │ (YANG/SNMP)   │  │
│  └──────┬──────┘  └───────┬───────┘  │
│         │                  │          │
│  ┌──────▼──────────────────▼───────┐  │
│  │    Unified Path Computation     │  │
│  │                                 │  │
│  │  Constraint: Minimize cost      │  │
│  │  Subject to: latency < 5ms     │  │
│  │             SRLG diverse        │  │
│  │             BW >= 100G          │  │
│  └─────────────────────────────────┘  │
└───────────────────────────────────────┘
         │
         ▼
  Decision: "Create new optical path on ROADM ring B
             to offload congested link A-C"
```

**Multi-Layer Use Cases:**

1. **Dynamic wavelength provisioning**: PCE detects congested packet link → requests new wavelength from optical layer → new IP link appears automatically
2. **Optical bypass**: High-volume flow between two routers three hops apart → provision direct wavelength → traffic bypasses intermediate routers → saves router forwarding capacity
3. **Maintenance coordination**: Planned fiber cut → orchestrator pre-provisions backup optical paths → reroutes packet traffic → zero impact maintenance
4. **Restoration coordination**: Fiber cut triggers optical protection → orchestrator notifies packet layer "optical restored, path changed" → packet layer adjusts TE tunnels to new latency/SRLG reality

### Control Plane Standards & Protocols

**OpenConfig:**

The industry's winning approach to multi-vendor optical management:

| Model | Purpose |
|-------|---------|
| `openconfig-platform` | Hardware inventory (transceivers, line cards) |
| `openconfig-terminal-device` | Coherent optic configuration (modulation, FEC, frequency) |
| `openconfig-transport-line-protection` | APS optical protection |
| `openconfig-wavelength-router` | ROADM media channel configuration |
| `openconfig-optical-amplifier` | EDFA/Raman gain, tilt, mode |
| `openconfig-channel-monitor` | Per-channel power monitoring (OCM) |

**OpenROADM:**

MSA (Multi-Source Agreement) defining interoperable ROADM networks:

- **Device models**: YANG models for every optical element (ROADM, amplifier, pluggable, OTN switch)
- **Network model**: Topology, service, routing constraint models
- **Service model**: End-to-end wavelength service provisioning
- **Current version**: OpenROADM 14.0 (2024)
- **Adopters**: AT&T (primary driver), Orange, Telefonica, Vodafone

**Transport API (TAPI):**

ONF's abstraction layer for transport network services:

```
       ┌─────────────────────────────────────┐
       │          TAPI Client (OSS/BSS)       │
       │    "Give me 100G from A to Z"        │
       └──────────────┬──────────────────────┘
                      │ TAPI NBI
       ┌──────────────▼──────────────────────┐
       │     Transport SDN Controller         │
       │     (ONOS, OpenDaylight, vendor)     │
       └──────────────┬──────────────────────┘
                      │ SBI (NETCONF/RESTCONF)
              ┌───────┴───────┐
              │               │
       ┌──────▼─────┐  ┌─────▼──────┐
       │  Optical    │  │  Packet    │
       │  Domain     │  │  Domain    │
       └────────────┘  └────────────┘
```

**TAPI Services:**
- Topology Service (discover + abstract multi-layer topology)
- Connectivity Service (create/modify/delete connections)
- Path Computation Service (constrained path computation)
- OAM Service (performance monitoring, fault management)
- Notification Service (alarm/event streaming)

### Alien Wavelengths

A critical deployment model: one vendor's transponder/pluggable lighting a wavelength over another vendor's DWDM line system.

**Concept:**
```
Vendor A Router (with ZR+)
         │
         │  Alien wavelength (Vendor A coherent signal)
         │
  ┌──────▼──────────────────────────────────────────┐
  │  Vendor B Open Line System                       │
  │  (amplifiers + ROADMs — no transponders)         │
  │                                                  │
  │  ┌────┐    ┌────┐    ┌────┐    ┌────┐           │
  │  │EDFA│────│ROADM│────│EDFA│────│ROADM│          │
  │  └────┘    └────┘    └────┘    └────┘           │
  └──────────────────────────────────────────────────┘
         │
         │  Alien wavelength received
         │
Vendor A Router (with ZR+)
```

**Alien wavelength challenges:**
- **Power management**: Line system doesn't know the alien signal's OSNR requirements → manual per-channel power leveling or optical monitoring feedback loops
- **Performance monitoring**: Line system sees power, not pre-FEC BER → limited visibility into signal quality
- **Wavelength assignment**: Must coordinate channel plan (frequency, spacing) between alien source and line system
- **Fault isolation**: "Is it my pluggable or your amplifier?" → need clear demarcation and PM exchange
- **Fiber characterization**: Alien transceivers may not work on all spans (loss budget, CD, PMD)

**Best practices for alien wavelengths:**
1. Characterize the optical path before deployment (OSNR, CD, PMD margin)
2. Use OpenConfig `openconfig-channel-monitor` on the line system for per-channel power tracking
3. Establish per-FEC BER thresholds on the alien transceiver (pre-FEC BER > 1e-3 = investigation)
4. Document power levels at every amplifier/ROADM for baseline
5. Use wavelengths near the center of the EDFA gain band (avoid edges where gain flatness degrades)

## Configuration

### IOS-XR: Coherent Pluggable (400ZR/ZR+)

```cisco
!! NCS 5500/5700 with QSFP-DD 400ZR
!! Configure the coherent optics on the port
!!
!! NOTE: 400ZR has FIXED modulation (dp-16QAM) and FEC (oFEC) per the OIF spec.
!! These are not configurable — the optic IS dp-16QAM with oFEC, period.
!! For ZR+ pluggables, use operational-mode <id> to select rate/modulation.

controller Optics 0/0/0/0
 !
 ! Transmit power (dBm) — tune based on span loss budget
 transmit-power -100    ! -10.0 dBm (value in 0.1 dBm units)
 !
 ! Wavelength/frequency selection
 dwdm-carrier 100MHz-grid frequency 1931000  ! 193.1 THz (ITU C-band center)
 !
 ! For ZR+ (multi-rate): select operational mode instead of fixed ZR
 ! operational-mode 1     ! Vendor-specific mode ID (check optics data sheet)
 !                        ! Mode determines: modulation, FEC, baud rate
 !
 ! Performance monitoring thresholds
 pm 30-sec optics threshold opt-min -280     ! -28.0 dBm Rx power minimum
 pm 30-sec optics threshold opt-max 10       ! +1.0 dBm Rx power maximum
 pm 30-sec optics threshold cd-min -2000     ! Chromatic dispersion min (ps/nm)
 pm 30-sec optics threshold cd-max 2000      ! Chromatic dispersion max
 pm 30-sec optics threshold dgd-max 30       ! DGD (PMD) max in ps
!

!! Associated Ethernet interface
!! NOTE: 400ZR QSFP-DD creates a FourHundredGigE interface, NOT HundredGigE
interface FourHundredGigE 0/0/0/0
 ipv4 address 10.0.12.1/30
 mtu 9216
 ! Add to IGP, MPLS, etc. as normal
!

!! Verify coherent state
! show controllers optics 0/0/0/0
! show controllers coherentDSP 0/0/0/0
```

### IOS-XR: SRLG Configuration

```cisco
!! Define SRLGs — configured under GLOBAL srlg block, NOT under the interface!
!! SRLG value = arbitrary 32-bit number (use a naming convention!)
!! Convention: SRLG 1000-1999 = fiber conduits
!!             SRLG 2000-2999 = duct routes
!!             SRLG 3000-3999 = building entries

srlg
 interface FourHundredGigE 0/0/0/0
  value 1001     ! Chicago-Dallas fiber conduit #1
  value 2001     ! I-35 corridor duct
 !
 interface FourHundredGigE 0/0/0/1
  value 1001     ! Same conduit — CSPF will avoid co-routing!
  value 2001     ! Same duct
 !
!

!! Enable SRLG advertisement in IS-IS
router isis CORE
 address-family ipv4 unicast
  advertise link attributes    ! Advertises SRLGs via IS-IS TE TLVs
 !
!

!! Enable SRLG advertisement in OSPF
router ospf CORE
 mpls traffic-eng router-id Loopback0
 mpls traffic-eng area 0
 ! SRLGs auto-advertised with OSPF TE opaque LSAs
!

!! SRLG-aware RSVP-TE tunnel — use attribute-set for SRLG exclusion
mpls traffic-eng
 attribute-set path-option SRLG-DIVERSE
  exclude-srlg                ! Excludes all SRLGs used by primary path
 !
!

interface tunnel-te 100
 ipv4 unnumbered Loopback0
 destination 10.0.0.5
 path-option 10 dynamic
 path-option 20 dynamic attribute-set SRLG-DIVERSE   ! SRLG-diverse backup
!

!! SRLG-aware SR-TE policy (preferred modern approach)
segment-routing
 traffic-eng
  policy SR-POL-DALLAS
   color 100 end-point ipv4 10.0.0.5
   candidate-paths
    preference 200
     dynamic
      metric type igp
     !
    !
    preference 100
     dynamic
      metric type igp
      disjoint-path group-id 1 type srlg   ! SRLG-diverse from group peer
     !
    !
   !
  !
 !
!
```

### Junos: Coherent Pluggable (400ZR/ZR+)

```junos
# MX/PTX with QSFP-DD 400ZR/ZR+
# NOTE: 400ZR modulation/FEC are fixed (dp-16QAM, oFEC) per OIF spec.
# ZR+ supports tunable modes — select via optics-options.

# Configure coherent optics under the interface (NOT under chassis)
interfaces {
    et-0/0/0 {
        optics-options {
            # Wavelength/frequency (193.1 THz)
            wavelength frequency 193.10thz;
            # Transmit power in dBm
            tx-power -10.00;
            # For ZR+ (multi-rate), modulation is selectable:
            # modulation dp-qpsk;       # 100G mode (longer reach)
            # modulation dp-16qam;      # 400G mode
        }
        mtu 9192;
        unit 0 {
            family inet {
                address 10.0.12.1/30;
            }
            family iso;
            family mpls;
        }
    }
}

# Verify
# show interfaces diagnostics optics et-0/0/0
# show interfaces et-0/0/0 media
```

### Junos: SRLG Configuration

```junos
# Define SRLG membership on interfaces
interfaces {
    et-0/0/0 {
        srlg {
            # SRLG name-based (Junos uses names, not just numbers)
            chicago-dallas-conduit-1;
            i35-corridor;
        }
    }
    et-0/0/1 {
        srlg {
            chicago-dallas-conduit-1;    # Same conduit!
            i35-corridor;
        }
    }
}

# SRLG definitions with numeric values (for interop with IOS-XR)
routing-options {
    srlg {
        chicago-dallas-conduit-1 {
            srlg-value 1001;
        }
        i35-corridor {
            srlg-value 2001;
        }
    }
}

# IS-IS SRLG advertisement
protocols {
    isis {
        traffic-engineering;       # SRLGs advertised automatically with TE enabled
    }
}

# SRLG-diverse RSVP-TE LSPs
protocols {
    mpls {
        label-switched-path to-dallas-primary {
            to 10.0.0.5;
            # CSPF computes primary path normally
        }
        label-switched-path to-dallas-diverse {
            to 10.0.0.5;
            # Exclude specific SRLGs from path computation
```

### Key Knobs

| Parameter | Default | Recommended | Why |
|-----------|---------|-------------|-----|
| Tx Power | Varies | Per span design | Over-power = nonlinear impairment; under-power = low OSNR |
| Frequency | None | Per channel plan | Must match ROADM/filter allocation |
| FEC | Vendor default | oFEC (ZR), SD-FEC (ZR+) | Match transceiver standard |
| Modulation | dp-16qam | Match reach requirement | Lower order = longer reach, less capacity |
| SRLG | None | Always configure! | Without SRLG, diverse paths aren't diverse |
| PM thresholds | Vendor default | Set Rx power, CD, DGD | Early warning before hard failure |
| CD compensation | Auto | Auto (verify) | Modern coherent DSP handles this — verify range covers your span |
| Loopback | Disabled | Disabled (enable for test) | Never leave coherent loopback enabled in production |

**Red flags to watch:**
- Pre-FEC BER > 1e-3: Signal degrading — investigate span loss, amplifier, or connector

- Q-factor < 8.5 dB: Marginal link — will fail with any additional impairment
- OSNR < 20 dB: Low margin (for dp-16qam) — check amplifier chain
- DGD > 10 ps: High PMD — check fiber (old fiber, aerial spans, temperature)
- CD outside ±2000 ps/nm: Unusual — check fiber type and span length

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Coherent optic won't lock | Frequency mismatch between Tx and Rx | Verify both ends configured for same ITU channel |
| High pre-FEC BER | Span loss exceeds budget, dirty connector, amplifier degradation | Check OSNR, clean connectors, verify amplifier gain |
| Link flapping | PMD on temperature-sensitive spans (aerial fiber) | Check DGD trending, consider reroute or fiber replacement |
| SRLG not advertised | TE extensions not enabled in IGP | Enable `traffic-engineering` in IS-IS/OSPF |
| BGP-LS session up but no topology | Missing `distribute link-state` in IGP | Add `distribute link-state` under `router isis` or `router ospf` |
| Diverse path still shares risk | SRLG not configured on one interface | Audit all interfaces in the path — SRLG must be on every link |
| ZR optic shows "unsupported" | Platform/IOS-XR version doesn't support CMIS for that pluggable | Check release notes for pluggable PID support matrix |
| Post-FEC errors present | Pre-FEC BER very close to FEC limit | Degraded link — reduce modulation order or fix underlying optical issue |
| Wavelength blocked at ROADM | ROADM channel not provisioned or wrong port | Verify ROADM cross-connect matches alien wavelength plan |
| Intermittent errors correlated with time of day | Thermal effects on aerial fiber (CD/PMD shift with temperature) | Monitor CD/PMD vs temperature, increase margin or reroute |

### Methodology

**Coherent optic troubleshooting flow:**

```
1. Is the optic recognized?
   └─ show inventory / show platform → CMIS/module info
   └─ If not: check firmware, PID support, seating

2. Is the laser on?
   └─ show controllers optics → Laser State
   └─ If not: check admin state, tx-shutdown, power supply

3. Is the frequency correct?
   └─ Both ends must match to within the grid tolerance
   └─ Common mistake: GHz vs MHz vs THz unit confusion

4. Is the signal reaching the far end?
   └─ Check Rx power at far end vs Tx power at near end
   └─ Delta = total span loss (should match link budget)
   └─ If Rx too low: check connectors, amplifiers, ROADM attenuation

5. Is the pre-FEC BER acceptable?
   └─ < 1e-4: Healthy
   └─ 1e-4 to 1e-3: Marginal — monitor closely
   └─ > 1e-3: Degraded — investigate OSNR, impairments
   └─ > FEC threshold: Errors passing through — service impact

6. Are there impairments?
   └─ CD > expected for fiber length: wrong fiber type or unexpected routing
   └─ DGD > 10 ps: PMD issue (old fiber, connectors, splices)
   └─ OSNR < required: amplifier noise, filter concatenation, loss

7. Is it intermittent?
   └─ Correlate with temperature (thermal PMD, fiber expansion)
   └─ Correlate with traffic (nonlinear impairments at high channel loading)
   └─ Correlate with maintenance (someone touching the fiber?)
```

**Multi-layer fault correlation:**

```
Router sees link down
  │
  ├─ Check optical controller state → "Rx Loss of Signal"
  │   │
  │   ├─ Far-end Tx laser still on? → Check far-end optic
  │   │
  │   └─ Far-end laser on but no signal → Fiber/path issue
  │       │
  │       ├─ ROADM alarms? → Cross-connect issue or ROADM failure
  │       │
  │       ├─ Amplifier alarms? → EDFA pump degradation, gain tilt
  │       │
  │       └─ No optical alarms? → Physical fiber cut or dirty connector
  │           └─ OTDR trace to locate break
  │
  └─ Controller shows coherent errors (not LOS)
      │
      ├─ Gradual BER increase → Amplifier aging, connector degradation
      │
      └─ Sudden BER spike → New channel added (XPM/FWM), 
                             equipment vibration, 
                             or environmental (temperature, water ingress)
```

### War Story: The Multi-Layer Restoration Loop

> A Tier-1 SP had their optical layer configured with 1+1 line protection and their routers running SR-TE with FRR. During a fiber cut on a major metro ring:
> 
> 1. Optical 1+1 protection switched in <50ms — traffic restored on the protect fiber
> 2. But the protect path was 15km longer, adding ~75μs latency
> 3. BFD on the router (configured for 3×10ms) didn't see the 50ms glitch — no packet-layer event
> 4. However, the latency change pushed one SR-TE policy outside its latency constraint (max 2ms for a financial customer)
> 5. The router's periodic TE re-optimization (every 60 seconds) detected the constraint violation
> 6. Router rerouted the SR-TE policy to a longer but lower-latency path through a different city
> 7. This path was on the SAME fiber conduit that was cut — but the optical 1+1 hadn't been configured on that route
> 8. Customer suffered a 60-second outage despite two layers of protection
>
> **Root cause**: No SRLG configuration. The router didn't know the "diverse" path shared the same physical conduit. After adding SRLGs and excluding them in SR-TE constraints, the re-optimization chose a genuinely diverse path.
>
> **Lesson**: Multi-layer protection without SRLG awareness is security theater. You think you're protected, but physics doesn't care about your control plane.

## Design Considerations

### Architecture Decision Framework

**When to use each integration model:**

| Factor | Ships in Night | Multi-Layer Coord. | GMPLS/UNI | IPoDWDM (ZR/ZR+) |
|--------|---------------|-------------------|-----------|-------------------|
| **Network size** | Any | Large SP | Large SP | Metro/DCI/Regional |
| **Vendor diversity** | Multi-vendor | Multi-vendor | Single optical vendor | Multi-vendor (OIF interop) |
| **Provisioning speed** | Days-weeks | Minutes-hours | Minutes | Seconds (router config only) |
| **CAPEX** | Highest (transponders) | High (orchestrator license) | Medium | Lowest |
| **OPEX** | Highest (manual) | Medium (automated) | Medium-Low | Lowest |
| **Reach** | Unlimited | Unlimited | Unlimited | 120 km (ZR), ~2000 km (ZR+) |
| **PM/OAM depth** | Best (OTN TCM) | Good | Good | Adequate (improving) |
| **Maturity** | Proven (20+ yr) | Mature (5-10 yr) | Mature but declining | Rapidly maturing |
| **Best for** | Legacy, regulated | Large-scale SP | Single-vendor optical | Greenfield, DCI, metro |

### IPoDWDM Migration Strategy

For SPs transitioning from traditional to IPoDWDM:

```
Phase 1: DCI First (Low Risk)
├─ Deploy ZR on data center interconnects (point-to-point, short reach)
├─ Keep long-haul on traditional transponders
├─ Build operational experience with coherent pluggable management
└─ Timeline: 0-12 months

Phase 2: Metro Expansion
├─ Extend ZR+ to metro ring/mesh (amplified, multi-span)
├─ Deploy open line systems (new builds) alongside legacy DWDM
├─ OpenConfig-based management for all new coherent optics
└─ Timeline: 12-24 months

Phase 3: Regional Reach
├─ ZR+ with lower modulation orders for regional (500-2000 km)
├─ Evaluate 800ZR (next-gen form factor) for higher capacity
├─ Begin decommissioning transponder shelves on migrated routes
└─ Timeline: 24-36 months

Phase 4: Full Convergence (Aspirational)
├─ Router-native coherent for all metro + regional
├─ Traditional transponders only for ultra-long-haul and submarine
├─ Full multi-layer SDN orchestration
└─ Timeline: 36+ months
```

### SRLG Design Best Practices

1. **Naming convention**: Use meaningful names that map to physical assets
   - `CHI-DAL-CONDUIT-001` → Chicago-Dallas fiber conduit #1
   - `I35-CORRIDOR` → Interstate 35 right-of-way
   - `EQUINIX-CH1-ENTRY` → Building entrance at Equinix Chicago
   
2. **Granularity**: Define SRLGs at the most granular shared-risk level
   - Same conduit → same SRLG
   - Same duct route → additional SRLG
   - Same building entrance → additional SRLG
   - Same CO/PoP → additional SRLG (for total site failure)
   
3. **Value ranges**: Reserve ranges for categories
   - 1000-1999: Fiber conduits
   - 2000-2999: Duct routes / rights-of-way
   - 3000-3999: Building entries / meet-me rooms
   - 4000-4999: PoP/CO shared risk
   
4. **Automation**: Import SRLGs from fiber plant GIS database
   - Don't maintain manually at scale — you WILL miss one
   - Use NETCONF/RESTCONF to push SRLG configs from source-of-truth

5. **Validation**: Regularly audit that SRLGs match physical reality
   - After fiber reroutes, SRLGs may be stale
   - Test with `show mpls traffic-eng topology path destination X exclude srlg` (IOS-XR)
